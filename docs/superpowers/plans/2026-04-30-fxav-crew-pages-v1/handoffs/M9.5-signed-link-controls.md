## 6. Watchpoints from prior adversarial review

These are pre-loaded concerns the implementer should expect Codex to raise; addressing them up front saves rounds.

1. **Single-holder advisory-lock topology.** The M5 R20 CRITICAL deadlock was caused by nested holders (JS wrapper + in-RPC). Both M9.5 RPCs MUST be the sole lock holder for their hashkey on their invocation path. The writing-plans session must enumerate the existing holders on each affected hashkey and confirm M9.5 is the sole holder for these specific paths. Extend the structural meta-test.
2. **Floor-bump atomicity.** The "Issue new link" UPDATE atomically bumps both `current_token_version` AND `max_issued_version` (spec §5.2 line 1095-1097). A non-atomic write — e.g., separate UPDATEs — leaves a window where `current_token_version > max_issued_version` (impossible per the invariant), or `current_token_version <= revoked_below_version` (the no-live-link state). Single-statement UPDATE only.
3. **Realtime invalidation timing.** The composite `viewer_version_token` race was fixed in M3+M4 batch-14 via the `crew_member_auth.last_changed_at` BEFORE-UPDATE trigger. Both new RPC UPDATEs fire that trigger by touching the row. The integration test must assert this is still true — a "shortcut" UPDATE that bypasses the trigger (e.g., direct catalog manipulation) would silently break realtime.
4. **No-live-link state UI affordance gating.** Spec line 1100: "It does NOT show 'Copy share link' until the row leaves the no-live-link state." Even though M9.5 doesn't ship copy-share, the state-machine condition (no-live-link ↔ `current_token_version === revoked_below_version`) must be computed and exposed so the BACKLOG copy-share work can consume it correctly. The "Revoke all" button must be **disabled** in no-live-link state (nothing to revoke); "Issue new link" is the only live action.
5. **Leaked-link middleware interaction.** Spec §7.2 Branch A (lines 1959-1975) handles the case where `?t=` leakage triggers an auto-floor-bump into the no-live-link state. The M9.5 admin UI must render correctly when a row arrives at no-live-link via the middleware path (not just via admin click). Existing `tests/auth/leakedLinkMiddleware.test.ts` is the pin; M9.5 may need to extend it with a UI-render assertion or add a sibling test.
6. **Admin-log structured entry.** Per the `BL-OPS-LOG` BACKLOG entry, durable admin-facing operator logging is not yet implemented. M9.5 can either (a) emit via `admin_alerts` (informational severity) which IS shipped, OR (b) emit via `console.error` with structured payload (acceptable until BL-OPS-LOG lands). The writing-plans session picks one; emitting nothing is not acceptable for security-sensitive admin actions.
7. **Two-tap confirm UX parity.** The M9 C4 R3 / C9 R10 lessons on `useFormStatus` (M9-D-C4-1 memory) apply: the Revoke confirm button derives "Revoking…" state from `useFormStatus().pending`, not a local UI state flag. Mirrors `components/admin/ResolveAlertButton.tsx` and `app/admin/settings/admins/RevokeRowButton.tsx`.
