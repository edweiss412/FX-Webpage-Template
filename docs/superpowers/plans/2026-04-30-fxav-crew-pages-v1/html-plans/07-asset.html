Imagine the crew showing up to a convention center basement with one bar of LTE. Stage diagram on the page? Has to load. Opening reel video? Has to play. Agenda PDF? Has to open. And the file Doug embedded in the sheet last Tuesday must look exactly like it did last Tuesday — even if he replaced it with a new version this morning.

So at every Apply we make a copy of every linked file into our own storage, tag the copy with the exact Drive revision it came from, and serve it through our own URL. Old copies get cleaned up by a janitor job a week later. If something went wrong during that copy, a recovery mode tries again — without ever touching the bytes that are already approved.

M7 · One image from Drive to the crew's phone

Why we copy instead of hot-linking

A simpler design would just stick Drive URLs into the crew page and let the phone fetch them directly. Three things kill that idea:

Venue wifi
Conference centers throttle outbound TLS. Cold-fetching a 50MB stage plot from Drive at showcall is a recipe for a black tile.
Drive permissions
Doug shares files differently across shows. A file world-readable last month becomes "anyone in FXAV" this month. The crew member opening the link is not in FXAV's Google org and gets a permission error.
Doug edits things
The whole point of "Apply" is that the operator approved these exact bytes. If Doug replaces stage-plot.png mid-show with the wrong version, the crew page must still show the approved one.
Invariant · immutable pin tuple

Every linked asset is pinned to a content-derived identifier — not a timestamp. Linked-folder files pin to (headRevisionId, md5Checksum); embedded sheet images pin to (sheetsRevisionId, base64url(SHA-256(bytes))). These tuples are byte-stable: equal inputs always produce equal outputs. Apply re-verifies the pin under the per-show advisory lock before any byte ever lands in canonical storage.

Why not drive_modified_time? Because there's a TOCTOU window between reading the modtime and downloading the bytes. The newly-edited file would get snapshotted as the "approved" version even though Apply was approving the older one. Modtime is informational. The pin is the fence.

The two-prefix dance

Every Apply writes to a temp prefix first. Only after every byte is on storage AND the DB transaction commits does a post-commit promoter rename the temp prefix to the canonical prefix in one atomic JSONB cutover. If anything fails — Storage 5xx, manifest mismatch, race with another Apply for the same show — the promoter reverse-renames everything back to temp. The canonical prefix is either fully populated with the new revision or untouched. A partial gallery is never live.

The ledger row, decoded

Every Apply attempt inserts exactly one row into pending_snapshot_uploads. That row is the rendezvous point for the post-commit promoter, the recovery cron, the GC janitor, and the stuck-rollback repair endpoint. Three timestamp markers do all the coordination — together they form a 3-state lifecycle the GC sweep uses to know whether bytes are touchable.

StateMarker shapeWho's responsibleGC behavior
uploaded promoted_at NULL · delete_started_at NULL · promote_started_at NULL Phase-2 just committed; promoter hasn't started yet (or rolled back cleanly). Eligible for claim by promoter OR GC after 1h.
promoting promote_started_at NOT NULL · promoted_at NULL Post-commit promoter is mid-Storage-rename. Heartbeats claim_expires_at. Refuses to reclaim or delete. The marker is the byte-protection invariant.
committing_delete delete_started_at NOT NULL GC has committed to delete this row's temp prefix; Storage DELETE is running. Refuses reclaim. Another worker cannot double-delete the same bytes.
promoted promoted_at NOT NULL Cutover landed; canonical revision advanced. Swept after 24h (keep for forensics, then DELETE the row).
stuck promote_started_at < now - '15min' · promoted_at NULL Operator-only via /api/admin/snapshot-rollback/[id]/repair. Refuses to touch — emits PENDING_SNAPSHOT_ROLLBACK_STUCK admin alert.

M7 invariants — never violated past this milestone

  1. Content-derived pins only. Approval evidence is SHA-256(bytes) or md5Checksum. NEVER an HTTP ETag, NEVER a HEAD-derived token, NEVER a modtime.
  2. The asset route reads only current. The pending JSONB sub-payload is the cutover staging slot and is invisible to crew. Literal equality on current.snapshot_revision_id is the only authoritative gate.
  3. Any missing-bytes outcome is 410, never 404. Single contract — caches invalidate consistently, regardless of cause (drift, GC, revoked link, never-existed).
  4. Per-show advisory lock is mandatory for every write path. Apply, asset_recovery, and the admin repair endpoint all acquire show:; the post-commit promoter also takes promote:. No two writers ever race for the same show.
  5. A partial gallery is never live. Promote is all-or-nothing across the manifest; rollback reverse-renames every successfully renamed asset back to temp before clearing pending.

The pin tuple, in code

// lib/sync/types.ts — per-entry approval evidence

export type LinkedFolderItem = {
  driveFileId: string;
  mimeType: 'image/png' | 'image/jpeg' | 'image/gif' | 'image/webp';
  alt: 'media';
  drive_modified_time: string;     // informational only — NEVER used as a fence
  headRevisionId: string;          // immutable per-revision Drive token
  md5Checksum: string;             // content hash — re-verified at Apply
  snapshotPath: string | null;     // 'diagram-snapshots/shows///' once promoted
};

export type EmbeddedImage = {
  sheetTab: string;                // resolved DIAGRAMS tab title (case-preserved)
  objectId: string;                // Sheets API embedded-object id
  mimeType: string;
  sheetsRevisionId: string;        // last entry of drive.revisions.list(spreadsheetId)
  embeddedFingerprint: string | null;        // base64url(SHA-256(bytes)) — null only if contentUrl unreachable
  recovery_disposition: 'normal' | 'restage_required';
  snapshotPath: string | null;
};

// shows.diagrams JSONB — two sub-payloads
export type DiagramsJsonb = {
  current: { snapshot_revision_id: string; embeddedImages: EmbeddedImage[];
             linkedFolderItems: LinkedFolderItem[]; snapshot_status: 'complete' | 'partial_failure' | 'partial_failure_restage_required'; };
  pending: { revision_id: string; embeddedImages: EmbeddedImage[];
             linkedFolderItems: LinkedFolderItem[]; snapshot_status: 'complete' | 'partial_failure'; } | null;
};
