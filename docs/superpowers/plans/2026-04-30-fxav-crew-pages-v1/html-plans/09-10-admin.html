Before this pair of milestones, the app technically works: sheets parse, the database fills, crew can read their personal pages. But there's no front door for a new operator, no place to retry the sheets that wouldn't parse, no consistent voice when something goes wrong. M9 and M10 fix all three.

M9 is the quality polish. The stale-data footer at the bottom of every crew page reads from a single message catalog. Every tile gets two error boundaries — one for server-render throws, one for client crashes — so a single broken section never takes down a whole page. Every empty state has a screenshot baseline.

M10 is the front door. When Doug first signs up, the system walks him through pointing at his Google Drive folder — share it, scan it, review the first sheets, then atomically promote it to "live." Once he's set up, /admin becomes his command center for fixing things when sheets are weird: a dashboard of active shows, a banner of unresolved alerts, a per-show parse panel, and a preview-as-crew impersonation surface.

Plan invariant 5 · No raw error codes in user-visible UI

Every code the system raises — DRIVE_FETCH_FAILED, STAGED_PARSE_OUTDATED, ONBOARDING_FOLDER_NOT_SHARED, all 70-odd of them — passes through lib/messages/lookup.ts before it ever reaches a person. The catalog returns the right voice for the audience: crew get short, human copy; Doug gets a longer "what does this mean?" explanation. Raw code text is admin-log-only. M9 builds the catalog; M10 wires the explainer.

Server vs client error paths — a tale of two boundaries

The most subtle bug Task 9.2 closes is also the easiest to miss in review: a developer adds await fetchExtraData() inside the tile's view component for a "quick" enhancement, the loader passes, the view throws, and because React invokes view components after the wrapper's try/catch returns, the throw escapes all the way to the route-level error.tsx. The plan adds a static-analysis test that walks every *TileView.tsx and refuses await keywords or imports from lib/db/**, lib/drive/**, or lib/sync/** inside view components.

Why two boundaries, not one

Client ErrorBoundary uses componentDidCatch which only fires after the component tree has hydrated. Server Components have no equivalent primitive. The split is necessary, not stylistic. Render-time throws → ErrorBoundary fallback. Handler-time throws → handler converts to error-state render or routes to Sentry. Async data-fetch → must live in the server wrapper's load callback, never inside the view.

M10 Onboarding wizard · Admin console

The three-step wizard

M10 opens with the question every new operator asks: where do I start? The wizard answers it in three screens. None of them touch the live database until the operator clicks Finalize on the last screen — and Finalize itself is a four-phase atomic promotion so a crash mid-way can never half-publish a folder.

M10 · Onboarding wizard flow

What atomic promotion actually means

The wizard reaches its final screen with N approved sheets ready to publish. But "publish" can't be a single big UPDATE — the wizard pre-uploads diagram assets to Storage, the operator may have approved 250 sheets, and a cron tick on the existing live folder can fire mid-promotion. The four-phase protocol handles each concern in isolation: Phase A does Drive re-verify and Storage uploads outside any per-show lock; Phase B commits per-row transactions in alphabetical order each holding the per-show lock for ~200ms; Phase C cleans up temp prefixes best-effort; Phase D is the final CAS that flips published = TRUE for every promoted row in one short transaction with no I/O.

Lock topology · finalize:<sid> vs show:<driveFileId>

Two distinct advisory locks. finalize:<sessionId> serializes /finalize against /finalize-cas and cleanup-abandoned-finalize for the same wizard. show:<driveFileId> is acquired inside each per-row Phase B sub-transaction, never during Phase A. The two locks never nest at the same layer — the single-holder rule from §2 invariants holds.

The admin console once setup is done

Once watched_folder_id is non-null, /admin renders the dashboard instead of the wizard. Same URL, different surface. The dashboard is Doug's command center for fixing things when sheets are weird.

M9 · M10 invariants — never violated past this milestone

  1. Every UI render-site that surfaces an error code calls messageFor(code, params). Raw code text is admin-log-only — never in crew-facing or Doug-facing UI.
  2. Server-component data-fetches throw inside <TileServerFallback load={…}>, never inside the pure view component. Static analysis enforces this.
  3. Onboarding writes are scoped by wizard_session_id; live writes by wizard_session_id IS NULL. The two partitions never share a row.
  4. Folder promotion is the four-phase atomic protocol. Phase D is the SINGLE CAS that flips watched_folder_id and published = TRUE; nothing else does either.
  5. The finalize:<sessionId> advisory lock serializes /finalize, /finalize-cas, and cleanup-abandoned-finalize. The per-show show:<driveFileId> lock is only acquired inside Phase B sub-transactions.
  6. Preview-as-crew uses identity-only { kind: 'admin_preview', crewMemberId }. Role flags re-derive server-side; the helper never accepts a caller-supplied role object.

Acceptance criteria

  • AC-9.1Stale-data footer renders with catalog-bound copy across every last_sync_status branch and every relative-time tier. Status precedence wins over age.
  • AC-9.2Every empty state defined in §8.3 has a Playwright screenshot baseline reachable from at least one fixture.
  • AC-9.3Per-tile server and client error boundaries — a tile data-fetch throw produces a tile fallback, not a route-level error.
  • AC-10.1First-visit /admin renders the wizard inline (both watched_folder_id AND pending_wizard_session_id NULL). No redirect to a non-existent /admin/onboarding.
  • AC-10.2Wizard step 2 surfaces every documented success/failure message via the catalog: invalid URL, folder-not-shared, operator-error, session-superseded.
  • AC-10.3Wizard step 3 renders all three status classes — parsed-and-ready, couldn't-parse, skipped-non-sheet — and Finalize blocks until every row is resolved.
  • AC-10.4Re-run Setup opens the wizard with empty pending_* while watched_folder_id stays untouched.
  • AC-10.5Mid-wizard abandonment doesn't blackout the live folder; cron continues against the existing watched_folder_id. The next Re-run Setup overwrites pending state.
  • AC-10.6Stale onboarding Apply re-scans inline. Edit a sheet during step 3, click Apply, the route re-verifies Drive modtime and re-stages the parse.
