M5

Tap the link. You're in.

No sign-up screen. No password reset email. No app to download. Crew members receive a personalized URL from Doug, tap it once, and the right show — with the right fields visible for their role — opens on their phone. M5 is the machinery that makes "tap the link, you're in" actually safe.

How does the system know who you are without making you sign up for an account? It doesn't, really — Doug already knows who you are. He has your name and your email in his show sheet. M5's job is to translate Doug's I know who this person is into a browser session, securely, without bothering you to make or remember anything.

There are two front doors. The signed link — Doug texts or emails you a URL with a tiny encrypted ticket baked into it. You tap, the page reads the ticket, hands you back a cookie, and forgets the ticket. The Google sign-in — for admins and anyone with an @fxav.net address — clicks through Google's "choose an account" screen. Either way, on the way in, the system asks the same question: which show is this, and which row in that show's crew list is you?

A The signed-link flow

From Doug's outbox to a rendered crew page

The signed-link path is the load-bearing piece of M5. It's how every crew member who isn't on Google's enterprise plan gets in. The whole flow is one round-trip — Doug sends a URL, the crew member taps it, the browser holds a cookie for the next 12 hours.

M5 · signed-link sequence — five lanes, one round-trip
The cookie is NOT the JWT

The signed link's JWT is exchanged exactly once, at the bootstrap shell. What ends up in the browser is an opaque UUID wrapped in a tiny JSON envelope ({ v: 1, token, show_id }) inside a __Host-fxav_session cookie. The JWT itself never lives in any cookie, never re-verifies on later requests. From here on, the validator only looks up the opaque token in link_sessions — twelve steps deep.

B validateLinkSession · the 12-step gate

Twelve checks before every page render

Every request to /show/<slug> from a cookie-bearing browser runs through this gate. It is the highest-stakes function in M5: failing any one step in the wrong direction is a CVE. Each step has a specific status code, a specific catalog message, and — for the failure paths — a mandatory side effect of deleting the offending link_sessions row so the cookie can't be reused.

12 steps · stops at the first match
C The auth chain · admin first

Three credentials, in this order

A request to /show/<slug> may carry an admin role, a Google session, a signed-link cookie, or any combination. The chain runs in a specific order — admin first, before the cookie validator gets a chance to short-circuit — because an admin clicking a share link during preview is still an admin. The single shared predicate isAdminSession() guards the runtime branch and the X.3 static audit, so the two can never disagree.

Invariant · admin precedence over crew-on-self

When the same person is both an admin and a row in crew_members for the same show, the admin role wins. Always. An earlier draft had the link cookie short-circuit ahead of the admin check; that silently downgraded admins to crew-mode every time they opened a preview link. The corrected chain runs isAdminSession first as a pure predicate, then validateLinkSession, then validateGoogleSession, then a final requireAdmin fallback for non-OAuth admin shapes.

The chain · first success wins
D When a link leaks · the ?t= handler

Two branches, one canonical code

A signed link in a URL fragment (#t=…) is invisible to servers and logs — that's the safe form. A signed link in a query string (?t=…) is logged everywhere — Vercel, proxies, browser history, screenshots. Any request with ?t= on any path under /show/<slug> is treated as a confirmed compromise event.

Branch A · no-live-link

Leaked token IS the current version

Insert a revoked_links row at the exact version. Set revoked_below_version = current_token_version. Do not auto-mint a fresh version — the spec demands Doug explicitly click "Issue new link" before any shareable token exists.

410 LEAKED_LINK_DETECTED
Branch B · surgical revoke only

Leaked token is OLDER than current

Insert a revoked_links row at the leaked version only. crew_member_auth is unchanged — the live current link stays usable. An old leak shouldn't kick everyone off the current version.

410 LEAKED_LINK_DETECTED
Branch C · lifted floor

Leaked token is from the FUTURE

Anomaly. Lift current, max, and floor all to the leaked version in one step so a single "Issue new link" click clears the floor. The retired half-fix needed multiple clicks — a real auth-lockout bug.

structured log only
Middleware scope · every /show/** path

Earlier drafts narrowed the ?t= scan to /show/[slug]/p. That was wrong: Vercel logs the full URL for every path the platform serves, and a copy-paste leak as /show/<slug>?t=... would bypass revocation. The middleware now matches ^/show/[^/]+ regardless of subroute — root crew page, bootstrap path, trailing-segment typos, and any future /show/<slug>/<new-thing> route.

E Google OAuth · admins and @fxav.net

The second front door

Admins and any crew member with an @fxav.net email sign in through Google. The flow is the standard PKCE OAuth dance via Supabase Auth. Two subtle pieces matter: a next-param canonicalizer that refuses to redirect anywhere off-app, and a /me page that uses a separate validator (validateGoogleIdentity) — because /me has no show parameter, and calling the show-bound validateGoogleSession from a show-less route would be a structural error.

validateNextParam · five guards then the allowlist

The OAuth next redirect is an open-redirect attack surface. The validator runs five guards in order — reject non-strings, reject backslashes / percent-encoded dot-segments / control characters, parse via new URL(), reject cross-origin, reject the /show/<slug>/p bootstrap surface — then matches against a canonicalized allowlist. Anything else falls back to /admin.

Acceptance criteria

  • AC-5.1Cookie / show binding: 12-step gate's steps 1–4 reject missing, wrong-show, and cross-show cookies with the right status + side effect.
  • AC-5.2Version mismatch: cookie's jwt_token_version ≠ crew_member_auth.current_token_version → 410 LINK_VERSION_MISMATCH + DELETE row.
  • AC-5.3Cross-show reuse: cookie minted for show A presented at show B → continue + clearCookie + row DELETEd; chain falls through.
  • AC-5.4Revoked floor: jwt_token_version <= revoked_below_version → 410 LINK_REVOKED_FLOOR.
  • AC-5.5Surgical revoke: exact (show_id, crew_name, token_version) match in revoked_links → 410 LINK_REVOKED_SURGICAL.
  • AC-5.6Idle timeout at 15 min → 401 SESSION_IDLE_TIMEOUT (NOT 410). Absolute timeout at 12h → 401 SESSION_ABSOLUTE_TIMEOUT.
  • AC-5.6aGlobal key rotation atomically invalidates redeemed sessions via link_sessions.signing_key_id ≠ app_settings.active_signing_key_id.
  • AC-5.7Google sign-in for an email not on the show's crew list → 403 GOOGLE_NO_CREW_MATCH.
  • AC-5.8Multi-match within the same show → 500 AMBIGUOUS_EMAIL_BINDING + UPSERT admin_alerts with coalesce semantics.
  • AC-5.9Payload introspection: LEAD response includes financials; non-LEAD does not. coi_status is universally visible.
  • AC-5.10Demote LEAD → A1 via DB update; refresh; Financials tile disappears within one render cycle — no token rotation needed.
  • AC-5.11?t= compromise handler runs on every path under /show/<slug>; branches A / B / C produce correct crew_member_auth state.
  • AC-5.12Synthesized duplicate-email runtime condition writes a banner; resolving the alert removes it.
  • AC-5.13Admin precedence: when both admin and crew credentials are present, admin wins; cookie left in place.
  • AC-5.14Google OAuth sign-in / callback / sign-out: PKCE state cleanup on every code path; next-param canonicalized through the five-guard validator; sign-out clears every cookie family atomically.

M5 invariants — never violated past this milestone

  1. The cookie carries an opaque token + show binding, never a JWT. JWTs verify exactly once at redeem time.
  2. Every cookie set/clear flows through lib/auth/cookies.ts. The literal string __Host-fxav_session outside that module fails X.3's banned-identifier scan.
  3. Admin precedence runs first. isAdminSession is a pure predicate shared between the runtime chain and the X.3 static audit.
  4. Validators return identity only. Role re-derived inside getShowForViewer every request — Task 4.3's locked contract.
  5. Client-side cookie parse/format faults are continue + clearCookie, never terminal_failure. The chain must be able to fall through.
  6. The ?t= handler matches ^/show/[^/]+ — every path, not a closed list of subroutes.

The validator contract, in code

Every later milestone that touches a server-rendered show page consumes this tri-state shape. Returning terminal_failure for anything other than a true server-side fault is a contract violation that X.3's audit will catch. 

// lib/auth/validateLinkSession.ts

export type ValidatorOutcome =
  | { kind: 'success'; viewer: { kind: 'crew'; crewMemberId: string; showId: string } }
  | {
      kind: 'continue';
      // Lossless: carry the prior failure metadata so the chain adapter
      // can render the most-actionable §12.4 code if ALL branches fall through.
      priorFailure?: { status: number; code: string; messageInterpolations?: Record<string, string> };
      clearCookie: boolean;
    }
  | { kind: 'terminal_failure'; status: number; code: string };
  // ^ SERVER-SIDE infrastructure faults ONLY.
  //   DB outage. Supabase down. Signing-key fetch failure.
  //   Client-side cookie corruption is `continue + clearCookie`, never this.

export async function validateLinkSession(req: Request): Promise<ValidatorOutcome>;
