Doug saves the Google Sheet. Within five minutes — or seconds, with a push notification — the crew member's webpage reflects the change. That's the whole milestone in one sentence.

Under the hood: four ways a sync can start (a scheduled cron tick, a Drive push, an admin clicking "Re-sync", or a fresh-folder onboarding scan), one shared processor that handles every show identically, and one ironclad rule — only ever one worker mutates a given show at a time.

M6 · A Four ways in, one way through

The four entry points

Cron, push, manual, onboarding-scan. Each one originates from a different surface — a Vercel cron cadence, an inbound webhook from Google, an admin button click, a wizard step in M10. But every one of them eventually calls the same shared processOneFile helper. That's the design: the entry points are thin shells; the processor is fat and shared.

M6 · Trigger sources feed one shared processor
Invariant · per-show advisory lock

Every mutation of shows, crew_members, pending_syncs, or pending_ingestions runs inside pg_try_advisory_xact_lock(hashtext('show:' || drive_file_id)) (cron / push) or the blocking pg_advisory_xact_lock(...) (admin paths). Two workers cannot process the same show at the same time. Period. The lock survives the Phase 1 → Phase 2 boundary because both phases run inside ONE transaction owned by the orchestrator.

Warn · single-holder rule (M5 R20 lesson)

For any given hash key, the lock is acquired at exactly ONE layer — never two. Nested holders deadlock under burst. Phase 1 and Phase 2 helpers receive the locked transaction as an argument and MUST NOT call pg_*advisory*_lock themselves. The LockedShowTx<T> brand makes this unforgeable: _unlocked helpers accept only the branded type, so a refactor that hands them a raw Tx fails to compile. A DEV-mode runtime check against pg_locks catches the residual force-cast.

M6 · B Inside the lock: two phases

Phase 1, Phase 2, and the wall between them

Phase 1 is read-mostly: it parses the markdown, runs the fourteen invariants from M1, and decides whether the sheet should auto-apply, stage for review, or hard-fail. It is allowed to write status columns and pending rows, but never the destructive ones. Phase 2 is the destructive transaction: old crew rows for this show are deleted, new rows inserted, hotel and transport snapshots wiped and reseeded — all atomic, all under the lock.

M6 · Two-phase commit under one advisory lock

Three monotonic guards, one per mode

Phase 2 won't let a stale write clobber a fresh one. Each mode picks a different comparison against last_seen_modified_time:

Ratified amendment 9 · first-seen routing

First-seen sheets dropped in the watched folder (mode = cron / push / manual) AUTO-APPLY when all MI-1..MI-14 pass — Phase 2 runs, the show goes live, a 24h unpublish_token is stamped, and SHOW_FIRST_PUBLISHED fires. Wizard onboarding-scan mode still stages with the ONBOARDING_SCAN_REVIEW sentinel — explicit human review is the point of the wizard. MI-1..MI-5b hard fails route to pending_ingestions regardless of mode. The retired FIRST_SEEN_REVIEW sentinel no longer fires from any code path.

The same-revision binding contract

Doug edits the sheet while a sync is mid-flight. The Drive head advances. The orchestrator captured markdown bytes at R1 but is now enriching pins at R2. Without protection, half the staged ParseResult describes a sheet that no longer exists.

The fix: every sync captures binding = { headRevisionId, modifiedTime } BEFORE exporting markdown, pins every subsequent Drive read to that revision, then runs a final re-verify after enrichment. Mismatch aborts as STAGED_PARSE_REVISION_RACE — no commit, no watermark advance. The next cron pass picks it up with a fresh binding. A cooldown table prevents a hot sheet from burning Drive API quota indefinitely: 60s, 120s, 240s, 480s, 600s capped, cleared on successful commit. 

// lib/sync/runScheduledCronSync.ts — processOneFile flow
const binding = await getDriveClient.files.get(driveFileId, {
  fields: 'headRevisionId,modifiedTime', supportsAllDrives: true
});
const markdown = await fetchSheetAsMarkdownAtRevision(driveFileId, binding.headRevisionId);
const parsed = parseSheet(markdown);
const parseResult = await enrichWithDrivePins(parsed, ..., { binding });

const reVerify = await getDriveClient.files.get(driveFileId, /* same fields */);
if (reVerify.headRevisionId !== binding.headRevisionId) {
  logSyncOutcome({ kind: 'skip', reason: 'STAGED_PARSE_REVISION_RACE', ... });
  return;  // no commit, no watermark advance
}

M6 carries the largest AC count in the plan — twenty-seven criteria across cron, push, manual, onboarding, and watch lifecycle. Here are the most representative:

  • AC-6.2Unchanged sheet → no advance of last_seen_modified_time; cron is a no-op.
  • AC-6.4Show A's parse failure does not skip Show B — per-file isolation across the cron loop.
  • AC-6.7Concurrent sync attempts: second worker hits CONCURRENT_SYNC_SKIPPED because the first holds the advisory lock.
  • AC-6.11First-seen routing per amendment 9: auto-apply on cron/push/manual + all-MI-pass; stage on wizard mode; hard-fail on MI-1..MI-5b.
  • AC-6.14Edit a sheet, webhook fires, last_seen_modified_time advances within ~5 seconds end-to-end.
  • AC-6.21Push that races cron rolls back as STALE_PUSH_ABORTED — the strict-< monotonic guard fires.
  • AC-6.26Source out of scope at Apply time → STAGED_PARSE_SOURCE_OUT_OF_SCOPE; existing-show stages restore prior status.

Two partitions, one drive_file_id

Once an admin starts the onboarding wizard against a candidate folder while the live folder is still cron-syncing, a single drive_file_id can have TWO pending rows — one in the live partition (wizard_session_id IS NULL), one in the wizard partition (wizard_session_id = $session). The schema enforces coexistence via two partial unique indexes treating NULL as distinct from any UUID. Every read and write in the sync pipeline carries the matching scope predicate, and the Apply / Discard routes demand a source_scope parameter from request context — never inferred from row content.

M6 invariants — non-negotiable past this milestone

  1. Every show mutation runs inside the per-show advisory lock. Single holder: lock acquired at exactly one layer.
  2. Phase 1 and Phase 2 share ONE transaction, owned by the orchestrator. Helpers never BEGIN / COMMIT themselves.
  3. The LockedShowTx<T> brand makes lock ownership unforgeable at compile time. DEV-mode pg_locks query catches force-casts.
  4. Same-revision binding pinned on every Drive read; post-enrichment re-verify before Phase 1 commits anything.
  5. Source-scope predicate (wizard_session_id IS NULL vs = $session) on every pending_* read and write.
  6. No global sync cursor. Per-show watermark only — shows.last_seen_modified_time is the source of truth.
