When a crew member clicks "something looks wrong," we file a ticket in our bug tracker. The hard part is making sure one click never creates two tickets, even with bad networks and retries.

The user sees: tap button → modal → type a sentence → submit → "report filed." The system does: authenticate, count quota, reserve a row in the database with a unique idempotency key, stamp an ownership token, call GitHub, write the URL back, recover from anything that broke in the middle.

PATTERN Reserve · call · stamp

The shape of the problem

A naive bug-report endpoint would just POST /api/report → octokit.issues.create and call it done. That endpoint has at least five duplicate-creating bugs in it. Two clients with flaky wifi both retry → two issues. A 5xx from GitHub that actually succeeded → the retry sees no record and creates a second issue. A request that finishes after the user gave up and refreshed → ditto. The fix is the reserve-then-call pattern: the database commits to the work before the network call, so every retry path can find the prior attempt and bind to its result.

M8 · happy path — reserve-then-call

The five states of a report row

Every reports row passes through a small state machine. The columns idempotency_key, processing_lease_until, lease_holder, and github_issue_url tell you which state the row is in at any moment. All four are authored once by Task 2.2's schema; M8 just defines the protocol.

The race that ate fifteen review rounds

Gotcha · the concurrent-retry race

A worker reserves the row, calls GitHub, GitHub succeeds, but the response is lost to a TCP hiccup. The lease expires. A retry claims the lease, creates a second issue, writes its URL to the row. Then the original's stalled HTTP response finally arrives — and the original tries to stamp its URL onto the row. If you just do UPDATE reports SET github_issue_url = $url WHERE id = $row, the original overwrites the retry's URL, then closes the wrong issue. Two open issues. Wrong one closed. Row points at a corpse.

The fix is the lease_holder uuid ownership token. Every URL-writing UPDATE carries AND lease_holder = $myToken. The original sees its tail UPDATE match 0 rows, re-reads the row, discovers another retry is in charge, closes its own orphan issue, and returns 200 with the retry's URL.

§13.2.3 Three ratified amendments

What 40 rounds of review changed

The spec's original §13.2.3 had three load-bearing mistakes. Each one looked fine at the spec level and only revealed itself under adversarial review. All three are now ratified amendments in 00-overview.md; scripts/verify-spec-amendment-3.sh exits non-zero if the spec text drifts from any of them.

Amendment 1 · recovery uses listForRepo, not code search

The original design recovered lost responses by GitHub code search (q="" in:body). The search index lags create writes by tens of seconds — exactly the window AC-8.12 must defend against. A freshly-created issue would be invisible to its own retry, which would then create a duplicate.

The amendment swaps to octokit.rest.issues.listForRepo({ creator, since, state: 'all', labels: 'fxav-app:report' }), which is immediately consistent with create writes. Pages are scanned for the <!-- fxav-report-id: KEY --> marker, bounded by a 1000-page sanity cap, and filtered client-side by issue.created_at >= cutoff because GitHub's since filters by last-updated. LookupInconclusive returns 502 and never authorizes createIssue.

Amendment 2 · retention horizon aligns on created_at

The original reaper deleted rows whose processing_lease_until was older than 24h. But a retry that refreshes the lease 23h after creation pushes the lease another 90s into the future — which under the lease-based predicate would keep the row alive almost another full day, while expiredLeaseRetry was already returning 410 REPORT_HORIZON_EXPIRED on it. Two gates, drifting apart.

The amendment aligns both gates on reports.created_at: 

-- reaper (Task 8.3f, daily cron)
DELETE FROM reports
 WHERE github_issue_url IS NULL
   AND created_at < now - interval '24 hours'
   AND processing_lease_until < now;  -- never reap a row a retry holds

-- retry lease-claim (Task 8.3e)
UPDATE reports SET processing_lease_until = now + interval '90 seconds',
                   lease_holder = $myToken
 WHERE idempotency_key = $1
   AND processing_lease_until < now
   AND github_issue_url IS NULL
   AND created_at >= now - interval '24 hours';  -- horizon fence

The processing_lease_until < now clause in the reaper is the lease-expired race fix — it prevents the reaper from yanking a row a retry is actively working.

Amendment 3 · lease_holder uuid ownership protocol

A new column on reports. Stamped at reservation with gen_random_uuid(). Rotated on every lease reacquisition. Required by every URL-writing tail UPDATE: 

UPDATE reports
   SET github_issue_url = $newUrl
 WHERE idempotency_key = $key
   AND github_issue_url IS NULL
   AND lease_holder    = $myToken::uuid  -- I still own this row
   AND created_at >= now - interval '24 hours'
RETURNING id;

A 0-row tail UPDATE means another retry stole the lease in the meantime. The losing worker enters handleTailUpdateMiss, re-reads the row, and dispatches: Case A the row's URL is mine (a retry's findByMarker recovered my issue — do NOT close it, return 200); Case B the row's URL is a different issue (mine is the orphan — close with state_reason: not_planned and label fxav-orphan-lost-lease, UPSERT admin_alerts.REPORT_ORPHANED_LOST_LEASE); Case C URL still NULL, another worker holds the lease (cleanup my issue, return 409); Case Reaped row gone (close my issue anyway — orphan must never leak at GitHub — return 410).

M8 invariants — never violated

  1. At most one open GitHub issue per idempotency_key, regardless of retry pattern, dropped responses, clock skew, or worker death.
  2. LookupInconclusive never authorizes createIssue. Any inconclusive recovery lookup returns 502; the row stays unresolved for the next retry.
  3. Quota is charged only when the idempotency claim is genuinely won. Duplicate retries, race losers, and rolled-back over-quota attempts increment the counter by zero.
  4. Every horizon decision uses Postgres now(), never Date.now. App/DB clock skew at the 24h boundary must not flip the verdict.
  5. An orphan GitHub issue never leaks even if the row was reaped. Case Reaped still closes the issue and writes the per-show admin alert.

What happens when GitHub goes weird

The recovery path is the most-reviewed code in the milestone. Here's the decision tree every retry walks before it's allowed to call createIssue a second time.

expiredLeaseRetry · decision tree (Task 8.3e)

Why fail-closed on LookupInconclusive matters

If findIssueByMarker can't be sure whether the prior attempt's issue exists, treating the inconclusive result as "no issue exists" would cause a duplicate-creating createIssue on every retry. The contract is hard: only a clean null return — pagination exhausted, no marker match — authorizes a second create call. Pagination errors, missing GITHUB_BOT_LOGIN, malformed responses, multiple live matches, open-issue-with-orphan-label — each routes to a different admin_alerts code so Eric sees the right diagnosis.

The alert catalog

M8 introduces six new admin_alerts codes. Each is keyed per-show via the §4.6 partial unique index (coalesce(show_id::text, ''), code) WHERE resolved_at IS NULL, except GITHUB_BOT_LOGIN_MISSING which is truly global.

CodeTriggerScopeClient gets
REPORT_ORPHANED_LOST_LEASE Tail UPDATE 0-rows; a competing retry stole the lease and my issue is the orphan. per-show 200 (with winner's URL) or 410 (Case Reaped)
REPORT_LOOKUP_INCONCLUSIVE Pagination error, shape error — generic "couldn't confirm whether your last report went through." per-show 502
GITHUB_BOT_LOGIN_MISSING The GITHUB_BOT_LOGIN env var is unset. Recovery cannot run at all. global 502
REPORT_DUPLICATE_LIVE_MATCHES Two open marker-bearing issues for one idempotency_key — data-integrity fault. per-show 502 (Eric must investigate manually)
REPORT_OPEN_ORPHAN_LABEL An open issue carries the fxav-orphan-lost-lease label — should be impossible. per-show 502
REPORT_LEASE_THRASHING expiredLeaseRetry recursed past depth 3 and the row is genuinely still stuck. per-show 503
