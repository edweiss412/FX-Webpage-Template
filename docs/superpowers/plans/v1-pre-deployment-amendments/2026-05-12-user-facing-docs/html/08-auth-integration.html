PHASE H

Prove the whole thing actually works.

Most testing in M12 is unit-level — Phase H is the integrated pass. Every test exercises the assembled /help as a real user would: real browser, real session, real MDX render, real mobile viewport. Each test starts red — proven to fail against the implementation it guards — then goes green.

By the time Phase H runs, the implementation is already done — A built the pipeline, D built the components, E wrote the content, G wired the affordances. The risk now is that a test asserting "feature X works" passes trivially because the implementation is already in place. So Phase H uses a verify-red-via-restore pattern: temporarily break the implementation in a controlled way, watch the test fail, restore, watch it pass. That proves the test would catch a real regression.

The verify-red-via-restore pattern

Seven-step pattern · proves each test catches its regression
No git stash in Phase H

At Phase H time the working tree is clean. git stash push on a clean tree creates no stash; git stash pop then either no-ops or pops the user’s unrelated previous stash — a data-loss risk. The plan calls this out explicitly and uses cp / mv instead.
