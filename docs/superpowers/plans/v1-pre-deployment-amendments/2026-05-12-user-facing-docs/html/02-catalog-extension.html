PHASE B

Give every error code a title, a long story, and a destination.

The app already has a central message catalog — one entry per code with short Doug-facing copy, crew-facing copy, follow-up text, and helpful context. Phase B adds three more fields, then reconciles the live catalog against the master spec so admin-log-only codes stop surfacing in user-visible UI.

Every error in this app — a missing day header in the sheet, a stale sync attempt, a malformed parse — has a stable code like WARN_DAY_HEADER_MISSING. The codes live in one big object. Each entry says, in code, “if this error happens, here’s what to show Doug, here’s what to show crew, here’s a follow-up button label, and here’s extra admin context.”

Phase B adds three more things to each entry: a short title (a heading on /help/errors), a long explanation (the body paragraph there), and a helpHref deep link. Then it cleans up codes that should never have been Doug-visible in the first place — the master spec lists them; the live catalog had drifted.

02 The master-spec alignment problem

Master-spec §12.4 distinguishes two classes of error code: those that should reach Doug visibly (banners, panels) and those that should only ever appear in logs. The live catalog has drifted — some admin-log-only codes still have dougFacing text, so M9’s AlertBanner surfaces them. Phase B’s B.2 parser derives the canonical set; B.3 makes the live catalog match.

02 One predicate, three callers
Predicate fan-out · B.4 makes it canonical
Why helpHref always points at /help/errors

You might expect codes about parsing to link to /help/admin/parse-warnings. They don’t. The §5.6 template-family contract requires one and only one destination for every code’s Learn more → link: /help/errors#<CODE>. Topic pages cross-link from the errors page to their detailed explainers, never the other way around. One catalog, one shape, one URL pattern.
