### `_metaInfraContract` grep-shape recognition contract (deep-dive — read before writing any registered helper)

The meta-test's grep-shape (`tests/admin/_metaInfraContract.test.ts:302-355`) is a regex heuristic, not a type checker. Every registered helper MUST satisfy it or the meta-test fails. Two hard rules learned from the live test body:
1. **A registered file MUST contain ≥1 grep-recognized supabase await** (`awaitLineNumbers.length > 0`, `:317-320`). Recognized = a line matching `await supabase` OR `await <builderVar>` where `<builderVar>` is assigned `const/let/var <builderVar> = supabase…` (or chained from another such var). **An indirect client access (`await resolvedClient.client.from(...)`, `await this.db.from(...)`) is NOT recognized** → bind a local `const supabase = <client>` and call `await supabase.from(...)`. This is why Task 5.1's `getActiveWatchedFolder` binds `const supabase = resolvedClient.client` and Task 5.3 names its client `supabase`.
2. **A pure WRAPPER with no direct supabase await** (e.g. `fetchEmbeddedAdminEmails` → `await listAdminEmails()`) CANNOT be an `infraRegistry` row (it has 0 recognized awaits → fails rule 1). It uses invariant 9's alternative: an inline `// not-subject-to-meta: <reason>` comment + a behavioral test (Task 6.1). The direct boundary it wraps (`listAdminEmails`) is the real Supabase surface (existing code, out of B1 scope).
3. Every recognized await AND every builder-assignment line (`<v> = supabase.from(...)`) must be inside a `try { … } catch` (`:322-354`) — `.from()` is a synchronous throw site, so the assignment, not just the await, must be wrapped.

Result: the 3 registered B1 reads (alertCount, fetchDriveConnectionHealth, getActiveWatchedFolder) each use a literal `supabase` local with the read inside try/catch; the 1 wrapper (fetchEmbeddedAdminEmails) is exempted inline. No registered surface trips the heuristic.

### Task 0.7: Add `getRequiredDougFacing(code): string` (typed non-null catalog accessor)

`getDougFacing` is typed `string | null` (`lib/messages/lookup.ts:42`) — passing it directly into a string-only context (a React `aria-label`/`title` attribute, or text that must not render empty) is a strict-typecheck failure even when the catalog row's `dougFacing` is non-null. Every NEW B1 catalog code (Tasks 0.1–0.5) has a non-null `dougFacing`, so B1 adds a typed accessor that returns `string` (throws if the code's `dougFacing` is null — a programmer error for these codes). **Convention (load-bearing for all later phases):** every new B1 surface sources catalog copy via `getRequiredDougFacing(code)` for string-typed contexts (`aria-label`, `title`, status-line text that must never render empty); plain `getDougFacing` is only acceptable where the consumer genuinely accepts `null` (a `ReactNode` child). This keeps invariant 5 (no raw codes / no literals) AND strict typecheck.

import { createSupabaseServerClient } from "@/lib/supabase/server";
import { MESSAGE_CATALOG, type MessageCatalogEntry } from "@/lib/messages/catalog";

export type AlertCountResult = { kind: "ok"; count: number } | { kind: "infra_error" };

// Mirror AlertBanner's exclusion: info-severity codes are notices, not banner-raising.
const INFO_SEVERITY_CODES: string[] = (Object.values(MESSAGE_CATALOG) as MessageCatalogEntry[])
  .filter((entry) => entry.severity === "info")
  .map((entry) => entry.code);

export async function fetchUnresolvedAlertCount(): Promise<AlertCountResult> {
  let supabase: Awaited<ReturnType<typeof createSupabaseServerClient>>;
  try {
    supabase = await createSupabaseServerClient();
  } catch {
    return { kind: "infra_error" };
  }
  try {
    let q = supabase
      .from("admin_alerts")
      .select("id", { count: "exact", head: true })
      .is("resolved_at", null);
    if (INFO_SEVERITY_CODES.length > 0) {
      q = q.not("code", "in", `(${INFO_SEVERITY_CODES.map((c) => `"${c}"`).join(",")})`);
    }
    const { data: _countData, count, error } = await q; // invariant 9: destructure { data, error }, not bare
    void _countData;
    if (error) return { kind: "infra_error" };
    // A null/undefined count with NO error is an integrity failure, NOT a clean zero — rendering it as
    // count:0 would hide a broken count path behind the no-badge state. Only a real number is "ok".
    if (typeof count !== "number") return { kind: "infra_error" };
    return { kind: "ok", count };
  } catch {
    return { kind: "infra_error" };
  }
