**Tagging convention (CRITICAL for cleanup correctness):** every harness-INSERTed row carries `validation_tag = 'm12-fixture-<outcome>'` somewhere addressable:

- `reports` rows → `context->>'validation_tag'` (jsonb field) — `reports.context` is `jsonb not null` per `supabase/migrations/20260501001000_internal_and_admin.sql:315`.
- `report_rate_limits` rows → no jsonb column; instead the `identity` field uses a reserved prefix `validation:m12-fixture-<outcome>:` so the cleanup pattern is `WHERE identity LIKE 'validation:m12-fixture-%'`. **Two outcomes break the synthetic-identity default** (R39 commit 76 F36 amendment — extends the R35 commit 73 F34 contract): (i) **`rate-limit-admin`** uses the dev's real canonical admin email (NOT the prefix) because live `enforceQuota` canonicalizes the admin identity at the UPSERT (`lib/reports/rateLimit.ts:76`) — a synthetic prefix cannot intercept the production quota path. Cleanup for both real-identity outcomes uses the **snapshot+restore contract** (not blanket deletion): the harness records the prior `(kind, identity, hour_bucket=<recorded>)` row state before seeding and the `--cleanup --include-admin-email <email>` / `--cleanup --include-crew-id <uuid>` flags scope their `report_rate_limits` writes strictly to the recorded `hour_bucket` (restoring the prior count, or DELETEing the row if no prior row existed). The cleanup recipe **never** touches cross-hour buckets and **never** issues a `WHERE kind=<…> AND identity=<…>` predicate without the `AND hour_bucket=<recorded>` clause — see F34 (admin) + F36 (crew) contracts below. The R39 F36 amendment specifically resolves the pre-R39 three-way contradiction where plan 04:70/107 named real fixture crew UUID but plan 04:131 Step 3 / spec §9.1.2:824 implied synthetic-prefix identity for crew; the real-identity contract wins per live-code citation at `lib/reports/submit.ts:168`.

**R35 commit 73 F34 cleanup safety contract (snapshot+restore — non-destructive of prod state).** Pre-R35 the cleanup recipe DELETEd `WHERE kind='admin' AND identity=canonicalize(<email>)` spanning ALL hour_buckets. The `report_rate_limits` primary key is `(kind, identity, hour_bucket)` per `supabase/migrations/20260501001000_internal_and_admin.sql:333` — production admin rate-limit state for the same canonical email in DIFFERENT hours would be erased by that predicate. R34 F34 (MEDIUM) flagged this as destructive cleanup affecting prod state. The repaired contract:

1. **Snapshot at seed time.** Before the harness UPSERTs the rate-limit-admin row, it `SELECT count FROM report_rate_limits WHERE kind='admin' AND identity=canonicalize($VALIDATION_ADMIN_EMAIL) AND hour_bucket=date_trunc('hour', now())` and stores `(snapshot_prior_count, recorded_hour_bucket)` for cleanup. If no row exists, `snapshot_prior_count = NULL`.

2. **Seed via UPSERT** at the recorded bucket — `INSERT ... ON CONFLICT (kind, identity, hour_bucket) DO UPDATE SET count = 11` — so the seed is idempotent and the bucket coordinates are pinned.

3. **Cleanup via restore** (NOT blanket DELETE). With `--cleanup --include-admin-email $VALIDATION_ADMIN_EMAIL`:
   - If `snapshot_prior_count IS NULL` → `DELETE FROM report_rate_limits WHERE kind='admin' AND identity=canonicalize(<email>) AND hour_bucket=<recorded_hour_bucket>` (scoped to the EXACT recorded bucket; never cross-hour).
   - If `snapshot_prior_count IS NOT NULL` → `UPDATE report_rate_limits SET count = <snapshot_prior_count> WHERE kind='admin' AND identity=canonicalize(<email>) AND hour_bucket=<recorded_hour_bucket>` (restores the prior count to the EXACT recorded bucket; never cross-hour).

4. **Snapshot persistence (R41 commit 78 F37 amendment — file-backed only).** The `(snapshot_prior_count, recorded_hour_bucket, email_canonical_form)` tuple must survive across the harness's seed → exercise → cleanup invocation boundary (CLI invocations are separate processes).    - **File-backed snapshot** at `.validation-state/rate-limit-admin-snapshot.json` (gitignored — `.validation-state/` is in repo-root `.gitignore` per R41 commit 78) — recipe persists / reads / removes the file across invocations.

   `validation_state` CANNOT be used as a snapshot backend because the table is defined with `CHECK (key = 'validation_seed')` (singleton constraint) per the master spec §3.3.2 DDL and `02-phase0-validation-state.md:139` / `:404` — any snapshot key like `'rate_limit_admin_snapshot'` would fail the CHECK constraint at INSERT time. Extending the `validation_state` schema to admit additional keys is out-of-scope for M12 (live migration change against a singleton-constraint table that is already RLS-locked + REVOKE-locked per R17 commit 37). 5. **Cleanup defense-in-depth.** If `--cleanup --include-admin-email <email>` is invoked WITHOUT a snapshot record (e.g., orphaned cleanup after a crash), the harness MUST refuse with stderr `no rate-limit-admin snapshot found; cleanup aborted to avoid touching prod buckets` (exit 1).
