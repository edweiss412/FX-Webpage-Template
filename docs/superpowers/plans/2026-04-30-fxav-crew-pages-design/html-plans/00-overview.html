OVERVIEW

The whole show, on one page.

Doug ships every show out of a giant Google Sheet. The crew has to thumb through it on a phone backstage. This plan turns every sheet into a personal, mobile-first webpage for every crew member — with sub-second sync, role-based field hiding, signed-link sharing, and an admin surface that lets Doug fix things from a folder, not a dashboard.

Picture a touring AV crew — audio, video, lighting techs — arriving at a venue they’ve never seen, on a tour where the schedule changed twice that morning. The owner of FXAV, Doug, has a Google Sheet for every show: hotel reservations, schedule, stage diagrams, pack lists, who’s LEAD, who flies in early, what time the show calls. Today every crew member squints at that whole sheet on a phone.

The product is a webpage personalized to each crew member: only their lodging, their schedule, the gear they pack, the people they need to call. Doug keeps editing the sheet. The page updates within a second. He can drop a brand-new sheet into a Drive folder and it goes live automatically, with a one-button undo email. Crew log in with a Google account, or just tap the signed link Doug texted them.

The nine plan-wide invariants — never violated, regardless of test status

  1. TDD per task. Every task: failing test → minimal implementation → passing test → commit. Skipping the failing test means the test isn’t actually covering what it claims.
  2. Per-show advisory lock. Every mutation of shows, crew_members, crew_member_auth, pending_syncs, or pending_ingestions runs inside pg_try_advisory_xact_lock(hashtext('show:' || drive_file_id)). Single-holder rule: per hashkey, the lock is acquired at exactly one layer — never two.
  3. Email canonicalization at every boundary. lib/email/canonicalize.ts is the only function permitted to touch raw emails. The schema-level CHECK is the safety net, not the primary mechanism.
  4. No global sync cursor. Per §3.2 / §5.2 / AC-X.4, no source file references lastPollAt. Each show tracks shows.last_seen_modified_time.
  5. No raw error codes in user-visible UI. §12.4 is the catalog. UI reads codes through lib/messages/lookup.ts which returns the appropriate copy.
  6. Commit per task. Conventional commits: feat(parser):, test(auth):, infra:. One task per commit; no batching.
  7. Spec is canonical. Nine ratified amendments below are the only places where the plan supersedes the spec. Elsewhere the spec wins — open a question.
  8. UI quality gate. Every UI surface ships only after /impeccable critique AND /impeccable audit pass; HIGH and CRITICAL findings either fixed or deferred via DEFERRED.md.
  9. Supabase call-boundary discipline. Every Supabase call destructures { data, error }; returned vs thrown errors are distinguished; infra faults surface as discriminable typed results — never silent continue or benign auth signals.
§2 The nine ratified amendments

Where the plan supersedes the spec

Every milestone references the spec section it implements. The plan and spec disagree in exactly nine places — each one survived between four and forty rounds of cross-model adversarial review. These are the only places where the plan wins.

Note · spec §12.4 catalog rows

Separately, two crew-facing catalog rows were ratified into spec §12.4 (AGENDA_GONE_FOR_CREW 410/403, AGENDA_UNAUTHENTICATED 401) covering the AgendaPdfViewer proxy. Integrated into the spec body by Task 9.0.A1; consumed by M7-D2.

A1 · §13.2.3 Ratified

Recovery uses listForRepo, not code search

GitHub’s code-search index lags tens of seconds — false-negatives drive duplicate-issue creation. Switched to issues.listForRepo({creator, since, state:'all'}), which is immediately consistent with create writes. Body marker stays; client-side filters by created_at >= T-24h.

api/report/route.tsTask 8.3d/8.3e
A2 · §13.2.3 Ratified

Reaper aligns on created_at + lease check

Retry rejects rows where created_at < now - 24h (returns 410). Reaper deletes only when github_issue_url IS NULL AND created_at < now - 24h AND processing_lease_until < now. The third clause prevents reaping a row a retry actively holds.

Task 8.3f
A3 · §13.2.3 / §4.1 Ratified

lease_holder ownership protocol

lease_holder uuid on reports, stamped at reservation, rotated on re-acquisition, required on every tail UPDATE. 0-row tail triggers orphan cleanup — close issue, label fxav-orphan-lost-lease, alert. Re-SELECT null → 410.

Task 8.3g · verify-only
A4 · §6.4 Ratified

Drop v3 from the version registry

Spec declared v3 marker as block:GEAR INVENTORY. Zero corpus fixtures contain that string. Parser treats versions as 'v1' | 'v2' | 'v4'. Re-introducible if a genuine v3 surfaces.

lib/parser/schema.ts
A5 · §6.4 Ratified

v4 collapses to a single marker

Strict AND-of-two (row:Contact Office & block:MAIN/SECONDARY) produced 50% false-negatives in the v4 corpus. Parser treats v4 as row:Contact Office single-marker. MAIN/SECONDARY is documentation-only.

lib/parser/schema.ts
A6 · §5.2 / §5.3 Ratified

Sheets revision binding → modtime CAS

Google Sheets are Workspace-native: headRevisionId and md5Checksum return null; drive.revisions.list is empty; Sheets API v4 has no revision parameter. Plan ratifies modtime CAS for the spreadsheet; per-asset binary binding (diagrams) is unaffected.

M6 · Pin-stop 1.5
A7 · §6.8 Ratified

MI-8/MI-8b modtime-stable debounce

Drive’s ~3-min modtime batching means a mid-edit "clear-and-retype" can fire an empty-cell push. MI-8 (financial collapse) and MI-8b (COI delta) fire only when now - modifiedTime ≥ 240s. Cron/push/recovery only — manual and onboarding bypass.

lib/sync/constants.ts
A8 · §6.8 Ratified

MI-9 narrowed to LEAD-bit changes

Spec staged on any role_flags delta. Narrowed to LEAD-bit-set membership change. Non-LEAD deltas (department, additive flags) auto-apply and emit info-severity ROLE_FLAGS_NOTICE. MI-9 and MI-10 collapse to a single LEAD-bit check.

lib/parser/invariants.ts
A9 · §5.2 / §6.8 Ratified

First-seen auto-publish + 24h email undo

Doug-validated 2026-05-09: "the folder IS the publish gate." First-seen + all MI pass → auto-apply via Phase 2 + signed 24h unpublish link in confirmation email. Wizard onboarding scan unchanged. FIRST_SEEN_REVIEW retired; SHOW_FIRST_PUBLISHED replaces it.

Why these survived adversarial review

Each amendment was probed by a cross-CLI Codex/Claude opposing-model review until convergence. A1–A3 needed forty rounds and a meta-test (tests/auth/_metaInfraContract.test.ts) before converging. A6–A9 are grounded in verified external facts: a grep of the corpus, a Drive API doc citation, a Doug-validation answer in doug-validation-questions.md. The plan earns the right to override the spec only with that level of evidence.
