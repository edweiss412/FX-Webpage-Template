Crew Pages · Implementation Plan · 2026-04-30

A mobile webpage for every crew member, every show.

Doug Larson runs FXAV — a touring AV company. He keeps a Google Sheet for every show: crew, hotels, schedules, stage diagrams, pack lists. This project turns each sheet into a personalized mobile webpage for every crew member on that show — synced live from Drive, with role-based field hiding, signed-link sharing, and an admin console for Doug to keep it all running. A touring AV crew can have 40 people on a single show — riggers, A1s, V1s, lighting, drivers, runners. Today, every one of them gets the same 60-page Google Sheet and is expected to mentally filter it down to the parts that apply to their job, on their day, in their assigned room.

This system replaces that with: scan a QR code → open a mobile webpage with only your schedule, your hotel, your transportation, your venue, and the diagrams for the rooms you actually work. When Doug edits the master sheet in Drive, your page updates within seconds.

Eight plan-wide invariants — non-negotiable

These hold across every milestone. Violating any of them is a P0 bug regardless of test status.

  1. TDD per task. Failing test → minimal implementation → passing test → commit. Always.
  2. Per-show advisory lock. Every code path that mutates show data runs inside pg_advisory_xact_lock on the show's drive_file_id. Single-holder rule — nested holders deadlock.
  3. Email canonicalization at every boundary. lib/email/canonicalize.ts is the only function allowed to touch raw emails.
  4. No global sync cursor. Each show tracks its own last_seen_modified_time. No lastPollAt anywhere.
  5. No raw error codes in user-visible UI. §12.4 is the catalog; UI reads it through lib/messages/lookup.ts.
  6. Commit per task with conventional-commits style: feat(scope): summary.
  7. Spec is canonical, with the nine ratified amendments documented in the overview.
  8. UI quality gate. Every UI surface ships only after /impeccable critique AND /impeccable audit pass on the diff.
  9. Supabase call-boundary discipline. Every Supabase call destructures {data, error} and surfaces infra faults as typed results, never silent continue.
