AC · X

Six robots that walk the whole codebase.

The other ten milestones each prove a feature works. This one proves something stranger: that a set of plan-wide promises still hold true everywhere, all at once. No single feature can guarantee them; only an audit can. Six audits ship here — one per invariant — each implemented as a script that scans every relevant file in the repo and fails CI the moment a line of code violates the contract.

These aren't features. They're guarantees. Each audit is a small robot that walks the entire codebase looking for one specific kind of violation. If the robot finds anything, CI turns red and the change cannot merge.

Why does this exist? Because in a system this big, individual milestone tests cannot catch class-level drift. A test proves that one route handler works; it can't prove that every route handler still calls the auth validator first. A test proves that one error message renders correctly; it can't prove that no raw error code ever leaks into a user-visible string anywhere on the site. The audits do.

Tests prove a feature works. Audits prove an invariant holds.

Every milestone before this one shipped behavior. M1 parses sheets. M4 renders tiles. M6 syncs Drive. M8 mints report links. Each was tested in isolation. But the spec also makes system-wide promises: no raw error codes in the UI; one place that touches raw emails; per-show watermarks only; a single auth chain on every protected surface. Those promises can only be defended by a script that scans the entire repo on every pull request. That's what this milestone builds.

AC-X · Six scanners over the codebase

What each audit defends, in plain words

AC-X.1 · catalog parity
Every error message Doug or a crew member could ever see is described in one place (§12.4 of the spec). The audit proves spec, catalog code, and at least one test scenario all agree about every code — and that retired codes never reappear.
AC-X.2 · no raw codes in UI
Internal error codes (LINK_REVOKED_FLOOR, MI-5b_DUPLICATE_CREW_EMAIL, parse_error) are for logs and admin alerts. They must never appear as text on a page Doug or crew see — not in body text, not in title attributes, not in a textarea controlled by React state.
AC-X.3 · one auth chain
Every protected URL passes through the same small set of validators in the same order, before touching protected data. Not just "imports the validator" — dominates every reachable database read.
AC-X.4 · no global cursor
Sync decisions are made per show, not per system. There is no single "when did we last poll?" timestamp anywhere — every watermark belongs to one row of one table.
AC-X.5 · email canonicalization
One function (lib/email/canonicalize.ts) is allowed to clean up raw email strings. Every other place that touches an email must call it first. The schema CHECKs are the safety net; this audit is the doorman.
AC-X.6 · spec ↔ code matrix
Every numbered section of the spec maps to at least one task in the plan via a structured HTML-comment anchor. Free-form prose mentions don't count. If the spec adds a new amendment and no task claims it, CI fails.
X.1 Three-way §12.4 parity

X.1 · No orphan error codes

Spec §12.4 is the canonical message catalog. Every code worth showing a human has a row there with Doug-facing copy, crew-facing copy, a follow-up suggestion, and helpful-context paragraph. This audit proves three things, all at once.

Invariant · spec ↔ catalog ↔ scenario

For every code X in §12.4: (1) SPEC_CODES[X] is extracted from the spec markdown, (2) catalog[X] in the codebase has byte-identical copy in all four columns, and (3) at least one named test scenario actually drives a code path that emits X. Drift in any one of the three fails the build.

What would break if this failed

Someone edits the Doug-facing copy in §12.4 to say something kinder, but never updates the catalog. Doug keeps seeing the old phrasing. Or a developer adds a new internal error code in lib/sync/ without adding it to §12.4 — orphan codes that the lookup helper can't translate ship into production. Or worst: a code marked retired in round 44 reappears in a TSX file. All three classes are caught here.

X.2 Substring leak detection

X.2 · No raw error codes in user-visible UI

An error code like LINK_REVOKED_FLOOR is meaningful in logs and Sentry. To a crew member in a hotel lobby at 2am, it's an alarming hex-string. This audit makes sure no rendered surface ever shows one.

Invariant · catalog is the only renderer

Internal codes (SPEC_CODES ∪ RETIRED_CODES ∪ INTERNAL_CODE_ENUMS) must reach the screen only through messageFor(), which substitutes catalog copy. Raw appearance — in text, attribute, or live DOM property — fails the build.

Three crawls per surface

Playwright drives every reachable route. For each element on each page, the audit runs three scans, because earlier drafts each missed a hole:

ScanWhat it readsThe hole it closes
1a · textContent every element's text node obvious leaks: <span>{errorCode}</span>
1b · user-visible attributes aria-label, title, alt, placeholder, value, aria-description, aria-roledescription title="LINK_REVOKED_FLOOR" — invisible to text scan, visible to screen readers
1c · live DOM properties input.value, textarea.value, select.selectedOptions[0], contenteditable.textContent controlled inputs whose value lives in React state, never in the HTML attribute
2 · AST scan ts-morph over every JsxText + JsxAttribute + JsxExpression codes referenced statically (defense in depth — catches before runtime can)
Known leak class — fixed here

A textarea bound to useState('LINK_REVOKED_FLOOR') is invisible to textContent (textarea internals aren't text) and invisible to getAttribute('value') (React never writes the attribute — it owns the property setter). Scan 1c reads textarea.value directly. A fixture in tests/cross-cutting/fixtures/x2-controlled-input/ reproduces this exact bug and the audit must flag it.

X.3 Trust-domain control-flow audit

X.3 · One auth chain, dominates every sink

The original draft of this audit checked: does the route file import a validator? That's worthless. A handler can import validateLinkSession, gate one branch on it, and read protected data in another. The corrected audit does dominator analysis on the call graph: every reachable database read on every reachable execution path must be preceded by a complete validator chain.

Eight trust domains

Every file under app/ is classified into exactly one domain. Each domain has a required auth contract. Adding a new file without classifying it fails CI.

crew-session

Show page · asset · report routes

Four valid terminal-success branches. B1: admin-precedence wins. B2: link wins. B3: link continue → google wins. B4: link continue → google continue → admin wins. The audit accepts any path matching any branch.

admin

Admin pages + admin API

requireAdmin only.

me

/me · cross-show user list

validateGoogleIdentity ONLY — not validateGoogleSession, which is show-bound and cannot run cross-show.

auth-library

The validators themselves

lib/auth/**, middleware.ts, and the cookie-mint route. Exempt — these are the chain.

public-bootstrap

Pre-cookie shell

app/show/[slug]/p/page.tsx renders before any cookie exists.

public-webhook

Drive push channel

Constant-time token compare, not the user-session validator chain.

cron-internal

Scheduled jobs

Vercel cron auth header. No user identity available.

server-action

'use server' (AST-detected)

Detected by directive scan before path-based classification, so an inline action in a component file doesn't escape audit. Trust domain inherited from the containing route subtree.

Why this is hard — full set of request entries

App Router invokes many server-side functions during a request: not just the page default export and HTTP-method handlers, but generateMetadata, generateViewport, head.tsx, loading.tsx, not-found.tsx, template.tsx. Each is its own entry into the dominator analysis. A generateMetadata that reads shows_internal without first calling the validator leaks data through Open Graph tags and page titles — and the audit catches it.

X.4 Per-show watermarks only

X.4 · No global cursor

Per spec §3.2, §5.2 and AC-X.4: there is no single timestamp that decides whether sync runs. Every watermark belongs to one row. Why? Because a global cursor coupled to wall-clock time has nasty failure modes — clock skew, crashed-mid-batch reprocessing storms, and "we missed everything modified between 2:04 and 2:05" silent gaps. Per-row watermarks are crash-safe by construction.

Invariant · three audit layers

(a) Name-based heuristic catches the obvious — anything matching /last_(seen|sync|poll|processed|run|cursor)|watermark|cursor/i must be on the allowlist. (b) Semantic data-flow audit walks the call graph from every sync entry point and asserts every modifiedTime-shape comparison reads from a per-row source. (c) DDL event trigger blocks new columns matching the heuristic at the database layer.

Two named sets

AUTHORITATIVE_GATING
Drive-derived per-row timestamps that may appear as the right-hand operand of a sync-decision comparison: shows.last_seen_modified_time, pending_syncs.base_modified_time, pending_syncs.staged_modified_time, fileMeta.modifiedTime, deferred_ingestions.deferred_at_modified_time.
DISPLAY_ONLY
Per-row timestamps rendered to the operator but never consulted by sync gating: shows.last_synced_at, pending_syncs.parsed_at, pending_ingestions.last_attempt_at. A sync-decision read of any of these fails the audit.
Why naming alone isn't enough

Someone introduces processedAt on a hypothetical singleton app_settings row and uses it as the watermark. The name passes every regex. The semantic layer catches it: the comparison's RHS resolves to a row that isn't on the per-show allowlist.

X.5 One canonicalization function

X.5 · Email canonicalization at every boundary

lib/email/canonicalize.ts (built in Task 1.2) lowercases, trims, and strips +suffix aliases. Every email entering the system must pass through it exactly once, at the earliest boundary. The schema-level *_email_canonical CHECK constraints are the safety net — they reject uncanonicalized writes — but this audit is the doorman, catching the problem at the call site rather than at INSERT time.

Failure mode the audit catches

A new email-bearing column is added without the matching CHECK; a parser site is added without calling canonicalize; the SQL helper auth_email_canonical() is renamed or has its arity changed; an admin_alerts.context payload starts capturing a raw requester email. All flagged by the pg_proc + ts-morph + AST scan trio.

X.6 Machine-generated coverage matrix

X.6 · Spec ↔ implementation traceability

The spec is the canonical contract. Every numbered section is a normative obligation. The plan is supposed to implement all of them. The earlier draft of this audit was a manual checklist — unmaintainable at the scale of 17 spec sections × ~80 amendments × 60-plus tasks. The corrected design generates the matrix.

How the generator works

  1. Walk every heading §N / §N.M / §N.M.O in the spec markdown.
  2. Walk every <!-- spec-id: kebab-slug --> anchor — these mark non-heading normative units like the §6.8 derivation table, §13.2.3 lease protocol amendment, and the immutable-pin amendment.
  3. Walk every AC-* row from spec §17.
  4. Walk every plan task block delimited by ### Task N.M:, collecting <!-- coverage: §X, AC-Y --> markers. Prose mentions don't count — only structured anchors.
  5. Emit docs/superpowers/plans/coverage.md with status planned / implemented / deferred / out-of-scope / MISSING per anchor.
Why prose doesn't count

Earlier rounds accepted a sentence like "this task uses ParsedSheet and enrichWithDrivePins" as evidence of coverage. That re-introduces the heuristic matching X.6 was redesigned to eliminate. A future amendment to the type split could ship with no task actually claiming it — and the audit would still pass on the strength of the old prose mention. Only structured <!-- coverage: ... --> markers count now.

CI · 7

verify-branch-protection-status

Lightweight reader. Confirms the privileged verify-branch-protection job ran successfully on main within the last 8 days. Uses only the auto-injected GITHUB_TOKEN — no secrets — so fork PRs don't deadlock.

Why a split reader job exists

GitHub does not send secrets to pull_request workflows triggered by forks; using pull_request_target would fix the secret access but expose secrets to untrusted PR code — a known security hole. The split: the privileged verify-branch-protection job runs only on trusted contexts (push to main, weekly cron); the lightweight verify-branch-protection-status reader runs on every PR and asserts the privileged job ran successfully recently. Two jobs, one guarantee, zero secret leak.

Acceptance criteria

  • AC-X.1Every §12.4 code has a catalog entry with byte-identical copy in all four columns AND at least one named scenario that emits it. Retired codes have no producer in any source surface.
  • AC-X.2No SPEC_CODES, RETIRED_CODES, or INTERNAL_CODE_ENUMS identifier appears as text content, user-visible attribute, or live DOM property on any reachable surface.
  • AC-X.3Every protected request entry — including generateMetadata, loading.tsx, not-found.tsx, and inline 'use server' actions — passes a complete validator chain on every reachable execution path before any protected sink.
  • AC-X.4No file, schema column, or string literal anywhere in the repo encodes a global sync watermark. All gating watermarks resolve to per-row sources on the explicit allowlist.
  • AC-X.5lib/email/canonicalize.ts is the sole function called at every parser, DB write, validator, and RLS boundary that handles a raw email. Every email-bearing schema element has a matching *_email_canonical CHECK.
  • AC-X.6Generator emits zero anchors at status MISSING. ParsedSheet/ParseResult split, /40 lease-holder amendments, and §16 secrets coverage are all explicitly mapped via structured <!-- coverage: ... --> markers. All seven required CI checks pass on every PR.

The point

Building one feature well is engineering. Keeping every feature still respecting the same set of system-wide promises, sprint after sprint, year after year — that's discipline. And discipline at this scale only survives if a robot enforces it. These six audits are that robot.

They have no end-user. No one ever sees their output unless something is wrong. They sit quietly in CI on every PR, walking the codebase one more time, asking the same six questions: is every error code accounted for? Is the UI clean of internal codes? Does every protected route still authenticate? Are watermarks still per-show? Are emails canonicalized? Does every spec section still have an owner?

Eleven milestones built the product. This one keeps it honest.
