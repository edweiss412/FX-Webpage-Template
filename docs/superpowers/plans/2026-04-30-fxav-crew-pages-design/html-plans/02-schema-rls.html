A schema is the blueprint for a database — what tables exist, what columns each table has, what relationships connect them. RLS (row-level security) is Postgres saying "even if the application is buggy or compromised, here are the rows each kind of user is allowed to see." Together they form the safety perimeter every later milestone trusts without re-checking.

For FXAV that perimeter matters because each show sheet contains two very different kinds of data: ordinary crew info (names, hotels, call times) that the crew themselves need to see, and LEAD-only financial data (payout rates, gross billings) that must never reach a regular crew member's phone — even if a bug rendered an admin page to the wrong audience. M2 makes that separation a property of the database itself, not a property of the UI code.

M2 What gets built

The shape

Three migration files, one seed script, one introspection harness. The migrations are CREATE-only — no ALTER TABLE, no IF NOT EXISTS — because additive fragments mask drift the audit harness is designed to catch.

M2 · Entity-relationship sketch

LEAD-only financial data — three independent locks

A crew page never accidentally exposes financials. Not because the UI is careful — because three different layers would each have to fail simultaneously:

  1. Physical isolation. Financial fields don't live on the shows table at all. They live in shows_internal, a separate 1:1 table. A SELECT against shows can't accidentally return what isn't there.
  2. RLS policy. shows_internal is admin-only. A request without the admin role gets zero rows back, regardless of what query it sends.
  3. Role-aware fetcher (M4). The application-side data layer refuses to join shows_internal unless the current viewer is a LEAD or admin. This is the layer crew code touches; the layers below it are the backstop.

AC-2.5 and AC-2.6 prove layers 1 and 2 work with exhaustive table×verb tests — every admin-only table is probed for SELECT / INSERT / UPDATE / DELETE under a non-admin JWT, with a service-role control proving the row actually exists and the operation is real.

Email canonicalization — at the schema, as a safety net

The plan-wide invariant says lib/email/canonicalize.ts (from M1) is the only function permitted to touch a raw email. M2 makes that a property the database itself enforces — every email-bearing column carries a CHECK that rejects any value the canonicalizer would have altered: 

-- supabase/migrations/…_initial_public_schema.sql

CREATE TABLE crew_members (
  id            uuid primary key default gen_random_uuid,
  show_id       uuid not null references shows(id) on delete cascade,
  name          text not null,
  email         text,                                       -- nullable per spec
  role          text not null,
  role_flags    text[] not null default '{}',
  ...
  CONSTRAINT crew_members_email_canonical
    CHECK (email IS NULL OR email = lower(btrim(email)))
);

-- partial unique index: one row per (show, canonical email) when present
CREATE UNIQUE INDEX crew_members_show_email_unique
  ON crew_members (show_id, email)
  WHERE email IS NOT NULL;

If any path ever forgets to canonicalize — a future API route, a manual SQL fix, a CSV import — Postgres rejects the row. The application doesn't have to be perfect; it has to be perfect or the database catches it.

The RLS pattern, in one helper

Naïvely, the RLS policy on crew_members wants to say "let this row through if the viewer is admin or a member of this show." But Postgres evaluates that policy against crew_members itself — which is the same table the predicate has to read to check membership. Self-referential RLS either recurses or denies-by-default.

The fix is a SECURITY DEFINER helper that runs with the migration role's privileges and bypasses RLS for the membership lookup. Three rules harden it: an explicit SET search_path with pg_temp placed last, schema-qualified references inside the body, and a REVOKE ALL FROM PUBLIC followed by an explicit grant to only the roles that need it. 

CREATE OR REPLACE FUNCTION can_read_show(p_show_id uuid)
RETURNS boolean
LANGUAGE sql
SECURITY DEFINER
SET search_path = public, pg_temp   -- pg_temp LAST closes the writable-schema attack
STABLE
AS $$
  SELECT public.is_admin
      OR EXISTS (
           SELECT 1 FROM public.crew_members c
            WHERE c.show_id = p_show_id
              AND c.email   = public.auth_email_canonical
         );
$$;

REVOKE ALL ON FUNCTION can_read_show(uuid) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION can_read_show(uuid) TO authenticated, anon;

Every crew-readable table's SELECT policy then reads: is_admin OR (can_read_show(table.show_id) AND EXISTS (SELECT 1 FROM shows WHERE id = table.show_id AND published = TRUE)). The published = TRUE gate has to be replicated on every peer — not relying on a join through shows — because PostgREST can query peer tables directly.

Invariant · the three canonical SQL helpers

is_admin (zero-arg, boolean), auth_email_canonical (zero-arg, text), and canonicalize_email(text) (one-arg, immutable). Every callsite uses parentheses; bare-identifier usage would silently return the function's oid instead of executing it. Task 2.5's introspection asserts pronargs on each.

Why introspection earns its own task

Migrations are write-once. A column rename, a weakened cascade rule (ON DELETE CASCADE → SET NULL), or a partial index that quietly loses its WHERE clause won't fail any normal test — the rows still flow, the queries still return data. Task 2.5 reads Postgres's own catalogs and asserts each constraint, foreign key, and partial index matches its documented definition exactly:

What it checksSource catalogFailure mode it catches
CHECK definitions pg_get_constraintdef A future migration relaxes the canonical-email CHECK to allow mixed-case emails.
FOREIGN KEY cascade rules introspect_fk(...) SECURITY DEFINER helper An FK silently changes from CASCADE to SET NULL, orphaning crew rows when a show is deleted.
Partial index WHERE clauses pg_get_indexdef The unique index loses its WHERE email IS NOT NULL predicate and starts blocking NULL-email inserts.
Function signatures (pronargs) pg_proc A helper gains a parameter; existing callsites still type-check but resolve to the wrong overload.
RLS trigger inventory (M2.2 add) pg_trigger The AFTER-STATEMENT publish trigger gets dropped — rows still update, but the realtime invalidation never fires.

Acceptance criteria

  • AC-2.1Every table from spec §4 exists with documented columns, constraints, FK cascades, and partial indexes. Drift fails the introspection test.
  • AC-2.2crew_members_show_email_unique rejects a duplicate (show_id, canonical email) pair.
  • AC-2.3crew_members_email_canonical rejects any value that isn't lower(btrim(value)) — mixed case, leading whitespace, surrounding spaces.
  • AC-2.4revoked_links.token_version > 0 CHECK rejects a zero token version.
  • AC-2.5Every one of the 21 admin-only tables denies SELECT / INSERT / UPDATE / DELETE under a non-admin JWT — 84 assertions, each with a service-role control proving the row exists and the payload is valid.
  • AC-2.6Crew-readable tables (shows + 5 peers) deny writes under a matching and non-matching crew JWT, and the published = FALSE gate hides interim-batch rows from peers.
  • AC-2.7The seed script loads all 10 fixtures via the production pipeline; every persisted field round-trips correctly, including the reel-pin quadruple and the structured diagrams JSONB.

One more thing M2 quietly establishes

M2 doesn't use per-show advisory locks — that's M5 and M6 — but it puts the surfaces in place so later code can. Every table that later milestones mutate under a pg_advisory_xact_lock(hashtext('show:' || drive_file_id)) is created here: shows, crew_members, crew_member_auth, pending_syncs, pending_ingestions. The plan-wide invariant — "single holder per hashkey" — is enforced later by the structural meta-test at tests/auth/advisoryLockRpcDeadlock.test.ts; the tables it pins exist starting here.
