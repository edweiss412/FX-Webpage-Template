Imagine Doug — owner of FXAV — keeps a Google Sheet for every show he produces. Each sheet has crew names, hotel reservations, schedules, transportation, stage diagrams, pack lists. Today his crew has to scroll through that giant sheet on a phone backstage. Our job is to turn each sheet into a personal mobile webpage for every crew member.

But before we touch the database or build a single screen, we have to prove one thing can be done reliably: given the text of one of those sheets, can we extract every piece of useful information from it as a clean data structure? That's all M1 does. M0 just gets the workshop ready to host it.

M1 Parser standalone

Reading the binder

Every FXAV show has a Google Sheet that follows one of four templates (v1, v2, v4 — v3 was dropped because no real sheet uses it). The sheet has named blocks for crew, hotels, transportation, stage diagrams, schedules, financials. A small army of typos and edge cases lives in there — sheets in the wild call things "Hotal" instead of "Hotel", say "MAIN | SECONDARY" in some versions but not others, and use shorthand like "GS - V1" to mean a crew member is V1 for the General Session room.

The parser is the brain that turns all that text into a single, predictable data structure. It runs as a pure function — no database, no Drive API, no network. You give it markdown; it gives you a ParseResult with warnings on the side. That purity is the point: every later milestone can call this function thousands of times in tests without spinning anything up.

M1 · parseSheet pipeline

Why pure?

Because every test in every later milestone — the database tests, the sync tests, the UI tests — calls parseSheet on a fixture and asserts on the result. If the parser had any side effect, those tests would have to mock or spin up infrastructure to even start. By forbidding side effects, the parser stays runnable in 50ms inside a Vitest worker.

Invariant · email canonicalization

Every email the parser emits passes through lib/email/canonicalize.ts exactly once. This is the only function in the codebase allowed to touch a raw email string. The schema-level CHECK in M2 is the safety net — but canonicalization happens here, at the very first boundary.

Ratified amendment · spec §6.4

The spec defined four versions (v1, v2, v3, v4). Zero real sheets have v3's marker (block:GEAR INVENTORY) — verified by grepping the entire fixture corpus. The plan treats versions as 'v1' | 'v2' | 'v4'. If a v3 sheet ever surfaces, it can be re-introduced. v4 was simplified from a two-marker AND to a single marker for the same reason: half the v4 corpus didn't have the second marker.

Foundation invariants — never violated past this milestone

  1. The parser is a pure function. No imports of node:fs, googleapis, or any Supabase client are ever allowed.
  2. lib/email/canonicalize.ts is the only function permitted to touch raw email strings. Every later layer enforces this with audits (AC-X.5).
  3. Version detection uses marker presence, never structural heuristics. The amendment retiring v3 and simplifying v4 lives in 00-overview.md.
  4. TDD per task. Each task adds a failing test first, then implementation. No exceptions.

Acceptance criteria

  • AC-1.1parseSheet runs against any of the 10 raw fixtures without throwing.
  • AC-1.2Crew rows extract correctly for every version, including compound role tokens like "GS - LEAD".
  • AC-1.3Day restrictions parse into explicit / unknown_asterisk / none; no silent guessing.
  • AC-1.4Stage restrictions enumerate work-phases canonically (Load In, Set, Show, Strike, Load Out).
  • AC-1.5Email outputs are all canonicalized before leaving the parser.
  • AC-1.6Canonicalization handles uppercase, whitespace, and +alias suffixes.
  • AC-1.7MI-1..MI-5b hard-fail invariants trigger ParseError, not silent partial results.
  • AC-1.8MI-6..MI-14 soft invariants produce warnings consumable by the M6 sync gate.
  • AC-1.9Slug derivation is stable: same show name → same slug across re-parses.
  • AC-1.10Slugs are URL-safe and collision-checked against the existing show set.

The contract, in code

Every milestone after this one consumes ParseResult. Here are the two most-referenced shapes — the ones M2 turns into Postgres tables and M4 turns into mobile tile components. 

// lib/parser/types.ts

export type ParseResult = {
  show: ShowRow;
  crew: CrewMemberRow[];
  hotels: HotelReservationRow[];
  rooms: RoomRow[];
  transportation: TransportationRow[];
  contacts: ContactRow[];
  pullSheet: PullSheetCase[];
  diagrams: DiagramReference[];
  warnings: ParseWarning[];
  errors: ParseError[];          // empty on a clean parse
};

export type CrewMemberRow = {
  name: string;
  email: string | null;          // canonicalized at the boundary
  phone: string | null;
  role: string;                  // raw display string from the sheet
  role_flags: RoleFlag[];        // ['LEAD', 'A1'] — atomic flags, no compounds
  date_restriction:  DateRestriction;
  stage_restriction: StageRestriction;
  flight_info: string | null;
};

export type WorkPhase = 'Load In' | 'Set' | 'Show' | 'Strike' | 'Load Out';
