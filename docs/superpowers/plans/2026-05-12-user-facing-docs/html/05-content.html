04 The shared pattern · every task follows it

Every Phase E task follows the same seven steps. The only thing that varies is the page-specific assertions in the test.

  1. Write the test (RED). Real-render through the MDX pipeline + page-specific source-shape assertions.
  2. Run test → fails. Phase A.7’s stub is just # <Title> — none of the page-specific assertions pass.
  3. Replace the stub content. File already exists from A.7; E.N edits in place.
  4. For E.5 – E.11, backfill the catalog. Populate title / longExplanation / helpHref on the corresponding catalog entries. helpHref always points at /help/errors#<CODE>.
  5. Run test → passes. Both real-render and source-shape assertions green.
  6. Manual sanity check. pnpm dev, visit as admin, scan for editorial drift.
  7. Commit. feat(help): /help/<slug> page (Task E.N).
Why the real-MDX-render step?

Earlier drafts only used readFileSync + regex to assert page shape. That lets invalid JSX or missing imports ship green — the regex still matches the source even if the page would crash at runtime. Compiling the page through @next/mdx in the test catches that class.

05 Content quality bar
Tone
Matches PRODUCT.md — professional · bold-modern · intuitive. Plain language. No jargon. No error codes in visible text (only as URL fragments). Glanceable on mobile.
Components
Every <Screenshot name="..."> resolves to a manifest entry. Every <RefAnchor> uses catalog-code-shaped IDs (non-catalog anchors use plain <h2 id="...">).
No artifacts
No <ScreenshotPlaceholder> references at v1 close-out (Phase H.5 lint enforces).
Cross-links
Every link from a page points at a real /help/... route or a real #anchor. Phase H.1’s anchor-resolver fails CI on any broken deep-link.
06 /help/errors · the special case

/help/errors is the only TSX page. It iterates the catalog and renders one section per code passing the predicate (severity ≠ "info" AND dougFacing ≠ null). Each section is a <RefAnchor id="CODE" as="h3"> followed by title and longExplanation. Codes appear in catalog declaration order.

Invariant #4 — no raw error codes in visible UI

/help/errors shows codes only as URL fragments (#REPORT_HORIZON_EXPIRED). The visible heading is entry.title, never the code itself.

E.13 also lands the live-catalog full-contract assertion — originally scoped as Phase H.6 but folded into E.13 for TDD compliance. The assertion uses contractViolations from catalogDocsValidator.ts (Phase B.4) — single source of truth between forced fixtures and the live-catalog gate.
