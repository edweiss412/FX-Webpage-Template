PHASE G

Plant the "Learn more →" seeds in /admin.

The help center only works if Doug can find it. Phase G retrofits twelve specific admin surfaces with little ? tooltips and Learn more → links that deep-link into /help/.... The whole thing is driven by a typed registry — the affordance matrix — so the doc-side and the surface-side can’t drift.

A help center buried inside the app is no help at all. So instead of expecting Doug to navigate to /help when he’s confused, Phase G plants tiny ? tooltips and Learn more → links right next to the things in the admin UI that are most likely to confuse him. Each one is a single-click jump into the exact relevant page or anchor.

The clever part: the same renderer needs to know not to show those affordances when Doug is using "preview as crew" mode — the previewed content has to look exactly like what crew sees, with no admin-only chrome leaking in.

What's an affordance?

FLAVOR 1

? tooltips on headers

Active Shows, Sync health, Crew preview links, wizard steps. Click → open the matching help page in a new tab.

FLAVOR 2

Learn more → on error messages

Every message rendered via messageFor(code) in admin contexts. Points at /help/errors#<CODE>.

FLAVOR 3

Take the tour footer link

Single link in the dashboard footer. Points at /help/tour.

02 The affordance matrix · 3 row classes

app/help/_affordanceMatrix.ts is the typed registry — one row per affordance. Three row classes:

Class What it means Example testid
concrete Single fixed data-testid on a known surface. Test #13 asserts it exists. help-affordance--dashboard-active-shows--tooltip
template-family Pattern for codes that come from the catalog. One row, walked per-code via catalog iteration. help-affordance--error-message--<code>--learn-more
negative Asserts the absence of any help-affordance--* testid on a given route — crew pages must never carry one. None on /show/<slug>

The 12 concrete rows live in M3 / M9 / M10 components (dashboard, per-show panel, preview-as-crew banner, onboarding wizard steps). The template-family row covers every code in the catalog. The negative row pins the crew-side guarantee.

03 The four-context renderer gate

The same messageFor(code) call site renders in four different contexts. The gate decides whether to emit the Learn more → link.

Context URL pattern Audience Emit Learn-more?
admin /admin/* Doug Yes
help-admin /help/* Doug Yes
crew /show/<slug> Crew No
preview-as-crew /admin/show/<slug>/preview/<member> Doug, pretending to be crew No (banner excepted)
The preview-as-crew exception

Preview-as-crew lives on an /admin/* URL but the content being previewed is what crew would see. The renderer treats it as crew context — the previewed content stays affordance-free — while the sticky admin banner above it still carries its ? tooltip. The URL alone can’t disambiguate; the renderer needs an explicit admin-context signal.

Pure function. Takes pathname + explicit previewAsCrew signal. Returns true/false. Unit tests cover all four contexts.
