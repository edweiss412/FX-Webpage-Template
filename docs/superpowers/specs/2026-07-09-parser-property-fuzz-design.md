# Parser Property-Fuzz Layer (fast-check) — Design

**Date:** 2026-07-09
**Provenance:** open half of `docs/audits/e2e-real-world-variation-preparedness-2026-07-07.md` §7 item 5 ("`fast-check` property fuzz layer … still absent"); original finding `docs/audits/edge-case-preparedness-audit-2026-07-04.md:67` ("a novel-but-valid shape has *no test analog by construction*").
**Approved direction (user, 2026-07-09):** both layers phased (markdown-first, xlsx second); Tier 1 robustness + Tier 2 plant-and-find properties; PR-bounded fixed-seed run + nightly deep run; model-first generator + chaos side-arbitrary.

---

## 1. Problem

The mutation harness (`tests/parser/mutation/`, PR #338) perturbs the **existing** fixture corpus with fixed operators and pins verdicts in an 8k-row known-holes ledger. It answers "does the parser signal when a known sheet is corrupted?" It cannot answer "does the parser handle a sheet shape nobody has written yet?" — the July-4 audit's residual: golden-file replay of 15 historical fixtures means a novel-but-valid coordinator variation (renamed headers within vocabulary, reordered sections, unusual-but-supported date/dims formats, headerless crew tables) has no test analog by construction.

A property-based layer generates those shapes and asserts invariants the parser must hold on **all** of them.

## 2. Scope

**Phase 1 (this milestone):** generate markdown directly, feed `parseSheet(markdown, filename?)` (`lib/parser/index.ts:546`). Tier 1 + Tier 2 properties, chaos arbitrary, PR CI job, nightly deep job, regression corpus.

**Phase 2 (separate follow-up milestone; designed here, not planned here):** generate workbook grids, `XLSX.write` to buffer, feed `synthesizeMarkdownFromXlsx(buffer, opts?)` (`lib/drive/exportSheetToMarkdown.ts:301`), then `parseSheet`. Adds exporter-normalization coverage (grid→table serialization, OLD-tab dropping, pull-sheet region detection). Same property set; exporter-specific Tier-1 additions (§6.1).

**Out of scope (both phases):** sync/apply layers (`runOnboardingScan`, `runScheduledCronSync`); DB; UI; the per-field provenance model (the other open half of audit §7 item 5); replacing or modifying the mutation harness; fuzzing `parsePullSheet` internals beyond what flows through the two entry points.

## 3. Architecture

New directory `tests/parser/fuzz/` (sibling of `tests/parser/mutation/`; shares only read-only imports from `tests/parser/mutation/oracle.ts`). New devDependency: `fast-check` (currently absent from `package.json` — verified).

```
tests/parser/fuzz/
  model.ts          ShowModel type + fc.Arbitrary<ShowModel>
  dials.ts          variation-dial registry (each dial cites its code contract)
  render.ts         ShowModel + DialChoices → markdown string
  chaos.ts          fc.Arbitrary<string> — hostile markdown, no model
  groundTruth.ts    Tier-2 oracle: planted-entity lookup + signal-accounting
  shape.ts          assertParsedSheetShape structural validator
  seeds.ts          PR_SEED, PR_NUM_RUNS, deep-run env plumbing (single source of numbers)
  robustness.fuzz.test.ts     Tier 1 over model-rendered + chaos inputs
  plantAndFind.fuzz.test.ts   Tier 2 over model-rendered inputs
  regressions/      committed shrunk counterexamples, replayed every PR run
  _metaDialRegistry.test.ts   walker meta-test (§8)
```

### 3.1 ShowModel (ground truth)

A semantic model of one show sheet, generated by fast-check arbitraries:

- `dates`: travel-in/show-days/travel-out as ISO calendar dates, years confined to 2020–2035 (inside the parser's 2000–2099 bound on ISO/dash/long-form shapes — `lib/parser/blocks/_helpers.ts:139-142`; the slash path is unbounded but the model stays inside the strictest bound so every rendering dial is in-contract). Model-internal representation is always unambiguous ISO; **rendering** format is a dial.
- `crew`: 1–12 members — `name` (letter-safe generated names, no pipes/newlines, **unique within the model** — `fc.uniqueArray` or equivalent; duplicates would let one parsed row satisfy two planted entities), `role` (drawn from a clean role vocab — screened so no role text matches the parser's restriction/personalization metasyntax: no `ONLY`, no `***`, no date-shaped tokens, no stage words, no parenthesized clauses; the `dayRestriction` dial is the ONLY producer of restriction clauses, so an unplanted `date_restriction` is always a real finding), `phone`, optional `email`, optional day-restriction clause.
- `hotels`: 0–3 — name, address (US suffix-bearing street shape only, matching the `address` dial's split contract — Canadian/suffixless shapes are out of Phase-1 model scope; see the dial table), guest list drawn from crew names.
- `rooms`: 0–6 — name + dims (width×depth values the dims grammar must accept). Room rendering is constrained to the structured room-table layout whose admission gates `parseRooms` actually implements — header-family recognition, uppercase/prefix gates, placeholder rejection, content gates — with the exact admissible name/layout envelope **enumerated from live `rooms.ts` at plan time before the dial ships** (same discipline as `dayRestriction`); generated room names are screened against header-family tokens and placeholder vocab so a "model-valid" room can never sit outside the cited contract.
- `venue`: name + address.
- `sections`: which of the modeled sections are present and in what order. **Presence is coupled to content:** crew, venue, and dates sections are always present (their entities are always generated); hotels and rooms sections are present **iff** their entity list is non-empty — a section is never omitted while carrying planted entities, and never rendered empty. `validateModel` (below) enforces the coupling; `sectionOrder`'s "omission of optional sections" means exactly the hotels/rooms-empty case.

**Model section scope (Phase 1):** the model plants exactly the five entity kinds above (crew, hotels, rooms, venue, dates). Other section kinds the parser handles (transportation, contacts, agenda, pull sheet, diagrams, opening reel, client, dress, event details) are OUT of Phase-1 model scope — they appear in fuzz input only via the chaos arbitrary (Tier 1). Extending the model to them is the natural Phase-1.5 increment; the oracle matrix (§4.2) grows a row per field when they land.

**Version scaffold (render-emitted, not planted) — v4 ONLY in Phase 1:** the confidence markers `classifyVersion` scores live in blocks outside the five modeled kinds (`lib/parser/schema.ts:91-100`, `V4_BLOCKS`/`V2_BLOCKS`) — but not all marker blocks are modeled-section-safe: the v2 GS-timing markers (`GS SET TIME`/`GS SETUP`) cause `parseRooms` to CREATE a General Session room (`lib/parser/blocks/rooms.ts:978-1019`), which would fabricate `rooms` payload for a zero-room model. Phase 1 therefore pins `model.version = "v4"` (a constant, not an arbitrary): `render.ts` emits a FIXED, byte-constant v4 scaffold — marker label rows for ≥2 of contact/rental/logistics, with constant placeholder values — whose parse products land only in non-modeled sections (contacts/transportation). v2 generation is deferred to Phase 1.5 and requires its own scaffold-interference analysis (finding a v2 marker pair that provably cannot touch modeled payload). Scaffold rows are NOT planted entities — the Tier-2 oracle asserts nothing about them, and the §7 zero-fabrication guard applies to modeled sections only. Being byte-constant, the scaffold satisfies `MIN_ABS`/`MIN_MARGIN`/`MIN_BLOCKS` deterministically.

**Model validity gate (`validateModel`, runtime-enforced honesty):** generator-honesty constraints are enforced by CODE, not prose — `model.ts` exports `validateModel(model)` which every property calls on each generated model BEFORE rendering; a violation throws immediately (generator bug, never a parser finding). Invariants: (a) all planted identity strings are unique within their section (crew names globally unique; hotel names, room names pairwise distinct; venue name distinct from all of them); (b) hotel guest lists PARTITION a subset of crew names — no crew name appears in two hotels' guest lists; (c) no planted string contains a version-marker literal, a pipe, or a newline; (d) dates within the 2020–2035 band; (e) dial composition exclusions (§3.2) hold; (f) section-presence coupling — crew/venue/dates present, hotels/rooms present iff non-empty, no section omitted while carrying planted entities. This is the structural defense that keeps value-containment attribution (§4.2) sound: any contained value identifies exactly one planted entity.

The model **is** the ground truth: `render.ts` serializes it; `groundTruth.ts` knows exactly what was planted where.

### 3.2 Dials — contractually-supported variation only

Every dial is registered in `dials.ts` with `{ name, contractFile, contractSymbol, note?, arbitrary }` — `contractFile` is the repo-relative file and `contractSymbol` a named top-level declaration in it (exported or module-private — several contracts are deliberately private constants, e.g. `STREET_ADDRESS_RE`, `CREW_COLUMN_VOCAB`) that promises to handle the variation (this is the normative registry format the §8 meta-test asserts; the table below adds `file:line` prose citations for the reader's convenience only — line numbers are not machine-checked). Structural dials whose contract is a behavior rather than one symbol (e.g. `sectionOrder`, `blankPadding`) anchor on `parseSheet` and carry a mandatory `note` stating the behavioral contract. A parse failure **inside dial range is a real parser finding by definition** — the dial registry is the honesty boundary. Initial dial set:

| Dial | Variation | Contract |
|---|---|---|
| `sectionHeader` | per-section **(token, layout-family) pairs** — tokens drawn from each block's own `SECTION_HEADER_TOKENS` export (crew: `["CREW","TECH"]` `crew.ts:31`; venue: `["VENUE"]` `venue.ts:8`; dates: `["DATES"]` `dates.ts:36`; rooms: `rooms.ts:108`), NOT the global `KNOWN_SECTION_HEADERS` registry (`knownSections.ts:34`, a known/unknown classifier containing e.g. `VENUES` which `parseVenue` never opens on). Tokens are NOT interchangeable openers: each dial value binds a token to the layout family that token actually opens, verified at plan time. Hotels Phase 1 uses ONLY the structured `HOTEL` opener (`hotels.ts:381-384`); the `HOTEL RESERVATION(S)`/`HOTEL STAY(S)` tokens are inline-row family matchers with a different layout (`hotels.ts:605-627`) — deferred to a follow-on dial with their own render layout. The walker meta-test (`tests/parser/_metaKnownSectionsWalker.test.ts`) pins that every block exports the token constant |
| `headerTypo` | Damerau distance-1 typos, ONLY within corrected vocab | `lib/parser/sectionHeaderNormalize.ts:16` (`LONG_SECTION_VOCAB`), `:25` (`SHORT_SECTION_VOCAB`), excluding `:26` (`SHORT_SECTION_VOCAB_EXCLUDE`) |
| `dateFormat` | slash M/D/YY(YY), dash M-D-YYYY, ISO, long-form MDY, long-form DMY | `lib/parser/blocks/_helpers.ts:127-147` (`normalizeDate` branches) |
| `dimsFormat` | unit-bearing (`50'`, `50′`, `50 ft`) and bare 2–3-digit operands, `x`/`×` separators | `lib/parser/blocks/_dimsToken.ts:21,27,40-47` |
| `crewColumns` | header-row column-order permutations of the crew vocab; headerless positional layout | `lib/parser/blocks/crew.ts:75` (`CREW_COLUMN_VOCAB`), `:80-84` (positional defaults) |
| `dayRestriction` | explicit per-crew day-clause shapes the crew parser strips from role and parses into `date_restriction` — exact supported clause grammar enumerated at plan time from live code before the dial ships | `lib/parser/blocks/crew.ts:308,386` (day-clause strip), `:416` (`date_restriction`) |
| `sectionOrder` | permutation of section order; omission of optional sections — GIVEN exporter-shaped separation (below) | anchor `lib/parser/index.ts:546` `parseSheet` — Step 3 dispatches every block parser over the whole markdown (`:594-600`); note: "order-independent given every section is separated by ≥1 blank line, the shape the exporter always emits (`lib/drive/exportSheetToMarkdown.ts:357` `tables.join("\n\n")`)" |
| `blankPadding` | **1–3** blank rows between sections (NEVER 0 — a zero-separation adjacency is not exporter-producible and sits inside the previous block's read window, so it is out of contract); trailing blank cells | anchor `lib/parser/index.ts:546` `parseSheet`; note: "render always emits ≥1 blank line between sections, matching the exporter's `join(\"\\n\\n\")`; block terminators (e.g. first blank line after table start, `index.ts:298-303`) then bound every read window" |
| `address` | US suffix-bearing streets ONLY (incl. Crescent/Commons/Mews-class suffixes) | `lib/parser/blocks/hotels.ts:319-320` (`STREET_ADDRESS_RE`) — the split contract. Suffixless `…, ST ZIP`/Canadian-postal tails (`STREET_ADDRESS_ZIP_RE`, `:325-326`) are deliberately EXCLUDED: that regex is a dash-discriminator only, never a split trigger (`hotels.ts:330-333` "NOT used to SPLIT"); suffixless addresses stay glued-safe in `hotel_name` by design, so they are not a round-trip contract |

**Ambiguity guard (dateFormat):** when the dial renders a slash/dash date whose day ≤ 12 (order-ambiguous), ground truth is the parser's MDY-first slash/dash order (`lib/parser/blocks/_helpers.ts:154-162` — `month = slash[1]`); the model may also receive the `DATE_ORDER_SUGGESTS_DMY` warning (ambiguity-warnings-v1, PR #367) — Tier 2 counts that as a signal, never as a failure. The dial never plants a date whose **intended** semantics are DMY in an ambiguous rendering.

**Typo-dial guard:** `headerTypo` applies at most one edit and never produces a string that collides with a different vocab entry or a `KNOWN_SUB_LABELS` entry (`lib/parser/knownSections.ts:99` inference would reclassify it — that's the known silent-exit class from audit §5 P1-5, out of contract).

**Dial composition constraints (render-enforced):** dial combinations must stay jointly in-contract, not just individually. Known exclusions, enforced in `render.ts` and documented per-dial in the registry: (a) `headerTypo` over `SHORT_SECTION_VOCAB` (CREW/TECH) applies ONLY when the section renders WITH its header row carrying the field-band corroboration — live `normalizeSectionHeaders` corrects short-vocab typos only with a field-header band on the same row, never label-only (`lib/parser/sectionHeaderNormalize.ts:95-108`); it therefore NEVER composes with the headerless `crewColumns` state (a typo'd bare `CREW` row is out of contract — not correctable, not positional-fallback-eligible). New exclusions discovered during implementation are added to the same per-dial registry documentation.

Dial growth rule: adding a dial for shape X requires the contract citation; if the parser has no contract for X, X is not a dial — file it in BACKLOG as a parser-capability gap instead.

### 3.3 Chaos arbitrary (`chaos.ts`)

Model-free hostile input for Tier 1 only: random line soup biased toward `|`, `\n`, header-ish tokens, zero-width/bidi/control characters, very long cells, degenerate tables (0-cell rows, wide rows), mixed CRLF. Hard caps, jointly enforced so the Cartesian worst case stays CI-safe: ≤ 10k chars/cell, ≤ 400 lines, ≤ 120 cells/row, AND a **total document cap of 256 KB** (the binding constraint — the per-cell/per-row caps alone would multiply to hundreds of MB before parse-twice + JSON round-trip). No ground truth exists, so only robustness properties apply.

## 4. Properties

### 4.1 Tier 1 — robustness (inputs: model-rendered AND chaos)

1. **Never throws:** `parseSheet` returns for every input. `parseSheet` has no internal catch-all — today the only net is the cron call-site guard (`runScheduledCronSync.ts:2865-2884`, `PARSE_SHEET_THREW` fail-closed); this property establishes empirically that the parser never needs it.
2. **Deterministic:** parsing the same input twice yields deeply-equal `ParsedSheet` (via `tests/parser/mutation/oracle.ts:103` `fingerprint` or vitest `toEqual`).
3. **Structurally valid:** output satisfies `assertParsedSheetShape` — every REQUIRED field of `ParsedSheet` (`lib/parser/types.ts:378-402`) present with correct container types; optional fields (`runOfShow?`) validated only when present; `warnings[]`/`hardErrors[]` entries carry non-empty string `code`; payload is JSON-round-trippable.
4. **Bounded:** completes within the property timeout (structural guard against catastrophic-backtracking regex regressions).

(No "sentinel leak" property: parser payload legitimately carries sentinel strings like `TBD` — hiding them is the UI contract enforced by `tests/components/tiles/_metaSentinelHidingContract.test.ts`, not a parser invariant.)

### 4.2 Tier 2 — plant-and-find (inputs: model-rendered only)

**Precondition — no hardErrors, ever, on model-rendered input.** Every dial is in-contract by construction, so a hardError on model-rendered input is itself a property failure. The model's version is pinned `"v4"` in Phase 1 (§3.1); render emits ONLY v4 confidence markers via the byte-constant version scaffold — ≥2 markers across ≥2 independent blocks (`lib/parser/schema.ts:88-101`, `MIN_ABS = 2`) and ZERO competing-version marker literals anywhere in the output (render must screen generated free text against both versions' marker lists), which satisfies `MIN_MARGIN = 2` (`lib/parser/schema.ts:101-102`) by construction. `VERSION_AMBIGUOUS` (or any other hardError) on generated input fails the property outright. This closes the vacuous-pass path where a sheet-level hardError would absolve every planted miss at once. hardErrors are therefore NOT an absolution channel in Tier 2 at all.

For every planted entity, **either** it round-trips correctly **or** an attributable non-fatal signal fired:

- **Found correctly:** entity appears in the payload section it was planted into with EVERY planted field matching, under **one-to-one matching** — each parsed row may satisfy at most one planted entity (planted identities are unique by construction, §3.1, and matching consumes rows). The oracle is **field-complete over the model**: every field the model plants has a match rule; a field with no rule is a spec bug. Phase-1 matrix:

| Planted field | Match rule | Contract |
|---|---|---|
| crew.name | verbatim (whitespace-canon) | verbatim-storage, audit §6 2.1 findings |
| crew.role | verbatim (day-clause stripped when restriction dial applied) | `crew.ts:308,386` cleanedRole |
| crew.phone | whitespace-canon equality | `crew.ts:190` near-verbatim `clean()` |
| crew.email | equality under `canonicalize()` — the parser lowercases at the boundary (invariant 3), so the oracle compares `canonicalize(planted)` to parsed; absent stays absent | `lib/email/canonicalize.ts:2-5`, `crew.ts:317-318` |
| crew.dayRestriction | parsed `date_restriction` kind `explicit` with day-token multiset equal to the model's days rendered as the parser's stored `M/D` token format (parser stores raw `M/D` tokens, NOT ISO — `groundTruth.ts` converts model ISO days to `M/D` before comparing); `{kind:'none'}` when no restriction planted | `lib/parser/personalization.ts:57-64,142-145`; `crew.ts:416`; storage shape pinned by `tests/parser/blocks/crew.test.ts:56` |
| dates (travel-in/show/travel-out) | ISO-equal to model | `_helpers.ts:127-190` |
| rooms.name + rooms.dims | name whitespace-canon; dims numerically equal after normalization | `_dimsToken.ts` |
| hotels.name | whitespace-canon equality | `hotels.ts` splitHotelNameAddress name side |
| hotels.address | non-null AND street-string equality (suffix-bearing planted shape must split) | `hotels.ts:319-320` `STREET_ADDRESS_RE` |
| hotels.guests | **multiset equal** to planted list (whitespace-canon) — subset NOT enough; empty/shortened fails unless signaled (e.g. `HOTEL_GUEST_SPLIT_AMBIGUOUS`) | `hotels.ts:129-151` |
| venue.name + venue.address | whitespace-canon equality | `venue.ts` parseVenue |

Field-level mismatches absolve exactly like entity misses: an entity-attributable or section-structural signal (below), else FAIL.
- **Attributable signal**, two tiers:
  1. **Entity-attributable warning:** same section AND identifies the entity — `blockRef.iso` matches the planted date, `blockRef.name`/`blockRef.index` matches the planted entity, or the warning context contains the entity's **identity value** (`ParseWarning.blockRef` fields: `lib/parser/types.ts:7-28`). Identity values are exactly the strings `validateModel` (§3.1) guarantees unique — crew name, hotel name, room name, venue name, date token. Containment of a NON-identity field value (role, phone, email, dims, address, guest-cell text) NEVER absolves: those are not uniqueness-guaranteed, so a warning about one entity could otherwise mask a different entity's miss sharing the same value.
  2. **Section-structural warning:** a code from an explicit allowlist in `groundTruth.ts` of codes that mean "this whole section is unreadable" (empty-section codes and their kin) — these absolve every entity of that section, because that is their documented meaning. The allowlist is enumerated in code with a comment per entry justifying whole-section semantics. `CREW_COLUMN_POSITIONAL_FALLBACK` is explicitly NOT on the allowlist: it means "columns read by position," not "section unreliable" — the headerless-crew dial renders values in the positional-default order (`crew.ts:80-84`), so the parse must STILL round-trip correctly; the oracle for that dial value requires BOTH the warning fires AND every crew entity is found (no absolution). A generic same-section warning does NOT absolve entities it doesn't identify (a single date-ambiguity warning must not mask an unrelated dropped crew member in the same section).
  3. **`raw_unrecognized`:** row's `block` field maps to the entity's section AND its content contains the entity's **identity value** (same restriction as tier 1 — non-identity field values never absolve).

An unrelated section's warning or `raw_unrecognized` row does NOT absolve a miss elsewhere, and within a section only the two attribution tiers above absolve — this prevents one noisy signal from masking silent drops (the mutation harness's `SILENT_WRONG` discipline, `tests/parser/mutation/oracle.ts:67-76`, applied with ground truth instead of baseline-diff).

Expected-signal cases are first-class: the headerless-crew dial value asserts `CREW_COLUMN_POSITIONAL_FALLBACK` (`lib/parser/blocks/crew.ts:159`) actually fires — the property net double-covers audit item 2.1's fix.

## 5. Execution model

All run-count/seed numbers live in `seeds.ts` — single source; the table below references it.

| Run | Trigger | Seed | numRuns/property | Merge-blocking |
|---|---|---|---|---|
| PR | default vitest suite | `PR_SEED` fixed constant | `PR_NUM_RUNS` = 100 | yes |
| Regression corpus | default vitest suite | n/a (explicit examples) | all committed cases | yes |
| Nightly deep | x-audits schedule + `workflow_dispatch` | random (logged) | `DEEP_NUM_RUNS` = 5000 (env-overridable) | no |

- PR run is deterministic (fixed seed) — it is a regression net, not exploration; runtime target < 60 s total.
- Fuzz test files live under `tests/parser/fuzz/*.fuzz.test.ts` and must be picked up by the default `vitest run` project globs — NOT by the mutation globs (`vitest.projects.ts` `MUTATION_TEST_GLOBS` = `tests/parser/mutationHarness.*.test.ts`; no collision) and NOT env-gated like the mutation project (`vitest.config.ts:98-109` `VITEST_INCLUDE_MUTATION_HARNESS`). Plan must verify the chosen glob actually includes the new directory in the default run.
- Deep run: `FUZZ_DEEP=1` env raises numRuns; seed is random by default but overridable via `FUZZ_SEED=<n>` (exact replay path). New x-audits job `parser-fuzz-deep` gated `if: github.event_name == 'schedule' || github.event_name == 'workflow_dispatch'` (workflow already has all three triggers, `.github/workflows/x-audits.yml:1-10`; per the local-passes-CI-fails discipline, `workflow_dispatch` verification at close-out is mandatory).
- On failure, the job summary surfaces seed, numRuns, fast-check version, and the shrunk counterexample. Triage step 1 is always exact local replay: `FUZZ_DEEP=1 FUZZ_SEED=<seed> vitest run tests/parser/fuzz` — the committed regression file is derived from the replayed (not just the printed) counterexample.

## 6. Findings lifecycle

1. Nightly (or bumped-seed) run fails → shrunk counterexample + seed in job output.
2. Counterexample committed under `tests/parser/fuzz/regressions/` as a plain explicit-input test (no fast-check runtime dependency for replay) — replayed by every PR run permanently.
3. Triage, exactly two exits:
   - **Parser bug** (input is inside dial contract): fix via TDD; the regression file is the failing test.
   - **Dial overreach** (generator claimed contract the code never made): narrow the dial with an inline `// contract-narrowed: <reason>` comment + BACKLOG entry if the capability gap is worth closing later.
4. No large known-holes ledger. The mutation harness pins an 8k-row verdict inventory because its input space is fixed; generated space is unbounded — the honesty mechanism is the dial contract, not a ledger.

### 6.1 Phase-2 exporter additions (designed, not planned here)

- Grid arbitrary: model → 2-D cell grid with merged-region simulation, then `XLSX.write` → buffer.
- Exporter Tier-1: `synthesizeMarkdownFromXlsx` never throws; deterministic; OLD-tab exclusion invariant (any `\bOLD\b`-named tab's content absent from markdown unless `includePullSheetFromTab` names it, `lib/drive/exportSheetToMarkdown.ts:323-349`).
- Chain Tier-2: plant-and-find through exporter+parser composed.

## 7. Non-goals / guard conditions

- **Zero model = valid:** the ShowModel arbitrary may generate 0 hotels/0 rooms — per §3.1 coupling the section is then OMITTED from the render entirely (never rendered empty). Tier 2 asserts strict absence-of-fabrication: a modeled section with nothing planted must have an EMPTY payload, no signal absolution (there is nothing a signal could legitimately explain — the section isn't in the input). Applies to MODELED sections only — version-scaffold sections (§3.1) legitimately parse into payload and are exempt.
- **PII:** generated names/phones/emails are synthetic; counterexample files contain only synthetic data — safe to commit.
- **No flakiness by construction:** PR path has zero randomness (fixed seed + explicit regressions). Only the non-blocking nightly explores.
- **No production code changes** in Phase 1 except a fix if the fuzz finds a real parser bug pre-merge (then: TDD, separate commit, class-sweep per AGENTS.md).
- **Telemetry invariant 10:** N/A — test-only surfaces, no mutation routes/actions.
- **UI invariant 8:** N/A — no UI files.
- **DB:** N/A — no schema/migrations.

## 8. Meta-test inventory (AGENTS.md writing-plans requirement)

- **CREATES** `tests/parser/fuzz/_metaDialRegistry.test.ts`: walks the `dials.ts` registry; every dial entry must carry `contractFile` (exists in repo) + `contractSymbol` (matched as a **declaration**, regex `^\s*(export\s+)?(const|let|function|class|type)\s+<symbol>\b` multiline against the file — a comment or string mention does NOT pass; not line-number pinning, line pins rot; a deleted/renamed contract declaration fails the walker), and any dial anchored on `parseSheet` (structural) must carry a non-empty `note`. Fails-by-default for a new dial added without a contract — mirrors the known-sections-walker pattern, `tests/parser/_metaKnownSectionsWalker.test.ts`, PR #351.
- **EXTENDS:** none. Mutation-harness registries untouched. Not applicable: `_metaInfraContract` (no Supabase calls), sentinel-hiding walker (no tiles), admin-alert catalog (no new codes — fuzz asserts existing codes fire; it introduces none).

## 9. Success criteria

1. PR suite: fuzz properties green at fixed seed, < 60 s, merge-blocking.
2. Nightly deep job exists, `workflow_dispatch`-verified green on real CI at least once before close-out.
3. Tier-2 oracle demonstrably catches a seeded silent-wrong: a deliberate sabotage check (temporarily break e.g. crew column mapping locally) produces a `plant-and-find` failure — anti-tautology proof, recorded in the plan's test task.
4. Dial registry meta-test fails when a dial lacks a contract citation.
5. `fast-check` in devDependencies only; no production import.

## 10. Resolved decisions (user, 2026-07-09)

- Both layers, phased; Phase 1 = parser/markdown only. (Q1)
- Phase 1 model version pinned v4 (v2 scaffold provably fabricates rooms via GS markers — deferred to Phase 1.5 with scaffold-interference analysis). Hotels Phase 1 = structured `HOTEL` opener only; inline-row hotel families deferred. (Adversarial R8)
- Tiers 1+2; Tier 3 metamorphic (mutation operators over generated baselines) explicitly deferred — revisit after Phase 2. (Q2)
- PR-bounded fixed-seed merge-blocking + nightly deep non-blocking. (Q3)
- Model-first generator + chaos side-arbitrary. (Q4)
