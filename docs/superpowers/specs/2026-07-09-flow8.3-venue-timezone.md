# Flow 8.3a — Venue Timezone Enrichment

**Status:** spec (autonomous ship). **Date:** 2026-07-09. **Branch:** `feat/flow8.3-venue-tz`.

**Goal:** Populate each show's `venue.timezone` (IANA) at ingest-time enrich, derived offline from the lat/lng already present in the Google Geocoding response, so `resolveShowTimezone` renders every show in its actual timezone instead of the `America/New_York` fallback. When a timezone can't be derived, emit a gate-exempt admin data-gap warning `VENUE_TIMEZONE_UNRESOLVED`, sibling to `VENUE_GEOCODE_UNRESOLVED`.

**Scope boundary:** This spec is Flow **8.3a (venue timezone) ONLY**. Flow 8.3b (transport id-resolution, `BL-TRANSPORT-ID-RESOLUTION` remainder — resolve driver/assigned names to crew ids at APPLY and match transport visibility by id) is a **separate later arc** with its own spec; it shares no code with this one and remains BACKLOG-deferred. Do not implement any transport/id/visibility change here.

Service-role only; PostgREST-locked (`migration:45-47`), registered in `tests/db/postgrest-dml-lockdown.test.ts`. cache.ts is registered in the Supabase call-boundary meta-test `tests/sync/_metaInfraContract.test.ts` (`cache.ts:11`).

- **No tz library present.** `package.json` has no `tz-lookup`/`geo-tz`/`luxon`/`date-fns-tz`. Timezone handling today is native `Intl` only.
- **Enrich order** (`lib/sync/enrichWithDrivePins.ts`): reel pins → embedded images → linked folder → `enrichAgenda` → `enrichVenueGeocode` → `enrichTransportAssignees`. All tz work lives *inside* the existing `enrichVenueGeocode` step; no new enrich step, no new pipeline wiring.

## 3. Architecture & data flow

```
Google Geocoding (already called once per uncached venue)
  → response results[0].geometry.location {lat, lng}      ← NEW: captured, was discarded
  → geocodeVenueCity returns { city, lat, lng }           ← widened result shape
  → enrichVenueGeocode:
       cache write persists {city, lat, lng}              ← widened cache row
       coordsToTimezone(lat, lng) → "America/Los_Angeles" ← NEW offline derivation
       venue.timezone = <iana>                             ← NEW write (field resolver already reads)
       (no tz derivable) → warnings.push(VENUE_TIMEZONE_UNRESOLVED)
  → resolveShowTimezone(venue) reads venue.timezone        ← UNCHANGED consumer
```

Data flows one way; every new write is best-effort and never throws out of enrich (the existing `try/catch` at `enrichVenueGeocode.ts:72,119` wraps it all).

### 4.2 `lib/time/coordsToTimezone.ts` (NEW — pure helper)
```ts
import tzlookup from "tz-lookup";

/**
 * Offline lat/lng → IANA timezone. Returns null on any invalid/unresolvable input
 * so the caller falls back to the ET default. Never throws.
 */
export function coordsToTimezone(
  lat: number | null | undefined,
  lng: number | null | undefined,
): string | null {
  if (typeof lat !== "number" || typeof lng !== "number") return null;
  if (!Number.isFinite(lat) || !Number.isFinite(lng)) return null;   // NaN / ±Infinity
  if (lat < -90 || lat > 90 || lng < -180 || lng > 180) return null; // out of range
  try {
    const zone = tzlookup(lat, lng);
    // Belt-and-suspenders: validate the IANA name the same way the resolver does.
    new Intl.DateTimeFormat("en-CA", { timeZone: zone });
    return zone;
  } catch {
    return null;
  }
}
```
**Guard conditions (every input):** `null`/`undefined` → null; non-number → null; `NaN`/`±Infinity` → null; `|lat|>90` or `|lng|>180` → null; `tzlookup` throw → null; unparseable IANA result → null. `(0, 0)` is a valid coordinate (returns `Etc/GMT`-family) — not special-cased; a real venue never geocodes to null-island, and if it somehow did, the resolver still validates the name downstream.

```ts
function applyTimezoneOrWarn(
  result: ParseResult,
  venue: NonNullable<ParseResult["show"]["venue"]>,
  lat: number | null,
  lng: number | null,
): void {
  const zone = coordsToTimezone(lat, lng);
  if (zone) {
    venue.timezone = zone;
    return;
  }
  // No coords / unresolvable → the show will render in the ET default. Surface a
  // gate-exempt data-gap so Doug sees it in staged review without a push alert.
  result.warnings.push({
    severity: "warn",
    code: "VENUE_TIMEZONE_UNRESOLVED",
    message:
      "Couldn't determine the venue's time zone, so times will show in Eastern Time",
  });
}
```

## 6. Warning emit matrix (D6, D8) — the load-bearing contract

`venue.timezone` is set ⇔ `coordsToTimezone` returned non-null. `VENUE_TIMEZONE_UNRESOLVED` fires ⇔ a geocode was attempted (cache-hit or live-success) AND no tz resulted. Per `enrichVenueGeocode` path:

| Path (line) | tz set? | `VENUE_TIMEZONE_UNRESOLVED`? | `VENUE_GEOCODE_UNRESOLVED`? | Rationale |
|-------------|---------|------------------------------|-----------------------------|-----------|
| no venue / no name (`:74`) | no | **no** | no | nothing to geocode |
| already-enriched re-entry (`:75`, `venue.city` preset) | unchanged | **no** | no | idempotent guard; pure return — **unreachable in the pipeline**, see §6.1 |
| unconfigured / no API key (`:76`) | no | **no** | no | mirrors geocode silence (local/dev) |
| breaker-open (`:93`) | no | **no** | no | transient infra; retries next sync |
| network-fail (`:98`) | no | **no** (dedup) | **yes** | geocode warning already signals unknown location; tz-unknown is implied |
| cache-hit, coords present (`:86`) | **yes** | no | no | resolved |
| cache-hit, legacy NULL coords (`:86`) | no | **yes** | no | transitional (row cached pre-feature); self-heals when the ~30-day TTL expires → miss → fresh geocode caches coords |
| live-success, coords present (`:113`) | **yes** | no | no | resolved |
| live-success, no coords — e.g. `ZERO_RESULTS` (`:113`) | no | **yes** | no | located nothing usable; ET fallback flagged |

**No ET false-positive:** a genuinely-Eastern venue geocodes to coords → `coordsToTimezone` returns `America/New_York` → `venue.timezone` is *set* → **no warning**. The warning means "we couldn't determine it," never "it happens to be Eastern."

**Dedup is structural:** network-fail returns at `:111` before any tz code; cache-hit/live-success never touch the geocode-warning branch. The two codes cannot both fire for one venue — no runtime flag.

### 6.1 The `:75` idempotency guard does not strand timezone (proof + why the guard is kept as-is)

The guard `if (venue.city?.trim()) return;` (`enrichVenueGeocode.ts:75`) short-circuits before any tz logic when `venue.city` is already set. This raises: could a show end up with `city` set but `timezone` unset, permanently skipping derivation? **No, in the pipeline:**

1. **The parser never sets `venue.city`** (it is pure — `lib/parser/types.ts:114-118` documents city as enrich-only). Every sync re-parses the sheet from scratch, producing a fresh `ParseResult` whose `venue.city` is **unset**.
2. **`enrichVenueGeocode` is called exactly once** per enrich pass (`enrichWithDrivePins.ts`, single call site). It is not re-invoked on the same `ParseResult`.
3. Therefore at real entry `venue.city` is always unset → the `:75` guard never trips → the cache-hit/live-success paths run → `city` **and** `timezone` are set together in the same pass. They are never persisted apart.

The guard is reachable only by **double-enriching the same object in memory** (a defensive re-entry that the pipeline does not perform). Keeping it a pure return is deliberate: (a) it preserves the existing geocode-skip idempotency contract unchanged; (b) even if reached, coords are **not** carried on `venue` (D4) — only in the cache keyed by name+address — so a second pass would re-read the cache and, if the row now has coords, set tz anyway; if it does not, emit `VENUE_TIMEZONE_UNRESOLVED` on the cache-hit path. We do **not** relax the guard to `city && timezone`, because that adds a behavior change to an existing, working guard for a path the pipeline cannot reach. A test pins the idempotency contract (§7).

**Persisted-row note:** a prior sync's `parse_result.show.venue.city` living in the DB is *not* an enrich input — the next sync overwrites the whole `parse_result` from a fresh parse (city unset) before enrich runs. There is no "already-city-in-DB → skip tz" path.

## 10. Disagreement-loop preempts (do-not-relitigate, with citations)

- **Publish-gating dropped intentionally.** The original ask said "warning on *published* shows." `enrichVenueGeocode` receives only the `ParseResult` and cannot read `shows.published` (`enrichVenueGeocode.ts:68-71`; `EnrichContext` carries no show-status field). The sibling `VENUE_GEOCODE_UNRESOLVED` is also publish-blind and unconditional. `gateExempt` already prevents push-digest spam on drafts. Decision ratified in brainstorming; do not add a publish gate.
- **Legacy-cache 30-day self-heal is accepted, not a bug.** A pre-feature cache row (city set, coords NULL) yields `VENUE_TIMEZONE_UNRESOLVED` until its TTL (`GEOCODE_CACHE_TTL_MS = 30 days`, `cache.ts:23`) expires → miss → fresh geocode caches coords → tz resolves. We deliberately do **not** force a re-geocode on a city-cache-hit-without-coords: that would add a network call to the cache-hit path, defeating the breaker/cache design (`enrichVenueGeocode.ts:83-93`). New/changed venues resolve immediately; the warning surfaces the transitional gap meanwhile.
- **Coords live in the cache, not on `venue` jsonb (D4).** tz is the durable artifact; coords are a derivation input. Not persisting coords on `venue` keeps the show payload minimal and avoids a second place to keep in sync.
- **`gateExempt` is correct** (mirrors geocode): a transient location/tz miss must not trip the push-regression gate (`dataGaps.ts:59-63` precedent).
- **`(0,0)` not special-cased** — real venues never geocode there; the downstream `Intl` validation is the safety net.
- **The `:75` city-idempotency guard is kept as-is, not relaxed to `city && timezone`.** §6.1 proves the guard is unreachable-with-city-preset in the pipeline (parser purity + single enrich call), so relaxing it would change a working guard for a path that cannot occur. Pinned by the idempotency test (§7). Do not relitigate as a "timezone stranding" bug.
