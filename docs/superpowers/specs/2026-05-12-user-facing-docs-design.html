1. Goal & scope

Build an in-app wiki at /help whose primary reader is Doug Larson (the sole admin of /admin). The site does three jobs at once:

  1. Adoption. Carries Doug across the gap from his Sheets-only workflow into the FXAV-augmented one — explicit, narrative, finite.
  2. Reference. One page per operator-facing surface (dashboard triage, review queues, parse warnings, per-show panel, preview-as-crew, signed-link distribution, onboarding).
  3. Tour. A single capability-showcase page Doug can use to orient himself or a future successor.

The crew-pages spec §9.0.1 already mandates contextual help affordances — "?" tooltips, "Take the tour", "What does this mean?" links. Those land with M4 / M9 / M10 as plain-text only. M12 builds the destination, and retrofits each affordance with a Learn more → link.

2. Out of scope (explicit deferrals)

Every entry below was considered and rejected for v1. Recording so the round-N reviewer doesn't relitigate.

  • Crew-facing pages — phase 2 only. Auth gating is admin-only across the whole tree in v1.
  • Full-text search — sidebar + Ctrl-F is sufficient for 13 pages. Phase-2 trigger: total pages > ~25.
  • In-app authoring / CMS — MDX/TSX in the repo, PR-reviewed. Doug reads; he does not edit.
  • Versioned docs — single "latest".
  • Changelog / release notes.
  • Dev / internals explainers — those stay in docs/superpowers/specs/*.
  • Concepts / vocabulary track — explanations live inline on operator pages.
  • External hosting (Notion / GitBook) — rejected: divergence from DESIGN.md tokens, loss of in-repo review gate.
  • Separate docs framework (Nextra / Docusaurus / Fumadocs) — rejected: too much framework for 13 pages.

3. Architecture

3.1 System overview

3.2 Pipeline choice corrected r4

@next/mdx (native). Three deps, all maintained by Next.js. Integrates via pageExtensions: ['ts','tsx','mdx'] and withMDX() in next.config.ts. Reuses app/globals.css Tailwind v4 theme tokens. 13 pages do not justify a framework.

Compilation, not prerender. pnpm build compiles every .mdx/.tsx page to an RSC chunk. It does NOT pre-generate static HTML because the requireAdmin() gate (§3.4) forces dynamic-at-request-time rendering. Compile verifies MDX validity + component resolution; the per-request render happens on every authenticated GET. app/help/layout.tsx exports export const dynamic = "force-dynamic" to make this explicit.

3.4 Rendering posture & auth corrected r3

Dynamic at request time. app/help/layout.tsx calls requireAdmin(), which runs Supabase queries (auth.getUser() + rpc("is_admin")) on every request. The auth gate forces the entire /help/* tree out of static prerender into the dynamic / on-demand path — same posture as /admin. MDX content itself is statically compiled to RSC; the gate is dynamic.

Client hydration only for: theme toggle, sidebar current-page highlight.

Mobile-first. Single-column under 768px with sidebar collapsed (same 390px baseline as the crew page).

Auth + infra-error mapping. The helper at lib/auth/requireAdmin.ts:52-126 uses forbidden() for confirmed non-admin and throws AdminInfraError (lib/auth/requireAdmin.ts:41-48) for infra faults. app/help/layout.tsx mirrors the existing app/admin/layout.tsx:47-71 pattern verbatim — try/catch around requireAdmin(), catch AdminInfraError, render the cataloged 500-class surface via messageFor("ADMIN_SESSION_LOOKUP_FAILED"). Phase 2 splits via route groups; static prerender restored for the public tree only.

3.5 Screenshot harness r2 + reproducibility hardening r3

Real screenshots ship in v1. Capture is scripted, deterministic, and reproducible. r3 hardens the pipeline against round-1 finding 4.

3.5.1 Pipeline

  1. Manifest at scripts/help-screenshots.manifest.ts — single source of truth. Per-entry: { key, route, fixture, viewport, theme, waitFor, captureSelector?, expectStableMs? }.
  2. Capture script at scripts/help-screenshots.ts drives a dedicated Playwright project through the reproducibility preconditions (3.5.2) and writes WebP (q90, pinned sharp encoder) to public/help/screenshots/<key>-{light,dark}.webp.
  3. <Screenshot key> MDX component renders <picture> with <source media="(prefers-color-scheme: dark)" srcset="…-dark.webp"> + a default light <img>. AC-12.25 is enforced by a component-level test (§7.1 test 10).
  4. Invocation: pnpm screenshot:help runs the full pipeline on a clean checkout.

3.5.3 CI drift gate

A CI step runs pnpm screenshot:help against a clean checkout, then git diff --exit-code public/help/screenshots/. Non-zero exit fails the PR. This is the load-bearing drift signal — UI changes that shift a documented surface cannot merge without regenerated screenshots. AC-12.19 (idempotency) is what makes this safe.

Page slugs and anchor IDs under /help/* are committed contracts. Renaming any requires a redirect entry in the same change. The build-time anchor resolver (§7 test 1) fails CI on any helpHref that points at a missing anchor or page.
