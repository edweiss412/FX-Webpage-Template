# Spec — Structured field-level From→To for the auto-applied `field_changed` row

**Date:** 2026-07-17
**Slug:** `autoapplied-field-structured-diff`
**Backlog:** `BL-AUTOAPPLIED-FIELD-STRUCTURED-DIFF` · **Deferral:** `DEFERRED.md` AUTOAPPLIED-REDESIGN-3 (P2)
**Branch:** `feat/autoapplied-field-structured-diff`
**Chosen option:** B — "From→To in the row" (structured summary; one aggregate row, no per-field row split, no undo/count semantics change). Options A (name-only) and C (per-field rows + before/after-image snapshots) were rejected — see §2. **Retention invariant (one story, §4):** no `TriggeredReviewItem` shape is widened and **no old financial value is ever persisted**; the newly-retained data is a bounded structured payload in `show_change_log.after_image` (COI old→new + MI-9 role flags/crew name + constant labels/notes) — a classified, accepted new retention surface, admin-only and telemetry-excluded. **MI-9 (LEAD/role delta) IS enriched (§3.2):** owner option B (2026-07-17, PR #439) ratified that a LEAD-bit change auto-applies and is audited (`ROLE_FLAGS_NOTICE` + durable `LEAD_ROLE_APPLIED`), so it legitimately auto-applies and this feature renders its role From→To like the other fields.

---

## 1. Problem

On the admin dashboard's "Recently auto-applied" strip (`components/admin/RecentAutoAppliedStrip.tsx`), a `field_changed` change renders a single generic sentence — **"A field changed on this sync"** (`lib/sync/changeLog/writeAutoApplyChanges.ts:156`). Doug accepts (acknowledges) the change without seeing *which* field changed or its old→new value (impeccable critique P2). The strip is an **acknowledge feed**, not an approval queue: the change is already applied to the DB during sync; "Accept" sets `acknowledged_at` (dismiss from feed) and `field_changed` is **never undoable** (`lib/admin/loadRecentAutoApplied.ts:70`, F17). So the only thing Doug does on the row is confirm he saw it — but today he cannot see what "it" was.

## 2. Goal & option analysis

**Goal:** the `field_changed` row **names each auto-applied show-level field** (and shows old→new where the value is safe and available), in one row, without new per-field rows and without touching undo/acknowledge semantics. The field-family invariant set and the set of syncs that emit a row are unchanged (§3); the enrichment covers **MI-8/8b/8c/9** — the show-level and role fields that auto-apply (MI-9's LEAD/role auto-apply is ratified + audited, §3.2), so it describes only genuinely-applied changes. **Fidelity per field is what is safely derivable without persisting sensitive values** (§3.1): COI shows old→new; a financial *clear* (MI-8) is named + "cleared" (the old number is neither shown nor stored — §4); a role change (MI-9) shows the crew member's role flags old→new; structural changes are named. (The bounded structured payload IS newly retained in `after_image` — §4.)

Three options were mocked (approved comparison artifact: the Option-B card). Decision:

| Option | What | Rejected because |
| --- | --- | --- |
| A — name only | Summary names fields, no values | Doug still can't see old→new; low payoff for near-same effort as B |
| **B — From→To in row (CHOSEN)** | One row names each field + shows old→new where safe (COI) / "cleared" for a financial clear | Delivers the value; reuses the reader's existing `DiffBlock` machinery; no undo/count change; no item-shape widening (only a bounded `after_image` payload is newly retained — §4) |
| C — per-field rows + stored before/after-image | One acknowledge row per field | **Rejected on the user's redundancy concern**: `field_changed` is non-undoable, so per-row granularity buys nothing actionable — it only multiplies dismissal taps for one sheet edit. The group's existing "Accept all" already clears the bucket. |

## 3. What triggers the row (predicate restated — behavior-preserving)

This spec governs only the **Phase-2 writer** `writeAutoApplyChanges` (`writeAutoApplyChanges.ts:143-160`), which today emits one `field_changed` row when **any** of `MI-8 | MI-8b | MI-8c | MI-9` is present in the `triggeredItems`/`notableItems` it receives. **The set of items the writer receives, and which items emit a row, are unchanged by this spec.** The structured enrichment covers **all four** — MI-8/MI-8b/MI-8c (show-level fields) and MI-9 (crew role/LEAD delta), each of which auto-applies once Phase-1 admits apply. Two Phase-1 gates run **before** the writer and are preserved untouched:

- **MI-8/MI-8b unstable-modifiedTime debounce** (`mi8DebounceReason`, `phase1.ts:278-292`): when Drive `modifiedTime` is still unstable and the only review items are MI-8/MI-8b, Phase-1 returns `defer` — the parse does **not** apply and the writer is **never reached** that pass, so **no `field_changed` row is written** for a deferred sync. An MI-8/MI-8b item can therefore exist in a pass that writes no row; this spec does not change that. (MI-9 is not part of the MI-8/MI-8b debounce — a sync carrying an MI-9 item routes to `pass` and reaches the writer.)
- **MI-9 routing is settled — auto-apply, ratified + audited** (§3.2): a LEAD-bit set-membership change auto-applies (owner option B, 2026-07-17, PR #439; ratified amendment #8, `00-overview.md:158-175`) and emits `ROLE_FLAGS_NOTICE` (info) + a durable `LEAD_ROLE_APPLIED` audit `app_event`. Because MI-9 legitimately auto-applies, this feature enriches it like the other field families — a per-crew role From→To entry (§3.1). This feature changes **neither** MI-9's routing nor its existing alert/audit emits; it only adds the strip's structured render.

The writer's implementation shape (fully **behavior-preserving** — a row appears in exactly the same passes as today):

0. Compute `hasFieldFamilyItem` = **any** of MI-8/MI-8b/MI-8c/**MI-9** present in the received items. This is the **row-emission predicate** — identical to today's trigger (`writeAutoApplyChanges.ts:143-160`) — and is computed **before/independently of** entry validation, so it does not depend on how many entries validate.
1. Build the validated `fieldChanges` entries from **MI-8/MI-8b/MI-8c/MI-9** (§3.1, §3.6), tracking `omittedCount` = **malformed** MI-8/8b/8c/9 items dropped by per-entry validation (§3.6). A well-formed item of any family produces its entry; only a corrupt/forward-schema item is skipped and counted.
2. **If ≥1 valid entry** → write the **structured** row (`after_image = { fieldChanges }`, `summary` names the fields, §5). If `omittedCount > 0`, `fieldChanges` also carries one **incompleteness marker** entry (§3.6) — so a co-occurring malformed item is never silently dropped.
3. **Else if `hasFieldFamilyItem`** (a field-family item was present but **every** one was malformed → no valid structured entry) → write the **generic fallback** row (`summary = "A field changed on this sync"`, `after_image = null`), preserving the acknowledge trail exactly as today.
4. **Else** (`!hasFieldFamilyItem`) → no row (same as today).

So the predicate is `hasFieldFamilyItem` (identical to today); only the row's *content* is enriched when any MI-8/8b/8c/9 entry validates. The generic summary is **not** retired — it is the fallback for an **all-malformed** sync (every field-family item corrupt) and for **pre-existing** DB rows (`after_image = null`, written before this deploy). A well-formed MI-9-only sync now writes a **structured** row with its role entry (not the generic fallback). **Do not** weaken or bypass the Phase-1 debounce to "reach" the writer.

### 3.1 Per-invariant enrichment matrix

For each triggering item present, the writer produces a structured **field-change entry**. Entry shape (new): `{ label: string; from: string | null; to: string | null; note: string | null }` — an entry is either a **From→To entry** (`from`/`to` set, `note` null) or a **note-only entry** (`note` set, `from`/`to` null).

| Invariant | Item shape (live: `lib/parser/types.ts`) | Entry produced | Guard |
| --- | --- | --- | --- |
| **MI-8** financial clear (`:537`) | `{ field: "po"\|"proposal"\|"invoice"\|"invoiceNotes" }` — **shape UNCHANGED** (not widened) | **Note-only**: `label` = display name (§3.3), `note` = `"cleared on this sync"`. MI-8 only fires had-value→now-empty (`invariants.ts:447-451`), so the change direction is fully conveyed without the old value. The old financial number is **deliberately not shown or persisted** (§4, privacy). | field enum valid (§3.6) |
| **MI-8b** COI status (`:538`) | `{ prior: string\|null; next: string\|null }` (already carried) | From→To: `label` = `"COI status"`, `from`/`to` via the **trim-aware** normalizer (§3.6): non-empty-after-trim value else `"(none)"` | none |
| **MI-8c** pull-sheet regression (`:539`) | `{ mode: "collapse"\|"ambiguous_format"\|"halved"\|"case_dropped"; details? }` — **live code emits one `case_dropped` item PER missing case** (`invariants.ts:506-515`), so a sync can carry many | **Aggregated by mode (§3.4a):** note-only, `label` = `"Pull sheet"`. Distinct modes each yield one entry; **all `case_dropped` items collapse into ONE count-bearing entry** ("N case(s) removed"). **Ordering:** entries appended in the writer's evaluation order: all MI-8 (financial) first (in the `financialFields` array order — po, proposal, invoice, invoiceNotes), then MI-8b, then the **mode-aggregated** MI-8c entries (§3.4a, in mode order collapse/ambiguous_format/halved/case_dropped), then the **MI-9 role entries in the writer's crew-iteration order** (`priorByName` Map order, `invariants.ts:530`; deterministic). Deterministic, fixture-stable.

### 3.5 No item-shape widening — no new triggered-item persistence (privacy, adversarial-review preempt)

_(The `after_image.fieldChanges` payload IS newly retained — that is the separate, classified surface in §4. This section is only about NOT widening `TriggeredReviewItem`.)_

`TriggeredReviewItem` values are persisted at multiple surfaces: `pending_syncs.triggered_review_items` (staged; **read by `pnpm observe staged --full`**), `sync_audit.triggered_review_items` (`applyStagedCore.ts:258`), and forwarded to the writer via the choice-aware staged path (`applyStagedCore.ts:578`, `:600-605`). **Therefore this spec does NOT widen any `TriggeredReviewItem` shape.** In particular MI-8 stays `{ id, invariant, field }` — carrying old PO/proposal/invoice/`invoiceNotes` values on the item would deposit internal financial values into `pending_syncs`, `sync_audit`, and the observe `staged` surface, all outside the `after_image` analysis in §4. Because MI-8 becomes **note-only** ("cleared on this sync", §3.1), no old financial value is ever read, shown, or stored anywhere new. MI-8b already carries `{ prior, next }` (COI status — low-sensitivity, already persisted today, no new footprint), and **MI-9 already carries `{ crew_name, prior_flags, new_flags }`** (`lib/parser/types.ts:547-550`) — the writer only *reads* these already-persisted fields into `after_image`; no MI-9 shape is widened either. This also means there is **no legacy-vs-new item shape** — staged MI-8/MI-9 items (pre- or post-deploy) are identical, so no transitional-window handling is needed. (MI-8b COI status and MI-9 role flags old→new are the field-values shown; §4 classifies + covers where they render.)

### 3.6 Malformed stored-item validation + value bounding (writer, fail-safe)

Because stored items reaching the writer (§3.5) are guarded only for `id`/`invariant` (`reviewPayloadGuards.ts:53,74`), the structured writer MUST NOT assume any other field is present or well-typed. The prior generic writer never dereferenced `field`/`crew_name`/`prior`/`next`, so a corrupt/forward-schema stored item was harmless; the new writer dereferences them and must **fail safe, never `undefined`, never throw**. Per-entry runtime validation (at the writer, before building each entry) — **a validation failure skips the entry entirely, never producing a placeholder/fake entry** (an all-malformed sync then falls back to the generic row, §3 step 3):

- **MI-8:** `item.field` must be one of `po | proposal | invoice | invoiceNotes`; otherwise **skip** (no partial/`undefined` label). Note-only entry — no value dereference.
- **MI-8c:** `item.mode` must be one of `collapse | ambiguous_format | halved | case_dropped`; otherwise **skip**.
- **MI-9:** `item.crew_name` must be a non-empty string (after trim) and `item.prior_flags`/`item.new_flags` must each be an **array of strings**; otherwise **skip** (no partial/`undefined` label, no fake role entry). Each flag string is coerced trim-aware and the set rendered via the flag-join (§3.4b); an empty array renders `"(none)"`. A well-formed MI-9 item produces its From→To entry and does **not** touch `omittedCount`; only a malformed one is skipped and counted.
- **MI-8b:** requires `prior` and `next` keys that are each `string | null`; then normalize both trim-aware (`x = typeof x === "string" && x.trim() !== "" ? x.trim() : "(none)"`). **Skip if the shape is invalid OR the two normalized values are equal** — this drops a corrupt `{ id, invariant:"MI-8b" }` (both → `"(none)"`, equal → skipped, **no fake `(none)→(none)` row**) and mirrors the live fire condition `priorCoi !== nextCoi` (`invariants.ts:460`). Otherwise emit From→To with the normalized values (never a blank cell).
- **Universal (trim-aware):** every string placed in an entry (`label`/`from`/`to`/`note`) is coerced through `typeof x === "string" && x.trim() !== "" ? x.trim() : <sentinel>` (`<sentinel>` = `"(none)"` for MI-8b values) so the stored JSON never contains `undefined`, `null`, `""`, or whitespace-only in a taken branch — upholding the "no blank cell" guarantee (§7). (`label` values are writer-controlled constants and always non-empty.)

**Omitted items → incompleteness marker (Codex R12/R15).** The writer tracks `omittedCount` = malformed MI-8/8b/8c/9 items skipped by per-entry validation (§3.6). A partial list must never be presented as complete on a whole-row-Accept feed:
- **≥1 valid entry AND `omittedCount > 0`** → after the valid entries, append **one bounded marker entry** `{ label: "Other changes", note: "<omittedCount> other field change(s) on this sync — details unavailable", from: null, to: null }`. The acknowledged row then shows the known changes AND signals that a corrupt item could not be rendered — not a partial list masquerading as complete.
- **0 valid entries** → no structured row; the **row-emission predicate `hasFieldFamilyItem` (§3 step 0)** decides: if any field-family item was present (i.e. every field-family item was malformed) → the **generic fallback row** (`summary = "A field changed on this sync"`, `after_image = null`); else no row.

The marker/fallback is an **in-band data signal** (not telemetry): the writer runs inside the sync advisory-lock txn, so per invariant 10 it emits no `log.*`/`logAdminOutcome` — the marker entry (and the generic-fallback row) carry the "something was omitted" signal in the persisted row itself. This is strictly fail-closed — never a throw that bricks staged apply/finalize, never a silent loss.

**Value bounding (storage + render):** each stored **`label`/`from`/`to`/`note`** string is **truncated at the writer** to **120 chars** with a trailing `…` when longer. The user-sourced strings are MI-8b COI status (`from`/`to`) and MI-9's `crew_name` (in the `label`) + role-flag join (`from`/`to`, §3.4b); the other labels/notes are writer constants (`"PO number"`, `"COI status"`, `"Pull sheet …"`, `"Role — …"` prefix, the marker note) already short. The cap is applied to every string (defence-in-depth, incl. `label` — so a pathological `crew_name` cannot bloat the MI-9 label) so no path is unbounded. This bounds `after_image` size and, combined with the render wrapping contract (§6), prevents one field row from dominating or breaking Doug's mobile-first dashboard.

### 3.2 MI-9 role enrichment (routing resolved — auto-apply, ratified + audited)

**MI-9 is enriched: a per-crew role From→To entry.** MI-9 fires on a **LEAD-bit set-membership delta** (`invariants.ts:530-544`), which is **auth-sensitive** (LEAD grants access to the internal ops surface and `shows_internal` financials — `00-overview.md:151`). Its auto-apply-vs-stage routing was previously contested; **that is now resolved** (owner option B, 2026-07-17, PR #439):

- **Ratified plan amendment #8** (`00-overview.md:158-175`, updated 2026-07-17): "a LEAD-bit change **AUTO-APPLIES** (it no longer stages) … a deliberate sheet edit … severs no access." All `role_flags` deltas auto-apply via the Phase-2 UPSERT and emit a `ROLE_FLAGS_NOTICE` (`admin_alerts`, info severity); a LEAD gain/loss additionally writes a durable, failure-visible `LEAD_ROLE_APPLIED` audit `app_event` (forensic code, recoverable via `observe events`).
- **Live "Phase 2 decision rule" code + passing tests** (`phase1.ts:504-511`; `tests/sync/phase1.test.ts` — MI-11 is the only gated invariant, every other invariant incl. MI-9 auto-applies) — code and the (now-reconciled) ratified plan **agree**: MI-9 auto-applies.

Because MI-9 legitimately auto-applies and is independently audited, its role change **belongs in the auto-applied strip** with rendered detail — consistent with the fix's whole purpose (LEAD-change visibility). **Therefore** the structured `fieldChanges` enrichment covers **MI-8, MI-8b, MI-8c, AND MI-9**. Each MI-9 item (one per crew member whose LEAD membership changed) yields **one From→To entry** — `label = "Role — {crew_name}"`, `from`/`to` = the crew member's role-flag set (§3.1, §3.4b). No omission signal is needed for a well-formed MI-9 item — it is now shown, not hidden. (A **malformed** MI-9 item — empty `crew_name`, non-array `flags` — is skipped and counts toward `omittedCount` like any other family, §3.6; it never fakes a role entry.)

This feature changes **neither** MI-9's routing (auto-apply, `phase1.ts:504-511`) **nor** its existing emits (`ROLE_FLAGS_NOTICE` + `LEAD_ROLE_APPLIED`, `phase2.ts`); it only adds the strip's structured render, reading the MI-9 item's already-carried `{ crew_name, prior_flags, new_flags }` into `after_image` (no item-shape widening, §3.5). The strip render is a redundant, admin-only *third* surface for the same delta (feed alert + durable audit + strip) — reinforcing, never the sole record.

### 3.3 Field display names (single source of truth)

Financial field → display name map (new constant, colocated with the writer):
`po` → `"PO number"`, `proposal` → `"Proposal"`, `invoice` → `"Invoice"`, `invoiceNotes` → `"Invoice notes"`. No raw enum token reaches the DOM (invariant 5 spirit; these are field names, not error codes).

### 3.4 MI-8c mode sentences

`collapse` → `"lost all rows"`; `ambiguous_format` → `"format became ambiguous"`; `halved` → `"lost more than half its cases"`; `case_dropped` → `"{N} case(s) removed"` (N = count of `case_dropped` items — §3.4a). Rendered as `"Pull sheet {sentence}"` in the note. (Descriptive English, no codes — invariant 5.)

### 3.4a MI-8c aggregation (cardinality bound — Codex R19)

The writer receives **one MI-8c item per triggering condition**, and `case_dropped` fires **once per missing pull-sheet case** (`invariants.ts:506-515`), so a large regression can carry hundreds of MI-8c items. Emitting one entry each would bloat `after_image`, overwhelm the strip, and could exceed `READ_FIELDS_ENTRY_CAP` — misclassifying legitimate output as corruption. **The writer aggregates MI-8c before building entries:** group the MI-8c items by `mode`; emit **at most one entry per distinct mode** (dedup `collapse`/`ambiguous_format`/`halved`), and collapse **all** `case_dropped` items into a **single** count-bearing entry `"Pull sheet: N case(s) removed"`. MI-8c thus contributes **≤4 entries** regardless of how many raw items arrive. (This is a writer-side aggregation, not a change to what invariants fire — invariant 7.) A malformed MI-8c item (bad `mode`) is skipped and counts toward `omittedCount` (§3.6) as before.

### 3.4b MI-9 role-flag render + cardinality

**Flag-join.** A role-flag set (`prior_flags`/`new_flags`, each `RoleFlag[]` — a bounded enum: `LEAD, A1, A2, V1, L1, GS, BO, CAM_OP, PTZ, LED, STREAM, GAV, FLOATER, FLOOR, SHOW_CALLER, GREEN_ROOM, OWNER, CONTENT_CREATION, FINANCIALS, ONLY`, `lib/parser/types.ts:113-142`) is rendered as **sorted, comma-joined tokens** (`[...flags].sort().join(", ")`), or `"(none)"` for an empty set. Sorting makes the stored/rendered value deterministic and fixture-stable regardless of the source array order. Role flags are the operational vocabulary Doug already uses (the crew-page role tokens), **not** error codes — rendering the token is consistent with invariant 5 (a field/role name, not a raw error code); no lookup-table mapping is introduced. Each joined string is subject to the universal 120-char cap (§3.6); the enum is small so a legitimate join is short, but the cap still applies defence-in-depth.

**Cardinality (roster-bounded).** MI-9 emits **one item per crew member** with a LEAD-bit delta (`invariants.ts:530-544`), so MI-9 contributes **one entry per such crew member** — bounded by the show's roster, not unbounded, and in realistic syncs 0–1 (rarely a handful). Unlike MI-8c, MI-9 entries are **not** aggregated: each names a distinct crew member and each carries an auth-sensitive change that must be individually visible (never collapsed behind a count). The overall writer entry bound is therefore `≤4 MI-8 + 1 MI-8b + ≤4 MI-8c + R MI-9 + ≤1 marker`, where `R` = crew-with-role-change (roster-bounded). This stays **far under** the `READ_FIELDS_ENTRY_CAP = 500` corruption ceiling (§5) for any real roster, so the cap still never truncates legitimate output.

## 4. Storage (Resolved Decision — no migration)

The structured entries are stored on the existing `show_change_log.after_image` **jsonb** column (`supabase/migrations/20260608000001_show_change_log.sql:17`; freeform, no CHECK — **no migration required**), as `after_image = { fieldChanges: FieldChangeEntry[] }`. `before_image` stays **null** (field_changed is non-undoable — no snapshot needed; consistent with today).

**New retention surface — classified (Codex R10).** `after_image.fieldChanges` **is a new retained copy** of the enriched change data on `show_change_log`, on **both** the live cron/manual path (in-memory `notableItems`) and the staged path. Today a `field_changed` row's `after_image` is `null`; this feature persists a bounded structured payload there. The retained data is deliberately minimized:
- **No old financial value.** MI-8 is note-only (§3.1) — no PO/invoice/proposal number or `invoiceNotes` content is ever read, shown, or stored. The user-sourced retained strings are **MI-8b COI status old→new** and **MI-9's crew name + role-flag sets old→new**. Everything else (financial labels, pull-sheet notes, marker) is a writer constant. **No PO/invoice/proposal numbers, no `invoiceNotes` content, no email/phone.**
- **COI carve-out (accepted).** COI status (`pending`/`received`/`expired`-style operational status) is low-sensitivity — not financial, not PII. Retaining its old→new value in this admin-only log is the minimal way to honor the user-approved Option B (show old→new); making MI-8b note-only too would collapse the feature to the rejected Option A (names only). **Decision: accept COI old→new retention here.**
- **MI-9 role/crew carve-out (accepted).** MI-9's entry retains the crew member's **name** (label) and **role-flag sets** old→new. Neither is a new data class for this table: `show_change_log` already stores crew name (and email/phone) in crew-domain images (see access note below), and role flags are the operational role vocabulary, not financial/contact PII. The same LEAD/role delta is **already** persisted durably by the ratified fix (`LEAD_ROLE_APPLIED` app_event carries `prior_flags`/`new_flags`; `ROLE_FLAGS_NOTICE` carries the delta) — so this adds a redundant admin-only copy on a surface that already holds richer crew data, not a new exposure. Retaining old→new is required to honor Option B for the role field. **Decision: accept MI-9 crew-name + role-flag old→new retention here.**

**Access + telemetry posture:**
- `show_change_log` is REVOKEd from `authenticated` and read only by the service-role loader (`loadRecentAutoApplied.ts:10-13`); the table already carries crew PII (name/email/phone) in crew-domain images — this adds no new PII class.
- The telemetry read-core **never selects `before_image`/`after_image`** (`lib/observe/query/changeLog.ts`, AGENTS.md telemetry §), so `fieldChanges` never reaches the observe CLI. Because no `TriggeredReviewItem` is widened (§3.5), no value enters `pending_syncs`/`sync_audit`/`observe staged`. So the **only** new retention surface is `show_change_log.after_image` (admin-only, telemetry-excluded, bounded per §3.6 + §5).
- `canonImage` (`writeAutoApplyChanges.ts:204`) only touches a top-level `email` key — `fieldChanges` has none, so canonicalization is a correct no-op.
- `undo_change` reads `after_image` only for `crew_added` (`undo_change_rpc.sql:39`); `field_changed` is never undoable, so the RPC never reads these rows. `cleanup_superseded_before_images` touches crew-domain rows only. **No write-path conflict.**

Rejected alternatives: (a) a new dedicated column — needs a migration + validation-parity round for no benefit over the freeform jsonb already read by the loader; (b) encoding into the `summary` string and re-parsing in the reader — brittle, forbidden shape-in-string. The `summary` string is **also** upgraded to a plain-English one-line fallback (§5) but is not the structured source of truth.

## 5. Read path

`lib/admin/loadRecentAutoApplied.ts`:
- Extend `AutoAppliedDiff` (`:24`) with a new variant: `| { kind: "fields"; entries: FieldChangeEntry[] }`.
- `deriveDiff` (`:83`) gains a `field_changed` branch that reads `after_image.fieldChanges` and **re-validates + re-bounds at read time** (defence-in-depth — the loader must not trust the stored payload, which could be a corrupt, manually-repaired, tampered, or future-bug service-role row):
  - Each entry must be an object with a non-empty string `label` and a valid `note` XOR (`from`+`to`); malformed entries are dropped. Each string re-truncated to **120** chars.
  - **Corruption/DoS ceiling — not a legit-data truncation:** `READ_FIELDS_ENTRY_CAP = 500`. The **writer's** entry count is roster-bounded: ≤4 MI-8 financial + 1 MI-8b COI + **≤4 MI-8c (mode-aggregated, §3.4a — `case_dropped` collapses to one count-bearing entry)** + **R MI-9 role entries (one per crew member with a LEAD delta, roster-bounded, §3.4b)** + ≤1 marker. This holds even for a pull-sheet regression dropping hundreds of cases (MI-8c aggregates) and even for a role reshuffle touching the whole roster (R ≤ crew count, dozens at most). So the 500 cap is unreachable by any legitimate writer output — **it never truncates a legit change** and only fires on external corruption/tampering.
  - **Over-cap → explicit visible invalid-payload marker (Codex R15):** a payload with >500 entries → `{ kind: "fields", entries: [ one marker ] }` where the marker is `{ label: "Unavailable", note: "This change record is too large to display safely (<observed length> entries) — review it in the change log", from: null, to: null }`. It uses the **observed** `fieldChanges` length, so the user sees an explicit indication that the record is oversized/invalid — **not** the row's arbitrary stored `summary` (which, on a corrupt row, may be the generic "A field changed on this sync" and would hide that hundreds of entries were suppressed). The loader also emits a `code:`-carrying `log.warn`. (Accept is not disabled — the visible marker is a sufficient signal; disabling acknowledge would be a larger UX change out of scope here.)
  - If ≥1 valid bounded entry remains (≤500) → `{ kind: "fields", entries }`. **If the reader dropped any malformed entry** (stored count > kept count), it appends the same **incompleteness marker** entry (`{ label:"Other changes", note:"some changes could not be displayed" }`) so a read-side drop also never presents a partial list as complete.
  - **Payload-shape classification (the ONLY paths to summary are legacy/empty):** `{ kind: "none" }` → summary is returned **only** when `after_image.fieldChanges` is **absent / `null` / an empty array `[]`** (legacy or generic-fallback rows). **Any other present value is an invalid structured payload** → the "Unavailable" marker (below), never summary:
    - **Present but not an array** (`{...}`, string, number, boolean — a future writer bug or manual repair, Codex R20) → invalid.
    - **Non-empty array whose every entry is malformed** (≤500) (Codex R16) → invalid.
    - **>500 entries** (Codex R15) → invalid (over-cap variant, marker states the observed length).

    For every invalid case, return `{ kind: "fields", entries: [ { label:"Unavailable", note:"This change record could not be displayed — review it in the change log", from:null, to:null } ] }` (the over-cap variant substitutes its observed-length text) + a `code:`-carrying `log.warn`. So a corrupt structured row of **any** shape is never acknowledged as a bare summary with no visible signal.
  - The read-side `log.warn` on an over-cap/invalid payload runs in the admin **read** path (`loadRecentAutoApplied`, not inside a sync lock), so it is a permitted emit — distinct from the writer, which stays telemetry-free inside the lock (§3.6, invariant 10).
- **Scope of the read cap (honest — Codex R14):** the loader `SELECT`s `after_image` **before** `deriveDiff` runs (`loadRecentAutoApplied.ts:147`), so the 500-cap bounds **rendering / DOM work**, NOT the JSONB select/transfer. It is a **render-side** guard. The **write side** is what keeps the select small in normal operation: the writer bounds every row it writes to a roster-bounded count of short entries (§3.4a/§3.4b/§3.6 — MI-8c mode-aggregated, MI-9 one-per-crew, ≪ 500), and the loader already selects `before_image`/`after_image` for every strip row today (crew images can be sizeable) — so an oversized `after_image` can only arise from external corruption/tampering or a future bug, a DB-integrity concern outside this feature's normal path. A DB-side bounded read (e.g. an RPC checking `jsonb_array_length(after_image->'fieldChanges')` before returning) is **deliberately deferred** as disproportionate for that pathological case; if it is ever wanted, it is an additive hardening, not required for this feature. This spec does **not** claim the render cap protects the select.
- The loader's strict `readName`-only posture is preserved for crew kinds; the new branch reads only the controlled `fieldChanges` shape (label/from/to/note strings), never arbitrary image fields.

**Summary (writer) — names field-TYPES + encodes omissions (Codex R16).** Summary-only consumers — notably the **monitor digest**, whose query selects only `scl.summary` (`lib/notify/monitorDigest.ts`) — never see `after_image.fieldChanges` or its marker. So the summary must name what changed (including an auth-sensitive role change) and encode any omission. To keep crew names out of the emailed digest, the summary names **distinct field-TYPE names**, not per-entry labels:
- The type-name for a valid entry is: the financial field name (`"PO number"`/`"Proposal"`/`"Invoice"`/`"Invoice notes"`, each distinct), `"COI status"` (MI-8b), `"Pull sheet"` (any MI-8c), `"Role"` (any MI-9 — **one `"Role"` token regardless of how many crew changed**; per-crew detail + names stay in the strip's `fieldChanges`, never the digest email).
- **Structured row (§3 step 2):** `summary` names up to the first **3** distinct type-names (in the entry ordering of §3.1), then a single trailing `"and M more field change(s)"` clause where **M = (distinctTypeCount − named) + omittedCount** — unnamed type overflow AND omitted (malformed) items collapse into one "M more" count. Examples: one COI + one MI-9 role change → `"COI status, Role changed on this sync"` (both named, M=0); a role reshuffle → `"Role changed on this sync"` (one type, many crew); four financial clears → `"PO number, Proposal, Invoice and 1 more field change changed on this sync"`. One clause, never a double "and …".
- **Generic fallback row (§3 step 3):** the legacy `"A field changed on this sync"` (nothing to name).

Either way a `summary` accompanies every row, names the changed field types (a role change is explicitly `"Role"`, never hidden), and encodes malformed omissions. Digest parity: the digest reads `summary` only, so this flows through with **no digest code change** — but a **new digest test** (§9) MUST assert an MI-8b+MI-9 row's summary **names `"Role"`** (and `"COI status"`) — i.e. the auth-sensitive role change is visible in the digest — and that a valid+malformed row's summary includes the "N other field change" clause. Confirm no digest test pins the old literal as the *only* possible value.

**Reader back-compat:** the `{ kind:"none" }` → summary path renders both the generic-fallback rows (§3 step 3) and **pre-existing** DB rows (written before this deploy, `after_image = null`). The structured `{ kind:"fields" }` path renders new well-formed rows.

## 7. Guard conditions summary (every input)

- **MI-8 (financial):** note-only ("cleared on this sync") — no value dereference, no old number shown/stored (§3.1); a bad/absent `field` enum → entry skipped (§3.6).
- **MI-8b `prior`/`next` null / empty / whitespace:** fires only on `priorCoi !== nextCoi` after normalize (`invariants.ts:460`), but the raw stored value can be `null`, `""`, or whitespace on a real change (`"" → "received"`); the **trim-aware** normalizer (§3.6) maps any of these to `"(none)"` — never a blank cell.
- **MI-9 role delta:** From→To of the crew member's role-flag set (§3.1/§3.4b); `label` = `"Role — {crew_name}"`, `from`/`to` = sorted flag-join or `"(none)"` for an empty set (new-crew LEAD grant → `(none) → LEAD, …`). Empty/non-string `crew_name` or non-array `flags` → entry skipped + `omittedCount++` (§3.6), never a fake role entry. Roster-bounded, one entry per changed crew member (§3.4b).
- **Malformed stored item (bad/absent `field`, `crew_name`, `mode`, `prior`/`next`, non-array `flags`):** writer skips that entry (§3.6) — never `undefined`, never throws. **Mixed** valid + malformed → the valid entries render PLUS one bounded **incompleteness marker** ("N other field change(s)… details unavailable"), so a partial list is never shown as complete (§3.6). **All** malformed → **generic fallback** row (§3 step 3). Never silent loss.
- **Over-long value:** writer truncates each stored `label`/`from`/`to`/`note` to 120 chars + `…` (§3.6); reader wraps via `wrap-break-word` (§6) — no horizontal overflow on mobile. (User-sourced strings: MI-8b COI and MI-9 crew_name + role-flag join; all capped.)
- **`fieldChanges` empty but a field-family item present:** generic fallback row (§3 step 3). Empty `fieldChanges` with **no** field-family item present → no row (same as today).
- **Deferred MI-8/MI-8b (unstable modifiedTime):** Phase-1 `mi8DebounceReason` returns `defer` before the writer runs (§3), so the writer is never reached and **no `field_changed` row** is written that pass — unchanged from today. This spec must not weaken that debounce.
- **Reader-side payload states:** absent/null/`[]` (legacy/generic) → `{ kind: "none" }` + summary; non-empty all-malformed → explicit "Unavailable" marker + warn; >500 → explicit over-cap marker + warn; mixed valid+dropped → valid entries + incompleteness marker (§5). Never a corrupt structured row acknowledged as a bare summary.

13. **No hidden entries in a real sync; reader-side 500-entry corruption/DoS ceiling** — §5/§6; a whole-row Accept must not dismiss changes hidden behind an overflow collapse, so real syncs render in full. The writer's output count = the real applied-item count (roster-bounded, ≪ 500), so the ceiling never truncates a legit change; an over-ceiling row is corrupt/tampered and renders an **explicit observed-length invalid-payload marker** + a warn — never a silent partial render or an arbitrary stored summary.
14. **Read-side re-validation + bounds (render-side only)** — §5; `deriveDiff` does not trust stored `after_image`, re-validating shape, re-truncating strings, capping entry count → an **explicit invalid-payload marker** (over-cap or non-empty-all-malformed) + warn; only absent/null/`[]` falls to summary. The cap is **render-side** (the loader selects `after_image` before `deriveDiff`), so it bounds DOM, not the select; the write-side roster-bounded entry count (§3.4a/§3.4b — MI-8c mode-aggregated, MI-9 one-per-crew) keeps the select small in normal operation. A DB-side bounded read is deferred (Codex R14).
15. **No silent omission on a complete-looking row** — §3.6/§5; `omittedCount` = malformed(MI-8/8b/8c/9) skipped by validation; any omission appends a bounded "N other field change(s)… details unavailable" marker (or, if no valid entry, the generic fallback row). A whole-row Accept never dismisses a silently-omitted change: a well-formed MI-9 role delta is **shown** (its own From→To entry), and a *corrupt* item of any family raises the marker. Over-cap corrupt rows render an explicit observed-length marker, not the arbitrary stored summary. In-band data signal (writer telemetry-free inside the lock, invariant 10; the read-side warn is a permitted read-path emit).
