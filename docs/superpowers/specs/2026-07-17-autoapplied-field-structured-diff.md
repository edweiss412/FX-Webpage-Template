# Spec ‚Äî Structured field-level From‚ÜíTo for the auto-applied `field_changed` row

**Date:** 2026-07-17
**Slug:** `autoapplied-field-structured-diff`
**Backlog:** `BL-AUTOAPPLIED-FIELD-STRUCTURED-DIFF` ¬∑ **Deferral:** `DEFERRED.md` AUTOAPPLIED-REDESIGN-3 (P2)
**Branch:** `feat/autoapplied-field-structured-diff`
**Chosen option:** B ‚Äî "From‚ÜíTo in the row" (structured summary; one aggregate row, no per-field row split, no undo/count semantics change). Options A (name-only) and C (per-field rows + before/after-image snapshots) were rejected ‚Äî see ¬ß2. **Retention invariant (one story, ¬ß4):** no `TriggeredReviewItem` shape is widened and **no old financial value is ever persisted**; the newly-retained data is a bounded structured payload in `show_change_log.after_image` (COI old‚Üínew + constant labels/notes; MI-9 out of scope, ¬ß3.2) ‚Äî a classified, accepted new retention surface, admin-only and telemetry-excluded.

---

## 1. Problem

On the admin dashboard's "Recently auto-applied" strip (`components/admin/RecentAutoAppliedStrip.tsx`), a `field_changed` change renders a single generic sentence ‚Äî **"A field changed on this sync"** (`lib/sync/changeLog/writeAutoApplyChanges.ts:156`). Doug accepts (acknowledges) the change without seeing *which* field changed or its old‚Üínew value (impeccable critique P2). The strip is an **acknowledge feed**, not an approval queue: the change is already applied to the DB during sync; "Accept" sets `acknowledged_at` (dismiss from feed) and `field_changed` is **never undoable** (`lib/admin/loadRecentAutoApplied.ts:70`, F17). So the only thing Doug does on the row is confirm he saw it ‚Äî but today he cannot see what "it" was.

## 2. Goal & option analysis

**Goal:** the `field_changed` row **names each auto-applied show-level field** (and shows old‚Üínew where the value is safe and available), in one row, without new per-field rows and without touching undo/acknowledge semantics. The field-family invariant set and the set of syncs that emit a row are unchanged (¬ß3); every field-family invariant (MI-8/8b/8c/9) auto-applies (¬ß3.2), so the enrichment describes only genuinely-applied changes. **Fidelity per field is what is safely derivable without persisting sensitive values** (¬ß3.1): COI shows old‚Üínew; a financial *clear* (MI-8) is named + "cleared" (the old number is neither shown nor stored ‚Äî ¬ß4); structural/role changes are named. (The bounded structured payload IS newly retained in `after_image` ‚Äî ¬ß4.)

Three options were mocked (approved comparison artifact: the Option-B card). Decision:

| Option | What | Rejected because |
| --- | --- | --- |
| A ‚Äî name only | Summary names fields, no values | Doug still can't see old‚Üínew; low payoff for near-same effort as B |
| **B ‚Äî From‚ÜíTo in row (CHOSEN)** | One row names each field + shows old‚Üínew where safe (COI) / "cleared" for a financial clear | Delivers the value; reuses the reader's existing `DiffBlock` machinery; no undo/count change; no item-shape widening (only a bounded `after_image` payload is newly retained ‚Äî ¬ß4) |
| C ‚Äî per-field rows + stored before/after-image | One acknowledge row per field | **Rejected on the user's redundancy concern**: `field_changed` is non-undoable, so per-row granularity buys nothing actionable ‚Äî it only multiplies dismissal taps for one sheet edit. The group's existing "Accept all" already clears the bucket. |

## 3. What triggers the row (predicate restated ‚Äî behavior-preserving)

This spec governs only the **Phase-2 writer** `writeAutoApplyChanges` (`writeAutoApplyChanges.ts:143-160`), which today emits one `field_changed` row when **any** of `MI-8 | MI-8b | MI-8c | MI-9` is present in the `triggeredItems`/`notableItems` it receives. **The set of items the writer receives, and which items emit a row, are unchanged by this spec.** The structured enrichment covers **MI-8/MI-8b/MI-8c** ‚Äî show-level fields that unambiguously auto-apply once Phase-1 admits apply; **MI-9 is excluded** (¬ß3.2, contested routing) and its contribution is left as-is. Two Phase-1 gates run **before** the writer and are preserved untouched:

- **MI-8/MI-8b unstable-modifiedTime debounce** (`mi8DebounceReason`, `phase1.ts:278-292`): when Drive `modifiedTime` is still unstable and the only review items are MI-8/MI-8b, Phase-1 returns `defer` ‚Äî the parse does **not** apply and the writer is **never reached** that pass, so **no `field_changed` row is written** for a deferred sync. An MI-8/MI-8b item can therefore exist in a pass that writes no row; this spec does not change that.
- **MI-9 out of scope** (¬ß3.2): its LEAD-bit routing (auto-apply vs stage) is a contested code-vs-ratified-plan discrepancy; this feature does not enrich MI-9 and does not change its existing behavior. MI-9 still counts toward row-presence (a MI-9-only sync yields the generic row, as today) but never a structured entry.

The writer's implementation shape (fully **behavior-preserving** ‚Äî a row appears in exactly the same passes as today):

0. Compute `hasFieldFamilyItem` = **any** of MI-8/MI-8b/MI-8c/**MI-9** present in the received items. This is the **row-emission predicate** ‚Äî identical to today's trigger (`writeAutoApplyChanges.ts:143-160`) ‚Äî and is computed **before/independently of** entry validation, so it does not depend on how many entries validate. (MI-9 counts here even though it is never enriched, ¬ß3.2.)
1. Build the validated `fieldChanges` entries from **MI-8/MI-8b/MI-8c** (¬ß3.1, ¬ß3.6), tracking `omittedCount` = (malformed MI-8/8b/8c dropped) + (MI-9 items present, which are never enriched, ¬ß3.2). MI-9 produces no structured entry but **does** increment `omittedCount`.
2. **If ‚â•1 valid entry** ‚Üí write the **structured** row (`after_image = { fieldChanges }`, `summary` names the fields, ¬ß5). If `omittedCount > 0`, `fieldChanges` also carries one **incompleteness marker** entry (¬ß3.6) ‚Äî so a co-occurring malformed OR MI-9 item is never silently dropped.
3. **Else if `hasFieldFamilyItem`** (a field-family item was present ‚Äî MI-9-only, all-malformed MI-8/8b/8c, or both ‚Äî but no valid structured entry) ‚Üí write the **generic fallback** row (`summary = "A field changed on this sync"`, `after_image = null`), preserving the acknowledge trail exactly as today.
4. **Else** (`!hasFieldFamilyItem`) ‚Üí no row (same as today).

So the predicate is `hasFieldFamilyItem` (identical to today); only the row's *content* is enriched when MI-8/8b/8c entries validate. Crucially, an **MI-9-only sync still writes the generic row** (step 3 via `hasFieldFamilyItem`, independent of entry validation) ‚Äî the acknowledge trail for an MI-9 change never disappears (Codex R14). The generic summary is **not** retired ‚Äî it is the fallback for MI-9-only, all-malformed, and pre-existing DB rows. **Do not** weaken or bypass the Phase-1 debounce to "reach" the writer.

### 3.1 Per-invariant enrichment matrix

For each triggering item present, the writer produces a structured **field-change entry**. Entry shape (new): `{ label: string; from: string | null; to: string | null; note: string | null }` ‚Äî an entry is either a **From‚ÜíTo entry** (`from`/`to` set, `note` null) or a **note-only entry** (`note` set, `from`/`to` null).

| Invariant | Item shape (live: `lib/parser/types.ts`) | Entry produced | Guard |
| --- | --- | --- | --- |
| **MI-8** financial clear (`:537`) | `{ field: "po"\|"proposal"\|"invoice"\|"invoiceNotes" }` ‚Äî **shape UNCHANGED** (not widened) | **Note-only**: `label` = display name (¬ß3.3), `note` = `"cleared on this sync"`. MI-8 only fires had-value‚Üínow-empty (`invariants.ts:447-451`), so the change direction is fully conveyed without the old value. The old financial number is **deliberately not shown or persisted** (¬ß4, privacy). | field enum valid (¬ß3.6) |
| **MI-8b** COI status (`:538`) | `{ prior: string\|null; next: string\|null }` (already carried) | From‚ÜíTo: `label` = `"COI status"`, `from`/`to` via the **trim-aware** normalizer (¬ß3.6): non-empty-after-trim value else `"(none)"` | none |
| **MI-8c** pull-sheet regression (`:539`) | `{ mode: "collapse"\|"ambiguous_format"\|"halved"\|"case_dropped"; details? }` | Note-only: `label` = `"Pull sheet"`, `note` = the mode's descriptive sentence (¬ß3.4). No clean from/to ‚Äî established (AUTOAPPLIED-REDESIGN-3 "MI-8c stays a sentence"). | none |

**Ordering:** entries appended in the writer's evaluation order: all MI-8 (financial) first (in the `financialFields` array order ‚Äî po, proposal, invoice, invoiceNotes), then MI-8b, then MI-8c entries. (MI-9 produces no entry ‚Äî ¬ß3.2.) Deterministic, fixture-stable.

### 3.5 No item-shape widening ‚Äî no new triggered-item persistence (privacy, adversarial-review preempt)

_(The `after_image.fieldChanges` payload IS newly retained ‚Äî that is the separate, classified surface in ¬ß4. This section is only about NOT widening `TriggeredReviewItem`.)_

`TriggeredReviewItem` values are persisted at multiple surfaces: `pending_syncs.triggered_review_items` (staged; **read by `pnpm observe staged --full`**), `sync_audit.triggered_review_items` (`applyStagedCore.ts:258`), and forwarded to the writer via the choice-aware staged path (`applyStagedCore.ts:578`, `:600-605`). **Therefore this spec does NOT widen any `TriggeredReviewItem` shape.** In particular MI-8 stays `{ id, invariant, field }` ‚Äî carrying old PO/proposal/invoice/`invoiceNotes` values on the item would deposit internal financial values into `pending_syncs`, `sync_audit`, and the observe `staged` surface, all outside the `after_image` analysis in ¬ß4. Because MI-8 becomes **note-only** ("cleared on this sync", ¬ß3.1), no old financial value is ever read, shown, or stored anywhere new. MI-8b already carries `{ prior, next }` (COI status ‚Äî low-sensitivity, already persisted today, no new footprint). This also means there is **no legacy-vs-new item shape** ‚Äî staged MI-8 items (pre- or post-deploy) are identical, so no transitional-window handling is needed. (COI values on MI-8b are the only field-value old‚Üínew shown; ¬ß4 covers where they render.)

### 3.6 Malformed stored-item validation + value bounding (writer, fail-safe)

Because stored items reaching the writer (¬ß3.5) are guarded only for `id`/`invariant` (`reviewPayloadGuards.ts:53,74`), the structured writer MUST NOT assume any other field is present or well-typed. The prior generic writer never dereferenced `field`/`crew_name`/`prior`/`next`, so a corrupt/forward-schema stored item was harmless; the new writer dereferences them and must **fail safe, never `undefined`, never throw**. Per-entry runtime validation (at the writer, before building each entry) ‚Äî **a validation failure skips the entry entirely, never producing a placeholder/fake entry** (an all-malformed sync then falls back to the generic row, ¬ß3 step 3):

- **MI-8:** `item.field` must be one of `po | proposal | invoice | invoiceNotes`; otherwise **skip** (no partial/`undefined` label). Note-only entry ‚Äî no value dereference.
- **MI-8c:** `item.mode` must be one of `collapse | ambiguous_format | halved | case_dropped`; otherwise **skip**.
- **MI-9:** produces no structured entry at all (¬ß3.2, out of scope) ‚Äî nothing to validate. It is not a *malformed* skip, but its presence **does** increment `omittedCount` (¬ß3.6) so it raises the incompleteness marker when co-occurring with valid entries (never silently omitted).
- **MI-8b:** requires `prior` and `next` keys that are each `string | null`; then normalize both trim-aware (`x = typeof x === "string" && x.trim() !== "" ? x.trim() : "(none)"`). **Skip if the shape is invalid OR the two normalized values are equal** ‚Äî this drops a corrupt `{ id, invariant:"MI-8b" }` (both ‚Üí `"(none)"`, equal ‚Üí skipped, **no fake `(none)‚Üí(none)` row**) and mirrors the live fire condition `priorCoi !== nextCoi` (`invariants.ts:460`). Otherwise emit From‚ÜíTo with the normalized values (never a blank cell).
- **Universal (trim-aware):** every string placed in an entry (`label`/`from`/`to`/`note`) is coerced through `typeof x === "string" && x.trim() !== "" ? x.trim() : <sentinel>` (`<sentinel>` = `"(none)"` for MI-8b values) so the stored JSON never contains `undefined`, `null`, `""`, or whitespace-only in a taken branch ‚Äî upholding the "no blank cell" guarantee (¬ß7). (`label` values are writer-controlled constants and always non-empty.)

**Omitted items ‚Üí incompleteness marker (Codex R12/R15).** The writer tracks `omittedCount` = malformed MI-8/8b/8c skipped + MI-9 items present (¬ß3.2). A partial list must never be presented as complete on a whole-row-Accept feed:
- **‚â•1 valid entry AND `omittedCount > 0`** ‚Üí after the valid entries, append **one bounded marker entry** `{ label: "Other changes", note: "<omittedCount> other field change(s) on this sync ‚Äî details unavailable", from: null, to: null }`. The acknowledged row then shows the known changes AND signals that more occurred (including an auth-sensitive MI-9 LEAD change) ‚Äî not a partial list masquerading as complete.
- **0 valid entries** ‚Üí no structured row; the **row-emission predicate `hasFieldFamilyItem` (¬ß3 step 0)** decides: if any field-family item was present (all-malformed MI-8/8b/8c, or MI-9-only) ‚Üí the **generic fallback row** (`summary = "A field changed on this sync"`, `after_image = null`); else no row.

The marker/fallback is an **in-band data signal** (not telemetry): the writer runs inside the sync advisory-lock txn, so per invariant 10 it emits no `log.*`/`logAdminOutcome` ‚Äî the marker entry (and the generic-fallback row) carry the "something was omitted" signal in the persisted row itself. This is strictly fail-closed ‚Äî never a throw that bricks staged apply/finalize, never a silent loss.

**Value bounding (storage + render):** each stored **`label`/`from`/`to`/`note`** string is **truncated at the writer** to **120 chars** with a trailing `‚Ä¶` when longer. With MI-9 out of scope (¬ß3.2), the only user-sourced strings are MI-8b COI status (`from`/`to`); all labels/notes are writer constants (`"PO number"`, `"COI status"`, `"Pull sheet ‚Ä¶"`, the marker note) already short. The cap is still applied to every string (defence-in-depth, incl. `label`) so no path is unbounded. This bounds `after_image` size and, combined with the render wrapping contract (¬ß6), prevents one field row from dominating or breaking Doug's mobile-first dashboard.

### 3.2 MI-9 is OUT OF SCOPE for structured enrichment (contested routing ‚Äî Codex R13)

**MI-9 is deliberately excluded from this feature's structured field enrichment.** MI-9 fires on a **LEAD-bit set-membership delta** (`invariants.ts:530-544`), which is **auth-sensitive** (LEAD grants access to the internal ops surface and `shows_internal` financials ‚Äî `00-overview.md:151`). Its auto-apply-vs-stage routing is **contested between two authoritative sources**:

- **Ratified plan amendment #8** (`00-overview.md:158-175`): "Phase 1's MI-9 check **stages** ‚Ä¶ when the LEAD-bit set membership differs" ‚Äî tests (a)/(d) assert a LEAD-bit delta **stages** with `MI-9_ROLE_FLAGS_DELTA`.
- **Live "Phase 2 decision rule" code + passing tests** (`phase1.ts:504-511`; `tests/sync/phase1.test.ts:807-808` ‚Äî "MI-11 routes to auto_apply_with_holds, **every other invariant auto-applies**", parametrized over MI-9 LEAD-gain and LEAD-loss): MI-9 **auto-applies**.

This is an unresolved code-vs-ratified-plan discrepancy on an auth-sensitive event, and resolving it is **out of scope for this read/enrichment feature** (it is a pre-existing divergence, surfaced separately ‚Äî see ¬ß11). Building rendered "role updated (applied)" detail on MI-9's contested applied-status would be wrong under either resolution: if the ratified plan wins, MI-9 stages and never auto-applies (so it must not appear in an *auto-applied* strip); if the code wins, MI-9 has its own dedicated `MI-9_ROLE_FLAGS_DELTA` surface.

**Therefore:** the structured `fieldChanges` enrichment covers **MI-8, MI-8b, MI-8c only**. MI-9 gets **no structured entry** (no rendered role/LEAD detail ‚Äî its applied-status is contested and auth-sensitive). But MI-9 is treated as an **always-omitted field-family item** that contributes to the incompleteness signal (¬ß3.6) so it is **never silently dropped from a row Doug accepts as complete** (Codex R15):

- **MI-9 present alongside ‚â•1 valid structured entry** (e.g. COI + MI-9) ‚Üí the row shows the valid entries **plus the bounded incompleteness marker** ("N other field change(s) on this sync ‚Äî details unavailable"), where N includes the MI-9 item(s). The auth-sensitive LEAD change is thus signaled (magnitude), never hidden behind a complete-looking COI/financial/pull-sheet row.
- **MI-9 only** (no MI-8/8b/8c) ‚Üí the **generic fallback row** (`"A field changed on this sync"`), via `hasFieldFamilyItem` (¬ß3 step 3), identical to today.

MI-9's *routing* (auto-apply vs stage) and its existing writer branch are **untouched** (this feature changes neither); it simply never receives structured enrichment and always raises the incompleteness signal when present. (`ROLE_FLAGS_NOTICE`, `phase2.ts:522`, the separate non-LEAD role path, emits no MI-9 item and is likewise out of scope.)

`po` ‚Üí `"PO number"`, `proposal` ‚Üí `"Proposal"`, `invoice` ‚Üí `"Invoice"`, `invoiceNotes` ‚Üí `"Invoice notes"`. No raw enum token reaches the DOM (invariant 5 spirit; these are field names, not error codes).

`collapse` ‚Üí `"lost all rows"`; `ambiguous_format` ‚Üí `"format became ambiguous"`; `halved` ‚Üí `"lost more than half its cases"`; `case_dropped` ‚Üí `"a case was dropped"`. Rendered as `"Pull sheet {sentence}"` in the note. (Descriptive English, no codes ‚Äî invariant 5.)

## 4. Storage (Resolved Decision ‚Äî no migration)

The structured entries are stored on the existing `show_change_log.after_image` **jsonb** column (`supabase/migrations/20260608000001_show_change_log.sql:17`; freeform, no CHECK ‚Äî **no migration required**), as `after_image = { fieldChanges: FieldChangeEntry[] }`. `before_image` stays **null** (field_changed is non-undoable ‚Äî no snapshot needed; consistent with today).

**New retention surface ‚Äî classified (Codex R10).** `after_image.fieldChanges` **is a new retained copy** of the enriched change data on `show_change_log`, on **both** the live cron/manual path (in-memory `notableItems`) and the staged path. Today a `field_changed` row's `after_image` is `null`; this feature persists a bounded structured payload there. The retained data is deliberately minimized:
- **No old financial value.** MI-8 is note-only (¬ß3.1) and MI-9 is out of scope (¬ß3.2) ‚Äî the only user-sourced retained string is **MI-8b COI status old‚Üínew**. Everything else (labels, notes, marker) is a writer constant. **No PO/invoice/proposal numbers, no `invoiceNotes` content, no crew names, no email/phone.**
- **COI carve-out (accepted).** COI status (`pending`/`received`/`expired`-style operational status) is low-sensitivity ‚Äî not financial, not PII. Retaining its old‚Üínew value in this admin-only log is the minimal way to honor the user-approved Option B (show old‚Üínew); making MI-8b note-only too would collapse the feature to the rejected Option A (names only). **Decision: accept COI old‚Üínew retention here.**

**Access + telemetry posture:**
- `show_change_log` is REVOKEd from `authenticated` and read only by the service-role loader (`loadRecentAutoApplied.ts:10-13`); the table already carries crew PII (name/email/phone) in crew-domain images ‚Äî this adds no new PII class.
- The telemetry read-core **never selects `before_image`/`after_image`** (`lib/observe/query/changeLog.ts`, AGENTS.md telemetry ¬ß), so `fieldChanges` never reaches the observe CLI. Because no `TriggeredReviewItem` is widened (¬ß3.5), no value enters `pending_syncs`/`sync_audit`/`observe staged`. So the **only** new retention surface is `show_change_log.after_image` (admin-only, telemetry-excluded, bounded per ¬ß3.6 + ¬ß5).
- `canonImage` (`writeAutoApplyChanges.ts:204`) only touches a top-level `email` key ‚Äî `fieldChanges` has none, so canonicalization is a correct no-op.
- `undo_change` reads `after_image` only for `crew_added` (`undo_change_rpc.sql:39`); `field_changed` is never undoable, so the RPC never reads these rows. `cleanup_superseded_before_images` touches crew-domain rows only. **No write-path conflict.**

Rejected alternatives: (a) a new dedicated column ‚Äî needs a migration + validation-parity round for no benefit over the freeform jsonb already read by the loader; (b) encoding into the `summary` string and re-parsing in the reader ‚Äî brittle, forbidden shape-in-string. The `summary` string is **also** upgraded to a plain-English one-line fallback (¬ß5) but is not the structured source of truth.

## 5. Read path

`lib/admin/loadRecentAutoApplied.ts`:
- Extend `AutoAppliedDiff` (`:24`) with a new variant: `| { kind: "fields"; entries: FieldChangeEntry[] }`.
- `deriveDiff` (`:83`) gains a `field_changed` branch that reads `after_image.fieldChanges` and **re-validates + re-bounds at read time** (defence-in-depth ‚Äî the loader must not trust the stored payload, which could be a corrupt, manually-repaired, tampered, or future-bug service-role row):
  - Each entry must be an object with a non-empty string `label` and a valid `note` XOR (`from`+`to`); malformed entries are dropped. Each string re-truncated to **120** chars.
  - **Corruption/DoS ceiling ‚Äî not a legit-data truncation:** `READ_FIELDS_ENTRY_CAP = 500`. The **writer** only ever emits the *real* count of applied field-family entries in a sync ‚Äî with MI-9 out of scope (¬ß3.2), that is ‚â§4 financial + COI + pull-sheet + a marker (‚â§7); well under 500. So the cap is unreachable by any legitimate writer output ‚Äî **it never truncates a legit change** and only fires on external corruption/tampering.
  - **Over-cap ‚Üí explicit visible invalid-payload marker (Codex R15):** a payload with >500 entries ‚Üí `{ kind: "fields", entries: [ one marker ] }` where the marker is `{ label: "Unavailable", note: "This change record is too large to display safely (<observed length> entries) ‚Äî review it in the change log", from: null, to: null }`. It uses the **observed** `fieldChanges` length, so the user sees an explicit indication that the record is oversized/invalid ‚Äî **not** the row's arbitrary stored `summary` (which, on a corrupt row, may be the generic "A field changed on this sync" and would hide that hundreds of entries were suppressed). The loader also emits a `code:`-carrying `log.warn`. (Accept is not disabled ‚Äî the visible marker is a sufficient signal; disabling acknowledge would be a larger UX change out of scope here.)
  - If ‚â•1 valid bounded entry remains (‚â§500) ‚Üí `{ kind: "fields", entries }`. **If the reader dropped any malformed entry** (stored count > kept count), it appends the same **incompleteness marker** entry (`{ label:"Other changes", note:"some changes could not be displayed" }`) so a read-side drop also never presents a partial list as complete.
  - **Non-empty but zero-valid (all entries malformed, ‚â§500) ‚Üí explicit marker, NOT summary (Codex R16):** distinguish an **absent / null / empty** `after_image.fieldChanges` (legacy or generic-fallback row ‚Üí `{ kind: "none" }` ‚Üí summary) from a **non-empty array whose every entry is malformed** (a corrupt/future-bug structured row). For the latter, return `{ kind: "fields", entries: [ { label:"Unavailable", note:"This change record could not be displayed ‚Äî review it in the change log", from:null, to:null } ] }` + a `code:` `log.warn` ‚Äî the same class of visible invalid-payload signal as the over-cap path, so a corrupt structured row is never acknowledged as if nothing was suppressed.
  - The read-side `log.warn` on an over-cap/invalid payload runs in the admin **read** path (`loadRecentAutoApplied`, not inside a sync lock), so it is a permitted emit ‚Äî distinct from the writer, which stays telemetry-free inside the lock (¬ß3.6, invariant 10).
- **Scope of the read cap (honest ‚Äî Codex R14):** the loader `SELECT`s `after_image` **before** `deriveDiff` runs (`loadRecentAutoApplied.ts:147`), so the 500-cap bounds **rendering / DOM work**, NOT the JSONB select/transfer. It is a **render-side** guard. The **write side** is what keeps the select small in normal operation: the writer bounds every row it writes to ‚â§7 short entries (¬ß3.6), and the loader already selects `before_image`/`after_image` for every strip row today (crew images can be sizeable) ‚Äî so an oversized `after_image` can only arise from external corruption/tampering or a future bug, a DB-integrity concern outside this feature's normal path. A DB-side bounded read (e.g. an RPC checking `jsonb_array_length(after_image->'fieldChanges')` before returning) is **deliberately deferred** as disproportionate for that pathological case; if it is ever wanted, it is an additive hardening, not required for this feature. This spec does **not** claim the render cap protects the select.
- The loader's strict `readName`-only posture is preserved for crew kinds; the new branch reads only the controlled `fieldChanges` shape (label/from/to/note strings), never arbitrary image fields.

**Summary (writer) ‚Äî must ALSO encode omissions (Codex R16).** Summary-only consumers ‚Äî notably the **monitor digest**, whose query selects only `scl.summary` (`lib/notify/monitorDigest.ts`) ‚Äî never see `after_image.fieldChanges` or its marker. So the no-silent-omission guarantee (¬ß3.6) MUST also hold in the `summary` string, or an MI-8b+MI-9 row would email a complete-looking `"COI status changed on this sync"` while the auth-sensitive LEAD delta is visible only in the strip marker. Rule:
- **Structured row (¬ß3 step 2):** `summary` names up to the first **3** valid field labels, then a single trailing `"and M more field change(s)"` clause where **M = (validCount ‚àí named) + omittedCount** ‚Äî i.e. unnamed valid overflow AND omitted (malformed + MI-9) items collapse into one "M more" count. Examples: one COI + one MI-9 ‚Üí `"COI status and 1 more field change changed on this sync"`; four financial clears ‚Üí `"PO number, Proposal, Invoice and 1 more field change changed on this sync"`. One clause, never a double "and ‚Ä¶". MI-9 and malformed omissions are thus conveyed to the digest too.
- **Generic fallback row (¬ß3 step 3):** the legacy `"A field changed on this sync"` (nothing to name).

Either way a `summary` accompanies every row and encodes omissions. Digest parity: the digest reads `summary` only, so this flows through with **no digest code change** ‚Äî but a **new digest test** (¬ß9) MUST assert an MI-8b+MI-9 (and malformed+valid) row's summary includes the "N other field change" clause (failure mode: a digest email hiding an auth-sensitive LEAD change). Confirm no digest test pins the old literal as the *only* possible value.

**Reader back-compat:** the `{ kind:"none" }` ‚Üí summary path renders both the generic-fallback rows (¬ß3 step 3) and **pre-existing** DB rows (written before this deploy, `after_image = null`). The structured `{ kind:"fields" }` path renders new well-formed rows.
