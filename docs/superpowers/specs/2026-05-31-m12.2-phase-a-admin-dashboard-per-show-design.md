## 3. "Live now" — data layer

**Owner ruling (2026-05-30):** a published show is "live" when today falls anywhere in its `travelIn → travelOut`
span, inclusive (FULL active window). Computed from `dates`, **not** `schedule_phases`.

**Timezone — canonical rule (resolves R6 finding 1):** "today" is evaluated in the **show's timezone** via
the shared `resolveShowTimezone(venue)` helper (§3.1(a)) — NOT the admin's browser timezone. This is the
load-bearing half of the owner ruling ("admin 'Live now' and crew 'right now' share ONE notion of live"): the
crew predicate evaluates each show in its own timezone (`buildRightNowContext` derives the tz it passes as
`selectRightNowState` `options.timezone`), so admin must use the SAME resolver, or a non-Eastern show would flip
live/not-live at a different midnight than its crew sees. §3.1(a) extracts that resolver so crew + admin share
it by construction.
(The brief/ruling's "viewer TZ" wording was imprecise shorthand; show-tz is what achieves the stated crew
parity. Since `venue.timezone` is unpopulated today, `resolveTimezone` returns `America/New_York` for every show
— so admin and crew currently use the identical clock, and this rule stays correct when venue tz lights up.)
A **date-boundary test** (§13) pins this: a show whose span boundary differs across tz at the test instant
classifies identically to the crew predicate.

**Why `dates`, not `schedule_phases` (all verified live):**
- `shows` has no `schedule_phases` column and no event-date column other than `dates jsonb`
  (`supabase/migrations/20260501000000_initial_public_schema.sql:12`).
- `schedule_phases` is empty in prod (cron writes `{}` — `lib/sync/runScheduledCronSync.ts:502`) and is
  *derived from* `dates` via `deriveSchedulePhases` (`lib/parser/index.ts:272-307`; fallback at
  `lib/data/getShowForViewer.ts:267-273`). It deliberately omits `travelIn` (`lib/parser/index.ts:266`),
  so its key-range would under-count the full window.
- `dates.{travelIn,set,showDays,travelOut}` are ISO `YYYY-MM-DD` — they become `schedule_phases`'s ISO keys
  and feed the crew span gate `compareIso(todayIso, travelIn) >= 0 && compareIso(todayIso, travelOut) <= 0`
  (`lib/time/rightNow.ts:267-268`).

### 3.1 Shared helpers (orchestrator constraint 1 — pure extraction + negative regression)

**(a) Shared timezone resolver — extract ONE exported helper (resolves R7 finding 1).** There are currently
**two duplicate** show-tz implementations and neither is reusable as cited: `resolveTimezone` is a **private,
unexported** function in `packList.ts:73`, and the **crew right-now path does its OWN inline extraction** in
`buildRightNowContext.ts:83-89` (`venueWithTz?.timezone ... : "America/New_York"`) — it never calls
`resolveTimezone`. Pinning admin parity to `resolveTimezone` would be impossible (unexported) AND tautological
(not the crew path). Fix: extract a single exported `resolveShowTimezone(venue): string` (e.g. `lib/time/showTimezone.ts`)
with the identical logic (`venue.timezone` if non-empty string else `America/New_York`), and **refactor BOTH
`buildRightNowContext.ts:83-89` AND `packList.ts:73` to call it** — behavior-preserving (identical logic today),
so the existing crew + packList test suites MUST stay green (negative-regression). The admin dashboard live
computation (§3.2) uses the SAME helper. Parity is then structural, not asserted.

**(b) Shared live predicate — MUST enforce the crew full-date guard (resolves R9 finding).** A span-only check
(`travelIn ≤ today ≤ travelOut`) is **insufficient for parity**: the crew predicate first rejects incomplete
date data via `hasFullDates` (`rightNow.ts:179-191`), which requires `travelIn` AND `travelOut` AND
**non-empty `showDays`** (itself a Codex round-22 fix — empty `showDays` with travel bounds present is
broken-sheet data that renders `unknown`, never a confident/live state). A span-only admin helper would badge
such a show "Live" while crew shows `unknown` → parity break + authoritative status on invalid data.

New `lib/time/showSpan.ts`:

```
export function hasFullShowDates(dates): boolean   // shared with rightNow.ts (extracted hasFullDates)
export function isShowLiveOnDate(dates: ShowRow["dates"], todayIso: string): boolean
```
- `hasFullShowDates(dates)` = `Boolean(travelIn) && Boolean(travelOut) && Array.isArray(showDays) && showDays.length > 0`
  — the SAME predicate as `rightNow.ts:185-190`. Extract it here (or to a shared module) and **refactor
  `rightNow.ts` to import it** (behavior-preserving). This subsumes orchestrator constraint 2 (null bound →
  not live) AND the dateless case (`countParseableDates === 0` ⊂ `!hasFullShowDates`).
- `isShowLiveOnDate(dates, todayIso)` = `hasFullShowDates(dates) && compareIso(todayIso, dates.travelIn) >= 0 &&
  compareIso(todayIso, dates.travelOut) <= 0`. Taking the **full `dates` object** (not loose bounds) so a caller
  cannot forget the completeness guard.
- **Non-circular topology (R8 finding 3):** `compareIso` is **private** in `rightNow.ts:118`. Move it to a new
  leaf module `lib/time/isoDate.ts` (exported); both `rightNow.ts` and `showSpan.ts` import from there.
  `isoDate.ts` depends on nothing → no cycle. (`showSpan.ts` → `isoDate.ts`; `rightNow.ts` → `showSpan.ts` +
  `isoDate.ts`; `showSpan.ts` never imports `rightNow.ts`.)
- **Behavior-preserving extraction:** `rightNow.ts` imports the shared `hasFullShowDates` + `compareIso`; its
  span gate at `:267-268` and `hasFullDates` call at `:220` keep identical behavior. The existing `rightNow.ts`
  test suite MUST stay green unchanged — the negative-regression proof for crew-side `selectRightNowState`.

### 3.2 Aggregate compute (server, in `fetchDashboardData`)

For each Active row (`archived = false`, §3.3), in **one request-scoped server pass** using a **single
`nowDate()`** read (the existing `now` already hoisted at `Dashboard.tsx:242`): compute
`isLive = published && isShowLiveOnDate(dates, todayIso)` (§3.1(b) — enforces the shared full-date guard, so a
show with travel bounds but empty `showDays` is NOT live, matching crew's `unknown`), where `todayIso` is
resolved in that show's timezone via the **shared `resolveShowTimezone(venue)` helper from §3.1(a)** (the same
helper the crew path now uses). **Unpublished rows are never live** (`isLive=false`) — they carry the publishing
badge instead (§3.3). Store the result as a per-row **`isLive: boolean`**.

**Single source of truth (R2 finding 2):** add `isLive: boolean` to the `ActiveShowRow` type
(`components/admin/ActiveShowsPanel.tsx`); `fetchDashboardData` computes it per row and `liveCount` = `Σ isLive`
over the same rows. `ShowsTable`'s row-level Live pill (§5.2) reads `row.isLive` — it MUST NOT recompute from
display date endpoints or take a second clock read. So the StatStrip `liveCount` and the rendered row Live pills
are computed once, from the same instant + helper, and cannot diverge.

- **TZ resolution (verified, not hedged):** `venue.timezone` does **not** exist on `ShowRow.venue` today
  (no `timezone` field in `lib/parser/types.ts:87-93`). Both existing tz readers default to `America/New_York`
  (`packList.ts:73` private `resolveTimezone`; `buildRightNowContext.ts:83-89` inline). After the §3.1(a)
  extraction, all three (crew right-now, pack-list, admin live) call the one `resolveShowTimezone`, so **today
  every surface uses `America/New_York` for every show** and stays one-clock-correct when venue tz lights up. Admin
  Live-now MUST reuse `resolveTimezone` (or its exact discipline) — do not invent a parallel tz path or a new
  field.
- **Plan-time verification V2:** `dates` is already selected (`Dashboard.tsx:59`); add `venue` (needed by
  `resolveTimezone`) and the per-show `last_synced_at`/`last_sync_status` (for the sync footer, §6) to the
  relevant `select(...)`.
- **Guard:** shows with empty/partial `dates` (e.g. `MI-3_NO_VALID_DATES`) → not live (helper returns false);
  never crash, never miscount.

### 3.3 Dashboard query contract (pin explicitly — R1 finding 1)

The current `fetchDashboardData` shows query (`Dashboard.tsx:56-62`) filters `published = true` and does **not**
filter `archived`. **Both are wrong per the canonical v1 spec** (line 186): "admin read paths (Task 10.6
dashboard, per-show parse panel) do NOT [scope to published] — admin needs to see in-flight finalize rows with a
yellow 'publishing…' badge." So the current `published = true` filter is an **implementation bug** (it hides
in-flight/multi-batch finalize rows + recovery surfaces from the admin); preserving it would violate invariant 7.
Phase A pins the spec-compliant contract:

- **"Active shows" row set = `archived = false`** (BOTH published and unpublished in-flight rows). **Drop** the
  `published = true` filter; **add** `.eq("archived", false)`. This aligns with the canonical admin-visibility
  rule AND keeps archived rows out (archived display + action are Phase B). A published-AND-archived row (dead
  crew links, hidden from crew) thus never leaks into Active shows.
- **Unpublished (`published = false`) non-archived rows render with a "publishing…" badge** (a title-area
  row-state badge, §5.2) — the canonical "yellow publishing… badge." The prototype already maps a `publishing`
  status (`dashboard.jsx:5` StatusCell `publishing → idle`).
- **`isLive` / `liveCount` require `published && !archived && today∈span`** (§3.1/§3.2). An unpublished in-flight
  show is not crew-visible, so it is **never** "Live" (no Live pill, not counted) — it shows the publishing
  badge instead. (Matches the owner ruling: "a *published* show counts as live.")
- **Regression tests (§13):** (a) a `published=true, archived=true` row is absent from Active shows and excluded
  from `liveCount`; (b) a `published=false, archived=false` row **IS shown** with a publishing badge and has
  `isLive=false` (spec-compliant; this REVERSES the earlier published-only assumption).
- `shows.archived`/`published` are real columns (`20260501000000_initial_public_schema.sql:25-26`); Phase A only
  *reads/filters* them, adds no write. (Still no DB writes — §11 holds.)

**First-seen existence set must span ALL shows + carry the slug (R2 finding 1 + R3 finding 2).** The display
query above is filtered (`archived = false`), but the needs-attention "is this a known show?" classification
(§5.3) MUST NOT use a filtered set: a `pending_sync` for an archived OR unpublished *existing* show must classify
as existing-show staged review, not first-seen (today first-seen is inferred by subtracting the published-shows
drive-id set, `Dashboard.tsx:195,218-231` — latent-buggy for unpublished). Phase A adds a **separate read**:
`select drive_file_id, slug, title, archived, published from shows` with **no published/archived filter**. A
`pending_sync` whose `drive_file_id` matches ANY row is `existing_staged`; its `slug` (carried by this read) builds
the `/admin/show/[slug]` link — selecting only `drive_file_id` would classify-but-not-route (broken link) for
exactly the archived/unpublished rows this protects. (Read-only; no DB write; §11 holds.)

### 5.2 `ShowsTable` (replaces `ActiveShowsPanel` row list)
Props: `{ rows: ActiveShowRow[]; now: Date }`. Dense table: **Show** (title + client/meta) / **Dates** /
**Crew** / **Sync status** (`StatusIndicator`) / chevron. Whole row links to
`/admin/show/${encodeURIComponent(slug)}`.
- **Guard conditions:** empty `rows` → existing empty-state copy (preserve `ActiveShowsPanel`'s current empty
  treatment); `title` null → fallback label (preserve current `ActiveShowsPanel` behavior — verify at plan
  time); null `showDateStart`/`showDateEnd` → dash/"dates TBD"; `crewCount` 0 → "0".
- **Sync status column = sync HEALTH only, decoupled from live + publishing (R1 finding 2).** "Live"
  (active-window) and "publishing" (publish-state) are orthogonal to "sync status" (Drive-read health); the
  Sync-status column MUST NOT render live/publishing in place of a sync problem, or a live-but-failing show would
  hide its failure. The column maps the **complete canonical `last_sync_status` enum** — authoritative set is
  master spec line 182: `ok | parse_error | drive_error | sheet_unavailable | pending_review | pending` (plus
  null / not-yet-set) — through a strict **ordered priority** (first match wins),
  independent of live/publishing — **every canonical value is mapped (no fall-through, R3 finding 3 + R5 finding 2):**
  1. `drive_error` → `warn` + "Couldn't reach Drive"
  2. `sheet_unavailable` → `warn` + "Sheet not in folder"
  3. `parse_error` → `warn` + "Couldn't read the sheet"
  4. `pending_review` → `review` + "Changes to review"
  5. `pending` (initial state, spec line 182) → `idle` + "Sync in progress"
  6. `ok` → `positive` (calm non-green) + "Synced {relative}"
  7. null / not-yet-set → `idle` + "Not synced yet"
  8. **any unrecognized value → defensive `warn` + "Unknown sync state"** (NOT idle/clean) — makes future enum
     drift VISIBLE to the operator rather than silently labeling it "not synced".

  (Staleness — an `ok` row whose `last_synced_at` is past a horizon — is a presentation overlay, NOT a
  `last_sync_status` value; deferred to plan-time only if a horizon constant exists, else omitted in Phase A.

  - A show can be Live (or Publishing) in the title AND `warn` in the sync column simultaneously; all must be
    visible. Requires `published` on the row contract (`ActiveShowRow` already has `published`,

- **Mobile:** collapse Dates/Crew/Sync into a stacked sub-line under the title (prototype `show-meta-mobile`,

### 5.3 `NeedsAttentionInbox`

- **Archived existing shows: staged work is SURFACED, not hidden (R8→R10 correction).** Archived shows **already

- **Guard conditions:** empty `items` → `Empty` state "Nothing waiting on you" (prototype `dashboard.jsx:122`).

## 7. `<sheet-name>` interpolation fix (carry-in bug)

`components/admin/PerShowAlertSection.tsx` renders alert copy via `safeDougFacing(code)` (`:52-55,161`) which
calls `messageFor(code)` with **no params** → catalog placeholders like `<sheet-name>` leak literally. The
global `AlertBanner` interpolates correctly by passing `alert.context` (`AlertBanner.tsx:205-218`).
`fetchPerShowAlerts` already selects `context` (`PerShowAlertSection.tsx:72`).

**Fix:** interpolate using the row's `context` — `messageFor(code, alert.context as MessageParams).dougFacing`
— and pass `context` as `params` to the per-show `HelpAffordance` (mirroring `AlertBanner.tsx:227-237`).
`lib/messages/lookup.ts:13-40` already accepts a params arg and normalizes hyphen↔underscore keys.

- **Contract constraint:** the fix must only *call* `messageFor` with its existing `(code, params)` signature —
  it must **not** change `lib/messages/lookup.ts`'s contract (signature/behavior); it touches only call sites
  (`PerShowAlertSection.tsx`, the dashboard inbox). (The brief's now-superseded carve-out hinged a Codex leg on
  whether `lookup.ts`'s contract changed; that's moot — §14 step 4 now reviews the entire Phase-A diff regardless.
  Keeping the fix call-site-only remains the design constraint so it stays a low-risk presentation fix.)
- **Class-sweep covers EVERY admin render path for catalog-backed copy (R4 finding), not just per-show
  alerts.** Phase A rebuilds the dashboard pending queue into `NeedsAttentionInbox` (§5.3), which renders
  `pending_ingestions` — and the current `PendingPanel` calls `messageFor(code)` with **no params**, so
  `last_error_code` rows using `<sheet-name>` (and any other `<…>` token) would still leak literally on the
  redesigned dashboard. Sweep: grep `lib/messages/catalog.ts` for every `<…>` placeholder, then audit every
  admin surface that renders those codes — **per-show alerts, the dashboard `NeedsAttentionInbox`
  pending-ingestion items, and any `admin_alerts` render** — and confirm each passes the matching params.
- **Dashboard pending-ingestion interpolation (§5.3):** pass `{ sheet_name: driveFileName }` (available at
  `Dashboard.tsx:165` → `PendingIngestionRow.driveFileName`) into `messageFor`/`HelpAffordance` for
  pending-ingestion items; when no param source is available, deliberately fall back to `last_error_message`
  rather than render a code or a raw `<…>` token.
- **Negative-regression tests:** (a) a per-show alert whose `context.sheet_name` is populated renders the real
  name, not `<sheet-name>`; (b) a **dashboard `NeedsAttentionInbox` pending-ingestion** item whose
