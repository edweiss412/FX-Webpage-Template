## 3. "Live now" — data layer

**Owner ruling (2026-05-30):** a published show is "live" when today (viewer TZ) falls anywhere in its
`travelIn → travelOut` span, inclusive (FULL active window). Computed from `dates`, **not** `schedule_phases`.

**Why `dates`, not `schedule_phases` (all verified live):**
- `shows` has no `schedule_phases` column and no event-date column other than `dates jsonb`
  (`supabase/migrations/20260501000000_initial_public_schema.sql:12`).
- `schedule_phases` is empty in prod (cron writes `{}` — `lib/sync/runScheduledCronSync.ts:502`) and is
  *derived from* `dates` via `deriveSchedulePhases` (`lib/parser/index.ts:272-307`; fallback at
  `lib/data/getShowForViewer.ts:267-273`). It deliberately omits `travelIn` (`lib/parser/index.ts:266`),
  so its key-range would under-count the full window.
- `dates.{travelIn,set,showDays,travelOut}` are ISO `YYYY-MM-DD` — they become `schedule_phases`'s ISO keys
  and feed the crew span gate `compareIso(todayIso, travelIn) >= 0 && compareIso(todayIso, travelOut) <= 0`
  (`lib/time/rightNow.ts:267-268`).

### 3.1 Shared span helper (orchestrator constraint 1 — pure extraction + negative regression)

New `lib/time/showSpan.ts`:

```
export function isTodayWithinShowSpan(
  todayIso: string,
  travelIn: string | null,
  travelOut: string | null,
): boolean
```
- **Guard (orchestrator constraint 2):** if `travelIn` OR `travelOut` is null/empty → return `false`
  (unbounded window ⇒ NOT live). Conservative default. (Plan-time: if §E or prototype implies a
  `showDays`-span fallback when travel bounds are missing, surface before building — default conservative.)
- Else: `compareIso(todayIso, travelIn) >= 0 && compareIso(todayIso, travelOut) <= 0` (reuse the existing
  `compareIso` from `rightNow.ts`).
- **Behavior-preserving extraction:** refactor `rightNow.ts:267-268` to call this helper. The existing
  `rightNow.ts` test suite MUST stay green unchanged — that green is the negative-regression proof that the
  crew-side `selectRightNowState` behavior is preserved.

### 3.2 Aggregate compute (server, in `fetchDashboardData`)

For each Active row (`archived = false`, §3.3), in **one request-scoped server pass** using a **single
`nowDate()`** read (the existing `now` already hoisted at `Dashboard.tsx:242`): compute
`isLive = published && isTodayWithinShowSpan(todayIso, travelIn, travelOut)`, where `todayIso` is resolved in
that show's timezone via the **existing canonical resolver `resolveTimezone(show)`** (`lib/visibility/packList.ts:72-76`).
**Unpublished rows are never live** (`isLive=false`) — they carry the publishing badge instead (§3.3). Store the
result as a per-row **`isLive: boolean`**.

**Single source of truth (R2 finding 2):** add `isLive: boolean` to the `ActiveShowRow` type
(`components/admin/ActiveShowsPanel.tsx`); `fetchDashboardData` computes it per row and `liveCount` = `Σ isLive`
over the same rows. `ShowsTable`'s row-level Live pill (§5.2) reads `row.isLive` — it MUST NOT recompute from
display date endpoints or take a second clock read. So the StatStrip `liveCount` and the rendered row Live pills
are computed once, from the same instant + helper, and cannot diverge.

- **TZ resolution (verified, not hedged):** `venue.timezone` does **not** exist on `ShowRow.venue` today
  (no `timezone` field in `lib/parser/types.ts:87-93`). The codebase already reads it *defensively* for a
  future task: `resolveTimezone` returns `venue?.timezone` when populated, else `DEFAULT_TZ = "America/New_York"`
  (`packList.ts:60-76`); the crew `selectRightNowState` likewise defaults to `America/New_York`
  (`rightNow.ts:209`). So **today both crew and admin use `America/New_York` for every show**, and reusing
  `resolveTimezone` keeps "one notion of live" true now AND forward-compatible when venue tz lights up. Admin
  Live-now MUST reuse `resolveTimezone` (or its exact discipline) — do not invent a parallel tz path or a new
  field.
- **Plan-time verification V2:** `dates` is already selected (`Dashboard.tsx:59`); add `venue` (needed by
  `resolveTimezone`) and the per-show `last_synced_at`/`last_sync_status` (for the sync footer, §6) to the
  relevant `select(...)`.
- **Guard:** shows with empty/partial `dates` (e.g. `MI-3_NO_VALID_DATES`) → not live (helper returns false);
  never crash, never miscount.

### 3.3 Dashboard query contract (pin explicitly — R1 finding 1)

The current `fetchDashboardData` shows query (`Dashboard.tsx:56-62`) filters `published = true` and does **not**
filter `archived`. **Both are wrong per the canonical v1 spec** (line 186): "admin read paths (Task 10.6
dashboard, per-show parse panel) do NOT [scope to published] — admin needs to see in-flight finalize rows with a
yellow 'publishing…' badge." So the current `published = true` filter is an **implementation bug** (it hides
in-flight/multi-batch finalize rows + recovery surfaces from the admin); preserving it would violate invariant 7.
Phase A pins the spec-compliant contract:

- **"Active shows" row set = `archived = false`** (BOTH published and unpublished in-flight rows). **Drop** the
  `published = true` filter; **add** `.eq("archived", false)`. This aligns with the canonical admin-visibility
  rule AND keeps archived rows out (archived display + action are Phase B). A published-AND-archived row (dead
  crew links, hidden from crew) thus never leaks into Active shows.
- **Unpublished (`published = false`) non-archived rows render with a "publishing…" badge** (a title-area
  row-state badge, §5.2) — the canonical "yellow publishing… badge." The prototype already maps a `publishing`
  status (`dashboard.jsx:5` StatusCell `publishing → idle`).
- **`isLive` / `liveCount` require `published && !archived && today∈span`** (§3.1/§3.2). An unpublished in-flight
  show is not crew-visible, so it is **never** "Live" (no Live pill, not counted) — it shows the publishing
  badge instead. (Matches the owner ruling: "a *published* show counts as live.")
- **Regression tests (§13):** (a) a `published=true, archived=true` row is absent from Active shows and excluded
  from `liveCount`; (b) a `published=false, archived=false` row **IS shown** with a publishing badge and has
  `isLive=false` (spec-compliant; this REVERSES the earlier published-only assumption).
- `shows.archived`/`published` are real columns (`20260501000000_initial_public_schema.sql:25-26`); Phase A only
  *reads/filters* them, adds no write. (Still no DB writes — §11 holds.)

**First-seen existence set must span ALL shows + carry the slug (R2 finding 1 + R3 finding 2).** The display
query above is filtered (`archived = false`), but the needs-attention "is this a known show?" classification
(§5.3) MUST NOT use a filtered set: a `pending_sync` for an archived OR unpublished *existing* show must classify
as existing-show staged review, not first-seen (today first-seen is inferred by subtracting the published-shows
drive-id set, `Dashboard.tsx:195,218-231` — latent-buggy for unpublished). Phase A adds a **separate read**:
`select drive_file_id, slug, title, archived, published from shows` with **no published/archived filter**. A
`pending_sync` whose `drive_file_id` matches ANY row is `existing_staged`; its `slug` (carried by this read) builds
the `/admin/show/[slug]` link — selecting only `drive_file_id` would classify-but-not-route (broken link) for
exactly the archived/unpublished rows this protects. (Read-only; no DB write; §11 holds.)

## 7. `<sheet-name>` interpolation fix (carry-in bug)

`components/admin/PerShowAlertSection.tsx` renders alert copy via `safeDougFacing(code)` (`:52-55,161`) which
calls `messageFor(code)` with **no params** → catalog placeholders like `<sheet-name>` leak literally. The
global `AlertBanner` interpolates correctly by passing `alert.context` (`AlertBanner.tsx:205-218`).
`fetchPerShowAlerts` already selects `context` (`PerShowAlertSection.tsx:72`).

**Fix:** interpolate using the row's `context` — `messageFor(code, alert.context as MessageParams).dougFacing`
— and pass `context` as `params` to the per-show `HelpAffordance` (mirroring `AlertBanner.tsx:227-237`).
`lib/messages/lookup.ts:13-40` already accepts a params arg and normalizes hyphen↔underscore keys.

- **Contract note (brief exception check):** this only *calls* `messageFor` with its existing
  `(code, params)` signature — it does **not** change `lib/messages/lookup.ts`'s contract. Therefore the
  Phase-A no-Codex-diff-review rule holds; **no Codex leg** on `lookup.ts`. Re-confirm at implementation that
  the fix touches only `PerShowAlertSection.tsx` (call sites), not the helper's signature/behavior.
- **Class-sweep covers EVERY admin render path for catalog-backed copy (R4 finding), not just per-show
  alerts.** Phase A rebuilds the dashboard pending queue into `NeedsAttentionInbox` (§5.3), which renders
  `pending_ingestions` — and the current `PendingPanel` calls `messageFor(code)` with **no params**, so
  `last_error_code` rows using `<sheet-name>` (and any other `<…>` token) would still leak literally on the
  redesigned dashboard. Sweep: grep `lib/messages/catalog.ts` for every `<…>` placeholder, then audit every
  admin surface that renders those codes — **per-show alerts, the dashboard `NeedsAttentionInbox`
  pending-ingestion items, and any `admin_alerts` render** — and confirm each passes the matching params.
- **Dashboard pending-ingestion interpolation (§5.3):** pass `{ sheet_name: driveFileName }` (available at
  `Dashboard.tsx:165` → `PendingIngestionRow.driveFileName`) into `messageFor`/`HelpAffordance` for
  pending-ingestion items; when no param source is available, deliberately fall back to `last_error_message`
  rather than render a code or a raw `<…>` token.
- **Negative-regression tests:** (a) a per-show alert whose `context.sheet_name` is populated renders the real
  name, not `<sheet-name>`; (b) a **dashboard `NeedsAttentionInbox` pending-ingestion** item whose
  `last_error_code` is a `<sheet-name>`-bearing MI code renders the real `driveFileName`, not `<sheet-name>`.
  Stash each fix → its test must fail (contract pinned).
