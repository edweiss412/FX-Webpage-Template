## 3. "Live now" — data layer

**Owner ruling (2026-05-30):** a published show is "live" when today falls anywhere in its `travelIn → travelOut`
span, inclusive (FULL active window). Computed from `dates`, **not** `schedule_phases`.

**Timezone — canonical rule (resolves R6 finding 1):** "today" is evaluated in the **show's timezone** via
the shared `resolveShowTimezone(venue)` helper (§3.1(a)) — NOT the admin's browser timezone. This is the
load-bearing half of the owner ruling ("admin 'Live now' and crew 'right now' share ONE notion of live"): the
crew predicate evaluates each show in its own timezone (`buildRightNowContext` derives the tz it passes as
`selectRightNowState` `options.timezone`), so admin must use the SAME resolver, or a non-Eastern show would flip
live/not-live at a different midnight than its crew sees. §3.1(a) extracts that resolver so crew + admin share
it by construction.
(The brief/ruling's "viewer TZ" wording was imprecise shorthand; show-tz is what achieves the stated crew
parity. Since `venue.timezone` is unpopulated today, `resolveTimezone` returns `America/New_York` for every show
— so admin and crew currently use the identical clock, and this rule stays correct when venue tz lights up.)
A **date-boundary test** (§13) pins this: a show whose span boundary differs across tz at the test instant
classifies identically to the crew predicate.

**Why `dates`, not `schedule_phases` (all verified live):**
- `shows` has no `schedule_phases` column and no event-date column other than `dates jsonb`
  (`supabase/migrations/20260501000000_initial_public_schema.sql:12`).
- `schedule_phases` is empty in prod (cron writes `{}` — `lib/sync/runScheduledCronSync.ts:502`) and is
  *derived from* `dates` via `deriveSchedulePhases` (`lib/parser/index.ts:272-307`; fallback at
  `lib/data/getShowForViewer.ts:267-273`). It deliberately omits `travelIn` (`lib/parser/index.ts:266`),
  so its key-range would under-count the full window.
- `dates.{travelIn,set,showDays,travelOut}` are ISO `YYYY-MM-DD` — they become `schedule_phases`'s ISO keys
  and feed the crew span gate `compareIso(todayIso, travelIn) >= 0 && compareIso(todayIso, travelOut) <= 0`
  (`lib/time/rightNow.ts:267-268`).

### 3.1 Shared helpers (orchestrator constraint 1 — pure extraction + negative regression)

**(a) Shared timezone resolver — extract ONE exported helper (resolves R7 finding 1).** There are currently
**two duplicate** show-tz implementations and neither is reusable as cited: `resolveTimezone` is a **private,
unexported** function in `packList.ts:73`, and the **crew right-now path does its OWN inline extraction** in
`buildRightNowContext.ts:83-89` (`venueWithTz?.timezone ... : "America/New_York"`) — it never calls
`resolveTimezone`. Pinning admin parity to `resolveTimezone` would be impossible (unexported) AND tautological
(not the crew path). Fix: extract a single exported `resolveShowTimezone(venue): string` (e.g. `lib/time/showTimezone.ts`)
and **refactor BOTH `buildRightNowContext.ts:83-89` AND `packList.ts:73` to call it**; the admin dashboard live
computation (§3.2) uses the SAME helper. Parity is then structural, not asserted.
- **Validate the tz string (R13 finding 2):** `Intl.DateTimeFormat(..., { timeZone })` (used by
  `formatIsoForTimezone`) **throws on an invalid IANA name**. The resolver MUST **trim** `venue.timezone` and
  return it ONLY if it is a valid IANA zone (e.g. `Intl.DateTimeFormat(undefined,{timeZone}).resolvedOptions()`
  in a try/catch, or a known-zone check); otherwise fall back to `America/New_York`. Without this, a malformed/
  whitespace `venue.timezone` (once that field is populated) would crash BOTH the admin live aggregate and the
  crew right-now path for that show. **Behavior-preserving today** (no row has `venue.timezone`, so every call
  returns `America/New_York` — crew + packList suites stay green); the validation only hardens the future
  populated path. Unit tests: blank, whitespace-only, bogus (`"Not/AZone"`), and valid (`"America/Los_Angeles"`).

**(b) Shared live predicate — MUST enforce the crew full-date guard (resolves R9 finding).** A span-only check
(`travelIn ≤ today ≤ travelOut`) is **insufficient for parity**: the crew predicate first rejects incomplete
date data via `hasFullDates` (`rightNow.ts:179-191`), which requires `travelIn` AND `travelOut` AND
**non-empty `showDays`** (itself a Codex round-22 fix — empty `showDays` with travel bounds present is
broken-sheet data that renders `unknown`, never a confident/live state). A span-only admin helper would badge
such a show "Live" while crew shows `unknown` → parity break + authoritative status on invalid data.

New `lib/time/showSpan.ts`:

```
export function hasFullShowDates(dates: ShowRow["dates"] | null | undefined): boolean
export function isShowLiveOnDate(dates: ShowRow["dates"] | null | undefined, todayIso: string): boolean
```
- **Null-safe (R12 finding 1):** `shows.dates` is **nullable** (`dates jsonb`; the existing dashboard already
  treats it as nullable). Both helpers accept `ShowRow["dates"] | null | undefined` and **return `false` for
  null / non-object dates as the FIRST branch** — never dereference `dates.travelIn` on a null row (that would
  crash the admin dashboard). 
- `hasFullShowDates(dates)` = (dates is a non-null object) `&& Boolean(travelIn) && Boolean(travelOut) &&
  Array.isArray(showDays) && showDays.length > 0` — the SAME predicate as `rightNow.ts:185-190` for non-null
  input. Extract it here (or to a shared module) and **refactor `rightNow.ts` to import it** (behavior-preserving;
  `selectRightNowState`'s `dates` param is already non-null by its callers, so the added null-guard is a
  superset that can't change crew behavior). Subsumes orchestrator constraint 2 (null bound → not live) AND the
  dateless case (`countParseableDates === 0` ⊂ `!hasFullShowDates`).
- `isShowLiveOnDate(dates, todayIso)` = `hasFullShowDates(dates) && compareIso(todayIso, dates.travelIn) >= 0 &&
  compareIso(todayIso, dates.travelOut) <= 0`. Taking the **full `dates` object** (not loose bounds) so a caller
  cannot forget the completeness guard.
- **Non-circular topology (R8 finding 3):** `compareIso` is **private** in `rightNow.ts:118`. Move it to a new
  leaf module `lib/time/isoDate.ts` (exported); both `rightNow.ts` and `showSpan.ts` import from there.
  `isoDate.ts` depends on nothing → no cycle. (`showSpan.ts` → `isoDate.ts`; `rightNow.ts` → `showSpan.ts` +
  `isoDate.ts`; `showSpan.ts` never imports `rightNow.ts`.)
- **Behavior-preserving extraction:** `rightNow.ts` imports the shared `hasFullShowDates` + `compareIso`; its
  span gate at `:267-268` and `hasFullDates` call at `:220` keep identical behavior. The existing `rightNow.ts`
  test suite MUST stay green unchanged — the negative-regression proof for crew-side `selectRightNowState`.

### 3.2 Aggregate compute (server, in `fetchDashboardData`)

For each Active row (`archived = false`, §3.3), in **one request-scoped server pass** using a **single
`nowDate()`** read (the existing `now` already hoisted at `Dashboard.tsx:242`): compute
`isLive = published && isShowLiveOnDate(dates, todayIso)` (§3.1(b) — enforces the shared full-date guard, so a
show with travel bounds but empty `showDays` is NOT live, matching crew's `unknown`), where `todayIso` is
resolved in that show's timezone via the **shared `resolveShowTimezone(venue)` helper from §3.1(a)** (the same
helper the crew path now uses). **Unpublished rows are never live** (`isLive=false`) — they carry the publishing
badge instead (§3.3). Store the result as a per-row **`isLive: boolean`**.

**Single source of truth (R2 finding 2):** add `isLive: boolean` to the `ActiveShowRow` type
(`components/admin/ActiveShowsPanel.tsx`); `fetchDashboardData` computes it per row and `liveCount` = `Σ isLive`

display date endpoints or take a second clock read. So the StatStrip `liveCount` and the rendered row Live pills
are computed once, from the same instant + helper, and cannot diverge.

- **TZ resolution (verified, not hedged):** `venue.timezone` does **not** exist on `ShowRow.venue` today
  (no `timezone` field in `lib/parser/types.ts:87-93`). Both existing tz readers default to `America/New_York`
  (`packList.ts:73` private `resolveTimezone`; `buildRightNowContext.ts:83-89` inline). After the §3.1(a)
  extraction, all three (crew right-now, pack-list, admin live) call the one `resolveShowTimezone`, so **today
  every surface uses `America/New_York` for every show** and stays one-clock-correct when venue tz lights up. Admin
  Live-now MUST reuse `resolveTimezone` (or its exact discipline) — do not invent a parallel tz path or a new
  field.
- **Plan-time verification V2:** `dates` is already selected (`Dashboard.tsx:59`); add `venue` (needed by
  `resolveTimezone`) and the per-show `last_synced_at`/`last_sync_status` (for the sync footer, §6) to the
  relevant `select(...)`.
- **Guard:** shows with empty/partial `dates` (e.g. `MI-3_NO_VALID_DATES`) → not live (helper returns false);
  never crash, never miscount.

### 3.3 Dashboard query contract (pin explicitly — R1 finding 1)

The current `fetchDashboardData` shows query (`Dashboard.tsx:56-62`) filters `published = true` and does **not**
filter `archived`. **Both are wrong per the canonical v1 spec** (line 186): "admin read paths (Task 10.6
dashboard, per-show parse panel) do NOT [scope to published] — admin needs to see in-flight finalize rows with a
yellow 'publishing…' badge." So the current `published = true` filter is an **implementation bug** (it hides
in-flight/multi-batch finalize rows + recovery surfaces from the admin); preserving it would violate invariant 7.
Phase A pins the spec-compliant contract:

- **"Active shows" row set = `archived = false`** (BOTH published and unpublished in-flight rows). **Drop** the
  `published = true` filter; **add** `.eq("archived", false)`. This aligns with the canonical admin-visibility
  rule AND keeps archived rows out (archived display + action are Phase B). A published-AND-archived row (dead
  crew links, hidden from crew) thus never leaks into Active shows.
- **Unpublished (`published = false`) non-archived rows render with a "publishing…" badge** (a title-area
  row-state badge, §5.2) — the canonical "yellow publishing… badge." The prototype already maps a `publishing`
  status (`dashboard.jsx:5` StatusCell `publishing → idle`).
- **`isLive` / `liveCount` require `published && !archived && today∈span`** (§3.1/§3.2). An unpublished in-flight
  show is not crew-visible, so it is **never** "Live" (no Live pill, not counted) — it shows the publishing
  badge instead. (Matches the owner ruling: "a *published* show counts as live.")
- **Regression tests (§13):** (a) a `published=true, archived=true` row is absent from Active shows and excluded
  from `liveCount`; (b) a `published=false, archived=false` row **IS shown** with a publishing badge and has
  `isLive=false` (spec-compliant; this REVERSES the earlier published-only assumption).
- `shows.archived`/`published` are real columns (`20260501000000_initial_public_schema.sql:25-26`); Phase A only
  *reads/filters* them, adds no write. (Still no DB writes — §11 holds.)
- **Bounded Active-shows read — never silently truncate (R15 finding 1).** The Active-shows query is the
  dashboard's primary read; an unbounded `select … from shows where archived = false` would hit PostgREST's row
  cap (~1000 default) and silently drop shows + undercount `liveCount`/`crewTotal` once the table grows. Phase A
  pins: (a) an **explicit ordered `.limit(ACTIVE_SHOWS_CAP)`** (`ACTIVE_SHOWS_CAP` set comfortably above any
  realistic FXAV active-show count — a single production company has tens, not thousands; pin the literal at
  plan time, e.g. 500); (b) an **exact `activeCount`** via `count: 'exact', head: true` so the "Active shows"
  stat is the TRUE total even if rendering is capped; (c) when `activeCount > ACTIVE_SHOWS_CAP`, a **visible
  "showing first N of M — contact the developer"** overflow notice on the table (truncation is NEVER silent).
  **`liveCount` and `crewTotal` are computed over the fetched active set** (`activeShowIds`). When
  `activeCount ≤ ACTIVE_SHOWS_CAP` — the realistic FXAV case, always — the fetched set IS the complete
  `archived=false` set, so both stats are exact + global. **When `activeCount > ACTIVE_SHOWS_CAP` (unreachable at
  this product's scale), the overflow notice degrades ALL THREE derived stats together** (Active table, Live now,
  Crew) — the strip labels Live/Crew as "across the N shown shows", and neither is claimed global (R16 finding).
  So a truncated read never presents a falsely-global Live/Crew number. **Full pagination / a snapshot-consistent
  aggregate RPC for truly-global Live/Crew counts over ALL rows is deferred** (documented in §15, not silent) —
  disproportionate to this product's scale. Regression: active shows beyond the cap → exact `activeCount` total,
  visible overflow notice, AND Live/Crew labeled as shown-row counts (no silent drop, no false-global).

**First-seen existence set must span ALL shows + carry the slug (R2 finding 1 + R3 finding 2).** The display
query above is filtered (`archived = false`), but the needs-attention "is this a known show?" classification
(§5.3) MUST NOT use a filtered set: a `pending_sync` for an archived OR unpublished *existing* show must classify
as existing-show staged review, not first-seen (today first-seen is inferred by subtracting the published-shows
drive-id set, `Dashboard.tsx:195,218-231` — latent-buggy for unpublished). Phase A adds a **separate read keyed
FROM the pending rows' `drive_file_id`s (R13 finding 1 — bounded, not a full-table scan):** collect the distinct
`drive_file_id`s across the `pending_ingestions` + `pending_syncs` rows, then
`select drive_file_id, slug, title, archived, published from shows where drive_file_id IN (<those ids>)` (no
published/archived filter). **Do NOT read `shows` unbounded** — PostgREST caps rows (~1000 default), so an
unbounded `select … from shows` would silently truncate and misclassify existing shows beyond the prefix as
`first_seen` once the table grows; batch the `.in(...)` if the id set ever exceeds a safe param limit. A
`pending_sync` whose `drive_file_id` is in the returned set is `existing_staged`; its `slug` (carried by this
read) builds the `/admin/show/[slug]` link — selecting only `drive_file_id` would classify-but-not-route (broken
link) for exactly the archived/unpublished rows this protects. (Read-only; no DB write; §11 holds.)

### 3.4 Bounded-read invariant — comprehensive sweep (R13→R15 same-vector; AGENTS.md class-closure)

Rounds 13–15 each surfaced the **same** PostgREST row-cap truncation vector on a different read (existence read,
pending reads, active-shows read). Per the project's "same-vector recurrence → comprehensive re-analysis +
structural defense in the same commit" discipline, this section enumerates **every** Supabase read Phase A
adds/touches and its bounding mechanism — so no further per-surface instance is left to a future round:

| Read | Bounding mechanism |
|---|---|
| Active shows (`shows WHERE archived=false`) | explicit `.limit(ACTIVE_SHOWS_CAP)` + exact `activeCount` (`head:true`) + visible overflow notice (§3.3) |
| `crewTotal` (StatStrip) | **exact count** `from('crew_members').select('show_id',{count:'exact',head:true}).in('show_id', activeShowIds)` — NO row fetch (a row-fetch + JS count, current `Dashboard.tsx:84-107`, truncates and undercounts). Exact + global when `activeCount ≤ ACTIVE_SHOWS_CAP`; in the unreachable overflow case it is over the shown set and the strip labels it "across the N shown shows" (§3.3 — not claimed global) |
| per-show `crewCount` (table column) | **`.in('show_id', activeShowIds)` does NOT bound child rows** (R17 — it bounds the parent count only; the matching `crew_members` rows can exceed the cap). Use a non-truncating strategy: **paginate the crew read via `.range()` until a short page is returned** (fetch-until-complete), then count per show client-side; OR a per-show `head:true` exact count per rendered show. NOT a single unbounded `.in()` row fetch (current `Dashboard.tsx:84-107` shape, which truncates) |
| `pending_ingestions` (inbox) | `.limit(RENDER_CAP+1)` + exact count (§5.3) |
| `pending_syncs` (inbox) | `.limit(RENDER_CAP+1)` + exact count (§5.3) |
| existence lookup | `.in(<pending drive_file_ids>)` — bounded by the rendered pending rows (§3.3) |
| per-show `pending_syncs` `.eq(drive_file_id)` | single show — naturally bounded |
| per-show `crew_members` `.eq(show_id)` | single show — naturally bounded |
| `PerShowAlertSection` `admin_alerts` `.eq(show_id)` | single show — naturally bounded |

**Parent-bounded ≠ child-row-bounded (R17 — the rule the R15 sweep missed).** An `.in('<fk>', parentIds)` filter
bounds the number of *parent* ids, NOT the number of matching rows in a **one-to-many child table**. For any
child-table read (e.g. `crew_members` keyed by `show_id`), "bounded" requires one of: a `head:true` **exact
count** (returns a number, never rows — never truncates), a **`.range()` paginate-until-complete** loop, or a
grouped-count RPC — NOT a bare `.in(parentIds)` row fetch. Single-parent `.eq('<fk>', oneId)` reads (per-show
pages) are naturally bounded only because one show's children are small; still prefer counts where only a count
is needed.

**Structural defense (ships with the implementation, not a future round):** the plan adds a task that
enumerates every Supabase read in the Phase-A diff and asserts each carries an explicit bound — AND, for
one-to-many child-table reads, that the bound is a count/paginate strategy (not merely `.in(parentIds)`). No bare
unbounded `.select()` on an unbounded table; no parent-keyed row fetch standing in for a child-row bound. The
bounded-read regression cases (§13) cover the primary surfaces incl. the per-show crew-count child read; the
enumeration task is the CI-time guard against re-introduction. **Vector status: structurally closed** — the
parent-vs-child distinction was the gap that let R17 surface after R15's sweep; with it encoded here + in the
enumeration task, the row-cap truncation class has no remaining un-ruled instance in Phase A.

### 5.2 `ShowsTable` (replaces `ActiveShowsPanel` row list)
Props: `{ rows: ActiveShowRow[]; now: Date }`. Dense table: **Show** (title + client/meta) / **Dates** /
**Crew** / **Sync status** (`StatusIndicator`) / chevron. Whole row links to
`/admin/show/${encodeURIComponent(slug)}`.
- **Guard conditions:** empty `rows` → existing empty-state copy (preserve `ActiveShowsPanel`'s current empty
  treatment); `title` null → fallback label (preserve current `ActiveShowsPanel` behavior — verify at plan
  time); null `showDateStart`/`showDateEnd` → dash/"dates TBD"; `crewCount` 0 → "0".
- **Sync status column = sync HEALTH only, decoupled from live + publishing (R1 finding 2).** "Live"
  (active-window) and "publishing" (publish-state) are orthogonal to "sync status" (Drive-read health); the
  Sync-status column MUST NOT render live/publishing in place of a sync problem, or a live-but-failing show would
  hide its failure. The column maps the **complete canonical `last_sync_status` enum** — authoritative set is
  master spec line 182: `ok | parse_error | drive_error | sheet_unavailable | pending_review | pending` (plus
  null / not-yet-set) — through a strict **ordered priority** (first match wins),
  independent of live/publishing — **every canonical value is mapped (no fall-through, R3 finding 3 + R5 finding 2):**
  1. `drive_error` → `warn` + "Couldn't reach Drive"
  2. `sheet_unavailable` → `warn` + "Sheet not in folder"
  3. `parse_error` → `warn` + "Couldn't read the sheet"
  4. `pending_review` → `review` + "Changes to review"
  5. `pending` (initial state, spec line 182) → `idle` + "Sync in progress"
  6. `ok` → `positive` (calm non-green) + "Synced {relative}"
  7. null / not-yet-set → `idle` + "Not synced yet"

### 5.3 `NeedsAttentionInbox`
Props: `{ items: NeedsAttentionItem[]; totalCount: number; renderedCount: number; overflowCount: number }`
(R18 finding 2 — `items` alone, once capped, cannot express the real total / overflow; the count fields carry
the exact totals from §5.3's count queries so the "+N more" affordance and the Need-review stat are driven by
real data, not the capped array length). `overflowCount = max(0, totalCount - renderedCount)`. Items consolidate
`pending_ingestions` (couldn't-auto-apply, `Dashboard.tsx:160-191`), first-seen `pending_syncs`, and
existing-show staged-review `pending_syncs`.
- **Distinct item variants (R2 finding 1) — different actions:**
  - `pending_ingestion` (hard-fail ingestion) → existing PendingPanel action.
  - `first_seen` (a `pending_sync` whose `drive_file_id` matches **no** show in the all-shows existence set,
    §3.3) → onboarding/staged path.
  - `existing_staged` (a `pending_sync` whose `drive_file_id` **matches any show**, published/unpublished/archived)
    → the **per-show review path** `/admin/show/[slug]` (archived shows route here too — the per-show page is
    archived-safe per §6).
  The variant decides the action; misrouting an existing-show staged change to the first-seen path is the
  concrete failure mode the §13 regression pins.
- **Archived existing shows: staged work is SURFACED, not hidden (R8→R10 correction).** Archived shows **already
  exist today** — `unpublishShow.ts:123-138` (`archiveAndConsumeUnpublishToken`) sets `archived=true` AND
  `published=false` together — and the sync path can stage `pending_syncs` for them by `drive_file_id`. So the
  R8 "silently exclude archived, it's a no-op" plan was wrong: it would hide an archived show's staged changes
  with no review/apply/discard path (sync state stuck, invisible). **Correction:** an archived existing show's
  `pending_sync` is classified `existing_staged` and **DOES surface in the inbox**, routed to `/admin/show/[slug]`
  (the per-show page loads regardless of published/archived — `page.tsx:111-115` selects by slug). The per-show
  page is made **archived-safe** in §6 (reads `archived`, renders an "Archived" indicator instead of mislabeling
  it "Publishing", and gates share controls on `published` so there's no dead crew link). Archived shows still
  never appear in the **Active shows table** (the `archived = false` display filter, §3.3). Regression: an
  archived existing show's `pending_sync` appears in the inbox (NOT `first_seen`), routes to per-show, and the
  staged change is reviewable.
- **Guard conditions:** empty `items` → `Empty` state "Nothing waiting on you" (prototype `dashboard.jsx:122`).
- **Bounded source reads + exact count (R14 finding 1 — same truncation class as §3.3).** Each pending source
  (`pending_ingestions`, `pending_syncs`) MUST be read with an explicit **ordered `.limit(RENDER_CAP + 1)`** (or
  `.range`), NOT an unbounded select that PostgREST silently truncates at its row cap. The **Need-review stat
  total** (§5.1) is the **exact count** per source (`select(..., { count: 'exact', head: true })`, mirroring
  `AlertBanner.tsx:142-156`), so the stat reflects the true total even when the rendered list is capped. The
  inbox renders up to `RENDER_CAP` items with a **"+N more"** affordance computed from the real total minus
  rendered (mirrors the AlertBanner queue-chip). Pin `RENDER_CAP` at plan time (V4). **Regression:** pending rows
  beyond the cap → the Need-review stat reflects the true total AND the inbox shows "+N more" (no silent
  undercount). Classification (first_seen/existing_staged) applies to the rendered rows; the existence `.in(...)`
  read (§3.3) is keyed off those rendered rows' ids.
- **Live-partition filter is mandatory (R7 finding 2).** Both pending reads — `pending_ingestions` AND
  `pending_syncs` — MUST keep `wizard_session_id IS NULL` (the current code already does:
  `Dashboard.tsx:167` and `:204`). The post-onboarding dashboard surfaces ONLY the live partition; wizard-scoped
  onboarding rows (in-flight candidate-folder sessions) MUST NOT leak into `NeedsAttentionInbox`, be counted as
  operator attention, or be routed through first-seen / per-show actions. The all-shows existence read (§3.3) is
  on `shows` (no wizard partition) and is unaffected.
- **Regression (R7 finding 2):** a fixture with a live (`wizard_session_id = NULL`) row AND a wizard-scoped row
  for the same `drive_file_id`; assert ONLY the live row enters `NeedsAttentionInbox`.
- Data sources are existing rows plus the one all-shows `drive_file_id` existence read (§3.3); no DB writes.
