# Spec — Narrow `ROLE_FLAGS_NOTICE` to capability changes (LEAD ∪ FINANCIALS) + reclassify `audience: health → doug`

**Date:** 2026-07-17
**Slug:** `role-flags-notice-lead-only-doug`
**Branch:** `fix/role-flags-notice-lead-doug`
**Class:** UX / alert-audience reconciliation (auth-audit-adjacent — LEAD grants admin/ops/financials)
**Owner decision (2026-07-17, post-ship review of #439):** Seeing the shipped `ROLE_FLAGS_NOTICE` bell alert live, the owner ratified: (a) the bell nudge should fire **only for a capability-role gain/loss — `LEAD` or `FINANCIALS`, the two flags that grant financial-data access (§2.0)** — not routine scope-tile (department) swaps; (b) it is a **Doug-facing operator audit nudge**, not a developer/health concern, so it should be **dismissible**, render a **calm non-alarming tone (not the red critical it shows today)**, and deep-link to the **sheet** (not dev telemetry). The precise tone is the **info/accent tone** (the design's orange accent, `data-tone="info"`) — semantically correct for an `info`-severity notice and calmer than the amber/warn `notice` tone; "amber" in the original framing meant "not red", now pinned precisely to the info/accent tone (Round 1, Codex F2). Forcing the warn/amber `notice` tone on an `info`-severity code would require a per-code `rowTone` special-case (the exact hack the catalog-derived cascade avoids), so it is rejected. This **supersedes `mi9-lead-autoapply-fyi` §7.2** (which ratified `audience: "health"` for minimal fanout). The durable `LEAD_ROLE_APPLIED` app_event — the authoritative, non-losable dev/forensic record — keeps its topology/failure-visibility but its trigger **broadens from LEAD to capability (LEAD ∪ FINANCIALS)** so a financial-access grant is durably audited (§2.1); this spec reduces feed noise, re-homes the operator nudge, and STRENGTHENS the audit (never weakens it).

## 1. Problem

`ROLE_FLAGS_NOTICE` (`lib/messages/catalog.ts:866-883`) is `audience: "health"`, `healthWeight: "notice"`, `severity: "info"`, `resolution: "manual"`. It was reused as a Doug-facing LEAD audit FYI by #439 (`mi9-lead-autoapply-fyi`) but kept its older `health` classification from the 2026-07-04 audience-split sweep. That classification, applied to a Doug-directed message, produces three incoherences (all catalog-derived, no per-code hardcoding):

1. **Not dismissible.** Both resolve routes 403 any health code:
   - `app/api/admin/admin-alerts/[id]/resolve/route.ts:117-118` → `ALERT_HEALTH_RESOLVE_FORBIDDEN`
   - `app/api/admin/show/[slug]/alerts/[id]/resolve/route.ts:123-124` → same

   `BellPanel.tsx:275` gates the Dismiss button on `!isHealth`, and `isHealth = HEALTH_CODES.includes(code)` (`lib/admin/bellFeed.ts:126`). So a health row shows only "View in telemetry" and never clears — and `ROLE_FLAGS_NOTICE` is not auto-resolving (`resolution: "manual"`, absent from `AUTO_RESOLVING_CODES`), so it has no lifecycle to ever leave the feed.

2. **Renders red critical.** `rowTone` (`BellPanel.tsx:128-132`) short-circuits `if (entry.isHealth) return "critical"` **before** consulting severity/weight, so a `notice`-weight, `info`-severity FYI paints as a red `CircleAlert` — contradicting the health rollup's own amber treatment of the same code (`healthRollup` reduces `notice` → amber, `degraded` → red).

3. **Dev-directed action.** `isHealth` rows render the "View in telemetry" link → `/admin/dev/telemetry` (dev-gated), instead of the `openSheet` action the code already carries in `ALERT_ACTION_CODES` (`lib/adminAlerts/alertActions.ts:85`).

Separately, the **producer fires for every applied role_flags change, capability or not.** `roleFlagChangesForNotice` (`lib/sync/phase2.ts:250-296`) pushes a change entry for any existing-crew role_flags diff (`:287-292`) — including **scope-tile-only** swaps (A1 → V1, additive BO) that only change which tile a crew member sees on their own page and carry **no cross-viewer capability** (§2.0). Those generate the same permanent, red, non-dismissible bell alert as a genuine capability change. This is the "notified every time roles change" noise. (Capability changes — LEAD or FINANCIALS — DO warrant the audit surface; §2.0.)

**Dev is already independently covered** by the durable `LEAD_ROLE_APPLIED` app_event (`lib/log/emitLeadRoleApplied.ts`, queryable via `pnpm observe events --code LEAD_ROLE_APPLIED`). The bell nudge is redundant for dev and, as classified, mis-serves Doug.

## 2. Fix (producer narrow + change-log coverage + audience reclassify)

### 2.0 Capability-flag model — narrow on CAPABILITY, not LEAD alone (Round 6, Codex F-financials)

The bell notice + durable event narrowing keys on **capability-affecting** flags, NOT LEAD alone. Verified against live code, TWO `role_flags` values gate sensitive cross-viewer data (the `shows_internal` financials), and both are equally audit-worthy:

- **`LEAD`** — gates FinancialsTile + all scope tiles (`scopeTiles.ts:141` `financialsVisible = isAdmin || LEAD || FINANCIALS`; plus audio/video/lighting scope).
- **`FINANCIALS`** — a distinct non-LEAD `RoleFlag` (`lib/parser/types.ts:140`) that **independently** grants financial-data read (`scopeTiles.ts:141`; `getShowForViewer.ts:375` `financialsEntitled`). Admin-granted via `role_token_mappings` (never from sheet content, per `types.ts:137-139`), so a `[] → [FINANCIALS]` applied-flags delta is a real financial-data entitlement grant.

The scope-tile flags — `A1`, `A2`, `V1`, `L1`, `BO`, `SHOP`, etc. — only change which tile the crew member sees on **their own** page (no cross-viewer / data capability), so they are the genuinely "tile-only" changes that get a change-log row but no bell/event.

Define **`isCapabilityFlag(f) = f === "LEAD" || f === "FINANCIALS"`** and **`capabilityDelta(prior, next) = hasCap(prior,"LEAD") !== hasCap(next,"LEAD") || hasCap(prior,"FINANCIALS") !== hasCap(next,"FINANCIALS")`**. The bell notice + durable event fire on `capabilityDelta`; the change-log row fires on ANY delta (§2.4). **`LEAD-only` anywhere in this spec means `capability-only` (LEAD ∪ FINANCIALS)** — the sections below use "capability" for the predicate and "LEAD" only where a claim is genuinely LEAD-specific.

### 2.1 Producer — narrow the notice to CAPABILITY changes (LEAD or FINANCIALS)

In `roleFlagChangesForNotice` (`lib/sync/phase2.ts:250`), guard the **existing-crew** push (`:287-292`) so it only fires when a **capability** flag toggles:

```ts
if (roleFlagsEqual(priorMember.role_flags, nextMember.role_flags)) continue;
if (!capabilityDelta(priorMember.role_flags, nextMember.role_flags)) continue; // NEW: scope-tile-only swap → no notice
changes.push({ crew_name: nextMember.name, prior_flags: [...], new_flags: [...] });
```

The new-crew branch (`:278`) currently gates on `hasLead(nextMember.role_flags)` — **broaden it to `nextMember.role_flags.some(isCapabilityFlag)`** so a new crew member added WITH `FINANCIALS` (or LEAD) also produces the notice (a new-crew capability grant). Rename the function `roleFlagChangesForNotice` → `capabilityRoleChangesForNotice` (it now returns capability changes) and sweep its single caller (`:547`) + the comment block (`:245-249`). Add `isCapabilityFlag`/`capabilityDelta` helpers near `hasLead` (`:241`).

**Scope isolation (verified, load-bearing):** the capability-only `capabilityRoleChangesForNotice`/`roleFlagsNotice` list feeds **only** the alert (`upsertAdminAlert`) and the durable event (`emitLeadRoleApplied`). **The durable event must be BROADENED from LEAD-only to capability-toggle (Round 6, Codex F-financials):** `emitLeadRoleApplied.ts:39` currently skips entries where `hasLead(prior) === hasLead(new)` — change that filter to `!capabilityDelta(prior, new)` so a `FINANCIALS` grant/loss ALSO emits the durable audit event (a financials entitlement change is as audit-worthy as LEAD). Keep the forensic code `LEAD_ROLE_APPLIED` (it is internal/observe-only, not a §12.4 user-facing code; its `context` already carries `prior_flags`/`new_flags`, so the specific capability + direction is recoverable) but generalize its `direction` derivation and its doc-comment to "capability role applied". This is a DELIBERATE change to the mi9 durable-event filter (§3 non-goal amended). **The change-log rows (§2.4) are a SEPARATE, FULL role diff** (all flags) computed by `writeRoleChangeLogRows` at the same `runPhase2` point — do NOT reuse the narrowed capability-only notice list for the change log, or scope-tile rows would be lost. Two derivations from the same applied lists: capability-only → alert+event; all-role → change-log.

**Non-LEAD audit gap this exposes (owner-decided: close it — §2.4).** Contrary to a naive reading, a non-LEAD-only role_flags change on an existing crew member does **NOT** produce a `show_change_log` row today. The `field_changed` change-log row (`writeAutoApplyChanges.ts:142-160`) is gated on the **MI-9 invariant, which is LEAD-specific** (`lib/parser/invariants.ts:530` — "LEAD-bit set-membership delta"; MI-10 `:547-561` is also a LEAD safety net). A pure non-LEAD change (A1 → V1) fires no MI-9/MI-10 and matches no `crew_added/removed/renamed` roster change, so it gets **no change-log row** — its only current structured trace is the `ROLE_FLAGS_NOTICE` alert we are narrowing away. Left as-is, narrowing would leave non-LEAD swaps with **only** the live `crew_members.role_flags` value and no history. The owner elected to **preserve change-log coverage** (§2.4) so non-LEAD swaps remain auditable in the change log without the bell nag.

### 2.4 Change-log coverage for role changes — IDENTIFIABLE per-member rows

**Write from the SHARED `runPhase2` point so BOTH apply paths are covered (Round 3, Codex F5).** The row is written by a shared helper **`writeRoleChangeLogRows(port, showId, driveFileId, previousCrewMembers, appliedCrewMembers, identityLinkRenames, occurredAt?)`** (mirrors `writeAutoApplyChanges`, which also takes `showId` + `driveFileId` explicitly — both available in `runPhase2` as `snapshot.showId` and `args.driveFileId`) invoked inside `runPhase2` at the point role changes are known (`phase2.ts:~547`, where `roleFlagChanges` is already computed, INSIDE the locked tx via `port`), **unconditional of `feedPolicy`** — NOT inside `writeAutoApplyChanges` (which is skipped on the staged path via `feedPolicy: { kind: "none" }`, `applyStaged.ts:1400`). **The signature takes `identityLinkRenames` (NOT `renames`/`notableItems`) and has NO `heldNames` parameter** — the writer MUST NOT read open holds and MUST NOT derive renames from `notableItems`; it consumes exactly the three inputs `capabilityRoleChangesForNotice` consumes (§2.4 below), so its coverage cannot diverge from the LEAD producer's. `runPhase2` is the shared apply core for BOTH the cron auto-apply path AND the staged-apply path (`applyStagedCore` → `runPhase2`), so a single write there covers every path with no per-path branching.

**Source is `auto_apply` on every path — accurate, no CHECK migration.** A `role_flags` change is NEVER individually held: only MI-11 email changes gate; a role change always auto-applies via Phase 2 UPSERT, even when the *sync* was triggered by a staged approval of a co-occurring held item. So labeling its change-log row `source: 'auto_apply'` is correct on both paths (the role change WAS auto-applied), it needs no new `show_change_log.source` value (CHECK `20260608000001_show_change_log.sql:26` = `('auto_apply','mi11_approve','mi11_reject','undo')` is untouched), and it correctly surfaces in the auto-applied "Earlier" feed + acknowledge/badge (Doug SHOULD see an auto-applied role change there). This closes the staged non-LEAD gap for every role change on a has-a-prior member (LEAD notice arm (a)) — one identifiable row on every path; §2.5 states the two documented exclusions (new-crew-with-capability arm (b); staged identity-linked rename, backlogged).

**(Round 1, Codex F1.)** A generic notification-only `field_changed` row (`entityRef: null`, summary "A field changed on this sync", null images) is NOT auditable — an operator cannot tell *which* crew member changed or *what* the role went from/to. The change-log feed (`readShowChangeFeed.ts`) and `observe changes` (`queryChangeLog`) render the **`summary`** (feed selects `summary, entity_ref, change_kind, …` at `:225`; `queryChangeLog` deliberately never selects `before_image`/`after_image`), so identifiability MUST live in the rendered `summary`, not in images. (`entity_ref` is kept **null** for supersession safety — §2.4 "Supersession safety" — so the summary is the sole identity carrier, exactly as existing MI-8 `field_changed` rows do.)

**Coverage principle (Round 4 comprehensive re-analysis, Codex F5 vector — 3rd surface: held folds).** The writer emits one identifiable row for **every `role_flags` delta between `previousCrewMembers` and `appliedCrewMembers` (rename-resolved), with NO ad-hoc exclusions** — it MIRRORS `capabilityRoleChangesForNotice` exactly (which itself does NOT skip held names). Anything the LEAD notice/event catches, the change-log row catches; the only difference is LEAD-vs-all scope. There is no held-skip, no path-skip, no invariant gate — the applied-list diff IS the complete, defensible rule:

The writer consumes the **EXACT SAME three inputs** as `capabilityRoleChangesForNotice` — `snapshot.previousCrewMembers`, `applyOutcome.appliedCrewMembers`, and **`args.identityLinkRenames`** (NOT `renames` derived from `notableItems`) — and computes its `priorNameForAdded` map identically (`phase2.ts:263-265`). This is the structural close (Round 4, same-vector — staged rename+role): the writer's coverage is provably **the LEAD producer's existing-member arm (a) minus the capability filter, on EVERY path** (new-crew arm (b) is a `crew_added` case, excluded — see the new-crew note above), because they diff the identical inputs. Whatever rename resolution the LEAD notice/event get on a given path (cron threads `identityLinkRenames`; staged/manual pass empty), the change-log writer gets the same — they cannot diverge.

```ts
// writeRoleChangeLogRows(port, showId, driveFileId, previousCrewMembers, appliedCrewMembers, identityLinkRenames, occurredAt?)
const prevByName = new Map(previousCrewMembers.map((m) => [m.name, m]));
const priorNameForAdded = new Map(identityLinkRenames.map((r) => [r.addedName, r.removedName])); // SAME as capabilityRoleChangesForNotice :263-265
for (const next of appliedCrewMembers) {

  const prior = prevByName.get(priorName);
  if (!prior) continue;                                                 // no prior → a crew_added, NOT a role change (see new-crew note below)
  if (roleFlagsSetEqual(prior.role_flags, next.role_flags)) continue;   // no delta → no row

  //   summary=`Crew member ${next.name} role assignment changed: ${fmt(prior.role_flags)} → ${fmt(next.role_flags)}`,

```

- **New-crew-with-capability is a notice/event-only exception — NOT part of the change-log parity (Round 5, Codex F-parity).** `capabilityRoleChangesForNotice` has TWO arms: (a) the existing-member diff (`:287-292`) and (b) a **new-crew-with-capability** branch (`:278`, `prior_flags: []`, gated on `some(isCapabilityFlag)`) that has NO prior row. The change-log writer covers arm (a) only — a genuinely-new crew member is a **`crew_added`** (logged by `writeAutoApplyChanges` on the auto path), NOT a role *change*, so the `!prior` skip is correct and intentional. New-crew-with-capability is therefore covered by `crew_added` + the capability notice + the durable `LEAD_ROLE_APPLIED` event — it does not need (and must not get) a role-CHANGE row. **The coverage-parity invariant is scoped to arm (a): the writer's row set == the EXISTING/rename-resolved (has-a-prior) members `capabilityRoleChangesForNotice` arm (a) would flag, minus the capability filter.** It is NOT parity against the full unfiltered producer (which includes arm (b)).

**Staged rename handling — no regression (Round 4, Codex F5-staged-rename).** On the staged path `args.identityLinkRenames` is empty (`applyStagedCore` does not thread it — same as it does for the LEAD notice). With no rename map, an identity-linked name change is applied by `applyParseResult` as a **remove-old + add-new** (not a retained-member rename), so the successor appears as a genuinely-new member (`!prior` → no role-CHANGE row) and the old name as a removal — which is the ACTUAL DB semantics on that path, not a silent role-change loss. Because BOTH producers share `args.identityLinkRenames`, the LEAD notice behaves identically; this spec introduces NO divergence and NO regression. If threading `identityLinkRenames` through `applyStagedCore` is later deemed necessary, it is a **pre-existing mi9 property affecting the LEAD notice/event identically** → filed as `BL-STAGED-IDENTITYLINK-RENAMES` scoped to BOTH producers, explicitly out of scope here. The common staged case — an **MI-11 email hold** — retains the member under its OLD name (no successor mismatch), so its held-fold role change IS caught by the plain name diff (§2.4 held-fold coverage).

- **NO held-name skip (Round 4, Codex F5 3rd surface).** An MI-11 email hold pins the *identity* but applies non-identity fields — including `role_flags` — onto the RETAINED row (`applied` list, mi9 §3.1). `capabilityRoleChangesForNotice` deliberately does NOT skip held names so a **held-fold LEAD change** is caught; the change-log writer MUST match, or a **held-fold non-LEAD role change** commits with neither the (narrowed) bell alert NOR a change-log row — the exact gap. The retained held row appears under its name in `appliedCrewMembers`, so the plain name diff catches it (no rename map needed; a held rename SUPPRESSES the new name, retaining the old). The `sync_holds` entry is identity-focused and does NOT carry the prior/new role delta, so it is NOT a substitute.
- **Identity-row vs field-row asymmetry (intentional).** `writeAutoApplyChanges` skips held names for `crew_added/removed/renamed` (IDENTITY changes surfaced separately via `sync_holds`). The role-`field` row is NOT an identity change and is NOT surfaced by `sync_holds`, so it is emitted regardless of hold status. Different concerns, different rules — documented so a reviewer does not "restore" a held-skip by analogy to the identity rows.
- `fmt` renders the flag array as a readable token list (`A1` / `A1, BO` / `none` for `[]`). Role tokens are crew-domain data (like the crew name already in `crew_added` summaries), NOT error/invariant codes — invariant 5 holds.
- **SUBSUMES the MI-9 arm of the generic `field_changed` block in `writeAutoApplyChanges.ts`** (`:142-160`): remove `MI-9` from that arm (keep `MI-8`/`MI-8b`/`MI-8c`) so a LEAD change no longer emits the anonymous "A field changed" row — the shared writer emits its identifiable row instead PLUS the durable `LEAD_ROLE_APPLIED` event. No double-emit: role rows come ONLY from the shared writer. Because the writer runs unconditionally in `runPhase2` (not gated by `feedPolicy`), removing the MI-9 arm does not lose the LEAD change-log row on the staged path — it gains it.
- **Supersession safety — `entity_ref` MUST be null (Round 5, Codex F5-supersession).** `cleanup_superseded_before_images` (`20260608000003_undo_change_rpc.sql:323-352`) supersedes an older applied `crew_added/removed/renamed` row (nulling its `before_image`, killing Doug's undo) whenever a NEWER row in the same show has a matching `entity_ref` (`:339`) — and it does **NOT** restrict the newer/triggering row to crew-domain kinds. So a role `field_changed` row carrying `entity_ref = "Alice"` WOULD supersede an older undoable `crew_added`/`crew_renamed` "Alice" row — a data-recovery regression. **Fix: set `entity_ref = null`** (a null is `is distinct from` any real name and from the `crew_renamed` successor-name/email branch `:341-350`, so it triggers no supersession), and carry WHO in the `summary`. This matches the existing MI-8 `field_changed` rows (`writeAutoApplyChanges.ts:155`, also `entity_ref = null`). The role row is also NOT a supersession *target* (`field_changed` ∉ the crew-domain `older.change_kind` set `:331`) and NOT undoable (`UNDOABLE_CHANGE_KINDS` = `["crew_added","crew_removed","crew_renamed"]`, `holds/types.ts:46`). Net: with `entity_ref = null`, the role row is fully inert to the undo/supersession machinery. **Do NOT set `entity_ref = next.name`** — that is the regression.
- **Applied rename + role co-change → TWO rows:** `crew_renamed` (entity_ref = old name, from `writeAutoApplyChanges`, only for APPLIED — non-held — renames) AND the `field_changed` role row (entity_ref = null; the summary names the successor). The role row's null entity_ref also avoids superseding the fresh `crew_renamed` row for the same person (which IS undoable). Genuinely-new crew emit no role row (a `crew_added`, not a *change*; a new LEAD member still covered by notice + event).
- **Multiple changed members → multiple rows** (one per member) — intentional; the feed paginates.
- `roleFlagsEqual` is private to `phase2.ts`; if the shared writer lives in its own module, add a small **local** `roleFlagsSetEqual` helper (avoid a cross-module cycle). Reuses the SAME three inputs the notice producer diffs — `snapshot.previousCrewMembers`, `applyOutcome.appliedCrewMembers` (`phase2.ts:502-506`), and `args.identityLinkRenames` (`:550`) — all already in `runPhase2` scope. Do NOT source renames from `notableItems` (path-dependent) — use `args.identityLinkRenames` so the writer and the LEAD producer are provably parallel.
- **Emit-topology note:** the writer is a change-log WRITE inside the locked tx (like `writeAutoApplyChanges`), NOT a post-commit tail emit — no invariant-10 interaction. It rides `runPhase2`, the single shared point every apply path funnels through.

### 2.5 Staged-apply coverage — parity with the LEAD notice, one bounded pre-existing limitation

Because the role-change row is written from the shared `runPhase2` point using the SAME inputs as `capabilityRoleChangesForNotice` (§2.4), the writer's coverage is **exactly the LEAD notice's existing-member arm (a) minus the capability filter, on every path** (new-crew-with-capability is a `crew_added` + notice/event case, not a role-change row — §2.4 new-crew note) — by construction, not by case enumeration. Concretely on the staged path:

- **LEAD via staged apply:** identifiable change-log row (shared writer) + durable `LEAD_ROLE_APPLIED` app_event (staged tail, unchanged mi9 topology). Doubly covered.
- **Non-LEAD via staged apply (MI-11 email hold, the common case):** the member is retained under its OLD name (identity pinned, `role_flags` folded onto the retained row), so the plain name diff catches it → identifiable change-log row. The per-member delta is a discrete row, not just the batch-level `sync_audit`.

**The single bounded limitation (honest scope — Codex F5-staged-rename).** On the staged path `args.identityLinkRenames` is empty (`applyStagedCore` never threads it), so a staged **identity-linked RENAME** (successor name differs from the prior) is not recognized as a rename: the successor has no matched prior (`!prior`), so no role-CHANGE row is emitted, and `writeAutoApplyChanges` (which would log crew_added/removed) is skipped on staged (`feedPolicy:none`) — so that member's change is captured only in the applied DB state + `sync_audit`. This is **NOT introduced by this spec**: the pre-existing `ROLE_FLAGS_NOTICE`/`LEAD_ROLE_APPLIED` producers use the same empty `args.identityLinkRenames` and already miss this exact case (even a LEAD change on a staged-renamed member is missed today). Because the change-log writer shares the LEAD producer's inputs, it inherits this limitation **identically — no new gap, no divergence.** Closing it (threading `identityLinkRenames` through `applyStagedCore`) improves BOTH producers and is filed as **`BL-STAGED-IDENTITYLINK-RENAMES`**, out of scope here. So the accurate claim is: **every applied role_flags delta on a member WITH A PRIOR ROW (LEAD notice arm (a)) gets a discrete identifiable change-log row — LEAD or non-LEAD, auto-apply or staged, held-fold or clean.** Two documented exclusions: (i) **new-crew-with-capability** (arm (b), no prior) is a `crew_added` + capability notice/event case, NOT a role-change row (§2.4 new-crew note); (ii) a **staged identity-linked rename** (empty `args.identityLinkRenames`) is a pre-existing bounded limitation neither producer recognizes, backlogged (`BL-STAGED-IDENTITYLINK-RENAMES`). No `staged_apply` source, no CHECK migration.
