## 1. Problem (two parts)

**1a. Security/audit gap (the real defect).** When a crew member gains or loses the `LEAD` role flag via a sheet edit (MI-9), the change **auto-applies** (`lib/sync/phase1.ts` "Phase 2 decision rule" — MI-11 is the only gated invariant) but emits **no code-carrying admin_alert / audit signal**. `nonLeadRoleFlagChanges` (`lib/sync/phase2.ts:245`) at `:257` does `if (hasLead(prior) !== hasLead(next)) continue;` — it explicitly **skips** the LEAD toggle, so the info-severity `ROLE_FLAGS_NOTICE` alert (`phase2.ts:518-528`) never fires for a LEAD change. `MI-9_ROLE_FLAGS_DELTA` (`lib/messages/catalog.ts:866`, `spec-codes.ts:581`) exists in the catalog but **no live code path emits it** — it is dead. Net: LEAD (an ops + `shows_internal` financial access grant/loss) auto-applies **silently**. A mistaken LEAD grant on the wrong crew member takes effect on their next page load with no audit entry for Doug to catch.

**1b. Canonical-spec divergence.** The master spec (§6.8 `docs/superpowers/specs/2026-04-30-fxav-crew-pages-v1.md:1624`, §12.4 `:2862`, help `:3155`) and ratified amendment 8 (`plans/…/00-overview.md:158-175`) say MI-9 LEAD **"Stage for approval."** The live code auto-applies it, per the **2026-06-08 sync-changes-feed-identity-gate** amendment (`plans/…/02-decision-rule-and-hold-aware-apply.md:39,42,50`) which is **not in the `00-overview` ratified list**. Per invariant 7 the spec is canonical, so today's code contradicts it. Owner ratifies the auto-apply behavior (option B) — the canon must be updated to match, and the 2026-06-08 amendment recorded as governing.

## 3. Fix — wire the LEAD FYI (design (a): reuse `ROLE_FLAGS_NOTICE`)

`ROLE_FLAGS_NOTICE` is already `severity: "info"`, `audience: "health"`, `healthWeight: "notice"`, `resolution: "manual"` (`catalog.ts:879`) and fully wired through every consumer (alert union, identity map, actions/openSheet, health, banner-exclude, audience contract). Reuse it for LEAD — **zero new catalog code**, minimal fanout.

**3.1 Producer — diff against the APPLIED crew list, not the raw parse (Codex R1).** In `nonLeadRoleFlagChanges` (`phase2.ts:245-262`): (a) **remove the LEAD-skip at `:257`** so a LEAD-bit toggle also produces a `roleFlagChanges` entry; (b) **change the `next` argument at the call site (`phase2.ts:514-516`) from raw `parseResult.crewMembers` to `applyOutcome.appliedCrewMembers`** — the ACTUALLY-committed crew list (already available at this site; `writeAutoApplyChanges` uses it at `:478` for the same reason, P2-F2). This is load-bearing: a held **MI-11 rename/fold** suppresses the renamed sheet row and applies its non-identity fields (incl. `role_flags`) onto the retained old-name row; diffing the raw parse by name would see no `priorMember` for the renamed row and emit **nothing**, leaving a LEAD change via a held fold silent (the exact gap this spec closes). Diffing `previousCrewMembers` → `appliedCrewMembers` by name catches the `role_flags` change on the retained row regardless of fold. Then the existing `roleFlagsNotice` emission (`phase2.ts:518-528`) fires with `context.changes = [{ crew_name, prior_flags, new_flags }]`. **Rename** the misnamed helper `nonLeadRoleFlagChanges` → `roleFlagChangesForNotice`; sweep the def + its single caller.

**New-crew-with-LEAD is NOT silent (Codex R2 F1).** A crew member added *with* `LEAD` grants ops/financial access, but the `crew_added` change-log image is only `{name,email}` (no role_flags), and the name-keyed diff skips no-prior members — so a new LEAD crew member would be silent. The producer therefore also emits a role-flags notice entry for a **newly-applied crew member whose `role_flags` include `LEAD`** (`prior_flags: []`, `new_flags: [...LEAD...]`) — treating a new LEAD grant as notice-worthy. (Non-LEAD new crew stay covered by `crew_added` only; only the LEAD case needs the extra signal.)

**Guard conditions:** `prior_flags`/`new_flags` are always arrays; the `roleFlagsEqual` no-op skip (`:258`) stays — an identical applied role set emits nothing. Held-identity crew (MI-11 email hold) whose role_flags still applied to the retained row ARE captured (the point of diffing the applied list).

### 3.4 Durable, non-coalescing LEAD audit record (Codex R2 F2)

The `ROLE_FLAGS_NOTICE` admin_alert is the operator **feed nudge**, but `upsert_admin_alert` coalesces unresolved alerts by `(show_id, code)` and **replaces** `context` (non-`failedKeys` producers) — so a later role change on the same show can overwrite the unresolved LEAD notice's `context.changes`, hiding the LEAD grant while only bumping `occurrence_count`. And today no per-change durable record of a LEAD delta exists (MI-9 → a `field_changed` change-log row with **null** image; `crew_added` image omits role_flags). For an **access grant/loss**, a lossy nudge is not an adequate audit.

**Therefore, in addition to the feed nudge, emit a durable code-carrying `app_event` per LEAD-bit change** via the **failure-visible** writer `persistAppEventStrict` (`lib/log/persist.ts:60` — returns `{ ok:true } | { ok:false, error }`, does NOT swallow like `log.info`/`persistAppEvent`), post-commit, outside the advisory-lock tx (invariant 10). `code: "LEAD_ROLE_APPLIED"` (forensic code; NOT a §12.4 user-facing code — must pass the forensic-code namespace scanner and not collide with reserved prefixes), context = `{ crew_name, prior_flags, new_flags, direction: "gained"|"lost" }`, `showId`/`driveFileId` set. Redaction-safe (crew name + flag tokens — no email/phone/token; `persistAppEventStrict` also runs `sanitizeContext`). Queryable via `pnpm observe events --code LEAD_ROLE_APPLIED`.

**Failure policy (Codex R4 — honest durability).** A post-commit event cannot be *transactionally* atomic with the already-committed LEAD change, so the guarantee is **durable + failure-visible**, not "impossible to lose": destructure the `{ ok, error }` return (invariant 9); on `{ ok:false }`, **surface the failure loudly** — a `persistHealth`/health-alert escalation or a `log.error` with a distinct code — so an audit-write failure is itself observable and alertable, never silently swallowed. Residual risk (both the strict insert AND the failure-escalation fail) is a documented double-fault, not a silent gap. This is the authoritative LEAD audit trail (subject to that honest caveat); the coalescing `ROLE_FLAGS_NOTICE` alert remains a best-effort feed heads-up. **New caller of `persistAppEventStrict`** → complies with invariant 9 (`{ok,error}` destructure) and the sole-writer meta-test (`tests/log/_metaAppEventsWriter.test.ts` — the emitter calls the writer, never inserts `app_events` directly); register per `tests/sync/_metaInfraContract.test.ts` if required.

**Cross-caller emission topology (MANDATORY — Codex R3).** `runPhase2` already returns `roleFlagsNotice` and **every** post-commit caller emits it later — the durable event MUST ride the SAME set of sites so no apply path is missed. The LEAD event derives from the SAME `roleFlagChanges` that feed `roleFlagsNotice` (the LEAD-bit subset: entries where `hasLead(prior) !== hasLead(new)`, plus a new-crew-with-LEAD entry). Contract: at **every** site that emits/upserts `roleFlagsNotice`, ALSO emit `LEAD_ROLE_APPLIED` for that site's LEAD-subset entries. Enumerate and cover ALL of them (the same wrappers that today propagate `roleFlagsNotice`): the cron/manual `processOneFile` tail (`runScheduledCronSync.ts`), the staged-apply tail (`applyStaged` / `applyStagedCore`), and any other `runPhase2` caller that surfaces `roleFlagsNotice`. Do NOT wire it in only the cron path. Grep `roleFlagsNotice` across `lib/sync/**` to produce the exhaustive caller list at plan time; a per-surface test (below) fails-by-default if a caller is missed.

**3.2 Copy (must broaden — currently LEAD-false).** `ROLE_FLAGS_NOTICE.dougFacing` (`catalog.ts:887`) currently says *"The LEAD bit is unchanged, so the change was applied automatically."* — this becomes **false**. Rewrite to cover both cases and carry the confirm-intent nuance for LEAD, e.g.:
> `dougFacing`: "A crew member's role flags changed and were applied automatically — this entry is here for your audit. If the change included LEAD status (which grants admin/ops/financials access), confirm it was intentional."

Update `helpfulContext` (`catalog.ts:890`) and `dougSummary` (`catalog.ts:884`) to drop the "doesn't affect LEAD" claim and note LEAD toggles are included (auto-applied because a sheet edit is deliberate; still logged for audit). **These are §12.4-catalog rows → the lockstep in §5 applies.**

**3.3 Retire the dead code — per the `RETIRED_CODES` contract (Codex R1).** `MI-9_ROLE_FLAGS_DELTA` has no producer under B. The existing retired-code contract (`tests/cross-cutting/codes.test.ts:152` — "retired §12.4 codes have no producer, runtime catalog entry, or scenario") requires a retired code to be **ABSENT from the runtime catalog**. So retirement means: (a) **remove the `MI-9_ROLE_FLAGS_DELTA` row from `lib/messages/catalog.ts:866`** and any scenario/registry/producer reference; (b) mark it retired in the **master spec §12.4** prose (strikethrough / retired-row with `dougFacing —`); (c) `pnpm gen:spec-codes` moves it into the generated `RETIRED_CODES` map (with `retiredIn: "§12.4"` metadata, mirroring `OAUTH_STATE_INVALID`/`OAUTH_REDIRECT_INVALID`).
