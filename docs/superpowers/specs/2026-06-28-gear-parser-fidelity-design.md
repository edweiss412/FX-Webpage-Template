## 1. Problem statement (verified diagnosis)

The crew "Gear" section (`components/crew/sections/GearSection.tsx`, `data-testid="section-gear"`) renders four data domains: per-room A/V/L scope, pack list, keynote requirements, opening reel. A user report ("gear is not reliably parsed and/or making it to the relevant surfaces") was investigated end-to-end and **confirmed**. Three distinct parsing bugs drop gear before it reaches the surface:

### Bug 1 (dominant) — the GEAR tab is exported but never parsed

Modern shows (the v4 "GEAR-tab" schema family — RPAS Central 2026, Fixed Income Trading Summit 2025, all 2025-10/2026 shows) put **all** per-room A/V/L equipment in a dedicated `GEAR` tab: an `Item | …rental-date columns…` date-grid. The INFO room blocks for these shows carry only Setup/Set/Show/Strike times, **no** A/V/L rows.

- The GEAR grid **reaches the parser's input.** Proof: `fixtures/shows/exporter-xlsx/rpas.md:147-148` (the production `synthesizeMarkdownFromXlsx` path) emits `| | | Rental Dates | … |` then `| Item | Item | 21-Mar | … | 27-Mar |`, followed by room sub-headers (`| GENERAL SESSION - GRAND BALLROOM A/B | … |`) and equipment rows (`DIGITAL AUDIO CONSOLE - QU32`, `(4) BLIZZARD LED BARS`, etc.). Same in `fixed-income.md:122-145`. The exporter (`lib/drive/exportSheetToMarkdown.ts:194-209`) emits every tab except `OLD`-named ones, but **discards tab names** (`:209` joins table markdown with no tab-name header).
- **No parser consumes it.** `parsePullSheet` accepts a table only when **all** header cells contain `"PULL SHEET"` (`lib/parser/pull-sheet.ts:60`); the GEAR header matches neither that nor the deliberate `PULLED`+`INITAL` rejection (`:52`), so the grid is silently skipped → `pullSheet = null`. `parseRooms` only reads `GENERAL SESSION` / `BREAKOUT N` / `ADDITIONAL ROOM` INFO blocks (`lib/parser/blocks/rooms.ts`), so room A/V/L scope is empty.

**Effect:** for the entire modern show family the Gear section is blank or near-blank. End-to-end render proof: the committed admin-preview screenshot `public/help/screenshots/crew-preview-gear-mobile-light.webp` (RPAS Central 2026, captured through the real `ln` projection + real `GearSection`) renders **only** "Opening reel: MAYBE" — no scope, no pack list, no keynote — despite the live GEAR tab listing a Barco projector, QU32 console, AB168, 17 mics, LED Lekos, and Blizzard uplights. (Full-corpus `parseSheet()` run over all 17 committed fixtures [10 raw + 7 exporter-xlsx]: 9 fixtures report `rooms > 0` but `scope 0/0/0`; 14/17 report `pullSheet = null`.)

### Bug 2 — EVENT DETAILS value block dropped on older "form-layout" sheets

Older 2025 shows (`2025-10-consultants-roundtable.md` [LIVE], `2025-03-dci-rpas-central.md` [fixture-only], `2025-04-asset-mgmt-cfo-coo.md` [fixture-only]) use a vertical "CLIENT INTAKE FORM" layout where EVENT-detail fields are a headerless `| field-label | value |` run (`2025-10-consultants-roundtable.md:261` = `| Keynote Requirements | TBD |`). `parseEventDetails` yields `event_details = {}` for all three — confirmed via `parseSheet()`: 0 event keys, while controls East Coast (13 keys, `keynote="NONE"`, `reel="YES - LOOP VIDEO"`) and RPAS (17 keys, `keynote="TBD"`, `reel="MAYBE"`) parse fully.

Two compounding causes (both verified in code):
1. **Header matches the wrong block, then breaks before the values.** `EVENT_DETAILS_HEADER_RE` (`lib/parser/blocks/event.ts:39`, matched at `:132`) matches the *label-only* `| DETAILS |` column (`2025-10-consultants-roundtable.md:17`). Its rows are all single-column, so every one is skipped (`event.ts:206`). The block-scan then **breaks at the first blank/non-pipe line** after that small table (`event.ts:142` — `if (inBlock) break`), so the real value block (`:235`+) is never reached. Net: zero keys harvested → `{}`. (Confirmed: `parseSheet` reports 0 event keys, so no value-block row is processed at all.)
2. **Terminator collision (secondary hazard for the fix).** The room-header terminators `/^GENERAL SESSION\b/` and `/^BREAKOUT \d/` (`lib/parser/blocks/event.ts:165`, alongside `TERMINATING_LABELS` at `:43`) match the *title-case form fields* `General Session Room Name` / `Breakout Room Name(s)` in the value block — so naive harvesting of that block would still cut off before `Keynote Requirements`.

**Effect:** keynote, opening reel, and ~13-15 other event fields drop for these shows (incl. live Consultants Roundtable 2025).

### Bug 3 — secondary, narrower drops

- **3a — East Coast lighting orphan row.** `2024-05-east-coast-family-office.md:57` holds `(2) Lekos for Stage Wash (6) Blizzard LED Uplights` in an **unlabeled continuation row** (col0 empty), between `GS Scenic` (`:56`) and `GS Other` (`:58`). `rooms.ts` keys A/V/L by the col0 label, so this lighting value is dropped (`parseSheet` reports East Coast `audio=3 video=3 lighting=0` in both fixture families).
- **3b — wide PULL SHEET degradation.** East Coast's real PULL SHEET rows are wider than 5 columns (raw `~8`; the **production exporter** `exporter-xlsx/east-coast.md` is **16 cells** — `FALSE | qty | item | '' | CAT | '' | 0 | seq | <8 trailing blanks>`, `:142-145`), but `parseDataRows` requires exactly 5 (`lib/parser/pull-sheet.ts:176` → `cells.length !== 5` → `PULL_SHEET_AMBIGUOUS_FORMAT` raw-snippet fallback, qty/cat lost). The exporter case also carries all-blank summary rows with a trailing total (`| … | 1728 |`, `:139/:141`).
- **3c — dash sentinel.** `-` / `—` placeholders (RIA breakout `BO Audio='-'`, `BO Lighting='-'`) are not in `GENERIC_OPTIONAL_HIDE` (`lib/visibility/emptyState.ts:52`) → render as visible empty values.

## 3. Architecture

### 3.1 New module: `lib/parser/blocks/gear.ts`

Exports `parseGearTab(markdown: string, agg?: ParseAggregator): GearRoom[]` where `GearRoom = { kind: RoomRow["kind"]; name: string; audio: string | null; video: string | null; lighting: string | null; scenic: string | null; other: string | null }`.

**Scope of supported shape (IMPORTANT — verified against committed fixtures).** The GEAR parser targets the **exporter-xlsx (prod `synthesizeMarkdownFromXlsx`) grid shape** — the production ingestion path. The **raw-family GEAR grid is explicitly out of scope**: the legacy MCP converter mangles it — item *names* are replaced by `NO_HEADER` headers and only bare quantity columns survive (`fixtures/shows/raw/2026-03-rpas-central-four-seasons.md:201-217`), so the data is unrecoverable. The parser must **gracefully no-op** on the raw shape (no corrupted scope), and GEAR-parser correctness is asserted against the exporter-xlsx fixtures (`consultants.md`, `rpas.md`, `fixed-income.md`). Tested explicitly (anti-corruption assertion on the raw shape).

**Detection — a SINGLE shared signature predicate.** Define one exported helper `hasGearDateGrid(rows)` (in `lib/parser/blocks/gear.ts`) that is the **sole** definition of "this is a GEAR date-grid," reused verbatim by both `parseGearTab` AND the `gear_scope` source-anchor gate (§3.6) — so "a date-grid was parsed" ⇔ "the gear_scope anchor exists" can never drift (Codex R5-M2). The signature: a row whose cells are all (or majority) `Rental Dates`, followed — allowing intervening markdown alignment `:---:` rows and blanks (NOT immediate adjacency; consultants interposes a `:---:` row, `exporter-xlsx/consultants.md:139-141`) — by an `| Item | Item | <date> … |` header row (≥1 date token like `4-Oct`/`21-Mar`). **Both the Rental Dates row AND the Item/date header are required** (a `Rental Dates` row alone is NOT sufficient). `parseGearTab` returns `[]` when `hasGearDateGrid` is false. There is no tab-name signal (exporter discards names), so content shape is the only key.

**Per-room segmentation (clean cell-width signal — verified).** The exporter renders each room as a **2-cell mini-table header** (`| ROOM NAME | ROOM NAME |` followed by a 2-cell `| :---: | :---: |`), whereas package headers and equipment rows are **full grid width (8-9 cells)** with date-quantity columns (`exporter-xlsx/consultants.md`: room header `:143` is 2 cells; `STRETCHED SPANDEX…`/`SOUND SYSTEM PACKAGE`/equipment are 8 cells). So between the `Item` header and the `BACK TO INFO` sentinel, a **room sub-header** is detected structurally as a 2-cell, non-`:---:`, non-empty-col0 row — NOT a hard-coded `GENERAL SESSION`/`BREAKOUT`/`ADDITIONAL` vocabulary (real fixtures use bare `FOYER`, `LUNCH SESSION …`, and **unnumbered** `BREAKOUT SESSION 1 - LASALLE`). Full-width rows are either package headers (closed set, §3.2) or equipment items. Markdown alignment rows (`| :---: | … |`, any width) are skipped everywhere. The duplicated item column is handled: take col0 as the item; if col1 equals col0 (exporter doubling) skip it; treat remaining non-empty numeric cells as per-date quantities. The exact rules are **pinned by tests against the exporter-xlsx fixtures**, which exhibit the alignment-row, bare/unnumbered room-header, package-header, and doubled-column variants.

**Room matching (match on the room-NAME token, NOT the breakout index — verified hazard).** The breakout **numbers are scrambled** between the INFO and GEAR tabs: consultants GEAR has `BREAKOUT SESSION 1 - LASALLE` / `2 - DELAWARE` while INFO has `BREAKOUT 1 DELAWARE` / `2 LASALLE` (`exporter-xlsx/consultants.md`). Matching by `BREAKOUT N` index would attach LASALLE's gear to DELAWARE's room — silent corruption. Therefore matching strips the room-type prefix (`BREAKOUT[ SESSION] \d+`, `GENERAL SESSION`, `LUNCH (ROOM|SESSION)`, `ADDITIONAL ROOM`) and any trailing `Dimensions`/`Floor` suffix (mirroring `rooms.ts:342`'s strip), then matches on the **normalized remaining room-name token** (e.g. `LASALLE`, `GRAND BALLROOM A/B`). **Exact normalized name-token match only** — on any non-exact match the GEAR room is appended as a NEW room (kind inferred from the header word: `breakout`/`general`/`additional`; default `additional`); no warning code (D7 — gear preserved as a new card). Never fuzzy-guess a match (a wrong match corrupts; an appended room is recoverable). Consequence: `FOYER` (no INFO peer) and `LUNCH SESSION - GRAND BALLROOM C` (INFO `LUNCH ROOM BALLROOM C` — name token differs: `GRAND BALLROOM C` ≠ `BALLROOM C`) append as new rooms; the four named breakouts and the GS room match by token. This is pinned by the consultants test.

### 3.2 Classification registry: `lib/parser/gearClassification.ts`

**Classification precedence — per-item allow-list FIRST, package-bucket as FALLBACK (verified hazard; Codex R1+R2 + independent fixture trace).** Each equipment item is classified in this strict order:

1. **Per-item closed allow-list (highest priority).** Match the item text against the tight closed keyword allow-lists below. If matched → that discipline, **regardless of the active bucket.**
   - `audio`: SPEAKER, CONSOLE, MIXER, SOUND SYSTEM, MICROPHONE, MIC, SNAKE, ANTENNA, QU32, QU24, QU16, AB168, KLA, K8, K10, GOOSNECK/GOOSENECK, AUDIO
   - `video`: PROJECTOR, SCREEN, MONITOR, SWITCHER, LAPTOP, CAMERA, EIKI, BARCO, POINTER, MATRIX, COUNTDOWN CLOCK, CONFIDENCE MONITOR, DLP
   - `lighting`: LEKO, UPLIGHT, LED BAR(S), DMX, LIGHTRONICS, LIGHTING, BLIZZARD, ROCKVILLE
   - `scenic`: SPANDEX, LOGO, BRANDING, BACKDROP, SCENIC, TRUSS PODIUM, PODIUM
2. **Active bucket (fallback for allow-list misses only).** A **bucket-setter** row sets the active bucket: text matching `/SOUND SYSTEM/i` → `audio`; `/STAGE LIGHTING/i` or `/UPLIGHTING/i` → `lighting`. This intentionally matches both the pure grouping header (`SOUND SYSTEM PACKAGE`) AND a real-gear variant (`SMALL SOUND SYSTEM`, consultants lunch room `:228`). An item with NO allow-list match inherits the active bucket (so `CABLING`, `SPEAKER STANDS`, `MOUNTING HARDWARE` following `SOUND SYSTEM`/`SMALL SOUND SYSTEM` → audio). The bucket resets to none at each room boundary.
3. **`other` (catch-all).** No allow-list match AND no active bucket → `other` (truss, ZOOM/PTZ backup, misc) — nothing dropped (D2).

**Emission vs grouping:** a bucket-setter row is ALSO emitted as an item via rule 1/2 **unless its text ends in `PACKAGE`** (a pure grouping label — `SOUND SYSTEM PACKAGE`, `STAGE LIGHTING PACKAGE`, `LED UPLIGHTING PACKAGE` — set the bucket but are NOT emitted). Real-gear bucket-setters like `SMALL SOUND SYSTEM` (no `PACKAGE` suffix) set the audio bucket AND emit as an audio item.

**Why allow-list-first (not bucket-first):** in `exporter-xlsx/consultants.md:166-169` the rows `COUNTDOWN CLOCK`, `DLP DATA PROJECTOR-BARCO W8`, projector lens, and `6'X10' … PROJECTION SCREEN` appear *after* `SOUND SYSTEM PACKAGE` (`:155`) and *before* `STAGE LIGHTING PACKAGE` (`:170`). A bucket-first rule would file these video items under audio. Allow-list-first puts them in `video` correctly.

Within a room+discipline the classified item summaries join into a single `(N) ITEM` string (D4), order-preserving.

**Collision tripwire** (`tests/parser/gearClassificationRegistry.test.ts`): asserts no keyword appears in more than one discipline allow-list and no allow-list keyword equals a package-header token. Same structural-defense shape as the typo-tolerance registry guard.

### 3.3 Integration point

In `parseSheet` (`lib/parser/index.ts`), after `const rooms = parseRooms(markdown, version, agg)` (`:389`):

```
const gearRooms = parseGearTab(markdown, agg);
const rooms = mergeGearIntoRooms(parsedRooms, gearRooms);  // fill-don't-clobber
```

`mergeGearIntoRooms`: for each GEAR room matched by `(kind,name)`, for each of `audio/video/lighting/scenic/other`, set the room column **only when the existing INFO-derived value is null** (fill-the-gap; never overwrite a non-null INFO value). Unmatched GEAR rooms append. In practice a show is INFO-inline *or* GEAR-tab (never both populated for the same discipline), so this is fill, not conflict resolution — but the rule is explicit and tested.

### 3.4 Bug 2 fix (`lib/parser/blocks/event.ts`)

Add a **form-layout harvest** to `parseEventDetails` that **runs unconditionally after the classic-header pass** (NOT gated on zero classic keys — Codex R4 verified that gate breaks mixed sheets like fixed-income, which has a classic `EVENT DETAILS` block AND a separate later `Opening Sizzle Reel | No` form block; gating off the form harvest would leave the stale classic `Opening Reel | TBD` in place). Locate the form block by **anchoring** on known event-field labels — a contiguous run of 2-cell `| label | value |` rows in which ≥3 col0 labels normalize to a `CANONICAL_KEY_MAP` key (`event.ts:60`/`EVENT_LABEL_VOCAB:107`) — then **harvest the ENTIRE contiguous run**, not only the known labels. Known labels → their canonical key; **unknown labels → `toCanonicalKey(col0)`** (`event.ts:244`, the exact normalize-and-keep fallback the classic path already uses at `:201`). This recovers the full intake form, not just the vocab hits — Codex R3 verified the consultants form carries `Backdrop`, `Power Requirements`, `Internet Requirements`, `Recording`, `Podium for General Session`, `Stage Size for General Session`, `Onsite Equipment Storage`, `Additional Notes`, most of which are NOT in `CANONICAL_KEY_MAP`.

**Sensitive-label exclusion (permission boundary — Codex R6, P0).** `event_details` lives on the crew-visible `shows` row and `getShowForViewer`/`ln` projects it to **every** viewer, whereas financials are physically separated into the LEAD/admin-gated `shows_internal` table (`getShowForViewer.ts` financials gate, `:29-37`). The whole-block harvest must NOT move financial/internal fields into `event_details`: the consultants form run interleaves `Budget` (`raw/2025-10-consultants-roundtable.md:274`) and `PO#` (`:275`) right before `Additional Notes` (`:276`). Decide sensitivity against the **CANONICAL KEY** (the post-`CANONICAL_KEY_MAP`/`toCanonicalKey` snake_case key that would be written), NOT the raw display label — Codex R7 verified that a label-only denylist misses `Invoice Notes`, which `toCanonicalKey` normalizes to `invoice_notes` (a master-spec `shows_internal` financial field — master spec `:23`: "PO#, Proposal $, Invoice, Invoice Notes ... filtered out of non-LEAD views"). Define a closed token set **`SENSITIVE_KEY_TOKENS`** = `budget`, `po`, `purchase`, `proposal`, `invoice`, `cost`, `price`, `quote`, `estimate`, `internal`. A harvested field is **SKIPPED** (never written to `event_details`) iff any underscore-delimited token of its canonical key ∈ `SENSITIVE_KEY_TOKENS`. This catches `invoice_notes` (token `invoice`), `po`/`po_number`/`purchase_order` (`po`/`purchase`), `proposal`, `budget`, etc. by construction — covering all four master-spec financial fields plus the consultants `Budget`. Skip-not-stop: `Additional Notes` (key `additional_notes`, no sensitive token) after `Budget`/`PO#` is still recovered. Apply at **both** harvest sites — the new form harvest AND the existing classic `toCanonicalKey` unknown-label fallback (`event.ts:201`) — to close the class (no current fixture's `event_details` carries a sensitive key, so the classic-path addition is regression-free; verified). The separate INTERNAL checklist block (`:278`+, `Create Proposal`/`COI Sent`/…) is excluded by both the contiguous-run bound (distinct table after a blank line) and the `internal` token.

**Merge semantics (the safety guarantee — Codex R8-M2).** The classic pass is unchanged (its existing `writeField`/`exactReal` semantics). The **form harvest uses a stricter write discipline — `fillIfAbsentOrSentinel`**: before writing a key it inspects the value already in `event_details` and writes ONLY when that value is absent OR a sentinel (`shouldHideGenericOptional`); it **never overwrites an existing REAL value**. (Plain `writeField` is last-real-wins, which would silently clobber a classic real value with a conflicting form real value — the data-corruption path R8 flagged.) Because the classic pass runs first, this yields a deterministic **first-real-wins** policy. Consequences, now provably true: (a) classic-only shows (East Coast/RPAS) — the form re-harvest finds every classic key already real → writes nothing → identical key set + values (negative-regression pinned); (b) sentinel→real upgrade still works: fixed-income's classic `Opening Reel = TBD` (sentinel) is upgraded by the form `Opening Sizzle Reel = No` (real); (c) **real→real conflict: the classic real value is preserved** (form skips it) — pinned by a synthetic test (classic `Opening Reel = YES` + form `Opening Sizzle Reel = No` → `opening_reel` stays `YES`). The form run is bounded by its own contiguity (ends at a blank/non-2-cell row), so it does not inherit the classic path's premature blank-line break, and the room-header terminators (`event.ts:165`) are tightened to fire only on an **ALL-CAPS** room-header shape (`GENERAL SESSION GRAND BALLROOM A/B`), never the title-case form field `General Session Room Name`.

**`Opening Sizzle Reel` alias (required for the target fixture).** The live form-layout opening-reel value is keyed on **`Opening Sizzle Reel`**, not `Opening Reel` (`fixtures/shows/raw/2025-10-consultants-roundtable.md:264` = the actual reel status; also present in `ria`, `fixed-income`, `redefining` raw fixtures). `CANONICAL_KEY_MAP` (`event.ts:64`) currently has only `"opening reel" → opening_reel`. **Add `"opening sizzle reel" → opening_reel`** so both the classic path and the form fallback recover it. This is a single CANONICAL_KEY_MAP addition (serves both paths) and flows into the derived `EVENT_LABEL_VOCAB` (`event.ts:107`) automatically. **Precedence when a sheet has BOTH labels** (e.g. `fixed-income` exporter: classic `Opening Reel | TBD` at `:83` AND form-block `Opening Sizzle Reel | No` at `:233`): because the form harvest now runs unconditionally and both labels map to `opening_reel`, the sentinel-aware merge resolves `opening_reel = "No"` (real displaces the classic `TBD`). Pinned by a fixed-income exporter test (`opening_reel === "No"`, and the later form-only fields are present).

### 3.6 Render (`components/crew/sections/GearSection.tsx`)

Extend the `DISCIPLINES` array (`:80-104`) with `scenic` (heading "Scenic", lucide `Frame` icon) and `other` (heading "Other gear", lucide `Boxes` icon) after `lighting`. Each new card reuses the exact existing machinery: `shouldHideGenericOptional` per-value filter, omit-when-zero-rows, `SourceLink`, `KeyValueRows`. **Emphasis stays A/V/L-only** — `viewerDisciplines` (`:113-119`) is unchanged, so Scenic/Other are always neutral (no `data-emphasis`).

**Scope-card source links — target the GEAR tab ONLY for GEAR-date-grid-derived data (Codex R3-M3 + R4-M2).** For the modern GEAR-tab family the A/V/L scope is parsed from the GEAR date-grid, but the existing `gear-scope-*` cards link to the `rooms` region → the **INFO** tab (`buildSheetDeepLink.ts:134-136`), which does not contain that equipment. The naive fix (link to GEAR whenever a GEAR *tab* exists) is WRONG: Codex R4 verified `exporter-xlsx/ria.md` has INFO-inline scope rows (`GS Audio`/`GS Video`/`GS Scenic`/`GS Lighting`/`GS Other`, `:95-99`) and **no** GEAR date-grid — its scope is INFO-derived, so it must keep the INFO link. Provenance, not tab presence, is the signal. Fix:
- Add a new region `gear_scope` to `REGION_ANCHOR_SPEC` + `REGION_IDS` (`buildSheetDeepLink.ts`): `{ tabs: ["GEAR"], strategy: "whole-tab" }`. (`GEAR` is already in `SOURCE_LINK_ALLOWLIST`, `:1`.)
- **Gate its emission on the GEAR date-grid signature, not tab presence — using the SAME `hasGearDateGrid` predicate as the parser (§3.1).** In the anchor builder (`lib/drive/sourceAnchors.ts`), `gear_scope` resolves to the GEAR tab **only if `hasGearDateGrid` returns true for that tab's grid** — i.e. it has the full `Rental Dates` + `Item | Item | <date>` signature, not merely a stray `Rental Dates` cell. Sharing the predicate makes "gear_scope anchor exists" ⇔ "`parseGearTab` parsed a grid" hold by construction (Codex R5-M2 — a `Rental Dates` row without a valid Item/date header must NOT emit the anchor). ## 5. Guard conditions (every new input)

- `parseGearTab(markdown)`: empty/whitespace markdown → `[]`. No GEAR signature → `[]`. Grid present but zero room sub-headers → `[]` (no warning, D7). Room block with zero classifiable items → that room contributes no columns. A date-qty cell that is non-numeric/blank → ignored for qty (item still listed name-only). Item name empty after trim → row skipped.
- `mergeGearIntoRooms(rooms, gearRooms)`: `gearRooms` empty → returns `rooms` unchanged. GEAR room matches multiple INFO rooms by name token → match the first (deterministic, no warning). Existing column non-null → not overwritten. **Appended (unmatched) GEAR rooms carry ONLY gear columns** (`audio/video/lighting/scenic/other`); `setup`/`set_time`/`show_time`/`strike_time`/`dimensions`/`floor` are null, so they stay inert in non-gear surfaces — `deriveScheduleBookends` keys Strike on `strike_time` (`lib/parser/index.ts:463`), which is null → no spurious schedule entry. Pinned by a defensive test (appending `FOYER` adds no run-of-show bookend).
- Classification: item matching no allow-list and under no package header → `other` (never dropped, never throws). `(N)` parse: malformed leading paren → treat as no-embedded-qty, fall back to date-column max; no qty anywhere → name-only.
- `parseEventDetails` form harvest: **always runs after the classic pass** (NOT gated on classic keys — §3.4). Merges every field through sentinel-aware `writeField`, so a classic-only show re-harvests its own DETAILS block idempotently (no double-write effect; real values never clobbered) and a mixed show upgrades a classic sentinel with the form's real value. No anchored form run found (fewer than 3 known-vocab labels in any contiguous 2-cell run) → no-op, classic result stands. A label that is both a known event field and a room header (none currently) → known-field set wins.
- Render: a room whose `scenic`/`other` is `null` or a sentinel → that card omits that room's row; a discipline with zero rows across all rooms → card omitted entirely (existing `:144` behavior, extended). `data.rooms` empty → all scope cards omit (existing).
