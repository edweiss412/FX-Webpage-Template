# Spec — Flow 6.2 monitor digest ("Applied automatically since your last digest")

**Date:** 2026-07-08
**Slug:** flow6.2-monitor-digest
**Audit item:** `docs/audits/e2e-real-world-variation-preparedness-2026-07-07.md` §6 Flow 6, item 6.2 (Medium). Cross-flow note (audit §6 "Cross-flow note"): 6.2 is the **monitoring layer** of the P0-2 confident-wrong-values class.

---

## 1. Intent

A Doug who **never opens the admin dashboard** currently learns nothing about what the pull-only (cron auto-apply) band silently did to his live shows. Flow 4 (shipped, `feat/flow4-auto-applied-strip`) built the *in-app* auto-applied strip (`lib/admin/loadRecentAutoApplied.ts`) — that serves *active* Doug. 6.2 is the **email counterpart** for *passive* Doug: extend the existing daily review digest with a section reporting what was auto-applied / auto-corrected / drifted since the last digest.

**Done-when (audit §6):** a Doug who never opens the app still learns about failures and drift within a day.

This spec adds **one new email section** to the **existing** daily digest. It does NOT build a new cron job, a new toggle, a new delivery path, or a new dashboard surface.

## 2. Scope

### In scope (v1)

**Prerequisite (in scope):** fix the latent cron `sync_log` persistence bug (§3.2) so signals 2 & 3 have a data source. User-ratified scope expansion (2026-07-08): the original "data already persisted" assumption was wrong for the cron path.

Three signals, appended as one new section ("Applied automatically since your last digest") below the existing needs-attention section of the daily digest:

1. **Auto-applied roster/field changes** — reuse Flow-4's data (`show_change_log` `source='auto_apply'`, `status='applied'`, un-acknowledged), windowed to "since last digest."
2. **Autocorrects applied** — the five `*_AUTOCORRECTED` classes counted by `summarizeAutoFixes` (`lib/parser/dataGaps.ts:119`), over `sync_log` rows in the window.
3. **Sub-threshold drift** — per-show net data-gap movement (`summarizeDataGaps`, `lib/parser/dataGaps.ts`) that did NOT cross the push-alert regression threshold (`isQualityRegression`, `dataGaps.ts:214`), over the window.

### Out of scope (deferred, with reason)

- **New data gaps as a first-class digest line** — deferred until the `ambiguity-warnings-v1` worktree lands (it reshapes `lib/parser/dataGaps.ts` `GAP_CLASSES` / adds ambiguity codes). v1 reads `summarizeDataGaps` only through the **sub-threshold drift** computation (which is delta-based and robust to class-set additions); it does NOT enumerate individual new-gap classes as their own section. Rationale: decoupling from `ambiguity-warnings-v1` so 6.2 has no merge-order dependency on `dataGaps.ts` structural changes. (audit 6.2 note: "new data gaps ... can be a followup when ambiguity-warnings ships" — user decision, 2026-07-08.)
- **Setting the Resend key / recipient** — that is audit item 6.1 (config-only, `lib/notify/config.ts`), separate. Until 6.1 lands, the digest (and this new section) is built and dedup-recorded but not delivered (recipient-safe). 6.2 does NOT unblock delivery.
- **New `admin_alert` code / §12.4 catalog row** — none. The section renders as bespoke email copy (mirroring how `SHOW_FIRST_PUBLISHED`'s `data_gaps` sub-line is bespoke, `runScheduledCronSync.ts:2282`), not via the message catalog.
- **A per-recipient monitor window** — the watermark is global (single-band assumption, §4.3). Multi-recipient correctness is handled by computing monitor content ONCE per run and advancing the watermark ONCE (§5).

### 3.2 Prerequisite: fix the cron `sync_log` persistence gap (latent bug)

The cron sync path injects `writeSyncLog` as the `logSync` sink (`app/api/cron/sync/route.ts:21`). `logSync` (`runScheduledCronSync.ts:2143-2162`) already **threads** the applied outcome's parse warnings into `entry.parseWarnings` (`:2161`, D-7/FIX-3 intent) and the applied result carries `showId` (`ProcessOneFileResult` `outcome:"applied"`, `:384-385`). **But the production sink drops both:** `makePostgresSyncLogSink` (`lib/sync/syncLog.ts:30-42`) inserts only `(drive_file_id, status, message, parse_warnings)` — no `show_id` column — and its `warningsFor` (`:25-28`) serializes ONLY the payload row, ignoring `entry.parseWarnings`. So on the live cron path, `sync_log` rows have `show_id = NULL` and `parse_warnings` with no `ParseWarning` elements. (The recovery-path sink `insertSyncLog`, `:1163-1181`, DOES persist both — so the two sinks are inconsistent; this is a pre-existing bug, not new scope.)

**Fix (single-source both sinks):** bring `makePostgresSyncLogSink` in line with `insertSyncLog`:
- Add `show_id` to the insert: `insert into public.sync_log (show_id, drive_file_id, status, message, parse_warnings) values ($1::uuid, $2, $3, $4, $5::jsonb)`; bind `entry.showId ?? null`.
- `warningsFor(entry)` returns `[ ...(entry.payload ? [{ ...entry.payload, outcome: entry.outcome, code: entry.code ?? null }] : []), ...(entry.parseWarnings ?? []) ]` (payload row PLUS the threaded parse warnings — matching `insertSyncLog`'s `:1174-1178` union).
- `SyncLogEntry` (`:421-429`) gains `showId?: string | null`; `logSync` sets `entry.showId` **ONLY for the applied outcome** — `if (result.outcome === "applied" && "showId" in result) entry.showId = result.showId;` (`:385`) — placed right beside the existing applied-only `parseWarnings` gate (`:2161`).

**Why applied-only (round-3 finding).** Non-applied cron outcomes ALSO carry `showId` and route through `logSync`: `hard_fail` (`:3220-3225`), fetch/drive failures (`:2591-2596` return `{ ...result, showId }`), shrink-held / source-gone / parse-error (`:376,:382,:412-417`). If `logSync` populated `show_id` for those, they would land in `sync_log` with a non-null `show_id` but **payload-only `parse_warnings`** (no `ParseWarning` elements — `parseWarnings` is threaded applied-only). §3.1's "latest row with `show_id`" could then pick such a row as baseline/current, whose `summarizeDataGaps` is `{total:0}` — **fabricating "quiet drift"** (real gaps look like they vanished) or **hiding real drift** (a subsequent applied row's gaps look new). Gating `showId` to the applied outcome makes **`show_id` populated ⟺ `parseWarnings` populated (1:1)**: the monitor's `show_id IS NOT NULL` filter then selects exactly the applied cron rows that carry real parse warnings. Non-applied cron rows keep `show_id = NULL` (unchanged from today). A broader "populate `show_id` for all outcomes" observability improvement is explicitly OUT of scope here (it would reintroduce the contamination).

This is a **prerequisite task** in the plan (its own TDD task + commit), landing before the monitor-read tasks. It touches sync-core (`lib/sync/syncLog.ts`, the `logSync`/`SyncLogEntry` threading in `runScheduledCronSync.ts`) and updates the sink's existing tests (`tests/sync/syncLog.test.ts` pins the positional insert params — the expected array gains a leading `showId` and, for applied outcomes, the appended parse warnings). No advisory-lock change (the sink runs on its own post-apply connection, `syncLog.ts:44`, outside the apply tx).

**`parse_warnings` shape guard.** `insertSyncLog` (`runScheduledCronSync.ts:1174-1178`) stores `[ {...payload, outcome} (optional), ...parseWarnings ]` — the leading element is a payload object, NOT a `ParseWarning`. Both `summarizeAutoFixes` and `summarizeDataGaps` iterate and gate on `w.severity` / `w.code` (`dataGaps.ts:119-134`), so a payload object (no `severity`/`code`) is skipped. The monitor query passes `row.parse_warnings` verbatim to these summarizers — no pre-filtering needed, and this is asserted by a test (§13).

**Reuse vs re-query.** The roster/field signal reuses **the exact `show_change_log` query shape** of `loadRecentAutoApplied` plus one extra predicate (`occurred_at > windowStart`). Because `loadRecentAutoApplied` also fetches the `roster_shift_counts` RPC and groups for the in-app strip (not needed for email prose), the monitor builder issues its own leaner query rather than calling `loadRecentAutoApplied` directly. The predicate list and column set are pinned identical by a test (§13) so the two never drift on the security-relevant filters (`source`, `status`, `acknowledged_at`).

### 3.1 Sub-threshold drift — guarded definition

Drift is **per published show**. Because `show_id` is populated ⟺ the row is an applied cron parse carrying real `parse_warnings` (§3.2, applied-only gate), every `sync_log` read below implicitly selects only applied rows via `show_id IS NOT NULL` — non-applied error/skip/stale rows never enter the baseline/current selection. For each published show `s`:

1. **Baseline row** = the latest `sync_log` row with `show_id = s AND occurred_at <= windowStart`. **Guard — no baseline:** if no such row exists (new show, or first-ever run), **SKIP drift for `s` entirely** — there is no prior to measure against, and a brand-new published show's initial gaps are already surfaced by `SHOW_FIRST_PUBLISHED` (`runScheduledCronSync.ts:2287`), not the monitor digest. Do NOT substitute a synthetic zero baseline (that would misreport every existing gap as "new drift").
2. **Current row** = the latest `sync_log` row with `show_id = s AND occurred_at > windowStart`. **Guard — no in-window apply:** if none, SKIP `s` (nothing changed this window).
3. `baseline = summarizeDataGaps(baselineRow.parse_warnings)`, `current = summarizeDataGaps(currentRow.parse_warnings)`.
4. **Report `s`** iff `baseline` and `current` differ on at least one **non-`gateExempt`** `GapCode` **AND** `isQualityRegression(baseline, current) === false`. Rationale:
   - Comparing only non-`gateExempt` classes mirrors `isQualityRegression`'s own exemption (`dataGaps.ts:186` `isGateExempt`) — a transient `VENUE_GEOCODE_UNRESOLVED` flap must not manufacture "quiet drift" noise (audit Flow 6 §4.2).
   - Excluding `isQualityRegression === true` avoids double-notify: a true regression already raised `RESYNC_QUALITY_REGRESSED` (`runScheduledCronSync.ts:323`) and appears in the realtime alert channel.
5. **Rows with `show_id IS NULL`** (`sync_log.show_id` is nullable — `insertSyncLog` maps a missing show to `null`, `runScheduledCronSync.ts:1170`) are excluded from every step (drift is per-show).

## 4. Windowing — the watermark

### 4.1 Storage

New nullable column `public.app_settings.last_monitor_digest_sent_at timestamptz` (singleton row `id='default'`). Migration mirrors `supabase/migrations/20260602000003_b3_app_settings_notify_columns.sql:7` (`add column if not exists ... timestamptz`, no default, no CHECK). Idempotent (apply-twice safe via `if not exists`).

### 4.2 Read / write helpers

Both live in **`lib/notify/monitorWatermark.ts`** (new) — under `lib/notify/` deliberately, so the notify infra-contract meta-test's recursive `lib/notify` directory walk (`tests/notify/_metaInfraContract.test.ts:22-45`) catches them fails-by-default (see §10/§11). They use the supabase-client pattern.

- **Read:** `getMonitorDigestWatermark(client?)` → `{ kind: 'value'; watermark: Date | null } | { kind: 'infra_error' }`. Mirrors `lib/appSettings/getDailyReviewDigest.ts` (supabase-client `.select(...).eq('id','default').maybeSingle()`, `{data,error}` destructure, try/catch → `infra_error`).
- **Write:** `writeMonitorDigestWatermark(when: Date, client?)` → `{ kind: 'ok' } | { kind: 'infra_error' }`. Mirrors `lib/appSettings/writeSyncCronHeartbeat.ts` (`.update({ last_monitor_digest_sent_at: when.toISOString() }).eq('id','default').select('id')`; empty `data` → `infra_error`).

### 4.3 Window computation & guard conditions

`windowStart` = the watermark value. Guard conditions:

| Watermark state | `windowStart` | Rationale |
|---|---|---|
| `NULL` (first-ever digest, or column just added) | `now - 24h` | Bounded lookback so the first digest is not a giant historical backlog dump; 24h matches the daily cadence. Documented so a reviewer doesn't read it as "silently drops history." |
| non-NULL | watermark timestamp | net "since last successful monitor digest send" |
| read `infra_error` | **abort the monitor section only** (needs-attention section still sends); do NOT advance watermark; surface as a delivery-summary `infra_error` source (`getMonitorDigestWatermark`) so the cron route 5xxs (`app/api/cron/notify/route.ts:statusFor`) and the scheduler sees the degradation. | fail-closed on the monitor section; never fabricate a window. |

### 4.4 Advancing the watermark (exactly-once semantics)

`deliverDigest` returns a per-recipient count breakdown `{ sent, failed, skipped, retryLater }` (`lib/notify/deliver.ts:21-23` / `DeliveryResult`), and `activeRecipients` may return multiple recipients (`lib/notify/recipients.ts:13`, already pre-filtered to non-revoked). The watermark advances to the run's `now` **once per run**, AFTER the recipient loop, and ONLY when **all** of:

(a) the monitor content was non-empty (there was something to report), AND
(b) at least one recipient received the monitor-bearing email THIS run (`sent > 0`), AND
(c) **no** recipient hit a genuine transient non-delivery this run — `failed === 0 && retryLater === 0` (accumulate across the loop).

**Why `skipped` is deliberately NOT in the block condition (round-2 correction).** `deliverOneRecipient` increments `skipped` for THREE distinct reasons (`lib/notify/deliver.ts:326,335,341,377`): (i) the recipient was **already sent today** (dedup ledger `status='sent'`, `:332`) — they already HAVE the content; (ii) the recipient is **revoked** at delivery time (`:341`, a race vs `activeRecipients`) — they SHOULD NOT get it; (iii) send **failed and exhausted** `SEND_RETRY_CAP` retries (`:333`) — a fresh daily dedup key retries tomorrow regardless of the watermark. NONE of these is "a recipient who should get it but didn't this window and could still get it via retry." Blocking advance on `skipped` would let a single already-sent or revoked recipient **permanently** stall the watermark → the active recipient (Doug) re-receives the same section every day forever. So `skipped` does NOT block. Only `failed`/`retryLater` — genuine this-run transient failures where a same-day retry (dedup ledger `status='failed'`, `attempt_count < CAP`, so NOT skipped, `:333`) can still deliver the identical recomputed window — block the advance.

Rationale — **advance on real delivery, else re-report, never drop:**

- `failed`/`retryLater` on any recipient → watermark stays → the same window re-reports next run (`sendEmail` faulted, Resend down, 6.1 not yet configured). At-least-once beats at-most-once.
- `sent > 0` required: a same-day re-fire where **all** recipients dedup-skip (`sent === 0`) must NOT advance — that run's window `[watermark, now)` was never delivered (the earlier successful send predated those events), so it must ride to the next digest. Requiring a real `sent` is exactly this guard.
- Empty monitor content → nothing to consume; NOT advancing avoids truncating a later window with an empty intervening run.
- Advancing ONCE (not per-recipient) is why monitor content is computed once before the loop (§5): a per-recipient advance would let recipient #1's send blank recipient #2's window.

**Residual multi-recipient edge (accepted, documented).** If recipient B succeeds in run 1 (with an un-advanced watermark because recipient A `retryLater`'d that run) and A then succeeds in run 2, B's run-1 email covered `[w, now1)` while A's run-2 email covered `[w, now2)`; advancing to `now2` leaves B without `[now1, now2)`. This requires a recipient to transiently fail exactly one cron run and is bounded to a few minutes of one day's events. In the single-recipient reality (the pull-only band is Doug — §2), it cannot occur. Not worth per-recipient watermarks (§2 out-of-scope).

**Watermark write failure after successful send:** if `writeMonitorDigestWatermark` returns `infra_error` after emails delivered, surface it as a delivery `infra_error` (source `writeMonitorDigestWatermark`) → cron 5xx. Consequence: next run re-reports the same window (duplicate section in tomorrow's email) — an acceptable at-least-once degradation, strictly better than dropping. Documented so it is not read as data loss.

   `MonitorDigestResult = { kind:'ok'; model: MonitorDigestModel } | { kind:'empty' } | { kind:'infra_error' }`
   and `MonitorDigestModel = { windowStart: string; autoApplied: MonitorShowGroup[]; autofix: AutoFixSummary; drift: MonitorDriftEntry[] }`.

2. **`lib/notify/runNotify.ts` `runDigestNotify`.** Compute `const monitor = await buildMonitorDigestModel(now)` ONCE, immediately before the recipient loop (`runNotify.ts:423`). Then:
   - If `monitor.kind === 'infra_error'` → do NOT block needs-attention: still run the recipient loop and send needs-attention emails **without** the monitor section (pass `null`), do NOT advance the watermark, and record the run's delivery as `infra_error` source `buildMonitorDigestModel` so the cron route 5xxs and the scheduler sees the degradation. (Consistent with §4.3: the monitor section is independent of needs-attention; a monitor read fault must not suppress the needs-attention email.)
   - Inside the loop, `buildDigestModel(recipient, now)` still builds needs-attention. **Send condition change:** send the email if `model.kind === 'ok'` **OR** `monitor.kind === 'ok'`. If `model.kind === 'no_send'` AND `monitor.kind !== 'ok'` → `continue` (unchanged behavior). When needs-attention is `no_send` but monitor is `ok`, synthesize an empty needs-attention model (`shows: []`) so the email renders with only the monitor section.
   - Pass `monitor.kind === 'ok' ? monitor.model : null` into `deliverDigest`. Accumulate the returned `{ sent, failed, skipped, retryLater }` into run totals (today only `sent` is summed — `runNotify.ts:436-449`; extend to accumulate all four).
   - After the loop: advance iff `monitor.kind === 'ok' && runTotals.sent > 0 && runTotals.failed === 0 && runTotals.retryLater === 0` (§4.4 — note `skipped` is deliberately excluded from the block condition) → `await writeMonitorDigestWatermark(now)`; on its `infra_error`, convert the run result to delivery `infra_error` source `writeMonitorDigestWatermark`. (When `monitor.kind === 'infra_error'`, the run is already `infra_error` from the step above and the watermark is untouched.)

3. **`lib/notify/deliver.ts` `deliverDigest`.** Accept an optional `monitor: MonitorDigestModel | null` on `input`. Thread it into `renderDigest`. **Dedup key unchanged** (`digest:${dateET}`) — the monitor section rides inside the same once-per-day digest email; it does not get its own dedup row. Add `monitor_totals` to the recorded `context` (counts only — no crew PII; see §8) alongside the existing `date_et` / `source_totals`.

4. **`lib/notify/digest.ts` `buildDigestModel`.** No change to its needs-attention logic. (Its `no_send` return is now handled by the caller's send-condition change, §5.2.)

5. **`lib/notify/templates/digest.ts` `renderDigest`.** Accept optional `monitor` on `DigestInput`. Render the new section (§8) after the needs-attention list. When `monitor` is null/absent, output is byte-identical to today (empty-safe). Subject line: when needs-attention is empty but monitor is present, use a monitor-appropriate subject (§8).

**No advisory lock.** This feature reads `show_change_log` / `sync_log` and writes only `app_settings` (singleton, not a per-show mutation). ## 8. Email content spec (rendered, not conceptual)

New section rendered by `renderDigest` after the needs-attention `<ul>`s / overflow note. All dynamic values HTML-escaped via `escapeHtml` (`lib/notify/templates/escapeHtml.ts`); all copy em-dash-free and plain-language (invariant 5 — no raw codes; use `AUTO_FIX_CLASSES` / `GAP_CLASSES` `.label` strings, never `.code`).

**Section header (HTML `<h2>` / text line):** `Applied automatically since your last digest`

**Sub-blocks (each omitted when its data is empty):**

1. **Auto-applied changes** — grouped by show (title links to `${origin}/admin/show/${slug}`, or `${origin}/admin` when slug null). Per show, a bulleted list of `summary` strings (already human copy in `show_change_log.summary`). Caps: reuse `DIGEST_MAX_SHOWS` (12) shows and `DIGEST_MAX_ITEMS_PER_SHOW` (5) rows/show (`lib/notify/constants.ts:16-17`); overflow → `+N more on this show` / `+M more shows` linking to `${origin}/admin`, exactly as the needs-attention section does.
2. **Autocorrects applied** — a single roll-up line: `We automatically corrected N values (<label>: n, ...)` using `AUTO_FIX_CLASSES` labels for nonzero classes. No per-show breakdown in v1 (roll-up only; cap = the 5 fixed classes, no unbounded growth).
3. **Quiet drift** — per show with sub-threshold drift: `<show title>: <label> <prior>→<curr>` for each changed `GapCode` (label from `GAP_CLASSES`). Cap: `DIGEST_MAX_SHOWS` shows; per show cap the class lines at 5 with `+K more` overflow. Copy frames it as informational ("worth a glance"), NOT an alarm (these are below the alert threshold by definition).

**Guard conditions (every input null/empty/zero):**

| Input | Empty/zero behavior |
|---|---|
| `monitor` null (needs-attention-only email) | section entirely absent; output byte-identical to pre-6.2 |
| `autoApplied` `[]` | sub-block 1 absent |
| `autofix.total === 0` | sub-block 2 absent |
| `drift` `[]` | sub-block 3 absent |
