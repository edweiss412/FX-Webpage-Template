## 1. Problem

The `WATCH_CHANNEL_ORPHANED` admin alert (Google Drive push-subscription registration failed) has five compounding defects, confirmed by live investigation on 2026-07-01:

1. **Jargon copy.** The shipped copy says "cron" (`lib/messages/catalog.ts:249`), "The cron job" (`catalog.ts:253,256`), and "until the developer reconciles the subscription" (`catalog.ts:253`). The audience is Doug, a non-technical admin (`crewFacing: null`, `catalog.ts:250`).
2. **Misleading verb.** The banner's "Resolve" button (`components/admin/ResolveAlertButton.tsx:108`) only stamps `resolved_at`/`resolved_by` on the alert row (`app/admin/actions.ts:95-103`). It performs zero remediation, yet clears the only signal that real-time sync is broken.
3. **No recovery path.** A failed subscription becomes `status='orphaned'` and is never re-attempted: the hourly renewal queries `WHERE status = 'active'` only (`lib/drive/watch.ts:191-196`), and the GC cron only tears orphaned rows down (`watch.ts:214`). Recovery requires a developer to re-run onboarding finalize â€” the only production subscribe caller (`app/api/admin/onboarding/finalize-cas/route.ts:831,894`).
4. **No admin self-service.** There is no in-app action that re-registers the channel.
5. **Live incident.** On the validation deployment, `DRIVE_WEBHOOK_BASE_URL` is not provisioned (verified via `vercel env ls`; 23 vars, none matching). `webhookPublicUrl()` throws before the Drive API is reached (`lib/drive/watch.ts:94-99`). All 10 channel rows ever created on validation are `stopped`, zero ever activated; the unresolved alert row shows `reason: "watch_create_failed"`, `occurrence_count: 10`, raised 2026-06-28. The underlying error is swallowed by bare `catch {}` blocks (`watch.ts:383,396`) and never logged.

## 3. Design

### 3.1 Error capture and classification (watch.ts)

The two bare `catch {}` branches in `subscribeToWatchedFolder` (`lib/drive/watch.ts:383-392, 396-408`) become `catch (err)` and:

1. Classify the error via a new pure function `classifyWatchError(err: unknown): WatchErrorClass` where `WatchErrorClass = "config" | "drive_api" | "db"`:
   - `"config"`: message matches the `DRIVE_WEBHOOK_BASE_URL is required` throw (`watch.ts:97`) or Google auth/credential shapes (`invalid_grant`, `Could not load the default credentials`, missing `GOOGLE_SERVICE_ACCOUNT_JSON`).
   - `"db"`: `err instanceof DriveWatchInfraError` (`watch.ts:10-22`) â€” tx-layer failures.
   - `"drive_api"`: everything else (Drive HTTP errors, timeouts, malformed watch response from `watch.ts:297-298`).
2. `SubscribeResult`'s orphaned arm widens to carry its reason: `{ outcome: "orphaned"; channelId: string; reason: "watch_create_failed" | "activate_failed_after_watch_created" }` (both reasons are already computed at the catch sites for the alert context â€” this surfaces them to callers; existing callers ignore the extra field). **Route mapping (R8 finding 2):** an orphaned outcome with `reason: "activate_failed_after_watch_created"` is a **DB-layer infra fault wearing an orphaned costume** â€” Drive granted the channel; *our* activation UPDATE failed (`watch.ts:394-407`). The refresh loop records it in `failures` (operation `"activate_pending"`) and reconcile's own subscribe maps it to a `"activate_write"` fault â†’ the route's 500 class, *in addition to* the alert the subscribe already raised. `watch_create_failed` orphans (Drive/config class) remain pure handled degradation (200). The admin retry action is unchanged either way (orphaned â†’ banner re-render; no throw â€” the action's user feedback is the banner, not an HTTP status).
3. Extend the alert context written by `markWatchOrphanedWithTx` (`watch.ts:348-360`) with two new keys, written by **every** producer (uniform shape is what makes render-time derivation safe, since the upsert RPC replaces context wholesale â€” `supabase/migrations/20260618000000_upsert_admin_alert_failedkeys_merge.sql:55-69`, non-failedKeys contexts are not merged):
   - `error_class`: the classification above.
   - `error_message`: `redactWatchError(String(err?.message ?? err), { webhookSecret })` truncated to 300 chars. **Redaction contract (R2 finding 6):** a new pure function `redactWatchError(message, secrets)` applied at the single chokepoint where `error_message` enters the context (inside the catch, before `markWatchOrphanedWithTx`): (a) replaces every occurrence of the channel's `webhookSecret` value (in scope at both catch sites â€” the one secret that is definitionally present in the failing `files.watch` request, `watch.ts:287-295`) with `[redacted]`; (b) replaces `Bearer <non-space run>` sequences and the values of `token`/`key`/`secret`/`authorization` `[:=]` pairs with `[redacted]` (belt-and-braces for Google client error strings echoing request headers); (c) truncates to 300 chars last. Unit-tested with a fixture error whose message embeds the literal webhook secret and a Bearer token, asserting neither survives (Â§4.1). Values like env-var names (`DRIVE_WEBHOOK_BASE_URL is requiredâ€¦`) and Drive HTTP status text pass through â€” that diagnostic content is the point.
4. Log the failure: `log.error("drive watch subscribe failed", { source: "drive.watch", errorMessage: <the Â§3.1.3 redacted string>, watchedFolderId, channelId, errorClass })` (`lib/log/logger.ts:100-105`; error-level always persists to `app_events`, `logger.ts:22`). **The raw `error: err` field is deliberately omitted on this call (R5 finding 1):** the logger persists serialized `Error.message`/`stack` to `app_events` and its sanitizer redacts only emails (`lib/log/sanitize.ts`), so passing the raw error would leak into the durable log exactly what `redactWatchError` scrubs from the banner context. Redaction happens once at the chokepoint; both sinks (alert context and log) receive only the redacted string. The Â§4.1 test asserts the *logged fields* (not just the context) contain no secret from the leak fixture. No `code:` field outside a log call (scanner rule, `tests/cross-cutting/codes.test.ts:32-45` + `stripLogEmissionCalls`).

Guard conditions: `err` may be anything (string throw, undefined) â€” `classifyWatchError` must total-function over `unknown` and default to `"drive_api"`. `error_message` of a non-Error stringifies via `String()`.

### 3.2 Reconcile pass (new function in watch.ts, called from the refresh-watch cron route)

New exported function `reconcileWatchChannels(refreshResult, deps): Promise<ReconcileResult>` in `lib/drive/watch.ts` (canonical signature â€” the first parameter is the same-cycle refresh result consumed by health condition (b) below), invoked by `app/api/cron/refresh-watch/route.ts` **after** `refreshWatchSubscriptions()` in the same `runCronRoute` handler (route currently 16 lines; `rejectUnauthorizedCron` guard at `app/api/cron/_auth.ts:3`).

Steps, in order:

1. **Stale-pending sweep (silent hygiene â€” no alert upsert).** `UPDATE drive_watch_channels SET status='orphaned' WHERE status='pending' AND created_at < now() - STALE_PENDING_MAX_AGE_MS` (parameterized), logged via `log.warn` with the swept ids. This closes the crash-window gap (rows abandoned between insert-pending and the catch, `watch.ts:374-383`). The sweep deliberately does **not** touch `admin_alerts`: if the folder is actually healthy a zombie `pending` row is not a user-visible problem, and if it is unhealthy, step 3's failed re-subscribe raises/bumps the alert. Keeping the automatic producers to exactly one per cycle also keeps `occurrence_count` bumps at +1 per hour (relevant to step 5).
2. **Health predicate.** Read the singleton configured folder via `getActiveWatchedFolder()` (`lib/appSettings/getWatchedFolderId.ts:43-45`; `app_settings` singleton, CHECK `id='default'`, `supabase/migrations/20260501001000_internal_and_admin.sql:246`). Outcomes:
   - `no_folder_configured` â†’ healthy-vacuous: nothing to watch. Auto-resolve any open alert (step 4) and return.
   - `infra_error` â†’ `ReconcileResult.outcome = "infra_error"`, fault `"folder_read"` recorded (recorded-not-thrown, matching the `runNotify` contract, `lib/notify/runNotify.ts:141-147`); do not attempt subscribe. HTTP mapping per the route contract below.
   - Folder present â†’ healthy iff **both**: (a) â‰¥1 row `WHERE watched_folder_id = <folder> AND status='active' AND expires_at > now()`, **and** (b) this cycle's refresh reported **no orphaned renewal** for that folder (the route passes the Â§"Hardening" refresh result into `reconcileWatchChannels`; condition (b) is `!refreshResult.orphaned.includes(folderId)`). Condition (b) exists because renewal runs up to 24h **before** expiry (`watch.ts:416`) and a failed renewal leaves the *old* channel active â€” without (b), reconcile would auto-resolve the alert the failed renewal just raised, hiding failing renewals until actual expiry and resetting the escalation counter every hour (R4 finding 1). (The join against `app_settings.watched_folder_id` in (a) matters: after a folder switch, an old folder's active channel must not count â€” `getWatchedFolderId.ts` gotcha.)
3. **Unhealthy â†’ re-attempt, at most one subscribe per folder per cycle.** If condition (a) failed (no live channel), call `subscribeToWatchedFolder(folderId)` once. If only condition (b) failed (renewal-failing state: live channel exists but this cycle's refresh already attempted and orphaned a renewal), do **not** subscribe again â€” refresh was the attempt; a second call would double Drive traffic and double-bump `occurrence_count` per hour, distorting the escalation cadence. Either way the failure evidence is the upsert the failed subscribe already performed (`watch.ts:362-409`) â€” reconcile adds **no new producer site** for the meta-test (`tests/messages/_metaAdminAlertCatalog.test.ts:151-154` continues to pin `lib/drive/watch.ts`).
4. **Healthy â†’ auto-resolve.** `resolveAdminAlert({ showId: null, code: "WATCH_CHANNEL_ORPHANED" })` (`lib/adminAlerts/resolveAdminAlert.ts:11-24`; sets `resolved_at` only; established detector pattern per `lib/notify/detect/stall.ts:15-17`). Resolve calls are unpinned by meta-tests (verified). Because of condition (b), a renewal-failure window never auto-resolves; recovery is a later successful renewal (refresh `refreshed` includes the folder â†’ (b) passes, and the new channel satisfies (a)) or a successful admin Retry.
5. **Escalation check** (only on the unhealthy path, after step 3's subscribe returns `orphaned`): read the unresolved alert row (`code='WATCH_CHANNEL_ORPHANED' AND show_id IS NULL AND resolved_at IS NULL` â€” unique by partial index `admin_alerts_one_unresolved_idx`, `20260501001000_internal_and_admin.sql:279-280`). Fire escalation **iff** (`occurrence_count >= ESCALATION_THRESHOLD` **or** `context.error_class === "config"`) **and** no prior escalation is recorded for this alert row. The fired-once record is an `app_events` row (`source = 'drive.watch.escalation'`, `context.alertId = <alert row id>`), checked via `app_events WHERE source = 'drive.watch.escalation' AND context->>'alertId' = <alert row id> LIMIT 1`. `app_events` is append-only â€” immune to the context wholesale-replacement hazard that rules out storing a flag on the alert row (Â§3.1.3), and durable across cron restarts. A `>=` trigger + durable guard is robust to multi-bump cycles (e.g., an admin manual retry bumping the count between reconcile runs), which an exact-equality trigger is not.

   **Guard write contract (R1 finding 1, writer-guard-compliant per R3 finding 1):** the guard row is written by a new exported function `persistAppEventStrict(record): Promise<{ ok: true } | { ok: false; error: unknown }>` in **`lib/log/persist.ts`** â€” the one file the structural writer guard permits to insert into `app_events` (`tests/log/_metaAppEventsWriter.test.ts:29` pins `["lib/log/persist.ts"]` as the sole insert site; a writer anywhere else fails CI). It reuses the same sanitization path as `persistAppEvent` but returns the error instead of swallowing it (failure-visible; registered Supabase call boundary destructuring `{ data, error }` â€” the best-effort sibling keeps its `// not-subject-to-meta` waiver, the strict variant gets a registry row). The guard row supplies all NOT-NULL columns (`supabase/migrations/20260629000002_app_events.sql`): `level: "info"`, `source: "drive.watch.escalation"`, `message: "watch escalation fired"`, `context: { alertId, errorClass, occurrenceCount }`. Escalation is NOT written via best-effort `log.*` (`lib/log/logger.ts` swallows persist failures by design). Ordering: guard insert FIRST; only on confirmed success do Sentry/email fire (at-most-once bias). Guard insert error â†’ recorded infra fault for this cycle, no notifications, retry next hour (fail-closed for duplication, fail-open only in time). The residual crash window (process death between guard insert and sends) under-notifies once and is accepted â€” the Resend idempotency key on the same alert row id makes true email loss require both failures at once. Both reads/writes here are new Supabase call boundaries â†’ registry rows (Â§4 meta-test inventory).

   **Concurrency bound (R2 finding 1):** the read-then-insert guard is not atomic â€” `app_events` deliberately carries no business uniqueness constraint, and watch surfaces carry no advisory locks (Â§6). The only duplicate window is two *overlapping* reconcile executions (hourly cadence vs. sub-minute runtime; possible via manual cron trigger or a >1h hung run). Within that window, duplicates are bounded per channel: **email** stays at-most-once because both executions derive the same Resend idempotency key from the same alert row id (`baseKey("watch_escalation", alert.id, recipient)`) and Resend serializes same-key sends (`lib/notify/send.ts:20-23`); **Sentry** may record a duplicate event, which Sentry's issue grouping collapses â€” cosmetic, accepted. The fired-once contract is therefore precisely: at-most-once email delivery per (alert row, recipient) per retention window; at-most-once Sentry escalation *modulo overlapping-run duplicates that group into one issue*. Making the guard strictly atomic would require either a business unique index on the observability table or an advisory lock â€” both rejected (Â§6).

   **Retention bound:** `app_events` rows are pruned after 60 days (`supabase/migrations/20260629000002_app_events.sql:58-62`). The fired-once guarantee is therefore scoped: **at most once per alert-row lifetime within the 60-day retention window**. An alert continuously unresolved for 60+ days re-escalates once per window â€” accepted and desirable (a quarterly-broken system should re-notify). A dismissed-then-re-raised alert is a new row id â†’ escalates again (correct: new incident window). `log.error` is still emitted alongside for observability, but it is not the guard.

`reconcileWatchChannels(refreshResult, deps)` takes the same-cycle refresh result (condition (b) above). `ReconcileResult` = `{ outcome: "healthy" | "recovered" | "still_orphaned" | "renewal_failing" | "vacuous" | "infra_error"; sweptPending: number; escalated: boolean; faults: string[] }` â€” `renewal_failing` is the only-(b)-failed state (live channel, failing renewal; observable in the route body and cron logs) â€” `faults` names each recorded failure step. **Complete fault inventory (R3 finding 2/3; R4-prep comprehensive fault-vector audit):** `"pending_sweep"` (sweep UPDATE threw), `"folder_read"` (`getActiveWatchedFolder` `infra_error`), `"channel_read"` (the active-channel health-predicate SELECT on `drive_watch_channels` threw/errored), `"subscribe_infra"` (step 3's `subscribeToWatchedFolder` **threw** `DriveWatchInfraError` â€” tx-layer; distinct from the non-throwing `orphaned` outcome, `watch.ts:106-112`), `"activate_write"` (step 3's subscribe returned `orphaned` with `reason: "activate_failed_after_watch_created"` â€” the DB-fault-in-orphaned-costume case, Â§3.1.2), `"alert_resolve_write"` (`resolveAdminAlert` threw, `lib/adminAlerts/resolveAdminAlert.ts:27`), `"alert_row_read"` (escalation-check read of the unresolved alert row failed), `"guard_read"` (the existing-guard `app_events` SELECT failed â€” distinct from the write), `"guard_write"` (`persistAppEventStrict` returned error), `"pref_read"` (`getAlertOnSyncProblems` `infra_error`), `"recipients_read"` (`activeRecipients` `infra_error`, `lib/notify/recipients.ts:4` â€” notify precedent treats this as an infra delivery fault, not a silent skip), `"email_send"` (a `sendEmail` result of `{ ok: false }` â€” either kind). One step-order note: `"guard_read"`/`"guard_write"` faults suppress sends (fail-closed for duplication). **Escalation step ordering (R5 finding 2, reordered per R6 finding 1):** the sequence is strictly (i) escalation condition met, (ii) **recheck** â€” re-read the alert row immediately before the guard insert and abort if no longer unresolved (`resolved_at IS NOT NULL`; closes the race where an admin Retry resolves the alert between reconcile's initial check and escalation), (iii) **guard insert**, (iv) **sends** (Sentry â†’ email), with nothing between (ii)â†’(iii)â†’(iv) but those calls. A recheck read failure (`"alert_row_read"` fault) or a guard-insert failure (`"guard_write"` fault) aborts **before anything is consumed** â€” the escalation stays fully retryable next cycle; this ordering exists precisely so no transient DB fault can burn the fired-once guard without notifications having fired (R6 finding 1). The only remaining under-notify window is a process crash between (iii) and (iv) â€” accepted per the crash-window trade above. The residual mis-notify window (admin resolves in the sub-second gap between (ii) and (iv)) is accepted as bounded and cosmetic. A benign recheck-abort is not a fault. Every step inside `reconcileWatchChannels` is individually try/caught into these names (recorded-not-thrown); an unhandled throw out of the route handler is a contract violation, not an accepted path. Any non-empty `faults` â†’ `outcome: "infra_error"` â†’ the route's 500 class.

**`sendEmail` result mapping** (`lib/notify/send.ts:13-17`): `{ ok: true }` â†’ sent; `{ ok: "retry_later" }` â†’ benign concurrent same-key send (an overlapping run is already delivering this exact email) â€” skip, **no fault**; `{ ok: false, kind: "idempotency_conflict" }` â†’ fault (same key with different content should be impossible when content derives from the same alert row â€” surfacing it indicates a bug); `{ ok: false, kind: "infra_error" }` â†’ fault. Send ordering: guard insert â†’ Sentry â†’ email. An email fault after the guard insert is **not** auto-retried (the guard row exists; at-most-once bias per Â§3.2.5) â€” it surfaces as a 500 + `"email_send"` fault while the banner remains up, which is the accepted trade documented there.

**Route HTTP contract (R1 finding 2), mirroring the notify route's `statusFor` precedent (`app/api/cron/notify/route.ts:30-38`):** the refresh-watch route runs refresh (per-row try/catch; failures recorded, loop continues) then reconcile, and returns:
