## 1. Summary

The published-show review modal's monitoring-only attention pill ("N monitoring") is currently a non-interactive `<span>` (`components/admin/showpage/PublishedReviewModal.tsx:837`), and even when the composite pill opens the attention menu, the Monitoring group renders a single summary row, never the items (`components/admin/showpage/AttentionMenu.tsx:218-238`). The user cannot see WHICH issues the system is watching.

This feature makes the monitoring-only pill an interactive button that opens the existing attention menu, and makes the Monitoring group enumerate one row per item ‚Äî hollow positive-tone dot + alert title + a per-row auto-resolve note ‚Äî in both the monitoring-only and composite cases. The one-line summary row is retired.

- The monitoring-only span branch (`PublishedReviewModal.tsx:831-856`) is DELETED. Its rendering folds into the interactive button:
  - When `actionable === 0 && needsLook === 0 && selfHeal > 0` ("monitoring-only"): button uses the quiet palette ‚Äî `bg-surface-sunken` / `text-text-subtle` on the button, hover `bg-surface-sunken/80`, AND every warning-toned descendant swaps in lockstep: the monitoring segment's `text-warning-text/80` span (`PublishedReviewModal.tsx:782`) renders `text-text-subtle` instead, and the chevron's `text-warning-text` (`PublishedReviewModal.tsx:805-810`) renders `text-text-subtle` instead; no `*warning*` class may remain on the button or any descendant. - Degraded state C is unchanged: `alertsDegraded && selfHeal.length === 0` renders the "Alerts unavailable" span. When `alertsDegraded && selfHeal.length > 0`, the monitoring-only INTERACTIVE pill wins (same precedence as today, where the monitoring span won).

### 3.3 Open-menu reconciliation (amends split-spec ¬ß11.5a)

- The non-interactive exit set shrinks to C (degraded, no monitoring) and D (in-sync). The ENTRY set grows: monitoring-only `(0,0,s)` is now a fourth interactive entry shape. Product: 4 entry shapes (`[1,0]`, `[0,1]`, `[1,1]` with selfHeal=1, plus `[0,0]` with selfHeal=1) √ó 2 exits = **8 cells**, in BOTH the jsdom matrix (`pillFocusReconcile.test.tsx`) and the Playwright matrix (`attention-pill-focus.spec.ts`). Outcome contract unchanged: menu closes, no stale `aria-expanded`, focus rescued to the dialog root.
- NEW stays-open behavior (passive data path only): any interactive shape (open) ‚Üí monitoring-only is NOT an exit. The menu stays open; groups reconcile instantly per split-spec ¬ß9 rule 1 (instant insert/remove); the pill recolors amber‚Üíquiet with a SPLIT deterministic contract: the ROOT button's own `background-color`/`color` cross-fade via its existing `transition-colors duration-fast` (`PublishedReviewModal.tsx:734`); DESCENDANT tone-class swaps (segment span, chevron) are INSTANT ‚Äî CSS transitions do not inherit, the descendants carry no transition class, and none is added (deliberate: content-level tone follows data instantly, ¬ß9 doctrine). Both directions. `aria-expanded` stays true.
- **Focus rescue for removed rows while the menu stays open (NEW mechanism ‚Äî none exists today):** the current rescue effect fires only when `menuEffectivelyOpen` flips false (`PublishedReviewModal.tsx:353-366`). When the focused actionable/needs-look row unmounts while the menu remains open, the browser drops focus to `<body>` (standard DOM behavior on removing the focused element), breaking the modal focus trap. Mechanism: extend the post-commit rescue effect to ALSO run while `menuEffectivelyOpen` is true ‚Äî if `document.activeElement` is `<body>` (or no longer inside the dialog), refocus the pill button (still mounted, still the menu's trigger; `aria-expanded` stays true so AT context is preserved). **Empirical probe (run 2026-07-22, Chromium 147.0.7727.15, Playwright 1.59.1; script: standalone page, focus target, remove it, sample `document.activeElement` at sync/microtask/rAF, then refocus):** removing the focused element ‚Äî a `<button>` AND an `<a>` ‚Äî drops focus to `<body>` synchronously, and it REMAINS on `<body>` through the microtask and rAF checkpoints (no browser auto-restoration); an imperative `pill.focus()` at any later timing lands and stays settled on the pill. Both element kinds behave identically. This is the recorded data the mechanism rests on. **Effect re-run guarantee:** the existing rescue effect is DEP-LESS (`useEffect(() => {...})` with no dependency array, `PublishedReviewModal.tsx:353-365`) ‚Äî it runs after EVERY commit, including the amber‚Üímonitoring reconciliation where `menuEffectivelyOpen` stays `true`; extending its body is sufficient, no dependency wiring exists to go stale. **Temporal contract:** focus PASSES THROUGH `<body>` for the interval between DOM removal and the post-commit effect ‚Äî that transient is inherent to a post-commit rescue and is the same doctrine as the existing close-path rescue (`PublishedReviewModal.tsx:353-366`, also post-commit). The asserted contract is the SETTLED state: after the update commits, focus rests on the pill, not `<body>`. Ratification is EMPIRICAL, not prose: the real-browser Playwright cell below (focus a row, drive `__setItems(0,0,1)`, then assert the settled state ‚Äî menu open, `aria-expanded="true"`, `document.activeElement` is the pill) is the probe that ratifies the mechanism; jsdom focus semantics on element removal differ from browsers, so the e2e cell is the authoritative proof and the jsdom twin is corroborating only.
- `onResolved` path (¬ß1.1): resolving the last actionable item still closes the menu regardless of remaining monitoring/needs-look items (`PublishedReviewModal.tsx:413-418`) ‚Äî no code change; the existing close + user-initiated focus handling applies.
- Rebound guard (`pillFocusReconcile.test.tsx:108-136`) still applies to the C/D paths.

### 3.4 Transition inventory (delta)

Pill states after this change: **A** interactive (composite OR monitoring-only, one button), **C** degraded, **D** in-sync. Pairs A‚ÜîC, A‚ÜîD, C‚ÜîD: instant, no animation (unchanged doctrine). Intra-A mode flip (warning‚Üîquiet as counts move, BOTH directions), full inventory: (a) root button background/text ‚Äî cross-fade via existing `transition-colors duration-fast` (¬ß3.3 same definition); (b) descendant tone-class swaps (segment span, chevron) ‚Äî instant, deliberate (transitions don't inherit; none added); (c) leading dot solid-review ‚Üî hollow-positive ‚Äî instant swap (distinct elements, content follows data); (d) segment text insertion/removal ("N to confirm"/"N to review" appearing or vanishing) ‚Äî instant (matches today's composite behavior as counts change); (e) middot separators insertion/removal ‚Äî instant, same doctrine. No other visual property changes between the two modes. Menu open motion unchanged (fade+scale entrance, instant close). Compound: palette flip while menu open ‚Äî covered in ¬ß3.3; group appear/disappear mid-open ‚Äî split-spec ¬ß9 rule 1 unchanged.

### 3.5 Guard conditions

- `selfHeal.length === 0`: no Monitoring group, no monitoring segment (unchanged).
- All counts 0 + degraded: state C (unchanged). All counts 0, not degraded: state D (unchanged).
- Unknown/uncatalogued alert code in a self-heal item: `menuTitle` falls back via `alertTitle` (`lib/admin/attentionItems.ts:235-239`), note falls back to the `autoResolveNote` generic line. Nothing renders a raw code (invariant 5).
- `selfHeal.length > 99`: pill caps at "99+", sr-only + `title` carry the exact count (unchanged contract, now on a button).
- 12+ rows: scroller handles overflow (existing `max-h-96`).
