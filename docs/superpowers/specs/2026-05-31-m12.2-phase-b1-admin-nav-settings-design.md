## 2. Nav shell

`NAV = [{ id: dashboard, label: "Dashboard", short: "Home", icon: grid }, { id: settings, label: "Settings", short: "Settings", icon: cog }]` (verbatim from `app.jsx:11-14`). ### 2.2 Modes & responsive boundary
| Mode | Trigger | Renders |
|---|---|---|
| **Desktop** | viewport ≥ 720px | Top bar: brand (FXAV wordmark + "Admin" badge) · Dashboard/Settings nav · spacer · NotifBell · dark toggle · UserMenu (avatar). Content below. |
| **Mobile** | viewport < 720px | Top bar: brand · spacer · NotifBell · dark toggle · UserMenu. **Fixed bottom tab bar**: Home / Settings (icon + `short` label). |

### 2.3 UserMenu (`app.jsx:25-65`)
Avatar button (initials) → popover with: identity head (full name + email) + divider + **Sign out** item. 
- **Identity source:** `requireAdminIdentity()` returns `{ email }` (`lib/auth/requireAdmin.ts:89-150`, email from the Supabase JWT canonicalized at `:134`). The prototype's "Doug Pemberton / Production Manager" is mock; **real data = the canonical admin email only.** Name/role are NOT stored — render the email as the primary identity line; omit the name/role lines (or derive a display name from the email local-part) rather than invent data. **Guard:** if identity lookup yields no email (should be impossible past `requireAdmin()`), render the avatar with a neutral glyph and the menu with Sign out only.
- **Sign out** posts to `app/auth/sign-out/route.ts` (POST, `:89-127` — same-origin gated, calls `supabase.auth.signOut()`, clears cookies, 303 → `/auth/sign-in`). The menu item submits a same-origin POST form (not a link — GET returns 405 at the route).

### 2.4 NotifBell (`app.jsx:16-23`)
Bell icon with a count badge. 
- **Count source:** number of unresolved, non-info `admin_alerts` — the SAME filter the AlertBanner uses (`.is("resolved_at", null)` + the `INFO_SEVERITY_CODES` exclusion derived from `MESSAGE_CATALOG`, `components/admin/AlertBanner.tsx:57-59,97-99`). Implement as a `head:true` exact count query mirroring that filter, in a shared helper so the bell count and the banner's "+N more" cannot drift.
- **Click behavior:** navigates to `/admin#alerts` (the existing scroll anchor wrapping the AlertBanner, `layout.tsx:92`). On the dashboard this scrolls to the banner; from any other route it routes to `/admin` then to the anchor. No dropdown panel in B1 (YAGNI; the banner is the alert surface).
- **Guards:** count 0 → bell renders with **no badge** (not hidden — the affordance is always present). Count query infra-error → render the bell with no badge (fail-quiet, same posture as AlertBanner's null-return on error, `AlertBanner.tsx:61-78,125-128`); never surface a raw error in the chrome (invariant 5).
- **Cap:** badge displays the exact count up to 9; **10+ renders "9+"**.

### 2.5 Dark-mode toggle
**Reuse the existing `components/layout/ThemeToggle.tsx`** (sets `document.documentElement.dataset.theme`, persists to `localStorage["fxav-theme"]`, `:114-116`; no-FOUC inline script already stamps `<html data-theme>` pre-hydration at `app/layout.tsx:48`; light/dark token swap at `app/globals.css:288-320`). The toggle is currently mounted in the help Header (`app/help/_components/Header.tsx:110`) and the global Footer (`components/layout/Footer.tsx:72`). B1 mounts the same component in the admin top bar / mobile bar. **No new theme mechanism.** This serves PRODUCT.md's bright-floor/dim-backstage scenario (`PRODUCT.md:5`).

### 2.6 Page header / breadcrumbs (`app.jsx:155-204`)

**Data-ownership contract (load-bearing — prevents re-fetch / stale-header bugs).** The prototype renders one global `PageHead` from a `HEADERS` map + `window.DATA` (`app.jsx:155-204`), but that only works because the prototype has global mock data. In the real app, per-show header data (the show row, `published`/`archived`, the share token, crew-link eligibility/URL) is **page-owned** — fetched inside `app/admin/show/[slug]/page.tsx`. Therefore:
- **Layout-owned** (`app/admin/layout.tsx`): the persistent nav chrome only (top bar / mobile bar / bottom tabs / UserMenu / NotifBell / dark toggle). Its only data needs are the admin identity (§2.3) and the alert count (§2.4) — neither requires per-show data.
- **Page-owned**: each page renders its own header via a shared presentational `<AdminPageHeader title sub crumb? backHref? rightSlot?>` component (new, `components/admin/nav/AdminPageHeader.tsx`) populated **from props the page already has** — no data fetching inside the header component, none in the layout. The per-show page passes its already-computed Published/Archived pill + crew-link chip into `rightSlot`; the dashboard/settings pages pass title/sub only. This guarantees the header and body render from the **same** show/token snapshot (no second query, no drift) and keeps the per-show pill/chip exactly where their data lives.
- The layout does **not** read the route to synthesize per-show headers; static routes (dashboard, settings) simply render `<AdminPageHeader>` with route-appropriate title/sub from the page.

Per-route header content (each rendered by its page via `<AdminPageHeader>`):
| Route | Header |
|---|---|
| `/admin` (dashboard) | Title "Dashboard" + sub "Your live shows and anything that needs review." (matches Phase A `Dashboard.tsx`). |
| `/admin/settings` | Title "Settings" + sub "Manage your Drive connection, who can administer, and how the app behaves." |
| `/admin/show/[slug]` | Breadcrumb "Admin › Active shows" + **Back to dashboard** ghost button + Published/Archived pill + crew-link chip. **Phase A already ships the in-body `← Admin home` back-link (`app/admin/show/[slug]/page.tsx:263`), the title (`:267`), and the Published/Archived `StatusIndicator` (`:243,:274`).** B1 consolidates the breadcrumb + back-link into the page's `<AdminPageHeader>` (via `crumb`/`backHref` props) and **removes the now-duplicated standalone in-body `← Admin home`** (`:263`); the Published/Archived pill + crew-link chip pass through `rightSlot` (single source — rendered once, in the header, from the page's own data). Mobile reorders: title → sub → crew-link chip → pill (`app.jsx:174-180`); desktop: title + pill inline, then sub (`:182-190`). |
| `/admin/show/staged/[stagedId]`, onboarding/staged | No breadcrumb-with-crumb; plain title (these are slug-less candidate surfaces, master spec §9.1 `:2613`). |

**Guard:** unknown route → render a plain title from a sensible default (no crash); the header block must tolerate a route with no `HEADERS` entry.

## 4. Revoke-hang robustness fix

**Bug** (carry-in, RECONCILIATION §B + ARCHITECT-UPDATE §6): `RevokeRowButton.tsx` drives a `confirm → resolving → (reset)` UI state machine via `useActionState`. The "Revoking…" → reset only fires when the action **returns** a non-ok result (`:62-69` — the `ui === "resolving"` snap reads the returned result). If `revokeAdminAction` **throws** or the request **hangs**, no result returns, the snap never fires, and the button is stranded on "Revoking…" indefinitely. (Env was confirmed NOT the cause — ARCHITECT-UPDATE §6.)

**Fix:** make the resolving state self-limiting — reset on error/rejection and on a timeout, not only on a returned non-ok. Specifically: wrap the action dispatch so a thrown/rejected action transitions `resolving → error` (with a plain-language retry message), and add a watchdog timeout that returns the button to its idle/confirm state if no result arrives within a bounded window. **Regression test must stash the fix and prove the button is stranded without it** (negative-regression discipline; the test injects a throwing/hanging action and asserts the button recovers).

## 6. Dimensional invariants (Tailwind v4 — `items-stretch` is NOT default; DESIGN.md §7)

- **Mobile bottom tab bar** is fixed to the bottom of the admin viewport and does not scroll with content; the scroll container's bottom padding ≥ the tab-bar height so the last content row is never occluded (prototype `.body { padding-bottom: 80px }`, `Admin Redesign.html:224`). - **Top bar** (desktop & mobile) is a fixed-height flex row; brand / nav / spacer / actions are vertically centered (`items-center`, explicit — not relying on default).
- Each **bottom tab** fills an equal fraction of the bar width (`flex-1`) and its icon+label stack is centered.
- **Settings panels** are ≤740px centered; the Drive-connection panel's two-column "info ⟷ pill" and "helper ⟷ buttons" rows wrap on narrow widths (`between wrap`) without the pill/buttons overflowing.
