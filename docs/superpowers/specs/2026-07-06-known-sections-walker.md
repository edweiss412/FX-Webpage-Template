# Spec: Known-section-header source walker (BL-KNOWN-SECTIONS-WALKER)

**Date:** 2026-07-06
**Type:** Parser refactor + structural meta-test (defense-in-depth)
**Origin:** audit rec-6 item (c); `BACKLOG.md` BL-KNOWN-SECTIONS-WALKER (audit line 72).

## 1. Problem

`lib/parser/knownSections.ts` holds `KNOWN_SECTION_HEADERS` — the registry the class-B unknown-section scan (`lib/parser/index.ts:686-704`) consults so a **parsed** section header is not false-flagged `UNKNOWN_SECTION_HEADER`. The scan fires only when col0 is an all-caps token NOT in the registry (and not a sub-label) AND ≥2 field-header-word columns follow (`index.ts:698-700`).

The companion `tests/parser/_metaKnownSectionsRegistry.test.ts` is a HAND-MAINTAINED pin: it asserts a hardcoded `REQUIRED_HEADERS` list ⊆ `KNOWN_SECTION_HEADERS`. It does NOT read the block parsers, so a **new block parser whose section-opener header is registered in NEITHER list passes CI green**, and its own rows would false-positive `UNKNOWN_SECTION_HEADER`. The block parsers each recognize their section opener via heterogeneous private literals/regexes with **no shared introspectable constant** (survey below), so there is no way to walk them today.

## 2. Goal

Route every block parser's **section-opener** recognition through a per-parser **exported, introspectable token constant**, derive each parser's matcher from that constant (behavior-preserving), then add a **source-walker meta-test** that:

1. enumerates every `lib/parser/blocks/*.ts` file (filesystem walk — a NEW file fails-by-default);
2. for each, imports its exported `SECTION_HEADER_TOKENS` (or confirms membership in an explicit, justified `NO_SECTION_OPENER` allowlist);
3. asserts the **union** of all parsers' `SECTION_HEADER_TOKENS` is a subset of `KNOWN_SECTION_HEADERS` (every parser-recognized section opener is registered);
4. asserts a proof case: a token present in a parser const but absent from the registry FAILS.

This closes the drift class structurally: adding a block parser forces either an exported `SECTION_HEADER_TOKENS` (whose tokens must be registered) or an explicit allowlist entry with rationale — the walker fails otherwise.

## 3. Definitions

- **Section opener** = a col0 token on which a parser OPENS a multi-row section block (a record table or a keyed block that the unknown-section scan could see). This is the ONLY category the registry governs.
- **NOT a section opener** (out of registry, out of `SECTION_HEADER_TOKENS`):
  - **Single-cell / scalar-value labels** — a label whose parser reads only the SINGLE adjacent value cell (`cells[1]`) as a scalar, scanning all rows, and never opens a multi-row block:
    - `ops.ts` `COI`/`PROPOSAL`/`PO`/`INVOICE`/`INVOICE NOTES` (2-cell scalar rows). Declared in an exported `METADATA_FIELD_TOKENS` const (introspectable, explicitly NOT required in the registry — the walker asserts DISJOINTness from `SECTION_HEADER_TOKENS`).
    - `contacts.ts` `IN HOUSE AV`, `VENUE/HOTEL/HOTAL CONTACT INFO/INFORMATION/DETAIL(S)`, `ONSITE AV CONTACT` — **R1 finding 1:** `contacts.ts:88-136` row-scans and treats `cells[1]` as a scalar contact value (gated by `hasContactSignal`); it does NOT open a section. Therefore these are NOT section openers and MUST NOT be added to `KNOWN_SECTION_HEADERS`: registering e.g. `VENUE CONTACT INFO` exact would MASK a genuinely-dropped section of that exact name whose row DID carry a ≥2-field-header-word band (`| VENUE CONTACT INFO | Contact Name | Phone # | Email Address |` — the shape pinned by `tests/parser/unknownSection.test.ts`). `contacts.ts` is therefore in the `NO_SECTION_OPENER` allowlist (reason: scalar contact-label detection, `cells[1]`-only). NOTE: `IN HOUSE AV` is ALREADY in `KNOWN_SECTION_HEADERS` (grandfathered, harmless, exact-match) and is LEFT AS-IS; it is simply not a walker-covered section opener.
  - **Block terminators** — `crew.ts` `TERMINATING_LABELS`, `venue.ts` `VENUE_BLOCK_TERMINATORS`, `transport.ts` lowercase terminators (`hotel stays`/`coi`/…): tokens a parser stops AT, not opens on. The walker does NOT require these ⊆ registry.
  - **Room-name exclusion / classification prefixes** — `rooms.ts` local `SECTION_PREFIX_FAMILIES` (`ADDITIONAL`, `LUNCH`, `DETAILS`, …, `rooms.ts:70-76`) are BROAD prefixes used to REJECT compound room names / route to dedicated paths — **R1 finding 3:** they are NOT the same as the registry's narrow prefix families (`ADDITIONAL ROOM`, `LUNCH ROOM`, `LUNCH SESSION`, `knownSections.ts:79-86`). rooms' EXPORTED `SECTION_HEADER_TOKENS` is the subset of those narrow families rooms actually OPENS on (`GENERAL SESSION`, `BREAKOUT`, `ADDITIONAL ROOM`, `LUNCH ROOM` — NOT `LUNCH SESSION`, §4/R9); registry-only families like `LUNCH SESSION` are not exported by any parser; the broad internal exclusion prefixes stay PRIVATE and are neither exported nor registered.
  - **Title/sentinel tokens** — `index.ts` `NO_HEADER`, `CLIENT`-prefix title exclusion: not sections.
  - **Column sub-labels / grid headers** — governed by `KNOWN_SUB_LABELS`, unchanged.

## 6. The walker meta-test (`tests/parser/_metaKnownSectionsWalker.test.ts`)

New structural meta-test (companion to the existing hand-maintained pin, which is RETAINED as a redundant deletion-guard).

**Guarantee model (R4/R5 — same-vector reframe; honestly scoped).** Multiple review rounds kept finding matcher mechanisms a source-text guard missed (RHS-only equality, reversed equality, the agenda leading-`\s*`/alternation regex, rooms' `/^GENERAL SESSION\b/.test(col0)` non-pipe prefix, `col0.startsWith(...)`). Chasing every SHAPE is the drift-prone heuristic BL-KNOWN-SECTIONS-WALKER warns against and CANNOT be proven complete by a static source scan — no check short of full dataflow analysis proves a file derives ALL its matchers exclusively from its exported tokens. This is a **low-severity defense-in-depth** item, so the walker is a strong best-effort guard, NOT a soundness proof, and it says so:

- **PRIMARY (fully enforced) gates — force annotation + registry discipline:** (step 1) every scanned file exports `SECTION_HEADER_TOKENS` or is allowlisted (fail-by-default filesystem walk); (step 2) non-empty for non-allowlisted; (step 3) EXACT subset ⊆ registry. These three, combined with **parser-suite fidelity** (§7), are the enforced core: every section-opener parser MUST be annotated with registry-checked tokens (so the COMMON drift — a new parser with an unregistered section header — cannot pass silently), and any matcher whose behavior actually affects parsed output on a fixture is pinned.
- **STRUCTURAL NUDGE (weakly enforced) — import-link (step 4):** every token-exporting file (except `IMPORT_LINK_EXEMPT`) must IMPORT the shared factory. **This proves the factory is imported, NOT that it is the exclusive matcher constructor (R12 finding 1):** a file could import it unused and still recognize its opener via an accounted-for inline equality (`col0Upper === "VENUE"`, VENUE ∈ its own tokens, which §6.6 exempts) or regex. Import-link is therefore a code-organization nudge + the anchor for the source-guard backstop, NOT a proof of factory USE. Proving exclusive use requires whole-program dataflow analysis (disproportionate — §6.7).
- What the gates do NOT prove (see residual, step 7): (a) that a token-exporting file does not ALSO privately match some OTHER header via an uncovered mechanism on input absent from all fixtures; (b) that a file which imports the factory actually USES it rather than an accounted-for inline equality/regex. Both fall under the declared, accepted residual.
- **BACKSTOP (defense-in-depth, explicitly NOT a completeness proof):** the source-text guards (steps 5-6) flag raw-matcher SHAPES present in the codebase today (pipe/leading-ws/alternation regex; non-pipe `col0.test` prefix regex; uppercase equality both orders; `startsWith`/`includes` literal) **whose hard-coded token is an EXACT `KNOWN_SECTION_HEADERS` member** (registry-keyed trigger — a plan-preflight refinement that avoids over-firing on sub-labels / column headers / generic char-class matchers), so an accidental hand-rolled matcher for a *registered* section opener is caught early.

0. **Scanned-file set.** All of `lib/parser/blocks/*.ts` (filesystem-walked, excluding `*.test.ts`) PLUS `lib/parser/index.ts` (R3 finding 1 — it owns the agenda matcher). Token-collection (step 1), the import-link check (step 4), and BOTH source guards (steps 5-6) run over this full set.
1. **Filesystem walk.** For each file in the scanned-file set:
   - dynamically `import()` it; if it exports `SECTION_HEADER_TOKENS`, collect the tokens; else assert the filename is in the `NO_SECTION_OPENER` allowlist (declared in the test with a per-file reason string). A file in NEITHER → **fail** ("new block parser must export SECTION_HEADER_TOKENS or be allowlisted with a reason").
2. **Non-empty for non-allowlisted (R1 finding 2a).** A file NOT in the allowlist that exports `SECTION_HEADER_TOKENS` MUST export a NON-EMPTY array. An empty `SECTION_HEADER_TOKENS = []` on a non-allowlisted file → **fail** (prevents a new parser from vacuously satisfying the subset check with zero tokens).
3. **Subset assertion (EXACT membership — R2 finding 1).** Every collected token, after `normalizeHeader`, is an EXACT member of `KNOWN_SECTION_HEADERS` (`KNOWN_SECTION_HEADERS.has(normalizeHeader(token))`), NOT via `isKnownSectionHeader` (which prefix-matches room families and would let a bogus `GENERAL SESSION CATERING` pass). Room families rooms OPENS on are exported as their canonical BASE tokens (`GENERAL SESSION`, `BREAKOUT`, `ADDITIONAL ROOM`, `LUNCH ROOM` — NOT `LUNCH SESSION`, which rooms does not open on, R9 finding 1), each of which IS an exact member of the registry. A token failing exact membership → fail, naming the file + token.
4. **IMPORT-LINK (STRUCTURAL NUDGE — not a proof of use, R12 finding 1).** The refactor centralizes construction of the SIMPLE presence/equality section-opener matchers in ONE shared factory module (§7, `lib/parser/blocks/_sectionHeaderMatch.ts`, exporting `buildCol0HeaderRe(tokens)`, `buildCol0HeaderAltRe(tokens)`, `matchesSectionHeader(col0, tokens)`). The walker asserts: every file that exports a NON-EMPTY `SECTION_HEADER_TOKENS` **AND is not in the named `IMPORT_LINK_EXEMPT` set** **imports** `_sectionHeaderMatch` (static `import` check on the source). A non-exempt file that exports tokens but does NOT import the factory → **fail**. **Honest scope:** this proves the factory is IMPORTED, NOT that it is the parser's EXCLUSIVE matcher constructor — a file could import it unused and still recognize its opener via an accounted-for inline equality (`col0Upper === "VENUE"`, exempted by §6.6 because VENUE is its own token) or an allowlisted regex. So import-link is a code-organization nudge + the anchor for the shape backstop (§6.5/§6.6), NOT an enforcement of factory USE; the "build matcher from the factory" step is executed by the refactor (§7) and its behavior is pinned by parser-suite fidelity, but is not statically PROVEN by this check (residual §6.7). **`IMPORT_LINK_EXEMPT = { "rooms.ts" }` (R6 finding 1):** rooms exports its banner tokens for registry coverage but its openers are capture-and-extract / shape matchers not buildable from the presence factory (§4), so it is exempt with a declared reason; the exemption is an explicit NAMED entry so a NEW token-exporter cannot silently skip the factory — adding one requires a deliberate `IMPORT_LINK_EXEMPT` edit with justification.
5. **Raw-header-regex source guard (BACKSTOP) — REGISTERED-TOKEN-KEYED trigger (plan-preflight refinement).** The walker reads each scanned file's SOURCE TEXT (blocks/*.ts + index.ts) and **fails if it finds a raw col0-header matcher literal whose hard-coded UPPERCASE TOKEN is an EXACT member of `KNOWN_SECTION_HEADERS`** (i.e. it looks like it is matching a *registered section opener*) in a file that neither exports that token in its `SECTION_HEADER_TOKENS` nor is in `RAW_HEADER_REGEX_ALLOWLIST`. The matcher shapes scanned: (i) pipe-anchored `/^\s*\|\s*<TOKEN>.../` (incl. optional leading `\s*` and alternation groups — the agenda shape `index.ts:339`); (ii) non-pipe col0-tested prefix regex `/^<TOKEN>.../.test(col0)` (the rooms shape `rooms.ts:621/639/657`); (iii) `startsWith`/`includes` against a `<TOKEN>` string literal. **Implementation notes (plan-preflight):** the scan is **CASE-SENSITIVE** (real openers are UPPERCASE; lowercase scalar-label regexes like `contacts.ts` `/^\s*(?:venue|hotel)\s+contact/i` do NOT fire), and anchored-regex source is **whitespace-normalized** (`\s`/`\s+`/`\s*`/literal spaces → single space) so `/^GENERAL\s+SESSION\b/` is caught as well as `/^GENERAL SESSION/`. A token the file OWNS as a `METADATA_FIELD_TOKENS` entry is exempt (e.g. `ops.ts` `COI` — `COI` is a registered header consumed as a scalar field; `ops.ts` `/^\s*COI\s*$/i` therefore does not fire). **Why keyed on registry membership (NOT any `[A-Z][A-Z /&]{2,}` literal):** a plan-time preflight over the live tree showed a broad char-class trigger over-fires on dozens of legitimate NON-opener matchers — `KNOWN_SUB_LABELS` (`START`/`NAME`/`PHONE`/`EMAIL`/`ROLE`/`TRAVEL`/`SET`/`SHOW DAY`/`FINISH`/`TRT`), column headers (`FLIGHT DETAILS`/`FLIGHT BOOKED`/`MAIN`/`SECONDARY`/`DATE`/`TIME`), generic char-class matchers (crew `BLOCK_LABEL_RE`, index `TABLE_ROW_RE`), and value sentinels (`TBD`). None of those is a registered section opener, so keying the trigger to `KNOWN_SECTION_HEADERS` membership makes the guard HIGH-SIGNAL (it fires only on a hidden opener for a *registered* section) while auto-exempting the benign cases — this is the proportionate design for a low-severity best-effort backstop, not the drift-prone chase-every-shape the BL warns against. A NEW parser hand-writing `/^\|\s*GENERAL SESSION\s*\|/`, `/^BREAKOUT\b/.test(col0)`, or `col0.startsWith("HOTEL")` for a registered token it doesn't own (and isn't allowlisted for) → **fail**. **`RAW_HEADER_REGEX_ALLOWLIST` is TOKEN-SPECIFIC (whole-diff review R1 [medium], upgraded from file-wide):** each entry is `file → { tokens: string[], reason }` and the guard asserts the DETECTED `{file, token}` is explicitly allowed — so a file already in the allowlist can no longer hide a NEW hand-rolled matcher for a DIFFERENT registered token it doesn't own (that token fires unless enumerated). A file's OWN section/metadata tokens are exempt via `ownTokens` and are NOT listed (so `hotels.ts` `/^HOTEL$/i` and `transport.ts` `/^TRANSPORTATION\//i` need NO entry — they match only owned tokens; `scheduleTimes.ts` `DATES` is covered by `EQUALITY_LITERAL_ALLOWLIST`, short-circuited before the raw check). The map therefore lists ONLY the cross-section boundary/classification references to another section's non-owned banner: `rooms.ts` → `[DATES, CREW, DRESS, TRANSPORTATION, HOTEL, VENUE, AGENDA, DETAILS]` (block-terminator alternations `:864/:1048/:1314` detecting the NEXT section's banner to end the room block; IMPORT_LINK_EXEMPT), `event.ts` → `[GENERAL SESSION, BREAKOUT]` (`:181` room-block boundary break), `gear.ts` → `[GENERAL SESSION, BREAKOUT]` (`:90/:98` room-family classification owned by rooms), `index.ts` → `[GENERAL SESSION, BREAKOUT]` (`ROOM_FAMILY_RE :434`; its agenda capture `:353` is index's OWN token). Proof (h) pins the tightening: an allowlisted file (event) is STILL flagged for a registered token (DATES) not in its enumerated list.
6. **Uppercase-equality-literal source guard (BACKSTOP) — same REGISTERED-TOKEN-KEYED trigger.** The walker scans each file's SOURCE TEXT for uppercase-literal equality in BOTH operand orders — `(===|!==)\s*["']<TOKEN>["']` and `["']<TOKEN>["']\s*(===|!==)` — where `<TOKEN>` normalizes to an EXACT `KNOWN_SECTION_HEADERS` member, and **fails** unless ACCOUNTED FOR: a member of that file's exported `SECTION_HEADER_TOKENS` ∪ `METADATA_FIELD_TOKENS` (e.g. dates' own `=== "DATES"` sites become `matchesSectionHeader`; venue's `col0Upper === "VENUE"` where VENUE ∈ its tokens; ops' `COI`), OR an explicit reasoned `EQUALITY_LITERAL_ALLOWLIST` (`scheduleTimes.ts: ["DATES"]` — consumes the dates-block boundary owned by `dates.ts`; `index.ts: ["CLIENT"]` — CLIENT-title sentinel; sentinels like `NO_HEADER`). Sub-label / column-header equalities (`=== "SET"`, `=== "FLIGHT DETAILS"`, `=== "MAIN"`) do NOT trigger — they are not registered section openers. A NEW parser hand-writing `label === "GENERAL SESSION"` (either order) for a registered opener it doesn't own → **fail**.
7. **Bounded residual (honestly declared — NOT a completeness proof).** The walker is a strong best-effort guard for a low-severity item, not a soundness proof. The source guards (5-6) flag the raw-matcher shapes present today (pipe/leading-ws/alternation regex; non-pipe `col0.test` prefix regex; uppercase equality both orders; `startsWith`/`includes` uppercase literal). Because the source guards are keyed to `KNOWN_SECTION_HEADERS` membership (§6.5/§6.6, plan-preflight refinement), a parser could STILL match an UNREGISTERED header via a plain raw matcher (`=== "CATERING"`, CATERING not yet in the registry) OR a registered header via a genuinely exotic mechanism the guards do not model (a computed/non-literal token, a `.match()` on a built regex, a lowercase or non-`[A-Z /&]` literal, indexing/slicing). If such a private match is on input NOT present in any committed fixture, parser-suite fidelity stays green and the walker does not catch it. **Import-link is likewise import, not use (R12 finding 1):** a token-exporting file may import `_sectionHeaderMatch` yet recognize its opener via an accounted-for inline equality (`col0Upper === "VENUE"`, VENUE ∈ its tokens so §6.6 exempts it) or an allowlisted regex — the walker does NOT prove the factory is the EXCLUSIVE constructor. **Both are the DECLARED, accepted residual.** It is acceptable because: (a) the PRIMARY gates still force EVERY block-parser file to be annotated-or-allowlisted and every exported token to be registry-checked — so the COMMON drift (a new parser with a simple header) cannot pass silently; (b) any private matcher that affects real parsed output on a fixture IS caught by parser-suite fidelity; (c) closing the residual fully requires whole-program dataflow analysis, which is disproportionate for a low-severity defense-in-depth guard and is the drift-prone over-engineering BL-KNOWN-SECTIONS-WALKER explicitly cautions against. The walker's header comment states this residual verbatim so a reviewer is pre-loaded and does not re-litigate it as an undiscovered hole.
8. **Disjointness.** For any file exporting BOTH `SECTION_HEADER_TOKENS` and `METADATA_FIELD_TOKENS`, assert they are disjoint (a token cannot be both a section opener and a scalar metadata field).
9. **Proof test (non-vacuity).** (a) synthetic `{ SECTION_HEADER_TOKENS: ["ZZZ_UNREGISTERED"] }` fails the exact-subset check; (b) a non-allowlisted file with `SECTION_HEADER_TOKENS: []` fails the non-empty check; (c) a source string exporting tokens but NOT importing `_sectionHeaderMatch` fails the import-link check; (d) a raw `/^\|\s*GENERAL SESSION\s*\|/` literal, a `/^GENERAL SESSION\b/.test(col0)` literal, and a `col0.startsWith("GENERAL SESSION")` — each referencing a REGISTERED token in a non-owning, non-allowlisted source — all fail the raw-matcher guard; (e) `label === "GENERAL SESSION"` fails the equality guard; (f) the REVERSED `"GENERAL SESSION" === label` also fails. **(g) negative control (proves the narrowed trigger is not vacuously broad):** a raw `/^\|\s*FLIGHT DETAILS\s*\|/` and `label === "FLIGHT DETAILS"` do NOT fail (FLIGHT DETAILS is not a `KNOWN_SECTION_HEADERS` member — it is a column header). Each proves its gate fires on the real threat and only on it. **(h) token-specific allowlist (whole-diff review R1):** an allowlisted file (event, which allowlists only `GENERAL SESSION`/`BREAKOUT`) is STILL flagged for a registered token NOT in its list (`DATES`, detected via both Form A and Form B) — proving the allowlist can no longer hide drift onto a different registered token. (Tokens are chosen as REGISTERED section openers, not `CATERING`, because the trigger is registry-keyed — §6.5/§6.6.)
10. **No-orphan (FAILS — whole-diff review R1 [medium], upgraded from warn):** **fail** if a `KNOWN_SECTION_HEADERS` entry is claimed by no parser const AND no `PREFIX_SECTION_FAMILIES` member AND is not in `EXPECTED_ORPHANS`. Rationale: a header added to the registry that NO parser opens silently suppresses `UNKNOWN_SECTION_HEADER` for that section (the unknown-section scan treats it as "known" and stops emitting) — a real observability / data-loss regression, not cosmetic. `EXPECTED_ORPHANS` documents every intentional non-opener, each verified as genuinely consumed (not dropped) with a reason travelling in-code: `VENUES` (plural alias of VENUE), `HOTELS` (plural alias of HOTEL), `IN HOUSE AV` (contacts scalar label, grandfathered §3), `LUNCH SESSION` (`PREFIX_SECTION_FAMILIES` room-family variant), `COI` (ops scalar `METADATA_FIELD_TOKENS`), `DOCUMENT FOLDER LINK` (scalar drive-anchor field, `lib/drive/unknownFieldAnchors.ts`), `PULL SHEET` (parsed by the dedicated `lib/parser/pull-sheet.ts`, outside `blocks/`), `FOYER` (room/area name, agenda room detection). Anything NOT in that set blocks CI.

## 7. Behavior preservation (the hard requirement)

This is a REFACTOR: parser OUTPUT must not change. Enforced by:

- **Single shared matcher factory (for the SIMPLE matchers).** `lib/parser/blocks/_sectionHeaderMatch.ts` is the SOLE constructor of the **simple presence/equality** section-opener matchers — crew, hotels, transport-col0, event, dates, venue, client, dress, agenda: `buildCol0HeaderRe(tokens)` (pipe-anchored, optional leading `\s*`), `buildCol0HeaderAltRe(tokens)` (alternation, for agenda/event multi-variant), `matchesSectionHeader(col0, tokens)` (equality/normalized). Each of those parsers' section-opener matcher is produced by these, from its exported `SECTION_HEADER_TOKENS`. The import-link check (§6.4) pins that the factory is IMPORTED (a structural nudge); it does NOT statically prove exclusive USE (R12 finding 1 / residual §6.7) — that the refactored matcher is factory-built is executed here and behavior-pinned by parser-suite fidelity. **`rooms.ts` is the declared exception (§4, R6 finding 1):** its capture-and-extract / shape room-banner matchers are NOT rebuilt from the factory (they are import-link-exempt and `RAW_HEADER_REGEX_ALLOWLIST`ed); their behavior is preserved by parser-suite fidelity, and their banner-token identity is registry-checked via rooms' exported `SECTION_HEADER_TOKENS`.
- **Parser-suite fidelity (behavior preservation for fixture-exercised forms).** The existing parser test suite — `tests/parser/exporterFixtures.test.ts` (targeted fidelity assertions over the 7 real `fixtures/shows/exporter-xlsx` shows — production Drive-XLSX→markdown output, `exporterFixtures.test.ts:1-11`), `tests/parser/unknownSection.test.ts` (its 7-fixture real-exporter corpus, `unknownSection.test.ts:105`), and every parser unit test — MUST stay green with NO test/assertion edits and NO snapshot regeneration. Any diff in parsed output is a refactor bug, not an accepted change. Scope of this proof (R6 finding 2 / R7 finding 2 — no false-green): these are **targeted assertions over real exporter output**, NOT a full-output golden byte-snapshot; they preserve behavior for every header form EXERCISED BY A FIXTURE — including rooms' capture-extract matchers, whose real outputs those assertions pin. They do NOT by themselves prove a rebuilt matcher is equivalent for accepted header forms ABSENT from all fixtures; that narrower equivalence is covered, for the refactored SIMPLE matchers, by the per-matcher derivation unit test (next bullet), and any residual beyond that falls under the honestly-declared bounded residual (§6.7). It is therefore a behavior-preservation net for the shipped fixture suite, NOT a completeness/anti-drift proof of full matcher equivalence.
- **Matcher-derivation equivalence.** Where a regex is rebuilt from tokens, a co-located unit test asserts the rebuilt regex matches EXACTLY the same set of header strings as the original (enumerate the accepted + a rejected near-miss).
- **normalizeHeader consistency.** All token comparisons use the existing `normalizeHeader` (upper, single-spaced, trimmed) so casing/spacing behavior is unchanged.

## 8. Guard conditions

- **Empty/whitespace col0:** matchers already guard (`if (!col0)` in the scan); token-derived matchers preserve the same guard.
- **A parser recognizing zero section openers** (agenda helpers, ops): allowlisted, not required to export tokens.
- **A regex that matched a SUPERSET of its declared tokens** (bug): the equivalence unit test (§7) fails, forcing the const to capture the true accepted set.
- **A new file added to `lib/parser/blocks/`:** walker fails until it exports tokens or is allowlisted — the core anti-drift guarantee.

## 9. Scope

- **In scope:** exported token consts in each block parser doing section-opener detection; derive matchers from them (behavior-preserving); registry additions (§5); new walker meta-test; retain the existing hand-maintained pin.
