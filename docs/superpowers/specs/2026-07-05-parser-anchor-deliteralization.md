## 2. Part A — Room-header shape predicate

### 2.1 Current brittle behavior (cited)

`rooms.ts` hangs v1 room recognition on two hardcoded venue names, `MABEL` and `LAUDERDALE` (from the single 2024 east-coast fixture), at three sites:

1. **`extractGsBlock` terminator** (`lib/parser/blocks/rooms.ts:687`): `/^\|\s*(GENERAL SESSION|BREAKOUT|ADDITIONAL|LUNCH|MABEL|LAUDERDALE|DETAILS)\b/i` — stops GS bare-field extraction at the next room/section header.
2. **v1 room loop** (`:825`): `const mabelRe = /^\|\s*(MABEL\s+\d[^|]*|LAUDERDALE[^|]*?)\s*\|/gim;` — the ONLY discovery path for a label-less venue-named room block.
3. **`NEXT_ROOM_HEADER_RE`** (`:880-881`): `/^\|\s*(GENERAL\s+SESSION|BREAKOUT|ADDITIONAL\s+ROOM|LUNCH\s+ROOM|MABEL|LAUDERDALE|DETAILS)\b/i` — the block-extraction terminator used by `extractBoBlock` (`:883-898`).

Consequence (audit #6): every new venue has new room names; a differently-named breakout header at a venue that isn't the east-coast one is **lost or mis-grouped** because the loop matches the two literal names. Part A closes this for the **`NAME + DAY-range` shape** — a future `GRAND BALLROOM DAY 1 & 2` (or any proper name carrying a trailing DAY-range) now parses exactly as MABEL/LAUDERDALE do. **Explicitly NOT closed (R31 f1, descoped — see §2 "Descoped"):** a dims-ONLY header with no DAY-range (`SALON ABCD&#10;60' x 45'`). It is indistinguishable from a dims-bearing asset (`PROJECTION SCREEN&#10;5' x 9'`), origin/main never parsed it, and every attempted admit/signal path for it (14 adversarial rounds) reopened asset fabrication / field theft — so it stays unsupported and is filed to BACKLOG rather than parsed unsafely or flagged as noise.

**Descoped (out of scope — R31 f1):** a dims-ONLY breakout header with NO trailing DAY-range (`SALON ABCD&#10;60' x 45'`, `MERIDIAN HALL&#10;50' x 30'`). Rationale: such a row is byte-for-byte the same SHAPE as a dims-bearing ASSET/equipment row (`PROJECTION SCREEN&#10;5' x 9'`, `4' X 8' RISER`), so a name-blind parser cannot tell "novel dims-only room" from "asset" — 14 adversarial rounds confirmed every admit/evidence/ownership gate that tried reopened asset fabrication or field theft. origin/main's loop never parsed this shape either (it matched only the two literal names), so leaving it unparsed is NOT a regression. A blanket data-gap/warning signal is ALSO rejected: it would fire on every dims-bearing gear row in every sheet (pure noise, worse than silent). Filed to `BACKLOG.md` (`BL-ROOM-DIMS-ONLY-NOVEL-HEADER`): if a future need arises, the correct gate is a positive room signal the sheets actually carry (a `BREAKOUT`/room-section context or an explicit room label), not a dims token. **What IS closed:** any novel breakout with a trailing DAY-range (the MABEL/LAUDERDALE shape) now parses.

Note: shape-based recognition already exists for the *GS* label-less case — `findGsBlockVenueHeader` (`:543-581`) requires "strong evidence" (in-cell `&#10;` newline OR a dims token, plus a section-banner exclusion list). Part A applies the analogous "positive structural signal, not a bare name" discipline to the v1 breakout-room loop, but uses the **trailing DAY-range** as that signal (not dims — R30 f1) because it uniquely marks a room and never an asset.

### 2.2 New behavior

Introduce the room-shape functions below (a pure name-shape test, a trailing-DAY-range test, the composed admit predicate, a merge-key, and the precomputed terminator model). All exported from `lib/parser/blocks/rooms.ts`:

```ts
//     that is NOT dims-leading, NOT a section banner, NOT a field label, AND has a NON-EMPTY

//     (i) TRAILING-on-its-line — the DAY line ENDS with the range, so a single-line DAY-note

//     (ii) LAST-content — every NON-EMPTY line AFTER the DAY line is a dims-only line (`\d+ ' x …`), so a
```

**Design note — STRUCTURAL CONVERGENCE of the room-discovery false-positive vector (adversarial-review R2 f1, R3 f2, R4 f1, R6 f1, R7 f1, R10 f1, R11 f1, R12 f1, R15 f1, R17 f2, R18 f1, R19 f1, R20 f1, R21 f1, R23 f1, R24 f1, R26 f1, R27 f1, R28 f1, R29 f1, R30 f1 — a recurring vector).** Per the AGENTS.md same-vector rule ("if the round after comprehensive re-analysis STILL surfaces a same-vector finding, stop patching, declare the vector unresolved, and deep-dive spec + diff together until convergence is STRUCTURAL, not per-instance"), the emit model was re-derived from origin/main's actual contract instead of patched again. **Root cause:** origin/main's `mabelRe` loop matched ONLY `MABEL \d…|LAUDERDALE…` (`rooms.ts:825`) — literally the two names — and BOTH are `NAME + trailing DAY-range` headers with NO dims (verified: `MABEL 1&#10;DAY 1 & 2` → `slice(1)=["DAY 1 & 2"]`, no dims; single-line `LAUDERDALE 1, 2, 3 DAY 1 & 2` → `slice(1)=[]`, no dims). The `secondLineDims` admit path (added to catch a hypothetical dims-only novel room `SALON ABCD&#10;60' x 45'`) admitted dims-bearing rows — which are **structurally indistinguishable from dims-bearing ASSETS** (`PROJECTION SCREEN&#10;5' x 9'`, `RISER`), the SOLE driver of R11/R18/R20/R23/R28/R29/R30 (asset fabrication, field theft, run-ownership leaks). **Resolution:** admit ONLY on the `NAME + trailing DAY-range` shape (drop `secondLineDims`). Consequences: (a) **corpus no-op holds** — MABEL/LAUDERDALE are DAY-range headers, so `isRoomHeaderShape` admits exactly `{MABEL 1, LAUDERDALE …}` among the east-coast rows, and neither carries dims so none are lost; (b) **the asset/note vector VANISHES** — an asset/equipment/DAY-note carries no trailing DAY-range, so it is never admitted, never emits, never terminates, and cannot steal a field — no emit-eligibility, field-evidence window, or run-ownership machinery is needed; (c) a dims-ONLY novel room (`SALON ABCD&#10;60' x 45'` with no DAY) is NOT parsed — but origin/main's loop never parsed it either (it is not one of the two literal names), so this is NOT a regression, it is identical to origin/main behavior, and such a shape is fundamentally ambiguous with an asset. This is a **DROP-IN for `mabelRe`**: only row SELECTION changes (name-match → DAY-shape-match) plus the R27 merge-key refinement; DISPLAY name (`buildEmptyRoom("breakout", firstLine)`), dims-from-`slice(1)`, `mergeBoFields(extractBoBlock(...))`, the `seen` dedup, and the `roomHasContent` emit gate are ALL preserved VERBATIM. The **two-stage** model is now simply admit-then-emit:

1. **ADMIT** a candidate iff `roomHeaderNameShape && headerDayMarker` — a proper NON-dims-leading NAME on line 1 PLUS a TRAILING `DAY N` day-range (a line ENDING in the range) anywhere in the cell. Matches the committed east-coast rooms (verified against `2024-05-east-coast-family-office.md`): `MABEL 1&#10;DAY 1 & 2` (`:26`, DAY on the SECOND line) and the SINGLE-LINE `LAUDERDALE 1, 2, 3 DAY 1 & 2` (`:35`, DAY INLINE, commas in the name). The DAY marker is checked on ANY line (LAUDERDALE carries it inline, R21 f1) but ONLY as a TRAILING day-range (R26 f1) so a DAY-note (`SPECIAL DAY 1 NOTES` — an alphabetic word after the number) is rejected. **Nothing else is an admit signal:** a plain note has no DAY-range; a multi-line note (`FOH POSITION&#10;Downstage`) has none; an equipment/asset row (`4' X 8' RISER`, `PROJECTION SCREEN&#10;5' x 9'`) has dims but no DAY-range; a field-label row is rejected by name-shape item 3. All are excluded because they carry NO trailing DAY-range.

   **De-literalization is a DROP-IN for the `mabelRe` loop (`rooms.ts:823-847`), guaranteeing the corpus no-op by construction.** The ONLY changes are (i) row SELECTION (`MABEL \d…|LAUDERDALE…` → `roomHeaderNameShape && headerDayMarker`) and (ii) the group KEY (R27 f1 + R34 f1). Everything else is VERBATIM: DISPLAY name is `firstLine` as-is (`buildEmptyRoom("breakout", firstLine)` — `MABEL 1` and the full `LAUDERDALE 1, 2, 3 DAY 1 & 2` stored EXACTLY as origin/main stores them), dims ride from `rawHeader.split("\n").slice(1)`, `mergeBoFields(extractBoBlock(...))`, the `seen` dedup, and the `roomHasContent` emit gate. **Group KEY (R27 f1 + R34 f1):** origin/main keys on `firstLine.toUpperCase()` (`rooms.ts:829`), baking an INLINE DAY-range into the key (harmless for two literal names, but under de-literalization it would split a same-name header that carries the DAY inline from one that carries it on a second line). The key is `roomGroupKey(col0Raw, firstLine)` = `roomBaseName(firstLine)` (name, trailing inline DAY stripped, uppercased) + the NORMALIZED `dayRangeOf(col0Raw)`. This merges the inline-DAY and second-line-DAY forms of the SAME name+day (R27) but keeps DISTINCT day-ranges apart (`SALON ABCD DAY 1` vs `SALON ABCD DAY 2` → two rooms, NOT one impossible merged room — R34 f1); DISPLAY name stays VERBATIM. No-op on the corpus (verified): `MABEL 1&#10;DAY 1 & 2` → key `MABEL 1 1&2` (singleton), `LAUDERDALE 1, 2, 3 DAY 1 & 2` → key `LAUDERDALE 1, 2, 3 1&2` (SINGLE block) → same single rooms, same verbatim display names.
2. **EMIT** a room for a key-group iff `roomHasContent` (the room has ≥1 populated field or dims — `rooms.ts:424-441`, origin/main's exact gate, UNCHANGED). Because only genuine `NAME + DAY-range` headers are admitted (never an asset/note), a lone empty DAY-header is simply dropped by `roomHasContent` exactly as origin/main drops an empty MABEL — there is NO asset to fabricate a room, so no field-evidence or run-ownership gate is required.

**`roomHeaderNameShape(col0Raw)` (pure) returns true iff ALL hold:**

1. **Non-empty, label-less proper name — CASE-INSENSITIVE (adversarial-review R10 finding 1).** After flattening `&#10;`→newline, taking the first line, trimmed, and **uppercasing**, it matches the proper-name shape `^[A-Z0-9][A-Z0-9 &',./-]*$` (letters/digits/spaces + a small punctuation set INCLUDING the comma — the committed east-coast `LAUDERDALE 1, 2, 3 DAY 1 & 2` single-line header carries commas, adversarial-review R21 f1; at least one letter) **AND is NOT dimension-leading — it does NOT match `^\d+\s*'\s*x`i (adversarial-review R20 f1).** An equipment/asset row leads with its dimensions (`4' X 8' RISER`, `5' X 9' WIDESCREEN…`); a real room NAME does not (`SALON ABCD`, `MABEL 1`, `Grand Foyer`). Rejecting a dims-leading first line excludes equipment at the name-shape gate. (Names that merely start with a digit but are not dims — `5TH AVENUE BALLROOM` — do NOT match `^\d+\s*'\s*x` and are unaffected.) The match is case-INSENSITIVE (the literal `mabelRe` it replaces used `/gim`), so a mixed-case novel venue name like `Salon ABCD` / `Grand Foyer` IS admitted — that is the feature's whole point. Mixed-case *field labels* (`Breakout Room Setup Date / Time`) are NOT excluded by casing; they are excluded by item 2 (`BREAKOUT`/`ADDITIONAL`/… banner) and item 3 (field-label), both of which uppercase before comparing. The TRAILING DAY-range admit gate (§2.2) is the real discriminator for anything else — a mixed-case non-room row carries no DAY-range and is not admitted; the corpus no-op test pins that no mixed-case non-room row in the 15 fixtures becomes a room. (The dims-leading rejection in this item is retained defensively for the name-shape predicate's own contract, though dims no longer drive admission — R30 f1.)
2. **Non-empty identity, and not a known section/room banner.** Compute the room IDENTITY = the uppercased first line with any TRAILING DAY-range STRIPPED, i.e. `roomBaseName(firstLine)` (adversarial-review R32 f1) — NOT the raw first line, so an INLINE-DAY header collapses to its bare name first. **The identity MUST be NON-EMPTY (adversarial-review R36 f1):** a pure day-range row (`| DAY 1 |`, `| DAYS 1 & 2 |`, `| DAY 1 & 2 |`) passes item 1's raw-first-line shape but strips to `""` — it is a sheet DAY MARKER, not a room name — so a non-empty base is required BEFORE the section-token check (else the empty identity trivially misses every exact-match token and the day-only row is admitted into `roomHeaderLines`, where it can terminate GS/BO extraction or, with field-looking rows beneath, emit a bogus room). Then the (non-empty) identity must NOT be a section banner. Two match modes (adversarial-review R33 f1 — a bare `startsWith` for generic tokens is TOO BROAD: it would drop real compound rooms like `Hotel Ballroom DAY 1 & 2` / `Crew Lounge DAY 1 & 2`):
   - **PREFIX-match (identity `startsWith`)** — ONLY the room/section FAMILIES that intentionally carry a suffix and are claimed by a dedicated path: `GENERAL SESSION`, `BREAKOUT`, `ADDITIONAL`, `LUNCH`, `DETAILS` (so `BREAKOUT 3`, `ADDITIONAL ROOM 2`, `LUNCH ROOM` are excluded — the BO/GS paths own them via `seen`).
   - **EXACT-match (identity `===`)** — the bare generic section tokens (a section header is EXACTLY the token, never a compound room name): `DOCUMENTS`, `DATES`, `CREW`, `DRESS`, `TRANSPORTATION`, `HOTEL`, `VENUE`, `AGENDA`, `CONTACTS`, plus every token in `KNOWN_SECTION_HEADERS` / `KNOWN_SUB_LABELS` (`lib/parser/knownSections.ts:34,96`) including `FOYER`. So `HOTEL DAY 1 & 2` → identity `HOTEL` → rejected, but `Hotel Ballroom DAY 1 & 2` → `HOTEL BALLROOM` ≠ `HOTEL` → ADMITTED; `FOYER DAY 1 & 2` → `FOYER` → rejected, `GRAND FOYER DAY 1 & 2` → `GRAND FOYER` → admitted.
   Also reject a `GS <label>` field row (`^GS\s`, regex). **Note:** this is intentionally NARROWER than the live `findGsBlockVenueHeader` exclusion (`:565-571`), which prefix-matches ALL tokens including the generic ones — that GS-path breadth would reject a compound `Hotel Ballroom` GS venue too, but it is a separate path; the breakout predicate uses the corrected exact-match for generic tokens so differently-named DAY-range rooms actually parse (the PR's core claim). (Banners never carry a DAY-range, so the strip is a no-op for them; it matters only for a bare token + DAY like `FOYER DAY 1 & 2`.)
3. **Not a room-FIELD label** (adversarial-review R4 finding 1). A sheet with UPPERCASE field labels (`BO SETUP`, `AUDIO`, `SET TIME`) — if any carried a trailing DAY-range — would otherwise satisfy the name-shape + DAY-range gate and fabricate a room named after a field label — a regression the literal MABEL/LAUDERDALE loop cannot cause. Reject the row if its uppercased first line — after stripping an optional leading `BO ` or `GS ` prefix — is one of the room-field labels that `applyBoFields`/`applyGsLabel` recognize (`rooms.ts:695-707` + the `applyBoFields` label set): `SETUP`, `SET TIME`, `SHOW TIME`, `STRIKE TIME`, `AUDIO`, `VIDEO`, `SCENIC`, `LIGHTING`, `LED`, `POWER`, `OTHER`, `DIMENSIONS`, `FLOOR`, `DIGITAL SIGNAGE`, `NAME(S)`, `NOTES`. The exclusion list is DERIVED from the live `applyBoFields`/`applyGsLabel` code (not hardcoded in the spec) so it cannot drift; the plan pins it to a shared constant.

**No emit-eligibility / field-evidence / run-ownership machinery (STRUCTURAL CONVERGENCE, R30 f1).** Because admit is gated on a TRAILING DAY-range (the room signal that no asset/note carries), every admitted candidate IS a room header — there is no admitted asset to fabricate a room or steal a field, so the entire field-evidence-window / run-ownership apparatus that earlier rounds bolted on (R12/R23/R28/R29) is removed. Emit is origin/main's `roomHasContent` gate, unchanged.

**Uses — admit-then-emit, with a shared terminator set:**

- **Pass 0 — `computeRoomHeaderModel(markdown)` (top-level, computed ONCE before any block parse; adversarial-review R17 finding 2).** Built at the TOP of `collectV2V1Rooms` (`rooms.ts:215`), BEFORE `parseGsRoom(markdown)` (`:218`) and `parseBoRooms(markdown)` (`:220`), and **INDEPENDENT of the BREAKOUT/LUNCH `seen` state** (which does not yet exist at `parseGsRoom` time — the cyclicity R17 f2 flags). It splits `markdown` into `lines` ONCE, scans every table row's col0, and ADMITs iff `isRoomHeaderShape(rawHeader)` (= `roomHeaderNameShape && headerDayMarker` — a proper non-dims-leading NAME on line 1 plus a TRAILING `DAY N` day-range that is the cell's LAST content, i.e. only dims lines may follow it; NOT an asset/note/single- or multi-line DAY-note — R26 f1 + R30 f1 + R35 f1). Because `roomHeaderNameShape` item 2 already rejects the `GENERAL SESSION`/`BREAKOUT`/`ADDITIONAL`/`LUNCH`/`DETAILS` banners, those structural rows are never admitted — so the model needs NO `seen` input. Each admitted candidate is recorded `{key = roomGroupKey(rawHeader, firstLine), displayName = firstLine, lineIndex = i}` (`roomGroupKey` = base name + normalized day-range, R27 f1 + R34 f1; `displayName` is `firstLine` VERBATIM for `buildEmptyRoom`; `lineIndex` is a LINE index — R24 f1), grouped by `key`. It returns `{ lines, roomHeaderLines, groups }` where **`roomHeaderLines` = the `lineIndex` of EVERY admitted candidate** (no emit-eligibility distinction — every admitted DAY-range header is a genuine room header and a valid extraction boundary). An asset/equipment row (`4' X 8' RISER`, `PROJECTION SCREEN&#10;5' x 9'`) or a note (`SPECIAL NOTE`, `FOH POSITION&#10;Downstage`, single-line `SPECIAL DAY 1 NOTES`, or multi-line `SPECIAL DAY 1&#10;NOTES`) carries NO DAY-range as its LAST content (a note's DAY-range, if any, is not trailing-on-its-line or is followed by a non-dims prose line) → NEVER admitted → not in `roomHeaderLines` → can neither emit nor terminate nor steal a field (R30 f1 + R35 f1). This one model is THREADED as a parameter into `parseGsRoom` → `extractGsBlock` AND `parseBoRooms` → `extractBoBlock` (both take the precomputed model), so GS and BO extraction share ONE terminator set regardless of call order.
- **Pass 1 — GS + BO extraction consume the shared model.** `parseGsRoom` (`:218`, runs FIRST) and `parseBoRooms` (`:220`) both receive the Pass-0 model. The BREAKOUT/LUNCH `seen` dedup stays exactly where it is in `parseBoRooms` (`:807,830`) — it governs which rooms are EMITTED/deduped AFTER extraction, NOT termination. Duplicate suppression is therefore decoupled from the terminator prepass (R17 f2): the terminator set is purely structural line indices, computed once, order-independent.
- **Pass 2 — extract each admitted member block, merge by key, emit iff `roomHasContent`.** For each key-group, extract EVERY member block INDEPENDENTLY via `extractBoBlock(model.lines, candidate.lineIndex, model)` — one call per admitted candidate, starting at the candidate's LINE (R24 f1). Behavior-identical to the live `mabelRe` loop's `extractBoBlock(markdown, m.index)` (`rooms.ts:826-843`) because `m.index` is the row's line-start, so `markdown.slice(m.index) === lines.slice(candidate.lineIndex).join("\n")` (verified). `mergeBoFields` the per-block results across the group (UNCHANGED, `:854-876`), build the room via `buildEmptyRoom("breakout", displayName)`, and **emit iff `roomHasContent` (origin/main's exact gate, `:844-846`, UNCHANGED)** — a lone DAY-range header with no fields/dims is dropped exactly as origin/main drops an empty MABEL. `MABEL 1&#10;DAY 1 & 2` / `LAUDERDALE …` have content → emitted; any asset/note is never admitted so never reaches Pass 2 (R30 f1).
- **Terminator (LINE-BASED, R24 f1)** — `extractGsBlock(lines, gsStartLine, model)` (`:670-693`) and `NEXT_ROOM_HEADER_RE` → `extractBoBlock(lines, startLine, model)` (`:880-898`), BOTH consulting the Pass-0 `roomHeaderLines` (the line of every admitted DAY-range header). extractBoBlock walks `lines` from `startLine`; at relative position `k` the ABSOLUTE line index is `startLine + k`. A line terminates the block iff it is a structural keyword (`GENERAL SESSION|BREAKOUT|ADDITIONAL ROOM|LUNCH ROOM|DETAILS`) OR its ABSOLUTE index ∈ `roomHeaderLines` — **EXCEPT the block's own starting line (`k === 0`) is NEVER a terminator, preserving the existing positional `k > 0` guard verbatim (`extractBoBlock:888-894`).** Two properties: (1) an asset/equipment/note row is NOT admitted (no DAY-range) → not in `roomHeaderLines` → never severs a preceding room's field extraction (R30 f1 subsumes the earlier R11 concern — an asset can't be a terminator because it's not a room header at all); (2) R17-f2 — because the model is precomputed before `parseGsRoom`, a novel DAY-range room AFTER the GS block terminates `extractGsBlock` (its line is in the shared `roomHeaderLines`) instead of bleeding into the GS room. A member never self-terminates on its own header line.

The **corpus-wide rooms no-op test** (§8) plus explicit **negative terminator, negative discovery, equipment/asset-not-admitted (no-truncation), and same-key merge tests** pin this — the highest-risk area in the PR, bounded by the DAY-range admit gate and a whole-corpus regression, not asserted blind.

**Known limitation (R11 finding 2, backlog):** a room whose name is EXACTLY a `KNOWN_SECTION_HEADERS` token — in practice only `FOYER` (`knownSections.ts:64`) is both a registered section token AND a plausible bare room name — is NOT admitted (item 2 excludes it; a bare `Foyer` is genuinely ambiguous with the section token). Rather than ship a venue-token carve-out list (the exact literal-list brittleness this PR removes), bare `Foyer`-as-room is documented as unsupported and filed as a backlog item; a test asserts `Grand Foyer DAY 1 & 2` works and bare `Foyer` is (currently) not a room.

There is no residual asset/note admit path (the dims-admit path that created one was removed — R30 f1), and the corpus-wide rooms no-op test is the backstop. Regression tests cover single-line-note (R18 f1), multi-line-note (R19 f1), dims-leading + name-first-with-dims equipment (R20 f1 + R30 f1), a DAY-note (R26 f1), and the single-line inline-DAY LAUDERDALE header (R21 f1).

### 2.4 Regression anchor (Part A)

## 3. Part B — Stage-restriction token grammar

### 3.1 Current brittle behavior (cited)

`extractStageRestriction` (`lib/parser/personalization.ts:158-169`) recognizes exactly three hand-picked phrasings via three regexes:

- `FULL_STAGE_ONLY_PATTERN` (`:53-54`) — `Load In / Set / Strike / Load Out ONLY` → all 4 physical stages.
- `LOAD_IN_SET_ONLY_PATTERN` (`:55`) — anchored `^- Load In / Set ONLY$` → `[Load In, Set]`.
- `LOAD_OUT_STRIKE_ONLY_PATTERN` (`:56`) — anchored `^- Load Out / Strike ONLY$` → `[Load Out, Strike]`.

`extractRoleFlags` (`:267-370`) DUPLICATES the same three patterns to strip the stage prefix before tokenizing role flags (`:279-293`).

Consequence (audit #7): a reordered or novel-but-valid subset — `Set / Strike ONLY`, `Load Out / Strike ONLY` (note: only `Load In/Set` and `Load Out/Strike` orderings are hardcoded), `Set / Show ONLY` — falls through `extractStageRestriction` to `{kind:"none"}` AND is then mis-tokenized by `extractRoleFlags` into an `UNKNOWN_ROLE_TOKEN` cascade (e.g. `Set / Strike ONLY` → tokens `SET`, `STRIKE ONLY` → two spurious unknown-role warnings), and the crew member sees the **whole show** (a valid restriction silently ignored). The grammar recognizes these reordered/subset stage clauses; a malformed one that mixes a real stage with an unreadable token (`Set / Rehearsal ONLY`) gets a precise `UNKNOWN_STAGE_RESTRICTION` instead of the confusing multi-`UNKNOWN_ROLE_TOKEN` cascade. (A clause with NO recognized stage — `Rehearsal ONLY`, `RIGGER ONLY` — is a role clause and keeps its existing `UNKNOWN_ROLE_TOKEN` behavior; §3.2.)

### 3.2 New behavior — one shared `parseStageClause`

Introduce one order-independent grammar function, shared by both consumers (kills the 3-pattern duplication):

```ts
// lib/parser/personalization.ts
const STAGE_RESTRICTION_VOCAB = ["LOAD IN", "SET", "SHOW", "STRIKE", "LOAD OUT"] as const;
//                                                  ^^^^ NEW: Show added (already a valid WorkPhase)

export type StageClause = {
  /** Recognized stage tokens in appearance order, deduped. Empty if none recognized. */
  stages: Array<"Load In" | "Set" | "Show" | "Strike" | "Load Out">;
  /** True iff a VALID trailing restriction marker was present — exactly bare `ONLY`
   *  or `ONLY***` (three stars). A trailing 1–2 stars (`ONLY*`/`ONLY**`) is NOT a
   *  valid marker (mirrors live `STAGE_TRAILING_MARKER_RE`, personalization.ts:183-187). */
  hasOnly: boolean;
  /** The role cell with the leading stage clause + ONLY marker removed, for role-flag tokenizing. */
  cleaned: string;
  /** True IFF the body has a valid ONLY marker AND ≥1 recognized STAGE token AND ≥1 UNKNOWN token
   *  (the malformed branch → UNKNOWN_STAGE_RESTRICTION). A ZERO-stage body (e.g. `Rehearsal ONLY`,
   *  `RIGGER ONLY`) is a ROLE clause and leaves this FALSE (adversarial-review R32 f2); an all-STAGE
   *  (no-unknown) body is a valid EXPLICIT restriction and also leaves this FALSE. */
  unrecognizedRestriction: boolean;
  warnings: ParseWarning[];
};
export function parseStageClause(roleCell: string): StageClause
```

**Grammar (evaluated on the role cell, after day-restriction extraction already ran upstream — see `crew.ts:299-304`):**

1. Strip an optional leading `-` and whitespace (preserved for `cleaned` reconstruction).
2. **Full-4-list lenient-star carve-out FIRST (backward-compat, adversarial-review R17 finding 1).** Before the strict marker search, test the retained live `FULL_STAGE_ONLY_PATTERN` — `/Load\s+In\s*\/\s*Set\s*\/\s*Strike\s*\/\s*Load\s+Out\s+ONLY\*{0,3}/i` (`personalization.ts:53-54`, the EXACT-ORDER full physical-4 list with a LENIENT `ONLY\*{0,3}` marker). The pattern is the VERBATIM live `FULL_STAGE_ONLY_PATTERN` (UNANCHORED, `ONLY\*{0,3}`), so it (a) can match a full-stage substring AFTER a leading role prefix, and (b) accepts ANY trailing star count including `****` — because `\*{0,3}` matches the `***` prefix of `****` (verified). Both are DELIBERATE origin/main mirroring (see the star-count note below). If it matches → `stages = [Load In, Set, Strike, Load Out]`, `hasOnly = true`, `unrecognizedRestriction = false`, `consumedOnlyClause = true`. **`cleaned` PRESERVES the ENTIRE prefix (adversarial-review R18 f2 + R19 f2):** let `prefix` = text BEFORE the matched span, `tail` = text after it; **`cleaned = prefix + tail`** — the matched full-stage-ONLY span is EXCISED and everything else is preserved VERBATIM (NOT segment-filtered). This flows ALL prefix segments to `extractRoleFlags`: recognized roles are kept AND unrecognized tokens still emit `UNKNOWN_ROLE_TOKEN`, exactly as origin/main (whose anchored `extractRoleFlags` strip left the whole prefix intact while `FULL_STAGE_ONLY_PATTERN` produced the 4-stage restriction). Examples: `A1 / Load In / … ONLY*` → `cleaned = "A1 / "` → role `A1`; `RIGGER / Load In / … ONLY` → `cleaned = "RIGGER / "` → `UNKNOWN_ROLE_TOKEN(RIGGER)` PRESERVED (R19 f2); `A1 / RIGGER / Load In / … ONLY` → `cleaned = "A1 / RIGGER / "` → `A1` + `UNKNOWN_ROLE_TOKEN(RIGGER)`; pure `Load In / … ONLY*` (no prefix) → `cleaned = tail`. **Star-count note (R17 f1 + R19 f3 — anti-expansion, do NOT add `(?!\*)`):** the full-4 carve-out intentionally accepts `ONLY*` / `ONLY**` / `ONLY***` / `ONLY****`+ as an explicit `[Load In,Set,Strike,Load Out]` restriction, mirroring origin/main's `FULL_STAGE_ONLY_PATTERN` EXACTLY. A full-4 restriction EXCLUDES pure `Show` days (now a distinct WorkPhase); demoting ANY star count to whole-show would EXPOSE Show days — the R17 f1 regression. So the marker-enumeration "4+ stars invalid → whole-show" rule governs the GENERALIZED grammar (partials) ONLY; the full-4 compat path is deliberately lenient for every star count. The lenient-star tolerance is scoped to THIS exact full-4 pattern; it does NOT generalize to subsets.
3. **Otherwise, find the STRICT ONLY marker.** Locate the first VALID marker `\bONLY\b(?:\s*\*{3})?(?!\s*\*)`. **A marker is valid ONLY as bare `ONLY` or `ONLY***` (exactly three stars, optional whitespace before them); a trailing 1–2 stars — `ONLY*` / `ONLY**` — is NOT a valid marker (adversarial-review R14 finding 1).** The negative lookahead `(?!\s*\*)` rejects any ONLY-token followed by a non-triple star run, mirroring the live `STAGE_TRAILING_MARKER_RE` (`personalization.ts:183-187`, which peels only `ONLY` / `ONLY***` / bare `***` and deliberately leaves `ONLY*`/`ONLY**` untouched — `***`-count tolerance is a deferred concern). If NO valid marker (ABSENT, including a malformed non-full-4 `ONLY*`/`ONLY**`) → `hasOnly = false`, `stages = []`, `unrecognizedRestriction = false`, `cleaned = roleCell` UNCHANGED, return. This is fail-safe: a malformed PARTIAL subset like `Set / Strike ONLY*` is NEVER consumed as a restriction (so it can never narrow — origin/main never narrowed a partial on a starred marker either, since its partial-subset narrowing used the star-free `LOAD_IN_SET_ONLY_PATTERN`/`LOAD_OUT_STRIKE_ONLY_PATTERN`), and `extractRoleFlags` then emits `UNKNOWN_ROLE_TOKEN` on the garbled tokens (whole-show + surfaced signal, never silent — adversarial-review R2 finding 2). Descriptive full lists w/o ONLY are handled by the retained `FULL_STAGE_PATTERN` strip in `extractRoleFlags`.
4. **ONLY present (non-full-4).** Let `body` = text between the leading dash and the ONLY marker; `tail` = text after the marker. **Comprehensive re-analysis of the stage-clause classification vector (adversarial-review R1 f1, R3 f1, R5 f1, R8 f1, R14 f1, R16 f1, R22 f1, R25 f1 — a recurring vector; per the AGENTS.md same-vector rule the model is now a TOTAL per-TOKEN classification anchored on the live tokenizer, not per-segment).** **Tokenize `body` EXACTLY as `extractRoleFlags` does, then classify each atomic token — NOT each `/`-segment (R25 f1).** The prior per-`/`-segment model mis-handled a hyphen-MIXED segment like `Set - LEAD`: it saw one recognized role (`LEAD`) and labeled the WHOLE segment ROLE, silently swallowing the embedded `Set` STAGE. Token-level classification extracts BOTH. Tokenization (single shared helper, pinned to the SAME constants `extractRoleFlags` uses so it cannot drift):
   1. **Extract MULTI_WORD_TOKENS FIRST** (e.g. `SHOW CALLER`) from `body`, recording each as a ROLE token and EXCISING it from the remaining text — so multi-word roles are matched before any split and can NEVER decompose into a stray `SHOW` STAGE token (preserves R1 f1: `SHOW CALLER` ≠ `SHOW`). The physical two-word STAGES `LOAD IN`/`LOAD OUT` contain a SPACE but no `/` or `-`, so they survive the split below intact and are recognized as single STAGE tokens.
   2. **Split the remaining text on `/` AND `-`** into atomic tokens (trimmed, non-empty). Classify each token into exactly one kind:
      - **STAGE** — trimmed-uppercase EXACTLY equals a `STAGE_RESTRICTION_VOCAB` member (`Load In`/`Set`/`Show`/`Strike`/`Load Out`).
      - **ROLE** — not STAGE, and matches `ROLE_NORMALIZATIONS` (the SAME table `extractRoleFlags` uses). Covers `A1`, `BO`, `LEAD`, `V1`, `GS`.
      - **UNKNOWN** — neither (e.g. `Rehearsal`, `Sho`).
   Collect `stageTokens` / `roleTokens` (incl. the multi-word roles from step 1) / `unknownTokens`.

   **Structural discriminator: (i) a STAGE token is the SOLE signal that separates a stage restriction from a role clause, and (ii) the presence of an UNKNOWN token is the SOLE signal that separates a VALID restriction from a malformed one — there is NO stage-word guessing/heuristic** (that would reintroduce the brittle literal lists this PR removes). A ROLE token NEVER makes a clause malformed (R22 f1); it is preserved to `cleaned`. Three mutually-exclusive branches, keyed on the token partition:
   - **Valid restriction (explicit)** — ≥1 STAGE token AND ZERO UNKNOWN tokens (every non-STAGE token is a recognized ROLE). → `stages` = the STAGE tokens (appearance order, deduped, canonical), `hasOnly = true`, `unrecognizedRestriction = false`, `consumedOnlyClause = true`. **`cleaned` PRESERVES the ROLE tokens: `cleaned = <the ROLE tokens, joined by ' / '> + tail`** (a role-prefixed valid restriction keeps BOTH — role AND the stage restriction). Covers `Set / Strike ONLY` (roles=[] → `[Set,Strike]`), `A1 / Set / Strike ONLY` (role `A1` + `[Set,Strike]`, R22 f1), `BO / Set / Show ONLY` (role `BO` + `[Set,Show]`), **`Load In / Set - LEAD ONLY` (hyphen-mixed → stages `[Load In,Set]` + role `LEAD`, R25 f1 — the `Set` day is NO longer dropped)**, `SHOW ONLY` (`[Show]`), and the clean full-4.
   - **Malformed (some-unknown)** — ≥1 STAGE token AND ≥1 UNKNOWN token. → `stages = []` (NEVER partial-narrow to the recognized subset, R3 f1 — the stage restriction is DELIBERATELY NOT applied), `hasOnly = true`, `unrecognizedRestriction = true`, `consumedOnlyClause = true`. **`cleaned` PRESERVES ALL NON-STAGE tokens — recognized ROLE tokens AND unknown/typo tokens (R16 f1 + R28 f2): `cleaned = <the ROLE and UNKNOWN tokens, in original order, joined by ' / '> + tail`.** Only STAGE tokens are excluded (they are accounted by the single `UNKNOWN_STAGE_RESTRICTION`, whose message embeds the full raw `roleCell`). Routing the non-stage tokens back through `extractRoleFlags` matches origin/main: recognized roles flag, multi-word typo-roles autocorrect (`ROLE_TOKEN_AUTOCORRECTED` — e.g. `CONTENT CRETION` → `CONTENT CREATION`, `personalization.ts:333-352`), and genuine unknown role-prefixes (`RIGGER`) emit `UNKNOWN_ROLE_TOKEN` — so no role or typo-role is silently dropped. → `UNKNOWN_STAGE_RESTRICTION` + whole-show. **The restriction is NOT applied — the fail-open choice (R28 f2 rebuttal):** an unknown token adjacent to a stage may itself be a MIS-TYPED STAGE (`Set / Shoo ONLY`, `Shoo` = `Show` typo); applying `[Set]` would HIDE the `Show` day — §9 forbids hiding days, so any unknown ⟹ whole-show. Covers `Set / Rehearsal ONLY` (`cleaned = "Rehearsal"` → `UNKNOWN_ROLE_TOKEN`), `A1 / Set / Rehearsal ONLY` (`cleaned = "A1 / Rehearsal"` → role `A1` + `UNKNOWN_ROLE_TOKEN(Rehearsal)`; STAGE `Set` excluded), `RIGGER / Set ONLY` (`cleaned = "RIGGER"` → `UNKNOWN_ROLE_TOKEN(RIGGER)`, R28 f2), `CONTENT CRETION / Set ONLY` (`cleaned = "CONTENT CRETION"` → autocorrected role, R28 f2), `Set - Rehearsal ONLY` (hyphen-mixed stage+unknown → malformed), `Load In / Rehearsal ONLY - LEAD` (`- LEAD` preserved via `tail`).
   - **No-stage (ROLE clause)** — body contains ZERO STAGE tokens. → role clause regardless of whether its roles are known; `stages = []`, `hasOnly = true`, `unrecognizedRestriction = false`, `consumedOnlyClause = false`, **`cleaned = roleCell` UNCHANGED** so `extractRoleFlags` handles it with its own tokenizer — PRESERVING recognized roles and emitting `UNKNOWN_ROLE_TOKEN` for unrecognized ones (R5 f1 hyphen decomposition, R8 f1 unknown-role preservation). Covers `A1 ONLY`, `SHOW CALLER ONLY`, `BO - V1 ONLY`, `GS - A1 ONLY` (roles preserved), `RIGGER ONLY` → `UNKNOWN_ROLE_TOKEN`, `A1 / RIGGER ONLY` → `A1` kept + `UNKNOWN_ROLE_TOKEN(RIGGER)`, `Rehearsal ONLY` → `UNKNOWN_ROLE_TOKEN` (matches `origin/main`; no stage-word guess). NO `UNKNOWN_STAGE_RESTRICTION` here.

   (A date token in `body` — `\d{1,2}/\d{1,2}` — is a leaked day restriction normally consumed upstream by `extractDayRestriction` (`crew.ts:299`); it has no recognized stage, so it falls into No-stage and is left to the existing role/day path. Defensive; not expected.)

**Structural invariant (the class-closing rule, R22 f1 + R25 f1):** with the body partitioned into STAGE / ROLE / UNKNOWN **tokens** (per-token, NOT per-`/`-segment) — (a) ZERO STAGE ⟹ role clause; (b) ≥1 STAGE AND ZERO UNKNOWN ⟹ EXPLICIT restriction on the STAGE tokens (ROLE tokens allowed and preserved); (c) ≥1 STAGE AND ≥1 UNKNOWN ⟹ `UNKNOWN_STAGE_RESTRICTION`. A ROLE token NEVER makes a clause malformed, and a STAGE token embedded in a hyphen-mixed group (`Set - LEAD`) is NEVER swallowed (R25 f1). **Role-preservation invariant (R16 f1 + R22 f1 + R28 f2):** in EVERY branch the non-stage content reaches `extractRoleFlags` — via `cleaned = roleCell` unchanged (No-stage), `cleaned = <ROLE tokens> + tail` (explicit — only recognized roles accompany a VALID restriction), or `cleaned = <ROLE + UNKNOWN tokens> + tail` (malformed — recognized roles AND unknown/typo tokens both preserved, so a typo-role autocorrects and an unknown role-prefix like `RIGGER` surfaces as `UNKNOWN_ROLE_TOKEN` instead of vanishing) — so no branch drops assignment metadata. This subsumes `SHOW CALLER` (multi-word role, no stage), hyphenated `BO - V1` (no stage → role), `RIGGER`/`Rehearsal` (no stage → role/UNKNOWN_ROLE_TOKEN), `Set / Strike` (stage, no unknown → `[Set,Strike]`), `A1 / Set / Strike` (role + stage → `[Set,Strike]` + role A1, R22 f1), `Load In / Set - LEAD` (hyphen-mixed → `[Load In,Set]` + role LEAD, R25 f1), `Set / Rehearsal` (stage + unknown → UNKNOWN_STAGE_RESTRICTION), and `A1 / Set / Rehearsal` (role + stage + unknown → UNKNOWN_STAGE_RESTRICTION + role A1 kept). A dedicated unit test asserts all three branches directly, parameterized over the decision table.

**Marker-validity enumeration (asterisk sub-vector, R14 f1 — pinned as a parameterized unit test in §8).** Marker validity is gated FIRST, orthogonally to the all/some/no-stage branch — an invalid marker collapses to the ABSENT branch (`hasOnly = false`) regardless of body:

**`extractStageRestriction` becomes:**

```ts
export function extractStageRestriction(roleCell: string): {
  restriction: StageRestriction;
  warnings: ParseWarning[];
  consumedOnlyClause: boolean; // true iff a VALID (bare ONLY or ONLY***) stage clause was parsed (explicit OR unrecognized)
} {
  const clause = parseStageClause(roleCell);
  if (clause.stages.length > 0 && clause.hasOnly) {
    return { restriction: { kind: "explicit", stages: clause.stages }, warnings: [], consumedOnlyClause: true };
  }
  if (clause.unrecognizedRestriction) {
    return {
      restriction: { kind: "none" },
      warnings: [{
        severity: "warn",
        code: "UNKNOWN_STAGE_RESTRICTION",
        message: `Role cell has a work-phase restriction we couldn't read: '${roleCell}'`,
        rawSnippet: roleCell,
      }],
      consumedOnlyClause: true,
    };
  }
  return { restriction: { kind: "none" }, warnings: [], consumedOnlyClause: false };
}
```

Signature change: `extractStageRestriction` now returns `{restriction, warnings, consumedOnlyClause}` (the warnings mirror `extractDayRestriction`'s `DayRestrictionResult` shape, `:68-73`). ## 4. Part C — Short-header typo tolerance (deferred P3 pickup)

### 4.1 Current behavior (cited)

`normalizeSectionHeaders` (`lib/parser/sectionHeaderNormalize.ts:44-97`) fuzz-corrects only three LONG headers — `LONG_SECTION_VOCAB = ["TRANSPORTATION", "EVENT DETAILS", "GS DETAILS"]` (`:16`) — behind a gate that accepts a label-only row OR a row with ≥1 field-header word (`:70-73`). The short routers `CREW`/`TECH`/`HOTEL`/`VENUE` were explicitly **deferred** as P3 in `docs/superpowers/specs/2026-06-27-parser-typo-tolerance-design.md` (§out-of-scope: "P3 (short section routers CREW/TECH/HOTEL/VENUE behind the field-band gate …)").

This part corrects the typo in a pre-pass so TECH parses AND, for CREW, pre-empts the cryptic MI hard-fail with a clean auto-correct + warning. ### 4.2 New behavior

Add a `SHORT_SECTION_VOCAB` corrected behind gating STRICTER than the long-vocab path.
```ts
const SHORT_SECTION_VOCAB = ["CREW", "TECH"] as const;
```

Scoped to the two short routers whose REAL header rows in the corpus carry `SECTION_FIELD_HEADER_WORDS` so the field-band gate actually fires (adversarial-review R7 finding 2): `CREW | NAME | ROLE | PHONE …` (`NAME`/`ROLE`/`PHONE` are field-header words) and `TECH | PHONE | ARRIVAL | DEPARTURE` (`PHONE`). **`HOTEL` and `VENUE` are intentionally excluded**: their real header rows are `HOTEL | RESERVATION #1 | | RESERVATION #2` and (v1) `VENUE | Four Seasons …` — NO cell is a `SECTION_FIELD_HEADER_WORD`, so the field-band gate would NEVER fire and a `HOTLE`/`VENEU` typo would be silently left uncorrected. ## 9. Disagreement-loop preempts (for the adversarial reviewer)

- **Fail-open posture on unreadable clauses is intentional.** A some-stage clause (`Set / Rehearsal ONLY`) → whole-show + `UNKNOWN_STAGE_RESTRICTION`; a no-stage clause (`Rehearsal ONLY`) → whole-show + `UNKNOWN_ROLE_TOKEN` (role clause). Both are whole-show + warning (NOT zero-days, NOT hard-fail), mirroring the existing `unknown_asterisk` fallback (`crew.ts:347-362`) and safer than hiding a show. Cited: §1 out-of-scope (Rehearsal) + the `effectiveViewerDateRestriction` zero-days trap (`stageSchedule.ts:66`). **Stage-token presence is the sole stage-vs-role discriminator — no stage-word heuristic (see §3.2 structural invariant); do not relitigate.**
- **Shape-recognized rooms are NOT signaled.** Deliberate (brainstorming chose "replace", declined "always-signal"). A newly-recognized room is visible on the review surface by its presence. §1 out-of-scope.
- **`minLen: 4` for the short vocab** (vs the established `minLen: 5` for field-alias fallback) is deliberate — `CREW`/`TECH` are 4 chars. The caller-side field-band corroboration (`countFieldHeaderWords ≥ 1`, `:73`) + `noExactSpellingElsewhere` + `EXCLUDE` are the compensating gates. Cited: typo-tolerance design §4.3 deferred these *behind the field-band gate*, which is exactly this gate. **Part C vocab is CREW/TECH only** — HOTEL/VENUE header rows carry no `SECTION_FIELD_HEADER_WORD` so the field-band gate can never fire for them (R7 f2); shipping them would be a hollow promise, so they are deferred to a per-router follow-up. `DATES` also excluded (P4, region-less, already-loud MI-3). See §4.2.
- **Room de-literalization admits ONLY on a trailing DAY-range; emit is origin/main's `roomHasContent` (STRUCTURAL CONVERGENCE, R30 f1).** The `secondLineDims` admit path was REMOVED: it admitted dims-bearing rows indistinguishable from dims-bearing assets, the sole driver of the recurring asset/note/run-ownership vector (R11/R18/R20/R23/R28/R29/R30). Admitting only `NAME + trailing DAY-range` (the exact MABEL/LAUDERDALE shape) excludes every asset/note structurally, so NO emit-eligibility / field-evidence / run-ownership gate exists — emit is simply `roomHasContent`, verbatim origin/main. Do not relitigate re-adding dims-based admission (it re-opens the entire vector); a dims-only novel room with no DAY is deliberately out of scope (origin/main never parsed it either).
- **The `consumedOnlyClause` flag on `extractStageRestriction`** exists solely to suppress the pre-existing `crew.ts` triple-asterisk `UNKNOWN_DAY_RESTRICTION` when an `ONLY***` stage clause already consumed the marker (R7 f3) — it is not a new restriction concept.
- **`UNKNOWN_STAGE_RESTRICTION` is BOTH counted (data-gap) AND anchored, by design.** It mirrors `UNKNOWN_ROLE_TOKEN`/`UNKNOWN_DAY_RESTRICTION`, which each appear in BOTH `GAP_CLASSES` and `OPERATOR_ACTIONABLE_ANCHORED` (`dataGaps.ts:36-37,164-165`) — a crew-role sheet-data gap Doug can fix at a resolvable cell. Membership in both sets is intentional and pinned by the completeness meta-test (§7 item 8); do not relitigate as double-classification. (R13 f1.)
- **`STAGE_VOCAB` (typo vocab, `:174`) stays 4-wide; `STAGE_RESTRICTION_VOCAB` (grammar, new) is 5-wide.** Two different vocabularies for two different jobs (spelling correction vs clause recognition). Not a contradiction.
- **Classification is per-TOKEN, not per-`/`-segment; a ROLE token is preserved and NEVER makes a clause malformed (R16 f1 + R22 f1 + R25 f1).** The token-level classifier (multi-word extraction FIRST, then split on `/` AND `-`, then classify each atomic token STAGE/ROLE/UNKNOWN) makes role preservation a structural property of EVERY branch — `cleaned` always routes recognized roles to `extractRoleFlags` — AND guarantees a STAGE token embedded in a hyphen-mixed group (`Set - LEAD`) is never swallowed by an adjacent role (R25 f1): `Load In / Set - LEAD ONLY` → explicit `[Load In,Set]` + role LEAD, NOT `[Load In]`. Malformed (`UNKNOWN_STAGE_RESTRICTION`) is reserved for ≥1 STAGE token **and ≥1 UNKNOWN token**; a role-prefixed VALID restriction (`A1 / Set / Strike ONLY`) produces EXPLICIT `[Set,Strike]` AND keeps role `A1` (R22 f1). A leading role on a genuinely malformed clause (`A1 / Set / Rehearsal ONLY`) keeps `A1` while `Rehearsal` is embedded in the single `UNKNOWN_STAGE_RESTRICTION` message. Do not relitigate per-segment classification or treating ROLE+STAGE bodies as malformed — both regressions (R22, R25) are closed structurally by the shared token-level model.
- **A malformed clause preserves its non-stage tokens but NEVER applies the partial restriction (R28 f2).** The malformed branch (≥1 stage + ≥1 unknown) routes ALL non-stage tokens (role + unknown) to `extractRoleFlags`, so typo-roles autocorrect and unknown role-prefixes (`RIGGER`) surface as `UNKNOWN_ROLE_TOKEN` (matching origin/main — no role lost). But it does NOT apply the recognized-stage subset as a restriction: an unknown token adjacent to a stage may be a MIS-TYPED STAGE (`Set / Shoo ONLY`), and narrowing to `[Set]` would HIDE the `Show` day — the §9 fail-open contract forbids hiding days. This is the deliberate rebuttal to "apply the valid partial restriction": on the OUTCOME (whole-show) it matches origin/main, and applying the restriction is strictly LESS safe. Do not relitigate applying a partial restriction when any unknown token is present.
- **Marker tolerance is NOT widened, and full-4 compat is NOT dropped (R14 f1 + R17 f1).** For the GENERALIZED grammar the valid ONLY-marker forms are exactly `ONLY` and `ONLY***` — identical to the live `STAGE_TRAILING_MARKER_RE` (`personalization.ts:183-187`); `ONLY*`/`ONLY**` are invalid on PARTIAL subsets (fail-safe to whole-show), and origin/main never narrowed a partial on a starred marker (its partial narrowing used the star-free `LOAD_IN_SET_ONLY_PATTERN`/`LOAD_OUT_STRIKE_ONLY_PATTERN`). SEPARATELY, the EXACT full-4 list keeps origin/main's VERBATIM lenient `FULL_STAGE_ONLY_PATTERN` `ONLY\*{0,3}` (step 2 carve-out) → ANY trailing star count (`ONLY*`…`ONLY****`+) on the full-4 list stays an explicit 4-stage restriction (excludes Show days). The full-4 carve-out sets `cleaned = prefix + tail` (excises only the matched span), so a leading role (`A1`) is kept AND an unknown leading token (`RIGGER`) still emits `UNKNOWN_ROLE_TOKEN` — matching origin/main (R18 f2 + R19 f2). Both directions mirror origin/main exactly; neither the generalized-marker strictness NOR the full-4 lenience is a behavior change. Do not relitigate `***`-count tolerance for partial subsets (deferred); do not relitigate demoting the full-4 lenient star or adding a `(?!\*)` guard (either would EXPOSE Show days — R17 f1 / R19 f3); do not relitigate the full-4 prefix preservation (R19 f2).
- **Corpus-wide rooms no-op is the room contract**, not "looks right." De-literalization must be deep-equal to `origin/main` `rooms` on ALL committed fixtures; a bogus room anywhere fails. Discovery/terminator gate is name-shape + a TRAILING-last-content DAY-range (dims-alone rejected — R30 f1; a DAY-note rejected — R26/R35). This whole-corpus regression is the structural defense that closed the recurring false-positive vector; do not relitigate the dims-alone admit branch.
- **A room needs a NON-EMPTY name after the DAY-range is stripped (R36 f1).** `roomHeaderNameShape` item 2 requires `roomBaseName(firstLine)` to be non-empty, so a pure day-marker row (`DAY 1`, `DAYS 1 & 2`, `DAY 1 & 2`) — which passes item 1's raw shape and `headerDayMarker` but strips to `""` — is rejected before it can enter `roomHeaderLines`. Without this, an empty identity trivially misses every exact-match section token and the day marker becomes a bogus terminator/room. Do not relitigate admitting a name-less day-range row.
- **Section-banner exclusion (item 2) is PREFIX-match for suffix families, EXACT-match for generic tokens, against the DAY-stripped identity (R32 f1 + R33 f1).** Prefix: `GENERAL SESSION`/`BREAKOUT`/`ADDITIONAL`/`LUNCH`/`DETAILS`. Exact: `HOTEL`/`VENUE`/`CREW`/`DATES`/`TRANSPORTATION`/`DOCUMENTS`/`DRESS`/`AGENDA`/`CONTACTS`/`FOYER` + `KNOWN_SECTION_HEADERS`/`KNOWN_SUB_LABELS`. This ADMITS compound DAY-range rooms (`Hotel Ballroom DAY 1 & 2`, `Crew Lounge DAY 1 & 2`) while rejecting bare tokens (`HOTEL DAY 1 & 2`). Do not relitigate a blanket `startsWith` for generic tokens (drops real compound rooms) — it is intentionally narrower than the live `findGsBlockVenueHeader` prefix regex.
- **Room ADMIT = a proper NON-dims-leading NAME on line 1 PLUS a TRAILING `DAY N` day-range that is the cell's LAST content (R21 f1 + R26 f1 + R30 f1 + R35 f1).** DAY-range ONLY — dims are NOT an admit signal (R30 f1). This matches the committed east-coast signatures EXACTLY: `MABEL 1&#10;DAY 1 & 2` (DAY second-line) and the single-line `LAUDERDALE 1, 2, 3 DAY 1 & 2` (DAY inline, commas). The DAY marker is checked on ANY line (LAUDERDALE is single-line, R21 f1) but ONLY as a TRAILING day-range (R26 f1) that is the cell's LAST content — only dims lines may follow it (R35 f1, so a multi-line `SPECIAL DAY 1&#10;NOTES` is rejected). It rejects EVERY non-room shape structurally: notes, single- and multi-line DAY-notes, dims-leading equipment (`4' X 8' RISER`), and name-first assets (`PROJECTION SCREEN&#10;5' x 9'`). The de-literalized loop is a DROP-IN for `mabelRe` — SAME `firstLine`-verbatim name / dims / merge / `seen` / `roomHasContent` emit. Do not relitigate admitting on dims/`&#10;`/field-beneath or `DAY N` on any line (non-trailing OR followed by prose); do not require a second-line-only DAY (would drop inline-DAY LAUDERDALE).
- **`headerDayMarker` requires a TRAILING day-range that is the cell's LAST content — two conditions (R26 f1 + R35 f1).** (i) TRAILING-on-its-line: the DAY line ENDS with the range (`\bDAYS?\s+\d[\d\s&,.\-–—]*$`), so a SINGLE-line DAY-note (`SPECIAL DAY 1 NOTES`, `DAY 1 NOTES` — alphabetic word AFTER the number) is rejected (R26 f1). (ii) LAST-content: every non-empty line AFTER the (last) DAY line is a dims-only line, so a MULTI-LINE note (`SPECIAL DAY 1&#10;NOTES`, `DAY 1&#10;NOTES` — line 1 ends in a range but a prose line follows) is rejected (R35 f1), while a room MAY carry dims after the DAY line (`MERIDIAN&#10;DAY 1 & 2&#10;60' x 45'` admitted). Both admit MABEL's 2nd-line and LAUDERDALE's inline day-range, closing a room-fabrication + field-theft path the corpus no-op alone could not guard (no fixture carries the note shape). Do not relitigate loosening this to `DAY N` on ANY line (a multi-line note would fabricate a room and steal the next field — R35 f1).
- **Room GROUP KEY = base name + NORMALIZED day-range; DISPLAY name stays verbatim (R27 f1 + R34 f1).** origin/main keys on raw `firstLine.toUpperCase()` (safe for two literal names); de-literalization would split an inline-DAY block from its second-line-DAY same-name+day sibling. `roomGroupKey(col0Raw, firstLine)` = `roomBaseName(firstLine)` (trailing inline DAY stripped, uppercased) + `dayRangeOf(col0Raw)` (normalized trailing range from any line) so the inline and second-line forms of the SAME name+day merge — but `SALON ABCD DAY 1` and `SALON ABCD DAY 2` stay SEPARATE (distinct-day blocks must not collapse into one impossible merged room). `buildEmptyRoom` still stores `firstLine` verbatim. No-op on the corpus (MABEL/LAUDERDALE are singletons: keys `MABEL 1 1&2` / `LAUDERDALE 1, 2, 3 1&2`) — verified. Do not relitigate (a) keying on raw `firstLine` (splits inline vs second-line same-day), NOR (b) a bare day-STRIPPED key (collapses distinct days — the R34 corruption), NOR (c) re-splitting the display name (breaks the east-coast no-op).
- **The room-header model and both extractors are LINE-BASED (R24 f1).** `RoomHeaderModel` carries `lines`, `roomHeaderLines` (line indices of all admitted headers), and `groups` (roomGroupKey → candidates); `extractBoBlock`/`extractGsBlock` take a start LINE and the model. This is behavior-identical to the live byte-offset `extractBoBlock(markdown, m.index)` because `mabelRe`'s `m.index` is the row's line-start (`markdown.slice(m.index) === lines.slice(startLine).join("\n")`, verified), so the block text is byte-for-byte the same while the start index is consistent with `roomHeaderLines`. Do not relitigate mixing line indices with byte offsets — the contract is uniformly line-based and the corpus no-op test pins it.
- **No emit-eligibility / field-evidence / run-ownership gate exists (R30 f1 removed it).** Earlier rounds (R12/R23/R28/R29) built a field-evidence window and then a run-ownership pass to keep dims-admitted ASSETS from fabricating rooms / stealing fields. Admitting only on a trailing DAY-range removes the asset from admission entirely, so all of that machinery is gone; emit is `roomHasContent`. Do not relitigate re-introducing an evidence/ownership gate — there is no admitted asset for it to guard against.
- **Same-key repeated-header termination is origin/main-identical and lossless (R15 f1).** Pass 2 extracts EACH admitted member of a key-group with its OWN `extractBoBlock` call and unions the results via `mergeBoFields` — exactly as the live `mabelRe` loop already does for `MABEL 1` dims/fields split blocks (`rooms.ts:826-843`). A same-key sibling header therefore terminates only the PRIOR member's block; the later field-bearing block is extracted independently and merged, so no same-room data is lost. The block's own starting header never self-terminates (the existing positional `k > 0` guard, `extractBoBlock:888-894`, is preserved verbatim). This is not a new terminator model — it is the de-literalized form of the one already shipping; do not relitigate per-member extraction vs a single flat block scan.
