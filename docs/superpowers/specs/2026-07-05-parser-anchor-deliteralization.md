## 2. Part A — Room-header shape predicate

### 2.1 Current brittle behavior (cited)

`rooms.ts` hangs v1 room recognition on two hardcoded venue names, `MABEL` and `LAUDERDALE` (from the single 2024 east-coast fixture), at three sites:

1. **`extractGsBlock` terminator** (`lib/parser/blocks/rooms.ts:687`): `/^\|\s*(GENERAL SESSION|BREAKOUT|ADDITIONAL|LUNCH|MABEL|LAUDERDALE|DETAILS)\b/i` — stops GS bare-field extraction at the next room/section header.
2. **v1 room loop** (`:825`): `const mabelRe = /^\|\s*(MABEL\s+\d[^|]*|LAUDERDALE[^|]*?)\s*\|/gim;` — the ONLY discovery path for a label-less venue-named room block.
3. **`NEXT_ROOM_HEADER_RE`** (`:880-881`): `/^\|\s*(GENERAL\s+SESSION|BREAKOUT|ADDITIONAL\s+ROOM|LUNCH\s+ROOM|MABEL|LAUDERDALE|DETAILS)\b/i` — the block-extraction terminator used by `extractBoBlock` (`:883-898`).

Consequence (audit #6): every new venue has new room names; a bare proper-name GS/breakout header (e.g. `SALON ABCD\n60' x 45'`) at a venue that isn't the east-coast one is **lost or mis-grouped** — no room row, no signal.

Note: shape-based recognition already exists for the *GS* label-less case — `findGsBlockVenueHeader` (`:543-581`) requires "strong evidence" (in-cell `&#10;` newline OR a dims token, plus a section-banner exclusion list). Part A generalizes that same discipline to the v1 breakout-room loop and de-literalizes the three sites above with one shared predicate.

### 2.2 New behavior

Introduce **three** functions with clean single-responsibility contracts (splitting the pure name-shape test from the contextual evidence test — a single-cell predicate cannot see the block beneath, per adversarial-review R1 finding 2). All exported from `lib/parser/blocks/rooms.ts`:

```ts
// (a) PURE — name-shape only, computable from the header cell alone.
export function roomHeaderNameShape(col0Raw: string): boolean
// (b) PURE — a dims token on a NON-FIRST line of the header cell ("NAME&#10;<dims>", e.g.
//     SALON ABCD&#10;60' x 45'). A dims-LEADING single line (equipment "4' X 8' RISER") does NOT
//     qualify — its dims are on the first/name line, and roomHeaderNameShape item 1 rejects it (R20 f1).
function secondLineDims(col0Raw: string): boolean     // a line AFTER the first matches /\d+\s*'\s*x/i
// (b') PURE — a DAY marker ANYWHERE in the cell — inline on the first line OR on a later line. The
//      east-coast rooms carry it BOTH ways: MABEL 1&#10;DAY 1 & 2 (second line) AND the single-line
//      LAUDERDALE 1, 2, 3 DAY 1 & 2 (inline). This is the room signal that admits day-marked headers
//      without dims. A note ("FOH POSITION&#10;Downstage") has no DAY marker (R19 f1); equipment has
//      no DAY marker (R20 f1). The marker must be a TRAILING DAY-RANGE (R26 f1): the line ENDS with
//      the day range, so a note like "DAY 1 NOTES" / "SPECIAL DAY 1 NOTES" (an alphabetic word AFTER
//      the number) does NOT qualify and cannot fabricate a room — closing the DAY-note admit path.
function headerDayMarker(col0Raw: string): boolean    // ANY line matches /\bDAYS?\s+\d[\d\s&,.\-–—]*$/im
//   (trailing day-range only: DAY(S) + digit followed by ONLY digits/space/&/,/./-/en/em-dash to
//    end-of-line. `m` flag → `$` matches per line, so MABEL's 2nd-line "DAY 1 & 2" and LAUDERDALE's
//    inline "… DAY 1 & 2" both qualify; "DAY 1 NOTES" never does. Robust to `&#10;` vs real `\n`:
//    the day-range sits at the string/line end under either encoding — verified.)
// (also: dimsInHeader(col0Raw) — /\d+\s*'\s*x/i across ANY line — is retained for splitRoomHeader
//  VALUE extraction only, NOT for admit.)
// (c) FIELD-ROW predicate — is this single table row a recognized breakout room field? (the exact
//      applyBoLabel set; floor/dims excluded — R23 f1). Used by the run-ownership pass (Pass-0) to
//      decide emit-eligibility; a room field is NOT an admit signal (R18 f1: field-beneath is too weak).
function isRoomFieldRow(rowText: string): boolean
// (d) ADMIT predicate — a proper NAME on line 1 (not dims-leading) PLUS a TRAILING DAY-range on any line OR
//     second-line dims. Independent of fields beneath. Admits MABEL/LAUDERDALE (DAY) + novel dims
//     headers (SALON); excludes single-line notes (R18 f1), multi-line notes (R19 f1), AND
//     dims-leading equipment (R20 f1).
export function isRoomHeaderShape(col0Raw: string): boolean
//   = roomHeaderNameShape(col0Raw) && (headerDayMarker(col0Raw) || secondLineDims(col0Raw))
// (d') MERGE KEY — separates room identity from display name (adversarial-review R27 f1). Strips a
//      TRAILING inline DAY-range from firstLine (same shape as headerDayMarker), then uppercases, so an
//      inline-DAY field-bearing block (`SALON ABCD DAY 1 & 2`) groups with its dims-only same-name
//      sibling (`SALON ABCD&#10;60' x 45'`) instead of splitting and dropping the dims. DISPLAY name
//      stays firstLine VERBATIM. No-op on the corpus (MABEL has no inline DAY; LAUDERDALE is one block).
function roomMergeKey(firstLine: string): string
//   = firstLine.replace(/\s*\bDAYS?\s+\d[\d\s&,.\-–—]*$/i, "").trim().toUpperCase()
// (e) TOP-LEVEL MODEL — computed ONCE before parseGsRoom/parseBoRooms (R17 f2); `seen`-independent.
//     LINE-BASED (adversarial-review R24 f1): all indices are indices into the shared `lines` array
//     (`markdown.split("\n")`), NOT byte offsets. extractBoBlock is switched from a byte `startOffset`
//     to a start LINE index — behavior-identical because the live `mabelRe` (a /^…/gim match) has
//     `m.index` at the row's line START, so `markdown.slice(m.index) === lines.slice(startLine).join("\n")`
//     (verified). This eliminates the line-index-vs-byte-offset mismatch.
export type RoomCandidate = { key: string; fieldBearing: boolean; lineIndex: number };
export type RoomHeaderModel = {
  lines: string[];                          // markdown.split("\n"), shared by all consumers
  roomHeaderLines: ReadonlySet<number>;     // absolute LINE indices of emit-eligible headers
  emitEligibleGroups: Map<string, RoomCandidate[]>;  // key → its admitted candidates (for Pass 2)
};
export function computeRoomHeaderModel(markdown: string): RoomHeaderModel
//   split into `lines`; scan every col0 row → admit (isRoomHeaderShape) → record {key, fieldBearing,
//   lineIndex} → group by key → emit-eligible (≥1 fieldBearing) → roomHeaderLines = the lineIndex set
//   of candidates in emit-eligible groups. Threaded into parseGsRoom→extractGsBlock(lines, startLine,
//   model) and parseBoRooms→extractBoBlock(lines, startLine, model) as the shared terminator set.
```

**Design note — comprehensive re-analysis of the room-discovery false-positive vector (adversarial-review R2 f1, R3 f2, R4 f1, R6 f1, R7 f1, R10 f1, R11 f1, R12 f1, R15 f1, R17 f2, R18 f1, R19 f1, R20 f1, R21 f1, R23 f1, R24 f1, R26 f1, R27 f1, R28 f1, R29 f1 — a recurring vector; per the AGENTS.md "comprehensive re-analysis + structural defense" rule the model is now anchored end-to-end against the LIVE extraction code AND the committed east-coast fixture bytes, not point-patched: admit/emit/terminate are three distinct gates; ADMIT requires a proper NON-dims-leading NAME on line 1 PLUS a TRAILING `DAY N` day-range on any line (LAUDERDALE is single-line, R21 f1; trailing-only so a DAY-note can't fabricate, R26 f1) OR a DIMS token on a NON-FIRST line — matching the committed `MABEL 1&#10;DAY 1 & 2` / `LAUDERDALE 1, 2, 3 DAY 1 & 2` / `NAME&#10;DIMS` shapes — so NONE of field-beneath (R18 f1), a bare multi-line note (R19 f1), a dims-leading equipment row (R20 f1), OR a DAY-note (R26 f1) can fabricate a room; the loop is a DROP-IN for `mabelRe` (same name/dims/merge) so the no-op holds by construction; and the corpus-wide no-op test is the mechanism-agnostic guard).** The literal `MABEL|LAUDERDALE` loop was safe because it matched two specific names; a shape scan is inherently more permissive. The resolution is a **two-stage admit-then-emit** model that separates "worth extracting a block for" from "actually a room":

1. **ADMIT** a candidate iff `roomHeaderNameShape && (headerDayMarker || secondLineDims)` — a proper NAME on line 1 (NOT dims-leading, item 1) PLUS a TRAILING `DAY N` day-range (a line ending in the range) ANYWHERE in the cell OR a DIMS token on a NON-FIRST line. This matches the exact signatures of the committed east-coast rooms (verified against `2024-05-east-coast-family-office.md`): `MABEL 1&#10;DAY 1 & 2` (`:26`, DAY on the SECOND line), the SINGLE-LINE `LAUDERDALE 1, 2, 3 DAY 1 & 2` (`:35`, DAY INLINE, commas in the name), and novel `NAME&#10;DIMS` headers (`SALON ABCD&#10;60' x 45'`). The DAY marker is checked on ANY line (not second-line-only) precisely because LAUDERDALE carries it inline (adversarial-review R21 f1), but ONLY as a TRAILING day-range (R26 f1) so an in-room DAY-NOTE cannot masquerade as a header; dims stay second-line-only to keep dims-leading equipment out. **None of field-evidence-beneath, bare multi-line, a dims-LEADING line, OR a DAY-NOTE is an admit signal (adversarial-review R18 f1 + R19 f1 + R20 f1 + R26 f1).** Field-beneath fabricates a room from an in-room note/equipment row before a real field; "any `&#10;`" fabricates one from a multi-line note (`FOH POSITION&#10;Downstage`); "any dims token" fabricates one from a dims-leading equipment row (`4' X 8' RISER`); "`DAY N` anywhere" fabricates one from a DAY-note (`SPECIAL DAY 1 NOTES`, `DAY 1 NOTES`) that has an alphabetic word AFTER the number. The predicate excludes ALL FOUR: a plain note has no `DAY N` and no second-line dims; an equipment row is dims-leading (item 1) with no `DAY N`; a DAY-note's day marker is NOT trailing (a word follows the number) so `headerDayMarker` rejects it. This ADMITS `MABEL 1`/`LAUDERDALE …` (trailing DAY-range), the dims-only-first block (`SALON ABCD&#10;60' x 45'`, second-line dims — dims captured for merge, R7 f1); it does NOT admit a single-line note, a multi-line note, a dims-leading equipment row, or a DAY-note.

   **De-literalization is a DROP-IN for the `mabelRe` loop (`rooms.ts:823-847`), guaranteeing the corpus no-op for MABEL/LAUDERDALE by construction.** The ONLY things that change are (i) row SELECTION: `mabelRe` (matches `MABEL \d…|LAUDERDALE…`) → `isRoomHeaderShape`, and (ii) the merge KEY derivation (R27 f1, below — a corpus no-op). Everything else is preserved VERBATIM: the room DISPLAY name is `firstLine` of the header cell as-is (`buildEmptyRoom("breakout", firstLine)` — so `MABEL 1` and the full `LAUDERDALE 1, 2, 3 DAY 1 & 2` are stored EXACTLY as origin/main stores them, NOT re-split), dims ride from `rawHeader.split("\n").slice(1)`, `mergeBoFields(extractBoBlock(...))`, and the `seen` dedup. **Merge KEY (R27 f1 — separate identity from display):** origin/main keys candidates on `firstLine.toUpperCase()` (`rooms.ts:829`), which bakes an INLINE DAY-range into the key — harmless for the two literal names, but under de-literalization it would split a dims-only second-line sibling (`SALON ABCD&#10;60' x 45'`, key `SALON ABCD`) from a later inline-DAY field-bearing block (`SALON ABCD DAY 1 & 2`, key `SALON ABCD DAY 1 & 2`), dropping the dims. So the candidate key is `roomMergeKey(firstLine)` = `firstLine` with a TRAILING inline DAY-range stripped (`/\s*\bDAYS?\s+\d[\d\s&,.\-–—]*$/i` — the same trailing-day-range shape as `headerDayMarker`), then uppercased; the DISPLAY name stays `firstLine` VERBATIM. This is a **no-op on the corpus** (verified): `MABEL 1` has no inline DAY (its DAY is on line 2, already excluded from `firstLine`) → key unchanged; `LAUDERDALE 1, 2, 3 DAY 1 & 2` is a SINGLE block → its group has one member, so keying by `LAUDERDALE 1, 2, 3` vs the full string yields the identical single room with the identical verbatim display name. Because `isRoomHeaderShape` admits exactly `{MABEL 1, LAUDERDALE …}` among the east-coast rows (the corpus no-op test pins this) and processes them through the identical name/dims/merge path, the parsed `rooms` are byte-identical for existing fixtures; only NOVEL non-corpus shapes add or (correctly) merge rooms.
2. **Group** admitted candidates by header key and merge dims + fields across the group (the existing `mergeBoFields` merge-by-key logic, `:823-847`, UNCHANGED).
3. **EMIT** a room for a key-group iff **≥1 candidate in that group is a field-bearing RUN OWNER** (the run-ownership scan: a real room field is attributed to the EARLIEST admitted header in its run — R28 f1 far-below field counts, R29 f1 a subordinate asset can't steal it). Dims-leading equipment is already excluded at ADMIT (item 1 + second-line rule), so the emit gate's remaining job is to drop (a) a LONE `NAME&#10;dims`/`NAME&#10;DAY` header whose run has NO room field and no field-bearing same-key sibling — e.g. a gear entry `PROJECTION SCREEN&#10;5' x 9'` whose run is all gear — and (b) a SUBORDINATE asset that shares a run with a real room above it (the field belongs to the room, R29 f1). `roomHasContent` alone is INSUFFICIENT because it counts `dimensions` as content (`rooms.ts:424-441`), so such a dims-bearing block would pass it — the emit gate requires a real *field*-bearing block in the group. A `SALON ABCD&#10;60' x 45'` dims-only block still emits BECAUSE its same-key day sibling is field-bearing (R7 f1).

Structural defense shipped this round: the **corpus-wide rooms-array no-op test** (§8) — de-literalization MUST be deep-equal to `origin/main` `rooms` on all 15 committed fixtures — is the mechanism-agnostic guard that catches ANY residual false positive OR dropped dimension across the whole corpus.

**`roomHeaderNameShape(col0Raw)` (pure) returns true iff ALL hold:**

1. **Non-empty, label-less proper name — CASE-INSENSITIVE (adversarial-review R10 finding 1).** After flattening `&#10;`→newline, taking the first line, trimmed, and **uppercasing**, it matches the proper-name shape `^[A-Z0-9][A-Z0-9 &',./-]*$` (letters/digits/spaces + a small punctuation set INCLUDING the comma — the committed east-coast `LAUDERDALE 1, 2, 3 DAY 1 & 2` single-line header carries commas, adversarial-review R21 f1; at least one letter) **AND is NOT dimension-leading — it does NOT match `^\d+\s*'\s*x`i (adversarial-review R20 f1).** An equipment/asset row leads with its dimensions (`4' X 8' RISER`, `5' X 9' WIDESCREEN…`); a real room NAME does not (`SALON ABCD`, `MABEL 1`, `Grand Foyer`). Rejecting a dims-leading first line excludes equipment at the name-shape gate. (Names that merely start with a digit but are not dims — `5TH AVENUE BALLROOM` — do NOT match `^\d+\s*'\s*x` and are unaffected.) The match is case-INSENSITIVE (the literal `mabelRe` it replaces used `/gim`), so a mixed-case novel venue name like `Salon ABCD` / `Grand Foyer` IS admitted — that is the feature's whole point. Mixed-case *field labels* (`Breakout Room Setup Date / Time`) are NOT excluded by casing; they are excluded by item 2 (`BREAKOUT`/`ADDITIONAL`/… banner) and item 3 (field-label), both of which uppercase before comparing. The header-cell-evidence admit gate (a dims token OR a `DAY N` marker) + the emit gate (§2.2) are the real discriminator for anything else; the corpus no-op test pins that no mixed-case non-room row in the 15 fixtures becomes a room.
2. **Not a known section/room banner.** The uppercased first line does NOT start with any structural keyword already handled by a dedicated path or that denotes a section: `GENERAL SESSION`, `BREAKOUT`, `ADDITIONAL`, `LUNCH`, `DETAILS`, plus the section banners `findGsBlockVenueHeader` already excludes (`:565-571`: `DOCUMENTS`, `DATES`, `CREW`, `DRESS`, `TRANSPORTATION`, `HOTEL`, `VENUE`, `AGENDA`, `CONTACTS`). Also NOT in `KNOWN_SECTION_HEADERS` / `KNOWN_SUB_LABELS` (`lib/parser/knownSections.ts:34,96`) and NOT a `GS <label>` field row (`^GS\s`).
3. **Not a room-FIELD label** (adversarial-review R4 finding 1). A sheet with UPPERCASE field labels (`BO SETUP`, `AUDIO`, `SET TIME`) — if any carried a dims token or `&#10;` — would otherwise satisfy the name-shape + header-cell-evidence gate and fabricate a room named after a field label — a regression the literal MABEL/LAUDERDALE loop cannot cause. Reject the row if its uppercased first line — after stripping an optional leading `BO ` or `GS ` prefix — is one of the room-field labels that `applyBoFields`/`applyGsLabel` recognize (`rooms.ts:695-707` + the `applyBoFields` label set): `SETUP`, `SET TIME`, `SHOW TIME`, `STRIKE TIME`, `AUDIO`, `VIDEO`, `SCENIC`, `LIGHTING`, `LED`, `POWER`, `OTHER`, `DIMENSIONS`, `FLOOR`, `DIGITAL SIGNAGE`, `NAME(S)`, `NOTES`. The exclusion list is DERIVED from the live `applyBoFields`/`applyGsLabel` code (not hardcoded in the spec) so it cannot drift; the plan pins it to a shared constant.

**Emit-eligibility is computed by a single-pass RUN-OWNERSHIP scan (adversarial-review R28 f1 + R29 f1)**, resolving BOTH "a real field can sit a few rows below the header" (R28 f1) AND "an admitted asset must not steal the next room's field" (R29 f1). Walk the admitted-candidate lines and the room-field rows top-to-bottom, tracking the current run's OWNER:
- An admitted header **STARTS a new run** (and becomes its OWNER) iff a room field has been seen SINCE the current owner; otherwise it **JOINS** the current run as a SUBORDINATE (owner unchanged). So two consecutive `NAME&#10;dims` headers with no field between them (`SALON ABCD` then the asset `PROJECTION SCREEN&#10;5' x 9'`) share ONE run owned by the FIRST.
- A room field (`isRoomFieldRow`) is attributed to the current run's OWNER → that owner is **field-bearing**. The owner's field scan therefore reaches ACROSS subordinate asset rows and non-field template rows (so a novel room's field a few rows down still counts — R28 f1), while a subordinate asset NEVER becomes field-bearing and NEVER emits (R29 f1 — it cannot steal the field that belongs to the earliest header in its run).
- A STRUCTURAL keyword (`GENERAL SESSION|BREAKOUT|ADDITIONAL ROOM|LUNCH ROOM|DETAILS`) ends the current run.

A key-group is EMIT-ELIGIBLE iff ≥1 of its candidates is a field-bearing OWNER. Because a field is seen immediately after MABEL's header, `LAUDERDALE` (later) STARTS a fresh run and owns its own fields — so the corpus emits BOTH (no-op preserved, verified). The recognized-field set (`isRoomFieldRow`) is **the exact set of labels the live ROW parsers `applyBoLabel`/`applyBoFields`(plain) map into a room field** (`rooms.ts:900-918,920-934`) — matched as `BO <field>` or bare label rows exactly as they do: `setup`/`set time`/`show time`/`strike time`/`audio`/`video`/`scenic`/`lighting`/`led`/`power`/`other`/**`digital signage`**/**`notes`**. **`floor` AND `dimensions` are EXCLUDED (adversarial-review R23 f1 + R12 f1)** — NEITHER is parsed from a ROW: both come only from the HEADER cell via `splitRoomHeader` (`floor` is stripped as a name-template word at `rooms.ts:418,1059`; `dims` is the equipment/asset signature). Counting a `Floor`/`Dimensions` label row as a field would let a `NAME&#10;dims` candidate followed by a header-only `Floor` label become field-bearing without a real captured field — a fabricated-room path. Defining the field set as the ACTUAL `applyBoLabel` label set (which already omits `floor`/`dimensions`) makes it drift-proof AND correct: a novel room whose only corroboration is `Digital Signage`/`Notes` still emits (row-parsed), but a floor-only/dims-only run does NOT. The plan pins the field set to a single shared constant sourced from `applyBoLabel`.

The admit pass runs FIRST (collecting the admitted-candidate lines), THEN the run-ownership scan walks those lines interleaved with the room-field rows to mark each field-bearing OWNER — a two-sub-pass structure inside `computeRoomHeaderModel`, non-circular (ownership never depends on emit-eligibility, only on admit order + field rows).

**Uses — THREE separate gates (adversarial-review R11 finding 1: discovery-admit ≠ emit ≠ terminate must NOT share one predicate):**

- **Pass 0 — `computeRoomHeaderModel(markdown)` (top-level, computed ONCE before any block parse; adversarial-review R17 finding 2).** The room-header model is built at the TOP of `collectV2V1Rooms` (`rooms.ts:215`), BEFORE `parseGsRoom(markdown)` (`:218`) and `parseBoRooms(markdown)` (`:220`), and is **INDEPENDENT of the BREAKOUT/LUNCH `seen` state** (which does not yet exist at `parseGsRoom` time — the cyclicity R17 f2 flags). It scans every table row's col0 and ADMITs iff `isRoomHeaderShape(rawHeader)` (= `roomHeaderNameShape && (headerDayMarker || secondLineDims)` — a proper non-dims-leading NAME on line 1 plus a TRAILING `DAY N` day-range on any line OR a dims token on a NON-FIRST line; NOT field-beneath, NOT bare multi-line, NOT dims-leading equipment, NOT a DAY-note — R18 f1 + R19 f1 + R20 f1 + R21 f1 + R26 f1). Because `roomHeaderNameShape` item 2 already rejects the `GENERAL SESSION`/`BREAKOUT`/`ADDITIONAL`/`LUNCH`/`DETAILS` banners, those structural rows are never admitted here — so the model needs NO `seen` input to avoid them. It splits `markdown` into `lines` ONCE, then runs TWO sub-passes: (0a) scan every col0 row and ADMIT each `isRoomHeaderShape` candidate, recording `{key = roomMergeKey(firstLine), displayName = firstLine, lineIndex = i}` (`roomMergeKey` strips a TRAILING inline DAY-range then uppercases, so an inline-DAY block groups with its dims-only same-name sibling — R27 f1; `displayName` is `firstLine` VERBATIM for `buildEmptyRoom`; `lineIndex` is a LINE index into `lines`, not a byte offset — R24 f1); (0b) the RUN-OWNERSHIP scan (above) walks the admitted lines interleaved with `isRoomFieldRow` rows to mark each field-bearing OWNER (R28 f1 + R29 f1 — a field belongs to the EARLIEST admitted header in its run, so a far-below field still counts but a subordinate asset can't steal it). Then group by key, mark a key **EMIT-ELIGIBLE iff ≥1 of its candidates is a field-bearing OWNER**, and returns `{ lines, roomHeaderLines, emitEligibleGroups }` where `roomHeaderLines` = the set of `lineIndex` of candidates in emit-eligible key-groups. A dims-LEADING equipment row (`5' X 9' WIDESCREEN…`, `4' X 8' RISER`) is NEVER admitted — its dims are on the first/name line, so `roomHeaderNameShape` item 1 rejects it and there is no second-line evidence (R20 f1) — even when a real field follows it; so it can neither emit nor terminate. An in-room note — single-line (`SPECIAL NOTE`), multi-line (`FOH POSITION&#10;Downstage`), OR a DAY-note (`SPECIAL DAY 1 NOTES`, `DAY 1 NOTES`, R26 f1) — is likewise NEVER admitted (no TRAILING `DAY N` day-range on any line and no second-line dims; a DAY-note's marker is non-trailing because an alphabetic word follows the number), so it can neither emit nor terminate (R18 f1 + R19 f1 + R26 f1). A LONE `NAME&#10;dims`/`NAME&#10;DAY` header with no field-bearing group member (a gear-list entry) is admitted but NOT emit-eligible → its line is NOT in `ROOM_HEADER_LINES`. This one model is THREADED as a parameter into `parseGsRoom` → `extractGsBlock` AND `parseBoRooms` → `extractBoBlock` (signature change: both take the precomputed model), so GS and BO extraction share ONE terminator set regardless of call order.
- **Pass 1 — GS + BO extraction consume the shared model.** `parseGsRoom` (`:218`, runs FIRST) and `parseBoRooms` (`:220`) both receive the Pass-0 model. The BREAKOUT/LUNCH `seen` dedup stays exactly where it is in `parseBoRooms` (`:807,830`) — it governs which rooms are EMITTED/deduped AFTER extraction, NOT termination. Duplicate suppression is therefore decoupled from the terminator prepass (R17 f2): the terminator set is purely structural line indices, computed once, order-independent.
- **Pass 2 — extract EACH emit-eligible member block separately, then merge + emit.** For each emit-eligible key-group, extract EVERY member block INDEPENDENTLY via `extractBoBlock(model.lines, candidate.lineIndex, model)` — one call per admitted candidate, starting at the candidate's LINE (R24 f1). This is behavior-identical to the live `mabelRe` loop's `extractBoBlock(markdown, m.index)` (`rooms.ts:826-843`) because `m.index` is the row's line-start, so `markdown.slice(m.index) === lines.slice(candidate.lineIndex).join("\n")` (verified) — the block text is byte-for-byte the same, but the start is now expressed as a line index consistent with `roomHeaderLines`. Each block is bounded by the terminator (below), then `mergeBoFields` the per-block results across the group (UNCHANGED, `:854-876`) and emit ONE room. Because each same-key member is extracted independently and then UNIONed, a same-key sibling header acting as the terminator of a PRIOR member block is CORRECT and lossless — the later field-bearing block is extracted by its OWN `extractBoBlock` call and its fields are merged in. `MABEL 1&#10;DAY 1 & 2` / `LAUDERDALE …` are field-bearing → emitted; a `SALON ABCD&#10;60' x 45'` dims-only block (its dims captured by `splitRoomHeader`/`mergeBoFields` on its own extraction) merges into its same-key day-block group and is emitted with dims preserved (R7 f1); the equipment row is DROPPED (R6 f1); an all-caps note is never admitted (R3 f2).
- **Terminator (LINE-BASED, R24 f1)** — `extractGsBlock(lines, gsStartLine, model)` (`:670-693`) and `NEXT_ROOM_HEADER_RE` → `extractBoBlock(lines, startLine, model)` (`:880-898`), BOTH now consulting the Pass-0 `roomHeaderLines`. extractBoBlock walks `lines` from `startLine`; at relative position `k` the ABSOLUTE line index is `startLine + k`. A line terminates the block iff it is a structural keyword (`GENERAL SESSION|BREAKOUT|ADDITIONAL ROOM|LUNCH ROOM|DETAILS`) OR its ABSOLUTE index ∈ `roomHeaderLines` (an EMIT-ELIGIBLE room header — NOT merely admitted) — **EXCEPT the block's own starting line (`k === 0`, absolute `startLine`) is NEVER a terminator, preserving the existing positional `k > 0` guard verbatim (`extractBoBlock:888-894`: "Stop at the next room/section header — but not the current block's own header on line 0").** Concretely: the line at `k=0` (its own header, whose index is of course in `roomHeaderLines`) does NOT self-terminate; every LATER line whose absolute index (`startLine + k`) ∈ `roomHeaderLines` (or is a structural keyword) does. Three properties this yields: (1) R11-f1 — a dims-only equipment row is admitted but NOT emit-eligible, so its index is NOT in `ROOM_HEADER_LINES` and it NEVER severs a preceding room's field extraction; (2) R15-f1 — a same-key repeated header (dims-first + fields-later) terminates the PRIOR sibling block but, because Pass 2 extracts each member independently and `mergeBoFields` unions them, NO same-room data is lost — identical to origin/main's `MABEL 1` dims/fields split-then-merge behavior; (3) R17-f2 — because the model is precomputed before `parseGsRoom`, a novel proper-name room AFTER the GS block now terminates `extractGsBlock` (its emit-eligible index is in the shared `ROOM_HEADER_LINES`) instead of bleeding into the GS room. Same-key members terminate each other's PRIOR block (correct, lossless via merge); a member never self-terminates on its own header line.

The **corpus-wide rooms no-op test** (§8) plus explicit **negative terminator, negative discovery, equipment-before-fields (no-truncation), and repeated-block tests** pin this — the highest-risk area in the PR, bounded by the three-gate split and a whole-corpus regression, not asserted blind.

**Known limitation (R11 finding 2, backlog):** a room whose name is EXACTLY a `KNOWN_SECTION_HEADERS` token — in practice only `FOYER` (`knownSections.ts:64`) is both a registered section token AND a plausible bare room name — is NOT admitted (item 2 excludes it; a bare `Foyer` is genuinely ambiguous with the section token). COMPOUND names are unaffected: `Grand Foyer` / `East Foyer` do not equal `FOYER` and ARE admitted when their header cell carries dims or a `&#10;`. Rather than ship a venue-token carve-out list (the exact literal-list brittleness this PR removes), bare `Foyer`-as-room is documented as unsupported and filed as a backlog item; a test asserts `Grand Foyer` works and bare `Foyer` is (currently) not a room.

**In-room notes AND equipment rows are structurally closed (R18 f1 + R19 f1 + R20 f1 + R21 f1 + R26 f1), NOT backlogged.** Admit requires a proper NON-dims-leading NAME on line 1 PLUS a TRAILING `DAY N` day-range on any line OR a dims token on a NON-FIRST line. A single-line note (`SPECIAL NOTE`) has no DAY and no second-line dims; a multi-line note (`FOH POSITION&#10;Downstage`) has neither; a dims-leading equipment row (`4' X 8' RISER`, `5' X 9' …`) fails item 1 (dims-leading name) and has no DAY; a DAY-note (`SPECIAL DAY 1 NOTES`) has a non-trailing DAY marker (a word follows the number) so `headerDayMarker` rejects it (R26 f1). All are rejected at admit, statelessly, without an "is-inside-an-open-room" positional tracker (added parser state this PR deliberately avoids). The only residual would be a row whose FIRST line is a proper non-dims-leading name AND which carries a TRAILING `DAY N` day-range or a second-line `NN' x NN'` dims token AND which sits immediately before a real field — a shape indistinguishable from a room header by construction and absent from the corpus (the corpus-wide rooms no-op test is the backstop). Regression tests cover single-line-note-then-field (R18 f1), multi-line-note-then-field (R19 f1), dims-leading-equipment-then-field (R20 f1), and the single-line inline-DAY LAUDERDALE header (R21 f1).

## 3. Part B — Stage-restriction token grammar

### 3.1 Current brittle behavior (cited)

`extractStageRestriction` (`lib/parser/personalization.ts:158-169`) recognizes exactly three hand-picked phrasings via three regexes:

- `FULL_STAGE_ONLY_PATTERN` (`:53-54`) — `Load In / Set / Strike / Load Out ONLY` → all 4 physical stages.
- `LOAD_IN_SET_ONLY_PATTERN` (`:55`) — anchored `^- Load In / Set ONLY$` → `[Load In, Set]`.
- `LOAD_OUT_STRIKE_ONLY_PATTERN` (`:56`) — anchored `^- Load Out / Strike ONLY$` → `[Load Out, Strike]`.

`extractRoleFlags` (`:267-370`) DUPLICATES the same three patterns to strip the stage prefix before tokenizing role flags (`:279-293`).

Consequence (audit #7): a reordered or novel-but-valid subset — `Set / Strike ONLY`, `Load Out / Strike ONLY` (note: only `Load In/Set` and `Load Out/Strike` orderings are hardcoded), `Set / Show ONLY` — falls through `extractStageRestriction` to `{kind:"none"}` AND is then mis-tokenized by `extractRoleFlags` into an `UNKNOWN_ROLE_TOKEN` cascade (e.g. `Set / Strike ONLY` → tokens `SET`, `STRIKE ONLY` → two spurious unknown-role warnings), and the crew member sees the **whole show** (a valid restriction silently ignored). The grammar recognizes these reordered/subset stage clauses; a malformed one that mixes a real stage with an unreadable token (`Set / Rehearsal ONLY`) gets a precise `UNKNOWN_STAGE_RESTRICTION` instead of the confusing multi-`UNKNOWN_ROLE_TOKEN` cascade. (A clause with NO recognized stage — `Rehearsal ONLY`, `RIGGER ONLY` — is a role clause and keeps its existing `UNKNOWN_ROLE_TOKEN` behavior; §3.2.)

### 3.2 New behavior — one shared `parseStageClause`

Introduce one order-independent grammar function, shared by both consumers (kills the 3-pattern duplication):

```ts
// lib/parser/personalization.ts
const STAGE_RESTRICTION_VOCAB = ["LOAD IN", "SET", "SHOW", "STRIKE", "LOAD OUT"] as const;
//                                                  ^^^^ NEW: Show added (already a valid WorkPhase)

export type StageClause = {
  /** Recognized stage tokens in appearance order, deduped. Empty if none recognized. */
  stages: Array<"Load In" | "Set" | "Show" | "Strike" | "Load Out">;
  /** True iff a VALID trailing restriction marker was present — exactly bare `ONLY`
   *  or `ONLY***` (three stars). A trailing 1–2 stars (`ONLY*`/`ONLY**`) is NOT a
   *  valid marker (mirrors live `STAGE_TRAILING_MARKER_RE`, personalization.ts:183-187). */
  hasOnly: boolean;
  /** The role cell with the leading stage clause + ONLY marker removed, for role-flag tokenizing. */
  cleaned: string;
  /** True iff the cell had an ONLY-clause that looked like a restriction but no stage resolved. */
  unrecognizedRestriction: boolean;
  warnings: ParseWarning[];
};
export function parseStageClause(roleCell: string): StageClause
```

**Grammar (evaluated on the role cell, after day-restriction extraction already ran upstream — see `crew.ts:299-304`):**

1. Strip an optional leading `-` and whitespace (preserved for `cleaned` reconstruction).
2. **Full-4-list lenient-star carve-out FIRST (backward-compat, adversarial-review R17 finding 1).** Before the strict marker search, test the retained live `FULL_STAGE_ONLY_PATTERN` — `/Load\s+In\s*\/\s*Set\s*\/\s*Strike\s*\/\s*Load\s+Out\s+ONLY\*{0,3}/i` (`personalization.ts:53-54`, the EXACT-ORDER full physical-4 list with a LENIENT `ONLY\*{0,3}` marker). The pattern is the VERBATIM live `FULL_STAGE_ONLY_PATTERN` (UNANCHORED, `ONLY\*{0,3}`), so it (a) can match a full-stage substring AFTER a leading role prefix, and (b) accepts ANY trailing star count including `****` — because `\*{0,3}` matches the `***` prefix of `****` (verified). Both are DELIBERATE origin/main mirroring (see the star-count note below). If it matches → `stages = [Load In, Set, Strike, Load Out]`, `hasOnly = true`, `unrecognizedRestriction = false`, `consumedOnlyClause = true`. **`cleaned` PRESERVES the ENTIRE prefix (adversarial-review R18 f2 + R19 f2):** let `prefix` = text BEFORE the matched span, `tail` = text after it; **`cleaned = prefix + tail`** — the matched full-stage-ONLY span is EXCISED and everything else is preserved VERBATIM (NOT segment-filtered). This flows ALL prefix segments to `extractRoleFlags`: recognized roles are kept AND unrecognized tokens still emit `UNKNOWN_ROLE_TOKEN`, exactly as origin/main (whose anchored `extractRoleFlags` strip left the whole prefix intact while `FULL_STAGE_ONLY_PATTERN` produced the 4-stage restriction). Examples: `A1 / Load In / … ONLY*` → `cleaned = "A1 / "` → role `A1`; `RIGGER / Load In / … ONLY` → `cleaned = "RIGGER / "` → `UNKNOWN_ROLE_TOKEN(RIGGER)` PRESERVED (R19 f2); `A1 / RIGGER / Load In / … ONLY` → `cleaned = "A1 / RIGGER / "` → `A1` + `UNKNOWN_ROLE_TOKEN(RIGGER)`; pure `Load In / … ONLY*` (no prefix) → `cleaned = tail`. **Star-count note (R17 f1 + R19 f3 — anti-expansion, do NOT add `(?!\*)`):** the full-4 carve-out intentionally accepts `ONLY*` / `ONLY**` / `ONLY***` / `ONLY****`+ as an explicit `[Load In,Set,Strike,Load Out]` restriction, mirroring origin/main's `FULL_STAGE_ONLY_PATTERN` EXACTLY. A full-4 restriction EXCLUDES pure `Show` days (now a distinct WorkPhase); demoting ANY star count to whole-show would EXPOSE Show days — the R17 f1 regression. So the marker-enumeration "4+ stars invalid → whole-show" rule governs the GENERALIZED grammar (partials) ONLY; the full-4 compat path is deliberately lenient for every star count. The lenient-star tolerance is scoped to THIS exact full-4 pattern; it does NOT generalize to subsets.
3. **Otherwise, find the STRICT ONLY marker.** Locate the first VALID marker `\bONLY\b(?:\s*\*{3})?(?!\s*\*)`. **A marker is valid ONLY as bare `ONLY` or `ONLY***` (exactly three stars, optional whitespace before them); a trailing 1–2 stars — `ONLY*` / `ONLY**` — is NOT a valid marker (adversarial-review R14 finding 1).** The negative lookahead `(?!\s*\*)` rejects any ONLY-token followed by a non-triple star run, mirroring the live `STAGE_TRAILING_MARKER_RE` (`personalization.ts:183-187`, which peels only `ONLY` / `ONLY***` / bare `***` and deliberately leaves `ONLY*`/`ONLY**` untouched — `***`-count tolerance is a deferred concern). If NO valid marker (ABSENT, including a malformed non-full-4 `ONLY*`/`ONLY**`) → `hasOnly = false`, `stages = []`, `unrecognizedRestriction = false`, `cleaned = roleCell` UNCHANGED, return. This is fail-safe: a malformed PARTIAL subset like `Set / Strike ONLY*` is NEVER consumed as a restriction (so it can never narrow — origin/main never narrowed a partial on a starred marker either, since its partial-subset narrowing used the star-free `LOAD_IN_SET_ONLY_PATTERN`/`LOAD_OUT_STRIKE_ONLY_PATTERN`), and `extractRoleFlags` then emits `UNKNOWN_ROLE_TOKEN` on the garbled tokens (whole-show + surfaced signal, never silent — adversarial-review R2 finding 2). Descriptive full lists w/o ONLY are handled by the retained `FULL_STAGE_PATTERN` strip in `extractRoleFlags`.
4. **ONLY present (non-full-4).** Let `body` = text between the leading dash and the ONLY marker; `tail` = text after the marker. **Comprehensive re-analysis of the stage-clause classification vector (adversarial-review R1 f1, R3 f1, R5 f1, R8 f1, R14 f1, R16 f1, R22 f1, R25 f1 — a recurring vector; per the AGENTS.md same-vector rule the model is now a TOTAL per-TOKEN classification anchored on the live tokenizer, not per-segment).** **Tokenize `body` EXACTLY as `extractRoleFlags` does, then classify each atomic token — NOT each `/`-segment (R25 f1).** The prior per-`/`-segment model mis-handled a hyphen-MIXED segment like `Set - LEAD`: it saw one recognized role (`LEAD`) and labeled the WHOLE segment ROLE, silently swallowing the embedded `Set` STAGE. Token-level classification extracts BOTH. Tokenization (single shared helper, pinned to the SAME constants `extractRoleFlags` uses so it cannot drift):
   1. **Extract MULTI_WORD_TOKENS FIRST** (e.g. `SHOW CALLER`) from `body`, recording each as a ROLE token and EXCISING it from the remaining text — so multi-word roles are matched before any split and can NEVER decompose into a stray `SHOW` STAGE token (preserves R1 f1: `SHOW CALLER` ≠ `SHOW`). The physical two-word STAGES `LOAD IN`/`LOAD OUT` contain a SPACE but no `/` or `-`, so they survive the split below intact and are recognized as single STAGE tokens.
   2. **Split the remaining text on `/` AND `-`** into atomic tokens (trimmed, non-empty). Classify each token into exactly one kind:
      - **STAGE** — trimmed-uppercase EXACTLY equals a `STAGE_RESTRICTION_VOCAB` member (`Load In`/`Set`/`Show`/`Strike`/`Load Out`).
      - **ROLE** — not STAGE, and matches `ROLE_NORMALIZATIONS` (the SAME table `extractRoleFlags` uses). Covers `A1`, `BO`, `LEAD`, `V1`, `GS`.
      - **UNKNOWN** — neither (e.g. `Rehearsal`, `Sho`).
   Collect `stageTokens` / `roleTokens` (incl. the multi-word roles from step 1) / `unknownTokens`.

   **Structural discriminator: (i) a STAGE token is the SOLE signal that separates a stage restriction from a role clause, and (ii) the presence of an UNKNOWN token is the SOLE signal that separates a VALID restriction from a malformed one — there is NO stage-word guessing/heuristic** (that would reintroduce the brittle literal lists this PR removes). A ROLE token NEVER makes a clause malformed (R22 f1); it is preserved to `cleaned`. Three mutually-exclusive branches, keyed on the token partition:
   - **Valid restriction (explicit)** — ≥1 STAGE token AND ZERO UNKNOWN tokens (every non-STAGE token is a recognized ROLE). → `stages` = the STAGE tokens (appearance order, deduped, canonical), `hasOnly = true`, `unrecognizedRestriction = false`, `consumedOnlyClause = true`. **`cleaned` PRESERVES the ROLE tokens: `cleaned = <the ROLE tokens, joined by ' / '> + tail`** (a role-prefixed valid restriction keeps BOTH — role AND the stage restriction). Covers `Set / Strike ONLY` (roles=[] → `[Set,Strike]`), `A1 / Set / Strike ONLY` (role `A1` + `[Set,Strike]`, R22 f1), `BO / Set / Show ONLY` (role `BO` + `[Set,Show]`), **`Load In / Set - LEAD ONLY` (hyphen-mixed → stages `[Load In,Set]` + role `LEAD`, R25 f1 — the `Set` day is NO longer dropped)**, `SHOW ONLY` (`[Show]`), and the clean full-4.
   - **Malformed (some-unknown)** — ≥1 STAGE token AND ≥1 UNKNOWN token. → `stages = []` (NEVER partial-narrow to the recognized subset, R3 f1 — the stage restriction is DELIBERATELY NOT applied), `hasOnly = true`, `unrecognizedRestriction = true`, `consumedOnlyClause = true`. **`cleaned` PRESERVES ALL NON-STAGE tokens — recognized ROLE tokens AND unknown/typo tokens (R16 f1 + R28 f2): `cleaned = <the ROLE and UNKNOWN tokens, in original order, joined by ' / '> + tail`.** Only STAGE tokens are excluded (they are accounted by the single `UNKNOWN_STAGE_RESTRICTION`, whose message embeds the full raw `roleCell`). Routing the non-stage tokens back through `extractRoleFlags` matches origin/main: recognized roles flag, multi-word typo-roles autocorrect (`ROLE_TOKEN_AUTOCORRECTED` — e.g. `CONTENT CRETION` → `CONTENT CREATION`, `personalization.ts:333-352`), and genuine unknown role-prefixes (`RIGGER`) emit `UNKNOWN_ROLE_TOKEN` — so no role or typo-role is silently dropped. → `UNKNOWN_STAGE_RESTRICTION` + whole-show. **The restriction is NOT applied — the fail-open choice (R28 f2 rebuttal):** an unknown token adjacent to a stage may itself be a MIS-TYPED STAGE (`Set / Shoo ONLY`, `Shoo` = `Show` typo); applying `[Set]` would HIDE the `Show` day — §9 forbids hiding days, so any unknown ⟹ whole-show. Covers `Set / Rehearsal ONLY` (`cleaned = "Rehearsal"` → `UNKNOWN_ROLE_TOKEN`), `A1 / Set / Rehearsal ONLY` (`cleaned = "A1 / Rehearsal"` → role `A1` + `UNKNOWN_ROLE_TOKEN(Rehearsal)`; STAGE `Set` excluded), `RIGGER / Set ONLY` (`cleaned = "RIGGER"` → `UNKNOWN_ROLE_TOKEN(RIGGER)`, R28 f2), `CONTENT CRETION / Set ONLY` (`cleaned = "CONTENT CRETION"` → autocorrected role, R28 f2), `Set - Rehearsal ONLY` (hyphen-mixed stage+unknown → malformed), `Load In / Rehearsal ONLY - LEAD` (`- LEAD` preserved via `tail`).
   - **No-stage (ROLE clause)** — body contains ZERO STAGE tokens. → role clause regardless of whether its roles are known; `stages = []`, `hasOnly = true`, `unrecognizedRestriction = false`, `consumedOnlyClause = false`, **`cleaned = roleCell` UNCHANGED** so `extractRoleFlags` handles it with its own tokenizer — PRESERVING recognized roles and emitting `UNKNOWN_ROLE_TOKEN` for unrecognized ones (R5 f1 hyphen decomposition, R8 f1 unknown-role preservation). Covers `A1 ONLY`, `SHOW CALLER ONLY`, `BO - V1 ONLY`, `GS - A1 ONLY` (roles preserved), `RIGGER ONLY` → `UNKNOWN_ROLE_TOKEN`, `A1 / RIGGER ONLY` → `A1` kept + `UNKNOWN_ROLE_TOKEN(RIGGER)`, `Rehearsal ONLY` → `UNKNOWN_ROLE_TOKEN` (matches `origin/main`; no stage-word guess). NO `UNKNOWN_STAGE_RESTRICTION` here.

   (A date token in `body` — `\d{1,2}/\d{1,2}` — is a leaked day restriction normally consumed upstream by `extractDayRestriction` (`crew.ts:299`); it has no recognized stage, so it falls into No-stage and is left to the existing role/day path. Defensive; not expected.)

**Structural invariant (the class-closing rule, R22 f1 + R25 f1):** with the body partitioned into STAGE / ROLE / UNKNOWN **tokens** (per-token, NOT per-`/`-segment) — (a) ZERO STAGE ⟹ role clause; (b) ≥1 STAGE AND ZERO UNKNOWN ⟹ EXPLICIT restriction on the STAGE tokens (ROLE tokens allowed and preserved); (c) ≥1 STAGE AND ≥1 UNKNOWN ⟹ `UNKNOWN_STAGE_RESTRICTION`. A ROLE token NEVER makes a clause malformed, and a STAGE token embedded in a hyphen-mixed group (`Set - LEAD`) is NEVER swallowed (R25 f1). **Role-preservation invariant (R16 f1 + R22 f1 + R28 f2):** in EVERY branch the non-stage content reaches `extractRoleFlags` — via `cleaned = roleCell` unchanged (No-stage), `cleaned = <ROLE tokens> + tail` (explicit — only recognized roles accompany a VALID restriction), or `cleaned = <ROLE + UNKNOWN tokens> + tail` (malformed — recognized roles AND unknown/typo tokens both preserved, so a typo-role autocorrects and an unknown role-prefix like `RIGGER` surfaces as `UNKNOWN_ROLE_TOKEN` instead of vanishing) — so no branch drops assignment metadata. This subsumes `SHOW CALLER` (multi-word role, no stage), hyphenated `BO - V1` (no stage → role), `RIGGER`/`Rehearsal` (no stage → role/UNKNOWN_ROLE_TOKEN), `Set / Strike` (stage, no unknown → `[Set,Strike]`), `A1 / Set / Strike` (role + stage → `[Set,Strike]` + role A1, R22 f1), `Load In / Set - LEAD` (hyphen-mixed → `[Load In,Set]` + role LEAD, R25 f1), `Set / Rehearsal` (stage + unknown → UNKNOWN_STAGE_RESTRICTION), and `A1 / Set / Rehearsal` (role + stage + unknown → UNKNOWN_STAGE_RESTRICTION + role A1 kept). A dedicated unit test asserts all three branches directly, parameterized over the decision table.

**Marker-validity enumeration (asterisk sub-vector, R14 f1 — pinned as a parameterized unit test in §8).** Marker validity is gated FIRST, orthogonally to the all/some/no-stage branch — an invalid marker collapses to the ABSENT branch (`hasOnly = false`) regardless of body:

Rationale: exactly mirrors the live `STAGE_TRAILING_MARKER_RE` contract (`ONLY` / `ONLY***` / bare `***`), so de-literalization does not widen marker tolerance beyond origin/main. **Exception — the full-4 lenient-star carve-out (step 2, R17 f1):** this enumeration governs the GENERALIZED grammar (step 3+); the EXACT-ORDER full physical-4 list `Load In / Set / Strike / Load Out ONLY\*{0,3}` is matched FIRST (step 2) and DOES accept ANY trailing star count (`ONLY*`/`ONLY**`/`ONLY***`/`ONLY****`+) as an explicit 4-stage restriction — the VERBATIM live `FULL_STAGE_ONLY_PATTERN` (whose `\*{0,3}` matches the `***`-prefix of `****`). **`extractStageRestriction` becomes:**

```ts
export function extractStageRestriction(roleCell: string): {
  restriction: StageRestriction;
  warnings: ParseWarning[];
  consumedOnlyClause: boolean; // true iff a VALID (bare ONLY or ONLY***) stage clause was parsed (explicit OR unrecognized)
} {
  const clause = parseStageClause(roleCell);
  if (clause.stages.length > 0 && clause.hasOnly) {
    return { restriction: { kind: "explicit", stages: clause.stages }, warnings: [], consumedOnlyClause: true };
  }
  if (clause.unrecognizedRestriction) {
    return {
      restriction: { kind: "none" },
      warnings: [{
        severity: "warn",
        code: "UNKNOWN_STAGE_RESTRICTION",
        message: `Role cell has a work-phase restriction we couldn't read: '${roleCell}'`,
        rawSnippet: roleCell,
      }],
      consumedOnlyClause: true,
    };
  }
  return { restriction: { kind: "none" }, warnings: [], consumedOnlyClause: false };
}
```

Signature change: `extractStageRestriction` now returns `{restriction, warnings, consumedOnlyClause}` (the warnings mirror `extractDayRestriction`'s `DayRestrictionResult` shape, `:68-73`). Caller `crew.ts:324` threads the warnings (stamped with `crewBlockRef`) into `warnings`/`agg.warnings` exactly as `dayResult.warnings` is threaded (`:300-301`).

## 4. Part C — Short-header typo tolerance (deferred P3 pickup)

### 4.1 Current behavior (cited)

`normalizeSectionHeaders` (`lib/parser/sectionHeaderNormalize.ts:44-97`) fuzz-corrects only three LONG headers — `LONG_SECTION_VOCAB = ["TRANSPORTATION", "EVENT DETAILS", "GS DETAILS"]` (`:16`) — behind a gate that accepts a label-only row OR a row with ≥1 field-header word (`:70-73`). The short routers `CREW`/`TECH`/`HOTEL`/`VENUE` were explicitly **deferred** as P3 in `docs/superpowers/specs/2026-06-27-parser-typo-tolerance-design.md` (§out-of-scope: "P3 (short section routers CREW/TECH/HOTEL/VENUE behind the field-band gate …)").

Consequence (audit #5): a one-edit typo in a short header — `TCEH` — silently vanishes the TECH grid; a `CRWE` typo instead trips the loud MI-3/4/5 hard fail. This part corrects the typo in a pre-pass so TECH parses AND, for CREW, pre-empts the cryptic MI hard-fail with a clean auto-correct + warning. (HOTEL/VENUE typos are also silent per the audit, but their header rows lack the field-header corroboration this gate needs — see §4.2 — so they are deferred to a per-router follow-up.)

### 4.2 New behavior

Add a `SHORT_SECTION_VOCAB` corrected behind gating STRICTER than the long-vocab path. The gates are: `minLen` + `exclude` (both real `gatedVocabCorrect` opts, `lib/parser/typoGate.ts:3-8`) + a **caller-side field-band gate** (`countFieldHeaderWords(otherCells) ≥ 1`, already at `sectionHeaderNormalize.ts:73`) + `noExactSpellingElsewhere` (caller, `:75-76`).

```ts
const SHORT_SECTION_VOCAB = ["CREW", "TECH"] as const;
```

Scoped to the two short routers whose REAL header rows in the corpus carry `SECTION_FIELD_HEADER_WORDS` so the field-band gate actually fires (adversarial-review R7 finding 2): `CREW | NAME | ROLE | PHONE …` (`NAME`/`ROLE`/`PHONE` are field-header words) and `TECH | PHONE | ARRIVAL | DEPARTURE` (`PHONE`). **`HOTEL` and `VENUE` are intentionally excluded**: their real header rows are `HOTEL | RESERVATION #1 | | RESERVATION #2` and (v1) `VENUE | Four Seasons …` — NO cell is a `SECTION_FIELD_HEADER_WORD`, so the field-band gate would NEVER fire and a `HOTLE`/`VENEU` typo would be silently left uncorrected. Shipping them in the vocab would be a hollow promise. They need a bespoke per-router corroboration (e.g. a `RESERVATION #`-shape or hotel/venue-value lookahead) that is out of scope here; filed as a backlog item. **`DATES` also excluded** (P4, region-less, DATE↔DATES delicacy, already-loud MI-3). (This narrows the informal "CREW/TECH/HOTEL/VENUE/DATES" list from the feature brief to the two routers the gate can actually serve; noted in the ship report + backlog.)

**Two-pass structure in `normalizeSectionHeaders`:** per row, try the existing LONG vocab first (unchanged gate: label-only OR ≥1 field word). - **Field-band gate (caller-side, stricter for short):** a short-header correction fires ONLY when the candidate row carries ≥1 `SECTION_FIELD_HEADER_WORDS` in its other cells (`countFieldHeaderWords`, `knownSections.ts:241`) — the `labelOnly` branch that the long vocab allows is NOT accepted for short headers (short headers are collision-prone; require field-header corroboration). But the Damerau-1 PLURALS of the short vocab — `CREWS` (CREW+S) and `TECHS` (TECH+S) — are NOT in any registry (verified: `rg CREWS lib/parser` → no match), so nothing currently blocks `CREWS | NAME | ROLE | PHONE` from autocorrecting to `CREW`. **Concrete code change:** add `const SHORT_SECTION_VOCAB_EXCLUDE = ["CREWS", "TECHS"] as const;` and pass `{ minLen: 4, exclude: [...EXCLUDE, ...SHORT_SECTION_VOCAB_EXCLUDE] }` to the SHORT-vocab `gatedVocabCorrect` call (the LONG-vocab call keeps its current `{ exclude: EXCLUDE }`). This makes the plural guard live-code, not an asserted-but-absent registry claim. `TECH`↔`TEAM` is Damerau-2 (safe without a guard). `DATE`/`DAY`/`ROOM` sub-labels remain in `EXCLUDE` harmlessly.

**Region mapping:** extend `CANON_TO_REGION` (`:18-22`) with the two short headers, using the canonical `RegionId` values from `REGION_ANCHOR_SPEC` (`lib/sheet-links/buildSheetDeepLink.ts:61-123`): `CREW`→`crew`, `TECH`→`crew` (the `crew` region header is `/^(CREW|TECH)$/i`, `:76`). ## 9. Disagreement-loop preempts (for the adversarial reviewer)

- **Fail-open posture on unreadable clauses is intentional.** A some-stage clause (`Set / Rehearsal ONLY`) → whole-show + `UNKNOWN_STAGE_RESTRICTION`; a no-stage clause (`Rehearsal ONLY`) → whole-show + `UNKNOWN_ROLE_TOKEN` (role clause). Both are whole-show + warning (NOT zero-days, NOT hard-fail), mirroring the existing `unknown_asterisk` fallback (`crew.ts:347-362`) and safer than hiding a show. Cited: §1 out-of-scope (Rehearsal) + the `effectiveViewerDateRestriction` zero-days trap (`stageSchedule.ts:66`). **Stage-token presence is the sole stage-vs-role discriminator — no stage-word heuristic (see §3.2 structural invariant); do not relitigate.**
- **Shape-recognized rooms are NOT signaled.** Deliberate (brainstorming chose "replace", declined "always-signal"). A newly-recognized room is visible on the review surface by its presence. §1 out-of-scope.
- **`minLen: 4` for the short vocab** (vs the established `minLen: 5` for field-alias fallback) is deliberate — `CREW`/`TECH` are 4 chars. The caller-side field-band corroboration (`countFieldHeaderWords ≥ 1`, `:73`) + `noExactSpellingElsewhere` + `EXCLUDE` are the compensating gates. Cited: typo-tolerance design §4.3 deferred these *behind the field-band gate*, which is exactly this gate. **Part C vocab is CREW/TECH only** — HOTEL/VENUE header rows carry no `SECTION_FIELD_HEADER_WORD` so the field-band gate can never fire for them (R7 f2); shipping them would be a hollow promise, so they are deferred to a per-router follow-up. `DATES` also excluded (P4, region-less, already-loud MI-3). See §4.2.
- **Room de-literalization is an admit-then-emit split, not a single predicate.** dims-only blocks are ADMITTED (to preserve dims via merge — R7 f1) but a room is EMITTED only when its key-group has a field-bearing block (so equipment with dims is dropped — R6 f1). `dimsInHeader` in the emit gate would reintroduce the equipment false-positive; the emit gate is field-evidence, not `roomHasContent` (which counts dims). Do not relitigate.
- **The `consumedOnlyClause` flag on `extractStageRestriction`** exists solely to suppress the pre-existing `crew.ts` triple-asterisk `UNKNOWN_DAY_RESTRICTION` when an `ONLY***` stage clause already consumed the marker (R7 f3) — it is not a new restriction concept.
- **`UNKNOWN_STAGE_RESTRICTION` is BOTH counted (data-gap) AND anchored, by design.** It mirrors `UNKNOWN_ROLE_TOKEN`/`UNKNOWN_DAY_RESTRICTION`, which each appear in BOTH `GAP_CLASSES` and `OPERATOR_ACTIONABLE_ANCHORED` (`dataGaps.ts:36-37,164-165`) — a crew-role sheet-data gap Doug can fix at a resolvable cell. Membership in both sets is intentional and pinned by the completeness meta-test (§7 item 8); do not relitigate as double-classification. (R13 f1.)
- **`STAGE_VOCAB` (typo vocab, `:174`) stays 4-wide; `STAGE_RESTRICTION_VOCAB` (grammar, new) is 5-wide.** Two different vocabularies for two different jobs (spelling correction vs clause recognition). Not a contradiction.
- **Classification is per-TOKEN, not per-`/`-segment; a ROLE token is preserved and NEVER makes a clause malformed (R16 f1 + R22 f1 + R25 f1).** The token-level classifier (multi-word extraction FIRST, then split on `/` AND `-`, then classify each atomic token STAGE/ROLE/UNKNOWN) makes role preservation a structural property of EVERY branch — `cleaned` always routes recognized roles to `extractRoleFlags` — AND guarantees a STAGE token embedded in a hyphen-mixed group (`Set - LEAD`) is never swallowed by an adjacent role (R25 f1): `Load In / Set - LEAD ONLY` → explicit `[Load In,Set]` + role LEAD, NOT `[Load In]`. Malformed (`UNKNOWN_STAGE_RESTRICTION`) is reserved for ≥1 STAGE token **and ≥1 UNKNOWN token**; a role-prefixed VALID restriction (`A1 / Set / Strike ONLY`) produces EXPLICIT `[Set,Strike]` AND keeps role `A1` (R22 f1). A leading role on a genuinely malformed clause (`A1 / Set / Rehearsal ONLY`) keeps `A1` while `Rehearsal` is embedded in the single `UNKNOWN_STAGE_RESTRICTION` message. Do not relitigate per-segment classification or treating ROLE+STAGE bodies as malformed — both regressions (R22, R25) are closed structurally by the shared token-level model.
- **A malformed clause preserves its non-stage tokens but NEVER applies the partial restriction (R28 f2).** The malformed branch (≥1 stage + ≥1 unknown) routes ALL non-stage tokens (role + unknown) to `extractRoleFlags`, so typo-roles autocorrect and unknown role-prefixes (`RIGGER`) surface as `UNKNOWN_ROLE_TOKEN` (matching origin/main — no role lost). But it does NOT apply the recognized-stage subset as a restriction: an unknown token adjacent to a stage may be a MIS-TYPED STAGE (`Set / Shoo ONLY`), and narrowing to `[Set]` would HIDE the `Show` day — the §9 fail-open contract forbids hiding days. This is the deliberate rebuttal to "apply the valid partial restriction": on the OUTCOME (whole-show) it matches origin/main, and applying the restriction is strictly LESS safe. Do not relitigate applying a partial restriction when any unknown token is present.
- **Marker tolerance is NOT widened, and full-4 compat is NOT dropped (R14 f1 + R17 f1).** For the GENERALIZED grammar the valid ONLY-marker forms are exactly `ONLY` and `ONLY***` — identical to the live `STAGE_TRAILING_MARKER_RE` (`personalization.ts:183-187`); `ONLY*`/`ONLY**` are invalid on PARTIAL subsets (fail-safe to whole-show), and origin/main never narrowed a partial on a starred marker (its partial narrowing used the star-free `LOAD_IN_SET_ONLY_PATTERN`/`LOAD_OUT_STRIKE_ONLY_PATTERN`). SEPARATELY, the EXACT full-4 list keeps origin/main's VERBATIM lenient `FULL_STAGE_ONLY_PATTERN` `ONLY\*{0,3}` (step 2 carve-out) → ANY trailing star count (`ONLY*`…`ONLY****`+) on the full-4 list stays an explicit 4-stage restriction (excludes Show days). The full-4 carve-out sets `cleaned = prefix + tail` (excises only the matched span), so a leading role (`A1`) is kept AND an unknown leading token (`RIGGER`) still emits `UNKNOWN_ROLE_TOKEN` — matching origin/main (R18 f2 + R19 f2). Both directions mirror origin/main exactly; neither the generalized-marker strictness NOR the full-4 lenience is a behavior change. Do not relitigate `***`-count tolerance for partial subsets (deferred); do not relitigate demoting the full-4 lenient star or adding a `(?!\*)` guard (either would EXPOSE Show days — R17 f1 / R19 f3); do not relitigate the full-4 prefix preservation (R19 f2).
- **Corpus-wide rooms no-op is the room contract**, not "looks right." De-literalization must be deep-equal to `origin/main` `rooms` on ALL committed fixtures; a bogus room anywhere fails. Discovery/terminator gate is name-shape + immediate-field-evidence (dims-alone rejected — R6). This whole-corpus regression is the structural defense that closed the three-round false-positive vector; do not relitigate the dims-alone branch.
- **Room ADMIT = a proper NON-dims-leading NAME on line 1 PLUS a TRAILING `DAY N` day-range on any line OR second-line dims (R18 f1 + R19 f1 + R20 f1 + R21 f1 + R26 f1).** This matches the committed east-coast signatures EXACTLY: `MABEL 1&#10;DAY 1 & 2` (DAY second-line), the single-line `LAUDERDALE 1, 2, 3 DAY 1 & 2` (DAY inline, commas), and novel `NAME&#10;DIMS`. The DAY marker is checked on ANY line (LAUDERDALE is single-line, R21 f1) but ONLY as a TRAILING day-range (R26 f1 — a DAY-note like `SPECIAL DAY 1 NOTES` has a word after the number and is rejected); dims stay second-line-only (equipment leads with dims). It rejects all fabrication shapes: field-beneath (single-line note), bare `&#10;` (multi-line note), dims-LEADING equipment (`4' X 8' RISER`), and DAY-notes. The de-literalized loop is a DROP-IN for `mabelRe` — SAME `firstLine`-verbatim name / dims / merge / `seen` — so the corpus no-op holds by construction for MABEL/LAUDERDALE. Do not relitigate admitting on field-beneath, bare `&#10;`, any-dims-token, or `DAY N` anywhere (non-trailing); do not require a second-line-only DAY (would drop the inline-DAY LAUDERDALE); do not ask for any of these to be backlogged.
- **`headerDayMarker` requires a TRAILING day-range, not `DAY N` anywhere (R26 f1).** A real room header's DAY marker is the trailing descriptor (`… DAY 1 & 2` at line end); an in-room DAY-NOTE (`SPECIAL DAY 1 NOTES`, `DAY 1 NOTES`) has an alphabetic word AFTER the number. The regex `\bDAYS?\s+\d[\d\s&,.\-–—]*$` (multiline `$`) admits MABEL's 2nd-line and LAUDERDALE's inline day-range while rejecting DAY-notes, closing a room-fabrication + field-theft path the corpus no-op alone could not guard (no fixture carries that shape). Do not relitigate loosening this to `DAY N` anywhere.
- **Room merge KEY strips a trailing inline DAY-range; DISPLAY name stays verbatim (R27 f1).** origin/main keys on raw `firstLine.toUpperCase()` (safe for two literal names); de-literalization makes an inline-DAY field-bearing block split from its dims-only same-name sibling and drop the dims. `roomMergeKey(firstLine)` strips the trailing inline DAY-range before uppercasing so they merge, while `buildEmptyRoom` still stores `firstLine` verbatim (LAUDERDALE's name unchanged). No-op on the corpus (MABEL has no inline DAY; LAUDERDALE is one block) — verified. Do not relitigate keying on raw `firstLine` (re-opens the dims-drop) or re-splitting the display name (breaks the east-coast no-op).
- **The room-header model and both extractors are LINE-BASED (R24 f1).** `RoomHeaderModel` carries `lines`, `roomHeaderLines` (line indices), and `emitEligibleGroups`; `extractBoBlock`/`extractGsBlock` take a start LINE and the model. This is behavior-identical to the live byte-offset `extractBoBlock(markdown, m.index)` because `mabelRe`'s `m.index` is the row's line-start (`markdown.slice(m.index) === lines.slice(startLine).join("\n")`, verified), so the block text is byte-for-byte the same while the start index is consistent with `roomHeaderLines`. Do not relitigate mixing line indices with byte offsets — the contract is uniformly line-based and the corpus no-op test pins it.
- **Emit-eligibility is RUN-OWNERSHIP: a field belongs to the EARLIEST admitted header in its run (R28 f1 + R29 f1).** A single top-to-bottom pass: an admitted header starts a new run (owner) iff a field has been seen since the current owner, else joins as a subordinate; a room field marks the current owner field-bearing. This makes a far-below field still count for its room (R28 f1) AND stops a subordinate `NAME&#10;dims` asset (`PROJECTION SCREEN`) between a room and its field from stealing that field (R29 f1) — the asset joins the room's run and never emits. Corpus no-op holds because a field appears right after MABEL, so LAUDERDALE starts a fresh run and owns its own fields. Do not relitigate a 1–2 row window (drops far-field rooms) or nearest-header attribution (re-opens the asset-theft).
- **Room EMIT evidence = the exact `applyBoLabel` row-field set — `floor` and `dimensions` are EXCLUDED (R23 f1 + R12 f1).** Both are HEADER-only (parsed by `splitRoomHeader`, never from a row), so a `Floor`/`Dimensions` label row is not a real captured field and must not make a candidate emit-eligible. `digital_signage`/`notes` ARE row-parsed by `applyBoLabel`, so they DO count. Sourcing the shared constant from `applyBoLabel` makes this drift-proof. Do not relitigate adding `floor` (or `dimensions`) to evidence — it opens a fabricated-room path.
- **Same-key repeated-header termination is origin/main-identical and lossless (R15 f1).** Pass 2 extracts EACH admitted member of a key-group with its OWN `extractBoBlock` call and unions the results via `mergeBoFields` — exactly as the live `mabelRe` loop already does for `MABEL 1` dims/fields split blocks (`rooms.ts:826-843`). A same-key sibling header therefore terminates only the PRIOR member's block; the later field-bearing block is extracted independently and merged, so no same-room data is lost. The block's own starting header never self-terminates (the existing positional `k > 0` guard, `extractBoBlock:888-894`, is preserved verbatim). This is not a new terminator model — it is the de-literalized form of the one already shipping; do not relitigate per-member extraction vs a single flat block scan.
