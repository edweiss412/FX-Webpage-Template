## 2. Part A — Room-header shape predicate

### 2.1 Current brittle behavior (cited)

`rooms.ts` hangs v1 room recognition on two hardcoded venue names, `MABEL` and `LAUDERDALE` (from the single 2024 east-coast fixture), at three sites:

1. **`extractGsBlock` terminator** (`lib/parser/blocks/rooms.ts:687`): `/^\|\s*(GENERAL SESSION|BREAKOUT|ADDITIONAL|LUNCH|MABEL|LAUDERDALE|DETAILS)\b/i` — stops GS bare-field extraction at the next room/section header.
2. **v1 room loop** (`:825`): `const mabelRe = /^\|\s*(MABEL\s+\d[^|]*|LAUDERDALE[^|]*?)\s*\|/gim;` — the ONLY discovery path for a label-less venue-named room block.
3. **`NEXT_ROOM_HEADER_RE`** (`:880-881`): `/^\|\s*(GENERAL\s+SESSION|BREAKOUT|ADDITIONAL\s+ROOM|LUNCH\s+ROOM|MABEL|LAUDERDALE|DETAILS)\b/i` — the block-extraction terminator used by `extractBoBlock` (`:883-898`).

Consequence (audit #6): every new venue has new room names; a bare proper-name GS/breakout header (e.g. `SALON ABCD\n60' x 45'`) at a venue that isn't the east-coast one is **lost or mis-grouped** — no room row, no signal.

Note: shape-based recognition already exists for the *GS* label-less case — `findGsBlockVenueHeader` (`:543-581`) requires "strong evidence" (in-cell `&#10;` newline OR a dims token, plus a section-banner exclusion list). Part A generalizes that same discipline to the v1 breakout-room loop and de-literalizes the three sites above with one shared predicate.

### 2.2 New behavior

Introduce **three** functions with clean single-responsibility contracts (splitting the pure name-shape test from the contextual evidence test — a single-cell predicate cannot see the block beneath, per adversarial-review R1 finding 2). All exported from `lib/parser/blocks/rooms.ts`:

```ts
// (a) PURE — name-shape only, computable from the header cell alone.
export function roomHeaderNameShape(col0Raw: string): boolean
// (b) PURE — a dims token on a NON-FIRST line of the header cell ("NAME&#10;<dims>", e.g.
//     SALON ABCD&#10;60' x 45'). A dims-LEADING single line (equipment "4' X 8' RISER") does NOT
//     qualify — its dims are on the first/name line, and roomHeaderNameShape item 1 rejects it (R20 f1).
function secondLineDims(col0Raw: string): boolean     // a line AFTER the first matches /\d+\s*'\s*x/i
// (b') PURE — a DAY marker ANYWHERE in the cell — inline on the first line OR on a later line. The
//      east-coast rooms carry it BOTH ways: MABEL 1&#10;DAY 1 & 2 (second line) AND the single-line
//      LAUDERDALE 1, 2, 3 DAY 1 & 2 (inline). This is the room signal that admits day-marked headers
//      without dims. A note ("FOH POSITION&#10;Downstage") has no DAY marker (R19 f1); equipment has
//      no DAY marker (R20 f1) — so DAY-anywhere does not re-open either false positive.
function headerDayMarker(col0Raw: string): boolean    // ANY line matches /\bDAYS?\s+\d/i
// (also: dimsInHeader(col0Raw) — /\d+\s*'\s*x/i across ANY line — is retained for splitRoomHeader
//  VALUE extraction only, NOT for admit.)
// (c) LOCAL — ≥1 recognized breakout room-field among the IMMEDIATE next 1–2 non-blank rows.
//      EMIT-gate signal ONLY — NOT an admit signal (adversarial-review R18 f1: field-beneath is too
//      weak; an in-room note/equipment row before a normal Audio/Notes field would fabricate a room).
function hasRoomFieldEvidence(nextRowsText: string): boolean
// (d) ADMIT predicate — a proper NAME on line 1 (not dims-leading) PLUS a DAY marker anywhere OR
//     second-line dims. Independent of fields beneath. Admits MABEL/LAUDERDALE (DAY) + novel dims
//     headers (SALON); excludes single-line notes (R18 f1), multi-line notes (R19 f1), AND
//     dims-leading equipment (R20 f1).
export function isRoomHeaderShape(col0Raw: string): boolean
//   = roomHeaderNameShape(col0Raw) && (headerDayMarker(col0Raw) || secondLineDims(col0Raw))
// (e) TOP-LEVEL MODEL — computed ONCE before parseGsRoom/parseBoRooms (R17 f2); `seen`-independent.
export type RoomHeaderModel = { roomHeaderLines: ReadonlySet<number>; /* + emit-eligible groups */ };
export function computeRoomHeaderModel(markdown: string): RoomHeaderModel
//   scans all col0 rows → admit (isRoomHeaderShape) → group by key → emit-eligible (≥1 fieldBearing)
//   → roomHeaderLines = lineIndex set of candidates in emit-eligible groups. Threaded into
//   parseGsRoom→extractGsBlock and parseBoRooms→extractBoBlock as the shared terminator set.
```

**Design note — comprehensive re-analysis of the room-discovery false-positive vector (adversarial-review R2 f1, R3 f2, R4 f1, R6 f1, R7 f1, R10 f1, R11 f1, R12 f1, R15 f1, R17 f2, R18 f1, R19 f1, R20 f1, R21 f1 — a recurring vector; per the AGENTS.md "comprehensive re-analysis + structural defense" rule the model is now anchored end-to-end against the LIVE extraction code AND the committed east-coast fixture bytes, not point-patched: admit/emit/terminate are three distinct gates; ADMIT requires a proper NON-dims-leading NAME on line 1 PLUS a `DAY N` marker ANYWHERE (LAUDERDALE is single-line, R21 f1) OR a DIMS token on a NON-FIRST line — matching the committed `MABEL 1&#10;DAY 1 & 2` / `LAUDERDALE 1, 2, 3 DAY 1 & 2` / `NAME&#10;DIMS` shapes — so NONE of field-beneath (R18 f1), a bare multi-line note (R19 f1), OR a dims-leading equipment row (R20 f1) can fabricate a room; the loop is a DROP-IN for `mabelRe` (same name/dims/merge) so the no-op holds by construction; and the corpus-wide no-op test is the mechanism-agnostic guard).** The literal `MABEL|LAUDERDALE` loop was safe because it matched two specific names; a shape scan is inherently more permissive. The resolution is a **two-stage admit-then-emit** model that separates "worth extracting a block for" from "actually a room":

1. **ADMIT** a candidate iff `roomHeaderNameShape && (headerDayMarker || secondLineDims)` — a proper NAME on line 1 (NOT dims-leading, item 1) PLUS a `DAY N` marker ANYWHERE in the cell OR a DIMS token on a NON-FIRST line. This matches the exact signatures of the committed east-coast rooms (verified against `2024-05-east-coast-family-office.md`): `MABEL 1&#10;DAY 1 & 2` (`:26`, DAY on the SECOND line), the SINGLE-LINE `LAUDERDALE 1, 2, 3 DAY 1 & 2` (`:35`, DAY INLINE, commas in the name), and novel `NAME&#10;DIMS` headers (`SALON ABCD&#10;60' x 45'`). The DAY marker is checked ANYWHERE (not second-line-only) precisely because LAUDERDALE carries it inline (adversarial-review R21 f1); dims stay second-line-only to keep dims-leading equipment out. **None of field-evidence-beneath, bare multi-line, OR a dims-LEADING line is an admit signal (adversarial-review R18 f1 + R19 f1 + R20 f1).** Field-beneath fabricates a room from an in-room note/equipment row before a real field; "any `&#10;`" fabricates one from a multi-line note (`FOH POSITION&#10;Downstage`); "any dims token" fabricates one from a dims-leading equipment row (`4' X 8' RISER`). The predicate excludes ALL THREE: a note has no `DAY N` and no second-line dims; an equipment row is dims-leading (item 1) with no `DAY N`. This ADMITS `MABEL 1`/`LAUDERDALE …` (DAY marker), the dims-only-first block (`SALON ABCD&#10;60' x 45'`, second-line dims — dims captured for merge, R7 f1); it does NOT admit a single-line note, a multi-line note, or a dims-leading equipment row.

   **De-literalization is a DROP-IN for the `mabelRe` loop (`rooms.ts:823-847`), guaranteeing the corpus no-op for MABEL/LAUDERDALE by construction.** The ONLY thing that changes is row SELECTION: `mabelRe` (matches `MABEL \d…|LAUDERDALE…`) → `isRoomHeaderShape`. Everything downstream is preserved VERBATIM: the room name is `firstLine` of the header cell as-is (`buildEmptyRoom("breakout", firstLine)` — so `MABEL 1` and the full `LAUDERDALE 1, 2, 3 DAY 1 & 2` are stored EXACTLY as origin/main stores them, NOT re-split), dims ride from `rawHeader.split("\n").slice(1)`, `headerKey = firstLine.toUpperCase()`, `mergeBoFields(extractBoBlock(...))`, and the `seen` dedup. Because `isRoomHeaderShape` admits exactly `{MABEL 1, LAUDERDALE …}` among the east-coast rows (the corpus no-op test pins this) and processes them through the identical name/dims/merge path, the parsed `rooms` are byte-identical for existing fixtures; only NOVEL non-corpus shapes add rooms.
2. **Group** admitted candidates by header key and merge dims + fields across the group (the existing `mergeBoFields` merge-by-key logic, `:823-847`, UNCHANGED).
3. **EMIT** a room for a key-group iff **≥1 admitted block in that group has `hasRoomFieldEvidence`** (an ACTUAL immediate room-field row). Dims-leading equipment is already excluded at ADMIT (item 1 + second-line rule), so the emit gate's remaining job is to drop a LONE `NAME&#10;dims`/`NAME&#10;DAY` header that has neither an immediate room field NOR a field-bearing same-key sibling — e.g. a gear-list entry `PROJECTION SCREEN&#10;5' x 9'` whose neighboring rows are more gear, not room fields. `roomHasContent` alone is INSUFFICIENT because it counts `dimensions` as content (`rooms.ts:424-441`), so such a dims-bearing block would pass it — the emit gate requires a real *field*-bearing block in the group. A `SALON ABCD&#10;60' x 45'` dims-only block still emits BECAUSE its same-key day sibling is field-bearing (R7 f1).

| False-positive / correctness class | Example | Closed by |
|-------|---------|-----------|
| All-caps NOTE inside a block (note text beneath) | `SPECIAL NOTE` then note text | not admitted (single-line: no second-line dims, no `DAY N`) |
| Single-line NOTE followed by a REAL FIELD (R18 f1) | `SPECIAL NOTE` then `Audio`/`Notes`/`Digital Signage` (fields of the OPEN room) | not admitted — no `DAY N` marker, no second-line dims, and field-beneath is NOT an admit signal → neither emits NOR enters `ROOM_HEADER_LINES` |
| MULTI-LINE NOTE followed by a REAL FIELD (R19 f1) | `FOH POSITION&#10;Downstage` then `BO Audio` (field of the OPEN room) | not admitted — the extra line is neither a dims token nor a `DAY N` marker → not a room, does not truncate the open room |
| Uppercase FIELD-LABEL row | `BO SETUP` / `AUDIO` | `roomHeaderNameShape` item 3 (field-label exclusion) |
| Dimension-LEADING EQUIPMENT/asset row (single-line) | `5' X 9' WIDESCREEN…` (`fixed-income:121`), `4' X 8' RISER` | not admitted — dims-leading first line fails `roomHeaderNameShape` item 1 (R20 f1); no `DAY N` marker and dims are on the first (name) line, not a second line |
| Dims-leading EQUIPMENT followed by a REAL FIELD (R20 f1) | `4' X 8' RISER` then `BO Audio`/`Notes` (fields of the OPEN room) | not admitted (item 1 rejects dims-leading) → neither emits NOR enters `ROOM_HEADER_LINES`; does not truncate the open room |
| Repeated room: dims-first + fields-later (CORRECTNESS) | `SALON ABCD&#10;60' x 45'` … `SALON ABCD&#10;DAY 1 & 2` + `BO Setup` | dims-only block admitted → merged into the same-key group → EMITTED because the day block is field-bearing; **dims survive** (R7 f1) |

east-coast holds: `MABEL 1&#10;DAY 1 & 2` (`:26`, DAY on the second line) + `BO Setup` (`:28`) and the SINGLE-LINE `LAUDERDALE 1, 2, 3 DAY 1 & 2` (`:35`, DAY INLINE, commas) + `BO Setup` (`:36`) are admitted (proper name + `headerDayMarker`) and field-bearing → emitted; equipment rows are dims-leading (no DAY) → never admitted. `dimsInHeader` (dims anywhere) is used ONLY for extracting the dims VALUE in `splitRoomHeader`; the ADMIT signals are `headerDayMarker` (DAY anywhere) and `secondLineDims` (dims on a non-first line only). Structural defense shipped this round: the **corpus-wide rooms-array no-op test** (§8) — de-literalization MUST be deep-equal to `origin/main` `rooms` on all 15 committed fixtures — is the mechanism-agnostic guard that catches ANY residual false positive OR dropped dimension across the whole corpus.

**`roomHeaderNameShape(col0Raw)` (pure) returns true iff ALL hold:**

1. **Non-empty, label-less proper name — CASE-INSENSITIVE (adversarial-review R10 finding 1).** After flattening `&#10;`→newline, taking the first line, trimmed, and **uppercasing**, it matches the proper-name shape `^[A-Z0-9][A-Z0-9 &',./-]*$` (letters/digits/spaces + a small punctuation set INCLUDING the comma — the committed east-coast `LAUDERDALE 1, 2, 3 DAY 1 & 2` single-line header carries commas, adversarial-review R21 f1; at least one letter) **AND is NOT dimension-leading — it does NOT match `^\d+\s*'\s*x`i (adversarial-review R20 f1).** An equipment/asset row leads with its dimensions (`4' X 8' RISER`, `5' X 9' WIDESCREEN…`); a real room NAME does not (`SALON ABCD`, `MABEL 1`, `Grand Foyer`). Rejecting a dims-leading first line excludes equipment at the name-shape gate. (Names that merely start with a digit but are not dims — `5TH AVENUE BALLROOM` — do NOT match `^\d+\s*'\s*x` and are unaffected.) The match is case-INSENSITIVE (the literal `mabelRe` it replaces used `/gim`), so a mixed-case novel venue name like `Salon ABCD` / `Grand Foyer` IS admitted — that is the feature's whole point. Mixed-case *field labels* (`Breakout Room Setup Date / Time`) are NOT excluded by casing; they are excluded by item 2 (`BREAKOUT`/`ADDITIONAL`/… banner) and item 3 (field-label), both of which uppercase before comparing. The header-cell-evidence admit gate (a dims token OR a `DAY N` marker) + the emit gate (§2.2) are the real discriminator for anything else; the corpus no-op test pins that no mixed-case non-room row in the 15 fixtures becomes a room.
2. **Not a known section/room banner.** The uppercased first line does NOT start with any structural keyword already handled by a dedicated path or that denotes a section: `GENERAL SESSION`, `BREAKOUT`, `ADDITIONAL`, `LUNCH`, `DETAILS`, plus the section banners `findGsBlockVenueHeader` already excludes (`:565-571`: `DOCUMENTS`, `DATES`, `CREW`, `DRESS`, `TRANSPORTATION`, `HOTEL`, `VENUE`, `AGENDA`, `CONTACTS`). Also NOT in `KNOWN_SECTION_HEADERS` / `KNOWN_SUB_LABELS` (`lib/parser/knownSections.ts:34,96`) and NOT a `GS <label>` field row (`^GS\s`).
3. **Not a room-FIELD label** (adversarial-review R4 finding 1). A sheet with UPPERCASE field labels (`BO SETUP`, `AUDIO`, `SET TIME`) — if any carried a dims token or `&#10;` — would otherwise satisfy the name-shape + header-cell-evidence gate and fabricate a room named after a field label — a regression the literal MABEL/LAUDERDALE loop cannot cause. Reject the row if its uppercased first line — after stripping an optional leading `BO ` or `GS ` prefix — is one of the room-field labels that `applyBoFields`/`applyGsLabel` recognize (`rooms.ts:695-707` + the `applyBoFields` label set): `SETUP`, `SET TIME`, `SHOW TIME`, `STRIKE TIME`, `AUDIO`, `VIDEO`, `SCENIC`, `LIGHTING`, `LED`, `POWER`, `OTHER`, `DIMENSIONS`, `FLOOR`, `DIGITAL SIGNAGE`, `NAME(S)`, `NOTES`. The exclusion list is DERIVED from the live `applyBoFields`/`applyGsLabel` code (not hardcoded in the spec) so it cannot drift; the plan pins it to a shared constant.

**`hasRoomFieldEvidence(nextRowsText)`**: ≥1 recognized breakout room-field row among the IMMEDIATE next 1–2 non-blank table rows following the header. The recognized-field set is **DERIVED from the shared `applyBoFields`/`applyGsLabel` label set** (not a spec-hardcoded list that could drift, adversarial-review R12 finding 1) — every field those functions map, matched as `BO <field>` or bare label rows exactly as they do: `setup`/`set time`/`show time`/`strike time`/`audio`/`video`/`scenic`/`lighting`/`led`/`power`/`other`/`floor`/**`digital signage`**/**`notes`**. **`dimensions` is the ONLY field EXCLUDED from evidence** — a dims value is what an equipment/asset row carries, so counting it would defeat the emit gate; every OTHER content field `roomHasContent` counts (`:424-441`) IS valid evidence, so a novel room whose immediate corroboration is only `Digital Signage`/`Notes` still emits. The plan pins this to a single shared constant (evidence set = `applyBoFields` labels ∖ `{dimensions}`) so it cannot drift from `roomHasContent`. Bounded to the immediate rows, NOT the whole block.

A tiny helper `nextNonBlankRows(lines, i, n=2)` returns the next `n` non-blank, non-separator table rows after index `i`, joined — used by both call sites so the window is identical.

**Uses — THREE separate gates (adversarial-review R11 finding 1: discovery-admit ≠ emit ≠ terminate must NOT share one predicate):**

- **Pass 0 — `computeRoomHeaderModel(markdown)` (top-level, computed ONCE before any block parse; adversarial-review R17 finding 2).** The room-header model is built at the TOP of `collectV2V1Rooms` (`rooms.ts:215`), BEFORE `parseGsRoom(markdown)` (`:218`) and `parseBoRooms(markdown)` (`:220`), and is **INDEPENDENT of the BREAKOUT/LUNCH `seen` state** (which does not yet exist at `parseGsRoom` time — the cyclicity R17 f2 flags). It scans every table row's col0 and ADMITs iff `isRoomHeaderShape(rawHeader)` (= `roomHeaderNameShape && (headerDayMarker || secondLineDims)` — a proper non-dims-leading NAME on line 1 plus a `DAY N` marker anywhere OR a dims token on a NON-FIRST line; NOT field-beneath, NOT bare multi-line, NOT dims-leading equipment — R18 f1 + R19 f1 + R20 f1 + R21 f1). Because `roomHeaderNameShape` item 2 already rejects the `GENERAL SESSION`/`BREAKOUT`/`ADDITIONAL`/`LUNCH`/`DETAILS` banners, those structural rows are never admitted here — so the model needs NO `seen` input to avoid them. It records per admitted candidate `{key = firstLine.toUpperCase(), fieldBearing = hasRoomFieldEvidence(nextNonBlankRows(lines, i)), lineIndex}` (the field-beneath check is used HERE only to set the emit-eligibility flag, not to admit), groups by key, marks a key **EMIT-ELIGIBLE iff ≥1 of its candidates is `fieldBearing`**, and returns `{ ROOM_HEADER_LINES, emitEligibleGroups }` where `ROOM_HEADER_LINES` = the set of `lineIndex` of candidates in emit-eligible key-groups. A dims-LEADING equipment row (`5' X 9' WIDESCREEN…`, `4' X 8' RISER`) is NEVER admitted — its dims are on the first/name line, so `roomHeaderNameShape` item 1 rejects it and there is no second-line evidence (R20 f1) — even when a real field follows it; so it can neither emit nor terminate. An in-room note — single-line (`SPECIAL NOTE`) OR multi-line (`FOH POSITION&#10;Downstage`) — is likewise NEVER admitted (no `DAY N` marker anywhere and no second-line dims), so it can neither emit nor terminate (R18 f1 + R19 f1). A LONE `NAME&#10;dims`/`NAME&#10;DAY` header with no field-bearing group member (a gear-list entry) is admitted but NOT emit-eligible → its line is NOT in `ROOM_HEADER_LINES`. This one model is THREADED as a parameter into `parseGsRoom` → `extractGsBlock` AND `parseBoRooms` → `extractBoBlock` (signature change: both take the precomputed model), so GS and BO extraction share ONE terminator set regardless of call order.
- **Pass 1 — GS + BO extraction consume the shared model.** `parseGsRoom` (`:218`, runs FIRST) and `parseBoRooms` (`:220`) both receive the Pass-0 model. The BREAKOUT/LUNCH `seen` dedup stays exactly where it is in `parseBoRooms` (`:807,830`) — it governs which rooms are EMITTED/deduped AFTER extraction, NOT termination. Duplicate suppression is therefore decoupled from the terminator prepass (R17 f2): the terminator set is purely structural line indices, computed once, order-independent.
- **Pass 2 — extract EACH emit-eligible member block separately, then merge + emit.** For each emit-eligible key-group, extract EVERY member block INDEPENDENTLY (`extractBoBlock`, one call per admitted candidate `lineIndex` in the group — exactly as the live `mabelRe` loop calls `extractBoBlock(markdown, m.index)` once per `MABEL`/`LAUDERDALE` occurrence, `rooms.ts:826-843`), each bounded by the terminator (below), then `mergeBoFields` the per-block results across the group (UNCHANGED, `:854-876`) and emit ONE room. Because each same-key member is extracted independently and then UNIONed, a same-key sibling header acting as the terminator of a PRIOR member block is CORRECT and lossless — the later field-bearing block is extracted by its OWN `extractBoBlock` call and its fields are merged in. `MABEL 1&#10;DAY 1 & 2` / `LAUDERDALE …` are field-bearing → emitted; a `SALON ABCD&#10;60' x 45'` dims-only block (its dims captured by `splitRoomHeader`/`mergeBoFields` on its own extraction) merges into its same-key day-block group and is emitted with dims preserved (R7 f1); the equipment row is DROPPED (R6 f1); an all-caps note is never admitted (R3 f2).
- **Terminator** — `extractGsBlock` (`:670-693`) and `NEXT_ROOM_HEADER_RE` → `extractBoBlock` (`:880-898`), BOTH now consulting the Pass-0 `ROOM_HEADER_LINES`: a line terminates the block iff it is a structural keyword (`GENERAL SESSION|BREAKOUT|ADDITIONAL ROOM|LUNCH ROOM|DETAILS`) OR its absolute index ∈ `ROOM_HEADER_LINES` (an EMIT-ELIGIBLE room header — NOT merely admitted) — **EXCEPT the block's own starting header line is NEVER a terminator, preserving the existing positional `k > 0` guard verbatim (`extractBoBlock:888-894`: "Stop at the next room/section header — but not the current block's own header on line 0").** Concretely: extraction starts at the candidate's offset; the line at position 0 (its own header, whose index is of course in `ROOM_HEADER_LINES`) does NOT self-terminate; every LATER line whose absolute index ∈ `ROOM_HEADER_LINES` (or is a structural keyword) does. Three properties this yields: (1) R11-f1 — a dims-only equipment row is admitted but NOT emit-eligible, so its index is NOT in `ROOM_HEADER_LINES` and it NEVER severs a preceding room's field extraction; (2) R15-f1 — a same-key repeated header (dims-first + fields-later) terminates the PRIOR sibling block but, because Pass 2 extracts each member independently and `mergeBoFields` unions them, NO same-room data is lost — identical to origin/main's `MABEL 1` dims/fields split-then-merge behavior; (3) R17-f2 — because the model is precomputed before `parseGsRoom`, a novel proper-name room AFTER the GS block now terminates `extractGsBlock` (its emit-eligible index is in the shared `ROOM_HEADER_LINES`) instead of bleeding into the GS room. Same-key members terminate each other's PRIOR block (correct, lossless via merge); a member never self-terminates on its own header line.

The **corpus-wide rooms no-op test** (§8) plus explicit **negative terminator, negative discovery, equipment-before-fields (no-truncation), and repeated-block tests** pin this — the highest-risk area in the PR, bounded by the three-gate split and a whole-corpus regression, not asserted blind.

**Known limitation (R11 finding 2, backlog):** a room whose name is EXACTLY a `KNOWN_SECTION_HEADERS` token — in practice only `FOYER` (`knownSections.ts:64`) is both a registered section token AND a plausible bare room name — is NOT admitted (item 2 excludes it; a bare `Foyer` is genuinely ambiguous with the section token). COMPOUND names are unaffected: `Grand Foyer` / `East Foyer` do not equal `FOYER` and ARE admitted when their header cell carries dims or a `&#10;`. Rather than ship a venue-token carve-out list (the exact literal-list brittleness this PR removes), bare `Foyer`-as-room is documented as unsupported and filed as a backlog item; a test asserts `Grand Foyer` works and bare `Foyer` is (currently) not a room.

**In-room notes AND equipment rows are structurally closed (R18 f1 + R19 f1 + R20 f1 + R21 f1), NOT backlogged.** Admit requires a proper NON-dims-leading NAME on line 1 PLUS a `DAY N` marker anywhere OR a dims token on a NON-FIRST line. A single-line note (`SPECIAL NOTE`) has no DAY and no second-line dims; a multi-line note (`FOH POSITION&#10;Downstage`) has neither; a dims-leading equipment row (`4' X 8' RISER`, `5' X 9' …`) fails item 1 (dims-leading name) and has no DAY. All are rejected at admit, statelessly, without an "is-inside-an-open-room" positional tracker (added parser state this PR deliberately avoids). The only residual would be a row whose FIRST line is a proper non-dims-leading name AND which carries a `DAY N` marker or a second-line `NN' x NN'` dims token AND which sits immediately before a real field — a shape indistinguishable from a room header by construction and absent from the corpus (the corpus-wide rooms no-op test is the backstop). Regression tests cover single-line-note-then-field (R18 f1), multi-line-note-then-field (R19 f1), dims-leading-equipment-then-field (R20 f1), and the single-line inline-DAY LAUDERDALE header (R21 f1).

## 3. Part B — Stage-restriction token grammar

### 3.1 Current brittle behavior (cited)

`extractStageRestriction` (`lib/parser/personalization.ts:158-169`) recognizes exactly three hand-picked phrasings via three regexes:

- `FULL_STAGE_ONLY_PATTERN` (`:53-54`) — `Load In / Set / Strike / Load Out ONLY` → all 4 physical stages.
- `LOAD_IN_SET_ONLY_PATTERN` (`:55`) — anchored `^- Load In / Set ONLY$` → `[Load In, Set]`.
- `LOAD_OUT_STRIKE_ONLY_PATTERN` (`:56`) — anchored `^- Load Out / Strike ONLY$` → `[Load Out, Strike]`.

`extractRoleFlags` (`:267-370`) DUPLICATES the same three patterns to strip the stage prefix before tokenizing role flags (`:279-293`).

Consequence (audit #7): a reordered or novel-but-valid subset — `Set / Strike ONLY`, `Load Out / Strike ONLY` (note: only `Load In/Set` and `Load Out/Strike` orderings are hardcoded), `Set / Show ONLY` — falls through `extractStageRestriction` to `{kind:"none"}` AND is then mis-tokenized by `extractRoleFlags` into an `UNKNOWN_ROLE_TOKEN` cascade (e.g. `Set / Strike ONLY` → tokens `SET`, `STRIKE ONLY` → two spurious unknown-role warnings), and the crew member sees the **whole show** (a valid restriction silently ignored). The grammar recognizes these reordered/subset stage clauses; a malformed one that mixes a real stage with an unreadable token (`Set / Rehearsal ONLY`) gets a precise `UNKNOWN_STAGE_RESTRICTION` instead of the confusing multi-`UNKNOWN_ROLE_TOKEN` cascade. (A clause with NO recognized stage — `Rehearsal ONLY`, `RIGGER ONLY` — is a role clause and keeps its existing `UNKNOWN_ROLE_TOKEN` behavior; §3.2.)

### 3.2 New behavior — one shared `parseStageClause`

Introduce one order-independent grammar function, shared by both consumers (kills the 3-pattern duplication):

```ts
// lib/parser/personalization.ts
const STAGE_RESTRICTION_VOCAB = ["LOAD IN", "SET", "SHOW", "STRIKE", "LOAD OUT"] as const;
//                                                  ^^^^ NEW: Show added (already a valid WorkPhase)

export type StageClause = {
  /** Recognized stage tokens in appearance order, deduped. Empty if none recognized. */
  stages: Array<"Load In" | "Set" | "Show" | "Strike" | "Load Out">;
  /** True iff a VALID trailing restriction marker was present — exactly bare `ONLY`
   *  or `ONLY***` (three stars). A trailing 1–2 stars (`ONLY*`/`ONLY**`) is NOT a
   *  valid marker (mirrors live `STAGE_TRAILING_MARKER_RE`, personalization.ts:183-187). */
  hasOnly: boolean;
  /** The role cell with the leading stage clause + ONLY marker removed, for role-flag tokenizing. */
  cleaned: string;
  /** True iff the cell had an ONLY-clause that looked like a restriction but no stage resolved. */
  unrecognizedRestriction: boolean;
  warnings: ParseWarning[];
};
export function parseStageClause(roleCell: string): StageClause
```

**Grammar (evaluated on the role cell, after day-restriction extraction already ran upstream — see `crew.ts:299-304`):**

1. Strip an optional leading `-` and whitespace (preserved for `cleaned` reconstruction).
2. **Full-4-list lenient-star carve-out FIRST (backward-compat, adversarial-review R17 finding 1).** Before the strict marker search, test the retained live `FULL_STAGE_ONLY_PATTERN` — `/Load\s+In\s*\/\s*Set\s*\/\s*Strike\s*\/\s*Load\s+Out\s+ONLY\*{0,3}/i` (`personalization.ts:53-54`, the EXACT-ORDER full physical-4 list with a LENIENT `ONLY\*{0,3}` marker). The pattern is the VERBATIM live `FULL_STAGE_ONLY_PATTERN` (UNANCHORED, `ONLY\*{0,3}`), so it (a) can match a full-stage substring AFTER a leading role prefix, and (b) accepts ANY trailing star count including `****` — because `\*{0,3}` matches the `***` prefix of `****` (verified). Both are DELIBERATE origin/main mirroring (see the star-count note below). If it matches → `stages = [Load In, Set, Strike, Load Out]`, `hasOnly = true`, `unrecognizedRestriction = false`, `consumedOnlyClause = true`. **`cleaned` PRESERVES the ENTIRE prefix (adversarial-review R18 f2 + R19 f2):** let `prefix` = text BEFORE the matched span, `tail` = text after it; **`cleaned = prefix + tail`** — the matched full-stage-ONLY span is EXCISED and everything else is preserved VERBATIM (NOT segment-filtered). This flows ALL prefix segments to `extractRoleFlags`: recognized roles are kept AND unrecognized tokens still emit `UNKNOWN_ROLE_TOKEN`, exactly as origin/main (whose anchored `extractRoleFlags` strip left the whole prefix intact while `FULL_STAGE_ONLY_PATTERN` produced the 4-stage restriction). Examples: `A1 / Load In / … ONLY*` → `cleaned = "A1 / "` → role `A1`; `RIGGER / Load In / … ONLY` → `cleaned = "RIGGER / "` → `UNKNOWN_ROLE_TOKEN(RIGGER)` PRESERVED (R19 f2); `A1 / RIGGER / Load In / … ONLY` → `cleaned = "A1 / RIGGER / "` → `A1` + `UNKNOWN_ROLE_TOKEN(RIGGER)`; pure `Load In / … ONLY*` (no prefix) → `cleaned = tail`. **Star-count note (R17 f1 + R19 f3 — anti-expansion, do NOT add `(?!\*)`):** the full-4 carve-out intentionally accepts `ONLY*` / `ONLY**` / `ONLY***` / `ONLY****`+ as an explicit `[Load In,Set,Strike,Load Out]` restriction, mirroring origin/main's `FULL_STAGE_ONLY_PATTERN` EXACTLY. A full-4 restriction EXCLUDES pure `Show` days (now a distinct WorkPhase); demoting ANY star count to whole-show would EXPOSE Show days — the R17 f1 regression. So the marker-enumeration "4+ stars invalid → whole-show" rule governs the GENERALIZED grammar (partials) ONLY; the full-4 compat path is deliberately lenient for every star count. The lenient-star tolerance is scoped to THIS exact full-4 pattern; it does NOT generalize to subsets.
3. **Otherwise, find the STRICT ONLY marker.** Locate the first VALID marker `\bONLY\b(?:\s*\*{3})?(?!\s*\*)`. **A marker is valid ONLY as bare `ONLY` or `ONLY***` (exactly three stars, optional whitespace before them); a trailing 1–2 stars — `ONLY*` / `ONLY**` — is NOT a valid marker (adversarial-review R14 finding 1).** The negative lookahead `(?!\s*\*)` rejects any ONLY-token followed by a non-triple star run, mirroring the live `STAGE_TRAILING_MARKER_RE` (`personalization.ts:183-187`, which peels only `ONLY` / `ONLY***` / bare `***` and deliberately leaves `ONLY*`/`ONLY**` untouched — `***`-count tolerance is a deferred concern). If NO valid marker (ABSENT, including a malformed non-full-4 `ONLY*`/`ONLY**`) → `hasOnly = false`, `stages = []`, `unrecognizedRestriction = false`, `cleaned = roleCell` UNCHANGED, return. This is fail-safe: a malformed PARTIAL subset like `Set / Strike ONLY*` is NEVER consumed as a restriction (so it can never narrow — origin/main never narrowed a partial on a starred marker either, since its partial-subset narrowing used the star-free `LOAD_IN_SET_ONLY_PATTERN`/`LOAD_OUT_STRIKE_ONLY_PATTERN`), and `extractRoleFlags` then emits `UNKNOWN_ROLE_TOKEN` on the garbled tokens (whole-show + surfaced signal, never silent — adversarial-review R2 finding 2). Descriptive full lists w/o ONLY are handled by the retained `FULL_STAGE_PATTERN` strip in `extractRoleFlags`.
4. **ONLY present (non-full-4).** Let `body` = text between the leading dash and the ONLY marker; `tail` = text after the marker. Split `body` on `/` into segments (trimmed). **Comprehensive re-analysis of the stage-clause classification vector (adversarial-review R1 f1, R3 f1, R5 f1, R8 f1, R14 f1, R16 f1 — a recurring vector; per the AGENTS.md rule the model is now a total per-segment classification anchored on the live role recognizer, not a set of special cases).** **Classify EACH segment** into exactly one of three kinds:
   - **STAGE** — trimmed-uppercase EXACTLY equals a `STAGE_RESTRICTION_VOCAB` member (full-segment-exact so `SHOW CALLER` ≠ `SHOW`, R1 f1).
   - **ROLE** — not STAGE, and the segment yields ≥1 recognized role via the SHARED role recognizer `segmentHasKnownRole(seg)` (tokenize the segment by `/`,`-` + multi-word extraction and test each against `ROLE_NORMALIZATIONS`/`MULTI_WORD_TOKENS` — the SAME tables `extractRoleFlags` uses, pinned to one shared constant so it cannot drift). Covers `A1`, `BO`, `LEAD`, and hyphen/multi-word role segments like `BO - V1`.
   - **UNKNOWN** — neither STAGE nor ROLE (e.g. `Rehearsal`, `Sho`).

   **Structural discriminator: the presence of a STAGE segment is the SOLE signal that separates a stage restriction from a role clause — there is NO stage-word guessing/heuristic** (that would reintroduce the brittle literal lists this PR removes). Three mutually-exclusive branches:
   - **All-stages** — EVERY segment is STAGE (≥1 segment). → `stages` = segments (appearance order, deduped, canonical), `hasOnly = true`, `unrecognizedRestriction = false`, `consumedOnlyClause = true`, `cleaned = tail`. Explicit restriction.
   - **Some-stage (malformed)** — ≥1 STAGE segment but NOT all are STAGE (i.e. ≥1 ROLE or UNKNOWN segment present). → `stages = []` (NEVER partial-narrow to the recognized prefix, R3 f1), `hasOnly = true`, `unrecognizedRestriction = true`, `consumedOnlyClause = true`. **`cleaned` PRESERVES the ROLE segments (adversarial-review R16 f1): `cleaned = <the ROLE segments, joined by ' / '> + tail`.** STAGE and UNKNOWN segments are ABSORBED into the single `UNKNOWN_STAGE_RESTRICTION` (whose message embeds the full raw `roleCell`, so nothing is lost) and are NOT re-tokenized → no `UNKNOWN_ROLE_TOKEN` cascade; the recognized ROLE segments flow to `extractRoleFlags` so assignment/filtering metadata is NEVER silently dropped. → `UNKNOWN_STAGE_RESTRICTION` + whole-show. Covers `Set / Rehearsal ONLY` (no ROLE seg → `cleaned = tail`), `A1 / Set / Rehearsal ONLY` (ROLE `A1` preserved → `cleaned = "A1"`), `BO / Load In / Sho ONLY` (ROLE `BO` preserved), `Load In / Rehearsal ONLY - LEAD` (no ROLE seg in body; `- LEAD` preserved via `tail`).
   - **No-stage (ROLE clause)** — body contains ZERO recognized stage tokens. → it is a role clause regardless of whether its roles are known; `stages = []`, `hasOnly = true`, `unrecognizedRestriction = false`, `consumedOnlyClause = false`, **`cleaned = roleCell` UNCHANGED** so `extractRoleFlags` handles it with its own tokenizer (multi-word extraction, then `/` and `-` split) — PRESERVING recognized roles and emitting `UNKNOWN_ROLE_TOKEN` for unrecognized ones (adversarial-review R5 f1 hyphen decomposition, R8 f1 unknown-role preservation). Covers `A1 ONLY`, `SHOW CALLER ONLY`, `BO - V1 ONLY`, `GS - A1 ONLY` (roles preserved), `RIGGER ONLY` → `UNKNOWN_ROLE_TOKEN`, `A1 / RIGGER ONLY` → `A1` kept + `UNKNOWN_ROLE_TOKEN(RIGGER)`, `Rehearsal ONLY` → `UNKNOWN_ROLE_TOKEN` (matches `origin/main`; no stage-word guess). NO `UNKNOWN_STAGE_RESTRICTION` here.

   (A date token in `body` — `\d{1,2}/\d{1,2}` — is a leaked day restriction normally consumed upstream by `extractDayRestriction` (`crew.ts:299`); it has no recognized stage, so it falls into No-stage and is left to the existing role/day path. Defensive; not expected.)

**Structural invariant (the class-closing rule):** `UNKNOWN_STAGE_RESTRICTION` fires **iff** the body has ≥1 STAGE segment AND is not all-STAGE. Zero STAGE segments ⟹ role clause (never a stage restriction); all-STAGE ⟹ explicit restriction. **Role-preservation invariant (R16 f1):** in EVERY branch, a recognized ROLE segment reaches `extractRoleFlags` — via `cleaned = roleCell` unchanged (No-stage) or via `cleaned = <ROLE segments> + tail` (Some-stage) — so no branch can silently drop assignment metadata. This subsumes SHOW CALLER (no stage → role), hyphenated `BO - V1` (no stage → role), `RIGGER`/`Rehearsal` (no stage → role/UNKNOWN_ROLE_TOKEN), `Set / Rehearsal` (some-stage, no role → UNKNOWN_STAGE_RESTRICTION only), and `A1 / Set / Rehearsal` (some-stage WITH a leading role → UNKNOWN_STAGE_RESTRICTION + role A1 kept). A dedicated unit test asserts both invariants directly, parameterized over the decision table.

**Marker-validity enumeration (asterisk sub-vector, R14 f1 — pinned as a parameterized unit test in §8).** Marker validity is gated FIRST, orthogonally to the all/some/no-stage branch — an invalid marker collapses to the ABSENT branch (`hasOnly = false`) regardless of body:

| ONLY-suffix | Valid marker? | Disposition |
|-------------|---------------|-------------|
| `ONLY` (0 stars) | ✅ yes | marker consumed; body classified all/some/no-stage |
| `ONLY*` (1) | ❌ no | ABSENT → whole-show + role path (`UNKNOWN_ROLE_TOKEN`) |
| `ONLY**` (2) | ❌ no | ABSENT → whole-show + role path |
| `ONLY***` (3) | ✅ yes | marker consumed; the `***` day-emphasis is absorbed (crew.ts triple-asterisk, suppressed via `consumedOnlyClause`) |
| `ONLY****` (4+) | ❌ no (generalized) / ✅ full-4 only | GENERALIZED grammar: ABSENT → whole-show + role path (safe). FULL-4 carve-out (step 2): explicit 4-stage (verbatim live regex; anti-expansion, R19 f3) |

Rationale: exactly mirrors the live `STAGE_TRAILING_MARKER_RE` contract (`ONLY` / `ONLY***` / bare `***`), so de-literalization does not widen marker tolerance beyond origin/main. **Exception — the full-4 lenient-star carve-out (step 2, R17 f1):** this enumeration governs the GENERALIZED grammar (step 3+); the EXACT-ORDER full physical-4 list `Load In / Set / Strike / Load Out ONLY\*{0,3}` is matched FIRST (step 2) and DOES accept ANY trailing star count (`ONLY*`/`ONLY**`/`ONLY***`/`ONLY****`+) as an explicit 4-stage restriction — the VERBATIM live `FULL_STAGE_ONLY_PATTERN` (whose `\*{0,3}` matches the `***`-prefix of `****`). A full-4 restriction excludes pure Show days, so demoting ANY star count to whole-show would EXPOSE Show days (R17 f1); a `(?!\*)` guard is deliberately NOT added (R19 f3). The lenient star is scoped to that one full-4 pattern; partial subsets (`Set / Strike ONLY*`) still fall to ABSENT/whole-show.

Note: `Sho` (typo of `Show`) is NOT auto-corrected by `normalizeStageWords` (its `STAGE_VOCAB` is the 4 physical stages, no `Show` — §3.3), so `Set / Sho ONLY` is some-stage-malformed (safe `UNKNOWN_STAGE_RESTRICTION`); extending stage-word typo correction to `Show` is out of scope.

**`extractStageRestriction` becomes:**

```ts
export function extractStageRestriction(roleCell: string): {
  restriction: StageRestriction;
  warnings: ParseWarning[];
  consumedOnlyClause: boolean; // true iff a VALID (bare ONLY or ONLY***) stage clause was parsed (explicit OR unrecognized)
} {
  const clause = parseStageClause(roleCell);
  if (clause.stages.length > 0 && clause.hasOnly) {
    return { restriction: { kind: "explicit", stages: clause.stages }, warnings: [], consumedOnlyClause: true };
  }
  if (clause.unrecognizedRestriction) {
    return {
      restriction: { kind: "none" },
      warnings: [{
        severity: "warn",
        code: "UNKNOWN_STAGE_RESTRICTION",
        message: `Role cell has a work-phase restriction we couldn't read: '${roleCell}'`,
        rawSnippet: roleCell,
      }],
      consumedOnlyClause: true,
    };
  }
  return { restriction: { kind: "none" }, warnings: [], consumedOnlyClause: false };
}
```

Signature change: `extractStageRestriction` now returns `{restriction, warnings, consumedOnlyClause}` (the warnings mirror `extractDayRestriction`'s `DayRestrictionResult` shape, `:68-73`). ## 4. Part C — Short-header typo tolerance (deferred P3 pickup)

### 4.1 Current behavior (cited)

`normalizeSectionHeaders` (`lib/parser/sectionHeaderNormalize.ts:44-97`) fuzz-corrects only three LONG headers — `LONG_SECTION_VOCAB = ["TRANSPORTATION", "EVENT DETAILS", "GS DETAILS"]` (`:16`) — behind a gate that accepts a label-only row OR a row with ≥1 field-header word (`:70-73`). The short routers `CREW`/`TECH`/`HOTEL`/`VENUE` were explicitly **deferred** as P3 in `docs/superpowers/specs/2026-06-27-parser-typo-tolerance-design.md` (§out-of-scope: "P3 (short section routers CREW/TECH/HOTEL/VENUE behind the field-band gate …)").

Consequence (audit #5): a one-edit typo in a short header — `TCEH` — silently vanishes the TECH grid; a `CRWE` typo instead trips the loud MI-3/4/5 hard fail. This part corrects the typo in a pre-pass so TECH parses AND, for CREW, pre-empts the cryptic MI hard-fail with a clean auto-correct + warning. (HOTEL/VENUE typos are also silent per the audit, but their header rows lack the field-header corroboration this gate needs — see §4.2 — so they are deferred to a per-router follow-up.)

### 4.2 New behavior

Add a `SHORT_SECTION_VOCAB` corrected behind gating STRICTER than the long-vocab path. The gates are: `minLen` + `exclude` (both real `gatedVocabCorrect` opts, `lib/parser/typoGate.ts:3-8`) + a **caller-side field-band gate** (`countFieldHeaderWords(otherCells) ≥ 1`, already at `sectionHeaderNormalize.ts:73`) + `noExactSpellingElsewhere` (caller, `:75-76`).

```ts
const SHORT_SECTION_VOCAB = ["CREW", "TECH"] as const;
```

Scoped to the two short routers whose REAL header rows in the corpus carry `SECTION_FIELD_HEADER_WORDS` so the field-band gate actually fires (adversarial-review R7 finding 2): `CREW | NAME | ROLE | PHONE …` (`NAME`/`ROLE`/`PHONE` are field-header words) and `TECH | PHONE | ARRIVAL | DEPARTURE` (`PHONE`). **`HOTEL` and `VENUE` are intentionally excluded**: their real header rows are `HOTEL | RESERVATION #1 | | RESERVATION #2` and (v1) `VENUE | Four Seasons …` — NO cell is a `SECTION_FIELD_HEADER_WORD`, so the field-band gate would NEVER fire and a `HOTLE`/`VENEU` typo would be silently left uncorrected. Shipping them in the vocab would be a hollow promise. (This narrows the informal "CREW/TECH/HOTEL/VENUE/DATES" list from the feature brief to the two routers the gate can actually serve; noted in the ship report + backlog.)

But the Damerau-1 PLURALS of the short vocab — `CREWS` (CREW+S) and `TECHS` (TECH+S) — are NOT in any registry (verified: `rg CREWS lib/parser` → no match), so nothing currently blocks `CREWS | NAME | ROLE | PHONE` from autocorrecting to `CREW`. **Concrete code change:** add `const SHORT_SECTION_VOCAB_EXCLUDE = ["CREWS", "TECHS"] as const;` and pass `{ minLen: 4, exclude: [...EXCLUDE, ...SHORT_SECTION_VOCAB_EXCLUDE] }` to the SHORT-vocab `gatedVocabCorrect` call (the LONG-vocab call keeps its current `{ exclude: EXCLUDE }`). This makes the plural guard live-code, not an asserted-but-absent registry claim. `TECH`↔`TEAM` is Damerau-2 (safe without a guard). `DATE`/`DAY`/`ROOM` sub-labels remain in `EXCLUDE` harmlessly.

**Region mapping:** extend `CANON_TO_REGION` (`:18-22`) with the two short headers, using the canonical `RegionId` values from `REGION_ANCHOR_SPEC` (`lib/sheet-links/buildSheetDeepLink.ts:61-123`): `CREW`→`crew`, `TECH`→`crew` (the `crew` region header is `/^(CREW|TECH)$/i`, `:76`). Both are valid `RegionId` union members (`REGION_IDS`, `:29-44`).

## 9. Disagreement-loop preempts (for the adversarial reviewer)

- **Fail-open posture on unreadable clauses is intentional.** A some-stage clause (`Set / Rehearsal ONLY`) → whole-show + `UNKNOWN_STAGE_RESTRICTION`; a no-stage clause (`Rehearsal ONLY`) → whole-show + `UNKNOWN_ROLE_TOKEN` (role clause). Both are whole-show + warning (NOT zero-days, NOT hard-fail), mirroring the existing `unknown_asterisk` fallback (`crew.ts:347-362`) and safer than hiding a show. Cited: §1 out-of-scope (Rehearsal) + the `effectiveViewerDateRestriction` zero-days trap (`stageSchedule.ts:66`). **Stage-token presence is the sole stage-vs-role discriminator — no stage-word heuristic (see §3.2 structural invariant); do not relitigate.**
- **Shape-recognized rooms are NOT signaled.** Deliberate (brainstorming chose "replace", declined "always-signal"). A newly-recognized room is visible on the review surface by its presence. §1 out-of-scope.
- **`minLen: 4` for the short vocab** (vs the established `minLen: 5` for field-alias fallback) is deliberate — `CREW`/`TECH` are 4 chars. The caller-side field-band corroboration (`countFieldHeaderWords ≥ 1`, `:73`) + `noExactSpellingElsewhere` + `EXCLUDE` are the compensating gates. Cited: typo-tolerance design §4.3 deferred these *behind the field-band gate*, which is exactly this gate. **Part C vocab is CREW/TECH only** — HOTEL/VENUE header rows carry no `SECTION_FIELD_HEADER_WORD` so the field-band gate can never fire for them (R7 f2); shipping them would be a hollow promise, so they are deferred to a per-router follow-up. `DATES` also excluded (P4, region-less, already-loud MI-3). See §4.2.
- **Room de-literalization is an admit-then-emit split, not a single predicate.** dims-only blocks are ADMITTED (to preserve dims via merge — R7 f1) but a room is EMITTED only when its key-group has a field-bearing block (so equipment with dims is dropped — R6 f1). `dimsInHeader` in the emit gate would reintroduce the equipment false-positive; the emit gate is field-evidence, not `roomHasContent` (which counts dims). Do not relitigate.
- **The `consumedOnlyClause` flag on `extractStageRestriction`** exists solely to suppress the pre-existing `crew.ts` triple-asterisk `UNKNOWN_DAY_RESTRICTION` when an `ONLY***` stage clause already consumed the marker (R7 f3) — it is not a new restriction concept.
- **`UNKNOWN_STAGE_RESTRICTION` is BOTH counted (data-gap) AND anchored, by design.** It mirrors `UNKNOWN_ROLE_TOKEN`/`UNKNOWN_DAY_RESTRICTION`, which each appear in BOTH `GAP_CLASSES` and `OPERATOR_ACTIONABLE_ANCHORED` (`dataGaps.ts:36-37,164-165`) — a crew-role sheet-data gap Doug can fix at a resolvable cell. Membership in both sets is intentional and pinned by the completeness meta-test (§7 item 8); do not relitigate as double-classification. (R13 f1.)
- **`STAGE_VOCAB` (typo vocab, `:174`) stays 4-wide; `STAGE_RESTRICTION_VOCAB` (grammar, new) is 5-wide.** Two different vocabularies for two different jobs (spelling correction vs clause recognition). Not a contradiction.
- **Some-stage preserves recognized ROLE segments; only STAGE+UNKNOWN segments are absorbed (R16 f1).** The per-segment classifier (STAGE/ROLE/UNKNOWN) makes role preservation a structural property of EVERY branch — `cleaned` always routes recognized roles to `extractRoleFlags`. A leading role on a malformed stage clause (`A1 / Set / Rehearsal ONLY`) keeps `A1` (origin/main did too, while cascading `UNKNOWN_ROLE_TOKEN`s). The unreadable token (`Rehearsal`) is not double-warned — it is embedded in the `UNKNOWN_STAGE_RESTRICTION` message. Do not relitigate whether the whole body should be consumed; role loss on the degraded path is the exact regression this fix closes.
- **Marker tolerance is NOT widened, and full-4 compat is NOT dropped (R14 f1 + R17 f1).** For the GENERALIZED grammar the valid ONLY-marker forms are exactly `ONLY` and `ONLY***` — identical to the live `STAGE_TRAILING_MARKER_RE` (`personalization.ts:183-187`); `ONLY*`/`ONLY**` are invalid on PARTIAL subsets (fail-safe to whole-show), and origin/main never narrowed a partial on a starred marker (its partial narrowing used the star-free `LOAD_IN_SET_ONLY_PATTERN`/`LOAD_OUT_STRIKE_ONLY_PATTERN`). SEPARATELY, the EXACT full-4 list keeps origin/main's VERBATIM lenient `FULL_STAGE_ONLY_PATTERN` `ONLY\*{0,3}` (step 2 carve-out) → ANY trailing star count (`ONLY*`…`ONLY****`+) on the full-4 list stays an explicit 4-stage restriction (excludes Show days). The full-4 carve-out sets `cleaned = prefix + tail` (excises only the matched span), so a leading role (`A1`) is kept AND an unknown leading token (`RIGGER`) still emits `UNKNOWN_ROLE_TOKEN` — matching origin/main (R18 f2 + R19 f2). Both directions mirror origin/main exactly; neither the generalized-marker strictness NOR the full-4 lenience is a behavior change. Do not relitigate `***`-count tolerance for partial subsets (deferred); do not relitigate demoting the full-4 lenient star or adding a `(?!\*)` guard (either would EXPOSE Show days — R17 f1 / R19 f3); do not relitigate the full-4 prefix preservation (R19 f2).
- **Corpus-wide rooms no-op is the room contract**, not "looks right." De-literalization must be deep-equal to `origin/main` `rooms` on ALL committed fixtures; a bogus room anywhere fails. Discovery/terminator gate is name-shape + immediate-field-evidence (dims-alone rejected — R6). This whole-corpus regression is the structural defense that closed the three-round false-positive vector; do not relitigate the dims-alone branch.
- **Room ADMIT = a proper NON-dims-leading NAME on line 1 PLUS a `DAY N` marker anywhere OR second-line dims (R18 f1 + R19 f1 + R20 f1 + R21 f1).** This matches the committed east-coast signatures EXACTLY: `MABEL 1&#10;DAY 1 & 2` (DAY second-line), the single-line `LAUDERDALE 1, 2, 3 DAY 1 & 2` (DAY inline, commas), and novel `NAME&#10;DIMS`. The DAY marker is checked ANYWHERE (LAUDERDALE is single-line, R21 f1); dims stay second-line-only (equipment leads with dims). It rejects all fabrication shapes: field-beneath (single-line note), bare `&#10;` (multi-line note), and dims-LEADING equipment (`4' X 8' RISER`). The de-literalized loop is a DROP-IN for `mabelRe` — SAME `firstLine`-verbatim name / dims / merge / `seen` — so the corpus no-op holds by construction for MABEL/LAUDERDALE. Do not relitigate admitting on field-beneath, bare `&#10;`, or any-dims-token; do not require a second-line-only DAY (would drop the inline-DAY LAUDERDALE); do not ask for any of these to be backlogged.
- **Same-key repeated-header termination is origin/main-identical and lossless (R15 f1).** Pass 2 extracts EACH admitted member of a key-group with its OWN `extractBoBlock` call and unions the results via `mergeBoFields` — exactly as the live `mabelRe` loop already does for `MABEL 1` dims/fields split blocks (`rooms.ts:826-843`). A same-key sibling header therefore terminates only the PRIOR member's block; the later field-bearing block is extracted independently and merged, so no same-room data is lost. The block's own starting header never self-terminates (the existing positional `k > 0` guard, `extractBoBlock:888-894`, is preserved verbatim). This is not a new terminator model — it is the de-literalized form of the one already shipping; do not relitigate per-member extraction vs a single flat block scan.
