# M12.4 — Admin dashboard + settings fidelity (cosmetic batch)

**Status:** implemented; gates in progress.
**Owner:** Opus / Claude Code (UI work — invariant: UI is always Opus).
**Source:** a second live review of the validation deploy against the design
bundle (`.validation-local/design-admin/fxav-crew-pages/project/` — `Admin
Redesign.html` + `screenshots/`), following M12.3. Eight cosmetic/layout items.

This is a fidelity batch, not a feature change: no data-model, RPC, or auth
surface changes. The one data-layer touch is additive (a timestamp field carried
onto an existing in-memory item type).

## Items

### Dashboard

- **D1 — Header divider.** `AdminPageHeader` gains a full-width hairline below
  the title block (`border-b border-border pb-6`), matching the design's header
  rule. Shared component → applies to every admin page header (dashboard,
  settings, per-show), which the design also shows.
- **D2 — Full-width shell.** The admin layout container cap goes
  `max-w-6xl` (1152px) → `max-w-[1600px]` so dashboard content fills the desktop
  viewport. Bounded at 1600px to avoid over-stretch on ultra-wide. Below 1152px
  the geometry is unchanged (the cap never bound there).
- **D3 — Needs-attention timestamps.** Each card shows a relative activity
  timestamp ("1h ago") pinned top-right. `NeedsAttentionItem` carries a new
  `activityAt: string | null` (sourced from the existing sort key —
  `pending_ingestion` → `last_attempt_at`; sync variants → `staged_modified_time`).
  Rendered via the shared `formatRelative`; the `<time>` is **omitted** when
  `activityAt` is null (never the bare "never" placeholder). `NeedsAttentionInbox`
  now takes `now: Date` (mirrors `ShowsTable`).
- **D4 — Find shares the toggle row.** The "Active shows" title, the Find input,
  and the Active/Archived segmented control share ONE header row. `ShowsTable`
  owns that header for the active bucket (the Find client-state lives there) and
  receives the control as a `bucketControl: ReactNode` prop; the archived bucket
  keeps its own title+control row in `Dashboard`.

### Settings

- **S1 — Drive health badge.** The Drive-connection health readout is a tinted
  pill badge (glyph + label) instead of a bare status dot: green "Healthy" when
  positive, amber "Needs attention" otherwise. Tints use `color-mix` over the
  existing `@theme` status tokens — no new tokens, no inline hex.
- **S2 — Explainer tooltip on non-healthy.** Every non-healthy badge carries a
  `HelpTooltip` ("?") disclosing a plain-language, reason-specific explainer
  (not-configured / watch lapsed / sync stale / sync_unknown / infra). The
  positive badge has no tooltip. Copy is plain language (same register as the
  panel's existing helper text), not an error code.
- **S3 — Buttons on the description row.** "Open folder" + "Re-run setup" sit on
  the same row as the helper text (already coded `justify-between`; the S4 width
  change gives them room to stop wrapping).
- **S4 — Full-width, left-aligned settings.** The settings page drops its
  `max-w-[740px]` cap for `w-full` (left-aligned), matching the design.

## Dimensional invariants (D2/D4 — fixed-parent layout)

The dashboard two-col split (`min-[1080px]:flex-row`) keeps a usable
`minmax(0,1fr)` title track in the shows column. The inbox column is **320px
(`w-80`) through the tight split band (1080–1279px)** and widens to **360px only
at ≥1280px** where the now-uncapped layout has slack. This keeps the worst-case
overhead at the 1080px activation width unchanged (~862px), so the title grid
track stays above the 120px floor at every band. Verified in a real browser
(`tests/e2e/admin-layout-dimensions.spec.ts` TITLE_BANDS up to 1280px; also
measured locally: title track 162px @1080 → 482px @1440, header overlap −16px at
every band).

## Guard conditions

- `activityAt === null` → no `<time>` rendered (no "never" leak). Covered by a
  NeedsAttentionInbox test (with-time + null-time rows in one render).
- `ShowsTable.title` defaults to "Active shows" (the component is the
  active-bucket renderer) so existing callers/tests need no churn.
- Drive health `deriveHealthExplainer` narrows the union: `"kind" in health` →
  infra copy; `positive` → "" (never shown — the tooltip only renders on
  non-healthy); otherwise reason-specific.
