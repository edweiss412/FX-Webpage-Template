# M12.2 RECON-1 — Admin AlertBanner quieting (design)

**Date:** 2026-06-02 · **Milestone:** M12.2 (admin redesign), close-out blocker RECON-1 · **Owner:** Opus + impeccable v3 dual-gate (UI invariant 8) · **Routing:** UI → Opus.

**Source of need:** M12.2 whole-milestone design-fidelity reconciliation (`.validation-local/reconciliation/RECONCILIATION-POST-IMPL.md`) found the global admin alert banner was never quieted to the prototype's compact treatment — it still renders the M5-era "loud" block on every admin route. Tracked as `DEFERRED.md` → RECON-1. This spec defines the quieting. It supersedes the visual treatment only; the alert data model, catalog, RLS, and fail-visible contract are unchanged.

## 1. Goal & non-goals

**Goal:** Replace the loud, tall `AlertBanner` block with a calm single-row strip that is collapsed by default and expands in place to reveal full context — matching prototype `01-cards2.png` / `settings2.png` while preserving every behavioral contract the current banner carries.

**Non-goals:** No change to the alert data model, `admin_alerts` schema/RLS, the §12.4 catalog, alert resolution semantics, the queue/detail surface at `/admin#alerts`, or which alerts are selected (top unresolved non-info, raised_at DESC). No new persisted settings/flags. Not folded into B3 (email scope).

## 3. Target design

### 3.1 Structure (approved: "C strip + expand" with "B count badge")

A calm amber strip. **Collapsed by default**; a "Details" toggle expands an in-place panel. Same `warning-bg`/`warning-text`/`border-border-strong` tokens, reduced vertical weight (single row when collapsed).

**Disclosure mechanism — native `<details>/<summary>` (no-JS first).** The collapse/expand is a server-rendered native `<details>` element; `<summary>` is the always-visible collapsed row, and the expanded panel is `<details>`'s remaining content. This makes the full message, help, raised-at, and "+N more" link present in the SSR DOM and reachable **without any JavaScript / before/without hydration** (the operator can always open the disclosure and reach the queue link). Animation is a progressive enhancement layered on top (§6); function never depends on it. **The exact DOM/CSS tree is normative in §3.3** — the `<section>` is a CSS grid; `<details>` is column 1, and the action is a **section-grid sibling of `<details>` in column 2** (NOT inside `<summary>`, NOT inside `<details>`, NOT in a flex container with `<summary>`). Interactive controls inside `<summary>` would conflict with its native toggle, and a non-sibling action would either drift on expand or break no-JS reachability — §3.3 resolves all three.

**Collapsed row** = the `<summary>` content only (column 1 of the §3.3 section grid). The action slot is the separate column-2 grid sibling (see below), visually on the same row but not in the `<summary>` and not in a shared flex container. Inside `<summary>` (itself a `flex items-center gap-3 min-w-0` row):
1. Leading warning icon — lucide-react `TriangleAlert` (the admin icon convention; cf. `components/admin/settings/DriveConnectionPanel.tsx`, `components/admin/nav/NotifBell.tsx`), decorative, `aria-hidden`, sized to the row.
2. Message — catalog `dougFacing` via the existing render path, truncated to one line (`truncate`, flex-grow, `min-w-0`).
3. "N alerts" count badge — **only when** `unresolvedCount > 1`; hidden for a single alert. Count from `fetchUnresolvedAlertCount()` (reuses `moreCount` source; badge shows total = `moreCount + 1`). **Bounded format (F11):** the count is clamped for display — any total ≥ 100 renders as `99+` (and the "+N more →" link likewise caps at `99+`), so the badge's glyph width is bounded (max ~5 chars, e.g., "99+ alerts") regardless of queue size; this keeps the §3.3 `fit-content(55%)`/non-overlap contract intact even for very large queues. (The `/admin#alerts` queue page, out of scope here, still shows the exact number.)
4. Disclosure caret/label ("Details ⌄" → "Hide ⌃") — the `<summary>`'s own affordance; the caret rotates via CSS off `details[open]` (no JS).

Action slot (column-2 section-grid sibling of `<details>`, §3.3; visually on the collapsed row) — always visible. **The slot is a complete server-rendered node, not a bare button:**
   - Global alert: the **entire** `<form action={resolveAdminAlertFormAction}>` containing the hidden `id` input (`name="id"`, the alert UUID) **and** `<ResolveAlertButton />` — exactly as wrapped today (`AlertBanner.tsx:285-288`). The form boundary is load-bearing: `ResolveAlertButton` reads `useFormStatus().pending` from its parent form, and the hidden `id` is what the Server Action consumes. The form is server-rendered **directly into the column-2 action cell** (§3.3) — there is no required client island around it; nothing may split the button from its form, drop the hidden `id`, or interpose a client boundary that severs the `useFormStatus` parent relationship.
   - Per-show alert: the server-rendered "View show" `<a>` (`AlertBanner.tsx:271-277`).

**Expanded panel** (revealed below the row, separated by `border-t border-border` inside the same section):
- Full (un-truncated) message.
- "What does this mean?" disclosure + "Learn more →" — the existing `<HelpAffordance>`, moved into the panel.
- Meta row: "Raised <relative>" (left) + "+N more →" link to `/admin#alerts` (right) — only when `moreCount > 0`.
- Toggle label flips to "Hide ⌃".

### 3.2 Server/client split (progressive enhancement)

`AlertBanner` stays a **server component** that renders the entire banner — `<details>`, `<summary>`, the full message, `<HelpAffordance>`, the meta row, the "+N more" link, and the action node — into SSR HTML. **Function requires no client component:** the native `<details>` disclosure works with JS disabled, before hydration, and even if a client enhancement fails. Copy stays catalog-sourced (invariant 5); no raw codes anywhere.

Progressive enhancements (all optional, all degrade to the working no-JS baseline):
- **Route/alert remount boundary (`AlertBannerRouteBoundary`, F9+F17+F19):** a thin `'use client'` wrapper that keys its subtree by **`${usePathname()}?${useSearchParams().toString()}:${alertId}`** — `usePathname()` excludes the query string, so `useSearchParams()` is included too (F19) because admin UI drives state via query params (`?alert_id`, `?step=`, `?bucket=`); a same-pathname query-only nav must still remount. The native `<details>` thus remounts collapsed on a path change, a query change, OR an alert change. It receives the server-rendered banner as `children` (passed through opaquely — no copy/codes cross into the client). Degrades safely: with no JS, every navigation is a full page load that already renders fresh-collapsed, so the boundary only matters under client-side nav. `alertId` is the one scalar prop it needs (not catalog copy).
- The open/close height-morph animation (§6) — CSS-first (`details` + `interpolate-size`/grid-rows technique), or a tiny optional client sprinkle; absence → instant native toggle, still fully functional.
- `ResolveAlertButton` is already a client component for its two-tap UX; it is rendered by the server **inside its `<form action={resolveAdminAlertFormAction}>` with the hidden `id`**, exactly as today, placed as the action sibling in the collapsed row. **Slot-integrity rule:** nothing in the quieting may relocate the button out of its form, split it from the hidden `id`, or interpose a wrapper that severs the `useFormStatus` parent relationship. If any client wrapper is introduced for animation, the `<form>`+button subtree is passed through as an opaque child — never re-mounted or destructured.

No new client data fetch; the count badge + "+N more" are server-rendered from `fetchUnresolvedAlertCount()`.

### 3.3 Explicit DOM/CSS tree (normal state)

The action must be (a) outside `<summary>` (no native toggle conflict), (b) kept with the native `<details>` so the panel is no-JS reachable, (c) visually pinned to the collapsed row even when expanded, AND (d) the expanded panel must span the **full** banner width (not be trapped in a narrow column, F18). A **CSS grid on the `<section>` with `<details class="contents">` (display:contents)** satisfies all four: `<summary>`, the action, and the panel become direct grid items, so `<summary>` = column 1 / row 1, the action = column 2 / row 1 (together they form the collapsed row), and the panel = column 1 / span 2 / row 2 (full width, below the row). `self-start` on the action keeps it pinned to the collapsed-row top; the panel opening adds row 2 and never shifts the action. `display:contents` does not change native `<details>` toggle semantics, and the UA rule `details:not([open]) > :not(summary){display:none}` still hides the panel when closed.

**Track sizing (mobile-safe, pins F6 + F10):** column 1 is `minmax(0,1fr)` (NOT `1fr`) with `min-w-0` on `<details>`/`<summary>`, so the message truncates to near-zero. **The cap is on the TRACK, not the item:** column 2 is `fit-content(55%)` — the track itself is bounded to ≤55% of the grid (so an `auto` track can't reserve the action's max-content width and compress/overlap column 1); the action cell stays `flex flex-wrap justify-end` so the two-tap **confirm** pair (`Confirm resolve` + `Cancel`) wraps to a second line *within column 2* (growing the row downward) rather than overflowing or overlapping at 390px. Idle/pending (single button) stay one line. Because grid columns never overlap, summary fixed children (icon/badge/toggle) cannot paint into the action column once the track is capped this way (asserted by the §11 non-overlap bounding-box check). If at 390px the summary's `shrink-0` children still can't fit ≥45%, the plan's narrow-width fallback applies: drop the "Details" word to caret-only and/or the badge to a bare count — tuning kept green by the §11 390px non-overlap + overflow assertions.

```
{/* Wrapped by <AlertBannerRouteBoundary alertId={alert.id}> (§3.2): keyed `${pathname}?${searchParams}:${alert.id}`
    → remounts fresh-collapsed on route change incl. query-only nav (F17/F19) OR alert change (F9). children = this server-rendered section. */}
<section data-alert-id={alert.id}
         data-testid="admin-alert-banner" role="status" aria-live="polite" aria-atomic="true"
         class="grid grid-cols-[minmax(0,1fr)_fit-content(55%)] items-start gap-x-3 rounded-md border border-border-strong
                bg-warning-bg p-tile-pad text-warning-text mb-section-gap">
  {/* <details> uses display:contents (F18) so <summary>, the action, and the panel all participate in the
      SECTION grid directly: summary = col 1 / row 1, action = col 2 / row 1 (the collapsed row), panel = full
      width (col 1 / -1) / row 2. Native toggle semantics are unaffected by display:contents, and the UA rule
      `details:not([open]) > :not(summary){display:none}` still hides the panel when closed. */}
  <details class="contents">
    <summary class="col-start-1 row-start-1 min-w-0 flex items-center gap-3 cursor-pointer list-none
                    [&::-webkit-details-marker]:hidden">   <!-- THE collapsed row content, col 1 / row 1 -->
      <TriangleAlert aria-hidden />
      <span class="flex-1 min-w-0 truncate">{message}</span>
      {unresolvedCount > 1 && (
        <span class="shrink-0">
          <span aria-hidden="true">{formatBoundedCount(unresolvedCount)} alerts</span>   {/* visible, bounded (F11) */}
          <span class="sr-only">{unresolvedCount} unresolved alerts</span>                {/* exact, exposed to AT (F14/F16) */}
        </span>
      )}
      <span class="caret shrink-0">Details ⌄  /* CSS swaps to "Hide ⌃" when [open] */</span>
    </summary>
    <div data-testid="admin-alert-panel"
         class="col-start-1 col-span-2 row-start-2 min-w-0 mt-3 border-t border-border pt-3">  <!-- FULL WIDTH (F18) -->
      {full (un-truncated) message}
      <HelpAffordance code params />                  <!-- What does this mean? + Learn more → -->
      <div class="flex justify-between gap-2 min-w-0">
        <span>Raised {relative}</span>
        {moreCount > 0 && <Link href="/admin#alerts" aria-label={`View ${moreCount} more unresolved alerts`}>+{formatBoundedCount(moreCount)} more →</Link>}  {/* visible bounded (F11/F12); aria-label keeps EXACT count, preserving AlertBanner.tsx:261 (F14) */}
      </div>
    </div>
  </details>

  <div data-testid="admin-alert-action"
       class="col-start-2 row-start-1 self-start min-w-0 flex flex-wrap justify-end gap-2">
    <!-- action: grid item at col 2 / row 1 (the collapsed row), sibling of the summary/panel via details' display:contents.
         min-w-0 + flex-wrap let the confirm pair STACK so the cell's MIN-CONTENT = widest single control (not the pair
         side-by-side), keeping the fit-content(55%) track's min-content floor well under 55% (F13). -->
    {/* global  */ <form action={resolveAdminAlertFormAction}><input type="hidden" name="id" value={alert.id} /><ResolveAlertButton /></form>}
    {/* per-show */ <a href={`/admin/show/${slug}?alert_id=${alert.id}`}>View show</a>}
  </div>
</section>
```

Properties this guarantees: the action `<form>`/`<a>` is NOT a descendant of `<summary>` → clicking it can never toggle the disclosure (even with no JS), and toggling `<summary>` can never activate it; the panel stays inside native `<details>` → reachable no-JS, AND spans the full banner width via `col-span-2` row 2 (F18); `self-start` keeps the action on the collapsed row (row 1) in both states; `minmax(0,1fr)` + `min-w-0` (col 1) + the `fit-content(55%)` capped track (col 2) keep the collapsed row 390px overflow-free **and overlap-free** in every action state; the `AlertBannerRouteBoundary` (keyed `${pathname}?${searchParams}:${alertId}`, §3.2) remounts a fresh collapsed `<details>` on alert change (F9) or any route/query change (F17/F19). The **degraded** variant (§4) is a separate, simpler render branch (no `<details>`, no action) — a plain row with the fixed copy + "View alerts →".

**`formatBoundedCount(n)`** (shared helper, used by both the badge and the "+N more" link): returns `String(n)` for `n < 100`, else `"99+"`. This bounds every count's glyph width (F11/F12) so neither the collapsed badge nor the expanded queue link can widen unboundedly.

**On `fit-content(55%)` (F13 — it is NOT an absolute cap):** CSS `fit-content(L)` resolves to `min(max-content, max(min-content, L))` — it still **floors at the track's min-content**, so the cap only holds if the action cell's min-content stays below 55%. The design guarantees that: the action cell is `min-w-0 flex flex-wrap`, so the `Confirm resolve`/`Cancel` controls **stack** when space is tight → the cell's min-content equals the **widest single control** (~one button), not the pair side-by-side, which is comfortably under 55% of 390px. (The exact `fit-content(...)` percentage and any narrow-width badge/toggle fallback are tuning details the plan may adjust, provided the §11 computed-column-width + overflow + non-overlap assertions stay green.)

## 5. Guard conditions

- `unresolvedCount` infra-error / unavailable → treat as 1 (badge hidden, "+N more" hidden); the top alert still renders. Mirrors current `moreCount` collapse-to-0 (`AlertBanner.tsx:175`).
- `unresolvedCount <= 1` → no badge, no "+N more" link.
- Per-show alert with missing slug → no "View show" link (current behavior, `AlertBanner.tsx:185,278`); panel/expand still available.
- `HelpAffordance` returns null (unknown code / null `dougFacing` / no `helpHref`) → expanded panel simply omits that block; raised-at/"+N more" still render. Expanding a panel whose only content would be empty is still valid (full message always present).
- Degraded variant → never expandable (no specific alert to detail).
- Default expand state = **collapsed on every mount** (no persistence); SSR renders `<details>` without `open`. Fresh-collapse is guaranteed on alert change AND any route/query change by the `AlertBannerRouteBoundary` keyed `${pathname}?${searchParams}:${alertId}` (§3.2/§3.3, F9+F17+F19); no-JS navigation collapses natively via full page load.

## 6. Transition inventory

Four states: **collapsed (C)**, **expanded (E)**, **degraded (D)**, **null (N)**. Only C↔E is a within-request UI transition (native `<details>` open/close); the rest are **server re-render swaps across requests** (a fresh page render), instant by nature.

**Ephemerality principle (load-bearing) + two reconciliation caveats (F9, F17):** `expanded` is native `<details open>` state owned by the **browser/DOM**, not React, so a re-render does NOT auto-reset it. Two distinct remount triggers are needed, because the banner is mounted in the **persistent** `app/admin/layout.tsx` (which Next.js does NOT remount on client-side navigation between admin routes):
- **Alert identity change (F9)** — resolve→next alert, new alert, read failure: after `revalidatePath`, React can reconcile the *same* `<details>` node in place and preserve `open` (alert B shows expanded). Fixed by **keying the remount boundary on the alert id**.
- **Same-alert route navigation (F17)** — operator expands on `/admin`, client-navigates to `/admin/settings` while the same alert stays topmost: the persistent layout is NOT remounted and the alert id is unchanged, so an alert-id-only key cannot reset it. Under **no-JS** this is a non-issue (every navigation is a full document load → fresh collapsed). Under **client-side nav** it needs a route-derived remount.

**Mechanism:** wrap the server-rendered banner in the thin `'use client'` boundary `AlertBannerRouteBoundary` whose remount key is **defined once in §3.2** (the single source of truth — this paragraph does NOT restate the literal). It remounts (collapsing the native `<details>`) on a path change, a query-string change, OR an alert change. The banner content is passed through as server-rendered `children` (copy/codes stay server-side, no raw codes in the client). With it, every path — toggle, resolve→next, new alert, read failure, **and route/query change** — lands in C/D/N freshly collapsed, no stale `open`/height. (`data-alert-id` retained for tests/observability; the §3.2 key drives the remount.)

All N×(N−1)/2 = 6 unordered pairs:

| Pair | Trigger | Treatment |
|---|---|---|
| C ↔ E | user toggles `<summary>` | **animated** height-morph at `--duration-fast` (`app/globals.css:176`, 120ms); caret/label swap instant. Native toggle is instant if the animation enhancement is absent. |
| C ↔ D | alert read fails (→D) / recovers (→C) | server re-render, **instant**; lands collapsed-fresh on recovery |
| C ↔ N | top alert resolved & none left (→N) / new alert arrives (→C) | server re-render, **instant**; N = `return null`, no chrome/region |
| E ↔ D | read fails while expanded | server re-render, **instant**; renders fresh **degraded** — expanded state NOT carried over (ephemerality principle) |
| E ↔ N | last alert resolved while expanded | server re-render, **instant**; renders fresh **null** (no chrome) — expanded NOT carried over |
| D ↔ N | degraded read clears to empty / vice-versa | server re-render, **instant** |
| **E(alert A) → C(alert B)** — alert-identity change | top alert A is expanded then resolved, but alert B remains queued → banner re-renders for B in the same slot | server re-render; the **`AlertBannerRouteBoundary` (`${pathname}?${searchParams}:${alertId}` key) forces a fresh collapsed `<details>` for B** (F9) — B must NOT inherit A's browser-owned `open`/height. |
| **E → C on same-alert route/query change** — operator expands, then client-navigates to another admin route OR a query-only change (`?bucket=`, `?alert_id=`, `?step=`), same alert topmost | client-side nav; persistent layout NOT remounted | the boundary's `${pathname}?${searchParams}` key changes → remounts collapsed (F17/F19). No-JS: full page load collapses natively. |

Cross-cutting:
- **`prefers-reduced-motion`:** C↔E is instant — `--duration-fast` already resolves to `0ms` under reduced-motion (`app/globals.css:337`), so reusing that token auto-satisfies it; still assert explicitly. (Animating native `<details>` to auto height needs `interpolate-size`/grid-rows `1fr↔0fr`, not opacity.)
- **Live region:** the banner keeps `role="status" aria-live="polite" aria-atomic="true"` (`AlertBanner.tsx:200-202`); degraded keeps `role="status" aria-live="polite"` (`:147`); N renders no region. Server swaps (C/D/N) change the region's content as one atomic announcement — verify no duplicate/stale announcement and no orphaned height classes after a swap.
- **Compound — expand while `ResolveAlertButton` mid-confirm/resolving:** independent state; toggling `<details>` must not reset the two-tap 3s timer, and resolving must not force-close `<details>`. Verify both.

## 7. Dimensional invariants

Tailwind v4 here does **not** default `.flex` to `align-items: stretch` (DESIGN §7) and has no global `md`. Explicit:

- `<summary>` is `flex items-center`; its icon, count badge, and Details caret/label each stay vertically centered and never exceed the row height (`shrink-0` on each; message is the only `flex-1 min-w-0 truncate` child). The action lives in grid column 2 (not in this flex), pinned per the grid contract below.
- Message truncation: inside `<summary>` the message cell is the sole flexible child; icon/badge/toggle are `shrink-0` so a long message truncates rather than pushing them out.
- **Grid contract is defined ONCE in §3.3 (single source of truth); this section references it, never restates the class literal.** Per §3.3 the section is a CSS grid with a **shrinkable first track `minmax(0,1fr)` + `min-w-0`** on `<details>`/`<summary>`, and a **`fit-content(55%)` capped second track** for the action (the track cap — not just `max-w` on the item — is what prevents 390px overflow AND summary/action overlap, F10). Via `<details class="contents">` (F18), `<summary>` = col 1 / row 1 and the action = col 2 / row 1 (the collapsed row), while the panel = col 1 / span 2 / row 2 (**full width** below the row, `border-t`). The action is `self-start` so it stays pinned to row 1 and does NOT shift when the panel opens (verified §11 test (d)). The banner is wrapped by `AlertBannerRouteBoundary` (keyed `${pathname}?${searchParams}:${alertId}`, §3.2) for fresh-collapse on route/query/alert change (F9/F17/F19). The non-shrinkable `grid-cols-[1fr_auto]` track is FORBIDDEN (reintroduces F6); the §11 grep guard enforces this.
- Action tap target ≥ `min-h-tap-min`/`min-w-tap-min` (unchanged from current).
- **Action state matrix (the global two-tap confirm).** `ResolveAlertButton` grows from one `Resolve` button (idle) into two controls — `Confirm resolve` + `Cancel` (confirm state, `ResolveAlertButton.tsx:59-61`) — and a `pending` state. The compact-row contract must hold across all three, and this is the exact regression RECON-1 removes, so it is explicitly in scope:
  - **Steady state (idle + pending):** the collapsed row is always exactly one line. Mechanism (§3.3): column 1 is `minmax(0,1fr)` with `min-w-0` on `<details>`/`<summary>` and the message is the sole `flex-1 min-w-0 truncate` sink, so it truncates first; the column-2 action is `shrink-0` (single button).
  - **Transient confirm state (user-initiated):** at the narrowest widths the `Confirm resolve` + `Cancel` pair may not fit on one line. Mechanism (§3.3): the message truncates first; if the pair still cannot fit, it wraps to a second line **within the `fit-content(55%)` column 2** (action cell `flex flex-wrap justify-end`), growing the strip downward only — never overflowing horizontally, clipping, or overlapping column 1. This is a documented, intentional exception, acceptable because the confirm state is transient, user-initiated, and auto-reverts after 3s (`ResolveAlertButton.tsx`).
- Real-browser (Playwright) assertion required in the plan: at a band sweep incl. the 720px boundary **and the smallest mobile width (390)** with a **long-message** fixture, assert for **each action state (idle, confirm, pending)** and **both single- and multiple-alert (badge present/absent)**: (a) in idle/pending the collapsed row is exactly one line (height ≈ single-row); (b) no horizontal overflow or clipping in any state; (c) badge/toggle/action share the row's vertical center within 0.5px in idle/pending; (d) in the confirm state, either still one line OR a clean downward wrap with no overflow (per the exception above); (e) **non-overlap (F10):** the bounding boxes of the `<summary>` fixed children (icon, badge, Details toggle) do not intersect `[data-testid="admin-alert-action"]` in any state — proves the `fit-content(55%)` track cap holds, not just that the document lacks horizontal overflow. jsdom is insufficient.

## 8. Invariants preserved

- **Invariant 5 (no raw codes):** all copy via `lib/messages/lookup` (`ErrorExplainer`/`HelpAffordance`/`getRequiredDougFacing`); the banner is server-rendered text, not codes. The `tests/**/no-raw-codes` scanner still applies.
- **Accessibility — exact queue count (F14/F16):** bounding is visual-only; assistive tech keeps the exact backlog size, exposed via mechanisms that actually reach the a11y tree:
