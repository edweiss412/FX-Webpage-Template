## 0. Problem statement & scope

Today a show's lifecycle is half-built:

- **Auto-publish** of a clean first-seen sheet works (amendment 9): the sheet goes live, mints a 24h `unpublish_token`, raises `SHOW_FIRST_PUBLISHED`.
- **Unpublish** works as a 24h token-gated undo (`lib/sync/unpublishShow.ts`): it sets `archived=true, published=false`, clears the token, raises `SHOW_UNPUBLISHED`.
- **Archive is one-way and side-effect-only** — it happens *only* as a consequence of unpublish. There is **no admin Archive action**, **no Unarchive/Restore path at all**, and **no manual Publish action**.
- The dashboard **filters archived shows out entirely** (`components/admin/Dashboard.tsx:110,130`), so a retired show vanishes with no way to browse or restore it.
- Several mutation paths can reach an archived show server-side, bypassing the read-only UI gating Phase A shipped (the **DEF-1/2/3** deferrals, plus a fourth path this spec adds).
- There is **no operator control** over whether clean first-seen sheets auto-publish; amendment 9 makes it unconditional.

B2 closes the lifecycle:

1. Explicit admin **Archive** for any show (not just the 24h auto-publish window).
2. **Unarchive** → a safe **Held** state, and a manual **Publish** to bring a Held show back to Live.
3. An **Archived dashboard bucket** (segmented Active / Archived control).
4. An **auto-publish-clean toggle** in settings, defaulting to today's behavior.
5. **Four archived-immutability guards** (DEF-1/2/3 + the cron/push path).

**In scope:** the items above + their schema, catalog codes, and tests.

**Out of scope (→ B3):** the email-delivery subsystem and any opt-in toggles for it. B2 *emits* the events B3 will deliver (§4.4, §12) but ships **no email code** — that would be a zombie until B3 lands.

**Out of scope (unchanged):** the 24h token Unpublish mechanics (already shipped — B2 only re-points its core mutation at the shared archive routine, §2.5), the wizard finalize flow, the picker/identity-selection internals.

## 1. The lifecycle state model

Every show carries two independent booleans — `shows.published` (default `true`, `supabase/migrations/20260501000000_initial_public_schema.sql:26`) and `shows.archived` (default `false`, `:25`) — plus a derived **finalize-owned** predicate (a wizard/pending-changes finalize checkpoint owns the row; `lib/sync/runManualSyncForShow.ts:98-132`). They compose into four operator-meaningful states:

| State | `published` | `archived` | finalize-owned | Crew reachable? | Origin |
|---|---|---|---|---|---|
| **Live** | `true` | `false` | no | **Yes** (with active share token) | auto-publish, Publish action, or staged Apply |
| **Publishing…** | `false` | `false` | **yes** | No | wizard finalize in flight (exists today) |
| **Held** (a.k.a. Draft) | `false` | `false` | no | No | **NEW** — where a show lands after Unarchive |
| **Archived** | `false` | `true` | — | No | Archive action, or 24h token Unpublish |

Crew reachability is enforced at exactly one gate — `lib/auth/picker/resolvePickerSelection.ts:87`: `if (showRow.archived || !showRow.published) return { kind: "show_unavailable" }`. Both `archived` and `!published` independently deny access; `resolveShowPageAccess.ts:109` selects the same two columns. **`picker_epoch` is NOT part of the access gate** — it governs picker identity-selection staleness only.

### 1.1 Transition inventory

All state-changing transitions B2 owns. Each row is a discrete admin or system action; every action acquires the per-show advisory lock (§2.1).

| # | From → To | Trigger | Mutation (summary) | User-facing code |
|---|---|---|---|---|
| T1 | Live → Archived | admin **Archive** (two-tap) | `archived=true, published=false, archived_at=now()`; clear unpublish token; **rotate share token + bump `picker_epoch`** | `SHOW_ARCHIVED_BY_ADMIN` (info) |
| T2 | Held → Archived | admin **Archive** (two-tap) | same as T1 (token already absent/rotated; rotate is idempotent-safe) | `SHOW_ARCHIVED_BY_ADMIN` (info) |
| T3 | Live → Archived | 24h token **Unpublish** | shared archive core via `unpublishShow` (existing entry point, §2.5) | `SHOW_UNPUBLISHED` (info, existing) |
| T4 | Archived → Held | admin **Unarchive** (one-tap) | `archived=false, archived_at=null`; `published` stays `false` | `SHOW_UNARCHIVED` (info) |
| T5 | Held → Live | admin **Publish** (one-tap) | `published=true` | `SHOW_PUBLISHED_BY_ADMIN` (info) |
| T6 | (new sheet) → Held-equivalent-staged | auto-publish **OFF** + clean first-seen | route to `pending_syncs` staged review (no `shows` row yet) | `SHOW_AWAITING_PUBLISH_APPROVAL` (info) |
| T7 | (staged) → Live | admin **Apply** on a T6 row | existing staged-Apply → publishes with 24h undo | `SHOW_FIRST_PUBLISHED` (info, existing) |

**Refused transitions** (guarded; §5): Archive on a Publishing… (finalize-owned) row → `FINALIZE_OWNED_SHOW`; Archive on an already-Archived row → no-op (idempotent success, §2.2); any mutation (rotate/reset, apply/discard, manual re-sync, cron/push) on an Archived row → `SHOW_ARCHIVED_IMMUTABLE` (or silent skip for cron/push, §5 DEF-4); Publish on an Archived or finalize-owned row → refused (`SHOW_ARCHIVED_IMMUTABLE` / `FINALIZE_OWNED_SHOW`).

### 1.2 Compound / interleaving transitions

- **Stale-tab Archive after the row already changed** — Archive re-reads `archived`/`published`/finalize-ownership inside the lock; if the row is already archived it returns idempotent success (T1/T2 both converge), if finalize-owned it returns `FINALIZE_OWNED_SHOW`. No partial write.
- **Cron/push edit arrives while Archived** — silent skip (§5 DEF-4); watermark frozen.
- **Unarchive then immediate cron** — once `archived=false`, the frozen watermark means the next cron/push observes `modifiedTime > last_seen_modified_time` and re-syncs the latest sheet content into the Held row (content refresh; does **not** auto-publish — the show is Held, not Live, and the routine-update path doesn't flip `published`).
- **Archive during a staged-review window** — the show's `pending_syncs` staged row becomes unapplyable (DEF-2 guard returns `SHOW_ARCHIVED_IMMUTABLE` on Apply); Phase A already renders such rows read-only in the per-show ParsePanel (`app/admin/show/[slug]/page.tsx:10-11`).

## 2. Lifecycle actions

### 2.1 Shared invariant — per-show advisory lock (single-holder)

Every mutation in §2.2–§2.6 runs inside `pg_advisory_xact_lock(hashtext('show:' || drive_file_id))` (blocking / admin path) at exactly one layer. The new RPCs (`archive_show`, `unarchive_show`, `publish_show`) acquire the lock **in-RPC** (mirroring `rotate_show_share_token` at `supabase/migrations/20260523000004:28` and `reset_picker_epoch_atomic` at `20260523000003:30`); their JS callers MUST NOT also wrap them in `withShowLock` (single-holder rule, AGENTS.md invariant 2). The topology is pinned in `tests/auth/advisoryLockRpcDeadlock.test.ts` (§8).

**Shared archive primitive (single-holder discipline).** The Archive mutation set is shared by the admin Archive RPC (§2.2) and the 24h-token Unpublish path (§2.5). To avoid a double-lock or auth-mismatch (R1 finding), it is factored into a SECURITY DEFINER SQL routine `_archive_show_core(p_show_id)` that **acquires NO advisory lock and performs NO `is_admin()` check** — it is documented and tested as "callable only from an already-locked AND already-authorized context." Its two callers each hold the lock exactly once: (a) `archive_show(p_show_id)` = `is_admin()` gate + in-RPC `pg_advisory_xact_lock` + `_archive_show_core`; (b) the token Unpublish path = the existing JS-side `withShowLock` (the single holder) + token validate/consume, then `_archive_show_core` in the same transaction (no second lock, token-authorized rather than admin-authorized). `_archive_show_core` is `revoke`d from `public, anon, authenticated, service_role` and granted to nobody callable directly via PostgREST (invoked only from the two SECURITY DEFINER wrappers / the locked pipeline txn). It appears in the §8 advisory-lock topology matrix as a lockless inner routine.

### 2.2 Archive (admin)

**Entry point:** the per-show admin page (`app/admin/show/[slug]/page.tsx`) — there is room there for the two-tap confirm + the links-dead warning. (The dashboard row links to the per-show page for archive; it does not carry its own inline archive button.)

**Precondition (server-side, re-read inside the lock):** allowed when **not finalize-owned**. Allowed on Live and Held. On an already-Archived row → idempotent success (returns the archived state, no second rotate). On a finalize-owned row → refused with `FINALIZE_OWNED_SHOW`.

**Mutation (the body of `_archive_show_core`, invoked by the `archive_show` RPC after its `is_admin()` gate + in-RPC lock — §2.1):**

```
update shows set
  archived = true,
  published = false,
  archived_at = now(),
  unpublish_token = null,
  unpublish_token_expires_at = null
where id = p_show_id;
-- rotate the crew link so previously-distributed URLs are permanently dead:
update show_share_tokens set share_token = encode(extensions.gen_random_bytes(32),'hex') where show_id = p_show_id;
update shows set picker_epoch = picker_epoch + 1, picker_epoch_bumped_at = clock_timestamp() where id = p_show_id;
-- kick active crew sessions NOW (mirrors rotate/reset, which call this — 20260523000004:42 / 20260523000003:38):
perform public.publish_show_invalidation(p_show_id);
```

**Active-session invalidation (R3 finding).** The `archived`/`!published` access gate (`resolvePickerSelection.ts:87`) denies *new* resolutions, but an **already-open** crew page holds subscribed state and would not stop until it refreshes. The live realtime triggers fire on `crew_member_auth`/`crew_members`, not on `shows`/`show_share_tokens`, so an UPDATE to `shows` alone does NOT push an invalidation. Therefore `_archive_show_core` MUST call `public.publish_show_invalidation(p_show_id)` (`supabase/migrations/20260503000000_publish_show_invalidation_helper.sql`) after the state/token/epoch change — exactly as `rotate_show_share_token` (`:42`) and `reset_picker_epoch_atomic` (`:38`) do — so active viewers are invalidated immediately. This is the literal mechanism behind the owner's "crew links stop working **now**" requirement.

The `unpublish_token`/`unpublish_token_expires_at` pair is nulled **together** (preserves the paired CHECK from `20260512082710_add_show_unpublish_token.sql:14-22`). Token rotation is what enforces the owner requirement that *old links must not silently resurrect on revival* — after a later Unarchive→Publish the show is Live on a fresh token the operator never distributed.

**Confirm gesture (UI):** two-tap, matching `components/admin/AlertBanner.tsx` Resolve. Tap 1 → the button morphs to a confirm affordance whose label explicitly states the consequence: **"Confirm archive — crew links stop working now and won't come back until you re-publish and issue a new link."** Tap 2 → dispatches. A 4s idle reverts the confirm to the resting state. The disable-on-`isPending` discipline applies (a self-disabling submit cancels the React 19 form-action dispatch — see B1's revoke-hang lesson; disable on `isPending`, never synchronously in the click handler).

**Success confirmation (UI):** no toast (none exists in the project — only two `// future toast` comments). The form action resolves and `router.refresh()` re-renders the show into its Archived presentation (Archived pill, moved into the dashboard's Archived segment, ParsePanel read-only with a persistent "crew links are dead" note). The state change is the confirmation. **No `admin_alert` row is created** for an admin's own deliberate archive. Consequently `SHOW_ARCHIVED_BY_ADMIN` (and likewise `SHOW_UNARCHIVED`, `SHOW_PUBLISHED_BY_ADMIN`) is an **action-result / persistent-disclosure copy code**, rendered inline via `messageFor()` — it is **NOT** an `admin_alerts` producer and is **NOT** registered in the admin-alert catalog meta-test. **B2 introduces no NEW `admin_alert` codes** — the OFF-path awaiting-approval event (`SHOW_AWAITING_PUBLISH_APPROVAL`) is a per-staged `pending_syncs` inbox row + B3 email hook keyed by `staged_id`, not an `admin_alert` (§4.4). (The OFF→Apply path *reuses* the **existing** `SHOW_FIRST_PUBLISHED` alert producer for the post-publish notice — a reuse, not a new code; §4.3.)

**Persistent disclosure (UI):** the archived show's per-show page and its Archived-segment row both render a persistent, plain-language note: **"This show is archived. Crew links are dead. Unarchive and re-publish to bring it back."** This is a rendered element (not just a description) — see §3.3 dimensional invariants.

### 2.3 Unarchive (admin)

**Entry points:** the Archived-segment row on the dashboard (§3) and the archived show's per-show page.

**Precondition:** the row is `archived=true`. (No finalize-ownership conflict is possible — an archived row is not finalize-owned.)

**Mutation (RPC `unarchive_show(p_show_id)`):** `archived=false, archived_at=null`. `published` stays `false` → the show lands in **Held**. The share token is **not** touched — it was already rotated to a fresh, undistributed value, either by the Archive action (B2 rows) or by the migration backfill (legacy pre-B2 archived rows, §6.1). So no previously-distributed crew URL can ever resurrect on revival, regardless of when the show was archived.

**Gesture (UI):** one tap — Unarchive is safe (it exposes nothing; the Held state is crew-unreachable via the `!published` gate). Code `SHOW_UNARCHIVED`.

**Post-unarchive disclosure (UI):** the now-Held show renders a "Held — not published. Publish to make it live, then issue a crew link." note with the **Publish** affordance adjacent.

### 2.4 Publish (admin) — NEW

This action did not exist; it is required by the choice that Unarchive lands in a Held state rather than going straight to Live.

**Precondition (re-read inside the lock):** allowed only when `published=false AND archived=false AND not finalize-owned` (the **Held** state). On an Archived row → `SHOW_ARCHIVED_IMMUTABLE`. On a finalize-owned row → `FINALIZE_OWNED_SHOW`. On an already-`published` row → idempotent success.

**Mutation (RPC `publish_show(p_show_id)`):** `published=true`, then `perform public.publish_show_invalidation(p_show_id)` (so any client still showing the "unavailable" state re-resolves and any cached picker state refreshes). Nothing else — in particular it does **not** mint a 24h `unpublish_token` (that token is the auto-publish safety net for sheets that went live *without* review; a deliberate manual Publish of a revived show needs no undo grace). (The show's content is already applied — Held shows are dark, not empty.) Code `SHOW_PUBLISHED_BY_ADMIN`. (`unarchive_show` does NOT invalidate — the show stays unreachable in Held, so there is no stale active session to kick.)

**Crew access after Publish:** the `archived || !published` gate now passes, so the show is reachable **via its current share token**. Because that token was rotated at archive, the operator must distribute/copy the current crew link (or rotate again) — the existing per-show share/rotate controls, gated on `published && !archived` (`app/admin/show/[slug]/page.tsx:230`), become available again exactly here.

**Gesture (UI):** one tap. Publishing a Held show does not by itself expose anything (the share link is undistributed), so no two-tap is required.

### 2.5 Unpublish (24h token) — UNCHANGED behavior, refactored core

The existing `POST /api/show/[slug]/unpublish?token=` path and its `SHOW_UNPUBLISHED` framing are unchanged from the operator's view. Internally, `lib/sync/unpublishShow.ts`'s `archiveAndConsumeUnpublishToken` (`:123-133`) is refactored to delegate to the same archive core as §2.2 (so token rotation + `picker_epoch` bump are added to the token-unpublish path too, closing the same old-link-resurrection gap). The token consumption guard (`and unpublish_token = $2::uuid`, `:132`) is preserved. `SHOW_UNPUBLISHED` / `UNPUBLISH_TOKEN_CONSUMED` / `UNPUBLISH_TOKEN_EXPIRED` are unchanged (`lib/messages/catalog.ts:665,675,685`).

### 2.6 Guard conditions for the action inputs (null/empty/missing)

- **Missing/garbage `slug`** at any action route → the show lookup returns no row → `ADMIN_LINK_SHOW_NOT_FOUND` (NB: this code is referenced in the master spec but **not yet present in `catalog.ts`** — B2 adds it via three-lockstep, §7).
- **`p_show_id` not found** in any RPC → the RPC raises the show-not-found path (mirrors `rotate_show_share_token` `:19-26`) → `ADMIN_LINK_SHOW_NOT_FOUND`.
- **Double-submit / stale tab** → idempotent success (Archive on archived, Publish on published) or a guarded refusal; never a partial write (all mutations are single-statement-equivalent inside one locked txn).
- **`show_share_tokens` row absent** for the show (shouldn't happen for a published show, but defensively) → the rotate UPDATE matches 0 rows; the archive still sets `archived/published` and the access gate still denies. The RPC does not error on a 0-row token rotate.

## 3. Dashboard archived bucket (segmented Active / Archived)

### 3.1 Layout & behavior

A two-state segmented control — **Active** (default) / **Archived (N)** — sits atop the show list in the dashboard (`components/admin/Dashboard.tsx`). The **needs-attention inbox stays pinned above** both segments regardless of selection (it already spans all shows — `:298-299`). The selected segment is a **URL search param** (`?bucket=archived`), so back/forward and refresh behave and the RSC re-fetches server-side.

- **Active segment** = `archived=false` (today's query at `:110,:130`, unchanged). Includes Live, Publishing…, and **Held** rows (Held rows carry a distinct pill, §3.2).
- **Archived segment** = `archived=true`, ordered by **`archived_at DESC NULLS LAST, id`** (deterministic). Each row is **read-only** except for an **Unarchive** action and an "Open" link to the per-show page. Each row shows the title, an **Archived pill**, and the archived-time line: "Archived {relative archived_at}" when `archived_at` is set, else the fallback copy **"Archived (date unknown)"**. **Pre-B2 archived rows** (retired via the current unpublish path before B2 shipped) have `archived=true` but **`archived_at = NULL`** — there is no reliable historical archival timestamp on `shows` (it has only `created_at`/`last_synced_at`, neither of which is the archive moment), so B2 does **not** fabricate one: those rows sort last (NULLS LAST) and render "Archived (date unknown)". (The plan MAY optionally best-effort-backfill `archived_at = coalesce(last_synced_at, created_at)` for existing archived rows as an approximation, but the contract is the null-safe ordering + fallback copy, so a null `archived_at` is always handled.)

Both segment **counts** are always computed (the inactive label needs its count), via two `count`-only queries (`archived=false` and `archived=true`); the row list is fetched only for the selected segment.

**Guard conditions:** Archived count `0` → the "Archived (0)" segment still renders but is disabled/muted (no empty mode to fall into); selecting it shows an empty-state line "No archived shows." A `null` count from an infra fault → the dashboard's existing `infra_error` path owns it (the segment control is not rendered in the infra-error branch).

### 3.2 Held-state pill

A Held row (Active segment, `published=false && archived=false && !finalize-owned`) renders a **"Held — not published"** pill (neutral/idle treatment) distinct from the existing "Publishing…" warn pill (which is finalize-owned, `app/admin/show/[slug]/page.tsx:243`). The per-show status-pill precedence becomes: `archived` → "Archived"; else finalize-owned `!published` → "Publishing…"; else `!published` → "Held"; else "Published". (Extends the existing archived-FIRST precedence at `:241-243`.)

### 3.3 Dimensional invariants (real-browser Playwright assertions required)

This project's Tailwind v4 does **not** default `.flex` to `align-items: stretch`, and has **no global `md` breakpoint** (use `min-[Npx]:` utilities). Each relationship below must be guaranteed by an explicit class/style and verified by a real-browser `getBoundingClientRect()` assertion (jsdom is insufficient):

| Parent → child | Invariant | Guaranteeing mechanism |
|---|---|---|
| Segmented control track → each segment button | both segments share the control's full height | `items-stretch` on the track + `h-full` on each button |
| Segmented control → list region | the active list fills the content column width at every swept viewport | parent width is the column; no child max-width cap |
| Archived row → Unarchive action + Archived pill | the action/pill stay within the row height; no overflow | row `items-center`; pill/action `self-center` |
| Two-tap confirm button (archive) | the morphed confirm label does not change the button's box height (no layout shift on tap 1) | fixed `min-h`/`min-w` sized to the longer label |

Viewport sweep for the layout-dimensions task: `[600, 719, 720, 860, 1024, 1280]` (covers the project's `min-[720px]:` boundary band, per the B1 band-sweep lesson).

| Pair | Treatment |
|---|---|

## 4. auto-publish-clean toggle

### 4.1 Storage

A new column on the singleton settings row: `app_settings.auto_publish_clean_first_seen boolean not null default true` (`app_settings` is the `id='default'` singleton, `supabase/migrations/20260501001000_internal_and_admin.sql:232-247`). **Default `true`** preserves amendment-9 behavior exactly; existing installs are unchanged on migration.

### 4.2 Scope

The toggle gates **only the first-seen new-show publish moment** — a brand-new clean sheet going Live without review. It does **not** gate routine clean *updates* to already-published shows (those keep auto-applying; the show is already Live and the update refreshes content).

### 4.3 Effect when OFF

A clean first-seen sheet (all MI-1..MI-14 pass) that would auto-apply via Phase 2 instead routes to **staged review**: a `pending_syncs` row with the sentinel **`FIRST_SEEN_AWAITING_APPROVAL`** in `triggered_review_items`, surfaced per-staged in the dashboard needs-attention inbox as "new show, ready to publish — approve?" (copy `SHOW_AWAITING_PUBLISH_APPROVAL`). Doug clicks **Apply** → it publishes with the **same 24h undo token + `SHOW_FIRST_PUBLISHED` notice as the auto-publish-ON path** (parity is the whole point — OFF just inserts a human approval gate before the same outcome). No `shows` row is created until Apply (so OFF does **not** create Held rows — Held arises only from Unarchive, §1). The awaiting-approval *staging* event is keyed by `pending_syncs.staged_id` (NOT an `admin_alert`, §4.4) so multiple waiting sheets stay independently visible; the *post-Apply* notice reuses the existing `SHOW_FIRST_PUBLISHED` producer.

**Apply-path wiring (R6/R7 findings — comprehensive re-analysis of the publish-notice path).** The `SHOW_FIRST_PUBLISHED` notice + 24h token are NOT emitted by `runPhase2`. They are emitted by the exported **`emitSuccessfulPhase2Tail`** (`runScheduledCronSync.ts:1624-1652`), which — when handed an `autoPublishFirstSeen: {unpublishToken, unpublishTokenExpiresAt}` payload — calls `emitFirstPublishedNotice` (`:1601-1622`: the `SHOW_FIRST_PUBLISHED` upsert + token context) plus `publishShowInvalidation` + `logSync`. The cron auto-publish path calls this tail; **`applyStaged` does NOT** — after `runPhase2` it only does `bumpReviewerAuthFloors` + `insertSyncAudit` + `deleteLivePendingSync` + return (`applyStaged.ts:1342-1355`). So passing `autoPublishFirstSeen` to `runPhase2` is insufficient — the notice/token live in the *tail*.

**Required wiring:** when `applyStaged` applies a staged row carrying the `FIRST_SEEN_AWAITING_APPROVAL` sentinel (a first-seen apply, no pre-existing `shows` row), after a successful `runPhase2` it MUST invoke the **same exported `emitSuccessfulPhase2Tail`** with `autoPublishFirstSeen: {unpublishToken, unpublishTokenExpiresAt = now()+24h}` — reusing the cron path's exact notice logic (no duplication, no new alert code). **Structural defense (R7 — vector recurred R6→R7):** the 24h `unpublish_token` is persisted **only** as part of the single `autoPublishFirstSeen` payload that the tail consumes — there is no separate token-mint statement — so "token written but no `SHOW_FIRST_PUBLISHED`" is impossible by construction: either the tail runs with the payload (both token + notice + invalidation) or it doesn't (neither). A parity meta-test pins that **every** code path that sets `autoPublishFirstSeen` routes through `emitSuccessfulPhase2Tail` (so cron and Apply can never diverge on this again).

When **ON** (default): unchanged — Phase 2 auto-applies, mints the 24h token, raises `SHOW_FIRST_PUBLISHED`.

### 4.4 Event for B3 (no email code in B2)

When OFF routes a clean first-seen sheet to staged review, the staged `pending_syncs` row carries a dedicated sentinel **`FIRST_SEEN_AWAITING_APPROVAL`** in `triggered_review_items` (a clean first-seen sheet held only because the toggle is off — distinct from the MI-tripped first-seen sentinels; conceptually the conditional re-enablement of the amendment-9-retired `FIRST_SEEN_REVIEW`, §4.6). This event is **keyed by `pending_syncs.staged_id`, NOT an `admin_alert`** (R4 finding):

- **In-app:** it surfaces per-staged in the existing needs-attention inbox, which reads `pending_syncs` **directly** and classifies each row as `first_seen`/`existing_staged` (`lib/admin/needsAttention.ts:6`) — so **two** waiting clean first-seen sheets show as **two distinct inbox rows**. `SHOW_AWAITING_PUBLISH_APPROVAL` is the user-facing **copy** for such a row.
- **Why NOT an `admin_alert`:** `admin_alerts` has a unique index `(coalesce(show_id::text,''), code) where resolved_at is null` (`supabase/migrations/20260501001000_internal_and_admin.sql:279-280`). Since no `shows` row exists yet (decision 3A — staged, not Held), `show_id` would be `null`, so every awaiting-approval sheet would collapse onto the single conflict key `('', 'SHOW_AWAITING_PUBLISH_APPROVAL')` — the second pending show would overwrite the first. Keying off `pending_syncs.staged_id` avoids the collapse entirely and needs no change to the alert uniqueness model.
- **B3 email hook:** B3's delivery reads the awaiting-approval `pending_syncs` rows (sentinel `FIRST_SEEN_AWAITING_APPROVAL`, keyed by `staged_id`) and emails per-show, gated by B3's opt-in.

**B2 ships the producer (the per-staged sentinel + inbox copy); B3 ships the delivery.** B3's opt-in taxonomy must cover "show awaiting approval," not only "sync problems" — flagged here so B3's spec accounts for it. **B2 introduces no NEW `admin_alert` codes** (the awaiting-approval *staging* event is the per-staged `pending_syncs` sentinel, not an alert). The OFF→Apply path **reuses the existing `SHOW_FIRST_PUBLISHED` `admin_alert` producer** for the post-publish notice (§4.3) — same as the ON path; that is a reuse, not a new code (R4/R6 resolution).

## 5. Archived-immutability guards (DEF-1/2/3 + DEF-4)

The unifying principle: **an archived show is immutable everywhere** — admin actions and the sync engine alike. Each guarded path re-reads the show inside its advisory-locked transaction and refuses before any write. The shared user-facing code is `SHOW_ARCHIVED_IMMUTABLE` (Doug-facing: *"This show is archived. Unarchive it before making changes."*). DEF-4 is the exception (silent skip, not a user-facing refusal — the sync engine has no user in the loop).

| ID | Path | File:line (current) | Today | B2 guard |

| **DEF-1** | `rotate_show_share_token` + `reset_picker_epoch_atomic` RPCs | `20260523000004:13-26`, `20260523000003:15-28` | `is_admin()` + show-exists only | reject `archived` → `SHOW_ARCHIVED_IMMUTABLE`; reject finalize-owned → `FINALIZE_OWNED_SHOW` (they lack this today, unlike re-sync). **Observability of DEF-4 skips.** The archived skip is **silent** — it reuses the existing `perFileProcessor` skip-outcome path (`{outcome:"skip", reason:"archived"}`), which the caller short-circuits without mutation, exactly like the `deferred_permanent`/`deferred_modtime` skips (`perFileProcessor.ts:174,178`) that write no per-pass record. **It does NOT write to `sync_audit`** — that table is an apply-path audit with required `staged_id`/`triggered_review_items`/`reviewer_choices`/`derived_side_effects`/`parse_result_summary`/`staged_modified_time` columns and **no `reason` column** (`supabase/migrations/20260501001000_internal_and_admin.sql:204-217`); forcing a skip row there would fabricate fake applied-audit data. If observability is later wanted, the correct sink is **`sync_log`** (which has a `status text` column, `:221`) with `status='skipped_archived'` — but B2 does **not** require a per-pass `sync_log` row (that would spam, since the watermark is intentionally not advanced and the file is re-detected each cycle); the skip is silent by contract. `SYNC_SKIPPED_ARCHIVED` is therefore a **conceptual outcome name, not a §12.4 user-visible code and not a persisted audit row**.

**DEF-4 also covers folder-removal / missing-file detection (R4 finding — a 5th mutation path).** The master-spec §5.2 *prose* says removal detection runs over `archived = false`, but the **live code does not implement that filter**: `listPostgresLiveShows` selects every row `where drive_file_id is not null` (`lib/sync/runScheduledCronSync.ts:1447`), and a row missing from the folder is routed to `markMissingShow_unlocked` (`:1782`, called at `:2318`) which mutates `shows.last_sync_status`, writes `sync_log`, and raises `SHEET_UNAVAILABLE`. So an **archived** show removed from the watched folder would still be mutated + alert-spammed — violating archived-immutability. B2 closes this: **filter `listPostgresLiveShows` to `archived = false`** (aligning the code to the §5.2 contract) **AND** re-read `archived` inside `markMissingShow_unlocked` before any update/log/alert (defense-in-depth for the archive-mid-pass race, same two-phase rationale as the per-file path). AC-B2.13c covers it.

### 5.1 PostgREST DML lockdown (RPC-gated tables)

The durable lifecycle state of a show lives in **`shows`** (`archived`/`published`/content) and **`show_share_tokens`** (the crew link). **Both are already DML-locked-down and registered** in `tests/db/postgrest-dml-lockdown.test.ts` (`RPC_GATED_TABLES` at `:139` and `:168`) — `INSERT/UPDATE/DELETE` are REVOKEd from `anon`/`authenticated`, so even an admin-authenticated caller cannot `from('shows').update({archived:false})` or mutate a share token directly via PostgREST; the only write path is a SECURITY DEFINER RPC. B2's new `archive_show`/`unarchive_show`/`publish_show`/`_archive_show_core` are *additional* RPC entry points on these already-gated tables — **no new REVOKE migration is needed for them**, and B2 must NOT add `shows`/`show_share_tokens` registry rows (they already exist). The plan's job is only to confirm these two tables remain in the registry (a regression guard) so the new RPCs are the sole lifecycle-state write path.

**Why `pending_syncs` / `pending_ingestions` are NOT locked down (deliberate).** Both carry table-level DML grants for `anon`/`authenticated` (`supabase/migrations/20260501002000_rls_policies.sql:83,91`) and are intentionally **RLS-`admin_only`-gated rather than RPC-gated** — they are *staged scratch* tables, not show-state. The DEF-2 archived guard lives in `applyStaged_unlocked`/`discardStaged_unlocked`: applying staged content mutates **`shows`** (already locked down → blocked at the table layer for direct PostgREST), so an archived show's durable state cannot be changed by bypassing the pipeline. A direct admin `from('pending_syncs').delete()` would only discard moot staged data for an already-immutable show — it cannot resurrect or mutate the show — so adding these to the RPC-gated registry is unnecessary and would in fact break the registry's "every entry has a matching REVOKE" meta-test (they have GRANTs by design). **`app_settings`** (the auto-publish toggle, §4) is likewise RLS-`admin_only` (`rls_policies.sql:134`), written by an admin-gated server action consistent with the existing folder-settings write path — RLS is the appropriate gate for an admin-only singleton, not RPC-gating. (See `reference_postgrest_dml_lockdown_classwide_pattern` for the registry + `describe.each` + Layer-4 reconciliation shape.)

## 6. Schema & migrations

### 6.1 New columns

```sql
-- migration: add lifecycle columns (idempotent: DROP ... IF EXISTS guards on any new constraint; ADD COLUMN IF NOT EXISTS)
alter table public.app_settings add column if not exists auto_publish_clean_first_seen boolean not null default true;
alter table public.shows        add column if not exists archived_at timestamptz;   -- set on archive, null on unarchive

-- R5 finding: legacy archived rows (retired via the pre-B2 unpublish path) still hold their
-- ORIGINAL distributed share_token — unpublishShow.ts:127-128 sets archived/published but never
-- rotated show_share_tokens, and resolve_show_by_slug_and_token (:14-18) matches purely on
-- slug+share_token before the archived/published gate. So a legacy archived row could resurrect
-- an old crew URL after Unarchive→Publish. One-shot backfill: rotate every existing archived row's
-- token + bump picker_epoch so legacy rows match the new archive contract (old links dead forever).
-- Safe/idempotent: archived shows are unreachable, so re-rotating is harmless.
update public.show_share_tokens t
   set share_token = encode(extensions.gen_random_bytes(32), 'hex')
  from public.shows s
 where s.id = t.show_id and s.archived = true;
update public.shows
   set picker_epoch = picker_epoch + 1, picker_epoch_bumped_at = clock_timestamp()
 where archived = true;
```

After this backfill, **every** archived row (legacy + future) has a dead distributed token, so the §2.3 "token was already rotated" assumption holds universally — Unarchive→Publish never resurrects a previously-shared URL regardless of when the row was archived.

`archived_at` is purely for display ("Archived {when}") + audit; it is nullable, set by `archive_show`, nulled by `unarchive_show`. The 30-day archived GC window (`lib/sync/diagramGc.ts:176`) already keys off the `archived` boolean and is unaffected.

## 12. Acceptance criteria

- **AC-B2.1 Archive (Live→Archived):** archive a Live show → `archived=true, published=false, archived_at` set, `unpublish_token` null, `show_share_tokens.share_token` changed, `picker_epoch` incremented; `resolvePickerSelection` returns `show_unavailable` for the old token; the `SHOW_ARCHIVED_BY_ADMIN` action-result copy renders inline AND **no `admin_alerts` row is created** (assert the alert table is unchanged). Two-tap confirm required (one tap does not mutate).
- **AC-B2.1a Active-session invalidation on Archive:** an active subscribed crew viewer of a Live show receives a realtime invalidation when the show is archived (assert `publish_show_invalidation` is invoked by `_archive_show_core`; assert the subscriber's channel receives the invalidation message). Negative-regression: remove the `publish_show_invalidation` call → the active viewer keeps stale state.
- **AC-B2.2 Archive refusals:** archive on a finalize-owned row → `FINALIZE_OWNED_SHOW`, no write; archive on an already-archived row → idempotent success, no second token rotation.
- **AC-B2.3 Unarchive (Archived→Held):** `archived=false, archived_at=null`, `published` stays `false`; crew still denied (`!published`); share token unchanged from the rotated value.
