## 0. Problem statement & scope

Today a show's lifecycle is half-built:

- **Auto-publish** of a clean first-seen sheet works (amendment 9): the sheet goes live, mints a 24h `unpublish_token`, raises `SHOW_FIRST_PUBLISHED`.
- **Unpublish** works as a 24h token-gated undo (`lib/sync/unpublishShow.ts`): it sets `archived=true, published=false`, clears the token, raises `SHOW_UNPUBLISHED`.
- **Archive is one-way and side-effect-only** — it happens *only* as a consequence of unpublish. There is **no admin Archive action**, **no Unarchive/Restore path at all**, and **no manual Publish action**.
- The dashboard **filters archived shows out entirely** (`components/admin/Dashboard.tsx:110,130`), so a retired show vanishes with no way to browse or restore it.
- Several mutation paths can reach an archived show server-side, bypassing the read-only UI gating Phase A shipped (the **DEF-1/2/3** deferrals, plus a fourth path this spec adds).
- There is **no operator control** over whether clean first-seen sheets auto-publish; amendment 9 makes it unconditional.

B2 closes the lifecycle:

1. Explicit admin **Archive** for any show (not just the 24h auto-publish window).
2. **Unarchive** → a safe **Held** state, and a manual **Publish** to bring a Held show back to Live.
3. An **Archived dashboard bucket** (segmented Active / Archived control).
4. An **auto-publish-clean toggle** in settings, defaulting to today's behavior.
5. **Five archived-immutability guards** (DEF-1/2/3 + DEF-4 cron/push/missing-file + DEF-5 pending-ingestion retry/discard routes).

**In scope:** the items above + their schema, catalog codes, and tests.

**Out of scope (→ B3):** the email-delivery subsystem and any opt-in toggles for it. B2 *emits* the events B3 will deliver (§4.4, §12) but ships **no email code** — that would be a zombie until B3 lands.

**Out of scope (unchanged):** the 24h token Unpublish mechanics (already shipped — B2 only re-points its core mutation at the shared archive routine, §2.5), the wizard finalize flow, the picker/identity-selection internals.

## 1. The lifecycle state model

Every show carries two independent booleans — `shows.published` (default `true`, `supabase/migrations/20260501000000_initial_public_schema.sql:26`) and `shows.archived` (default `false`, `:25`) — plus a derived **finalize-owned** predicate (a wizard/pending-changes finalize checkpoint owns the row; `lib/sync/runManualSyncForShow.ts:98-132`). They compose into four operator-meaningful states:

| State | `published` | `archived` | finalize-owned | Crew reachable? | Origin |
|---|---|---|---|---|---|
| **Live** | `true` | `false` | no | **Yes** (with active share token) | auto-publish, Publish action, or staged Apply |
| **Publishing…** | `false` | `false` | **yes** | No | wizard finalize in flight (exists today) |
| **Held** (a.k.a. Draft) | `false` | `false` | no | No | **NEW** — where a show lands after Unarchive |
| **Archived** | `false` | `true` | — | No | Archive action, or 24h token Unpublish |

Crew reachability is enforced at exactly one gate — `lib/auth/picker/resolvePickerSelection.ts:87`: `if (showRow.archived || !showRow.published) return { kind: "show_unavailable" }`. Both `archived` and `!published` independently deny access; `resolveShowPageAccess.ts:109` selects the same two columns. **`picker_epoch` is NOT part of the access gate** — it governs picker identity-selection staleness only.

### 1.1 Transition inventory

All state-changing transitions B2 owns. Each row is a discrete admin or system action; every action acquires the per-show advisory lock (§2.1).

| # | From → To | Trigger | Mutation (summary) | User-facing code |
|---|---|---|---|---|
| T1 | Live → Archived | admin **Archive** (two-tap) | `archived=true, published=false, archived_at=now()`; clear unpublish token; **rotate share token + bump `picker_epoch`** | `SHOW_ARCHIVED_BY_ADMIN` (info) |
| T2 | Held → Archived | admin **Archive** (two-tap) | same as T1 (token already absent/rotated; rotate is idempotent-safe) | `SHOW_ARCHIVED_BY_ADMIN` (info) |
| T3 | Live → Archived | 24h token **Unpublish** | shared archive core via `unpublishShow` (existing entry point, §2.5) | `SHOW_UNPUBLISHED` (info, existing) |
| T4 | Archived → Held | admin **Unarchive** (one-tap) | `archived=false, archived_at=null`; `published` stays `false` | `SHOW_UNARCHIVED` (info) |
| T5 | Held → Live | admin **Publish** (one-tap) | `published=true` | `SHOW_PUBLISHED_BY_ADMIN` (info) |
| T6 | (new sheet) → staged review | auto-publish **OFF** + clean first-seen (cron/push OR manual retry) | route to `pending_syncs` staged review (reused `FIRST_SEEN_REVIEW` sentinel; no `shows` row yet) | `SHOW_AWAITING_PUBLISH_APPROVAL` (inbox copy) |
| T7 | (staged) → Live | admin **Apply** on a T6 row | existing staged-Apply → publishes with 24h undo | `SHOW_FIRST_PUBLISHED` (info, existing) |

**Refused transitions** (guarded; §5): Archive on a Publishing… (finalize-owned) row → `FINALIZE_OWNED_SHOW`; Archive on an already-Archived row → no-op (idempotent success, §2.2); any mutation (rotate/reset, apply/discard, manual re-sync, cron/push) on an Archived row → `SHOW_ARCHIVED_IMMUTABLE` (or silent skip for cron/push, §5 DEF-4); Publish on an Archived or finalize-owned row → refused (`SHOW_ARCHIVED_IMMUTABLE` / `FINALIZE_OWNED_SHOW`).

### 1.2 Compound / interleaving transitions

- **Stale-tab Archive after the row already changed** — Archive re-reads `archived`/`published`/finalize-ownership inside the lock; if the row is already archived it returns idempotent success (T1/T2 both converge), if finalize-owned it returns `FINALIZE_OWNED_SHOW`. No partial write.
- **Cron/push edit arrives while Archived** — silent skip (§5 DEF-4); watermark frozen.
- **Unarchive then immediate cron** — once `archived=false`, the frozen watermark means the next cron/push observes `modifiedTime > last_seen_modified_time` and re-syncs the latest sheet content into the Held row (content refresh; does **not** auto-publish — the show is Held, not Live, and the routine-update path doesn't flip `published`).
- **Archive during a staged-review window** — to avoid a stuck inbox row (R9/R10: an archived show whose staged row can be neither applied nor discarded would linger forever in needs-attention), `_archive_show_core` **deletes the show's live (non-wizard) `pending_syncs` AND `pending_ingestions` rows inside the archive transaction** (both are inbox sources — `lib/admin/needsAttention.ts:1-7`). So the inbox row disappears at archive time; DEF-4 + the pending-ingestion route guards (§5) then prevent any new actionable row from accruing for an archived show. Phase A's read-only ParsePanel rendering (`app/admin/show/[slug]/page.tsx:10-11`) remains the belt-and-suspenders for the vanishing race window. There is no dead-end.

## 2. Lifecycle actions

### 2.1 Shared invariant — per-show advisory lock (single-holder)

Every mutation in §2.2–§2.6 runs inside `pg_advisory_xact_lock(hashtext('show:' || drive_file_id))` (blocking / admin path) at exactly one layer. The new RPCs (`archive_show`, `unarchive_show`, `publish_show`) acquire the lock **in-RPC** (mirroring `rotate_show_share_token` at `supabase/migrations/20260523000004:28` and `reset_picker_epoch_atomic` at `20260523000003:30`); their JS callers MUST NOT also wrap them in `withShowLock` (single-holder rule, AGENTS.md invariant 2). The topology is pinned in `tests/auth/advisoryLockRpcDeadlock.test.ts` (§8).

**Archive mutation core — private to the admin RPC; the token path mirrors it inline (R1 + R13 callable-boundary fix).** The Archive mutation set (set archived/published/archived_at, clear unpublish token, rotate share token + bump `picker_epoch`, delete live scratch rows, invalidate sessions — §2.2) is captured as a **PRIVATE SECURITY DEFINER routine `_archive_show_core(p_show_id)`** that acquires NO advisory lock and performs NO `is_admin()` check, and is `revoke`d from `public, anon, authenticated, service_role` so it is **not callable via PostgREST or from any JS client**. It is invoked **only internally** (via `perform`) by **`archive_show`** — the single admin wrapper that holds the lock once (`is_admin()` + in-RPC `pg_advisory_xact_lock` + `perform _archive_show_core`); a same-owner SECURITY DEFINER → SECURITY DEFINER internal call needs no external grant, so the revoke does not block it. **The 24h-token Unpublish path does NOT call `_archive_show_core`** (R13: it is a raw `DATABASE_URL` transaction + `withShowLock` in JS — `lib/sync/unpublishShow.ts:52,241` — not a SECURITY DEFINER context, so sharing the authless core across it would force an external grant and expose the primitive). Instead, the token path keeps its existing single `withShowLock` holder and **implements the identical mutation set inline** in its locked raw-SQL transaction (§2.5). Drift between the two implementations is prevented by an **end-state parity meta-test** (§8): both paths must leave a show in byte-identical archived state (archived/published/token-null/share_token-rotated/picker_epoch-bumped/scratch-cleared). `_archive_show_core` appears in the §8 advisory-lock topology matrix as a lockless inner routine, and a privilege test asserts it is not callable via PostgREST by any role.

### 2.2 Archive (admin)

**Entry point:** the per-show admin page (`app/admin/show/[slug]/page.tsx`) — there is room there for the two-tap confirm + the links-dead warning. (The dashboard row links to the per-show page for archive; it does not carry its own inline archive button.)

**Precondition (server-side, re-read inside the lock):** allowed when **not finalize-owned**. Allowed on Live and Held. On a finalize-owned row → refused with `FINALIZE_OWNED_SHOW`.

**Idempotency early-return (R12 finding).** `archive_show` re-reads `archived` inside the lock and, **if `archived = true` already, returns idempotent success WITHOUT calling `_archive_show_core`** — so a stale-tab / double-submit Archive does NOT re-rotate the share token or re-bump `picker_epoch` (which would churn the future crew link). `_archive_show_core`'s mutation set below runs **only on the `false → true` transition**. (AC-B2.2 asserts a second Archive leaves `share_token` + `picker_epoch` unchanged.)

**Mutation (the body of `_archive_show_core`, invoked by `archive_show` ONLY when the row was not already archived, after its `is_admin()` gate + in-RPC lock — §2.1):**

```
update shows set
  archived = true,
  published = false,
  archived_at = now(),
  unpublish_token = null,
  unpublish_token_expires_at = null
where id = p_show_id;
-- rotate the crew link so previously-distributed URLs are permanently dead:
update show_share_tokens set share_token = encode(extensions.gen_random_bytes(32),'hex') where show_id = p_show_id;
update shows set picker_epoch = picker_epoch + 1, picker_epoch_bumped_at = clock_timestamp() where id = p_show_id;
-- R9/R10 finding: clear the show's live staged scratch (BOTH inbox sources — pending_syncs AND
-- pending_ingestions) so it can't sit forever in the needs-attention inbox with no allowed action.
-- Wizard-owned rows (wizard_session_id IS NOT NULL) are untouched (archive is refused on finalize-owned shows).
delete from pending_syncs      where drive_file_id = (select drive_file_id from shows where id = p_show_id) and wizard_session_id is null;
delete from pending_ingestions where drive_file_id = (select drive_file_id from shows where id = p_show_id) and wizard_session_id is null;
-- kick active crew sessions NOW (mirrors rotate/reset, which call this — 20260523000004:42 / 20260523000003:38):
perform public.publish_show_invalidation(p_show_id);
```

**Active-session invalidation (R3 finding).** The `archived`/`!published` access gate (`resolvePickerSelection.ts:87`) denies *new* resolutions, but an **already-open** crew page holds subscribed state and would not stop until it refreshes. The live realtime triggers fire on `crew_member_auth`/`crew_members`, not on `shows`/`show_share_tokens`, so an UPDATE to `shows` alone does NOT push an invalidation. Therefore `_archive_show_core` MUST call `public.publish_show_invalidation(p_show_id)` (`supabase/migrations/20260503000000_publish_show_invalidation_helper.sql`) after the state/token/epoch change — exactly as `rotate_show_share_token` (`:42`) and `reset_picker_epoch_atomic` (`:38`) do — so active viewers are invalidated immediately. This is the literal mechanism behind the owner's "crew links stop working **now**" requirement.

The `unpublish_token`/`unpublish_token_expires_at` pair is nulled **together** (preserves the paired CHECK from `20260512082710_add_show_unpublish_token.sql:14-22`). Token rotation is what enforces the owner requirement that *old links must not silently resurrect on revival* — after a later Unarchive→Publish the show is Live on a fresh token the operator never distributed.

**Confirm gesture (UI):** two-tap, matching `components/admin/AlertBanner.tsx` Resolve. Tap 1 → the button morphs to a confirm affordance whose label explicitly states the consequence: **"Confirm archive — crew links stop working now and won't come back until you re-publish and issue a new link."** Tap 2 → dispatches. A 4s idle reverts the confirm to the resting state. The disable-on-`isPending` discipline applies (a self-disabling submit cancels the React 19 form-action dispatch — see B1's revoke-hang lesson; disable on `isPending`, never synchronously in the click handler).

**Success confirmation (UI):** no toast (none exists in the project — only two `// future toast` comments). The form action resolves and `router.refresh()` re-renders the show into its Archived presentation (Archived pill, moved into the dashboard's Archived segment, ParsePanel read-only with a persistent "crew links are dead" note). The state change is the confirmation. **No `admin_alert` row is created** for an admin's own deliberate archive. Consequently `SHOW_ARCHIVED_BY_ADMIN` (and likewise `SHOW_UNARCHIVED`, `SHOW_PUBLISHED_BY_ADMIN`) is an **action-result / persistent-disclosure copy code**, rendered inline via `messageFor()` — it is **NOT** an `admin_alerts` producer and is **NOT** registered in the admin-alert catalog meta-test. **B2 introduces no NEW `admin_alert` codes** — the OFF-path awaiting-approval event (`SHOW_AWAITING_PUBLISH_APPROVAL`) is a per-staged `pending_syncs` inbox row + B3 email hook keyed by `staged_id`, not an `admin_alert` (§4.4). (The OFF→Apply path *reuses* the **existing** `SHOW_FIRST_PUBLISHED` alert producer for the post-publish notice — a reuse, not a new code; §4.3.)

**Persistent disclosure (UI):** the archived show's per-show page and its Archived-segment row both render a persistent, plain-language note: **"This show is archived. Crew links are dead. Unarchive and re-publish to bring it back."** This is a rendered element (not just a description) — see §3.3 dimensional invariants.

### 2.3 Unarchive (admin)

**Entry points:** the Archived-segment row on the dashboard (§3) and the archived show's per-show page.

**Precondition:** the row is `archived=true`. (No finalize-ownership conflict is possible — an archived row is not finalize-owned.)

**Mutation (RPC `unarchive_show(p_show_id)`):** `archived=false, archived_at=null`, **and `requires_resync=true`** (the durable publish-blocking marker, below). `published` stays `false` → the show lands in **Held**. The share token is **not** touched — it was already rotated to a fresh, undistributed value, either by the Archive action (B2 rows) or by the migration backfill (legacy pre-B2 archived rows, §6.1). So no previously-distributed crew URL can ever resurrect on revival, regardless of when the show was archived.

**Durable `requires_resync` marker — the STRUCTURAL closure of the Publish-freshness vector (R12 vector + R13 representation fix).** The single source of truth for "is this Held show safe to publish?" is a **dedicated boolean column `shows.requires_resync`** (NOT a `last_sync_status` literal — R13: a new status literal would read as "Unknown sync state" through the Drive-health recognized-status set, `lib/admin/driveConnectionHealth.ts` / catalog `SYNC_STATUS_UNKNOWN:1303-1307`, and would contradict §6.3's no-status-change claim). `unarchive_show` durably sets **`requires_resync=true` in the same committed transaction as `archived=false`**. It is cleared to `false` by **either valid resolution of the post-unarchive review (R14 — both branches, so no dead-end):** (1) a **successful Phase-2 apply** (Apply of a staged review OR a clean catch-up) — which already writes `last_sync_status='ok'` (`runScheduledCronSync.ts:983,1008,1033`) and additionally clears `requires_resync` at that single site; OR (2) a **discard** of the staged review — `discardStaged` (`lib/sync/discardStaged.ts:231-237`, which already updates `shows`) additionally clears `requires_resync`, because discarding is the operator deliberately choosing to keep the existing (pre-edit) content (an intentional resolution, not stale content). Clearing on discard is safe: `requires_resync` is only ever set `true` by `unarchive_show`, so a discard on a non-Held show is a no-op. Therefore, after Unarchive the show is publish-blocked until the operator resolves via Apply, Discard, or a successful Re-sync; a catch-up that stages a review (pending row remains), errors, **or never runs at all** (server-action crash/timeout) leaves the flag `true` so Publish stays blocked until a resolution runs. This closes the R12 hole where a crash before catch-up left the pre-archive `last_sync_status='ok'` standing and let Publish ship stale content: the boolean is set by unarchive itself, so it is *never* stale-clean. `requires_resync` is a boolean column (default `false`), so it does **not** alter the `last_sync_status` recognized-status contract and needs no `syncStatusBucket`/health/UI changes.

**Catch-up sync on Unarchive (where during-archive edits are pulled).** The Unarchive **server action** runs, *after* the `unarchive_show` RPC, a catch-up re-sync via the existing self-locking `runManualSyncForShow` (a Held show is `!archived` and not finalize-owned, so DEF-3/DEF-4 don't refuse it; the frozen watermark guarantees any during-archive edit is detected). These are **two separate, separately-locked transactions** — safe because the show is Held (crew-unreachable) throughout, so the gap exposes nothing, and the `requires_resync=true` marker (committed with unarchive) durably blocks Publish across the gap. A clean catch-up clears `requires_resync` to `false` + writes current content; an MI-tripping edit stages a `pending_syncs` review that Doug resolves via **Apply** (takes new content → clears) or **Discard** (keeps existing content → clears) — both allowed since `!archived` (§2.3 clearing rule); a Drive error or a never-run catch-up leaves `requires_resync=true`, so Doug must Re-sync (a successful one clears it) before Publish. Putting catch-up here (not in Publish) keeps the advisory lock off the Drive-fetch path and lets Publish be a single atomic locked RPC (§2.4).

**Gesture (UI):** one tap — Unarchive is safe (it exposes nothing; the Held state is crew-unreachable via the `!published` gate). Code `SHOW_UNARCHIVED`.

**Post-unarchive disclosure (UI):** the now-Held show renders a "Held — not published. Publish to make it live, then issue a crew link." note with the **Publish** affordance adjacent.

### 2.4 Publish (admin) — NEW

This action did not exist; it is required by the choice that Unarchive lands in a Held state rather than going straight to Live.

**Single atomic locked RPC — NOT a server-action orchestration (R11 lock-topology fix).** The R10 design ("the Publish server action runs `runManualSyncForShow` then `publish_show` under the show lock") was unsound: `runManualSyncForShow` is itself a self-locking wrapper and `publish_show` locks in-RPC, so composing them either double-holds the lock (violates the single-holder invariant) or runs in two transactions with a race window between the clean catch-up and the `published=true` flip. The catch-up therefore moves to **Unarchive** (§2.3), and **Publish is a single self-locking RPC `publish_show(p_show_id)` whose gate-check + flip are atomic under one lock** (mirroring `archive_show`: `is_admin()` + in-RPC `pg_advisory_xact_lock` + the lockless `_publish_show_core`). The Publish server action does nothing but call this one RPC — no orchestration, single holder.

**Gate (inside `_publish_show_core`, after the lock is held — the freshness contract):** re-read the show and **refuse unless ALL hold**: (a) `archived = false` (else `SHOW_ARCHIVED_IMMUTABLE`); (b) not finalize-owned (else `FINALIZE_OWNED_SHOW`); (c) **`requires_resync = false`** (the durable post-unarchive marker, §2.3) AND **no live (non-wizard) `pending_syncs` AND no live (non-wizard) `pending_ingestions` row for the show** → else `PUBLISH_BLOCKED_PENDING_REVIEW`. On an already-`published` row → idempotent success. This is the freshness guarantee: the Unarchive catch-up (§2.3) is what pulls during-archive edits and clears `requires_resync` on clean success; the Publish gate confirms the flag is cleared and no review/ingestion is pending **atomically with** the flip, so nothing can change the show between the check and `published=true`. (Gating on `requires_resync` — set by unarchive, cleared only by a successful sync — rather than on an ambient `last_sync_status='ok'` is what makes the gate crash-safe: a never-run catch-up cannot leave a stale-clean signal.)

**Mutation (after the gate passes, same locked txn):** `published=true`, then `perform public.publish_show_invalidation(p_show_id)` (so any client still showing the "unavailable" state re-resolves and any cached picker state refreshes). It does **not** mint a 24h `unpublish_token` (that token is the auto-publish safety net for sheets that went live *without* review; a deliberate manual Publish needs no undo grace). Code `SHOW_PUBLISHED_BY_ADMIN`. (`unarchive_show` does NOT invalidate — the show stays unreachable in Held.)

**Crew access after Publish:** the `archived || !published` gate now passes, so the show is reachable **via its current share token**. Because that token was rotated at archive, the operator must distribute/copy the current crew link (or rotate again) — the existing per-show share/rotate controls, gated on `published && !archived` (`app/admin/show/[slug]/page.tsx:230`), become available again exactly here.

**Gesture (UI):** one tap. Publishing a Held show does not by itself expose anything (the share link is undistributed), so no two-tap is required.

### 2.5 Unpublish (24h token) — UNCHANGED behavior, refactored core

The existing `POST /api/show/[slug]/unpublish?token=` path and its `SHOW_UNPUBLISHED` framing are unchanged from the operator's view, and its existing single-holder lock (`withShowLock`, `lib/sync/unpublishShow.ts:241`) is preserved. Internally, `archiveAndConsumeUnpublishToken` (`:123-133`) is extended to perform the **same archive mutation set inline** within that one locked raw-SQL transaction — adding share-token rotation + `picker_epoch` bump + `archived_at` + the live `pending_syncs`/`pending_ingestions` cleanup + `publish_show_invalidation`, so the token path closes the same old-link-resurrection and stuck-inbox gaps as admin Archive. It does **NOT** call `_archive_show_core` (R13 — that private SECURITY DEFINER core is admin-RPC-internal-only and not JS-callable, §2.1); the two implementations are kept in lockstep by the end-state **parity meta-test** (§8). The token consumption guard (`and unpublish_token = $2::uuid`, `:132`) is preserved. `SHOW_UNPUBLISHED` / `UNPUBLISH_TOKEN_CONSUMED` / `UNPUBLISH_TOKEN_EXPIRED` are unchanged (`lib/messages/catalog.ts:665,675,685`).

### 2.6 Guard conditions for the action inputs (null/empty/missing)

- **Missing/garbage `slug`** at any action route → the show lookup returns no row → `ADMIN_LINK_SHOW_NOT_FOUND` (NB: this code is referenced in the master spec but **not yet present in `catalog.ts`** — B2 adds it via three-lockstep, §7).
- **`p_show_id` not found** in any RPC → the RPC raises the show-not-found path (mirrors `rotate_show_share_token` `:19-26`) → `ADMIN_LINK_SHOW_NOT_FOUND`.
- **Double-submit / stale tab** → idempotent success (Archive on archived, Publish on published) or a guarded refusal; never a partial write (all mutations are single-statement-equivalent inside one locked txn).
- **`show_share_tokens` row absent** for the show (shouldn't happen for a published show, but defensively) → the rotate UPDATE matches 0 rows; the archive still sets `archived/published` and the access gate still denies. The RPC does not error on a 0-row token rotate.

## 3. Dashboard archived bucket (segmented Active / Archived)

### 3.1 Layout & behavior

A two-state segmented control — **Active** (default) / **Archived (N)** — sits atop the show list in the dashboard (`components/admin/Dashboard.tsx`). The **needs-attention inbox stays pinned above** both segments regardless of selection (it already spans all shows — `:298-299`). The selected segment is a **URL search param** (`?bucket=archived`), so back/forward and refresh behave and the RSC re-fetches server-side.

- **Active segment** = `archived=false` (today's query at `:110,:130`, unchanged). Includes Live, Publishing…, and **Held** rows (Held rows carry a distinct pill, §3.2).
- **Archived segment** = `archived=true`, ordered by **`archived_at DESC NULLS LAST, id`** (deterministic). Each row is **read-only** except for an **Unarchive** action and an "Open" link to the per-show page. Each row shows the title, an **Archived pill**, and the archived-time line: "Archived {relative archived_at}" when `archived_at` is set, else the fallback copy **"Archived (date unknown)"**. **Pre-B2 archived rows** (retired via the current unpublish path before B2 shipped) have `archived=true` but **`archived_at = NULL`** — there is no reliable historical archival timestamp on `shows` (it has only `created_at`/`last_synced_at`, neither of which is the archive moment), so B2 does **not** fabricate one: those rows sort last (NULLS LAST) and render "Archived (date unknown)". (The plan MAY optionally best-effort-backfill `archived_at = coalesce(last_synced_at, created_at)` for existing archived rows as an approximation, but the contract is the null-safe ordering + fallback copy, so a null `archived_at` is always handled.)

Both segment **counts** are always computed (the inactive label needs its count), via two `count`-only queries (`archived=false` and `archived=true`); the row list is fetched only for the selected segment.

## 4. auto-publish-clean toggle

### 4.1 Storage

A new column on the singleton settings row: `app_settings.auto_publish_clean_first_seen boolean not null default true` (`app_settings` is the `id='default'` singleton, `supabase/migrations/20260501001000_internal_and_admin.sql:232-247`). **Default `true`** preserves amendment-9 behavior exactly; existing installs are unchanged on migration.

### 4.2 Scope

The toggle gates **only the first-seen new-show publish moment** — a brand-new clean sheet going Live without review. It does **not** gate routine clean *updates* to already-published shows (those keep auto-applying; the show is already Live and the update refreshes content).

**It gates EVERY first-seen auto-publish entry point, not just cron/push (R8 finding).** There are TWO live paths that auto-publish a clean first-seen sheet, and the toggle must gate **both**: (a) the cron/push pipeline (`perFileProcessor` → Phase 2, `phase1.outcome === "auto_publish_ready"`); and (b) the **manual pending-ingestion Retry** — `lib/sync/runManualStageForFirstSeen.ts:73-119`, which on `auto_publish_ready` mints the 24h token + runs Phase 2 + calls `emitSuccessfulPhase2Tail`. Without gating (b), Doug could turn auto-publish OFF and still publish a clean first-seen sheet by clicking **Retry** on its `pending_ingestion` — bypassing the approval gate. When OFF, both paths instead stage a `FIRST_SEEN_REVIEW` `pending_syncs` row.

### 4.3 Effect when OFF

A clean first-seen sheet (all MI-1..MI-14 pass) that would auto-apply via Phase 2 instead routes to **staged review**: a `pending_syncs` row carrying the **reused `FIRST_SEEN_REVIEW` sentinel** in `triggered_review_items` — the exact sentinel amendment 9 *retired the emission of* but left fully scaffolded: it is still a member of the `TriggeredReviewItem` union (`lib/parser/types.ts:372-373`) and is already handled by `StagedReviewCard`'s `FIRST_SEEN_INVARIANTS` set + `describeItem` (`:74,:132`). B2 **conditionally re-enables its emission** when the toggle is OFF (§4.6) — so **no new sentinel, no typed-union change, no new StagedReviewCard branch** (this is why R8 replaced the earlier `FIRST_SEEN_AWAITING_APPROVAL` proposal — reusing the scaffolded sentinel avoids rebuilding the typed/UI contract). The row surfaces per-staged in the dashboard needs-attention inbox, rendered with the `SHOW_AWAITING_PUBLISH_APPROVAL` copy via `messageFor()`. Doug clicks **Apply** → it publishes with the **same 24h undo token + `SHOW_FIRST_PUBLISHED` notice as the auto-publish-ON path** (parity is the whole point — OFF just inserts a human approval gate before the same outcome). No `shows` row is created until Apply (so OFF does **not** create Held rows — Held arises only from Unarchive, §1). The awaiting-approval *staging* event is keyed by `pending_syncs.staged_id` (NOT an `admin_alert`, §4.4) so multiple waiting sheets stay independently visible; the *post-Apply* notice reuses the existing `SHOW_FIRST_PUBLISHED` producer.

**Apply-path wiring (R6/R7 findings — comprehensive re-analysis of the publish-notice path).** The `SHOW_FIRST_PUBLISHED` notice + 24h token are NOT emitted by `runPhase2`. They are emitted by the exported **`emitSuccessfulPhase2Tail`** (`runScheduledCronSync.ts:1624-1652`), which — when handed an `autoPublishFirstSeen: {unpublishToken, unpublishTokenExpiresAt}` payload — calls `emitFirstPublishedNotice` (`:1601-1622`: the `SHOW_FIRST_PUBLISHED` upsert + token context) plus `publishShowInvalidation` + `logSync`. The cron auto-publish path calls this tail; **`applyStaged` does NOT** — after `runPhase2` it only does `bumpReviewerAuthFloors` + `insertSyncAudit` + `deleteLivePendingSync` + return (`applyStaged.ts:1342-1355`). So passing `autoPublishFirstSeen` to `runPhase2` is insufficient — the notice/token live in the *tail*.

**Required wiring:** when `applyStaged` applies a staged row carrying the `FIRST_SEEN_REVIEW` sentinel (a first-seen apply, no pre-existing `shows` row), after a successful `runPhase2` it MUST invoke the **same exported `emitSuccessfulPhase2Tail`** with `autoPublishFirstSeen: {unpublishToken, unpublishTokenExpiresAt = now()+24h}` — reusing the cron path's exact notice logic (no duplication, no new alert code). **Structural defense (R7 — vector recurred R6→R7):** the 24h `unpublish_token` is persisted **only** as part of the single `autoPublishFirstSeen` payload that the tail consumes — there is no separate token-mint statement — so "token written but no `SHOW_FIRST_PUBLISHED`" is impossible by construction: either the tail runs with the payload (both token + notice + invalidation) or it doesn't (neither). A parity meta-test pins that **every** code path that sets `autoPublishFirstSeen` routes through `emitSuccessfulPhase2Tail` (so cron and Apply can never diverge on this again).

When **ON** (default): unchanged — Phase 2 auto-applies, mints the 24h token, raises `SHOW_FIRST_PUBLISHED`.

### 4.4 Event for B3 (no email code in B2)

When OFF routes a clean first-seen sheet to staged review, the staged `pending_syncs` row carries the **reused `FIRST_SEEN_REVIEW` sentinel** in `triggered_review_items` (the clean-first-seen-held semantics amendment 9 retired the emission of; B2 re-enables emission conditionally, §4.6). This event is **keyed by `pending_syncs.staged_id`, NOT an `admin_alert`** (R4 finding):

- **In-app:** it surfaces per-staged in the existing needs-attention inbox, which reads `pending_syncs` **directly** and classifies each row as `first_seen`/`existing_staged` (`lib/admin/needsAttention.ts:6`) — so **two** waiting clean first-seen sheets show as **two distinct inbox rows**. `SHOW_AWAITING_PUBLISH_APPROVAL` is the user-facing **copy** rendered for such a row.
- **Why NOT an `admin_alert`:** `admin_alerts` has a unique index `(coalesce(show_id::text,''), code) where resolved_at is null` (`supabase/migrations/20260501001000_internal_and_admin.sql:279-280`). **Consumer wiring (R8 finding — the new emission must be threaded through the existing typed + UI surfaces).** Re-enabling `FIRST_SEEN_REVIEW` requires these explicit updates (enumerated so the plan covers them — the typed union + `StagedReviewCard` branch already exist and need NO change):

2. **Dashboard / inbox query** — `lib/admin/needsAttention.ts` (and the `Dashboard` pending query) must **select `triggered_review_items`** so a `FIRST_SEEN_REVIEW` row classifies as first-seen review (today the query does not select it).
3. **`components/admin/PendingPanel.tsx:126`** — replace the hard-coded `"First-time review needed"` string with the `SHOW_AWAITING_PUBLISH_APPROVAL` copy via `messageFor()` for `FIRST_SEEN_REVIEW` rows (Opus/UI; impeccable-gated).
4. **`StagedReviewCard` `describeItem` (`:132`)** — the `FIRST_SEEN_REVIEW` case copy is reviewed for the approval-gate context (UI; may reuse or refine).
No change to `TriggeredReviewItem` (`lib/parser/types.ts:372`) or `FIRST_SEEN_INVARIANTS` (`StagedReviewCard.tsx:74`) — both already include `FIRST_SEEN_REVIEW`.

**B2 ships the producer (the conditionally re-enabled `FIRST_SEEN_REVIEW` row + inbox copy); B3 ships the delivery.** B3's opt-in taxonomy must cover "show awaiting approval," not only "sync problems" — flagged here so B3's spec accounts for it. **B2 introduces no NEW `admin_alert` codes** (the awaiting-approval *staging* event is the per-staged `pending_syncs` `FIRST_SEEN_REVIEW` row, not an alert). The OFF→Apply path **reuses the existing `SHOW_FIRST_PUBLISHED` `admin_alert` producer** for the post-publish notice (§4.3) — same as the ON path; that is a reuse, not a new code (R4/R6 resolution).

## 5. Archived-immutability guards (DEF-1/2/3 + DEF-4 cron/push/missing-file + DEF-5 pending-ingestion routes)

The unifying principle: **an archived show is immutable everywhere** — admin actions and the sync engine alike. Each guarded path re-reads the show inside its advisory-locked transaction and refuses before any write. The shared user-facing code is `SHOW_ARCHIVED_IMMUTABLE` (Doug-facing: *"This show is archived. Unarchive it before making changes."*). DEF-4 is the exception (silent skip, not a user-facing refusal — the sync engine has no user in the loop).

**Observability of DEF-4 skips — genuinely silent (R17 control-flow correction).** The archived skip writes **nothing**: no Drive fetch, no mutation, no `last_seen_modified_time`/`last_sync_status` change, **no `sync_log` row, and no `sync_audit` row**. It must NOT route through the normal skip path (which calls `logSync` at `:1896-1898`; that path's `"skipped" in result` guard at `:1517` does not actually suppress `{outcome:"skipped"}` results, so normal skips write a `sync_log` row every pass) — the archived short-circuit returns *before* the `logSync` call on both cron and push (§5 DEF-4 (a)). **It does NOT write to `sync_audit`** either — that table is an apply-path audit with required `staged_id`/`triggered_review_items`/`reviewer_choices`/`derived_side_effects`/`parse_result_summary`/`staged_modified_time` columns and **no `reason` column** (`supabase/migrations/20260501001000_internal_and_admin.sql:204-217`). `SYNC_SKIPPED_ARCHIVED` is therefore a **conceptual outcome name only — not a §12.4 code, not a persisted `sync_log`/`sync_audit` row**. If per-pass observability is ever wanted later, a deliberately throttled `sync_log` write would be a separate, opt-in change; B2's contract is silent.

**DEF-4 also covers folder-removal / missing-file detection (R4 finding — a 5th mutation path).** The master-spec §5.2 *prose* says removal detection runs over `archived = false`, but the **live code does not implement that filter**: `listPostgresLiveShows` selects every row `where drive_file_id is not null` (`lib/sync/runScheduledCronSync.ts:1447`), and a row missing from the folder is routed to `markMissingShow_unlocked` (`:1782`, called at `:2318`) which mutates `shows.last_sync_status`, writes `sync_log`, and raises `SHEET_UNAVAILABLE`. So an **archived** show removed from the watched folder would still be mutated + alert-spammed — violating archived-immutability. B2 closes this: **filter `listPostgresLiveShows` to `archived = false`** (aligning the code to the §5.2 contract) **AND** re-read `archived` inside `markMissingShow_unlocked` before any update/log/alert (defense-in-depth for the archive-mid-pass race, same two-phase rationale as the per-file path). AC-B2.13c covers it.

### 5.1 PostgREST DML lockdown (RPC-gated tables)

The durable lifecycle state of a show lives in **`shows`** (`archived`/`published`/content) and **`show_share_tokens`** (the crew link). **Both are already DML-locked-down and registered** in `tests/db/postgrest-dml-lockdown.test.ts` (`RPC_GATED_TABLES` at `:139` and `:168`) — `INSERT/UPDATE/DELETE` are REVOKEd from `anon`/`authenticated`, so even an admin-authenticated caller cannot `from('shows').update({archived:false})` or mutate a share token directly via PostgREST; the only write path is a SECURITY DEFINER RPC. B2's new `archive_show`/`unarchive_show`/`publish_show`/`_archive_show_core` are *additional* RPC entry points on these already-gated tables — **no new REVOKE migration is needed for them**, and B2 must NOT add `shows`/`show_share_tokens` registry rows (they already exist). The plan's job is only to confirm these two tables remain in the registry (a regression guard) so the new RPCs are the sole lifecycle-state write path.

**`pending_syncs` / `pending_ingestions` ARE locked down by B2 (R18 — reverses the R3 narrowing).** These tables currently carry table-level DML grants for `anon`/`authenticated` (`supabase/migrations/20260501002000_rls_policies.sql:83,91`). The R3 reasoning ("RLS-only scratch; direct deletes harmless") is **invalidated** by the §2.4 Publish gate: the gate treats *absence of a live `pending_syncs`/`pending_ingestions` row* as part of the freshness guarantee, so an admin-authenticated `from('pending_syncs').delete()` of a Held show's staged review would remove the only remaining publish blocker and let `publish_show` flip `published=true` over unresolved/stale content. B2 therefore **REVOKEs `INSERT/UPDATE/DELETE` on both tables from `anon` + `authenticated`** (service_role retains; the sync pipeline writes via the raw `DATABASE_URL` connection, unaffected) and **adds both to `RPC_GATED_TABLES`** (the REVOKE satisfies the registry's "every entry has a matching REVOKE" meta-test). This is **safe**: a repo sweep confirms there are **no** authenticated-client `from("pending_syncs"|"pending_ingestions")` INSERT/UPDATE/DELETE calls anywhere in `app/`/`components/`/`lib/` — every mutation already flows through the raw pipeline or SECURITY DEFINER RPCs (apply/discard/stage are pipeline `tx` writes; the routes call those, not `from()`), so the REVOKE breaks no legitimate path. After it, the Publish gate's "no live pending rows" component is **un-bypassable**: the only way to remove a pending row is Apply/Discard, which properly clears `requires_resync` — closing both the bypass and the no-row-to-resolve dead-end. **`app_settings`** (the auto-publish toggle, §4) remains RLS-`admin_only` (`rls_policies.sql:134`) — it is an admin-only singleton not consumed by any freshness gate, so RLS is the appropriate gate (not RPC-gating). (See `reference_postgrest_dml_lockdown_classwide_pattern` for the registry + `describe.each` + Layer-4 reconciliation shape.)

```sql
-- so direct PostgREST deletion of a Held show's staged review must be impossible. Lock these tables'
-- DML behind the pipeline/RPCs (service_role retains; raw DATABASE_URL pipeline unaffected). Verified
-- safe: no authenticated-client from("pending_syncs"|"pending_ingestions") INSERT/UPDATE/DELETE exists.
revoke insert, update, delete on table public.pending_syncs      from anon, authenticated;
revoke insert, update, delete on table public.pending_ingestions from anon, authenticated;
```

After this backfill, **every** archived row (legacy + future) has a dead distributed token, so the §2.3 "token was already rotated" assumption holds universally — Unarchive→Publish never resurrects a previously-shared URL regardless of when the row was archived.

`archived_at` is purely for display ("Archived {when}") + audit; it is nullable, set by `archive_show`, nulled by `unarchive_show`. The 30-day archived GC window (`lib/sync/diagramGc.ts:176`) already keys off the `archived` boolean and is unaffected.

### 6.2 New RPCs

Three admin-callable RPCs — `archive_show(p_show_id uuid)`, `unarchive_show(p_show_id uuid)`, `publish_show(p_show_id uuid)` — all SECURITY DEFINER, `set search_path = public, pg_temp`, `is_admin()` gated, `pg_advisory_xact_lock(hashtext('show:'||drive_file_id))` in-RPC, with `revoke all ... from public, anon, authenticated, service_role; grant execute ... to authenticated;` (mirroring `20260523000004:48-49`). Each holds the lock at **exactly one layer** (in-RPC); their JS callers never re-wrap them. Two **internal lockless** routines — `_archive_show_core(p_show_id)` (the archive mutation set, §2.1/§2.2) and `_publish_show_core(p_show_id)` (the Publish gate + flip + invalidation, §2.4) — are SECURITY DEFINER, `set search_path`, **no lock, no `is_admin()`**, `revoke`d from `public, anon, authenticated, service_role` (not PostgREST/JS-callable; invoked only via internal `perform` by their same-owner SECURITY DEFINER wrappers — `_archive_show_core` from `archive_show` ONLY, `_publish_show_core` from `publish_show` ONLY). **The token-Unpublish path does NOT call `_archive_show_core`** — it mirrors the mutation inline in its own locked raw-SQL txn (§2.5), kept in lockstep by the parity meta-test (§8). `unarchive_show` is a single-statement mutation inlined in its own RPC. **The Publish catch-up sync is NOT part of `publish_show`** — it runs on Unarchive as a separate self-locked `runManualSyncForShow` (§2.3), so `publish_show` never holds the advisory lock across a Drive fetch and never double-locks.
