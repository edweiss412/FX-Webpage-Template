## 1. Scope and boundaries

### 1.1 Problem
The "Alert me about sync problems" + "Daily review digest" toggles are operator **attention surfaces**: Doug's surface is Drive, not the admin dashboard, so a stalled sync or a staged event that lives only in-app is functionally invisible. The in-app `admin_alerts` surface (AlertBanner / NotifBell) is the system of record (pull); B3 adds the email **push** layer over it. Per **AGENTS.md no-zombie-flag**, neither toggle ships without its working backend — hence the toggles ship *with* the email subsystem.

### 1.2 In scope
First outbound-email path (`lib/notify/` channel-agnostic seam, email-only impl, Resend, secrets); the two real-time + digest email tiers; a simple `email_deliveries` dedup/audit ledger; the global `SYNC_STALLED` heartbeat detector (+ a small **cross-into-sync-path** heartbeat write and recovery-resolution calls, Codex backend); two `app_settings` toggles + read/write + UI; three §12.4 codes; `app/api/cron/notify/route.ts` + two pg_cron schedules; TDD tests + meta-test registrations.

### 1.3 Out of scope (deferred — §12)
Two-way feedback, inbound webhook, apply-from-email, unpublish tokens, SMS impl, per-admin prefs, digest→realtime promotion, and (RD-9) the exactly-once crash-recovery outbox machinery. The 2026-05-09 `notification-design-memo.md` is cited for **principles** only (push-not-pull, tiering, quiet-success, coalescing).

   ─ realtime sched ─────► │  app/api/cron/notify/route.ts (CRON_SECRET)  │
   ─ digest sched   ─────► │   ?job=realtime | ?job=digest                │
                           └───────────────┬─────────────────────────────┘
                                           │ service-role client
   ALWAYS (maintenance, no provider/toggle needed):
     • global SYNC_STALLED detect+resolve (heartbeat)   • recovery-resolution recheck
   GATED delivery (provider configured AND tier toggle ON):
     realtime: read candidates (admin_alerts ∈ SYNC_PROBLEM_CODES aged ≥1h;
               global SYNC_STALLED; pending_ingestions >1h) → for each × active

### 3.2 `email_deliveries` ledger (new table, B3 migration)
```sql
create table public.email_deliveries (
  id              uuid primary key default gen_random_uuid(),
  kind            text not null check (kind in ('realtime_problem','digest')),
  channel         text not null default 'email' check (channel in ('email','sms','webhook')),
  dedup_key       text not null,   -- realtime show-level '<show_id>:<code>:<epoch(raised_at)>'; global 'global:SYNC_STALLED:<epoch(raised_at)>'; ingestion 'ingestion:<drive_file_id>:<epoch(first_seen_at)>'; digest 'digest:<ET_date>'
  show_id         uuid references public.shows(id) on delete set null,
  recipient       text not null,
  triggered_codes text[] not null default '{}',
  status          text not null check (status in ('sent','failed')),
  provider_message_id text,
  error           text,
  attempt_count   int  not null default 1,
  created_at      timestamptz not null default now(),
  sent_at         timestamptz,
  constraint email_deliveries_recipient_email_canonical
    check (recipient = lower(trim(recipient)) and recipient <> '')   -- invariant 3 safety net
);
create unique index email_deliveries_dedup on public.email_deliveries (kind, dedup_key, recipient);
```
- **DML lockdown:** REVOKE insert/update/delete from `anon`, `authenticated`; grant to `service_role` only; add a service-role-only entry to `tests/db/postgrest-dml-lockdown.test.ts`.
- **Provider Idempotency-Key (the sole dup guard):** `fxav:<kind>:<sha256(kind || ':' || dedup_key || ':' || recipient)>` — a **bounded** ~90-char hash (Resend caps the key at **256 chars**; the raw email-bearing concat is hashed). Verified Resend contract (docs 2026-06-02): a repeat with the same key within **24h** returns the original response (no second email); same key + different payload → `409 invalid_idempotent_request`.
- **Dedup + at-least-once delivery (RD-9):** before sending a candidate × recipient, skip if a `status='sent'` row already exists for `(kind, dedup_key, recipient)`. Otherwise send via Resend with the idempotency key, then upsert the row to `sent` (+`provider_message_id`,`sent_at`) or `failed` (+`error`, `attempt_count`). **No claim/lock/terminal state.** Two concurrent ticks may both see no `sent` row and both call Resend — the **idempotency key makes the second a provider-side no-op**, and both upserts converge the row to `sent` (one email). A crash mid-send is handled the same way: the next tick re-selects (no `sent` row) and re-sends with the same key → Resend returns the original within 24h (no duplicate); beyond 24h a duplicate is *possible* and accepted (RD-9).
  - **`409 invalid_idempotent_request` is treated as SUCCESS, not failure.** Resend returns `409` when the same key is replayed with a DIFFERENT payload — which means the key was already used, i.e. the original DID send. `sendEmail` maps a `409` to `{ ok:true, alreadySent:true }` so the ledger records `sent` (NOT `failed` + a spurious `EMAIL_DELIVERY_FAILED`). This makes a payload snapshot unnecessary: realtime content can shift while `raised_at`/the key stays fixed, and the digest re-renders fresh per tick — a post-delivery retry with changed content just `409`s → `sent`; a not-yet-delivered key (a rejection does NOT register the key at Resend) sends the current content fresh.
  - **The ledger upsert NEVER downgrades a `sent` row to `failed`** (the failure upsert is conditional `WHERE status <> 'sent'`), so a late/racing failure can't clobber a recorded success.
- **Re-arm:** `dedup_key` includes the occurrence epoch (`raised_at` / `first_seen_at` / ET-date), so a resolved-then-reopened problem (fresh `raised_at`, via §4.3 recovery-resolution) is a NEW key → a new email. `email_deliveries_recipient_email_canonical` keeps the recipient canonical (it is part of the dedup identity).
- **Retry:** a `failed` row under `SEND_RETRY_CAP (3)` is re-sent on a later tick (same idempotency key); at the cap it stays `failed` and `EMAIL_DELIVERY_FAILED` (§6) remains until the underlying problem clears.

## 4. Detection (real-time tier)

### 4.1 `SYNC_PROBLEM_CODES` (single source of truth — `lib/notify/constants.ts`)
`['DRIVE_FETCH_FAILED','SHEET_PROCESS_FAILED','SHEET_UNAVAILABLE','PARSE_ERROR_LAST_GOOD']` (the show-level consumed error codes). `STAGED_PARSE_*` + info-severity codes are excluded. Referenced by the query and the tests, never re-listed inline. NOTE: `SYNC_STALLED` (show-wide) + pending-ingestion are handled by their own detectors, not this set.

### 4.2 Global sync-stall detector (`SYNC_STALLED`, show-wide)
`shows.last_synced_at` is the wrong staleness signal (bumped on failures; not bumped for quiet shows). Per-show failures are already covered by their error alerts (§4.3); the only gap is the whole pipeline stopping. So `SYNC_STALLED` is a single show-wide alert driven by a sync-run heartbeat:
- **Heartbeat (B3 adds — cross-into-sync-path):** the scheduled sync run writes `app_settings.last_sync_cron_completed_at = now()` at the END of EVERY completed run — success, all-quiet, AND the **`no_folder_configured` / skipped** completion (the heartbeat means "the cron executed", which it did). This prevents a false `SYNC_STALLED` on a pre-onboarding / no-watched-folder deployment (the cron runs hourly, skips, and bumps the heartbeat, so it never goes stale). Trigger-agnostic.
- **Detector (always, in `?job=realtime`):** if `now() - coalesce(last_sync_cron_completed_at,'-infinity') > STALENESS_THRESHOLD_MS` → `upsert_admin_alert(NULL,'SYNC_STALLED',…)` (one show-wide row). **Resolve** when the heartbeat is fresh again.
- **Common-mode limitation (documented):** the detector runs on the same scheduler it watches, so a total pg_cron/pg_net outage can't self-detect → covered by an external uptime monitor (B3-DEF-8, out of scope).

### 4.3 Gap-fill audit + recovery-resolution (implementation task — cross-into-sync-path)
Confirm each `SYNC_PROBLEM_CODES` condition both (a) UPSERTS its code's `admin_alerts` row on entry AND (b) RESOLVES that row whenever its **exact code predicate** no longer holds. **Per-code, not just "became OK":** the specific failure code is in `shows.last_sync_error` (`runScheduledCronSync.ts:661/674/694/715` set `last_sync_status` + `last_sync_error` together), and a show can switch `DRIVE_FETCH_FAILED`→`PARSE_ERROR_LAST_GOOD` while STILL in an error status. So each `SYNC_PROBLEM_CODES` alert resolves when the show's current `(last_sync_status, last_sync_error)` no longer maps to that alert's code (OK, or a different error code — the new code upserts its own alert). Today resolution is only on manual admin-resolve routes; **B3 ADDS the per-code recovery-resolution** to the sync path. This is load-bearing: without it, a cleared/switched error still emails its stale code 1h later AND `upsert_admin_alert` coalesces future occurrences into the wrong never-resolved row (the re-arm model depends on resolution refreshing `raised_at`). **Notify-side recheck:** before emailing a show-level candidate, compare the show's current `last_sync_status` + `last_sync_error` against the candidate alert's code; if they no longer match (recovered OR switched code), resolve the alert and skip. The plan documents the code→`(status,error)` predicate map.

### 4.4 Delivery loop (realtime)
Candidates from three sources: §4.2 staleness (→ `admin_alerts.SYNC_STALLED`), §4.3 consumed error alerts, §4.5 pending-ingestion.
0. **Maintenance (ALWAYS — no provider/toggle needed):** global `SYNC_STALLED` detect+resolve (§4.2) and the recovery-resolution recheck (§4.3) — they upsert/resolve IN-APP `admin_alerts` so the dashboard signal stands even when email is OFF/unconfigured.
1. **Provider gate:** unconfigured (§6) → maintain `EMAIL_NOT_CONFIGURED`, skip all sends. Else continue.
2. **Tier-toggle gate:** read `alert_on_sync_problems` fail-closed (`lib/appSettings/getAlertOnSyncProblems.ts`, mirroring `getAutoPublishCleanFirstSeen.ts`); false/read-error → skip realtime sends (maintenance already ran).
3. **Candidates:**
   - **(a) Show-level:** unresolved `admin_alerts` JOIN `shows` where `code = any(SYNC_PROBLEM_CODES)` AND `raised_at <= now()-interval '1 hour'` AND `shows.published = true` AND `shows.archived = false` (never email problems for inactive shows). Key `'<show_id>:<code>:<epoch(raised_at)>'`.
   - **(b) Global stall:** the unresolved show-wide `SYNC_STALLED` (`show_id IS NULL`); NO `shows` join; NO extra age gate (the 1h is already in §4.2's threshold). Key `'global:SYNC_STALLED:<epoch(raised_at)>'`, link → `/admin`.
   - **(c) Pending-ingestion (§4.5):** `pending_ingestions` where `now()-first_seen_at > interval '1 hour'` AND `wizard_session_id IS NULL`. Key `'ingestion:<drive_file_id>:<epoch(first_seen_at)>'`, `show_id NULL`, `triggered_codes=[last_error_code]`, link → `/admin`.
4. **Send:** for each candidate × each active recipient, **re-check the recipient is still active** (`admin_emails.revoked_at IS NULL`, close to the send; revoked → skip, never email a revoked admin); skip if a `status='sent'` ledger row exists; else `lib/notify.sendEmail(...)` → Resend (Idempotency-Key §3.2); upsert `sent`/`failed` (+`EMAIL_DELIVERY_FAILED` on failure, §6).
5. **Advisory-lock posture:** the loop mutates only `admin_alerts` (idempotent upsert) + `email_deliveries` and READS `shows`/`pending_ingestions`; it does not mutate the invariant-2 protected tables, so it holds **no** per-show lock (`// not-subject-to-meta: notify job mutates only admin_alerts + email_deliveries`).

### 4.5 Stuck-ingestion detector (pre-show — second documented exception to RD-6)
Pending ingestions are pre-show (no `show_id`), so `admin_alerts`' `(show_id,code)` upsert can't dedup per-file; this class is read directly and dedup'd by `drive_file_id`.
- **Select:** `pending_ingestions` older than `STALENESS_THRESHOLD_MS` AND `wizard_session_id IS NULL` (exclude active onboarding, matching the dashboard live-scope `components/admin/Dashboard.tsx:328-331`). A row's presence = the sheet hasn't become a show; a successful re-ingestion deletes the row → no email (quiet-success).
- **Dedup/re-arm:** `'ingestion:<drive_file_id>:<epoch(first_seen_at)>'`. Re-arms on a fresh `first_seen_at`.
- **Email:** body uses the row's `last_error_code` `dougFacing` + `drive_file_name`; link → `/admin`. Gated by `alert_on_sync_problems`.
- **No advisory lock (RD-9):** a lock-less read may *rarely* email a stuck sheet just as recovery deletes the row; for a >1h-stuck sheet that is a real, mildly-stale heads-up, not spam — accepted.

## 5. Delivery infrastructure

### 5.1 `lib/notify/`
- `lib/notify/send.ts` — `sendEmail({to, subject, html, text, idempotencyKey})`; `NotifyChannel='email'|'sms'|'webhook'` (email-only impl) → Resend SDK. **Call-boundary discipline (invariant 9):** returns `{ ok:true; messageId } | { ok:true; alreadySent:true } | { ok:false; kind:'infra_error'; message }`, never throws into the cron loop. A Resend `200` → `{ok:true, messageId}`; a `409 invalid_idempotent_request` → `{ok:true, alreadySent:true}` (the key already sent — §3.2); any other error / network fault → `{ok:false, kind:'infra_error'}`. (Pure provider wrapper; carries an inline `// not-subject-to-meta` for the Supabase-boundary meta-test.)
- `lib/notify/recipients.ts` — `activeRecipients()` reads `admin_emails where revoked_at is null` with a **service-role** client (NOT the cookie/RLS-bound `listAdminEmails` at `lib/data/adminEmails.ts:60-66`, which returns zero rows under cron); re-canonicalizes via `lib/email/canonicalize.ts` (invariant 3); call-boundary discipline applies.
- `lib/notify/templates/realtimeProblem.ts`, `templates/digest.ts` — pure renderers → `{subject, html, text}`. **Copy via `messageFor()` / catalog `dougFacing`** (invariant 5 — no raw codes); **no em dashes** (DESIGN.md §9). **HTML-escaping contract (mandatory):** every dynamic value interpolated into `html` (show name, `drive_file_name`, any sheet/admin `context`) passes through a trusted escaping primitive (a JSX/email renderer that escapes text nodes, or a central `escapeHtml()`); no dynamic value reaches `html` un-escaped. Subjects (em-dash-free): realtime `"FXAV · <show>: sync problem"`; digest `"FXAV: <m> shows have <n> items to review"`.
- `lib/notify/constants.ts` — single source of truth: `SYNC_PROBLEM_THRESHOLD_MS = 3_600_000`, `STALENESS_THRESHOLD_MS = 3_600_000`, `DIGEST_HOUR_LOCAL = 7`, `DIGEST_TIMEZONE = 'America/New_York'`, `DIGEST_RETRY_WINDOW_HOURS = 3` (eligible ET-hours 7..10), `SEND_RETRY_CAP = 3`, `SYNC_PROBLEM_CODES = [...]`.

### 5.2 Provider + secrets
**Resend** (Vercel-native). Secrets `RESEND_API_KEY` + `EMAIL_FROM` (a verified sending domain) in Vercel env; absence → unconfigured path (§6). Provider swappable behind `lib/notify/send.ts`; no provider type leaks above the seam. The 24h Idempotency-Key replay (§3.2) is the at-least-once dup guard.

### 5.3 Cron route + scheduling
- `app/api/cron/notify/route.ts` — first line `const denied = rejectUnauthorizedCron(request); if (denied) return denied;` (`app/api/cron/_auth.ts:3-12`, `Bearer $CRON_SECRET`). Branch on `?job=realtime|digest`; unknown/missing → 400.
- **Two pg_cron schedules** (B3 migration, per the 2026-05-26 pg-cron pivot; `pg_net` POST with `Authorization: Bearer <CRON_SECRET>`, secret mirrored in Supabase Vault per the pivot §2.3):
  - `realtime` — own schedule aligned to the sync cadence (RD-8c).
  - `digest` — **hourly**; the route proceeds only when the current `America/New_York` hour ∈ `[DIGEST_HOUR_LOCAL, DIGEST_HOUR_LOCAL + DIGEST_RETRY_WINDOW_HOURS)`. **Hourly + ET-hour window is DST-safe** (explicit `America/New_York`, never host TZ).
- **Digest content:** built fresh from the live needs-attention inbox at send time (reuse the dashboard's filters incl. `pending_ingestions.wizard_session_id IS NULL` AND `pending_syncs.wizard_session_id IS NULL`, `components/admin/Dashboard.tsx:328-331,364-369` / `lib/admin/needsAttention.ts`), executed via the **service-role** client. Per-recipient exclusion: omit an item the recipient already has a `sent` realtime delivery for (matched by the current occurrence's realtime `dedup_key`). **The digest query MUST select every field needed to derive each item's realtime occurrence key** — for pending-ingestion items `pending_ingestions.first_seen_at` (to reconstruct `ingestion:<drive_file_id>:<epoch(first_seen_at)>`), for show-level items the matching `admin_alerts.raised_at`. The dashboard's default needs-attention select omits `first_seen_at`, so the digest query adds it (else a stuck sheet already emailed realtime would still appear in the digest). Dedup_key `'digest:<ET_date>'` → one digest per ET day per recipient (idempotency-key replay covers a same-day retry). 0 open items → no send. Gated by `daily_review_digest`; maintenance runs first (§4.4 step 0).

Realtime → `${NEXT_PUBLIC_SITE_ORIGIN}/admin/show/<slug>` (show-level) or `/admin` (global / ingestion); digest → `/admin`. Read-only notices, no action tokens.

## 6. Failure & unconfigured-provider handling
- **Send failure:** Resend `{ ok:false, kind:'infra_error' }` → upsert the `email_deliveries` row `status='failed'` (+`error`, `attempt_count+1`) **only when the row is not already `sent`** (never downgrade a success); emit `upsert_admin_alert(show_id,'EMAIL_DELIVERY_FAILED',…)` (in-app only — **never** email about email failing). (A `409` is `{ok:true,alreadySent}` per §3.2, recorded `sent`, NOT a failure.) Retried next tick while `attempt_count < SEND_RETRY_CAP (3)`; at the cap it stays `failed` (alert persists).
- **`EMAIL_DELIVERY_FAILED` recovery:** resolve when no `failed`-at-cap delivery remains for the alert's scope (a later success / the underlying problem resolving clears it). For pending-ingestion (no source alert), it stops counting once the `pending_ingestions` row is gone.
- **Unconfigured provider:** missing `RESEND_API_KEY`/`EMAIL_FROM` → **only the send loops skip**; maintenance (§4.4 step 0) still runs (`SYNC_STALLED` detect/resolve, recovery-resolution) so the in-app dashboard signal is never blinded; emit `EMAIL_NOT_CONFIGURED` once (resolve when configured). Returns 200 (delivery skipped). Toggles still settable; settings UI MAY show a server-derived "email not configured" hint (never leaks the secret).
- **Fail-closed reads:** both toggle reads fail-closed (read error → "off") so an infra hiccup never spams.

## 9. Acceptance criteria
- **AC-B3.1** `alert_on_sync_problems=true`, an unresolved `SYNC_PROBLEM_CODES` alert with `raised_at ≤ now()-1h` → exactly one Resend send + one `sent` row per active recipient; a second tick → no resend (the `sent` row skips it).
- **AC-B3.1a (concurrency)** Two overlapping realtime ticks for the same alert+recipient → exactly ONE email actually delivered (both may call Resend, but the Idempotency-Key makes the second a provider no-op); the ledger converges to one `sent` row.
- **AC-B3.1b (multi-recipient)** One alert + two active admins → two distinct idempotency keys (include `recipient`), two `sent` rows, two emails; no recipient is provider-deduped away.
- **AC-B3.2** `alert_on_sync_problems=false`/read-error → zero realtime emails; in-app alerts unaffected.
- **AC-B3.3** An alert that resolves then re-opens (fresh `raised_at`) re-arms → a new email. (Negative-regression: stash the `raised_at` epoch in the key → second email fails to fire.)
- **AC-B3.3a (recovery-resolution, per-code)** An error alert whose show recovers before 1h is resolved → no email; the notify-side recheck resolves+skips a recovered-but-unresolved alert. **Code-switch:** a show going `DRIVE_FETCH_FAILED`→`PARSE_ERROR_LAST_GOOD` WITHOUT an OK interlude resolves the old code's alert (current `last_sync_error` ≠ that code) and upserts the new code's alert → the old code is never emailed stale and future occurrences coalesce into the right row. (Negative-regressions: drop recovery-resolution → stale email + suppressed recurrence; resolve only on "became OK" → the code-switch case emails the stale code.)
- **AC-B3.4** A transient that clears before 1h → no email (quiet-success).
- **AC-B3.5 (global stall)** Heartbeat older than `STALENESS_THRESHOLD_MS` (or NULL) → exactly one show-wide `SYNC_STALLED`; fresh heartbeat resolves it. Negative-regressions: a perpetually-FAILING show (fresh `last_synced_at` + error alert) does NOT trigger `SYNC_STALLED`; a quiet healthy show does NOT.
- **AC-B3.5a (fresh-deploy)** Right after the migration (heartbeat = now()), the first tick before any sync run raises NO `SYNC_STALLED`, even with toggles ON.
- **AC-B3.5c (no-folder no false stall)** With no watched folder configured, the scheduled sync's `no_folder_configured` completion bumps `last_sync_cron_completed_at`, so after >1h the detector raises NO `SYNC_STALLED`. (Test: no-folder deployment + >1h of skipped runs → zero stall email.)
- **AC-B3.8a (409 / no-downgrade)** A retry whose original already delivered returns Resend `409` → the row is recorded/kept `sent` (NOT `failed`, NO `EMAIL_DELIVERY_FAILED`); a late failure never downgrades an existing `sent` row. (Test: deliver, then retry with changed content → `409` → still one `sent` row, no spurious failure alert.)
- **AC-B3.5b (pending-ingestion)** A `pending_ingestions` row aged >1h, `wizard_session_id IS NULL` → one email per recipient dedup'd by `drive_file_id`; row deleted before 1h → no email; fresh `first_seen_at` re-arms; wizard-scoped rows excluded.
- **AC-B3.6 (digest)** First in-window ET tick (hour 07) with ≥1 open item → one digest per active recipient, grouped by show, excluding items already realtime-`sent`; later in-window tick → no additional send to a `sent` recipient (dedup_key replay). 0 open items → no send.
- **AC-B3.6a (digest live-scope)** Digest applies `wizard_session_id IS NULL` on `pending_ingestions` + `pending_syncs` (service-role). (Negative-regression: drop the filter → wizard rows leak.)
- **AC-B3.6b (digest occurrence-key derivation)** The digest query selects `pending_ingestions.first_seen_at` (+ the show-level `raised_at`) so it can match a `sent` realtime delivery's occurrence key. (Test: seed a `sent` realtime pending-ingestion delivery for a recipient → that exact occurrence is excluded from their digest, while a fresh re-armed occurrence — new `first_seen_at` — is still included. Negative-regression: omit `first_seen_at` from the digest select → the already-emailed sheet leaks into the digest.)
- **AC-B3.7 (DST)** Across spring-forward / fall-back, the digest fires within the 07:00-ET window (injected ET clock).
- **AC-B3.7a (inactive-show suppression)** A `SYNC_PROBLEM_CODES` alert unresolved after its show is archived/unpublished → no realtime email (the `shows` join requires `published ∧ ¬archived`).
- **AC-B3.8** Resend `infra_error` → a `failed` row + `EMAIL_DELIVERY_FAILED` + retry next tick; at `attempt_count=3` retries stop; no email about the failure. `EMAIL_DELIVERY_FAILED` resolves when no at-cap failure remains for the scope.
- **AC-B3.9** Missing provider env → send loops skip (200, no throw), `EMAIL_NOT_CONFIGURED` once; **maintenance still runs** — with a stale heartbeat + missing provider, both `SYNC_STALLED` and `EMAIL_NOT_CONFIGURED` are raised in-app and zero sends attempted; `EMAIL_NOT_CONFIGURED` resolves once configured.
- **AC-B3.10** Each toggle: write flips `app_settings`; fail-closed read reflects it; the gated effect changes — full flag-lifecycle, no zombie cell.
- **AC-B3.11** `email_deliveries` rejects `INSERT/UPDATE/DELETE` from `authenticated`/`anon` (service-role-only); the lockdown meta-test pins it.
- **AC-B3.11a** `email_deliveries.recipient` rejects non-canonical (`'Doug@X'`, spaces, empty) via the CHECK; the email-canonicalization schema-CHECK audit covers it.
- **AC-B3.12** No raw error code in any email body or toggle copy (invariant 5); no em dash in new copy (DESIGN.md §9).
- **AC-B3.12a (HTML escaping)** A show title / `drive_file_name` / context value containing `<script>` / `"><img onerror>` / a deceptive link renders escaped in the email `html` (no injected tags).
- **AC-B3.13** The cron route 401s a request without `Authorization: Bearer $CRON_SECRET` before any work.
- **AC-B3.14 (cron-context recipients)** The route invoked with `CRON_SECRET` + NO cookies + seeded active admins finds them and sends. (Negative-regression: route `activeRecipients()` through the RLS-bound `listAdminEmails` → zero recipients under cron.)
- **AC-B3.15 (revoked recipient)** A recipient revoked before the send is not emailed (re-check just before the provider call → skip).

## 13. Watchpoints / do-not-relitigate
- **At-least-once + the Resend Idempotency-Key is the chosen delivery contract (RD-9, owner-approved).** A rare duplicate after a process crash beyond the 24h replay window is **accepted**, not a defect. Do NOT re-introduce the claim-before-send / session-lock / terminal-state / submission-state / window-expiry / attempt-scoped-key machinery — that was explicitly trimmed (B3-DEF-9).
- **Accepted residual edges (all benign for a single-operator notifier):** a crash mid-send may rarely duplicate one email (idem-key covers the 24h common case); a >1h-stuck sheet recovering the same minute may get one heads-up; a digest retried after a crash may show slightly-stale content. None cause spam, silent misses, or privacy leaks.
- **No per-show advisory lock in the notify job** (§4.4 step 5) — it mutates only `admin_alerts` (idempotent upsert) + `email_deliveries`.
- **`SYNC_STALLED` is a show-wide cron-heartbeat alert, NOT per-show `last_synced_at`** (§4.2) — `last_synced_at` bumps on failures + skips quiet shows; per-show failures are covered by their error alerts.
- **`activeRecipients()` uses the service-role client, NOT `listAdminEmails`** (§5.1) — the RLS-bound reader returns zero rows under cron. Recipient-active re-check is on the send path (privacy: never email a revoked admin).
- **Provider Idempotency-Key is a BOUNDED hash including `recipient`** (§3.2) — under Resend's 256-char cap; `recipient` must be in the key or the provider would dedup-away the 2nd admin.
- **Recovery-resolution is PER-CODE, not just "became OK"** (§4.3) — resolve a `SYNC_PROBLEM_CODES` alert when the show's `(last_sync_status, last_sync_error)` no longer maps to that code, including an error-code SWITCH with no OK interlude; the notify-side recheck compares against `last_sync_error`. Without it, cleared/switched errors email stale + recurrences coalesce into the wrong row. `EMAIL_DELIVERY_FAILED`/`EMAIL_NOT_CONFIGURED` resolve symmetrically (§6).
- **The digest query must select the realtime occurrence-key inputs** (§5.3) — `pending_ingestions.first_seen_at` (+ show-level `raised_at`); the dashboard's default select omits `first_seen_at`, so an already-realtime-emailed stuck sheet would leak into the digest without it.
- **Maintenance always runs; only DELIVERY is gated** (§4.4 step 0, §6) — `SYNC_STALLED` detect/resolve must not be blinded by a toggle-off / unconfigured provider.
- **Digest + realtime-ingestion both apply `wizard_session_id IS NULL`** (§4.5, §5.3); digest content is built fresh from the live inbox via the service-role client.
- **The 2026-05-09 notification memo is principles-only** (§1.3) — its signed-token/feedback machinery is superseded by the crew-auth pivot + B2.
- **Both toggles default ON; digest hourly-fire + ET-hour window is the DST-safe design** (RD-8, §5.3).
