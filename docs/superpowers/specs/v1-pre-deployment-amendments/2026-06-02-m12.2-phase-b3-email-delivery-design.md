## 1. Scope and boundaries

### 1.1 Problem
The "Alert me about sync problems" Preferences toggle (admin-redesign prototype) and the *Daily review digest* push are both **operator attention surfaces**: Doug's natural surface is Drive, not the admin dashboard, so a staged event or a stalled sync that lives only in-app is functionally invisible (notification-design-memo §"The problem this solves"). The in-app `admin_alerts` surface (AlertBanner / NotifBell) is the *system of record* (pull); B3 adds the *push* layer (email) over it. Per **AGENTS.md flag-lifecycle / no-zombie-flag** rule, neither toggle may ship without its working backend — hence the toggles ship *with* the email subsystem in this milestone.

### 1.2 In scope
- First outbound-email path in the app: `lib/notify/` channel-agnostic seam (email-only impl) + Resend integration + secrets.
- Real-time sync-problem email tier (RD-5/RD-6): consume `admin_alerts` + staleness detector (§4.2) + stuck-ingestion detector (§4.5), delivered via an atomic claim-before-send ledger (§3.2).
- Daily review-digest email tier (RD-7), DST-safe scheduling.
- `email_deliveries` dedup/audit ledger.
- Two new `app_settings` boolean columns + read paths + write server-actions + the two Preferences toggles (UI).
- Three new §12.4 codes (`SYNC_STALLED`, `EMAIL_DELIVERY_FAILED`, `EMAIL_NOT_CONFIGURED`).
- New `app/api/cron/notify/route.ts` + two pg_cron schedules.
- TDD tests + meta-test registrations.

### 1.3 Out of scope (deferred — see §12)
Two-way feedback, inbound webhook, apply-from-email, unpublish tokens, SMS impl, per-admin prefs, digest→realtime promotion, crew-facing notifications, multi-operator routing. The 2026-05-09 `notification-design-memo.md` is cited for **principles only** (push-not-pull; severity tiering; quiet-success; coalescing). Its signed-token/feedback machinery is superseded by the 2026-05-23 crew-auth pivot and the B2 lifecycle redesign and is **not** built here.

## 2. Architecture overview

```
                         ┌──────────────────────────────────────────────┐
   pg_cron (pg_net)      │  app/api/cron/notify/route.ts  (CRON_SECRET)  │
   ─ realtime sched ───► │   ?job=realtime                              │
   ─ digest sched  ───►  │   ?job=digest                                │
                         └───────────────┬──────────────────────────────┘
                                         │ service-role Supabase client
              ┌──────────────────────────┼───────────────────────────────┐
   realtime:  │ 1. staleness detect → upsert_admin_alert(SYNC_STALLED)    │
              │    / resolve fresh shows                                  │
              │ 2. read unresolved admin_alerts ∈ SYNC_PROBLEM_CODES,     │
              │    raised_at ≤ now()-1h, gated by alert_on_sync_problems  │
              │ 3. skip any (show,code) already in email_deliveries       │
              │ 4. lib/notify.send(email) → Resend                       │
              │ 5. insert email_deliveries row (sent|failed)             │
   digest:    │ 1. gated by daily_review_digest; ET-hour==7 & not-sent  │
              │ 2. build needs-attention mirror (group by show, exclude  │
              │    real-time-alerted); 0 items → return (no send)        │
              │ 3. lib/notify.send(one email) → Resend; ledger row       │
              └──────────────────────────────────────────────────────────┘
```

Recipients: `listAdminEmails()` where `revoked_at IS NULL`. The job mutates only `admin_alerts` (via the idempotent SECURITY DEFINER `upsert_admin_alert`) and the new `email_deliveries` table — **not** the invariant-2 protected set — so it holds **no** per-show advisory lock (documented `// not-subject-to-meta` rationale in §4.4).

## 3. Data model

### 3.1 `app_settings` columns (B3 migration)

```sql
alter table public.app_settings

```
- Idempotent (`add column if not exists`), apply-twice safe.
- `app_settings` is a singleton (`id='default'`, `app_settings_singleton` CHECK at `supabase/migrations/20260501001000_internal_and_admin.sql:246`) gated by the `admin_only` RLS policy (`20260501002000_rls_policies.sql:131-134`). ### 3.2 `email_deliveries` ledger (new table, B3 migration)
```sql
create table public.email_deliveries (
  id                  uuid primary key default gen_random_uuid(),
  kind                text not null check (kind in ('realtime_problem','digest')),
  channel             text not null default 'email' check (channel in ('email','sms','webhook')),
  dedup_key           text not null,        -- see dedup-key shapes below
  show_id             uuid references public.shows(id) on delete set null,
  recipient           text not null,
  triggered_codes     text[] not null default '{}',
  status              text not null default 'claimed' check (status in ('claimed','sent','failed')),
  provider_message_id text,
  error               text,
  attempt_count       int  not null default 1,
  source_raised_at    timestamptz,          -- realtime: admin_alert.raised_at OR pending_ingestions.first_seen_at this delivery covers
  claimed_at          timestamptz not null default now(),
  created_at          timestamptz not null default now(),
  sent_at             timestamptz,
  constraint email_deliveries_recipient_email_canonical
    check (recipient = lower(trim(recipient)) and recipient <> '')
);
-- ONE delivery per logical message per recipient, REGARDLESS of status — this full
-- (not partial) unique index is the atomic-claim guard: a 'claimed' row blocks a
-- concurrent worker from sending the same message again.
create unique index email_deliveries_dedup
  on public.email_deliveries (kind, dedup_key, recipient);
```
- **DML lockdown:** REVOKE insert/update/delete from `anon`, `authenticated`; grant to `service_role` only; the cron route writes with the service-role client. Add an explicit service-role-only entry in `tests/db/postgrest-dml-lockdown.test.ts` so the lockdown class is pinned (no `authenticated` DML path exists).
- **Recipient canonicalization (invariant 3):** the schema CHECK above is the safety net; the writer ALSO canonicalizes via `lib/email/canonicalize.ts` before insert (recipients already arrive canonical from `listAdminEmails`, but the notify boundary re-canonicalizes defense-in-depth). The canonical recipient is part of the row identity, so a non-canonical slip would otherwise bypass dedup and duplicate-send.
- **Dedup-key shapes (single source of truth):**
  - *Realtime, show-level:* `'<show_id>:<code>:<floor(epoch(raised_at))>'` — `raised_at` from the `admin_alerts` row. `upsert_admin_alert` preserves `raised_at` across re-occurrence; only a *new* unresolved row (after the prior resolved) gets a fresh `raised_at`, so the epoch changes exactly when the problem genuinely recurs → re-arm.
  - *Realtime, pending-ingestion (pre-show, §4.5):* `'ingestion:<drive_file_id>:<floor(epoch(first_seen_at))>'`, `show_id = NULL`. Pre-show rows cannot use the `(show_id,code)` shape; `first_seen_at` re-arms when a later ingestion of the same file fails afresh.
  - *Digest:* `'digest:' || to_char((now() at time zone 'America/New_York')::date, 'YYYY-MM-DD')` → one digest per ET calendar day per recipient.
- **Atomic claim-before-send (concurrency-safe — closes the duplicate-send race):** the full unique index makes the ledger the lock. Per candidate `(kind, dedup_key, recipient)` the worker runs ONE atomic upsert that wins for a brand-new message OR a retryable `failed` row, and no-ops for an already-`claimed`/`sent`/cap-reached row:
  ```sql
  insert into public.email_deliveries
    (kind, dedup_key, recipient, show_id, triggered_codes, source_raised_at, status)
  values ($1,$2,$3,$4,$5,$6,'claimed')
  on conflict (kind, dedup_key, recipient) do update
    set status='claimed',
        attempt_count = public.email_deliveries.attempt_count + 1,
        claimed_at = now()
    where public.email_deliveries.status='failed'
      and public.email_deliveries.attempt_count < <SEND_RETRY_CAP>
  returning id, attempt_count;
  ```
  Only the worker that gets a returned row sends — `lib/notify.send()` → Resend with the provider **Idempotency-Key = `dedup_key`** — then `update`s its row to `sent` (+`provider_message_id`,`sent_at`) or `failed` (+`error`). A worker that gets no row skips (another worker owns it, or it is already sent / cap-reached). The send happens strictly **after** the claim, never before. Two concurrent ticks: one wins the INSERT, the other's conflict-update WHERE is false (row is `claimed`, not `failed`) → 0 rows → no send. Exactly-once.
- **Stale-claim recovery:** a row stuck in `claimed` (worker crashed after claiming, possibly after the Resend call) older than `CLAIM_STALE_MS` is made re-eligible (a sweep flips it to `failed` so the claim upsert can re-acquire it). Because every send passes Idempotency-Key = `dedup_key`, any resend is a provider-side no-op → no duplicate leaves the system even across a crash.
- **Guard conditions:** `show_id` nullable (digest + pending-ingestion rows carry NULL); `triggered_codes` defaults `{}` (never NULL); `provider_message_id`/`error` nullable by status; `recipient` NOT NULL + canonical CHECK.

## 4. Detection (real-time tier)

### 4.1 `SYNC_PROBLEM_CODES` set (single source of truth — `lib/notify/constants.ts`)
`['DRIVE_FETCH_FAILED','SHEET_PROCESS_FAILED','SHEET_UNAVAILABLE','PARSE_ERROR_LAST_GOOD','SYNC_STALLED']`. `STAGED_PARSE_*` race codes and all info-severity codes are **excluded** (dashboard-only). This array is referenced by both the detection query and the tests — never re-listed inline. NOTE: this set governs only the `admin_alerts`-consumption path (§4.3/§4.4 step 2); the pre-show pending-ingestion path (§4.5) is detected directly and is NOT keyed by this set.

### 4.2 Staleness detector (the only new detection logic)
- Runs first in `?job=realtime`. For each show that is `published = true` AND `archived = false` (watched, expected to sync) AND `now() - coalesce(last_synced_at, '-infinity') > STALENESS_THRESHOLD_MS`: `upsert_admin_alert(show_id, 'SYNC_STALLED', jsonb_build_object('last_synced_at', last_synced_at))`.
- **Resolve path:** when a show that previously had an unresolved `SYNC_STALLED` row syncs successfully (`now() - last_synced_at <= threshold`), set `resolved_at = now()` on that row (so it both clears the in-app banner and re-arms the email dedup for a future stall).
- **Guard conditions:** `last_synced_at IS NULL` (never synced) on a published show counts as stalled (coalesce to `-infinity`). A show flips `published=false` or `archived=true` → no longer evaluated; its open `SYNC_STALLED` is resolved on the same tick.

### 4.3 Gap-fill audit (implementation task, not a runtime path)
Confirm each non-staleness condition in `SYNC_PROBLEM_CODES` already upserts an `admin_alerts` row on the sync/ingestion paths (call sites: `lib/sync/runScheduledCronSync.ts:745`, `lib/sync/applyStaged.ts:1642`, `lib/sync/promoteSnapshot.ts:132`, `lib/sync/runOnboardingScan.ts:455`, `lib/adminAlerts/upsertAdminAlert.ts:39`, plus push-path). Where a condition does not produce an alert today, add the `upsert_admin_alert` call **in B3** so the email tier never re-detects. The audit's findings + any added call sites are documented in the plan; the email tier reads only `admin_alerts`.

### 4.4 Delivery loop (realtime)
Candidates come from THREE sources: §4.2 staleness (→ `admin_alerts.SYNC_STALLED`), §4.3 consumed error alerts (other `SYNC_PROBLEM_CODES`), and §4.5 pending-ingestion (read directly).
1. Gate: read `alert_on_sync_problems` fail-closed (`lib/appSettings/getAlertOnSyncProblems.ts`, mirroring `getAutoPublishCleanFirstSeen.ts`). False / read-error → no realtime emails this tick.
2. **Show-level candidates:** select unresolved `admin_alerts` where `code = any(SYNC_PROBLEM_CODES)` AND `raised_at <= now() - interval '1 hour'`; candidate key `'<show_id>:<code>:<epoch(raised_at)>'`, `source_raised_at = raised_at`.
3. **Pending-ingestion candidates (§4.5):** select `pending_ingestions` where `now() - first_seen_at > interval '1 hour'`; candidate key `'ingestion:<drive_file_id>:<epoch(first_seen_at)>'`, `show_id = NULL`, `triggered_codes = ARRAY[last_error_code]`, `source_raised_at = first_seen_at`.
4. For each candidate × each active recipient: run the **atomic claim** (§3.2). Only the claim winner calls `lib/notify.send()` → Resend (Idempotency-Key = `dedup_key`), then updates its row `sent` / `failed`. On failure emit `EMAIL_DELIVERY_FAILED` (§6). No claim row returned → skip.
5. **Advisory-lock posture:** the loop mutates only `admin_alerts` (idempotent SECURITY DEFINER upsert) + `email_deliveries`, and READS `shows`/`pending_ingestions`; it does not MUTATE `shows`/`crew_members`/`crew_member_auth`/`pending_syncs`/`pending_ingestions`, so per **invariant 2** it holds **no** per-show advisory lock. Where a meta-test or reviewer expects a lock, the code carries `// not-subject-to-meta: notify job mutates only admin_alerts (via idempotent upsert_admin_alert) + email_deliveries; no invariant-2 table mutation`.

### 4.5 Stuck-ingestion detector (pre-show — second documented exception to RD-6)
Pending ingestions are pre-show (no `show_id`), so an `admin_alerts` upsert (`coalesce(show_id::text,''), code` conflict key) would collapse every stuck file of the same `code` into ONE row and cannot dedup per-file. Therefore this class is detected by reading `pending_ingestions` **directly** and dedup'd in `email_deliveries` by `drive_file_id` — NOT routed through `admin_alerts`. This is the second deliberate exception to "consume admin_alerts" (the first, staleness §4.2, conversely ADDS to admin_alerts). Both exceptions are enumerated in §13 do-not-relitigate.
- **Select:** `pending_ingestions` rows older than `STALENESS_THRESHOLD_MS` (`now() - first_seen_at > 1h`). A row's mere presence means the sheet has not become a show; a successful re-ingestion deletes the row → no email (quiet-success). Sub-1h transients send nothing.
- **Dedup / re-arm:** `'ingestion:<drive_file_id>:<floor(epoch(first_seen_at))>'`, `show_id = NULL`. Re-arms only if a later ingestion of the same `drive_file_id` fails afresh (new `first_seen_at`).
- **Email:** body renders the row's `last_error_code` `dougFacing` (e.g. `SHEET_PROCESS_FAILED`) + `drive_file_name`; link → `${NEXT_PUBLIC_SITE_ORIGIN}/admin` (no show slug exists yet). Gated by the same `alert_on_sync_problems` toggle.
- **Guard:** `last_error_code` is NOT NULL (schema); presence + age is the trigger, `attempt_count` informational. `wizard_session_id` rows mid-onboarding are still pre-show and eligible once aged.

### 5.3 Cron route + scheduling

- **Two pg_cron schedules** (B3 migration, per the 2026-05-26 pg-cron pivot; `pg_net` HTTP POST with the `Authorization: Bearer <CRON_SECRET>` header — the secret mirrored in Supabase Vault per the pivot spec §2.3):

  - `digest` — **hourly**; the route computes the current `America/New_York` hour and proceeds only if `hour === DIGEST_HOUR_LOCAL` AND no `digest:<ET_date>` ledger row exists yet. **Hourly + ET-hour check is DST-safe** (no UTC offset hardcoded), honoring the validation-tooling-tz-pin discipline. The ET-date/hour computation uses an explicit `America/New_York` zone, never the host TZ.

## 6. Failure & unconfigured-provider handling

- **Send failure:** the claim winner (§3.2) gets `{ ok:false, kind:'infra_error' }` from Resend → `update` its claimed row to `status='failed'` (+`error`; `attempt_count` was already incremented at claim time); emit `upsert_admin_alert(show_id, 'EMAIL_DELIVERY_FAILED', …)` (in-app only — **never** email about email failing). A later tick re-acquires the row via the §3.2 atomic claim while `attempt_count < SEND_RETRY_CAP (3)`; at the cap the claim's `WHERE` predicate excludes it (no further send) and the in-app alert persists. Stale `claimed` rows (crash after claim) are recovered per §3.2 (flip to `failed` after `CLAIM_STALE_MS`; Idempotency-Key makes any resend a provider no-op). Quiet-success is preserved (no "still failing" emails).
- **Unconfigured provider:** missing `RESEND_API_KEY` or `EMAIL_FROM` → the job no-ops gracefully (no throw, no crash of the cron tick), emits `upsert_admin_alert(NULL, 'EMAIL_NOT_CONFIGURED', …)` **once** (deduped by the upsert conflict key), and returns 200 with a body indicating skipped. Toggles remain settable but inert. The settings UI MAY render a small "email not configured" hint when the env is absent (server-derived boolean prop; never leaks the secret).
- **Fail-closed reads:** both toggle reads fail-closed (a read error is treated as "off") so an infra hiccup never spams.

## 9. Acceptance criteria

- **AC-B3.1** With `alert_on_sync_problems=true`, an unresolved `SYNC_PROBLEM_CODES` alert whose `raised_at ≤ now()-1h` produces exactly one `sent` `email_deliveries` row per active recipient and one Resend send; a second realtime tick produces **no** additional send (dedup).
- **AC-B3.1a (concurrency)** Two overlapping realtime ticks for the same alert+recipient produce **exactly one** Resend send and one `sent` row: the claim winner sends, the loser's conflict-update `WHERE` is false → no send. (Test: two workers race the §3.2 atomic claim against one candidate.)
- **AC-B3.2** With `alert_on_sync_problems=false` (or read error), the same alert produces **zero** realtime emails; the in-app alert is unaffected.
- **AC-B3.3** An alert that resolves and later re-opens (new `raised_at`) re-arms: a new realtime email is sent (negative-regression: stashing the `raised_at` component of the dedup key makes the second email fail to fire).
- **AC-B3.4** A transient problem that clears before 1h produces **no** email (quiet-success).
- **AC-B3.5** A published, non-archived show with `now()-last_synced_at > 1h` upserts a `SYNC_STALLED` alert; a subsequent successful sync resolves it. A `last_synced_at IS NULL` published show is treated as stalled.
- **AC-B3.5a (pending-ingestion)** A `pending_ingestions` row with `now()-first_seen_at > 1h` and `alert_on_sync_problems=true` produces exactly one realtime email per active recipient, dedup'd by `'ingestion:<drive_file_id>:<epoch(first_seen_at)>'` (show_id NULL); a second tick sends nothing; deleting the row (successful re-ingestion) before 1h sends nothing; a fresh ingestion of the same file (new `first_seen_at`) re-arms.
- **AC-B3.6** With `daily_review_digest=true`, at ET-hour 07 with ≥1 open needs-attention item across shows, exactly one digest email is sent, grouped by show, excluding items already real-time-alerted; a second run that same ET day sends nothing (per-day dedup). Zero open items → no send.
- **AC-B3.7** Digest scheduling is DST-safe: across a spring-forward / fall-back boundary, the digest still fires at 07:00 ET exactly once (test injects the ET clock).
- **AC-B3.8** Resend `infra_error` → a `failed` ledger row + `EMAIL_DELIVERY_FAILED` in-app alert + retry next tick; at `attempt_count = 3` retries stop. No email is sent about the failure.
- **AC-B3.9** Missing `RESEND_API_KEY`/`EMAIL_FROM` → the cron job no-ops (200, no throw), emits `EMAIL_NOT_CONFIGURED` once; toggles still writable.
- **AC-B3.10** Each toggle: writing via its server action flips `app_settings`; the fail-closed read reflects it; the gated effect (email send / suppress) changes accordingly — full flag-lifecycle, no zombie cell.
- **AC-B3.11** `email_deliveries` rejects `INSERT/UPDATE/DELETE` from `authenticated`/`anon` (service-role-only); the lockdown meta-test pins it.
- **AC-B3.11a** `email_deliveries.recipient` rejects a non-canonical value (`'Doug@X'`, leading/trailing space, empty) via the `email_deliveries_recipient_email_canonical` CHECK; the email-canonicalization schema-CHECK audit covers the new column.
- **AC-B3.12** No raw error code appears in any email body or toggle copy (invariant 5); no em dash in any new copy (DESIGN.md §9).
- **AC-B3.13** The cron route rejects a request without `Authorization: Bearer $CRON_SECRET` (401) before any work.

## 10. Testing & meta-tests

**TDD per task** (red → minimal green → commit). Each test states its concrete failure mode.

- **Staleness detector:** stale published show → upserts `SYNC_STALLED`; fresh → resolves; NULL `last_synced_at` → stalled; archived/unpublished → not evaluated + open alert resolved. (Failure mode: a silently-dead cron never alerts.)
- **Realtime flush:** AC-B3.1/.2/.3/.4 — derive ages from fixture `raised_at`, never hardcode; **negative-regression** on the dedup key (stash `raised_at` epoch → assert second email fires) and on the toggle gate (flip OFF → assert zero sends). (Failure mode: dedup that never re-arms, or a gate that doesn't gate.)
- **Concurrency (claim-before-send):** AC-B3.1a — two workers race the §3.2 atomic claim against one candidate (real-DB / two-connection test, à la B2's `raceArchiveAgainst`); assert exactly one `sent` row + one mocked-Resend call. **Negative-regression:** revert to the partial `where status='sent'` index (or move the ledger insert after the send) → assert the test now sees two sends. (Failure mode: the duplicate-send race Codex flagged.)
- **Pending-ingestion realtime:** AC-B3.5a — a hard-failed `pending_ingestions` row aged >1h emails once, dedup'd by `drive_file_id`; row deleted before 1h → no email; re-armed on fresh `first_seen_at`. (Failure mode: RD-5's promised class never delivered.)
- **Recipient canonical CHECK:** AC-B3.11a — insert non-canonical recipient → CHECK violation; extend `tests/cross-cutting/email-canonicalization.test.ts` schema-CHECK coverage to `email_deliveries.recipient`. (Failure mode: a non-canonical recipient bypasses dedup → duplicate send + corrupt audit.)
- **Digest:** AC-B3.6/.7 — assert grouping + exclusion against the **data source** (the needs-attention query result), not the rendered email container (anti-tautology); inject an `America/New_York` clock for the DST test. ## 13. Watchpoints / disagreement-loop preempts (do-not-relitigate)

For the adversarial reviewer — each contract below is intentional and cited; do not relitigate without a new owner decision:

- **No per-show advisory lock in the notify job** is correct (§4.4): the job mutates only `admin_alerts` (idempotent SECURITY DEFINER upsert) + `email_deliveries`, neither in the invariant-2 protected set. Inline `// not-subject-to-meta` documents it.
- **Two deliberate detection exceptions to "consume admin_alerts"** (RD-6, §4.2, §4.5): staleness ADDS a `SYNC_STALLED` row to `admin_alerts`; pending-ingestion reads `pending_ingestions` DIRECTLY (because pre-show rows have no `show_id` and the `(show_id,code)` upsert cannot dedup per-file). Both are intentional, not gaps.
- **Atomic claim-before-send + full (non-partial) unique index** (§3.2) is the correct duplicate-send defense, not over-engineering: the ledger is the concurrency lock; the provider Idempotency-Key covers the crash-after-send window. Do not "simplify" back to insert-after-send.
- **`app_settings` writes are RLS `admin_only`-gated, not RPC-gated** (§3.1) — this matches the ratified B2 `setAutoPublish` pattern (`app/admin/settings/_actions/setAutoPublish.ts:46`); B3 does not introduce an RPC for the toggles.
- **Email failures never trigger emails** (§6) — `EMAIL_DELIVERY_FAILED` / `EMAIL_NOT_CONFIGURED` are in-app only, by design (no feedback loop, no spam).
- **The 2026-05-09 notification memo is principles-only** (§1.3) — its signed-token/feedback/unpublish machinery is superseded by the crew-auth pivot + B2 and is explicitly deferred, not omitted by oversight.
- **Both toggles default ON** (RD-8a) — owner decision; quiet-success keeps them silent when nothing is wrong.
- **Digest hourly-fire + ET-hour check is the DST-safe design** (§5.3), not an inefficiency to "optimize" into a single UTC cron time.
- **Server actions under `app/` are Opus-owned** (§11) per the UI hard rule, even though they touch `app_settings`.
