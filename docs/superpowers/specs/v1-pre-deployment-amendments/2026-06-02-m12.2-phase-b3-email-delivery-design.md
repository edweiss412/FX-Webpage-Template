## 1. Scope and boundaries

### 1.1 Problem
The "Alert me about sync problems" Preferences toggle (admin-redesign prototype) and the *Daily review digest* push are both **operator attention surfaces**: Doug's natural surface is Drive, not the admin dashboard, so a staged event or a stalled sync that lives only in-app is functionally invisible (notification-design-memo §"The problem this solves"). The in-app `admin_alerts` surface (AlertBanner / NotifBell) is the *system of record* (pull); B3 adds the *push* layer (email) over it. Per **AGENTS.md flag-lifecycle / no-zombie-flag** rule, neither toggle may ship without its working backend — hence the toggles ship *with* the email subsystem in this milestone.

### 1.2 In scope
- First outbound-email path in the app: `lib/notify/` channel-agnostic seam (email-only impl) + Resend integration + secrets.
- Real-time sync-problem email tier (RD-5/RD-6): consume `admin_alerts` + global stall detector (§4.2) + stuck-ingestion detector (§4.5), delivered via an atomic claim-before-send ledger (§3.2).
- Small **cross-into-sync-path** changes (Codex backend): (a) a sync-run heartbeat write (`app_settings.last_sync_cron_completed_at`) powering the global stall detector; (b) recovery-resolution calls that resolve `SYNC_PROBLEM_CODES` `admin_alerts` when a show exits its error state (§4.3 — required for the re-arm/no-stale-email contract).
- Daily review-digest email tier (RD-7), DST-safe scheduling.
- `email_deliveries` dedup/audit ledger.
- Two new `app_settings` boolean columns + read paths + write server-actions + the two Preferences toggles (UI).
- Three new §12.4 codes (`SYNC_STALLED`, `EMAIL_DELIVERY_FAILED`, `EMAIL_NOT_CONFIGURED`).
- New `app/api/cron/notify/route.ts` + two pg_cron schedules.
- TDD tests + meta-test registrations.

### 1.3 Out of scope (deferred — see §12)
Two-way feedback, inbound webhook, apply-from-email, unpublish tokens, SMS impl, per-admin prefs, digest→realtime promotion, crew-facing notifications, multi-operator routing. The 2026-05-09 `notification-design-memo.md` is cited for **principles only** (push-not-pull; severity tiering; quiet-success; coalescing). Its signed-token/feedback machinery is superseded by the 2026-05-23 crew-auth pivot and the B2 lifecycle redesign and is **not** built here.

## 2. Architecture overview

```

   pg_cron (pg_net)      │  app/api/cron/notify/route.ts  (CRON_SECRET)  │
   ─ realtime sched ───► │   ?job=realtime                              │
   ─ digest sched  ───►  │   ?job=digest                                │
                         └───────────────┬──────────────────────────────┘
                                         │ service-role Supabase client
              ┌──────────────────────────┼───────────────────────────────┐
   realtime:  │ 0. stale-claim sweep (claimed>CLAIM_STALE_MS → failed)   │
              │ 1. global stall detect (app_settings heartbeat) →        │
              │    show-wide SYNC_STALLED upsert / resolve when fresh     │
              │ 2. candidates (gated by alert_on_sync_problems):         │
              │    2a show-level admin_alerts ∈ SYNC_PROBLEM_CODES        │
              │       JOIN shows (published ∧ ¬archived), raised_at≤-1h   │
              │    2b global SYNC_STALLED (show_id NULL)                  │
              │    3  pending_ingestions >1h, wizard_session_id IS NULL   │
              │       (finalize under per-file advisory lock, §4.5)       │
              │ 4. per recipient: atomic CLAIM → Resend (idem-key         │
              │    kind:dedup_key:recipient) → ledger sent|failed         │
   digest:    │ 0. stale-claim sweep; ET-hour ∈ [7,10) else no-op        │
              │ 1. gated by daily_review_digest                          │
              │ 2. needs-attention mirror (group by show, exclude RT-    │
              │    alerted, wizard_session_id IS NULL); 0 items → no send │
              │ 3. per recipient: atomic CLAIM → Resend → ledger         │
              └──────────────────────────────────────────────────────────┘
```

Recipients: a **service-role** read of `admin_emails` where `revoked_at IS NULL` (§5.1 — NOT the cookie/RLS-bound `listAdminEmails`, which returns zero rows under cron). The job mutates only `admin_alerts` (via the idempotent SECURITY DEFINER `upsert_admin_alert`) and the new `email_deliveries` table — **not** the invariant-2 protected set — so it holds **no** per-show advisory lock (documented `// not-subject-to-meta` rationale in §4.4).

### 3.2 `email_deliveries` ledger (new table, B3 migration)
```sql
create table public.email_deliveries (
  id                  uuid primary key default gen_random_uuid(),
  kind                text not null check (kind in ('realtime_problem','digest')),
  channel             text not null default 'email' check (channel in ('email','sms','webhook')),
  dedup_key           text not null,        -- see dedup-key shapes below
  show_id             uuid references public.shows(id) on delete set null,
  recipient           text not null,
  triggered_codes     text[] not null default '{}',
  context             jsonb not null default '{}',   -- render inputs (show/file name, codes) so a retry can re-issue from the ledger ALONE, without the source row
  idempotency_key     text generated always as ('fxav:' || kind || ':' || encode(sha256((kind || ':' || dedup_key || ':' || recipient)::bytea), 'hex')) stored,  -- Resend Idempotency-Key: BOUNDED (~90 chars, well under Resend's 256-char cap) deterministic hash; the raw (kind,dedup_key,recipient) stay as columns for dedup/audit
  status              text not null default 'claimed' check (status in ('claimed','sent','failed')),
  provider_message_id text,
  error               text,
  terminal_reason     text,   -- non-null => TERMINAL ('recipient_revoked' | 'idempotency_window_expired' | 'digest_window_passed' | 'digest_superseded'); excluded from the claim predicate AND reconciliation so it is never re-sent
  attempt_count       int  not null default 0,   -- COMPLETED provider attempts (incremented on real send-failure, NOT at claim)
  source_raised_at    timestamptz,          -- realtime: admin_alert.raised_at OR pending_ingestions.first_seen_at this delivery covers
  claimed_at          timestamptz not null default now(),
  first_attempt_at    timestamptz,   -- set IMMUTABLY before the FIRST provider send (only if null); the idempotency-window guard (§4.4 step 1.5) ages against THIS, never the mutable claimed_at
  created_at          timestamptz not null default now(),
  sent_at             timestamptz,
  constraint email_deliveries_recipient_email_canonical
    check (recipient = lower(trim(recipient)) and recipient <> '')
);
-- ONE delivery per logical message per recipient, REGARDLESS of status — this full
-- (not partial) unique index is the atomic-claim guard: a 'claimed' row blocks a
-- concurrent worker from sending the same message again.
create unique index email_deliveries_dedup
  on public.email_deliveries (kind, dedup_key, recipient);
```
- **DML lockdown:** REVOKE insert/update/delete from `anon`, `authenticated`; grant to `service_role` only; the cron route writes with the service-role client. Add an explicit service-role-only entry in `tests/db/postgrest-dml-lockdown.test.ts` so the lockdown class is pinned (no `authenticated` DML path exists).
- **Recipient canonicalization (invariant 3):** the schema CHECK above is the safety net; the writer ALSO canonicalizes via `lib/email/canonicalize.ts` before insert (recipients already arrive canonical from `listAdminEmails`, but the notify boundary re-canonicalizes defense-in-depth). The canonical recipient is part of the row identity, so a non-canonical slip would otherwise bypass dedup and duplicate-send.
- **Dedup-key shapes (single source of truth):**
  - *Realtime, global stall:* `'global:SYNC_STALLED:<floor(epoch(raised_at))>'`, `show_id = NULL` (the show-wide `SYNC_STALLED`, §4.2/§4.4 step 2b).
  - *Realtime, show-level:* `'<show_id>:<code>:<floor(epoch(raised_at))>'` — `raised_at` from the `admin_alerts` row. `upsert_admin_alert` preserves `raised_at` across re-occurrence; only a *new* unresolved row (after the prior resolved) gets a fresh `raised_at`, so the epoch changes exactly when the problem genuinely recurs → re-arm.
  - *Realtime, pending-ingestion (pre-show, §4.5):* `'ingestion:<drive_file_id>:<floor(epoch(first_seen_at))>'`, `show_id = NULL`. Pre-show rows cannot use the `(show_id,code)` shape; `first_seen_at` re-arms when a later ingestion of the same file fails afresh.
  - *Digest:* `'digest:' || to_char((now() at time zone 'America/New_York')::date, 'YYYY-MM-DD')` → one digest per ET calendar day per recipient.
- **Atomic claim-before-send (concurrency-safe — closes the duplicate-send race):** the full unique index makes the ledger the lock. Per candidate `(kind, dedup_key, recipient)` the worker runs ONE atomic upsert that wins for a brand-new message OR a retryable `failed` row, and no-ops for an already-`claimed`/`sent`/cap-reached row:
  ```sql
  insert into public.email_deliveries
    (kind, dedup_key, recipient, show_id, triggered_codes, context, source_raised_at, status)
  values ($1,$2,$3,$4,$5,$6,$7,'claimed')          -- $6 = context jsonb (render inputs); MUST be populated at claim
  on conflict (kind, dedup_key, recipient) do update
    set status='claimed', claimed_at = now()        -- NOTE: does NOT increment attempt_count
    where public.email_deliveries.status='failed'
      and public.email_deliveries.attempt_count < <SEND_RETRY_CAP>
      and public.email_deliveries.terminal_reason is null   -- never reclaim a TERMINAL row (revoked / idempotency-window-expired)
  returning id, attempt_count;
  ```
  **`context` MUST be populated at claim time** (not left `{}`) — it is the render input the ledger-driven retry uses after the source row is gone (else AC-B3.5e cannot re-render). Shape per kind: show-level = `{showName, slug, codes}`; pending-ingestion = `{driveFileName, lastErrorCode}`; digest = `{groupedItems}` (or a compact summary sufficient to re-render). The conflict-retry branch keeps the existing `context` (re-render uses the stored inputs).
  This combined upsert (DO UPDATE WHERE `failed`) is the **reconciliation/retry** claim (§4.4 step 1.5). NEW candidates (§4.4 step 4) instead use a plain `INSERT … ON CONFLICT DO NOTHING` (brand-new only), so retry of an existing `failed` row happens in exactly ONE place per tick (≤1 provider attempt per row per tick). The recipient-active re-check (close to the send) applies on BOTH paths.
  **`attempt_count` is incremented ONLY by an actual provider attempt** (the post-send `failed` update, below) — NEVER at claim time. This is load-bearing: if the claim incremented, a worker that crashed after claiming the FINAL allowed attempt would be swept to `failed` at `attempt_count = CAP` and excluded from retry forever, even though that send never happened (the crash consumed a budget slot it never used). With increment-on-attempt, a crash-after-claim leaves `attempt_count` unchanged → the row stays retryable.
  Only the worker that gets a returned row sends — `lib/notify.send()` → Resend with the provider **Idempotency-Key = the full per-recipient ledger identity `kind || ':' || dedup_key || ':' || recipient`** (a stable hash of those three fields) — then `update`s its row to `sent` (+`provider_message_id`,`sent_at`) or `failed` (+`error`, `attempt_count = attempt_count + 1`). The key MUST include `recipient`: `dedup_key` alone is shared by every recipient of the same logical message, so a recipient-less key would let Resend suppress all but the first admin while their ledger rows still flip to `sent` (a lying audit + silently-skipped admins). A worker that gets no row skips (another worker owns it, or it is already sent / cap-reached). The send happens strictly **after** the claim, never before. Two concurrent ticks: one wins the INSERT, the other's conflict-update WHERE is false (row is `claimed`, not `failed`) → 0 rows → no send. Exactly-once.
- **Stale-claim recovery (explicit, pinned as each job's FIRST step):** every notify job (realtime AND digest), before any candidate work, runs:
  ```sql
  update public.email_deliveries
    set status='failed', error='stale_claim'   -- does NOT touch attempt_count (a stale claim was not a real attempt)
    where status='claimed' and claimed_at < now() - interval '<CLAIM_STALE_MS> milliseconds';
  ```
  The sweep deliberately does NOT increment `attempt_count` (a crashed claim made no provider attempt), so a swept row stays retryable while `attempt_count < SEND_RETRY_CAP`.
  This frees a row where a worker crashed after claiming but before `sent`/`failed` — otherwise the full unique index + the "claim only re-acquires `failed` rows" rule would block that `(kind,dedup_key,recipient)` message forever (silent permanent suppression). A FRESH `claimed` row (recent `claimed_at`) is NOT swept and correctly stays blocked to concurrent workers. Because every send passes the per-recipient Idempotency-Key (above), any resend of a recovered row is a provider-side no-op → no duplicate leaves the system even across a crash. **The sweep is safe against a merely-SLOW (not dead) worker** because every send is bounded by `SEND_TIMEOUT_MS (30s) ≪ CLAIM_STALE_MS (10 min)`: a live worker resolves its row to `sent`/`failed` within 30s, so a row only remains `claimed` past 10 min if the worker truly died — never because a send is still in flight. The sweep must never reclaim a row whose send could still be running.
- **Ledger-driven convergence (outbox pattern — survives source-row deletion):** retry/convergence of a `claimed`(swept→`failed`)/`failed` row is driven ENTIRELY by the LEDGER row (its `recipient` + idem-key `kind:dedup_key:recipient` + stored `context`), NOT by re-discovering the source candidate. This closes the crash-after-send + recovery-deletes-source ordering: if a worker crashes after Resend accepted but before the `sent` update, then the recovery path deletes the `pending_ingestions` row (or the source `admin_alert` resolves), a later tick still reconciles the durable ledger row by RE-ISSUING the send with the same `idempotency_key` — per the verified Resend contract (§5.2): a `200` replay returns the original id (no second email) → `sent`; a `409 invalid_idempotent_request` means it already sent → `sent`. Either way the row converges, audit preserved. (Crash-BEFORE-send re-issues and actually sends — a bounded, accepted "real >1h problem resolving moments later" per the AC-B3.5b notify-wins semantics.) The `context` column makes the row self-sufficient to re-render. The provider Idempotency-Key is the exactly-once guarantee; the ledger is the durable outbox.
- **Guard conditions:** `show_id` nullable (digest + pending-ingestion rows carry NULL); `triggered_codes` defaults `{}` (never NULL); `provider_message_id`/`error` nullable by status; `recipient` NOT NULL + canonical CHECK.

## 4. Detection (real-time tier)

### 4.1 `SYNC_PROBLEM_CODES` set (single source of truth — `lib/notify/constants.ts`)
`['DRIVE_FETCH_FAILED','SHEET_PROCESS_FAILED','SHEET_UNAVAILABLE','PARSE_ERROR_LAST_GOOD']` (the show-level consumed error codes). `STAGED_PARSE_*` race codes and all info-severity codes are **excluded** (dashboard-only). This array is referenced by both the detection query and the tests — never re-listed inline. NOTE: (a) `SYNC_STALLED` is NOT in this set — it is a show-WIDE (`show_id IS NULL`) alert with its own global heartbeat detector (§4.2) + candidate path (§4.4 step 2b); (b) the pre-show pending-ingestion path (§4.5) is detected directly and is NOT keyed by this set.

### 4.2 Global sync-stall detector (`SYNC_STALLED`, show-wide)
`shows.last_synced_at` is the WRONG signal for staleness: the sync path bumps it on every PROCESSED outcome **including failures** (`lib/sync/runScheduledCronSync.ts:661/674/694/715` — `parse_error`/`pending_review`/`sheet_unavailable`/`drive_error` each `set last_synced_at = now()`), AND change-driven polling never bumps it for quiet/unchanged shows. A per-show `now()-last_synced_at` predicate therefore both MISSES a perpetually-failing show and FALSE-POSITIVES on a healthy quiet show. Per-show failures are already covered by their error alerts (§4.3); the only uncovered gap is the whole pipeline stopping. So `SYNC_STALLED` is a single SHOW-WIDE alert driven by a sync-run heartbeat:
- **Heartbeat (B3 adds — cross-into-sync-path, Codex backend):** the scheduled sync run writes `app_settings.last_sync_cron_completed_at = now()` at the END of every completed run (success OR all-quiet, regardless of whether any sheet changed — it signals "the pipeline ran", not "a show changed"). New `app_settings` column (B3 migration). Trigger-mechanism-agnostic (works whether the run is Vercel-cron- or pg_cron-triggered).
- **Detector (first in `?job=realtime`, after the stale-claim sweep):** if `now() - coalesce(last_sync_cron_completed_at, '-infinity') > STALENESS_THRESHOLD_MS` → `upsert_admin_alert(NULL, 'SYNC_STALLED', jsonb_build_object('last_completed_at', last_sync_cron_completed_at))` (one show-wide row; `upsert_admin_alert`'s `(coalesce(show_id::text,''), code)` conflict key collapses to exactly one unresolved global row, which is correct here).
- **Resolve:** when the heartbeat is fresh again (`<= threshold`), set `resolved_at = now()` on the open `SYNC_STALLED` (clears the banner + re-arms email dedup for a future stall).
- **Guard:** A perpetually-FAILING but still-RUNNING cron keeps the heartbeat fresh → NO false `SYNC_STALLED` (those shows surface via their per-show error alerts §4.3 instead). **Fresh-deploy:** the B3 migration initializes the heartbeat to `now()` on the singleton (§3.1), so a just-deployed system is NOT NULL and does not false-blast before its first sync run (AC-B3.5d); the `coalesce(..., '-infinity')` NULL→stalled rule is only a defensive fallback for an impossible-in-practice NULL.
- **Common-mode limitation (documented, narrowed contract):** the stall detector runs on the SAME scheduler (pg_cron/pg_net → notify route) it watches, so a TOTAL scheduler / pg_net / cron-auth outage stops the notify job too and SYNC_STALLED cannot self-detect a failure of its own scheduler. `SYNC_STALLED` therefore covers "the sync pipeline stopped completing runs while the notify scheduler still runs" (sync-job-specific failure, Drive disconnect, watch loss) — NOT a total scheduler outage. Detecting that requires an INDEPENDENT external uptime monitor pinging the notify/health endpoint; that is an ops concern tracked as **B3-DEF-8** (§12), deliberately out of the email subsystem's scope.

### 4.3 Gap-fill audit + recovery-resolution (implementation task — cross-into-sync-path)
Confirm each non-staleness `SYNC_PROBLEM_CODES` condition both (a) UPSERTS an `admin_alerts` row on entry (failure), AND (b) RESOLVES it (`resolved_at = now()`) on exit (the show next syncs OK / leaves the error `last_sync_status`). Upsert call sites: `lib/sync/runScheduledCronSync.ts:745`, `lib/sync/applyStaged.ts:1642`, `lib/sync/promoteSnapshot.ts:132`, `lib/sync/runOnboardingScan.ts:455`, `lib/adminAlerts/upsertAdminAlert.ts:39`, plus push-path. **Today resolution happens only on manual admin-resolve routes, NOT on sync recovery (verified).** So **B3 ADDS the recovery-resolution call** to the sync path for each code (when a show transitions OUT of the matching error state, resolve the open alert). This is load-bearing, not cosmetic: WITHOUT it (i) a transient error that already cleared still emails 1h later (stale), and (ii) because `upsert_admin_alert` coalesces into the never-resolved row (`raised_at`/`dedup_key` never change), a later GENUINE recurrence is SUPPRESSED — the re-arm model (resolved→reopened ⇒ fresh `raised_at` ⇒ new `dedup_key`) depends entirely on this resolution.
- **Notify-side defensive recheck (belt-and-suspenders):** before emailing a show-level candidate (§4.4 step 2a), the notify job re-reads the show's CURRENT `last_sync_status`; if the show no longer matches the alert's code-condition (it recovered), the job RESOLVES the alert and SKIPS the email. So even a missed sync-side resolve cannot produce a stale email.
- The audit's findings + every added upsert/resolve call site + the code→error-state map are documented in the plan; the email tier reads only `admin_alerts`.

### 4.4 Delivery loop (realtime)
Candidates come from THREE sources: §4.2 staleness (→ `admin_alerts.SYNC_STALLED`), §4.3 consumed error alerts (other `SYNC_PROBLEM_CODES`), and §4.5 pending-ingestion (read directly).
0. **Maintenance — ALWAYS runs (no provider/toggle needed; NEVER suppressed by an email-config mistake):** (a) stale-claim sweep — §3.2 recovery `update` (`status='claimed' AND claimed_at < now()-CLAIM_STALE_MS → 'failed'`); (b) global `SYNC_STALLED` detect + resolve (§4.2) and show-level recovery-resolution recheck (§4.3) — these upsert/resolve IN-APP `admin_alerts` so the dashboard stall/recovery signal stands even when email is OFF or unconfigured.
1. **Provider gate:** if the provider is unconfigured (§6) → maintain `EMAIL_NOT_CONFIGURED` and SKIP everything below (Resend unreachable). Else continue.
1.5. **Ledger-driven reconciliation (outbox §3.2 — provider-gated; each reissue gated on the ROW'S-KIND tier toggle):** re-issue `failed` rows with `terminal_reason IS NULL AND attempt_count < SEND_RETRY_CAP` from the row itself (recipient + idem-key + `context`) to converge crash-after-send rows — regardless of source-candidate existence, but **only when that row's KIND tier toggle is ON** (`kind='realtime_problem'` → `alert_on_sync_problems`; `kind='digest'` → `daily_review_digest`). A reissue is a provider call that MAY deliver an email (a crash-after-send row whose original did NOT actually reach Resend would send on replay), so OFF must gate it — "OFF = no emails" is strict, including retries. (Only step-0 maintenance is toggle-independent.) **KIND-SCOPED:** the **realtime job** reconciles ONLY `kind='realtime_problem'` rows (never digest). The **digest job** reconciles BOTH (a) `kind='realtime_problem'` rows — to converge crash-after-send realtime rows so its per-recipient exclusion is accurate (§5.3, AC-B3.6d) — AND (b) its OWN `kind='digest'` rows whose `dedup_key` is TODAY's ET date, inside the digest window. A `kind='digest'` row whose `dedup_key` ET-date ≠ today (or outside the window) is NOT reissued → `terminal_reason='digest_window_passed'` (audit-only). **Digest reissue REBUILDS content from the LIVE needs-attention inbox** (NOT the stored `context` — a 7am failure retried at 8am must reflect items Doug may have cleared since); if the rebuilt per-recipient content has ZERO current items → `terminal_reason='digest_superseded'` (no stale digest of already-handled work). (Realtime/ingestion reissues use stored `context`; only DIGEST reissues rebuild, because digest content is a point-in-time mirror.) Net: digest emails are sent only by the digest job, only for today, in-window; the realtime job never sends a digest row; a stale/off-window digest never goes out (nor a next-day reissue within the 24h idempotency window). (No contradiction with AC-B3.6d: the digest job DOES reconcile realtime rows — for exclusion accuracy — it just never reconciles a digest row for another day.) These are ALREADY-decided sends being confirmed, not new sends; the idem-key prevents duplicates. Two GUARDS before each reissue, each setting `terminal_reason` (which permanently excludes the row from BOTH the claim predicate §3.2 AND future reconciliation — so a terminal row is NEVER re-sent):
   - **(i) Recipient re-check (privacy):** confirm `recipient` is STILL an active `admin_emails` row (`revoked_at IS NULL`, service-role read). If revoked → set `terminal_reason='recipient_revoked'` (status stays `failed`) WITHOUT sending — never email a revoked admin, even on replay.
   - **(ii) Idempotency-window guard (no late duplicate):** only reissue if `first_attempt_at IS NULL OR first_attempt_at > now() - PROVIDER_IDEMPOTENCY_WINDOW_MS` (Resend's 24h replay window, minus a safety margin). Age against the IMMUTABLE `first_attempt_at` (set before the first provider call, never overwritten), NOT the mutable `claimed_at` — else a chain of pre-expiry retries each refreshing `claimed_at` could push a reissue past the real 24h-from-original mark and duplicate. The send path sets `first_attempt_at = now()` only when null, immediately before the first Resend call. A row whose possible original attempt is OLDER than the window is NOT auto-reissued (a reissue would fall outside Resend's retention → a NEW send → duplicate irreversible email). Set `terminal_reason='idempotency_window_expired'`, keep `EMAIL_DELIVERY_FAILED` open as **audit-uncertain**, surface for manual reconciliation (B3-DEF-9).
   Resolve `EMAIL_DELIVERY_FAILED` per §6 as rows converge. (While a row's-kind toggle is OFF, a crash-after-send row stays `failed`/audit-uncertain and converges on the next ON tick via idempotent replay — no email is sent meanwhile, the correct OFF semantics. The KIND gating also keeps AC-B3.6d consistent: if `alert_on_sync_problems` is OFF, realtime rows are not reissued, the admin never got a realtime email, and the item legitimately appears in the digest.)
1.7. **New-send toggle gate:** read the job's tier toggle fail-closed (realtime: `alert_on_sync_problems` via `lib/appSettings/getAlertOnSyncProblems.ts`, mirroring `getAutoPublishCleanFirstSeen.ts`; digest: `daily_review_digest`); false/read-error → skip NEW candidate sends (2a/2b/3/4). Maintenance (step 0) already ran; reconciliation (1.5) ran per-kind-toggle-gated.
2a. **Show-level candidates:** select unresolved `admin_alerts` JOIN `shows` ON `admin_alerts.show_id = shows.id` where `code = any(SYNC_PROBLEM_CODES)` AND `raised_at <= now() - interval '1 hour'` AND `shows.published = true` AND `shows.archived = false`. The `published ∧ ¬archived` filter is REQUIRED: an alert raised before a show is archived/unpublished stays unresolved, and emailing sync problems for an inactive show contradicts the silent-skip posture of archived sync paths (RD-5 scopes realtime to active shows). Candidate key `'<show_id>:<code>:<epoch(raised_at)>'`, `source_raised_at = raised_at`.
2b. **Global stall candidate:** select the unresolved show-wide `SYNC_STALLED` alert (`show_id IS NULL`, `code='SYNC_STALLED'`) with **NO additional age gate** — the 1h is ALREADY encoded in §4.2's heartbeat threshold at raise time, so applying another `raised_at <= now()-1h` would delay the first email to ~2h and violate RD-5's ~1h SLA. NO `shows` join (it is show-wide). Candidate key `'global:SYNC_STALLED:<epoch(raised_at)>'`, `show_id = NULL`, link → `/admin`. (The show-level path 2a keeps its +1h gate because those alerts' `raised_at` = the error time, so +1h = "persists >1h"; SYNC_STALLED's raise already means >1h.)
3. **Pending-ingestion candidates (§4.5):** select `pending_ingestions` where `now() - first_seen_at > interval '1 hour'` **AND `wizard_session_id IS NULL`** (live-scope — exclude active onboarding, matching §4.5 + the dashboard); candidate key `'ingestion:<drive_file_id>:<epoch(first_seen_at)>'`, `show_id = NULL`, `triggered_codes = ARRAY[last_error_code]`, `source_raised_at = first_seen_at`. The §4.5 lock re-read RE-asserts `wizard_session_id IS NULL` (a row that became wizard-scoped between selection and finalization is dropped, not emailed).
4. For each candidate × each active recipient: insert a BRAND-NEW claim with `ON CONFLICT (kind,dedup_key,recipient) DO NOTHING` — retry of a PRE-EXISTING `failed` row is EXCLUSIVELY step 1.5's job, so any one `email_deliveries` row gets **≤1 provider attempt per tick** (a candidate whose row already exists is left to 1.5; this prevents 1.5 + step 4 from each burning an `attempt_count` slot in the same run). The insert winner: (a) **re-check the recipient is still active** (`admin_emails.revoked_at IS NULL`, service-role, as close as possible to the send — this closes the revoke-after-`activeRecipients()`-snapshot leak on FIRST sends, not just replay); revoked → `terminal_reason='recipient_revoked'`, no send; (b) set `first_attempt_at=now()`; (c) `lib/notify.send()` → Resend (Idempotency-Key §3.2); (d) update `sent`/`failed` (+`EMAIL_DELIVERY_FAILED` on failure, §6). No row inserted (conflict) → skip.
5. **Advisory-lock posture:** the show-level (step 2) and staleness paths mutate only `admin_alerts` (idempotent SECURITY DEFINER upsert) + `email_deliveries`, and READ `shows`; they do not MUTATE `shows`/`crew_members`/`crew_member_auth`/`pending_syncs`/`pending_ingestions`, so per **invariant 2** they hold **no** per-show advisory lock (code carries `// not-subject-to-meta: notify job mutates only admin_alerts + email_deliveries; no invariant-2 table mutation`). **EXCEPTION:** the pending-ingestion finalization (§4.5) DOES acquire a SESSION-level `pg_try_advisory_lock(hashtext('show:'||drive_file_id))` to close the read-to-send delete race (and commit the claim durably before the send) — a separate connection acquiring the same hashkey the sync path uses (mutual exclusion, not nested), documented in §4.5's advisory-lock topology note + §10 + §13.

### 4.5 Stuck-ingestion detector (pre-show — second documented exception to RD-6)
Pending ingestions are pre-show (no `show_id`), so an `admin_alerts` upsert (`coalesce(show_id::text,''), code` conflict key) would collapse every stuck file of the same `code` into ONE row and cannot dedup per-file. Therefore this class is detected by reading `pending_ingestions` **directly** and dedup'd in `email_deliveries` by `drive_file_id` — NOT routed through `admin_alerts`. This is the second deliberate exception to "consume admin_alerts" (the first, staleness §4.2, conversely ADDS to admin_alerts). Both exceptions are enumerated in §13 do-not-relitigate.
- **Select:** `pending_ingestions` rows older than `STALENESS_THRESHOLD_MS` (`now() - first_seen_at > 1h`) **AND `wizard_session_id IS NULL`** (exclude rows in active onboarding — an admin mid-wizard is already looking at the file; this matches the dashboard live-scope at `components/admin/Dashboard.tsx:328-331`). A qualifying row's mere presence means the sheet has not become a show; a successful re-ingestion deletes the row → no email (quiet-success). Sub-1h transients send nothing.
- **Dedup / re-arm:** `'ingestion:<drive_file_id>:<floor(epoch(first_seen_at))>'`, `show_id = NULL`. Re-arms only if a later ingestion of the same `drive_file_id` fails afresh (new `first_seen_at`).
- **Email:** body renders the row's `last_error_code` `dougFacing` (e.g. `SHEET_PROCESS_FAILED`) + `drive_file_name`; link → `${NEXT_PUBLIC_SITE_ORIGIN}/admin` (no show slug exists yet). Gated by the same `alert_on_sync_problems` toggle.
- **Race-safe finalization (read-to-send guard):** the sync/recovery path DELETEs a successful `pending_ingestions` row while holding `pg_advisory_xact_lock(hashtext('show:'||drive_file_id))` (e.g. `lib/sync/runScheduledCronSync.ts:604-610`). A lock-less notify tick could read an aged row and send the *irreversible* "stuck" email just as the recovery commits the delete — a false email violating quiet-success. So the ingestion finalization, per candidate, acquires a SESSION-level `pg_try_advisory_lock(hashtext('show:'||drive_file_id))` (non-blocking: if held → skip this file this tick, retry next), RE-reads the row (still present AND aged?), commits the atomic claim, THEN sends + updates the ledger while still holding the session lock (mechanism below). This serializes against the recovery delete while keeping the claim durable. It is a deliberate, scoped exception to §4.4 step 5's no-lock posture.
  - **Lock-holding mechanism (PINNED — SESSION-level lock, claim committed BEFORE the send):** runs on a dedicated **postgres.js** connection using a SESSION-level advisory lock (NOT xact-level), so the claim is durably committed before Resend is called while mutual exclusion is still held:
    1. `select pg_try_advisory_lock(hashtext('show:'||drive_file_id))` — session-level, non-blocking; not acquired → skip this file this tick.
    2. `begin; <re-read row: present ∧ aged ∧ wizard_session_id IS NULL>; <atomic claim §3.2>; commit;` — the `claimed` row is now DURABLE. (Row gone on re-read → commit empty, jump to step 4.)
    3. `await Resend send` (Idempotency-Key §3.2) — the SESSION lock is still held, so the recovery DELETE (which takes `pg_try_advisory_xact_lock` on the same key namespace) stays blocked; then `update` the ledger row `sent`/`failed` in its own short tx.
    4. `finally: select pg_advisory_unlock(hashtext('show:'||drive_file_id))` (a dropped connection also auto-releases a session lock).
    **Why session- not xact-level (R13):** holding the UNCOMMITTED claim in one xact across the send means a crash/commit-failure AFTER Resend accepts the email rolls the claim back → a later tick re-sends and the audit row is lost. Committing the claim first (durable) + the provider Idempotency-Key makes crash-after-send safe: the durable `claimed` row is swept→retried, the resend is a provider no-op, the ledger converges to `sent`, audit preserved. Must NOT use the PostgREST/autocommit client (no session-lock control). Bounded `statement_timeout` + Resend timeout cap how long the session lock is held (low-frequency per-file path, ≤15 shows, hourly). postgres.js is the project's transactional DB layer (B2's `_b2Helpers`/`tx.unsafe`).
  - **Advisory-lock topology (invariant 2, single-holder):** the existing holder of `hashtext('show:'||drive_file_id)` is the sync/recovery path (cron `pg_try_advisory_xact_lock`; admin/blocking `pg_advisory_xact_lock`). The notify ingestion finalization acquires the SAME hashkey via a SESSION-level `pg_try_advisory_lock` on a SEPARATE connection — session and xact advisory locks share one namespace, so they mutually exclude correctly; it is NOT a nested acquisition within one call stack, so the single-holder/no-nested-deadlock rule holds. The session lock is always released (`pg_advisory_unlock` in `finally`, or connection-close on crash). Enumerated in `tests/auth/advisoryLockRpcDeadlock.test.ts` (§10). The show-level (§4.4 step 2) and staleness (§4.2) paths stay lock-free: their underlying state changes non-destructively (a just-resolved 1h+ alert emailing once is a mild, defensible over-notify, not a false success).
- **Guard:** `last_error_code` is NOT NULL (schema); presence + age is the trigger, `attempt_count` informational. `wizard_session_id IS NOT NULL` rows (active onboarding) are **EXCLUDED** (live-scope), consistent with both the dashboard inbox and the digest (§5.3).

### 5.2 Provider + secrets
- **Resend** (Vercel-native integration). Secrets `RESEND_API_KEY` + `EMAIL_FROM` (a verified sending domain, e.g. `notifications@<fxav-domain>`) in Vercel env (all environments). `EMAIL_FROM` is operator-configured; absence → unconfigured path (§6). Sending-domain DNS/verification is an operator prerequisite noted in the plan's deployment-prereqs.
- Provider swappable behind `lib/notify/send.ts`; no provider type leaks above the seam.
- **Resend idempotency contract (verified against Resend docs 2026-06-02 — the outbox §3.2 depends on this; the provider-boundary test re-verifies it, §10):** the send endpoint accepts an `Idempotency-Key` header (= the row's `idempotency_key`, a bounded ~90-char hash — Resend caps the key at **256 chars**, so the raw email-bearing concat is hashed, §3.2). Within a **24h** window: a repeat with the SAME key + SAME payload returns the ORIGINAL response (same email `id`) WITHOUT re-sending; a repeat with the same key + DIFFERENT payload returns **`409 invalid_idempotent_request`**. **Reconciliation handling (§3.2/§4.4 step 1.5):** re-issue with the row's `idempotency_key` + a payload re-rendered deterministically from `context`; on `200` → `status='sent'` (+ returned `provider_message_id`); on `409 invalid_idempotent_request` → the original DID send (key already used) → `status='sent'` (`provider_message_id` may stay null/unknown — audit still correctly records "sent"). Either branch converges with NO duplicate email. **Window invariant:** `CLAIM_STALE_MS (10min) ≪ 24h`, so reconciliation always occurs within Resend's idempotency window. If a future provider lacks idempotent replay, AC-B3.5e narrows to "row stays `failed` → `EMAIL_DELIVERY_FAILED` persists" (no false `sent`) rather than silently dropping the contract.

### 5.3 Cron route + scheduling
- `app/api/cron/notify/route.ts` — first line `const denied = rejectUnauthorizedCron(request); if (denied) return denied;` (`app/api/cron/_auth.ts:3-12`). Branch on `?job=` (`'realtime' | 'digest'`); unknown/missing job → 400.
- **Two pg_cron schedules** (B3 migration, per the 2026-05-26 pg-cron pivot; `pg_net` HTTP POST with the `Authorization: Bearer <CRON_SECRET>` header — the secret mirrored in Supabase Vault per the pivot spec §2.3):
  - `realtime` — aligned to the sync cadence (own schedule, RD-8c).
  - `digest` — **hourly**; the route proceeds when the current `America/New_York` hour ∈ `[DIGEST_HOUR_LOCAL, DIGEST_HOUR_LOCAL + DIGEST_RETRY_WINDOW_HOURS)` (default `7..10`). It does **NOT** gate on a day-level ledger row (that would let one recipient's row skip the whole day and block same-day retry). Instead it runs the digest loop unconditionally within the window and lets the **per-recipient atomic claim** (`dedup_key = 'digest:<ET_date>'`, §3.2) decide per recipient: at the first in-window tick each active recipient is claimed + sent; on later in-window ticks already-`sent` recipients no-op and `failed` recipients (under cap) retry. Outside the window the route no-ops. (Like the realtime job, the digest job runs the §3.2 stale-claim sweep AND the ledger-driven reconciliation §4.4 step 1.5 BEFORE building content — so a realtime row that crashed after Resend accepted but before its `sent` update converges to `sent` first, and the per-recipient exclusion below correctly excludes it. Without reconciliation-first, such a row would be `failed` (not `sent`) and the digest would duplicate the realtime item the recipient already received.) **Digest data source (live-scope, pinned):** the digest content MUST apply the SAME live-scope filters as the dashboard needs-attention inbox — `pending_ingestions.wizard_session_id IS NULL` AND `pending_syncs.wizard_session_id IS NULL` (`components/admin/Dashboard.tsx:328-331,364-369`; `lib/admin/needsAttention.ts`) — so wizard-scoped (mid-onboarding) rows are never emailed. Reuse the dashboard's needs-attention filter logic, but execute it via the cron-safe **service-role** client (§5.1), NOT the cookie-bound dashboard reader (which returns zero under cron, finding-class §5.1). **Per-recipient, occurrence-precise realtime exclusion:** when building recipient R's digest, exclude an item if R has a realtime `email_deliveries` row for the item's CURRENT realtime dedup_key (`'<show_id>:<code>:<epoch(raised_at)>'` show-level; `'ingestion:<drive_file_id>:<epoch(first_seen_at)>'` ingestion) that is EITHER `sent` OR **audit-uncertain** (`first_attempt_at IS NOT NULL` AND it is NOT a clean provider rejection — i.e. a crash-swept row where Resend MAY have accepted). A clean provider-FAILURE row (`error` = a provider error, definitely NOT delivered, `alert_on_sync_problems` may be off so it was never reconciled) does NOT exclude — the admin did not receive it, so the item belongs in the digest. **Why audit-uncertain excludes:** if `alert_on_sync_problems` is turned OFF after a crash-after-send (realtime row stuck `failed`, never reconcilable while OFF) but `daily_review_digest` is ON, a naive `sent`-only exclusion would re-send the already-delivered item in the digest (AC-B3.6d violation). Conservatively excluding possibly-delivered items prevents that duplicate; the item is still visible in the dashboard inbox. Two WRONG simplifications to avoid: (a) a GLOBAL exclusion — if A got the realtime email but B's realtime send FAILED, it would drop the item from B's digest too, so B silently loses a review item it never received; (b) matching on `show_id+code` / `drive_file_id` ALONE — a resolved-then-reopened problem is a NEW occurrence (fresh `raised_at`/`first_seen_at`), so an OLD `sent` row must NOT exclude the re-armed item from a recipient whose realtime failed on the new occurrence. So the exclusion JOINs per-recipient delivery state on the exact current `dedup_key`. **Hourly + ET-hour window is DST-safe** (no UTC offset hardcoded; explicit `America/New_York` zone, never host TZ), honoring the validation-tooling-tz-pin discipline.

### 5.4 Email link targets (read-only notices)
Realtime email → `${NEXT_PUBLIC_SITE_ORIGIN}/admin/show/<slug>`; digest email → `${NEXT_PUBLIC_SITE_ORIGIN}/admin`. No action tokens (deferred). Links use the real origin env, never a hardcoded host.

## 6. Failure & unconfigured-provider handling

- **Send failure:** the claim winner (§3.2) gets `{ ok:false, kind:'infra_error' }` from Resend → `update` its claimed row to `status='failed'` (+`error`, `attempt_count = attempt_count + 1` — incremented HERE on the actual attempt, NOT at claim); emit `upsert_admin_alert(show_id, 'EMAIL_DELIVERY_FAILED', …)` (in-app only — **never** email about email failing). A later tick re-acquires the row via the §3.2 atomic claim while `attempt_count < SEND_RETRY_CAP (3)`; at the cap the claim's `WHERE` predicate excludes it (no further send) and the in-app alert persists. Stale `claimed` rows (crash after claim) are recovered per §3.2 (flip to `failed` after `CLAIM_STALE_MS`; Idempotency-Key makes any resend a provider no-op). Quiet-success is preserved (no "still failing" emails).
- **Unconfigured provider:** missing `RESEND_API_KEY` or `EMAIL_FROM` → **only the claim/send loops skip**; the job's ALERT-MAINTENANCE (§4.4 step 0) STILL runs: stale-claim sweep, global `SYNC_STALLED` detect+resolve (§4.2), show-level recovery-resolution recheck (§4.3), and `EMAIL_NOT_CONFIGURED` upsert **once** (+ its resolve when the provider later becomes configured). So an email-config mistake NEVER blinds the in-app dashboard signal for a stopped sync pipeline. Returns 200 (delivery skipped). Toggles remain settable but inert; the settings UI MAY render a server-derived "email not configured" hint (never leaks the secret).
- **Recovery-resolution of email-infra alerts (symmetric, mandatory):** these in-app alerts must be RESOLVED when the condition clears, else stale warning banners persist after delivery is healthy. `EMAIL_NOT_CONFIGURED` → resolved on any notify tick where the provider IS configured. `EMAIL_DELIVERY_FAILED` → resolved **aggregate-aware**: resolve ONLY when NO `failed` or cap-reached `email_deliveries` rows remain for the alert's scope (re-check on each delivery state-change). A single retry success must NOT clear the alert while another delivery for the same scope is still failed/cap-reached — `admin_alerts` dedups by `(show_id, code)`, so the one aggregate row must reflect "any outstanding failure," not "the last one I touched." **The aggregate counts only ACTIONABLE outstanding failures:** `failed` rows that are non-terminal OR `terminal_reason='idempotency_window_expired'` (the intentionally audit-uncertain case). It EXCLUDES `terminal_reason IN ('recipient_revoked','digest_window_passed','digest_superseded')` — those are deliberate skips/supersedes, not provider failures, and must not keep `EMAIL_DELIVERY_FAILED` lit. **Cap-reached currency (avoids a permanent banner):** a cap-reached `failed` row stops counting toward the aggregate once its OCCURRENCE is no longer current — i.e. its source problem has resolved (recovery-resolution §4.3 resolved the source `admin_alert`); or, for a **pending-ingestion** delivery (which bypasses `admin_alerts`), once no `pending_ingestions` row with the same `drive_file_id` + `first_seen_at` occurrence exists (the sheet recovered / the row was deleted); or, for a **digest** delivery, once its ET-day has passed. So a transient provider outage that burns 3 attempts does NOT leave a permanent `EMAIL_DELIVERY_FAILED` after delivery is healthy: when the underlying problem clears (or the day rolls over), the stale cap-reached row drops out and the alert resolves. **B3-DEF-9 (BACKLOG):** a manual resend/ack for a cap-reached delivery whose occurrence is STILL current (lets an operator retry after a long outage without waiting for recurrence).
- **Fail-closed reads:** both toggle reads fail-closed (a read error is treated as "off") so an infra hiccup never spams.

## 9. Acceptance criteria

- **AC-B3.1** With `alert_on_sync_problems=true`, an unresolved `SYNC_PROBLEM_CODES` alert whose `raised_at ≤ now()-1h` produces exactly one `sent` `email_deliveries` row per active recipient and one Resend send; a second realtime tick produces **no** additional send (dedup).
- **AC-B3.1a (concurrency)** Two overlapping realtime ticks for the same alert+recipient produce **exactly one** Resend send and one `sent` row: the claim winner sends, the loser's conflict-update `WHERE` is false → no send. (Test: two workers race the §3.2 atomic claim against one candidate.)
- **AC-B3.1b (multi-recipient)** One alert + two active admins → **two** distinct provider Idempotency-Keys (`kind:dedup_key:recipientA` ≠ `…:recipientB`), two `sent` rows, two Resend sends. No recipient is suppressed by provider-side dedup. (Negative-regression: set the key to `dedup_key` alone → assert the second send is dropped.)
- **AC-B3.2** With `alert_on_sync_problems=false` (or read error), the same alert produces **zero** realtime emails; the in-app alert is unaffected.
- **AC-B3.3** An alert that resolves and later re-opens (new `raised_at`) re-arms: a new realtime email is sent (negative-regression: stashing the `raised_at` component of the dedup key makes the second email fail to fire).
- **AC-B3.3a (recovery-resolution)** (i) An error alert whose show recovers (exits the error `last_sync_status`) before 1h is resolved → no email. (ii) An error emailed, then recovered (alert resolved on recovery), then RE-occurs (fresh `raised_at`) → a NEW email fires — proving re-arm works *because* recovery resolved the prior row. (iii) Notify-side recheck: an alert still unresolved whose show has recovered is resolved + skipped (no stale email). **Negative-regression:** remove the recovery-resolution call → assert (a) a stale email fires for a cleared error and (b) a recurrence is SUPPRESSED (the never-resolved row coalesces).
- **AC-B3.4** A transient problem that clears before 1h produces **no** email (quiet-success).
- **AC-B3.5 (global stall)** When `app_settings.last_sync_cron_completed_at` is older than `STALENESS_THRESHOLD_MS` (or NULL), the realtime job upserts exactly ONE show-wide `SYNC_STALLED` alert (`show_id NULL`); a fresh heartbeat resolves it. **Negative-regressions:** (a) a show that FAILS every tick (its `last_synced_at` stays fresh + an error alert is present) does NOT trigger `SYNC_STALLED` while the heartbeat is fresh; (b) a QUIET healthy show (no changes, heartbeat fresh) does NOT trigger `SYNC_STALLED`. (Failure mode: keying staleness off `last_synced_at` misses failing shows AND false-positives quiet shows.)
- **AC-B3.5c (stall email timing, ~1h not ~2h)** A heartbeat just past `STALENESS_THRESHOLD_MS` (e.g. `now()-61min`) raises `SYNC_STALLED` and the realtime job emails it on that SAME tick — step 2b applies NO extra age gate. (Test: heartbeat = now−61min → assert the stall email sends this tick, not a tick ~1h later.)
- **AC-B3.5d (fresh-deploy no false stall)** Immediately after the B3 migration (heartbeat initialized to `now()`), the first notify tick — before any post-deploy sync run — raises NO `SYNC_STALLED` and sends no stall email, even with both toggles ON. (Test: apply migration, run a notify tick at deploy-time → zero stall sends.)
- **AC-B3.5a (pending-ingestion)** A `pending_ingestions` row with `now()-first_seen_at > 1h` and `alert_on_sync_problems=true` produces exactly one realtime email per active recipient, dedup'd by `'ingestion:<drive_file_id>:<epoch(first_seen_at)>'` (show_id NULL); a second tick sends nothing; deleting the row (successful re-ingestion) before 1h sends nothing; a fresh ingestion of the same file (new `first_seen_at`) re-arms.
- **AC-B3.5b (ingestion read-to-send race — both orderings)** *Recovery-wins:* when the recovery path commits its successful-re-ingestion DELETE BEFORE notify acquires the lock, notify re-reads under the lock, finds the row gone, sends nothing. *Notify-wins:* when notify acquires the lock first, it sends — and that email is **legitimately real** (the sheet was genuinely >1h-stuck and not-yet-recovered at send time; quiet-success suppresses sub-1h transients, not a real >1h problem that resolves moments later). (Two-connection test asserts BOTH orderings: recovery-first → no email; notify-first → exactly one email, with the SESSION lock held across the paused send so recovery's DELETE blocks until notify releases it. ## 13. Watchpoints / disagreement-loop preempts (do-not-relitigate)

For the adversarial reviewer — each contract below is intentional and cited; do not relitigate without a new owner decision:

- **No per-show advisory lock in the notify job** is correct (§4.4): the job mutates only `admin_alerts` (idempotent SECURITY DEFINER upsert) + `email_deliveries`, neither in the invariant-2 protected set. Inline `// not-subject-to-meta` documents it.
- **Two deliberate detection exceptions to "consume admin_alerts"** (RD-6, §4.2, §4.5): staleness ADDS a `SYNC_STALLED` row to `admin_alerts`; pending-ingestion reads `pending_ingestions` DIRECTLY (because pre-show rows have no `show_id` and the `(show_id,code)` upsert cannot dedup per-file). Both are intentional, not gaps.
- **Atomic claim-before-send + full (non-partial) unique index** (§3.2) is the correct duplicate-send defense, not over-engineering: the ledger is the concurrency lock; the provider Idempotency-Key covers the crash-after-send window. Do not "simplify" back to insert-after-send.
- **`activeRecipients()` MUST use the service-role client, NOT `listAdminEmails`** (§5.1): the cron route has no admin session, so the cookie/RLS-bound reader returns zero rows and silently suppresses all email. Do not "reuse the existing helper."
- **Provider Idempotency-Key MUST include `recipient`** (`kind:dedup_key:recipient`, §3.2): `dedup_key` alone is shared across recipients and would let the provider drop all but the first admin while their ledger rows flip to `sent`.
- **Digest is per-recipient-claim-gated, NOT day-level-gated** (§5.3): the route runs the loop across the ET-hour window and lets the per-recipient claim decide; a day-level "any digest row exists → skip" guard would suppress other recipients + block same-day retry. Do not reintroduce it.
- **Realtime show-level candidates require `published ∧ ¬archived`** (§4.4 step 2): an alert raised before archive/unpublish stays unresolved, so the `shows` join is load-bearing to avoid emailing problems for inactive shows.
- **The pending-ingestion finalization advisory lock is correct, not an invariant-2 violation** (§4.5): it is a deliberate, scoped exception to the notify job's otherwise lock-free posture, acquiring the sync path's hashkey as a SEPARATE top-level transaction (mutual exclusion, not nested) to close the irreversible-email read-to-send race. Do not "remove the lock for consistency with the other paths."
- **HTML escaping of interpolated values is mandatory** (§5.1): show names / Drive file names / context are untrusted; every dynamic value into `html` is escaped. Do not treat sheet/admin data as trusted.
- **The ingestion finalization uses a SESSION-level lock on postgres.js with the claim COMMITTED before the send** (§4.5, R13): the claim must be durable before the irreversible Resend send (else a crash-after-send loses the claim → re-send + lost audit), while the session lock (released in `finally`) keeps mutual exclusion across the send. NOT one xact holding an uncommitted claim across the send; NOT the PostgREST/autocommit client. The test pins both: exclusion-across-send AND crash-after-send durability.
- **Outbox convergence depends on Resend's VERIFIED idempotency replay** (§5.2, confirmed against Resend docs 2026-06-02): same key within 24h → original response; same key + different payload → `409 invalid_idempotent_request` (handled as already-sent). `CLAIM_STALE_MS ≪ 24h`. The required probe (§10) re-verifies at implementation; don't treat the contract as assumed.
- **Recipient-active re-check is on EVERY send path, close to the provider call** (§4.4 step 4 + 1.5): not just replay — a first send must also re-check after the `activeRecipients()` snapshot, or a just-revoked admin gets emailed.
- **Failed-row retry happens in EXACTLY ONE place per tick** (§4.4: step 1.5 retries existing `failed`; step 4 is `ON CONFLICT DO NOTHING` brand-new only): ≤1 provider attempt per row per tick, so a transient failure doesn't burn the whole `SEND_RETRY_CAP` in one run. Do not let step 4 reclaim failed rows.
- **The provider Idempotency-Key is a BOUNDED hash, not the raw email-bearing concat** (§3.2): Resend caps it at 256 chars; a 254-char admin email would otherwise blow the budget. The raw `(kind,dedup_key,recipient)` stay as columns for dedup/audit; the hash is the provider key.
- **The idempotency-window guard ages against the IMMUTABLE `first_attempt_at`, never the mutable `claimed_at`** (§4.4 step 1.5): a chain of pre-expiry retries refreshing `claimed_at` must not let a reissue slip past the real 24h-from-original window.
- **Terminal rows (`terminal_reason` set) are excluded from the claim predicate AND reconciliation** (§3.2/§4.4 step 1.5): a `recipient_revoked` / `idempotency_window_expired` row must never be re-acquired or re-sent, even while the source occurrence is current. Status alone (`failed`) is insufficient — the terminal column is load-bearing.
- **Reconciliation is KIND-SCOPED for digest freshness** (§4.4 step 1.5): the realtime job never reissues `digest` rows; the digest job reconciles realtime rows (for exclusion accuracy, AC-B3.6d) PLUS its own today-in-window digest rows; an off-date/off-window digest row → terminal `digest_window_passed`. A generic all-kinds reconciliation would send stale/off-window digests; conversely a digest-only scope would break AC-B3.6d.
- **`EMAIL_DELIVERY_FAILED` aggregate excludes non-actionable terminal rows** (§6): `recipient_revoked` / `digest_window_passed` are deliberate skips, not provider failures, and must not keep the banner lit; `idempotency_window_expired` stays counted (audit-uncertain).
- **The digest job runs reconciliation BEFORE building content** (§5.3): a realtime row that crashed after Resend accepted (still `failed`, not `sent`) must converge first, or the digest duplicates the realtime item. Reconciliation-first is mandatory, not just a realtime-loop step.
- **Reconciliation re-checks recipient-active AND the idempotency window before any reissue** (§4.4 step 1.5): never email a revoked admin on replay; never auto-reissue a row older than `PROVIDER_IDEMPOTENCY_WINDOW_MS` (post-window reissue would duplicate). Both are reissue-time guards, not just claim-time.
- **Retry/convergence is LEDGER-DRIVEN (outbox), not candidate-driven** (§3.2): a stale `claimed`/`failed` row re-issues from the ledger row (recipient + idem-key + `context`), so a crash-after-send followed by source-row deletion still converges the durable row to `sent`. Do not make retry require the source candidate to still exist.
- **`context` is populated at claim time** (§3.2): the ledger-driven retry re-renders from `context` after the source row is gone; an empty `{}` context breaks post-deletion convergence (AC-B3.5e). Do not leave `context` defaulted.
- **Ledger reconciliation reissue is gated on the ROW'S-KIND tier toggle** (§4.4 step 1.5): OFF means that kind sends NOTHING, including retries/reissues (a reissue may deliver an email if the original never reached Resend). A crash-after-send row defers audit convergence to the next ON tick (idempotent replay, no duplicate). Only step-0 maintenance is toggle-independent. (Corrects the earlier "toggle-independent reconciliation" framing, which leaked retry emails while OFF.)
- **Alert MAINTENANCE always runs; only DELIVERY is gated** (§4.4 step 0, §6): the stale-claim sweep, `SYNC_STALLED` detect/resolve, recovery-resolution recheck, and `EMAIL_NOT_CONFIGURED` upsert/resolve run even when the toggle is off or the provider is unconfigured. Do not early-return the whole job on the provider/toggle gate — that would blind the in-app dashboard signal.
- **Stale-claim recovery is the first step of every notify job** (§3.2/§4.4 step 0/§5.3): without it a crash-after-claim permanently silences a `(kind,dedup_key,recipient)` message. Do not omit or reorder it.
- **The notify-wins ordering sends a legitimately-real ingestion email, not a false one** (AC-B3.5b): a sheet stuck >1h that recovers moments after the alert was a real >1h problem at send time; quiet-success governs sub-1h transients only. The advisory lock's role is to kill the read-stale-after-delete (recovery-wins) race, not to suppress real >1h alerts. Do not re-frame notify-wins as a quiet-success violation.
- **Digest + realtime-ingestion both apply `wizard_session_id IS NULL`** (§4.4 step 3, §4.5 SELECT + lock re-read, §5.3): wizard-scoped rows are mid-onboarding (admin is already looking) and are excluded from email, matching the dashboard live-scope. The filter is in the OPERATIVE predicate AND re-asserted at the lock re-read (mid-flight wizard-scoping is dropped). Execute via the service-role client (not the cookie-bound dashboard reader).
- **`SYNC_STALLED` is a show-wide cron-heartbeat alert, NOT per-show `last_synced_at`** (§4.2): `last_synced_at` bumps on failures (`runScheduledCronSync.ts:661-715`) and skips quiet shows, so it both misses failing shows and false-positives quiet ones. Per-show failures are covered by their error alerts; staleness = the pipeline stopped (heartbeat). Do not re-key staleness off `last_synced_at`.
- **`SEND_TIMEOUT_MS ≪ CLAIM_STALE_MS` is an invariant** (§3.2/§5.1): the stale-claim sweep must never reclaim a live in-flight send, so the bounded send timeout (30s) sits far below the stale window (10 min). Do not widen the send timeout toward the stale window.
- **`SYNC_STALLED` is email-eligible immediately on raise — no second +1h gate** (§4.4 step 2b): the 1h is already in §4.2's heartbeat threshold, so the first stall email fires at ~1h, not ~2h. The +1h age gate applies ONLY to show-level error alerts (step 2a), whose `raised_at` = the error time.
- **`SYNC_STALLED`'s common-mode limitation is an accepted, documented narrowing** (§4.2 / B3-DEF-8): it cannot detect a total outage of its own scheduler (pg_cron/pg_net); that needs an independent external monitor (BACKLOG, ops). Do not relitigate as a B3 gap.
- **Digest excludes AUDIT-UNCERTAIN realtime rows, not just `sent`** (§5.3): a crash-after-send realtime row (Resend may have accepted) while `alert_on_sync_problems` is OFF can't be reconciled, so the digest must conservatively exclude it or it duplicates a delivered item. A clean provider-failure row is NOT excluded (the admin didn't get it).
- **Digest reissues REBUILD content from the live inbox; realtime/ingestion reissues use stored `context`** (§4.4 step 1.5): a stale stored digest must not be re-sent after Doug cleared the items; zero current items → `terminal_reason='digest_superseded'`.
- **Pending-ingestion cap-reached failures stop counting when the source `pending_ingestions` row is gone** (§6): they bypass `admin_alerts`, so the source-resolution currency rule doesn't apply; key the currency on the `drive_file_id`+`first_seen_at` row existence.
- **Digest realtime-exclusion is PER-RECIPIENT and OCCURRENCE-PRECISE** (§5.3, RD-7): exclude an item from recipient R only if R has a `sent` realtime delivery whose `dedup_key` matches the item's CURRENT occurrence. A global exclusion hides items from a recipient whose realtime failed; an occurrence-blind (`show_id+code` only) match lets an old sent row hide a re-armed item. Keep both per-recipient AND the occurrence epoch in the match.
- **The heartbeat is initialized to `now()` at migration time** (§3.1): a fresh B3 deploy must NOT false-blast `SYNC_STALLED` before its first post-deploy sync run. Do not drop the init `update`.
- **`attempt_count` is incremented on a real provider attempt, NOT at claim** (§3.2): claim-time increment would permanently suppress a row whose worker crashed on the final allowed attempt. The stale-claim sweep likewise does not increment. Do not move the increment into the claim/sweep.
- **`EMAIL_DELIVERY_FAILED` resolution is aggregate-aware** (§6): one retry success must not clear the alert while another delivery for the same `(show_id,code)` scope is still failed/cap-reached.
- **Recovery-resolution of `SYNC_PROBLEM_CODES` alerts on sync recovery is mandatory** (§4.3): without it, cleared errors email stale AND recurrences are suppressed (the re-arm model depends on resolution refreshing `raised_at`/`dedup_key`). The notify-side current-state recheck is the defensive net. `EMAIL_DELIVERY_FAILED`/`EMAIL_NOT_CONFIGURED` resolve symmetrically (§6). Do not treat resolution as optional polish.
- **`app_settings` writes are RLS `admin_only`-gated, not RPC-gated** (§3.1) — this matches the ratified B2 `setAutoPublish` pattern (`app/admin/settings/_actions/setAutoPublish.ts:46`); B3 does not introduce an RPC for the toggles.
- **Email failures never trigger emails** (§6) — `EMAIL_DELIVERY_FAILED` / `EMAIL_NOT_CONFIGURED` are in-app only, by design (no feedback loop, no spam).
- **The 2026-05-09 notification memo is principles-only** (§1.3) — its signed-token/feedback/unpublish machinery is superseded by the crew-auth pivot + B2 and is explicitly deferred, not omitted by oversight.
- **Both toggles default ON** (RD-8a) — owner decision; quiet-success keeps them silent when nothing is wrong.
- **Digest hourly-fire + ET-hour check is the DST-safe design** (§5.3), not an inefficiency to "optimize" into a single UTC cron time.
- **Server actions under `app/` are Opus-owned** (§11) per the UI hard rule, even though they touch `app_settings`.
