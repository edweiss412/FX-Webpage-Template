## 1. Scope and boundaries

### 1.1 Problem
The "Alert me about sync problems" + "Daily review digest" toggles are operator **attention surfaces**: Doug's surface is Drive, not the admin dashboard, so a stalled sync or a staged event that lives only in-app is functionally invisible. The in-app `admin_alerts` surface (AlertBanner / NotifBell) is the system of record (pull); B3 adds the email **push** layer over it. Per **AGENTS.md no-zombie-flag**, neither toggle ships without its working backend ‚Äî hence the toggles ship *with* the email subsystem.

### 1.2 In scope
First outbound-email path (`lib/notify/` channel-agnostic seam, email-only impl, Resend, secrets); the two real-time + digest email tiers; a simple `email_deliveries` dedup/audit ledger; the global `SYNC_STALLED` heartbeat detector (+ a small **cross-into-sync-path** heartbeat write, recovery-resolution calls, and **two new `admin_alerts` producers** ‚Äî `DRIVE_FETCH_FAILED`/`PARSE_ERROR_LAST_GOOD`, which have no producer today so two of three RD-5 realtime classes can't notify until added (¬ß4.1) ‚Äî Codex backend); two `app_settings` toggles + read/write + UI; three new ¬ß12.4 codes (`SYNC_STALLED`/`EMAIL_DELIVERY_FAILED`/`EMAIL_NOT_CONFIGURED`) plus two `AdminAlertCode`-union additions for the already-cataloged `DRIVE_FETCH_FAILED`/`PARSE_ERROR_LAST_GOOD`; `app/api/cron/notify/route.ts` + two pg_cron schedules; TDD tests + meta-test registrations.

### 1.3 Out of scope (deferred ‚Äî ¬ß12)
Two-way feedback, inbound webhook, apply-from-email, unpublish tokens, SMS impl, per-admin prefs, digest‚Üírealtime promotion, and (RD-9) the exactly-once crash-recovery outbox machinery. The 2026-05-09 `notification-design-memo.md` is cited for **principles** only (push-not-pull, tiering, quiet-success, coalescing).

### 3.2 `email_deliveries` ledger (new table, B3 migration)
```sql
create table public.email_deliveries (
  id              uuid primary key default gen_random_uuid(),
  kind            text not null check (kind in ('realtime_problem','digest')),
  channel         text not null default 'email' check (channel in ('email','sms','webhook')),
  dedup_key       text not null,   -- realtime show-level '<show_id>:<code>:<epoch(raised_at)>'; global 'global:SYNC_STALLED:<epoch(raised_at)>'; ingestion 'ingestion:<drive_file_id>:<epoch(first_seen_at)>'; digest 'digest:<ET_date>'
  -- CANONICAL OCCURRENCE-EPOCH PRECISION: every `<epoch(<ts>)>` token above is the MICROSECOND-precision integer of the source timestamptz ‚Äî `floor(extract(epoch from <ts>) * 1e6)::bigint` ‚Äî NOT `floor(epoch(...))` (second-granular). Rationale: a resolve-then-reopen within the SAME SECOND mints a fresh occurrence with a new `raised_at`; a second-granular key would COLLIDE with the prior `sent` row and SUPPRESS the re-armed email (violates AC-B3.3). Microsecond precision distinguishes distinct INSERT timestamps. Compute identically on BOTH sides (the producer-written key AND the currentness recompute); to avoid the JS Date-ms-truncation boundary (postgres.js returns timestamptz as a Date), derive the integer in SQL (`extract(epoch ...)*1e6`) or from the ISO string, never via `Date.parse(<Date>)`.
  show_id         uuid references public.shows(id) on delete set null,
  recipient       text not null,
  triggered_codes text[] not null default '{}',
  context         jsonb not null default '{}',   -- OPTIONAL audit only (e.g. render inputs); NOT load-bearing for delivery ‚Äî payloads are re-rendered each attempt and a 409-invalid reissues with an attempt-scoped key (¬ß3.2)
  status          text not null check (status in ('sent','failed')),
  provider_message_id text,
  error           text,
  attempt_count   int  not null default 1,
  created_at      timestamptz not null default now(),
  sent_at         timestamptz,
  constraint email_deliveries_recipient_email_canonical
    check (recipient = lower(trim(recipient)) and recipient <> '')   -- invariant 3 safety net
);
create unique index email_deliveries_dedup on public.email_deliveries (kind, dedup_key, recipient);
```
- **Access lockdown ‚Äî service-role-only, INCLUDING SELECT (privacy):** the ledger holds recipient emails + `provider_message_id` + provider `error` text, and NO product UI reads it in v1, so it is fully service-role-only (stricter than the repo's admin-only-read tables): `alter table public.email_deliveries enable row level security;` (deny-by-default ‚Äî NO anon/authenticated policy), `revoke all on table public.email_deliveries from anon, authenticated;`, `grant all on table public.email_deliveries to service_role;`. The cron job reads/writes it with the service-role client. The `tests/db/postgrest-dml-lockdown.test.ts` entry declares SELECT=**denied** for anon/authenticated (not just DML) and asserts a non-admin SELECT is rejected (no read-leak of recipient emails / audit data).
- **Canonical admin-only-table registry (AC-2.5 lockstep ‚Äî REQUIRED, beyond the PostgREST lockdown):** `email_deliveries` is a new admin-only/service-role-only table holding PII, so it MUST also enter the project-wide admin-table inventory, NOT only the lockdown test. The `ADMIN_TABLES` registry (`lib/audit/admin-tables.generated.ts`) is GENERATED from the master spec ¬ß4.3 prose by `scripts/generate-admin-tables.ts` (it extracts backtick names from the `- **Admin-only tables‚Ä¶**` line AND requires a `create table <name>` block to exist in the master spec ‚Äî `generate-admin-tables.ts:18-42`). So B3 MUST, in the SAME commit as the migration: (a) add `` `email_deliveries` `` to the master spec ¬ß4.3 admin-only-tables prose line (`docs/superpowers/specs/2026-04-30-fxav-crew-pages-v1.md:639`) WITH a brief per-table note + a `create table email_deliveries (‚Ä¶)` block in the master spec (mirroring `app_settings` at master-spec:666 ‚Äî the generator needs the definition or it filters the name out); (b) bump the ¬ß4.3 "**22 tables**" count to **23** (+ its footnote); (c) run `pnpm gen:admin-tables` and commit the regenerated `lib/audit/admin-tables.generated.ts` (count 18‚Üí19). This is AC-2.5 / the master-spec-is-canonical invariant ‚Äî a privacy table must not ship outside the canonical inventory even though its bespoke lockdown test passes.
- **Provider Idempotency-Key (the sole dup guard):** `fxav:<kind>:<sha256(kind || ':' || dedup_key || ':' || recipient)>` ‚Äî a **bounded** ~90-char hash (Resend caps the key at **256 chars**; the raw email-bearing concat is hashed). Verified Resend contract (docs 2026-06-02): a repeat with the same key within **24h** returns the original response (no second email); same key + different payload ‚Üí `409 invalid_idempotent_request`.
- **Dedup + at-least-once delivery (RD-9):** before sending a candidate √ó recipient, skip if a `status='sent'` row already exists for `(kind, dedup_key, recipient)`. Otherwise re-render the payload from the current source and send via Resend with the base idempotency key, then upsert the row to `sent` (+`provider_message_id`,`sent_at`) or `failed` (+`error`, `attempt_count`). **No claim/lock/terminal state, and NO payload-stability requirement** (payloads MAY change between attempts ‚Äî `upsert_admin_alert` overwrites `context` on re-occurrence (`supabase/migrations/20260505000000_upsert_admin_alert.sql:16-19`), show titles/catalog/templates are mutable, and the digest is point-in-time).
  - **Idempotency outcomes (UNIFORM across realtime / ingestion / digest):**
    - `200` ‚Üí delivered ‚Üí `sent` (incl. a same-key/same-payload replay returning the original id ‚Äî the common no-change retry after a transient `infra_error`).
    - `409 concurrent_idempotent_requests` (the base key is still IN-FLIGHT in an overlapping tick) ‚Üí `retry_later`: skip, NO ledger write, NO `attempt_count` bump, NO alert; the in-flight tick finishes (re-checked next tick).
    - `409 invalid_idempotent_request` (the base key was reserved with a DIFFERENT payload ‚Äî the original attempt rendered different bytes, then crashed/failed before recording) ‚Üí **NOT proof of delivery and NOT a terminal failure.** REISSUE with an ATTEMPT-SCOPED provider key built with the SAME bounded, recipient-scoped constructor as the base key ‚Äî `'fxav:'||kind||':'||sha256(kind||':'||dedup_key||':'||recipient||':r:'||<nonce>)` (bounded ~90 chars, still includes `kind`+`recipient` so two admins get DISTINCT keys and the 256-char cap holds) ‚Üí `200` ‚Üí `sent`. **`<nonce>` is a per-SUBMISSION unique value ‚Äî a freshly generated UUID/high-resolution timestamp at the moment of the reissue call, NOT persisted and NOT derived from `attempt_count`.** This is load-bearing: a reissue key derived from the stored `attempt_count` would be RE-DERIVED IDENTICALLY on the next tick after a crash-before-record (which doesn't advance the count), and if the payload changed again that re-derived key would itself `409 invalid` forever ‚Üí a STUCK/MISSED notification (violating "deliver, not miss" ‚Äî worse than RD-9's accepted duplicate). A fresh per-submission nonce guarantees every reissue gets a UNIQUE provider key that has never been poisoned ‚Üí it always delivers. The ONLY residual cost is an RD-9-accepted DUPLICATE (if a prior reissue delivered but crashed before recording, the next tick's fresh-nonce reissue delivers again) ‚Äî never a miss. `attempt_count` is DECOUPLED from the key: it is solely the retry-CAP counter (¬ß5.1). The ledger `dedup_key` stays the base (one `sent` row per logical message per recipient); the attempt-scoped hash is ONLY the provider Idempotency-Key for that reissue.
    - `infra_error` / `400 invalid_idempotency_key` / 5xx / network ‚Üí `failed`/retryable + `EMAIL_DELIVERY_FAILED` (¬ß6).
  - **Why uniform (no per-kind split):** because no payload is guaranteed byte-stable, ANY kind can hit `409 invalid` on a re-render; treating it as a loud failure for realtime/ingestion would MISS a real sync-problem email (RD-9 accepts dups, not misses). So all kinds deliver-via-reissue. The base idempotency key still kills the COMMON duplicate (overlapping ticks + unchanged-payload retries); the attempt-scoped reissue is the rare changed-payload-after-reservation path. `context` (DDL) is optional audit only ‚Äî NOT load-bearing for delivery.
  - **The ledger upsert NEVER downgrades a `sent` row to `failed`** (the failure upsert is conditional `WHERE status <> 'sent'`), so a late/racing failure can't clobber a recorded success.
- **Re-arm:** `dedup_key` includes the occurrence epoch (`raised_at` / `first_seen_at` / ET-date), so a resolved-then-reopened problem (fresh `raised_at`, via ¬ß4.3 recovery-resolution) is a NEW key ‚Üí a new email. `email_deliveries_recipient_email_canonical` keeps the recipient canonical (it is part of the dedup identity).
- **Retry:** a `failed` row under `SEND_RETRY_CAP (3)` is re-sent on a later tick (same idempotency key); at the cap it stays `failed` and `EMAIL_DELIVERY_FAILED` (¬ß6) remains until the underlying problem clears.

### 4.2 Global sync-stall detector (`SYNC_STALLED`, show-wide)
`shows.last_synced_at` is the wrong staleness signal (bumped on failures; not bumped for quiet shows). Per-show failures are already covered by their error alerts (¬ß4.3); the only gap is the whole pipeline stopping. So `SYNC_STALLED` is a single show-wide alert driven by a sync-run heartbeat:
- **Heartbeat (B3 adds ‚Äî cross-into-sync-path):** the scheduled sync run writes `app_settings.last_sync_cron_completed_at = now()` at the END of EVERY completed run ‚Äî success, all-quiet, AND the **`no_folder_configured` / skipped** completion (the heartbeat means "the cron executed", which it did). This prevents a false `SYNC_STALLED` on a pre-onboarding / no-watched-folder deployment (the cron runs hourly, skips, and bumps the heartbeat, so it never goes stale). Trigger-agnostic.
- **Detector (always, in `?job=realtime`):** if `now() - coalesce(last_sync_cron_completed_at,'-infinity') > STALENESS_THRESHOLD_MS` ‚Üí `upsert_admin_alert(NULL,'SYNC_STALLED',‚Ä¶)` (one show-wide row). **Resolve** when the heartbeat is fresh again.
- **Common-mode limitation (documented):** the detector runs on the same scheduler it watches, so a total pg_cron/pg_net outage can't self-detect ‚Üí covered by an external uptime monitor (B3-DEF-8, out of scope).

### 4.3 Gap-fill audit + recovery-resolution (implementation task ‚Äî cross-into-sync-path)
Confirm each `SYNC_PROBLEM_CODES` condition both (a) UPSERTS its code's `admin_alerts` row on entry ‚Äî INCLUDING the two producers B3 adds for `DRIVE_FETCH_FAILED` + `PARSE_ERROR_LAST_GOOD` (¬ß4.1 producer gap), not just the pre-existing `SHEET_UNAVAILABLE` ‚Äî AND (b) RESOLVES that row whenever its **exact code predicate** no longer holds. **Per-code, not just "became OK":** a show can switch `DRIVE_FETCH_FAILED`‚Üí`PARSE_ERROR_LAST_GOOD` while STILL in an error status, so each `SYNC_PROBLEM_CODES` alert must resolve when the show's current state no longer maps to that alert's code. **CRITICAL ‚Äî `shows.last_sync_error` stores RAW causes, NOT catalog codes:** `drive_error` rows store e.g. `SYNC_FILE_FAILED`, and the admin-health layer MAPS the `last_sync_status` bucket to the catalog code (`drive_error`‚Üí`DRIVE_FETCH_FAILED`, `lib/admin/driveConnectionHealth.ts`). So the resolve/recheck predicate uses an explicit **`last_sync_status` ‚Üí catalog-code MAP** (NOT raw `last_sync_error` equality, which would never match `DRIVE_FETCH_FAILED`/`PARSE_ERROR_LAST_GOOD` and would falsely resolve/skip every real alert). Map (documented in full in the plan, derived from the health layer): `drive_error`‚Üí`DRIVE_FETCH_FAILED`; `parse_error`‚Üí`PARSE_ERROR_LAST_GOOD`; `sheet_unavailable`‚Üí`SHEET_UNAVAILABLE` **ALWAYS** (this status maps to a SINGLE alert code ‚Äî the live producer upserts the alert as `SHEET_UNAVAILABLE` for BOTH raw `last_sync_error` causes, `SHEET_UNAVAILABLE` AND `STAGED_PARSE_SOURCE_GONE`; `STAGED_PARSE_SOURCE_GONE` is a raw cause stored in `last_sync_error`, NOT a distinct alert/email code ‚Äî `runScheduledCronSync.ts:1850-1851,1869-1872`). So the map keys ONLY on `last_sync_status` here ‚Äî do NOT branch on `last_sync_error` for `sheet_unavailable`, else a source-gone show (`last_sync_error=STAGED_PARSE_SOURCE_GONE`) would map to a phantom `STAGED_PARSE_SOURCE_GONE` code, falsely resolve the open `SHEET_UNAVAILABLE` alert, and skip its email. `SHEET_PROCESS_FAILED` is pre-show (¬ß4.5). An alert resolves when the show's current `last_sync_status` maps to a DIFFERENT code (or OK); the new code upserts its own alert. Today resolution is only on manual admin-resolve routes; **B3 ADDS the per-code recovery-resolution** to the sync path. Load-bearing: without it, a cleared/switched error still emails its stale code 1h later AND `upsert_admin_alert` coalesces future occurrences into the wrong never-resolved row (re-arm depends on resolution refreshing `raised_at`). **Notify-side recheck (atomic, race-safe ‚Äî NO lock):** the PRIMARY resolver is the sync path (above, under the show lock). The notify-side recheck is a DEFENSIVE conditional resolve that must NOT clobber a current alert on a stale read (a concurrent sync can re-set the error + re-upsert the same row between notify's read and its resolve). So it is a single atomic statement that re-reads the show's CURRENT state inside the UPDATE: `update admin_alerts set resolved_at=now() where id=$alertId and resolved_at is null and not exists (select 1 from shows s where s.id=$showId and <s.last_sync_status maps to this alert's code>)`. If a concurrent sync has re-set the error, the `not exists` is false ‚Üí notify does NOT resolve (correct); if notify resolves first, sync's later `upsert_admin_alert` sees `resolved_at` set ‚Üí INSERTs a fresh row (re-armed). This closes the TOCTOU WITHOUT the notify job taking the per-show advisory lock (preserving ¬ß4.4's no-lock posture). The email is skipped only when this conditional resolve fired (or the alert was already resolved). The plan documents the full `last_sync_status`‚Üícode map (keyed on status alone ‚Äî no status disambiguates on `last_sync_error`; `sheet_unavailable`‚Üí`SHEET_UNAVAILABLE` covers both the `SHEET_UNAVAILABLE` and `STAGED_PARSE_SOURCE_GONE` raw causes).

### 4.4 Delivery loop (realtime)
Candidates from three sources: ¬ß4.2 staleness (‚Üí `admin_alerts.SYNC_STALLED`), ¬ß4.3 consumed error alerts, ¬ß4.5 pending-ingestion.
0. **Maintenance (ALWAYS ‚Äî no provider/toggle needed):** global `SYNC_STALLED` detect+resolve (¬ß4.2); the recovery-resolution recheck (¬ß4.3); AND the **`EMAIL_DELIVERY_FAILED` bidirectional reconciliation** (¬ß6 ‚Äî per scope, OPEN if ‚â•1 in-scope `failed` row is CURRENT, resolve if none; catches the false‚Üítrue currentness re-open that no new send-failure event would trigger after the retry cap). 1. **Provider gate:** config invalid (¬ß5.4 canonical predicate ‚Äî `RESEND_API_KEY` OR `EMAIL_FROM` OR validated site origin missing/invalid) ‚Üí skip all sends; raise `EMAIL_NOT_CONFIGURED` ONLY when at least one tier toggle is ON (`alert_on_sync_problems OR daily_review_digest`), resolve it when config becomes valid OR both toggles are OFF (¬ß6 positive predicate ‚Äî don't warn about unconfigured email when Doug has turned both tiers off). 3. **Candidates:**

5. **Advisory-lock posture:** the loop mutates only `admin_alerts` (idempotent upsert) + `email_deliveries` and READS `shows`/`pending_ingestions`; it does not mutate the invariant-2 protected tables, so it holds **no** per-show lock (`// not-subject-to-meta: notify job mutates only admin_alerts + email_deliveries`).

### 4.5 Stuck-ingestion detector (pre-show ‚Äî second documented exception to RD-6)

- **No advisory lock (RD-9):** a lock-less read may *rarely* email a stuck sheet just as recovery deletes the row; for a >1h-stuck sheet that is a real, mildly-stale heads-up, not spam ‚Äî accepted.

## 6. Failure & unconfigured-provider handling
- **Send failure:** Resend `{ ok:false, kind:'infra_error' }` ‚Üí upsert the `email_deliveries` row `status='failed'` (+`error`) **only when the row is not already `sent`** (never downgrade a success). **`attempt_count` semantics (exact ‚Äî retry-CAP counter ONLY; the reissue key does NOT derive from it ‚Äî ¬ß3.2 uses a per-submission nonce):** on the FIRST failure the row is INSERTed with `attempt_count = 1` (the DDL default, ¬ß3.2 line 91 `default 1`); each SUBSEQUENT failed attempt on a not-yet-`sent` row UPDATEs `attempt_count = email_deliveries.attempt_count + 1` (so the running count is 1 ‚Üí 2 ‚Üí 3, matching AC-B3.8/AC-B3.8b ‚Äî never starts at 2). The row is retried next tick ONLY while the STORED `attempt_count < SEND_RETRY_CAP (3)`. Emit `upsert_admin_alert(show_id,'EMAIL_DELIVERY_FAILED',‚Ä¶)` (in-app only ‚Äî **never** email about email failing). (Per ¬ß3.2/¬ß5.1: ONLY a Resend `200` records `sent`. `409 invalid_idempotent_request` (`idempotency_conflict`) is NEVER recorded `sent` and is NEVER a terminal failure ‚Äî for ALL kinds it REISSUES with an attempt-scoped key to deliver (accept a possible duplicate, ¬ß3.2). `409 concurrent_idempotent_requests` ‚Üí `retry_later`: skip, NO ledger write, NO `attempt_count` bump, NO alert.) An `infra_error`/5xx/network failure is retried next tick while `attempt_count < SEND_RETRY_CAP (3)`; at the cap it stays `failed` (alert persists per ¬ß6 lifecycle).
- **`EMAIL_DELIVERY_FAILED` lifecycle (BIDIRECTIONAL RECONCILIATION each maintenance pass, NOT emit-on-failure-only):** because retries cap at `attempt_count=3`, no NEW provider-failure event fires after the cap ‚Äî so the alert state must be RE-DERIVED on every notify maintenance pass (¬ß4.4 step 0), not only written at send-failure time. Each pass, PER SCOPE: if ‚â•1 in-scope `failed` row is CURRENT ‚Üí ensure `EMAIL_DELIVERY_FAILED` is OPEN for that scope (`upsert_admin_alert`); if NONE is current ‚Üí resolve it. This is bidirectional: currentness can flip false (alert resolves) then TRUE AGAIN with no new send failure ‚Äî a capped `failed` row whose tier toggle was turned OFF (resolved) then back ON while the occurrence is still live, or an archived‚Üípublished flip, or a same-day digest backlog that cleared then reappeared ‚Äî and the reconciliation MUST re-open the alert in that pass. An emit-on-failure-plus-resolve-only implementation would leave a now-current failure invisible. (The reconciliation re-derive itself runs every pass like the other maintenance, ¬ß4.4 step 0; the currentness predicate it evaluates is the toggle-aware one below.) `upsert_admin_alert(show_id,'EMAIL_DELIVERY_FAILED',‚Ä¶)` is keyed `(coalesce(show_id::text,''), code)` ‚Äî so a SHOW-LEVEL failure raises a SHOW-SCOPED alert (`show_id = A`), while digest/global-stall/pending-ingestion failures (no `show_id`) raise the SINGLE NULL-show alert. **Each alert row's OPEN/RESOLVE is scoped to its OWN `show_id`:** the show-A alert is OPEN iff ‚â•1 CURRENT `failed` row maps to show A, RESOLVES when none does ‚Äî it must NOT be kept open by a failure for show B, a digest, or a pending row (those drive show-B's alert or the NULL-show alert). The NULL-show alert is OPEN iff ‚â•1 CURRENT digest/global-stall/pending `failed` row remains. (A failed row maps to its alert by the `show_id` the producer used: show-level rows ‚Üí their `shows.id`; digest/global/pending ‚Üí NULL.) Per scope, the alert is OPEN whenever ‚â•1 in-scope `failed` row is **CURRENT**, and RESOLVES when none is ‚Äî independent of `attempt_count` (emitted on the FIRST failure, kept open through retries 1/2/3, never flickers off mid-retry; `SEND_RETRY_CAP` only stops further *retries*, not the alert). A `failed` row is **CURRENT** iff ALL THREE hold (state the LIVE conditions positively, do NOT invert): (x) its `recipient` is ACTIVE ‚Äî `admin_emails.revoked_at IS NULL`; (w) **its tier toggle is ON** ‚Äî `alert_on_sync_problems` for realtime kinds (show-level/global-stall/pending-ingestion), `daily_review_digest` for the digest kind (CURRENT must match the SAME eligibility gate as the send loop, ¬ß4.4: a row whose tier is now OFF would never be re-sent, so its failure is moot and the alert must resolve); AND (y) its occurrence is a LIVE candidate ‚Äî one branch per `kind`/source, each **OCCURRENCE-EXACT** (the live source's occurrence epoch MUST equal the epoch embedded in the failed row's `dedup_key`, so an OLD failed row never re-becomes current when a fresh occurrence opens): **digest** ‚Üí `dedup_key` IS today's ET date *and* today's digest STILL has ‚â•1 open needs-attention item for this recipient (if the backlog cleared, the digest wouldn't send ‚Üí resolve); **show-level** (`admin_alerts.show_id` NOT NULL) ‚Üí there is an UNRESOLVED `admin_alert` for this `show_id`+`code` whose MICROSECOND-precision `raised_at` epoch (`floor(extract(epoch from raised_at)*1e6)::bigint` ‚Äî ¬ß3.2 canonical precision, NOT second-granular `floor(epoch(...))`) EQUALS the row's `dedup_key` epoch, *and* the show is `published ‚àß ¬¨archived`; **global stall** (`show_id IS NULL`) ‚Üí the UNRESOLVED `SYNC_STALLED` alert's `raised_at` epoch EQUALS the row's `dedup_key` epoch; **pending-ingestion** ‚Üí a `pending_ingestions` row with this `drive_file_id` whose `first_seen_at` epoch EQUALS the row's `dedup_key` epoch EXISTS *and* `wizard_session_id IS NULL`. (These mirror ¬ß4.4 step 3 (a/b/c) / ¬ß4.5 / ¬ß5.3 ‚Äî every realtime candidate kind has a matching, occurrence-exact branch.) Equivalently, a failure STOPS being current ‚Äî and the alert resolves once none of its scope is current ‚Äî when the recipient is revoked, **its tier toggle is turned OFF**, the source alert resolves, the show is archived/unpublished, the ET-day rolls over, **the digest backlog clears**, the pending row is deleted or becomes wizard-scoped, OR a later delivery for that row succeeds (flips it to `sent`). No terminal-state column (RD-9 trim).
- **Unconfigured provider:** missing `RESEND_API_KEY`/`EMAIL_FROM` **OR a missing/blank/malformed canonical site origin (¬ß5.4 ‚Äî required for absolute links)** ‚Üí **only the send loops skip**; maintenance (¬ß4.4 step 0) still runs (`SYNC_STALLED` detect/resolve, recovery-resolution) so the in-app dashboard signal is never blinded. Returns 200 (delivery skipped). Toggles still settable; settings UI MAY show a server-derived "email not configured" hint (never leaks the secret). **`EMAIL_NOT_CONFIGURED` is a POSITIVE predicate: OPEN iff `(config invalid) AND (alert_on_sync_problems OR daily_review_digest)`; RESOLVE when config becomes valid OR both toggles are OFF.** Rationale: if Doug intentionally disables BOTH email tiers, no send is supposed to occur, so a config-not-set alert is a FALSE health signal ‚Äî the toggles must be able to silence their own configuration warning. (config valid = the ¬ß5.4 canonical predicate: `RESEND_API_KEY` present AND `EMAIL_FROM` present AND a validated absolute site origin present ‚Äî the THREE-way conjunction; any one missing/invalid ‚áí config invalid.)
- **Fail-closed reads:** both toggle reads fail-closed (read error ‚Üí "off") so an infra hiccup never spams.

## 13. Watchpoints / do-not-relitigate
- **At-least-once + the Resend Idempotency-Key is the chosen delivery contract (RD-9, owner-approved).** A rare duplicate after a process crash beyond the 24h replay window is **accepted**, not a defect. Do NOT re-introduce the claim-before-send / session-lock / terminal-state / submission-state / persistent-outbox window-expiry re-keying machinery ‚Äî that was explicitly trimmed (B3-DEF-9). (This is distinct from ‚Äî and does NOT trim ‚Äî the in-scope single-attempt provider reissue key `fxav:<kind>:<sha256(‚Ä¶:r:<nonce>)>` on the 409-invalid path, which is REQUIRED B3 work per ¬ß3.2/¬ß5.1/¬ß6 and B3-DEF-9's NOTE.)
- **Accepted residual edges (all benign for a single-operator notifier):** a crash mid-send may rarely duplicate one email (idem-key covers the 24h common case); a >1h-stuck sheet recovering the same minute may get one heads-up; a digest retried after a crash may show slightly-stale content.
