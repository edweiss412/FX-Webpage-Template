## 1. Scope and boundaries

### 1.1 Problem
The "Alert me about sync problems" + "Daily review digest" toggles are operator **attention surfaces**: Doug's surface is Drive, not the admin dashboard, so a stalled sync or a staged event that lives only in-app is functionally invisible. The in-app `admin_alerts` surface (AlertBanner / NotifBell) is the system of record (pull); B3 adds the email **push** layer over it. Per **AGENTS.md no-zombie-flag**, neither toggle ships without its working backend ‚Äî hence the toggles ship *with* the email subsystem.

### 1.2 In scope
First outbound-email path (`lib/notify/` channel-agnostic seam, email-only impl, Resend, secrets); the two real-time + digest email tiers; a simple `email_deliveries` dedup/audit ledger; the global `SYNC_STALLED` heartbeat detector (+ a small **cross-into-sync-path** heartbeat write and recovery-resolution calls, Codex backend); two `app_settings` toggles + read/write + UI; three ¬ß12.4 codes; `app/api/cron/notify/route.ts` + two pg_cron schedules; TDD tests + meta-test registrations.

### 1.3 Out of scope (deferred ‚Äî ¬ß12)
Two-way feedback, inbound webhook, apply-from-email, unpublish tokens, SMS impl, per-admin prefs, digest‚Üírealtime promotion, and (RD-9) the exactly-once crash-recovery outbox machinery. The 2026-05-09 `notification-design-memo.md` is cited for **principles** only (push-not-pull, tiering, quiet-success, coalescing).

### 3.2 `email_deliveries` ledger (new table, B3 migration)
```sql
create table public.email_deliveries (
  id              uuid primary key default gen_random_uuid(),
  kind            text not null check (kind in ('realtime_problem','digest')),
  channel         text not null default 'email' check (channel in ('email','sms','webhook')),
  dedup_key       text not null,   -- realtime show-level '<show_id>:<code>:<epoch(raised_at)>'; global 'global:SYNC_STALLED:<epoch(raised_at)>'; ingestion 'ingestion:<drive_file_id>:<epoch(first_seen_at)>'; digest 'digest:<ET_date>'
  show_id         uuid references public.shows(id) on delete set null,
  recipient       text not null,
  triggered_codes text[] not null default '{}',
  context         jsonb not null default '{}',   -- digest rows store the rendered {subject,html,text} at first attempt ‚Üí reused byte-for-byte on same-day retries (payload stability ¬ß3.2); realtime/ingestion may leave it '{}' (deterministic re-render)
  status          text not null check (status in ('sent','failed')),
  provider_message_id text,
  error           text,
  attempt_count   int  not null default 1,
  created_at      timestamptz not null default now(),
  sent_at         timestamptz,
  constraint email_deliveries_recipient_email_canonical
    check (recipient = lower(trim(recipient)) and recipient <> '')   -- invariant 3 safety net
);
create unique index email_deliveries_dedup on public.email_deliveries (kind, dedup_key, recipient);
```
- **Access lockdown ‚Äî service-role-only, INCLUDING SELECT (privacy):** the ledger holds recipient emails + `provider_message_id` + provider `error` text, and NO product UI reads it in v1, so it is fully service-role-only (stricter than the repo's admin-only-read tables): `alter table public.email_deliveries enable row level security;` (deny-by-default ‚Äî NO anon/authenticated policy), `revoke all on table public.email_deliveries from anon, authenticated;`, `grant all on table public.email_deliveries to service_role;`. The cron job reads/writes it with the service-role client. The `tests/db/postgrest-dml-lockdown.test.ts` entry declares SELECT=**denied** for anon/authenticated (not just DML) and asserts a non-admin SELECT is rejected (no read-leak of recipient emails / audit data).
- **Provider Idempotency-Key (the sole dup guard):** `fxav:<kind>:<sha256(kind || ':' || dedup_key || ':' || recipient)>` ‚Äî a **bounded** ~90-char hash (Resend caps the key at **256 chars**; the raw email-bearing concat is hashed). Verified Resend contract (docs 2026-06-02): a repeat with the same key within **24h** returns the original response (no second email); same key + different payload ‚Üí `409 invalid_idempotent_request`.
- **Dedup + at-least-once delivery (RD-9):** before sending a candidate √ó recipient, skip if a `status='sent'` row already exists for `(kind, dedup_key, recipient)`. Otherwise send via Resend with the idempotency key, then upsert the row to `sent` (+`provider_message_id`,`sent_at`) or `failed` (+`error`, `attempt_count`). **No claim/lock/terminal state.** Two concurrent ticks may both see no `sent` row and both call Resend ‚Äî the **idempotency key makes the second a provider-side no-op**, and both upserts converge the row to `sent` (one email). A crash mid-send is handled the same way: the next tick re-selects (no `sent` row) and re-sends with the same key ‚Üí Resend returns the original within 24h (no duplicate); beyond 24h a duplicate is *possible* and accepted (RD-9).
  - **Only a Resend `200` means delivered ‚Üí `sent`.** A `409 invalid_idempotent_request` (same key, DIFFERENT payload) is NOT proof of delivery ‚Äî a prior FAILED/5xx request can reserve the key without delivering, so a changed-payload retry could 409. Treating it as `sent` would silently drop a notification. So `409 invalid_idempotent_request` ‚Üí **FAILURE** (`infra_error` ‚Üí `failed`/retryable + `EMAIL_DELIVERY_FAILED`), never `sent`.
  - **Payload stability per idempotency key (so legitimate retries `200`-replay instead of 409ing):** the rendered payload for a given `(kind, dedup_key, recipient)` MUST be byte-stable across retries. Realtime/ingestion render deterministically from the FIXED occurrence (the `raised_at`/`first_seen_at` in the key); **digest renders ONCE at the first attempt and stores the rendered `{subject, html, text}` in `context`** ‚Äî retries reuse the stored body (NOT a fresh rebuild), so a same-day digest retry sends the identical payload ‚Üí `200` replay ‚Üí `sent`. With stable payloads, a `200`-replay confirms a real delivery (crash-after-send-same-payload converges correctly); a `409 invalid_idempotent_request` then signals a genuine payload-stability bug and fails LOUDLY rather than silently marking `sent`. (This is a minimal payload snapshot in an existing `context` column ‚Äî NOT the trimmed crash-recovery machinery, RD-9.)
  - **`409 concurrent_idempotent_requests`** (same key still IN-FLIGHT ‚Äî the OTHER overlapping tick is sending it now) ‚Üí **`retry_later`**: the loser tick does NOT write a `failed` row, does NOT increment `attempt_count`, does NOT emit `EMAIL_DELIVERY_FAILED` ‚Äî it leaves the row untouched and lets the in-flight tick finish (re-checked next tick). Correct for AC-B3.1a's overlapping-ticks case: exactly one email, no false failure.
  - **The ledger upsert NEVER downgrades a `sent` row to `failed`** (the failure upsert is conditional `WHERE status <> 'sent'`), so a late/racing failure can't clobber a recorded success.
- **Re-arm:** `dedup_key` includes the occurrence epoch (`raised_at` / `first_seen_at` / ET-date), so a resolved-then-reopened problem (fresh `raised_at`, via ¬ß4.3 recovery-resolution) is a NEW key ‚Üí a new email. `email_deliveries_recipient_email_canonical` keeps the recipient canonical (it is part of the dedup identity).
- **Retry:** a `failed` row under `SEND_RETRY_CAP (3)` is re-sent on a later tick (same idempotency key); at the cap it stays `failed` and `EMAIL_DELIVERY_FAILED` (¬ß6) remains until the underlying problem clears.

## 4. Detection (real-time tier)

### 4.1 `SYNC_PROBLEM_CODES` (single source of truth ‚Äî `lib/notify/constants.ts`)
`['DRIVE_FETCH_FAILED','SHEET_UNAVAILABLE','PARSE_ERROR_LAST_GOOD']` (the show-level consumed error codes ‚Äî each has a REAL show-level `admin_alerts` producer). `STAGED_PARSE_*` + info-severity codes are excluded. **`SHEET_PROCESS_FAILED` is NOT in this set** ‚Äî it is the pre-show pending-ingestion safe-copy FALLBACK (¬ß4.5), not a show-level alert producer; including it would make the show-level `admin_alerts` JOIN `shows` query search for a code no show-level path emits. A constants meta-test asserts every code in `SYNC_PROBLEM_CODES` has a real show/admin_alert producer. Referenced by the query and the tests, never re-listed inline. `SYNC_STALLED` (show-wide) + pending-ingestion are handled by their own detectors, not this set.

### 4.2 Global sync-stall detector (`SYNC_STALLED`, show-wide)
`shows.last_synced_at` is the wrong staleness signal (bumped on failures; not bumped for quiet shows). Per-show failures are already covered by their error alerts (¬ß4.3); the only gap is the whole pipeline stopping. So `SYNC_STALLED` is a single show-wide alert driven by a sync-run heartbeat:
- **Heartbeat (B3 adds ‚Äî cross-into-sync-path):** the scheduled sync run writes `app_settings.last_sync_cron_completed_at = now()` at the END of EVERY completed run ‚Äî success, all-quiet, AND the **`no_folder_configured` / skipped** completion (the heartbeat means "the cron executed", which it did). This prevents a false `SYNC_STALLED` on a pre-onboarding / no-watched-folder deployment (the cron runs hourly, skips, and bumps the heartbeat, so it never goes stale). Trigger-agnostic.
- **Detector (always, in `?job=realtime`):** if `now() - coalesce(last_sync_cron_completed_at,'-infinity') > STALENESS_THRESHOLD_MS` ‚Üí `upsert_admin_alert(NULL,'SYNC_STALLED',‚Ä¶)` (one show-wide row). **Resolve** when the heartbeat is fresh again.
- **Common-mode limitation (documented):** the detector runs on the same scheduler it watches, so a total pg_cron/pg_net outage can't self-detect ‚Üí covered by an external uptime monitor (B3-DEF-8, out of scope).

### 4.3 Gap-fill audit + recovery-resolution (implementation task ‚Äî cross-into-sync-path)
Confirm each `SYNC_PROBLEM_CODES` condition both (a) UPSERTS its code's `admin_alerts` row on entry AND (b) RESOLVES that row whenever its **exact code predicate** no longer holds. **Per-code, not just "became OK":** a show can switch `DRIVE_FETCH_FAILED`‚Üí`PARSE_ERROR_LAST_GOOD` while STILL in an error status, so each `SYNC_PROBLEM_CODES` alert must resolve when the show's current state no longer maps to that alert's code. **CRITICAL ‚Äî `shows.last_sync_error` stores RAW causes, NOT catalog codes:** `drive_error` rows store e.g. `SYNC_FILE_FAILED`, and the admin-health layer MAPS the `last_sync_status` bucket to the catalog code (`drive_error`‚Üí`DRIVE_FETCH_FAILED`, `lib/admin/driveConnectionHealth.ts`). So the resolve/recheck predicate uses an explicit **`last_sync_status` ‚Üí catalog-code MAP** (NOT raw `last_sync_error` equality, which would never match `DRIVE_FETCH_FAILED`/`PARSE_ERROR_LAST_GOOD` and would falsely resolve/skip every real alert). Map (documented in full in the plan, derived from the health layer): `drive_error`‚Üí`DRIVE_FETCH_FAILED`; `parse_error`‚Üí`PARSE_ERROR_LAST_GOOD`; `sheet_unavailable`‚Üí`SHEET_UNAVAILABLE` (or `STAGED_PARSE_SOURCE_GONE` ‚Äî disambiguated by `last_sync_error` where one status maps to multiple codes); `SHEET_PROCESS_FAILED` is pre-show (¬ß4.5). An alert resolves when the show's current `last_sync_status` maps to a DIFFERENT code (or OK); the new code upserts its own alert. Today resolution is only on manual admin-resolve routes; **B3 ADDS the per-code recovery-resolution** to the sync path. Load-bearing: without it, a cleared/switched error still emails its stale code 1h later AND `upsert_admin_alert` coalesces future occurrences into the wrong never-resolved row (re-arm depends on resolution refreshing `raised_at`). **Notify-side recheck (atomic, race-safe ‚Äî NO lock):** the PRIMARY resolver is the sync path (above, under the show lock). The notify-side recheck is a DEFENSIVE conditional resolve that must NOT clobber a current alert on a stale read (a concurrent sync can re-set the error + re-upsert the same row between notify's read and its resolve). So it is a single atomic statement that re-reads the show's CURRENT state inside the UPDATE: `update admin_alerts set resolved_at=now() where id=$alertId and resolved_at is null and not exists (select 1 from shows s where s.id=$showId and <s.last_sync_status (+ s.last_sync_error) maps to this alert's code>)`. If a concurrent sync has re-set the error, the `not exists` is false ‚Üí notify does NOT resolve (correct); if notify resolves first, sync's later `upsert_admin_alert` sees `resolved_at` set ‚Üí INSERTs a fresh row (re-armed). This closes the TOCTOU WITHOUT the notify job taking the per-show advisory lock (preserving ¬ß4.4's no-lock posture). The email is skipped only when this conditional resolve fired (or the alert was already resolved). The plan documents the full `(status, last_sync_error)`‚Üícode map.

### 4.4 Delivery loop (realtime)
Candidates from three sources: ¬ß4.2 staleness (‚Üí `admin_alerts.SYNC_STALLED`), ¬ß4.3 consumed error alerts, ¬ß4.5 pending-ingestion.
0. **Maintenance (ALWAYS ‚Äî no provider/toggle needed):** global `SYNC_STALLED` detect+resolve (¬ß4.2) and the recovery-resolution recheck (¬ß4.3) ‚Äî they upsert/resolve IN-APP `admin_alerts` so the dashboard signal stands even when email is OFF/unconfigured.
1. **Provider gate:** unconfigured (¬ß6) ‚Üí maintain `EMAIL_NOT_CONFIGURED`, skip all sends. Else continue.
2. **Tier-toggle gate:** read `alert_on_sync_problems` fail-closed (`lib/appSettings/getAlertOnSyncProblems.ts`, mirroring `getAutoPublishCleanFirstSeen.ts`); false/read-error ‚Üí skip realtime sends (maintenance already ran).
3. **Candidates:**
   - **(a) Show-level:** unresolved `admin_alerts` JOIN `shows` where `code = any(SYNC_PROBLEM_CODES)` AND `raised_at <= now()-interval '1 hour'` AND `shows.published = true` AND `shows.archived = false` (never email problems for inactive shows). Key `'<show_id>:<code>:<epoch(raised_at)>'`.
   - **(b) Global stall:** the unresolved show-wide `SYNC_STALLED` (`show_id IS NULL`); NO `shows` join; NO extra age gate (the 1h is already in ¬ß4.2's threshold). Key `'global:SYNC_STALLED:<epoch(raised_at)>'`, link ‚Üí `/admin`.
   - **(c) Pending-ingestion (¬ß4.5):** `pending_ingestions` where `now()-first_seen_at > interval '1 hour'` AND `wizard_session_id IS NULL`. Key `'ingestion:<drive_file_id>:<epoch(first_seen_at)>'`, `show_id NULL`, `triggered_codes=[last_error_code]`, link ‚Üí `/admin`.
4. **Send:** for each candidate √ó each active recipient, **re-check the recipient is still active** (`admin_emails.revoked_at IS NULL`, close to the send; revoked ‚Üí skip, never email a revoked admin); skip if a `status='sent'` ledger row exists; else `lib/notify.sendEmail(...)` ‚Üí Resend (Idempotency-Key ¬ß3.2); upsert `sent`/`failed` (+`EMAIL_DELIVERY_FAILED` on failure, ¬ß6).
5. **Advisory-lock posture:** the loop mutates only `admin_alerts` (idempotent upsert) + `email_deliveries` and READS `shows`/`pending_ingestions`; it does not mutate the invariant-2 protected tables, so it holds **no** per-show lock (`// not-subject-to-meta: notify job mutates only admin_alerts + email_deliveries`).

### 4.5 Stuck-ingestion detector (pre-show ‚Äî second documented exception to RD-6)
Pending ingestions are pre-show (no `show_id`), so `admin_alerts`' `(show_id,code)` upsert can't dedup per-file; this class is read directly and dedup'd by `drive_file_id`.
- **Select:** `pending_ingestions` older than `STALENESS_THRESHOLD_MS` AND `wizard_session_id IS NULL` (exclude active onboarding, matching the dashboard live-scope `components/admin/Dashboard.tsx:328-331`). A row's presence = the sheet hasn't become a show; a successful re-ingestion deletes the row ‚Üí no email (quiet-success).
- **Dedup/re-arm:** `'ingestion:<drive_file_id>:<epoch(first_seen_at)>'`. Re-arms on a fresh `first_seen_at`.
- **Email (safe copy ‚Äî invariant 5):** `pending_ingestions.last_error_code` is UNCONSTRAINED text, so the body MUST resolve it through the SAME shared safe-copy resolver the dashboard NeedsAttentionInbox uses (the Phase A `resolveIngestionCopy`-equivalent), passing `sheet_name = drive_file_name`, and FALL BACK to the generic `SHEET_PROCESS_FAILED` `dougFacing` for an unknown / null-dougFacing / crew-only / unresolved-`<‚Ä¶>`-placeholder code. A raw `messageFor(last_error_code)` lookup would crash rendering or leak a raw code / `<sheet-name>` placeholder into the email (and could halt the cron tick before later sends). Link ‚Üí `/admin`. Gated by `alert_on_sync_problems`.
- **No advisory lock (RD-9):** a lock-less read may *rarely* email a stuck sheet just as recovery deletes the row; for a >1h-stuck sheet that is a real, mildly-stale heads-up, not spam ‚Äî accepted.

## 6. Failure & unconfigured-provider handling
- **Send failure:** Resend `{ ok:false, kind:'infra_error' }` ‚Üí upsert the `email_deliveries` row `status='failed'` (+`error`, `attempt_count+1`) **only when the row is not already `sent`** (never downgrade a success); emit `upsert_admin_alert(show_id,'EMAIL_DELIVERY_FAILED',‚Ä¶)` (in-app only ‚Äî **never** email about email failing). (Per ¬ß3.2/¬ß5.1: ONLY a Resend `200` records `sent`. `409 invalid_idempotent_request` IS a failure ‚Üí `infra_error` ‚Üí this same `failed`/retryable + `EMAIL_DELIVERY_FAILED` path (NEVER `sent`). `409 concurrent_idempotent_requests` ‚Üí `retry_later`: skip, NO ledger write, NO `attempt_count` bump, NO alert.) Retried next tick while `attempt_count < SEND_RETRY_CAP (3)`; at the cap it stays `failed` (alert persists per ¬ß6 lifecycle).
- **`EMAIL_DELIVERY_FAILED` lifecycle (single POSITIVE predicate):** the alert is OPEN whenever ‚â•1 `failed` row is **CURRENT**, and RESOLVES when none is ‚Äî independent of `attempt_count` (emitted on the FIRST failure, kept open through retries 1/2/3, never flickers off mid-retry; `SEND_RETRY_CAP` only stops further *retries*, not the alert). A `failed` row is **CURRENT** iff BOTH hold (state the LIVE conditions positively, do NOT invert): (x) its `recipient` is ACTIVE ‚Äî `admin_emails.revoked_at IS NULL`; AND (y) its occurrence is a LIVE candidate ‚Äî one branch per `kind`/source, each **OCCURRENCE-EXACT** (the live source's occurrence epoch MUST equal the epoch embedded in the failed row's `dedup_key`, so an OLD failed row never re-becomes current when a fresh occurrence opens): **digest** ‚Üí `dedup_key` IS today's ET date; **show-level** (`admin_alerts.show_id` NOT NULL) ‚Üí there is an UNRESOLVED `admin_alert` for this `show_id`+`code` whose `floor(epoch(raised_at))` EQUALS the row's `dedup_key` epoch, *and* the show is `published ‚àß ¬¨archived`; **global stall** (`show_id IS NULL`) ‚Üí the UNRESOLVED `SYNC_STALLED` alert's `raised_at` epoch EQUALS the row's `dedup_key` epoch; **pending-ingestion** ‚Üí a `pending_ingestions` row with this `drive_file_id` whose `first_seen_at` epoch EQUALS the row's `dedup_key` epoch EXISTS *and* `wizard_session_id IS NULL`. (These mirror ¬ß4.4 step 3 (a/b/c) / ¬ß4.5 / ¬ß5.3 ‚Äî every realtime candidate kind has a matching, occurrence-exact branch.) Equivalently, a failure STOPS being current ‚Äî and the alert resolves once none of its scope is current ‚Äî when the recipient is revoked, the source alert resolves, the show is archived/unpublished, the ET-day rolls over, the pending row is deleted or becomes wizard-scoped, OR a later delivery for that row succeeds (flips it to `sent`). No terminal-state column (RD-9 trim).
- **Unconfigured provider:** missing `RESEND_API_KEY`/`EMAIL_FROM` ‚Üí **only the send loops skip**; maintenance (¬ß4.4 step 0) still runs (`SYNC_STALLED` detect/resolve, recovery-resolution) so the in-app dashboard signal is never blinded; emit `EMAIL_NOT_CONFIGURED` once (resolve when configured). Returns 200 (delivery skipped). Toggles still settable; settings UI MAY show a server-derived "email not configured" hint (never leaks the secret).
- **Fail-closed reads:** both toggle reads fail-closed (read error ‚Üí "off") so an infra hiccup never spams.

## 9. Acceptance criteria
- **AC-B3.1** `alert_on_sync_problems=true`, an unresolved `SYNC_PROBLEM_CODES` alert with `raised_at ‚â§ now()-1h` ‚Üí exactly one Resend send + one `sent` row per active recipient; a second tick ‚Üí no resend (the `sent` row skips it).
- **AC-B3.1a (concurrency)** Two overlapping realtime ticks for the same alert+recipient (identical, stable payload) ‚Üí exactly ONE email delivered; the second hits Resend's `409 concurrent_idempotent_requests` (still in-flight) ‚Üí `retry_later` (NO `failed` row, NO `EMAIL_DELIVERY_FAILED`, NO `attempt_count` bump); or, if the first already completed, a `200` SAME-payload replay (original id) ‚Üí `sent`. Either way ONE `sent` row, no false failure. (Test both orderings: in-flight ‚Üí `retry_later`; completed ‚Üí `200` replay ‚Üí one `sent` row.)
- **AC-B3.1b (multi-recipient)** One alert + two active admins ‚Üí two distinct idempotency keys (include `recipient`), two `sent` rows, two emails; no recipient is provider-deduped away.
- **AC-B3.2** `alert_on_sync_problems=false`/read-error ‚Üí zero realtime emails; in-app alerts unaffected.
- **AC-B3.3** An alert that resolves then re-opens (fresh `raised_at`) re-arms ‚Üí a new email. (Negative-regression: stash the `raised_at` epoch in the key ‚Üí second email fails to fire.)
- **AC-B3.3a (recovery-resolution, per-code)** An error alert whose show recovers before 1h is resolved ‚Üí no email; the notify-side recheck resolves+skips a recovered-but-unresolved alert. **Code-switch:** a show going `DRIVE_FETCH_FAILED`‚Üí`PARSE_ERROR_LAST_GOOD` WITHOUT an OK interlude resolves the old code's alert (current `last_sync_error` ‚âÝ that code) and upserts the new code's alert ‚Üí the old code is never emailed stale and future occurrences coalesce into the right row. (Negative-regressions: drop recovery-resolution ‚Üí stale email + suppressed recurrence; resolve only on "became OK" ‚Üí the code-switch case emails the stale code; **compare the alert code to raw `last_sync_error` instead of the status‚Üícode map ‚Üí a real `drive_error`/`SYNC_FILE_FAILED` row is falsely resolved/skipped and never emailed** ‚Äî test with the `SYNC_FILE_FAILED` drive_error fixture and assert the `DRIVE_FETCH_FAILED` alert is NOT resolved/skipped.) **Race (notify-side resolve vs sync re-upsert):** a notify recheck that read "recovered" must NOT resolve the alert if a concurrent sync has re-set the error ‚Äî the atomic conditional resolve (re-reading `shows` in the `UPDATE`) blocks it; the alert stays current. (Two-connection test: notify reads recovered, sync re-sets `drive_error` + re-upserts, notify's conditional resolve ‚Üí no-op; the alert remains unresolved.)
- **AC-B3.4** A transient that clears before 1h ‚Üí no email (quiet-success).
- **AC-B3.5 (global stall)** Heartbeat older than `STALENESS_THRESHOLD_MS` (or NULL) ‚Üí exactly one show-wide `SYNC_STALLED`; fresh heartbeat resolves it. Negative-regressions: a perpetually-FAILING show (fresh `last_synced_at` + error alert) does NOT trigger `SYNC_STALLED`; a quiet healthy show does NOT.
- **AC-B3.5a (fresh-deploy)** Right after the migration (heartbeat = now()), the first tick before any sync run raises NO `SYNC_STALLED`, even with toggles ON.
- **AC-B3.5c (no-folder no false stall)** With no watched folder configured, the scheduled sync's `no_folder_configured` completion bumps `last_sync_cron_completed_at`, so after >1h the detector raises NO `SYNC_STALLED`. (Test: no-folder deployment + >1h of skipped runs ‚Üí zero stall email.)
- **AC-B3.8a (200-only delivered; 409-invalid is a failure; no-downgrade)** ONLY a Resend `200` records `sent` ‚Äî incl. a same-key/SAME-payload replay after a crash-before-record (`200` returns the original id ‚Üí `sent`, no second email). A `409 invalid_idempotent_request` (different payload) is NOT recorded `sent` ‚Äî it is `failed`/retryable + `EMAIL_DELIVERY_FAILED` (a prior failed/5xx may have reserved the key without delivering; never silently mark `sent`). A late failure never downgrades an existing `sent` row. (Tests: crash-before-record + same-payload retry ‚Üí `200` replay ‚Üí one `sent` row; a `409 invalid` ‚Üí `failed` + alert, NOT `sent`; payload stability holds so a legitimate retry doesn't 409.)
- **AC-B3.8b (failure-alert lifecycle)** `EMAIL_DELIVERY_FAILED` is emitted on the FIRST failure and stays open through retries (attempt_count 1, 2, 3 ‚Äî NOT resolved mid-retry); it resolves only when no current-occurrence ELIGIBLE `failed` row remains: a later success clears it, OR the occurrence is no longer a current eligible candidate (digest PRIOR ET-date / source alert resolved OR show unpublished/archived / pending_ingestions row deleted OR now wizard-scoped). (Tests: assert the alert persists at attempt_count 1 and 2; assert an old capped digest failure does NOT keep it open after a later day's digest succeeds; assert a failed show-level delivery clears once the show is archived/unpublished; assert a failed pending-ingestion delivery clears once the row is deleted OR becomes wizard-scoped; assert a failed delivery clears once its recipient is revoked, and that a mixed active+revoked-recipient failure keeps the alert open ONLY while an active-recipient failure remains; assert a failed GLOBAL `SYNC_STALLED` delivery keeps the alert open until the `SYNC_STALLED` alert resolves / recipient revoked / a later send succeeds; assert OCCURRENCE-EXACTNESS ‚Äî an old failed show/global/pending row that resolved then REOPENED with a fresh occurrence (new `raised_at`/`first_seen_at` epoch) does NOT re-become current and does NOT reactivate the alert for the new occurrence unless a delivery for THAT occurrence failed.)
- **AC-B3.8c (cron coverage)** Both notify jobs (`fxav_cron_notify_realtime`, `fxav_cron_notify_digest`) appear in the x6 pg-cron coverage with `net.http_get` (NOT POST), the Authorization-bearer header, and `active=true`. (The x6 guard fails if a job uses POST or is missing.)
- **AC-B3.5b (pending-ingestion)** A `pending_ingestions` row aged >1h, `wizard_session_id IS NULL` ‚Üí one email per recipient dedup'd by `drive_file_id`; row deleted before 1h ‚Üí no email; fresh `first_seen_at` re-arms; wizard-scoped rows excluded.

## 13. Watchpoints / do-not-relitigate
- **At-least-once + the Resend Idempotency-Key is the chosen delivery contract (RD-9, owner-approved).** A rare duplicate after a process crash beyond the 24h replay window is **accepted**, not a defect. Do NOT re-introduce the claim-before-send / session-lock / terminal-state / submission-state / window-expiry / attempt-scoped-key machinery ‚Äî that was explicitly trimmed (B3-DEF-9).
- **Accepted residual edges (all benign for a single-operator notifier):** a crash mid-send may rarely duplicate one email (idem-key covers the 24h common case); a >1h-stuck sheet recovering the same minute may get one heads-up; a digest retried after a crash may show slightly-stale content. None cause spam, silent misses, or privacy leaks.
- **No per-show advisory lock in the notify job** (¬ß4.4 step 5) ‚Äî it mutates only `admin_alerts` (idempotent upsert) + `email_deliveries`.
- **`SYNC_STALLED` is a show-wide cron-heartbeat alert, NOT per-show `last_synced_at`** (¬ß4.2) ‚Äî `last_synced_at` bumps on failures + skips quiet shows; per-show failures are covered by their error alerts.
- **`activeRecipients()` uses the service-role client, NOT `listAdminEmails`** (¬ß5.1) ‚Äî the RLS-bound reader returns zero rows under cron. Recipient-active re-check is on the send path (privacy: never email a revoked admin).
- **ONLY a Resend `200` records `sent`** (¬ß3.2/¬ß5.1) ‚Äî a `409 invalid_idempotent_request` is NOT proof of delivery (a failed/5xx request can reserve the key without delivering), so it is a FAILURE (`failed` + alert), NEVER silently `sent`. `409 concurrent_idempotent_requests` (in-flight) ‚Üí `retry_later` (not a failure). Do not treat `invalid_idempotent_request` as success.
- **Payloads are byte-stable per idempotency key** (¬ß3.2): realtime/ingestion render deterministically from the fixed occurrence; the digest renders ONCE and stores `{subject,html,text}` in `context`, reused on retry. So legitimate retries `200`-replay (never `409 invalid`); a `409 invalid` then means a real payload-stability bug and fails loudly. Do not rebuild digest content on retry.
- **Provider Idempotency-Key is a BOUNDED hash including `recipient`** (¬ß3.2) ‚Äî under Resend's 256-char cap; `recipient` must be in the key or the provider would dedup-away the 2nd admin.
- **The notify-side recovery-resolve is an ATOMIC CONDITIONAL update (re-reads `shows` in the `UPDATE`), NOT a read-then-resolve** (¬ß4.3) ‚Äî else a stale "recovered" read could clobber a current alert that a concurrent sync just re-set. This closes the TOCTOU without the notify job taking the per-show lock (the sync path is the primary resolver, under its lock).
- **Pending-ingestion email/digest copy goes through the shared safe resolver + `SHEET_PROCESS_FAILED` fallback** (¬ß4.5/¬ß5.3) ‚Äî `last_error_code` is unconstrained text; a raw `messageFor` lookup leaks a raw code/placeholder (invariant 5) or crashes the tick.
- **Recovery-resolution maps `last_sync_status`‚Üícatalog code, NOT raw `last_sync_error` equality** (¬ß4.3) ‚Äî `last_sync_error` holds raw causes (`SYNC_FILE_FAILED`), so a raw equality check would falsely resolve/skip every real alert; use the health-layer status‚Üícode map (`driveConnectionHealth.ts`). Per-code (resolve on code-switch too), not just "became OK". Without it, cleared/switched/raw-cause errors email stale + recurrences coalesce into the wrong row. `EMAIL_DELIVERY_FAILED`/`EMAIL_NOT_CONFIGURED` resolve symmetrically (¬ß6).
- **The digest body is CAPPED** (¬ß5.3) ‚Äî `DIGEST_MAX_SHOWS`/`DIGEST_MAX_ITEMS_PER_SHOW` with "+N/+M more ‚Üí /admin" overflow; counts from the SOURCE totals (anti-tautology), not the truncated render list. Prevents an unbounded body (provider failure) or silent omission.
- **Digest realtime-exclusion is `pending_ingestion`-ONLY** (¬ß5.3, RD-7) ‚Äî that is the only inbox item type the realtime tier emails; `first_seen`/`existing_staged` (pending_syncs) items have NO realtime counterpart and are ALWAYS included with no `admin_alert` join (don't invent a phantom alert join or drop them). The digest query selects `pending_ingestions.first_seen_at` (the dashboard default omits it) to match the ingestion occurrence key.
- **`SHEET_PROCESS_FAILED` is the pending-ingestion fallback, NOT a show-level `SYNC_PROBLEM_CODES` member** (¬ß4.1/¬ß4.5) ‚Äî the show-level set contains only codes with real show/admin_alert producers (a constants meta-test asserts this).
- **Maintenance always runs; only DELIVERY is gated** (¬ß4.4 step 0, ¬ß6) ‚Äî `SYNC_STALLED` detect/resolve must not be blinded by a toggle-off / unconfigured provider.
- **Digest + realtime-ingestion both apply `wizard_session_id IS NULL`** (¬ß4.5, ¬ß5.3); digest content is built fresh from the live inbox via the service-role client.
- **The notify cron jobs use `net.http_get`, not POST** (¬ß5.3, RD-3) ‚Äî the x6 pg-cron-coverage guard asserts GET; the route reads `?job=` from the query string and the Vault `CRON_SECRET` from the Authorization-bearer header. Adding the two canonical `fxav_cron_notify_*` jobs to x6 coverage is mandatory.
- **`EMAIL_DELIVERY_FAILED` is a single POSITIVE-predicate lifecycle** (¬ß6) ‚Äî OPEN while ‚â•1 `failed` row is CURRENT, RESOLVE when none is, independent of `attempt_count` (NOT "only at cap"; emitted on first failure, kept open through retries). CURRENT ‚ü∫ recipient active AND occurrence live ‚Äî ONE OCCURRENCE-EXACT branch per realtime kind incl. global stall (the live source's `raised_at`/`first_seen_at` epoch MUST equal the failed row's `dedup_key` epoch ‚Äî an old failed row must NOT re-become current when a fresh occurrence opens; digest = today's ET date; show-level/global = unresolved alert of the SAME `raised_at` + show published‚àß¬¨archived; pending = row of the SAME `first_seen_at` + wizard NULL). Phrase it positively (the live conditions), not as the resolved/non-current cases, to avoid inverting it; cover every candidate kind (don't drop global-stall). No terminal-state column (RD-9 trim).
- **Revoked-recipient protection is BEST-EFFORT** (¬ß4.4 step 4, AC-B3.15) ‚Äî the re-check close to the send closes the common case; a revoke in the sub-second check-to-send window MAY deliver one final email. This is an accepted residual (RD-9), not a defect to fix with a per-send lock.
- **The three ¬ß12.4 codes are full predicate rows** (¬ß8) ‚Äî `dougFacing`/`crewFacing`/`followUp`/`helpfulContext`/`title`/`longExplanation`/`helpHref`, agreeing across the three-lockstep layers (x1).
- **The 2026-05-09 notification memo is principles-only** (¬ß1.3) ‚Äî its signed-token/feedback machinery is superseded by the crew-auth pivot + B2.
- **Both toggles default ON; digest hourly-fire + ET-hour window is the DST-safe design** (RD-8, ¬ß5.3).
