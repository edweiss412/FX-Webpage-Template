## 1. Scope and boundaries

### 1.1 Problem
The "Alert me about sync problems" + "Daily review digest" toggles are operator **attention surfaces**: Doug's surface is Drive, not the admin dashboard, so a stalled sync or a staged event that lives only in-app is functionally invisible. The in-app `admin_alerts` surface (AlertBanner / NotifBell) is the system of record (pull); B3 adds the email **push** layer over it. Per **AGENTS.md no-zombie-flag**, neither toggle ships without its working backend — hence the toggles ship *with* the email subsystem.

### 1.2 In scope
First outbound-email path (`lib/notify/` channel-agnostic seam, email-only impl, Resend, secrets); the two real-time + digest email tiers; a simple `email_deliveries` dedup/audit ledger; the global `SYNC_STALLED` heartbeat detector (+ a small **cross-into-sync-path** heartbeat write, recovery-resolution calls, and **two new `admin_alerts` producers** — `DRIVE_FETCH_FAILED`/`PARSE_ERROR_LAST_GOOD`, which have no producer today so two of three RD-5 realtime classes can't notify until added (§4.1) — Codex backend); two `app_settings` toggles + read/write + UI; three new §12.4 codes (`SYNC_STALLED`/`EMAIL_DELIVERY_FAILED`/`EMAIL_NOT_CONFIGURED`) plus two `AdminAlertCode`-union additions for the already-cataloged `DRIVE_FETCH_FAILED`/`PARSE_ERROR_LAST_GOOD`; `app/api/cron/notify/route.ts` + two pg_cron schedules; TDD tests + meta-test registrations.

### 1.3 Out of scope (deferred — §12)
Two-way feedback, inbound webhook, apply-from-email, unpublish tokens, SMS impl, per-admin prefs, digest→realtime promotion, and (RD-9) the exactly-once crash-recovery outbox machinery. The 2026-05-09 `notification-design-memo.md` is cited for **principles** only (push-not-pull, tiering, quiet-success, coalescing).

## 2. Architecture overview

```
   pg_cron (pg_net)        ┌─────────────────────────────────────────────┐
   ─ realtime sched ─────► │  app/api/cron/notify/route.ts (CRON_SECRET)  │
   ─ digest sched   ─────► │   ?job=realtime | ?job=digest                │
                           └───────────────┬─────────────────────────────┘
                                           │ service-role client
   ALWAYS (maintenance, no provider/toggle needed):
     • global SYNC_STALLED detect+resolve (heartbeat)   • recovery-resolution recheck
   GATED delivery (provider configured AND tier toggle ON):
     realtime: read candidates (admin_alerts ∈ SYNC_PROBLEM_CODES aged ≥1h;
               global SYNC_STALLED; pending_ingestions >1h) → for each × active
               recipient: skip if a 'sent' ledger row exists, else send via Resend
               (Idempotency-Key) → upsert email_deliveries sent|failed
     digest:   ET-hour ∈ [7,10) → build needs-attention mirror (grouped, exclude
               already-'sent'-realtime, wizard_session_id IS NULL); 0 items → no send;
               per recipient send (Idempotency-Key 'digest:<ET_date>') → ledger
```

Recipients: service-role read of `admin_emails` where `revoked_at IS NULL` (§5.1). The job mutates only `admin_alerts` (via the idempotent SECURITY DEFINER `upsert_admin_alert`) + `email_deliveries` — not the invariant-2 protected set — so it holds **no** per-show advisory lock (`// not-subject-to-meta` in §4).

### 3.2 `email_deliveries` ledger (new table, B3 migration)
```sql
create table public.email_deliveries (
  id              uuid primary key default gen_random_uuid(),
  kind            text not null check (kind in ('realtime_problem','digest')),
  channel         text not null default 'email' check (channel in ('email','sms','webhook')),
  dedup_key       text not null,   -- realtime show-level '<show_id>:<code>:<epoch(raised_at)>'; global 'global:SYNC_STALLED:<epoch(raised_at)>'; ingestion 'ingestion:<drive_file_id>:<epoch(first_seen_at)>'; digest 'digest:<ET_date>'
  show_id         uuid references public.shows(id) on delete set null,
  recipient       text not null,
  triggered_codes text[] not null default '{}',
  context         jsonb not null default '{}',   -- OPTIONAL audit only (e.g. render inputs); NOT load-bearing for delivery — payloads are re-rendered each attempt and a 409-invalid reissues with an attempt-scoped key (§3.2)
  status          text not null check (status in ('sent','failed')),
  provider_message_id text,
  error           text,
  attempt_count   int  not null default 1,
  created_at      timestamptz not null default now(),
  sent_at         timestamptz,
  constraint email_deliveries_recipient_email_canonical
    check (recipient = lower(trim(recipient)) and recipient <> '')   -- invariant 3 safety net
);
create unique index email_deliveries_dedup on public.email_deliveries (kind, dedup_key, recipient);
```
- **Access lockdown — service-role-only, INCLUDING SELECT (privacy):** the ledger holds recipient emails + `provider_message_id` + provider `error` text, and NO product UI reads it in v1, so it is fully service-role-only (stricter than the repo's admin-only-read tables): `alter table public.email_deliveries enable row level security;` (deny-by-default — NO anon/authenticated policy), `revoke all on table public.email_deliveries from anon, authenticated;`, `grant all on table public.email_deliveries to service_role;`. The cron job reads/writes it with the service-role client. The `tests/db/postgrest-dml-lockdown.test.ts` entry declares SELECT=**denied** for anon/authenticated (not just DML) and asserts a non-admin SELECT is rejected (no read-leak of recipient emails / audit data).
- **Provider Idempotency-Key (the sole dup guard):** `fxav:<kind>:<sha256(kind || ':' || dedup_key || ':' || recipient)>` — a **bounded** ~90-char hash (Resend caps the key at **256 chars**; the raw email-bearing concat is hashed). Verified Resend contract (docs 2026-06-02): a repeat with the same key within **24h** returns the original response (no second email); same key + different payload → `409 invalid_idempotent_request`.
- **Dedup + at-least-once delivery (RD-9):** before sending a candidate × recipient, skip if a `status='sent'` row already exists for `(kind, dedup_key, recipient)`. Otherwise re-render the payload from the current source and send via Resend with the base idempotency key, then upsert the row to `sent` (+`provider_message_id`,`sent_at`) or `failed` (+`error`, `attempt_count`). **No claim/lock/terminal state, and NO payload-stability requirement** (payloads MAY change between attempts — `upsert_admin_alert` overwrites `context` on re-occurrence (`supabase/migrations/20260505000000_upsert_admin_alert.sql:16-19`), show titles/catalog/templates are mutable, and the digest is point-in-time).
  - **Idempotency outcomes (UNIFORM across realtime / ingestion / digest):**
    - `200` → delivered → `sent` (incl. a same-key/same-payload replay returning the original id — the common no-change retry after a transient `infra_error`).
    - `409 concurrent_idempotent_requests` (the base key is still IN-FLIGHT in an overlapping tick) → `retry_later`: skip, NO ledger write, NO `attempt_count` bump, NO alert; the in-flight tick finishes (re-checked next tick).
    - `409 invalid_idempotent_request` (the base key was reserved with a DIFFERENT payload — the original attempt rendered different bytes, then crashed/failed before recording) → **NOT proof of delivery and NOT a terminal failure.** REISSUE with an ATTEMPT-SCOPED provider key built with the SAME bounded, recipient-scoped constructor as the base key — `'fxav:'||kind||':'||sha256(kind||':'||dedup_key||':'||recipient||':r:'||<nonce>)` (bounded ~90 chars, still includes `kind`+`recipient` so two admins get DISTINCT keys and the 256-char cap holds) → `200` → `sent`. **`<nonce>` is a per-SUBMISSION unique value — a freshly generated UUID/high-resolution timestamp at the moment of the reissue call, NOT persisted and NOT derived from `attempt_count`.** This is load-bearing: a reissue key derived from the stored `attempt_count` would be RE-DERIVED IDENTICALLY on the next tick after a crash-before-record (which doesn't advance the count), and if the payload changed again that re-derived key would itself `409 invalid` forever → a STUCK/MISSED notification (violating "deliver, not miss" — worse than RD-9's accepted duplicate). A fresh per-submission nonce guarantees every reissue gets a UNIQUE provider key that has never been poisoned → it always delivers. The ONLY residual cost is an RD-9-accepted DUPLICATE (if a prior reissue delivered but crashed before recording, the next tick's fresh-nonce reissue delivers again) — never a miss. `attempt_count` is DECOUPLED from the key: it is solely the retry-CAP counter (§5.1). The ledger `dedup_key` stays the base (one `sent` row per logical message per recipient); the attempt-scoped hash is ONLY the provider Idempotency-Key for that reissue.
    - `infra_error` / `400 invalid_idempotency_key` / 5xx / network → `failed`/retryable + `EMAIL_DELIVERY_FAILED` (§6).
  - **Why uniform (no per-kind split):** because no payload is guaranteed byte-stable, ANY kind can hit `409 invalid` on a re-render; treating it as a loud failure for realtime/ingestion would MISS a real sync-problem email (RD-9 accepts dups, not misses). So all kinds deliver-via-reissue. The base idempotency key still kills the COMMON duplicate (overlapping ticks + unchanged-payload retries); the attempt-scoped reissue is the rare changed-payload-after-reservation path. `context` (DDL) is optional audit only — NOT load-bearing for delivery.
  - **The ledger upsert NEVER downgrades a `sent` row to `failed`** (the failure upsert is conditional `WHERE status <> 'sent'`), so a late/racing failure can't clobber a recorded success.
- **Re-arm:** `dedup_key` includes the occurrence epoch (`raised_at` / `first_seen_at` / ET-date), so a resolved-then-reopened problem (fresh `raised_at`, via §4.3 recovery-resolution) is a NEW key → a new email. `email_deliveries_recipient_email_canonical` keeps the recipient canonical (it is part of the dedup identity).
- **Retry:** a `failed` row under `SEND_RETRY_CAP (3)` is re-sent on a later tick (same idempotency key); at the cap it stays `failed` and `EMAIL_DELIVERY_FAILED` (§6) remains until the underlying problem clears.

## 4. Detection (real-time tier)

### 4.1 `SYNC_PROBLEM_CODES` (single source of truth — `lib/notify/constants.ts`)
`['DRIVE_FETCH_FAILED','SHEET_UNAVAILABLE','PARSE_ERROR_LAST_GOOD']` (the show-level consumed error codes). **PRODUCER GAP — B3 MUST ADD TWO PRODUCERS (live-code-verified):** today only `SHEET_UNAVAILABLE` has a show-level `admin_alerts` producer (`runScheduledCronSync.ts:1819,1872`; also `runManualSyncForShow.ts:172`), and the `AdminAlertCode` union (`lib/adminAlerts/upsertAdminAlert.ts:3-29`) contains `SHEET_UNAVAILABLE` but NOT `DRIVE_FETCH_FAILED`/`PARSE_ERROR_LAST_GOOD` — those two are §12.4 catalog codes (`lib/messages/catalog.ts:54,93`) DERIVED for admin-health DISPLAY by `lib/admin/driveConnectionHealth.ts:66,70`, but NO `admin_alerts` row is ever written for them. Since the realtime tier consumes `admin_alerts` (RD-6) and aging uses `admin_alerts.raised_at` (the only error-onset timestamp — `shows.last_synced_at` bumps every tick incl. failures, so direct-from-`shows` detection can't do the ≥1h aging), **B3 adds show-level `upsertAdminAlert` producers for `DRIVE_FETCH_FAILED` (at the `markShowDriveError`/`drive_error` transition, `runScheduledCronSync.ts:709,1852`) and `PARSE_ERROR_LAST_GOOD` (at the `parse_error` transition)** — mirroring the existing `SHEET_UNAVAILABLE` producer (context carries `drive_file_id`+`sheet_name` for §12.4 AlertBanner interpolation). This requires: (a) extend the `AdminAlertCode` union with both codes; (b) the §12.4 catalog rows already EXIST (`catalog.ts:54,93`) — no new rows, but if AlertBanner copy/context changes, the §12.4 three-lockstep applies (master spec §12.4 prose + `gen:spec-codes` + `catalog.ts`); (c) companion-surface parity — add the same producer to the manual-sync path (`runManualSyncForShow.ts`, which today produces only `SHEET_UNAVAILABLE`) so a manually-triggered sync surfaces the same alerts; (d) producer regression tests landing BEFORE the notify-consumer tests (§10). `STAGED_PARSE_*` + info-severity codes are excluded (`STAGED_PARSE_SOURCE_GONE` is a raw `last_sync_error` cause that maps to the `SHEET_UNAVAILABLE` producer, §4.3 — NOT its own code). **`SHEET_PROCESS_FAILED` is NOT in this set** — it is the pre-show pending-ingestion safe-copy FALLBACK (§4.5), not a show-level alert producer; including it would make the show-level `admin_alerts` JOIN `shows` query search for a code no show-level path emits. A constants meta-test asserts every code in `SYNC_PROBLEM_CODES` has a real show-level `admin_alerts` producer — this test FAILS today for the two missing codes and PASSES only once B3 adds their producers (it is both the B3 deliverable's acceptance gate AND the anti-drift guard). Referenced by the query and the tests, never re-listed inline. `SYNC_STALLED` (show-wide) + pending-ingestion are handled by their own detectors, not this set.

### 4.2 Global sync-stall detector (`SYNC_STALLED`, show-wide)
`shows.last_synced_at` is the wrong staleness signal (bumped on failures; not bumped for quiet shows). Per-show failures are already covered by their error alerts (§4.3); the only gap is the whole pipeline stopping. So `SYNC_STALLED` is a single show-wide alert driven by a sync-run heartbeat:
- **Heartbeat (B3 adds — cross-into-sync-path):** the scheduled sync run writes `app_settings.last_sync_cron_completed_at = now()` at the END of EVERY completed run — success, all-quiet, AND the **`no_folder_configured` / skipped** completion (the heartbeat means "the cron executed", which it did). This prevents a false `SYNC_STALLED` on a pre-onboarding / no-watched-folder deployment (the cron runs hourly, skips, and bumps the heartbeat, so it never goes stale). Trigger-agnostic.
- **Detector (always, in `?job=realtime`):** if `now() - coalesce(last_sync_cron_completed_at,'-infinity') > STALENESS_THRESHOLD_MS` → `upsert_admin_alert(NULL,'SYNC_STALLED',…)` (one show-wide row). **Resolve** when the heartbeat is fresh again.
- **Common-mode limitation (documented):** the detector runs on the same scheduler it watches, so a total pg_cron/pg_net outage can't self-detect → covered by an external uptime monitor (B3-DEF-8, out of scope).

### 4.3 Gap-fill audit + recovery-resolution (implementation task — cross-into-sync-path)
Confirm each `SYNC_PROBLEM_CODES` condition both (a) UPSERTS its code's `admin_alerts` row on entry — INCLUDING the two producers B3 adds for `DRIVE_FETCH_FAILED` + `PARSE_ERROR_LAST_GOOD` (§4.1 producer gap), not just the pre-existing `SHEET_UNAVAILABLE` — AND (b) RESOLVES that row whenever its **exact code predicate** no longer holds. **Per-code, not just "became OK":** a show can switch `DRIVE_FETCH_FAILED`→`PARSE_ERROR_LAST_GOOD` while STILL in an error status, so each `SYNC_PROBLEM_CODES` alert must resolve when the show's current state no longer maps to that alert's code. **CRITICAL — `shows.last_sync_error` stores RAW causes, NOT catalog codes:** `drive_error` rows store e.g. `SYNC_FILE_FAILED`, and the admin-health layer MAPS the `last_sync_status` bucket to the catalog code (`drive_error`→`DRIVE_FETCH_FAILED`, `lib/admin/driveConnectionHealth.ts`). So the resolve/recheck predicate uses an explicit **`last_sync_status` → catalog-code MAP** (NOT raw `last_sync_error` equality, which would never match `DRIVE_FETCH_FAILED`/`PARSE_ERROR_LAST_GOOD` and would falsely resolve/skip every real alert). Map (documented in full in the plan, derived from the health layer): `drive_error`→`DRIVE_FETCH_FAILED`; `parse_error`→`PARSE_ERROR_LAST_GOOD`; `sheet_unavailable`→`SHEET_UNAVAILABLE` **ALWAYS** (this status maps to a SINGLE alert code — the live producer upserts the alert as `SHEET_UNAVAILABLE` for BOTH raw `last_sync_error` causes, `SHEET_UNAVAILABLE` AND `STAGED_PARSE_SOURCE_GONE`; `STAGED_PARSE_SOURCE_GONE` is a raw cause stored in `last_sync_error`, NOT a distinct alert/email code — `runScheduledCronSync.ts:1850-1851,1869-1872`). So the map keys ONLY on `last_sync_status` here — do NOT branch on `last_sync_error` for `sheet_unavailable`, else a source-gone show (`last_sync_error=STAGED_PARSE_SOURCE_GONE`) would map to a phantom `STAGED_PARSE_SOURCE_GONE` code, falsely resolve the open `SHEET_UNAVAILABLE` alert, and skip its email. `SHEET_PROCESS_FAILED` is pre-show (§4.5). An alert resolves when the show's current `last_sync_status` maps to a DIFFERENT code (or OK); the new code upserts its own alert. Today resolution is only on manual admin-resolve routes; **B3 ADDS the per-code recovery-resolution** to the sync path. Load-bearing: without it, a cleared/switched error still emails its stale code 1h later AND `upsert_admin_alert` coalesces future occurrences into the wrong never-resolved row (re-arm depends on resolution refreshing `raised_at`). **Notify-side recheck (atomic, race-safe — NO lock):** the PRIMARY resolver is the sync path (above, under the show lock). The notify-side recheck is a DEFENSIVE conditional resolve that must NOT clobber a current alert on a stale read (a concurrent sync can re-set the error + re-upsert the same row between notify's read and its resolve). So it is a single atomic statement that re-reads the show's CURRENT state inside the UPDATE: `update admin_alerts set resolved_at=now() where id=$alertId and resolved_at is null and not exists (select 1 from shows s where s.id=$showId and <s.last_sync_status maps to this alert's code>)`. If a concurrent sync has re-set the error, the `not exists` is false → notify does NOT resolve (correct); if notify resolves first, sync's later `upsert_admin_alert` sees `resolved_at` set → INSERTs a fresh row (re-armed). This closes the TOCTOU WITHOUT the notify job taking the per-show advisory lock (preserving §4.4's no-lock posture). The email is skipped only when this conditional resolve fired (or the alert was already resolved). The plan documents the full `last_sync_status`→code map (keyed on status alone — no status disambiguates on `last_sync_error`; `sheet_unavailable`→`SHEET_UNAVAILABLE` covers both the `SHEET_UNAVAILABLE` and `STAGED_PARSE_SOURCE_GONE` raw causes).

### 4.4 Delivery loop (realtime)
Candidates from three sources: §4.2 staleness (→ `admin_alerts.SYNC_STALLED`), §4.3 consumed error alerts, §4.5 pending-ingestion.
0. **Maintenance (ALWAYS — no provider/toggle needed):** global `SYNC_STALLED` detect+resolve (§4.2) and the recovery-resolution recheck (§4.3) — they upsert/resolve IN-APP `admin_alerts` so the dashboard signal stands even when email is OFF/unconfigured.
1. **Provider gate:** config invalid (§6 — missing Resend secret OR invalid site origin) → skip all sends; raise `EMAIL_NOT_CONFIGURED` ONLY when at least one tier toggle is ON (`alert_on_sync_problems OR daily_review_digest`), resolve it when config becomes valid OR both toggles are OFF (§6 positive predicate — don't warn about unconfigured email when Doug has turned both tiers off). Else (config valid) continue.
2. **Tier-toggle gate:** read `alert_on_sync_problems` fail-closed (`lib/appSettings/getAlertOnSyncProblems.ts`, mirroring `getAutoPublishCleanFirstSeen.ts`); false/read-error → skip realtime sends (maintenance already ran).
3. **Candidates:**
   - **(a) Show-level:** unresolved `admin_alerts` JOIN `shows` where `code = any(SYNC_PROBLEM_CODES)` AND `raised_at <= now()-interval '1 hour'` AND `shows.published = true` AND `shows.archived = false` (never email problems for inactive shows). Key `'<show_id>:<code>:<epoch(raised_at)>'`.
   - **(b) Global stall:** the unresolved show-wide `SYNC_STALLED` (`show_id IS NULL`); NO `shows` join; NO extra age gate (the 1h is already in §4.2's threshold). Key `'global:SYNC_STALLED:<epoch(raised_at)>'`, link → `/admin`.
   - **(c) Pending-ingestion (§4.5):** `pending_ingestions` where `now()-first_seen_at > interval '1 hour'` AND `wizard_session_id IS NULL`. Key `'ingestion:<drive_file_id>:<epoch(first_seen_at)>'`, `show_id NULL`, `triggered_codes=[last_error_code]`, link → `/admin`.
4. **Send:** for each candidate × each active recipient, **re-check the recipient is still active** (`admin_emails.revoked_at IS NULL`, close to the send; revoked → skip, never email a revoked admin); skip if a `status='sent'` ledger row exists; else `lib/notify.sendEmail(...)` → Resend (Idempotency-Key §3.2); upsert `sent`/`failed` (+`EMAIL_DELIVERY_FAILED` on failure, §6).
5. **Advisory-lock posture:** the loop mutates only `admin_alerts` (idempotent upsert) + `email_deliveries` and READS `shows`/`pending_ingestions`; it does not mutate the invariant-2 protected tables, so it holds **no** per-show lock (`// not-subject-to-meta: notify job mutates only admin_alerts + email_deliveries`).

### 4.5 Stuck-ingestion detector (pre-show — second documented exception to RD-6)
Pending ingestions are pre-show (no `show_id`), so `admin_alerts`' `(show_id,code)` upsert can't dedup per-file; this class is read directly and dedup'd by `drive_file_id`.
- **Select:** `pending_ingestions` older than `STALENESS_THRESHOLD_MS` AND `wizard_session_id IS NULL` (exclude active onboarding, matching the dashboard live-scope `components/admin/Dashboard.tsx:328-331`). A row's presence = the sheet hasn't become a show; a successful re-ingestion deletes the row → no email (quiet-success).
- **Dedup/re-arm:** `'ingestion:<drive_file_id>:<epoch(first_seen_at)>'`. Re-arms on a fresh `first_seen_at`.
- **Email (safe copy — invariant 5):** `pending_ingestions.last_error_code` is UNCONSTRAINED text, so the body MUST resolve it through the SAME shared safe-copy resolver the dashboard NeedsAttentionInbox uses (the Phase A `resolveIngestionCopy`-equivalent), passing `sheet_name = drive_file_name`, and FALL BACK to the generic `SHEET_PROCESS_FAILED` `dougFacing` for an unknown / null-dougFacing / crew-only / unresolved-`<…>`-placeholder code. A raw `messageFor(last_error_code)` lookup would crash rendering or leak a raw code / `<sheet-name>` placeholder into the email (and could halt the cron tick before later sends). Link → `/admin`. Gated by `alert_on_sync_problems`.
- **No advisory lock (RD-9):** a lock-less read may *rarely* email a stuck sheet just as recovery deletes the row; for a >1h-stuck sheet that is a real, mildly-stale heads-up, not spam — accepted.

### 5.1 `lib/notify/`

## 6. Failure & unconfigured-provider handling
- **Send failure:** Resend `{ ok:false, kind:'infra_error' }` → upsert the `email_deliveries` row `status='failed'` (+`error`) **only when the row is not already `sent`** (never downgrade a success). **`attempt_count` semantics (exact — retry-CAP counter ONLY; the reissue key does NOT derive from it — §3.2 uses a per-submission nonce):** on the FIRST failure the row is INSERTed with `attempt_count = 1` (the DDL default, §3.2 line 91 `default 1`); each SUBSEQUENT failed attempt on a not-yet-`sent` row UPDATEs `attempt_count = email_deliveries.attempt_count + 1` (so the running count is 1 → 2 → 3, matching AC-B3.8/AC-B3.8b — never starts at 2). The row is retried next tick ONLY while the STORED `attempt_count < SEND_RETRY_CAP (3)`. Emit `upsert_admin_alert(show_id,'EMAIL_DELIVERY_FAILED',…)` (in-app only — **never** email about email failing). (Per §3.2/§5.1: ONLY a Resend `200` records `sent`. `409 invalid_idempotent_request` (`idempotency_conflict`) is NEVER recorded `sent` and is NEVER a terminal failure — for ALL kinds it REISSUES with an attempt-scoped key to deliver (accept a possible duplicate, §3.2). `409 concurrent_idempotent_requests` → `retry_later`: skip, NO ledger write, NO `attempt_count` bump, NO alert.) An `infra_error`/5xx/network failure is retried next tick while `attempt_count < SEND_RETRY_CAP (3)`; at the cap it stays `failed` (alert persists per §6 lifecycle).
- **`EMAIL_DELIVERY_FAILED` lifecycle (single POSITIVE predicate):** the alert is OPEN whenever ≥1 `failed` row is **CURRENT**, and RESOLVES when none is — independent of `attempt_count` (emitted on the FIRST failure, kept open through retries 1/2/3, never flickers off mid-retry; `SEND_RETRY_CAP` only stops further *retries*, not the alert). A `failed` row is **CURRENT** iff ALL THREE hold (state the LIVE conditions positively, do NOT invert): (x) its `recipient` is ACTIVE — `admin_emails.revoked_at IS NULL`; (w) **its tier toggle is ON** — `alert_on_sync_problems` for realtime kinds (show-level/global-stall/pending-ingestion), `daily_review_digest` for the digest kind (CURRENT must match the SAME eligibility gate as the send loop, §4.4: a row whose tier is now OFF would never be re-sent, so its failure is moot and the alert must resolve); AND (y) its occurrence is a LIVE candidate — one branch per `kind`/source, each **OCCURRENCE-EXACT** (the live source's occurrence epoch MUST equal the epoch embedded in the failed row's `dedup_key`, so an OLD failed row never re-becomes current when a fresh occurrence opens): **digest** → `dedup_key` IS today's ET date *and* today's digest STILL has ≥1 open needs-attention item for this recipient (if the backlog cleared, the digest wouldn't send → resolve); **show-level** (`admin_alerts.show_id` NOT NULL) → there is an UNRESOLVED `admin_alert` for this `show_id`+`code` whose `floor(epoch(raised_at))` EQUALS the row's `dedup_key` epoch, *and* the show is `published ∧ ¬archived`; **global stall** (`show_id IS NULL`) → the UNRESOLVED `SYNC_STALLED` alert's `raised_at` epoch EQUALS the row's `dedup_key` epoch; **pending-ingestion** → a `pending_ingestions` row with this `drive_file_id` whose `first_seen_at` epoch EQUALS the row's `dedup_key` epoch EXISTS *and* `wizard_session_id IS NULL`. (These mirror §4.4 step 3 (a/b/c) / §4.5 / §5.3 — every realtime candidate kind has a matching, occurrence-exact branch.) Equivalently, a failure STOPS being current — and the alert resolves once none of its scope is current — when the recipient is revoked, **its tier toggle is turned OFF**, the source alert resolves, the show is archived/unpublished, the ET-day rolls over, **the digest backlog clears**, the pending row is deleted or becomes wizard-scoped, OR a later delivery for that row succeeds (flips it to `sent`). No terminal-state column (RD-9 trim).
- **Unconfigured provider:** missing `RESEND_API_KEY`/`EMAIL_FROM` **OR a missing/blank/malformed canonical site origin (§5.4 — required for absolute links)** → **only the send loops skip**; maintenance (§4.4 step 0) still runs (`SYNC_STALLED` detect/resolve, recovery-resolution) so the in-app dashboard signal is never blinded. Returns 200 (delivery skipped). Toggles still settable; settings UI MAY show a server-derived "email not configured" hint (never leaks the secret). **`EMAIL_NOT_CONFIGURED` is a POSITIVE predicate: OPEN iff `(config invalid) AND (alert_on_sync_problems OR daily_review_digest)`; RESOLVE when config becomes valid OR both toggles are OFF.** Rationale: if Doug intentionally disables BOTH email tiers, no send is supposed to occur, so a config-not-set alert is a FALSE health signal — the toggles must be able to silence their own configuration warning. (config valid = Resend secrets present AND a valid absolute site origin present — the CONJUNCTION; any one missing ⇒ config invalid.)
- **Fail-closed reads:** both toggle reads fail-closed (read error → "off") so an infra hiccup never spams.

## 9. Acceptance criteria
- **AC-B3.1** `alert_on_sync_problems=true`, an unresolved `SYNC_PROBLEM_CODES` alert with `raised_at ≤ now()-1h` → exactly one Resend send + one `sent` row per active recipient; a second tick → no resend (the `sent` row skips it).
- **AC-B3.1c (producers exist for ALL three codes — B3 adds the two missing)** A show entering `drive_error` upserts a `DRIVE_FETCH_FAILED` `admin_alerts` row, and a show entering `parse_error` upserts a `PARSE_ERROR_LAST_GOOD` row (both via `upsertAdminAlert`, both codes added to the `AdminAlertCode` union, mirroring the existing `SHEET_UNAVAILABLE` producer); each row, once aged ≥1h with the toggle on + show active, emails its code → so all THREE RD-5 realtime classes actually notify, not just `SHEET_UNAVAILABLE`. The same producers fire on the manual-sync path (companion-surface parity). (Tests — land BEFORE the consumer tests: a `drive_error` transition writes a `DRIVE_FETCH_FAILED` row; a `parse_error` transition writes a `PARSE_ERROR_LAST_GOOD` row; the `lib/notify/constants` meta-test that asserts every `SYNC_PROBLEM_CODES` member has a producer FAILS before these are added and PASSES after. **Concrete failure mode caught:** without these producers the realtime tier silently emails only 1 of 3 advertised failure classes — drive-fetch + parse failures never notify.)
- **AC-B3.1a (concurrency)** Two overlapping realtime ticks for the same alert+recipient (identical, stable payload) → exactly ONE email delivered; the second hits Resend's `409 concurrent_idempotent_requests` (still in-flight) → `retry_later` (NO `failed` row, NO `EMAIL_DELIVERY_FAILED`, NO `attempt_count` bump); or, if the first already completed, a `200` SAME-payload replay (original id) → `sent`. Either way ONE `sent` row, no false failure. (Test both orderings: in-flight → `retry_later`; completed → `200` replay → one `sent` row.)
- **AC-B3.1b (multi-recipient — base AND reissue keys)** One alert + two active admins → two distinct idempotency keys (BOTH the base key AND, on the 409-invalid reissue path, the attempt-scoped key include `recipient`+`kind` via the bounded `fxav:<kind>:<sha256(…)>` constructor of §3.2), two `sent` rows, two emails; no recipient is provider-deduped away on either path, and every key (base + attempt-scoped) is < 256 chars (no `400 invalid_idempotency_key`). (Negative-regression: a recipient-less key constructor collapses the two admins to one send — the test must fail if `recipient` is dropped from EITHER key.)
- **AC-B3.2** `alert_on_sync_problems=false`/read-error → zero realtime emails; in-app alerts unaffected.
- **AC-B3.3** An alert that resolves then re-opens (fresh `raised_at`) re-arms → a new email. The dedup_key MUST embed the occurrence epoch (`epoch(raised_at)`) so the reopened problem is a NEW logical message keyed distinctly from the prior `sent` row. (Negative-regression: OMIT the occurrence epoch from the key — a show+code-only key — or reuse the prior occurrence's key → the old `sent` row suppresses the reopened email → second email fails to fire. The test asserts the two occurrences produce DIFFERENT dedup_keys.)
- **AC-B3.3a (recovery-resolution, per-code)** An error alert whose show recovers before 1h is resolved → no email; the notify-side recheck resolves+skips a recovered-but-unresolved alert. **Code-switch:** a show going `DRIVE_FETCH_FAILED`→`PARSE_ERROR_LAST_GOOD` WITHOUT an OK interlude resolves the old code's alert (the show's CURRENT mapped catalog code — the status→code map keyed on `last_sync_status`, NOT a raw `last_sync_error` string comparison — differs from the alert's code) and upserts the new code's alert → the old code is never emailed stale and future occurrences coalesce into the right row. (Negative-regressions: drop recovery-resolution → stale email + suppressed recurrence; resolve only on "became OK" → the code-switch case emails the stale code; **compare the alert code to raw `last_sync_error` instead of the status→code map → a real `drive_error`/`SYNC_FILE_FAILED` row is falsely resolved/skipped and never emailed** — test with the `SYNC_FILE_FAILED` drive_error fixture and assert the `DRIVE_FETCH_FAILED` alert is NOT resolved/skipped; **source-gone fixture: `last_sync_status='sheet_unavailable'` + `last_sync_error='STAGED_PARSE_SOURCE_GONE'` maps to `SHEET_UNAVAILABLE` (the only alert code this status emits — `runScheduledCronSync.ts:1869-1872`), so the open `SHEET_UNAVAILABLE` alert is NOT resolved/skipped and IS emailed** — fails if the map branches on `last_sync_error` and invents a phantom `STAGED_PARSE_SOURCE_GONE` code.) **Race (notify-side resolve vs sync re-upsert):** a notify recheck that read "recovered" must NOT resolve the alert if a concurrent sync has re-set the error — the atomic conditional resolve (re-reading `shows` in the `UPDATE`) blocks it; the alert stays current. (Two-connection test: notify reads recovered, sync re-sets `drive_error` + re-upserts, notify's conditional resolve → no-op; the alert remains unresolved.)
- **AC-B3.4** A transient that clears before 1h → no email (quiet-success).
- **AC-B3.5 (global stall)** Heartbeat older than `STALENESS_THRESHOLD_MS` (or NULL) → exactly one show-wide `SYNC_STALLED`; fresh heartbeat resolves it. Negative-regressions: a perpetually-FAILING show (fresh `last_synced_at` + error alert) does NOT trigger `SYNC_STALLED`; a quiet healthy show does NOT.
- **AC-B3.5a (fresh-deploy)** Right after the migration (heartbeat = now()), the first tick before any sync run raises NO `SYNC_STALLED`, even with toggles ON.
- **AC-B3.5c (no-folder no false stall)** With no watched folder configured, the scheduled sync's `no_folder_configured` completion bumps `last_sync_cron_completed_at`, so after >1h the detector raises NO `SYNC_STALLED`. (Test: no-folder deployment + >1h of skipped runs → zero stall email.)
- **AC-B3.8a (200-only delivered; 409-invalid → reissue, never false-sent/miss; no-downgrade)** ONLY a Resend `200` records `sent` — incl. a same-key/SAME-payload replay after a crash-before-record (returns the original id → one `sent`, no second email). A `409 invalid_idempotent_request` (different payload — the original rendered different bytes) is NEVER recorded `sent` on its own AND is NEVER a terminal failure: the row reissues with the bounded, recipient-scoped attempt-scoped key (§3.2 — `fxav:<kind>:<sha256(…:r:<nonce>)>`) → `200` → `sent` (delivers; accepts a possible duplicate if the original delivered). A late failure never downgrades an existing `sent` row. (Tests: crash-before-record + same-payload retry → `200` replay → one `sent` row; a `409 invalid` (changed payload) → attempt-scoped reissue → delivered `sent`, NOT a stuck `failed` and NOT a `sent` without a real `200`; **TWO active recipients both on the 409-invalid path get DISTINCT attempt-scoped keys (neither deduped away) and both keys are < 256 chars** — AC-B3.1b; **repeated-reissue / crash freshness: base 409-invalid → reissue (nonce N1) crashes/infra-fails before recording `sent` → NEXT tick base still 409-invalid (payload changed AGAIN) → reissue uses a FRESH nonce N2 ≠ N1 and DELIVERS** — fails if the reissue key is derived from the stored `attempt_count` (which a crash-before-record leaves unadvanced → N2 == N1 → permanently 409-invalid → stuck/missed). This pins the §3.2 per-submission-nonce contract.)
- **AC-B3.8b (failure-alert lifecycle)** `EMAIL_DELIVERY_FAILED` is emitted on the FIRST failure and stays open through retries (attempt_count 1, 2, 3 — NOT resolved mid-retry); it resolves only when no current-occurrence ELIGIBLE `failed` row remains: a later success clears it, OR the occurrence is no longer a current eligible candidate (digest PRIOR ET-date / source alert resolved OR show unpublished/archived / pending_ingestions row deleted OR now wizard-scoped). (Tests: assert the alert persists at attempt_count 1 and 2; assert an old capped digest failure does NOT keep it open after a later day's digest succeeds; assert a failed show-level delivery clears once the show is archived/unpublished; assert a failed pending-ingestion delivery clears once the row is deleted OR becomes wizard-scoped; assert a failed delivery clears once its recipient is revoked, and that a mixed active+revoked-recipient failure keeps the alert open ONLY while an active-recipient failure remains; **assert turning the failed row's TIER TOGGLE off (realtime → `alert_on_sync_problems=false`; digest → `daily_review_digest=false`) clears it from currentness → the alert resolves (matches the send-loop eligibility gate); assert a failed DIGEST row resolves once today's needs-attention backlog clears (no open item), even before the ET-day rollover;** assert a failed GLOBAL `SYNC_STALLED` delivery keeps the alert open until the `SYNC_STALLED` alert resolves / recipient revoked / a later send succeeds; assert OCCURRENCE-EXACTNESS — an old failed show/global/pending row that resolved then REOPENED with a fresh occurrence (new `raised_at`/`first_seen_at` epoch) does NOT re-become current and does NOT reactivate the alert for the new occurrence unless a delivery for THAT occurrence failed.)
- **AC-B3.8c (cron coverage)** Both notify jobs (`fxav_cron_notify_realtime`, `fxav_cron_notify_digest`) appear in the x6 pg-cron coverage with `net.http_get` (NOT POST), the Authorization-bearer header, and `active=true`. (The x6 guard fails if a job uses POST or is missing.)
- **AC-B3.5b (pending-ingestion)** A `pending_ingestions` row aged >1h, `wizard_session_id IS NULL` → one email per recipient dedup'd by `drive_file_id`; row deleted before 1h → no email; fresh `first_seen_at` re-arms; wizard-scoped rows excluded.
- **AC-B3.6 (digest)** First in-window ET tick (hour 07) with ≥1 open item → one digest per active recipient, grouped by show, excluding items already realtime-`sent`; later in-window tick → no additional send to a `sent` recipient (dedup_key replay). 0 open items → no send.
- **AC-B3.6a (digest live-scope)** Digest applies `wizard_session_id IS NULL` on `pending_ingestions` + `pending_syncs` (service-role). (Negative-regression: drop the filter → wizard rows leak.)
- **AC-B3.6d (changed-payload retry → deliver, not miss — applies to all kinds; digest is the common case)** When a retry re-renders DIFFERENT bytes (mutable alert `context`/show title for realtime; point-in-time rebuild for digest) and the base key was already reserved → `409 invalid_idempotent_request` → REISSUE with an attempt-scoped key → DELIVERS (one `sent` row keyed by the base `dedup_key`), accepting a possible duplicate rather than a silent miss. (Test: reserve the base key with payload A, then a retry rendering payload B → assert delivery via the attempt-scoped key, NOT a stuck `failed`; verify for a realtime alert whose `context` changed AND for a rebuilt digest.)
- **AC-B3.6c (digest caps + overflow)** With more open items than the caps (`DIGEST_MAX_SHOWS`/`DIGEST_MAX_ITEMS_PER_SHOW`), the digest renders bounded content with `"+N more on this show"` / `"+M more shows"` linking to `/admin`, and the subject + overflow counts are computed from the SOURCE totals (NOT the truncated render list). (Test: seed > caps items → assert bounded render, correct "+N"/"+M" counts derived from the source query, no unbounded body.)
- **AC-B3.6b (pending-ingestion-only exclusion + key derivation)** The realtime exclusion applies ONLY to `pending_ingestion` digest items; the digest query selects `pending_ingestions.first_seen_at` to match a `sent` realtime delivery's `ingestion:<drive_file_id>:<epoch(first_seen_at)>` key. (Tests: a `sent` realtime pending-ingestion delivery → that exact occurrence excluded, a fresh re-armed occurrence still included; a `first_seen`/`existing_staged` pending_syncs item is ALWAYS included with NO `admin_alert` join (never dropped for lack of an alert). Negative-regression: omit `first_seen_at` → the already-emailed sheet leaks into the digest.)
- **AC-B3.7 (DST)** Across spring-forward / fall-back, the digest fires within the 07:00-ET window (injected ET clock).
- **AC-B3.7a (inactive-show suppression)** A `SYNC_PROBLEM_CODES` alert unresolved after its show is archived/unpublished → no realtime email (the `shows` join requires `published ∧ ¬archived`).
- **AC-B3.8** Resend `infra_error` → a `failed` row + `EMAIL_DELIVERY_FAILED` (emitted on the FIRST failure) + retry next tick; at `attempt_count=3` retries stop; no email about the failure. **`attempt_count` value test (explicit, §5.1):** first failure INSERTs `attempt_count=1`; second failed attempt UPDATEs to `2`; third to `3`; the fourth tick does NOT retry (stored count is at the cap). Assert all three values + the no-fourth-retry boundary (catches an off-by-one that would start at 2, consume a retry, and shift the attempt-scoped reissue key). `EMAIL_DELIVERY_FAILED` resolves when no CURRENT failed row remains (the §6 positive predicate), independent of `attempt_count` — NOT "only at cap".
- **AC-B3.9** Missing provider env → send loops skip (200, no throw), `EMAIL_NOT_CONFIGURED` once; **maintenance still runs** — with a stale heartbeat + missing provider, both `SYNC_STALLED` and `EMAIL_NOT_CONFIGURED` are raised in-app and zero sends attempted; `EMAIL_NOT_CONFIGURED` resolves once configured. **Site-origin parity:** a missing/blank/malformed canonical site origin (§5.4) with Resend secrets PRESENT ALSO raises `EMAIL_NOT_CONFIGURED` and blocks all sends (no dead-link emails); resolves once a valid absolute origin is configured. **Both-toggles-off suppression:** with config invalid AND `alert_on_sync_problems=false` AND `daily_review_digest=false`, NO `EMAIL_NOT_CONFIGURED` alert is raised (and an existing one resolves) — the toggles silence their own config warning; raised again only if a tier is re-enabled while config stays invalid.
- **AC-B3.9b (no unresolved placeholders in email)** A `SHEET_UNAVAILABLE` (or `PARSE_ERROR_LAST_GOOD`) `admin_alerts` row whose `context` LACKS `sheet_name` (or has it null — e.g. a legacy/manual-sync row) emails copy with the placeholder RESOLVED to the JOINed `shows.title`, never the literal `_<sheet-name>_`. The renderer's final guard rejects any subject/body still containing a `<…>` token. (Tests: a row with null `context.sheet_name` → body shows the title, asserts no `<` literal; a hypothetical unresolvable row → caught by the guard, not emailed with a placeholder.)
- **AC-B3.9a (absolute links — all tiers)** Every rendered email href is ABSOLUTE (`http(s)://host/…`), for ALL five link surfaces: show-level realtime, global-stall realtime, pending-ingestion realtime, digest body, and digest `"+N"/"+M"` overflow links. (Test: render each tier with a fixture origin and assert NO link starts with `/`; assert the localhost dev fallback is rejected in the email path. Derive the expected prefix from the injected origin, not a hardcoded literal — anti-tautology.)
- **AC-B3.10** Each toggle: write flips `app_settings`; fail-closed read reflects it; the gated effect changes — full flag-lifecycle, no zombie cell.
- **AC-B3.11** `email_deliveries` rejects `INSERT/UPDATE/DELETE` **AND `SELECT`** from `authenticated`/`anon` (fully service-role-only, RLS deny-by-default); the lockdown meta-test pins both the DML and the SELECT-denied posture (no read-leak of recipient emails / audit data).
- **AC-B3.11a** `email_deliveries.recipient` rejects non-canonical (`'Doug@X'`, spaces, empty) via the CHECK; the email-canonicalization schema-CHECK audit covers it.
- **AC-B3.12** No raw error code in any email body or toggle copy (invariant 5); no em dash in new copy (DESIGN.md §9).
- **AC-B3.12a (HTML escaping)** A show title / `drive_file_name` / context value containing `<script>` / `"><img onerror>` / a deceptive link renders escaped in the email `html` (no injected tags).
- **AC-B3.12b (safe ingestion copy)** A pending-ingestion email/digest item whose `last_error_code` is unknown / has null `dougFacing` / is crew-only / carries an unresolved `<sheet-name>` placeholder renders the generic `SHEET_PROCESS_FAILED` copy (with `sheet_name` filled), NEVER a raw code or `<…>` placeholder, and does NOT crash the tick. (Tests: unknown code, null/crew-only code, unresolved-placeholder code → safe fallback copy.)
- **AC-B3.13** The cron route 401s a request without `Authorization: Bearer $CRON_SECRET` before any work.
- **AC-B3.14 (cron-context recipients)** The route invoked with `CRON_SECRET` + NO cookies + seeded active admins finds them and sends. (Negative-regression: route `activeRecipients()` through the RLS-bound `listAdminEmails` → zero recipients under cron.)
- **AC-B3.15 (revoked recipient — best-effort)** A recipient revoked BEFORE the notify tick's recipient re-check is not emailed (the re-check close to the provider call → skip). **Residual (accepted, §13):** a revoke committing in the sub-second window between the re-check and Resend accepting MAY deliver one final email — perfect atomicity would need a per-send lock coordinated with the revoke action, deliberately out of scope (RD-9). (Test: revoke before the tick → no email. The narrow check-to-send race is documented, not asserted-against.)

## 13. Watchpoints / do-not-relitigate
- **At-least-once + the Resend Idempotency-Key is the chosen delivery contract (RD-9, owner-approved).** A rare duplicate after a process crash beyond the 24h replay window is **accepted**, not a defect. Do NOT re-introduce the claim-before-send / session-lock / terminal-state / submission-state / persistent-outbox window-expiry re-keying machinery — that was explicitly trimmed (B3-DEF-9). (This is distinct from — and does NOT trim — the in-scope single-attempt provider reissue key `fxav:<kind>:<sha256(…:r:<nonce>)>` on the 409-invalid path, which is REQUIRED B3 work per §3.2/§5.1/§6 and B3-DEF-9's NOTE.)
- **Accepted residual edges (all benign for a single-operator notifier):** a crash mid-send may rarely duplicate one email (idem-key covers the 24h common case); a >1h-stuck sheet recovering the same minute may get one heads-up; a digest retried after a crash may show slightly-stale content. None cause spam, silent misses, or privacy leaks.
- **No per-show advisory lock in the notify job** (§4.4 step 5) — it mutates only `admin_alerts` (idempotent upsert) + `email_deliveries`.
- **`SYNC_STALLED` is a show-wide cron-heartbeat alert, NOT per-show `last_synced_at`** (§4.2) — `last_synced_at` bumps on failures + skips quiet shows; per-show failures are covered by their error alerts.
- **`activeRecipients()` uses the service-role client, NOT `listAdminEmails`** (§5.1) — the RLS-bound reader returns zero rows under cron. Recipient-active re-check is on the send path (privacy: never email a revoked admin).
- **The 409-invalid reissue key uses a PER-SUBMISSION NONCE, never the stored `attempt_count`** (§3.2) — an `attempt_count`-derived key is re-derived identically after a crash-before-record (count unadvanced) and, if the payload changed, 409-invalids forever → a STUCK/MISSED notification (violates "deliver, not miss"). A fresh nonce per submission guarantees a never-poisoned key → always delivers; the only residual is an RD-9-accepted duplicate. `attempt_count` is solely the retry-cap counter. Do NOT relitigate the nonce as "non-determinism to remove" — determinism here re-creates the stuck case.
- **The email renderer resolves EVERY `<…>` placeholder; none ever ships** (§8) — `messageFor()` leaves a placeholder intact on a null/absent param, and B3 emails EXISTING `admin_alerts` `dougFacing` copy (incl. legacy/manual-sync `SHEET_UNAVAILABLE` rows with null `context.sheet_name`). Resolve `<sheet-name>` from `context.sheet_name` then fall back to the JOINed `shows.title`; new producers supply non-null `sheet_name`; a final guard rejects any residual `<…>` token before send (AC-B3.9b). Invariant-5 spirit: never a raw placeholder in user-visible output.
- **`EMAIL_NOT_CONFIGURED` copy names ALL THREE blocking inputs** (§8) — provider key, sending address, AND canonical site origin; the predicate is the conjunction, so naming only the provider key misdirects Doug when the origin is the fault.
- **EVERY email href is ABSOLUTE via a validated canonical site origin** (§5.4) — relative `/admin` links are unresolvable in email clients. ALL five tiers (show-level/global/ingestion realtime, digest body, digest overflow) build `${origin}<path>`; `origin` comes from `NEXT_PUBLIC_SITE_ORIGIN` validated as an absolute URL — NEVER the `http://localhost:3000` dev fallback (`validateNextParam.ts:21`). A missing/invalid origin is an `EMAIL_NOT_CONFIGURED` config fault that BLOCKS sends (config predicate = Resend secrets AND valid origin). Do NOT ship any tier with a relative link.
- **ONLY a Resend `200` records `sent`** (§3.2/§5.1) — a `409 invalid_idempotent_request` is NOT proof of delivery (a failed/5xx request can reserve the key without delivering), so it is NEVER recorded `sent` on its own. It is ALSO NOT a terminal failure: the caller (ALL kinds) REISSUES with the bounded recipient-scoped attempt-scoped key (§3.2 — `fxav:<kind>:<sha256(kind||':'||dedup_key||':'||recipient||':r:'||<nonce>)>`, NOT a raw `<dedup_key>:r<attempt>` concat that would drop `recipient`/`kind` and risk the 256-char cap) and records `sent` ONLY on the reissue's `200` (delivers; accepts a rare duplicate, RD-9). A row is recorded `failed` only after the reissue itself fails (`infra_error`). `409 concurrent_idempotent_requests` (in-flight) → `retry_later` (not a failure). Do not record a `409 invalid` as `sent` (no real `200`) NOR as a terminal `failed` (that would miss the email).
- **Idempotency handling is UNIFORM across kinds; payloads are NOT assumed byte-stable** (§3.2): `upsert_admin_alert` overwrites `context` on re-occurrence and digest content is point-in-time, so ANY retry may render different bytes. `200` → `sent`; `409 concurrent` → `retry_later`; `409 invalid` (changed payload) → reissue with the bounded recipient-scoped attempt-scoped key (§3.2 — `fxav:<kind>:<sha256(…:r:<nonce>)>`) → DELIVER (accept a rare duplicate, RD-9 — never a silent miss, never a `sent` without a real `200`); `infra_error` → failed/retry. The ledger `dedup_key` stays the base. Do NOT re-add a per-kind split, a payload-stability mandate, or a pre-send snapshot/claim.
- **Provider Idempotency-Key is a BOUNDED hash including `recipient`** (§3.2) — under Resend's 256-char cap; `recipient` must be in the key or the provider would dedup-away the 2nd admin.
- **The notify-side recovery-resolve is an ATOMIC CONDITIONAL update (re-reads `shows` in the `UPDATE`), NOT a read-then-resolve** (§4.3) — else a stale "recovered" read could clobber a current alert that a concurrent sync just re-set. This closes the TOCTOU without the notify job taking the per-show lock (the sync path is the primary resolver, under its lock).
- **Pending-ingestion email/digest copy goes through the shared safe resolver + `SHEET_PROCESS_FAILED` fallback** (§4.5/§5.3) — `last_error_code` is unconstrained text; a raw `messageFor` lookup leaks a raw code/placeholder (invariant 5) or crashes the tick.
- **Recovery-resolution maps `last_sync_status`→catalog code, NOT raw `last_sync_error` equality** (§4.3) — `last_sync_error` holds raw causes (`SYNC_FILE_FAILED`), so a raw equality check would falsely resolve/skip every real alert; use the health-layer status→code map (`driveConnectionHealth.ts`). The map keys on `last_sync_status` ALONE — **no status disambiguates on `last_sync_error`.** In particular `sheet_unavailable`→`SHEET_UNAVAILABLE` for BOTH raw causes (`SHEET_UNAVAILABLE` AND `STAGED_PARSE_SOURCE_GONE`); the live producer emits only the `SHEET_UNAVAILABLE` alert for this status (`runScheduledCronSync.ts:1869-1872`), and `STAGED_PARSE_SOURCE_GONE` is NOT a show-level alert/email code (it's a raw cause; §4.1 excludes `STAGED_PARSE_*` from `SYNC_PROBLEM_CODES`). Branching the map on `last_sync_error` invents a phantom code → falsely resolves the open `SHEET_UNAVAILABLE` alert → a moved/unshared sheet misses its email. Per-code (resolve on code-switch too), not just "became OK". Without it, cleared/switched/raw-cause errors email stale + recurrences coalesce into the wrong row. `EMAIL_DELIVERY_FAILED`/`EMAIL_NOT_CONFIGURED` resolve symmetrically (§6).
- **The digest body is CAPPED** (§5.3) — `DIGEST_MAX_SHOWS`/`DIGEST_MAX_ITEMS_PER_SHOW` with "+N/+M more → /admin" overflow; counts from the SOURCE totals (anti-tautology), not the truncated render list. Prevents an unbounded body (provider failure) or silent omission.
- **Digest realtime-exclusion is `pending_ingestion`-ONLY** (§5.3, RD-7) — that is the only inbox item type the realtime tier emails; `first_seen`/`existing_staged` (pending_syncs) items have NO realtime counterpart and are ALWAYS included with no `admin_alert` join (don't invent a phantom alert join or drop them). The digest query selects `pending_ingestions.first_seen_at` (the dashboard default omits it) to match the ingestion occurrence key.
- **`SHEET_PROCESS_FAILED` is the pending-ingestion fallback, NOT a show-level `SYNC_PROBLEM_CODES` member** (§4.1/§4.5) — the show-level set contains only codes with real show/admin_alert producers (a constants meta-test asserts this).
- **Two of three `SYNC_PROBLEM_CODES` have NO producer today — B3 ADDS them; this is in-scope, not deferred** (§4.1, AC-B3.1c). Live code (`upsertAdminAlert.ts:3-29` union; only `SHEET_UNAVAILABLE` producers at `runScheduledCronSync.ts:1819,1872`) writes no `admin_alerts` row for `DRIVE_FETCH_FAILED`/`PARSE_ERROR_LAST_GOOD` — they're §12.4 catalog codes (`catalog.ts:54,93`) used only for admin-health DISPLAY (`driveConnectionHealth.ts:66,70`). Adding the two producers REALIZES RD-5's already-ratified three-class intent; the consumer/aging design (consume `admin_alerts`, age on `raised_at`) is unchanged and correct (direct-from-`shows` can't age — `last_synced_at` bumps every tick). Do NOT relitigate this as a scope expansion or propose switching the detection source away from `admin_alerts`; the `lib/notify/constants` meta-test (fails-until-added) is the structural gate.
- **Maintenance always runs; only DELIVERY is gated** (§4.4 step 0, §6) — `SYNC_STALLED` detect/resolve must not be blinded by a toggle-off / unconfigured provider.
- **Digest + realtime-ingestion both apply `wizard_session_id IS NULL`** (§4.5, §5.3); digest content is built fresh from the live inbox via the service-role client.
- **The notify cron jobs use `net.http_get`, not POST** (§5.3, RD-3) — the x6 pg-cron-coverage guard asserts GET; the route reads `?job=` from the query string and the Vault `CRON_SECRET` from the Authorization-bearer header. Adding the two canonical `fxav_cron_notify_*` jobs to x6 coverage is mandatory.
- **`EMAIL_DELIVERY_FAILED` is a single POSITIVE-predicate lifecycle** (§6) — OPEN while ≥1 `failed` row is CURRENT, RESOLVE when none is, independent of `attempt_count` (NOT "only at cap"; emitted on first failure, kept open through retries). CURRENT ⟺ recipient active AND **tier toggle ON** AND occurrence live — CURRENT must match the SAME eligibility gate as the send loop, so a row whose tier is now OFF (or whose digest backlog cleared) resolves. ONE OCCURRENCE-EXACT branch per realtime kind incl. global stall (the live source's `raised_at`/`first_seen_at` epoch MUST equal the failed row's `dedup_key` epoch — an old failed row must NOT re-become current when a fresh occurrence opens; digest = today's ET date AND ≥1 open item; show-level/global = unresolved alert of the SAME `raised_at` + show published∧¬archived; pending = row of the SAME `first_seen_at` + wizard NULL). Phrase it positively (the live conditions), not as the resolved/non-current cases, to avoid inverting it; cover every candidate kind (don't drop global-stall). No terminal-state column (RD-9 trim).
- **`EMAIL_NOT_CONFIGURED` is gated on at least one tier being ON** (§6/§4.4 step 1) — OPEN iff `config invalid AND (alert_on_sync_problems OR daily_review_digest)`; RESOLVE on valid config OR both toggles OFF. If Doug disables both email tiers, the config warning must go silent (no send is intended) — the toggles silence their own warning. Do NOT raise it unconditionally on missing config.
- **Revoked-recipient protection is BEST-EFFORT** (§4.4 step 4, AC-B3.15) — the re-check close to the send closes the common case; a revoke in the sub-second check-to-send window MAY deliver one final email. This is an accepted residual (RD-9), not a defect to fix with a per-send lock.
- **The three §12.4 codes are full predicate rows** (§8) — `dougFacing`/`crewFacing`/`followUp`/`helpfulContext`/`title`/`longExplanation`/`helpHref`, agreeing across the three-lockstep layers (x1).
- **The 2026-05-09 notification memo is principles-only** (§1.3) — its signed-token/feedback machinery is superseded by the crew-auth pivot + B2.
- **Both toggles default ON; digest hourly-fire + ET-hour window is the DST-safe design** (RD-8, §5.3).
