## 1. Scope and boundaries

### 1.1 Problem
The "Alert me about sync problems" Preferences toggle (admin-redesign prototype) and the *Daily review digest* push are both **operator attention surfaces**: Doug's natural surface is Drive, not the admin dashboard, so a staged event or a stalled sync that lives only in-app is functionally invisible (notification-design-memo ¬ß"The problem this solves"). The in-app `admin_alerts` surface (AlertBanner / NotifBell) is the *system of record* (pull); B3 adds the *push* layer (email) over it. Per **AGENTS.md flag-lifecycle / no-zombie-flag** rule, neither toggle may ship without its working backend ‚Äî hence the toggles ship *with* the email subsystem in this milestone.

### 1.2 In scope
- First outbound-email path in the app: `lib/notify/` channel-agnostic seam (email-only impl) + Resend integration + secrets.
- Real-time sync-problem email tier (RD-5/RD-6): consume `admin_alerts` + staleness detector (¬ß4.2) + stuck-ingestion detector (¬ß4.5), delivered via an atomic claim-before-send ledger (¬ß3.2).
- Daily review-digest email tier (RD-7), DST-safe scheduling.
- `email_deliveries` dedup/audit ledger.
- Two new `app_settings` boolean columns + read paths + write server-actions + the two Preferences toggles (UI).
- Three new ¬ß12.4 codes (`SYNC_STALLED`, `EMAIL_DELIVERY_FAILED`, `EMAIL_NOT_CONFIGURED`).
- New `app/api/cron/notify/route.ts` + two pg_cron schedules.
- TDD tests + meta-test registrations.

### 1.3 Out of scope (deferred ‚Äî see ¬ß12)
Two-way feedback, inbound webhook, apply-from-email, unpublish tokens, SMS impl, per-admin prefs, digest‚Üírealtime promotion, crew-facing notifications, multi-operator routing. The 2026-05-09 `notification-design-memo.md` is cited for **principles only** (push-not-pull; severity tiering; quiet-success; coalescing). Its signed-token/feedback machinery is superseded by the 2026-05-23 crew-auth pivot and the B2 lifecycle redesign and is **not** built here.

```sql
alter table public.app_settings

```
- Idempotent (`add column if not exists`), apply-twice safe.
- `app_settings` is a singleton (`id='default'`, `app_settings_singleton` CHECK at `supabase/migrations/20260501001000_internal_and_admin.sql:246`) gated by the `admin_only` RLS policy (`20260501002000_rls_policies.sql:131-134`). Writes are direct `.update()` under that policy ‚Äî **mirrors the B2 `auto_publish_clean_first_seen` pattern** (`app/admin/settings/_actions/setAutoPublish.ts:46`), **not** an RPC. `app_settings` is therefore *not* in the PostgREST-DML-lockdown RPC registry; its write protection is the `admin_only` RLS policy (declare `// not-subject-to-meta: app_settings writes are RLS admin_only-gated, not RPC-gated` where the lockdown registry is touched).

### 3.2 `email_deliveries` ledger (new table, B3 migration)
```sql
create table public.email_deliveries (
  id                  uuid primary key default gen_random_uuid(),
  kind                text not null check (kind in ('realtime_problem','digest')),
  channel             text not null default 'email' check (channel in ('email','sms','webhook')),
  dedup_key           text not null,        -- see dedup-key shapes below
  show_id             uuid references public.shows(id) on delete set null,
  recipient           text not null,
  triggered_codes     text[] not null default '{}',
  status              text not null default 'claimed' check (status in ('claimed','sent','failed')),
  provider_message_id text,
  error               text,
  attempt_count       int  not null default 1,
  source_raised_at    timestamptz,          -- realtime: admin_alert.raised_at OR pending_ingestions.first_seen_at this delivery covers
  claimed_at          timestamptz not null default now(),
  created_at          timestamptz not null default now(),
  sent_at             timestamptz,
  constraint email_deliveries_recipient_email_canonical
    check (recipient = lower(trim(recipient)) and recipient <> '')
);
-- ONE delivery per logical message per recipient, REGARDLESS of status ‚Äî this full
-- (not partial) unique index is the atomic-claim guard: a 'claimed' row blocks a
-- concurrent worker from sending the same message again.
create unique index email_deliveries_dedup
  on public.email_deliveries (kind, dedup_key, recipient);
```
- **DML lockdown:** REVOKE insert/update/delete from `anon`, `authenticated`; grant to `service_role` only; the cron route writes with the service-role client. Add an explicit service-role-only entry in `tests/db/postgrest-dml-lockdown.test.ts` so the lockdown class is pinned (no `authenticated` DML path exists).
- **Recipient canonicalization (invariant 3):** the schema CHECK above is the safety net; the writer ALSO canonicalizes via `lib/email/canonicalize.ts` before insert (recipients already arrive canonical from `listAdminEmails`, but the notify boundary re-canonicalizes defense-in-depth). The canonical recipient is part of the row identity, so a non-canonical slip would otherwise bypass dedup and duplicate-send.
- **Dedup-key shapes (single source of truth):**
  - *Realtime, show-level:* `'<show_id>:<code>:<floor(epoch(raised_at))>'` ‚Äî `raised_at` from the `admin_alerts` row. `upsert_admin_alert` preserves `raised_at` across re-occurrence; only a *new* unresolved row (after the prior resolved) gets a fresh `raised_at`, so the epoch changes exactly when the problem genuinely recurs ‚Üí re-arm.
  - *Realtime, pending-ingestion (pre-show, ¬ß4.5):* `'ingestion:<drive_file_id>:<floor(epoch(first_seen_at))>'`, `show_id = NULL`. Pre-show rows cannot use the `(show_id,code)` shape; `first_seen_at` re-arms when a later ingestion of the same file fails afresh.
  - *Digest:* `'digest:' || to_char((now() at time zone 'America/New_York')::date, 'YYYY-MM-DD')` ‚Üí one digest per ET calendar day per recipient.
- **Atomic claim-before-send (concurrency-safe ‚Äî closes the duplicate-send race):** the full unique index makes the ledger the lock. Per candidate `(kind, dedup_key, recipient)` the worker runs ONE atomic upsert that wins for a brand-new message OR a retryable `failed` row, and no-ops for an already-`claimed`/`sent`/cap-reached row:
  ```sql
  insert into public.email_deliveries
    (kind, dedup_key, recipient, show_id, triggered_codes, source_raised_at, status)
  values ($1,$2,$3,$4,$5,$6,'claimed')
  on conflict (kind, dedup_key, recipient) do update
    set status='claimed',
        attempt_count = public.email_deliveries.attempt_count + 1,
        claimed_at = now()
    where public.email_deliveries.status='failed'
      and public.email_deliveries.attempt_count < <SEND_RETRY_CAP>
  returning id, attempt_count;
  ```
  Only the worker that gets a returned row sends ‚Äî `lib/notify.send()` ‚Üí Resend with the provider **Idempotency-Key = the full per-recipient ledger identity `kind || ':' || dedup_key || ':' || recipient`** (a stable hash of those three fields) ‚Äî then `update`s its row to `sent` (+`provider_message_id`,`sent_at`) or `failed` (+`error`). The key MUST include `recipient`: `dedup_key` alone is shared by every recipient of the same logical message, so a recipient-less key would let Resend suppress all but the first admin while their ledger rows still flip to `sent` (a lying audit + silently-skipped admins). A worker that gets no row skips (another worker owns it, or it is already sent / cap-reached). The send happens strictly **after** the claim, never before. Two concurrent ticks: one wins the INSERT, the other's conflict-update WHERE is false (row is `claimed`, not `failed`) ‚Üí 0 rows ‚Üí no send. Exactly-once.
- **Stale-claim recovery (explicit, pinned as each job's FIRST step):** every notify job (realtime AND digest), before any candidate work, runs:
  ```sql
  update public.email_deliveries
    set status='failed', error='stale_claim'
    where status='claimed' and claimed_at < now() - interval '<CLAIM_STALE_MS> milliseconds';
  ```
  This frees a row where a worker crashed after claiming but before `sent`/`failed` ‚Äî otherwise the full unique index + the "claim only re-acquires `failed` rows" rule would block that `(kind,dedup_key,recipient)` message forever (silent permanent suppression). A FRESH `claimed` row (recent `claimed_at`) is NOT swept and correctly stays blocked to concurrent workers. Because every send passes the per-recipient Idempotency-Key (above), any resend of a recovered row is a provider-side no-op ‚Üí no duplicate leaves the system even across a crash.
- **Guard conditions:** `show_id` nullable (digest + pending-ingestion rows carry NULL); `triggered_codes` defaults `{}` (never NULL); `provider_message_id`/`error` nullable by status; `recipient` NOT NULL + canonical CHECK.

## 4. Detection (real-time tier)

### 4.1 `SYNC_PROBLEM_CODES` set (single source of truth ‚Äî `lib/notify/constants.ts`)
`['DRIVE_FETCH_FAILED','SHEET_PROCESS_FAILED','SHEET_UNAVAILABLE','PARSE_ERROR_LAST_GOOD','SYNC_STALLED']`. `STAGED_PARSE_*` race codes and all info-severity codes are **excluded** (dashboard-only). This array is referenced by both the detection query and the tests ‚Äî never re-listed inline. NOTE: this set governs only the `admin_alerts`-consumption path (¬ß4.3/¬ß4.4 step 2); the pre-show pending-ingestion path (¬ß4.5) is detected directly and is NOT keyed by this set.

### 4.2 Staleness detector (the only new detection logic)
- Runs first in `?job=realtime`. For each show that is `published = true` AND `archived = false` (watched, expected to sync) AND `now() - coalesce(last_synced_at, '-infinity') > STALENESS_THRESHOLD_MS`: `upsert_admin_alert(show_id, 'SYNC_STALLED', jsonb_build_object('last_synced_at', last_synced_at))`.
- **Resolve path:** when a show that previously had an unresolved `SYNC_STALLED` row syncs successfully (`now() - last_synced_at <= threshold`), set `resolved_at = now()` on that row (so it both clears the in-app banner and re-arms the email dedup for a future stall).
- **Guard conditions:** `last_synced_at IS NULL` (never synced) on a published show counts as stalled (coalesce to `-infinity`). A show flips `published=false` or `archived=true` ‚Üí no longer evaluated; its open `SYNC_STALLED` is resolved on the same tick.

### 4.3 Gap-fill audit (implementation task, not a runtime path)
Confirm each non-staleness condition in `SYNC_PROBLEM_CODES` already upserts an `admin_alerts` row on the sync/ingestion paths (call sites: `lib/sync/runScheduledCronSync.ts:745`, `lib/sync/applyStaged.ts:1642`, `lib/sync/promoteSnapshot.ts:132`, `lib/sync/runOnboardingScan.ts:455`, `lib/adminAlerts/upsertAdminAlert.ts:39`, plus push-path). Where a condition does not produce an alert today, add the `upsert_admin_alert` call **in B3** so the email tier never re-detects. The audit's findings + any added call sites are documented in the plan; the email tier reads only `admin_alerts`.

### 4.4 Delivery loop (realtime)
Candidates come from THREE sources: ¬ß4.2 staleness (‚Üí `admin_alerts.SYNC_STALLED`), ¬ß4.3 consumed error alerts (other `SYNC_PROBLEM_CODES`), and ¬ß4.5 pending-ingestion (read directly).
0. **Stale-claim sweep (FIRST):** run the ¬ß3.2 recovery `update` (`status='claimed' AND claimed_at < now()-CLAIM_STALE_MS ‚Üí 'failed'`) before any candidate selection.
1. Gate: read `alert_on_sync_problems` fail-closed (`lib/appSettings/getAlertOnSyncProblems.ts`, mirroring `getAutoPublishCleanFirstSeen.ts`). False / read-error ‚Üí no realtime emails this tick.
2. **Show-level candidates:** select unresolved `admin_alerts` JOIN `shows` ON `admin_alerts.show_id = shows.id` where `code = any(SYNC_PROBLEM_CODES)` AND `raised_at <= now() - interval '1 hour'` AND `shows.published = true` AND `shows.archived = false`. The `published ‚àß ¬¨archived` filter is REQUIRED: an alert raised before a show is archived/unpublished stays unresolved, and emailing sync problems for an inactive show contradicts the silent-skip posture of archived sync paths (RD-5 scopes realtime to active shows). `show_id IS NULL` alerts are out of scope for this path. Candidate key `'<show_id>:<code>:<epoch(raised_at)>'`, `source_raised_at = raised_at`.
3. **Pending-ingestion candidates (¬ß4.5):** select `pending_ingestions` where `now() - first_seen_at > interval '1 hour'`; candidate key `'ingestion:<drive_file_id>:<epoch(first_seen_at)>'`, `show_id = NULL`, `triggered_codes = ARRAY[last_error_code]`, `source_raised_at = first_seen_at`.
4. For each candidate √ó each active recipient: run the **atomic claim** (¬ß3.2). Only the claim winner calls `lib/notify.send()` ‚Üí Resend (per-recipient Idempotency-Key = `kind:dedup_key:recipient`, ¬ß3.2), then updates its row `sent` / `failed`. On failure emit `EMAIL_DELIVERY_FAILED` (¬ß6). No claim row returned ‚Üí skip.
5. **Advisory-lock posture:** the show-level (step 2) and staleness paths mutate only `admin_alerts` (idempotent SECURITY DEFINER upsert) + `email_deliveries`, and READ `shows`; they do not MUTATE `shows`/`crew_members`/`crew_member_auth`/`pending_syncs`/`pending_ingestions`, so per **invariant 2** they hold **no** per-show advisory lock (code carries `// not-subject-to-meta: notify job mutates only admin_alerts + email_deliveries; no invariant-2 table mutation`). **EXCEPTION:** the pending-ingestion finalization (¬ß4.5) DOES acquire `pg_try_advisory_xact_lock(hashtext('show:'||drive_file_id))` to close the read-to-send delete race ‚Äî a separate top-level transaction acquiring the same hashkey the sync path uses (mutual exclusion, not nested), documented in ¬ß4.5's advisory-lock topology note + ¬ß10 + ¬ß13.

### 4.5 Stuck-ingestion detector (pre-show ‚Äî second documented exception to RD-6)
Pending ingestions are pre-show (no `show_id`), so an `admin_alerts` upsert (`coalesce(show_id::text,''), code` conflict key) would collapse every stuck file of the same `code` into ONE row and cannot dedup per-file. Therefore this class is detected by reading `pending_ingestions` **directly** and dedup'd in `email_deliveries` by `drive_file_id` ‚Äî NOT routed through `admin_alerts`. This is the second deliberate exception to "consume admin_alerts" (the first, staleness ¬ß4.2, conversely ADDS to admin_alerts). Both exceptions are enumerated in ¬ß13 do-not-relitigate.
- **Select:** `pending_ingestions` rows older than `STALENESS_THRESHOLD_MS` (`now() - first_seen_at > 1h`). A row's mere presence means the sheet has not become a show; a successful re-ingestion deletes the row ‚Üí no email (quiet-success). Sub-1h transients send nothing.
- **Dedup / re-arm:** `'ingestion:<drive_file_id>:<floor(epoch(first_seen_at))>'`, `show_id = NULL`. Re-arms only if a later ingestion of the same `drive_file_id` fails afresh (new `first_seen_at`).
- **Email:** body renders the row's `last_error_code` `dougFacing` (e.g. `SHEET_PROCESS_FAILED`) + `drive_file_name`; link ‚Üí `${NEXT_PUBLIC_SITE_ORIGIN}/admin` (no show slug exists yet). Gated by the same `alert_on_sync_problems` toggle.
- **Race-safe finalization (read-to-send guard):** the sync/recovery path DELETEs a successful `pending_ingestions` row while holding `pg_advisory_xact_lock(hashtext('show:'||drive_file_id))` (e.g. `lib/sync/runScheduledCronSync.ts:604-610`). A lock-less notify tick could read an aged row and send the *irreversible* "stuck" email just as the recovery commits the delete ‚Äî a false email violating quiet-success. So the ingestion finalization, per candidate, acquires `pg_try_advisory_xact_lock(hashtext('show:'||drive_file_id))` (non-blocking: if held ‚Üí skip this file this tick, retry next), RE-reads the row (still present AND aged?), and only then runs the atomic claim + Resend send + ledger update **within the held lock** (released at transaction end, after the send). This serializes against the recovery delete. It is a deliberate, scoped exception to ¬ß4.4 step 5's no-lock posture.
  - **Lock-holding mechanism (PINNED ‚Äî must not use the PostgREST client):** this critical section runs on a dedicated **postgres.js** connection in ONE transaction: `begin; select pg_try_advisory_xact_lock(hashtext('show:'||drive_file_id)); -- re-read row present+aged; -- atomic claim (¬ß3.2); -- await Resend send; -- update ledger sent/failed; commit;`. It must NOT use the service-role **PostgREST/Supabase** client, whose per-call autocommit releases the xact-scoped lock before the send and reopens the race. A bounded `set local statement_timeout` plus a bounded Resend client timeout cap how long the lock + connection are held; holding one transaction across a single external send is an accepted, bounded trade-off for this low-frequency per-file path (‚â§15 shows, hourly). (postgres.js is the project's transactional DB layer ‚Äî same as B2's `_b2Helpers`/`tx.unsafe` paths.)
  - **Advisory-lock topology (invariant 2, single-holder):** the existing holder of `hashtext('show:'||drive_file_id)` is the sync/recovery path (cron `pg_try_advisory_xact_lock`; admin/blocking `pg_advisory_xact_lock`). The notify ingestion finalization acquires the SAME hashkey as a SEPARATE top-level transaction at the route/JS layer ‚Äî mutual exclusion across independent transactions, NOT a nested second acquisition within one call stack, so the single-holder/no-nested-deadlock rule holds. Enumerated in `tests/auth/advisoryLockRpcDeadlock.test.ts` (¬ß10). The show-level (¬ß4.4 step 2) and staleness (¬ß4.2) paths stay lock-free: their underlying state changes non-destructively (a just-resolved 1h+ alert emailing once is a mild, defensible over-notify, not a false success).
- **Guard:** `last_error_code` is NOT NULL (schema); presence + age is the trigger, `attempt_count` informational. `wizard_session_id` rows mid-onboarding are still pre-show and eligible once aged.

## 6. Failure & unconfigured-provider handling

- **Send failure:** the claim winner (¬ß3.2) gets `{ ok:false, kind:'infra_error' }` from Resend ‚Üí `update` its claimed row to `status='failed'` (+`error`; `attempt_count` was already incremented at claim time); emit `upsert_admin_alert(show_id, 'EMAIL_DELIVERY_FAILED', ‚Ä¶)` (in-app only ‚Äî **never** email about email failing). A later tick re-acquires the row via the ¬ß3.2 atomic claim while `attempt_count < SEND_RETRY_CAP (3)`; at the cap the claim's `WHERE` predicate excludes it (no further send) and the in-app alert persists. Stale `claimed` rows (crash after claim) are recovered per ¬ß3.2 (flip to `failed` after `CLAIM_STALE_MS`; Idempotency-Key makes any resend a provider no-op). Quiet-success is preserved (no "still failing" emails).
- **Unconfigured provider:** missing `RESEND_API_KEY` or `EMAIL_FROM` ‚Üí the job no-ops gracefully (no throw, no crash of the cron tick), emits `upsert_admin_alert(NULL, 'EMAIL_NOT_CONFIGURED', ‚Ä¶)` **once** (deduped by the upsert conflict key), and returns 200 with a body indicating skipped. Toggles remain settable but inert. The settings UI MAY render a small "email not configured" hint when the env is absent (server-derived boolean prop; never leaks the secret).
- **Fail-closed reads:** both toggle reads fail-closed (a read error is treated as "off") so an infra hiccup never spams.

## 13. Watchpoints / disagreement-loop preempts (do-not-relitigate)

For the adversarial reviewer ‚Äî each contract below is intentional and cited; do not relitigate without a new owner decision:

- **No per-show advisory lock in the notify job** is correct (¬ß4.4): the job mutates only `admin_alerts` (idempotent SECURITY DEFINER upsert) + `email_deliveries`, neither in the invariant-2 protected set. Inline `// not-subject-to-meta` documents it.
- **Two deliberate detection exceptions to "consume admin_alerts"** (RD-6, ¬ß4.2, ¬ß4.5): staleness ADDS a `SYNC_STALLED` row to `admin_alerts`; pending-ingestion reads `pending_ingestions` DIRECTLY (because pre-show rows have no `show_id` and the `(show_id,code)` upsert cannot dedup per-file). Both are intentional, not gaps.
- **Atomic claim-before-send + full (non-partial) unique index** (¬ß3.2) is the correct duplicate-send defense, not over-engineering: the ledger is the concurrency lock; the provider Idempotency-Key covers the crash-after-send window. Do not "simplify" back to insert-after-send.
- **`activeRecipients()` MUST use the service-role client, NOT `listAdminEmails`** (¬ß5.1): the cron route has no admin session, so the cookie/RLS-bound reader returns zero rows and silently suppresses all email. Do not "reuse the existing helper."
- **Provider Idempotency-Key MUST include `recipient`** (`kind:dedup_key:recipient`, ¬ß3.2): `dedup_key` alone is shared across recipients and would let the provider drop all but the first admin while their ledger rows flip to `sent`.
- **Digest is per-recipient-claim-gated, NOT day-level-gated** (¬ß5.3): the route runs the loop across the ET-hour window and lets the per-recipient claim decide; a day-level "any digest row exists ‚Üí skip" guard would suppress other recipients + block same-day retry. Do not reintroduce it.
- **Realtime show-level candidates require `published ‚àß ¬¨archived`** (¬ß4.4 step 2): an alert raised before archive/unpublish stays unresolved, so the `shows` join is load-bearing to avoid emailing problems for inactive shows.
- **The pending-ingestion finalization advisory lock is correct, not an invariant-2 violation** (¬ß4.5): it is a deliberate, scoped exception to the notify job's otherwise lock-free posture, acquiring the sync path's hashkey as a SEPARATE top-level transaction (mutual exclusion, not nested) to close the irreversible-email read-to-send race. Do not "remove the lock for consistency with the other paths."
- **HTML escaping of interpolated values is mandatory** (¬ß5.1): show names / Drive file names / context are untrusted; every dynamic value into `html` is escaped. Do not treat sheet/admin data as trusted.
- **The ingestion finalization lock MUST run on postgres.js, not PostgREST** (¬ß4.5): the xact-scoped advisory lock has to stay held across the awaited Resend send, which requires one explicit transaction on a dedicated postgres.js connection. A PostgREST/autocommit implementation releases the lock before the send and silently reopens the race; the test pins this by pausing mid-send.
- **Stale-claim recovery is the first step of every notify job** (¬ß3.2/¬ß4.4 step 0/¬ß5.3): without it a crash-after-claim permanently silences a `(kind,dedup_key,recipient)` message. Do not omit or reorder it.
- **`app_settings` writes are RLS `admin_only`-gated, not RPC-gated** (¬ß3.1) ‚Äî this matches the ratified B2 `setAutoPublish` pattern (`app/admin/settings/_actions/setAutoPublish.ts:46`); B3 does not introduce an RPC for the toggles.
- **Email failures never trigger emails** (¬ß6) ‚Äî `EMAIL_DELIVERY_FAILED` / `EMAIL_NOT_CONFIGURED` are in-app only, by design (no feedback loop, no spam).
- **The 2026-05-09 notification memo is principles-only** (¬ß1.3) ‚Äî its signed-token/feedback/unpublish machinery is superseded by the crew-auth pivot + B2 and is explicitly deferred, not omitted by oversight.
- **Both toggles default ON** (RD-8a) ‚Äî owner decision; quiet-success keeps them silent when nothing is wrong.
- **Digest hourly-fire + ET-hour check is the DST-safe design** (¬ß5.3), not an inefficiency to "optimize" into a single UTC cron time.
- **Server actions under `app/` are Opus-owned** (¬ß11) per the UI hard rule, even though they touch `app_settings`.
