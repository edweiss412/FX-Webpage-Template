### 2.3 UserMenu (`app.jsx:25-65`)
Avatar button (initials) в†’ popover with: identity head (full name + email) + divider + **Sign out** item. 
- **Identity source + fault-boundary contract (load-bearing):** the UserMenu's identity MUST be obtained **inside the layout's existing `AdminInfraError` try/catch** (`app/admin/layout.tsx:47-71`), NOT via a second independent read during UserMenu render вЂ” otherwise a Supabase `getUser()`/RPC fault while rendering the menu falls into Next's generic error path instead of the cataloged admin-session-failure surface. **Mechanism:** the layout replaces its `requireAdmin()` call with a single **`requireAdminIdentity()`** (`lib/auth/requireAdmin.ts:89-150`; returns `{ email }`, email from the Supabase JWT canonicalized at `:134`; it gates identically AND yields the email) inside the same try/catch, and passes the resulting `email` down into the nav shell / UserMenu as a **prop**. **Scope of "one read":** this is the LAYOUT's identity read, for the nav-chrome/UserMenu only. In Next App Router a server layout CANNOT pass server-only props to its child page, so a page that needs identity for its OWN logic (notably the embedded Administrators self-revoke policy, В§3.2) does its OWN `requireAdminIdentity()` inside that page вЂ” covered by that segment's error boundary (В§2.7: `app/admin/settings/error.tsx`). That is NOT a forbidden "second fetch of the UserMenu's identity"; it is the page owning its own data (consistent with the В§2.6 page-owned-header contract). The rule being enforced is narrow: the UserMenu must not trigger an independent identity read OUTSIDE the layout catch. The prototype's "Doug Pemberton / Production Manager" is mock; **real data = the canonical admin email only** вЂ” render the email as the primary identity line; omit name/role (or derive a display name from the email local-part) rather than invent data. **Guard + correct-audience copy (load-bearing):** an `AdminInfraError` from `requireAdminIdentity()` renders the layout's cataloged error surface (`:51-67`). **But the current layout catch renders `messageFor(err.code)` and falls back to `entry.crewFacing` when `dougFacing` is null** вЂ” and the identity-fault code (`ADMIN_SESSION_LOOKUP_FAILED`) has `dougFacing: null` + a crew/show-misconfiguration `crewFacing`, so as-is it would show Doug *crew* copy on the admin shell. B1 changes the layout catch to render the **new `ADMIN_ROUTE_LOAD_FAILED`** (Doug-facing, В§8.1) for the identity-read fault вЂ” NOT `messageFor(err.code)`'s wrong-audience fallback. (This supersedes the R19 "layout can use err.code" note: server-side serialization is fine, but the *content* of that code is wrong-audience вЂ” so the layout uses the fixed Doug-facing code too, consistent with the client boundaries.) **Test:** a `getUser()`/identity infra fault renders the cataloged surface with the **actual `ADMIN_ROUTE_LOAD_FAILED` Doug-facing copy** (assert the copy text, not just the `admin-layout-infra-error` test id), never crew/show-misconfiguration copy, never Next's generic page.
- **Sign out** posts to `app/auth/sign-out/route.ts` (POST, `:89-127` вЂ” same-origin gated, calls `supabase.auth.signOut()`, clears cookies, 303 в†’ `/auth/sign-in`). The menu item submits a same-origin POST form (not a link вЂ” GET returns 405 at the route).

### 2.4 NotifBell (`app.jsx:16-23`)
Bell icon with a count badge. 
- **Count source:** number of unresolved, non-info `admin_alerts` вЂ” the SAME filter the AlertBanner uses (`.is("resolved_at", null)` + the `INFO_SEVERITY_CODES` exclusion derived from `MESSAGE_CATALOG`, `components/admin/AlertBanner.tsx:57-59,97-99`). The AlertBanner **already runs this exact `head:true` count inline** for its "+N more" chip (`AlertBanner.tsx:142-147`); B1 **extracts that count query into the shared helper** (В§8.1 row 1) that feeds BOTH the bell badge and the banner chip, so they cannot drift (not net-new query logic вЂ” a refactor-to-shared).
- **Click behavior:** navigates to `/admin#alerts` (the existing scroll anchor wrapping the AlertBanner, `layout.tsx:92`). On the dashboard this scrolls to the banner; from any other route it routes to `/admin` then to the anchor. No dropdown panel in B1 (YAGNI; the banner is the alert surface).
- **Guards (infra error MUST be visually distinct from zero вЂ” no silent failure):** the helper returns a typed result so the bell can distinguish three states: (a) **count = 0** в†’ bell renders with **no badge** (clean state; affordance always present); (b) **count > 0** в†’ numeric badge; (c) **infra_error** в†’ a **distinct degraded bell** (a small warn-tone indicator / "!" affordance, NOT a clean no-badge bell) that links to `/admin#alerts` вЂ” so a broken `admin_alerts`/RLS/schema path is *visible* exactly when operators need it, never disguised as "no unresolved alerts." Its accessible label + any visible copy resolve through a **NEW catalog code `ADMIN_ALERT_COUNT_FAILED`** вЂ” no existing code fits (`ADMIN_ALERT_NOT_FOUND` / `ALERT_REQUIRES_SHOW_SCOPED_RESOLVE` are unrelated), so B1 adds it via the В§12.4 three-lockstep (see В§8.1); copy via `lib/messages/lookup.ts`, no raw codes (invariant 5).
- **Banner-detail read must ALSO be fail-visible (the bell creates a NEW cross-read contradiction B1 must close):** the bell count is a `head:true` count, but the AlertBanner's topmost-alert DETAIL SELECT joins `shows(slug)` and currently returns **null on error** (`AlertBanner.tsx:61-78,125-128`). Different failure domains: a join/RLS/schema fault could make the count succeed (positive bell в†’ routes to `/admin#alerts`) while the banner shows **nothing** вЂ” hiding alert context exactly when degraded. So **B1 changes the AlertBanner**: when the detail SELECT errors, render a **cataloged degraded banner** (via `ADMIN_ALERT_COUNT_FAILED`, the same shared alert-summary surface) instead of `null`. A positive bell count must never route to an empty alert surface. (This reverses the earlier "AlertBanner unchanged" stance вЂ” the new bell makes the banner's silent-null a real contradiction; the no-silent-failure principle now applies to BOTH the bell count and the banner detail read.) **Test:** count succeeds but the banner detail SELECT throws/returns error в†’ degraded banner renders (not empty), surfaced via the cataloged code.
- **Cap:** badge displays the exact count up to 9; **10+ renders "9+"**.

### 2.6 Page header / breadcrumbs (`app.jsx:155-204`)

**Data-ownership contract (load-bearing вЂ” prevents re-fetch / stale-header bugs).** The prototype renders one global `PageHead` from a `HEADERS` map + `window.DATA` (`app.jsx:155-204`), but that only works because the prototype has global mock data. In the real app, per-show header data (the show row, `published`/`archived`, the share token, crew-link eligibility/URL) is **page-owned** вЂ” fetched inside `app/admin/show/[slug]/page.tsx`. Therefore:
- **Layout-owned** (`app/admin/layout.tsx`): the persistent nav chrome only (top bar / mobile bar / bottom tabs / UserMenu / NotifBell / dark toggle). Its only data needs are the admin identity (В§2.3) and the alert count (В§2.4) вЂ” neither requires per-show data.
- **Page-owned**: each page renders its own header via a shared presentational `<AdminPageHeader title sub crumb? backHref? rightSlot?>` component (new, `components/admin/nav/AdminPageHeader.tsx`) populated **from props the page already has** вЂ” no data fetching inside the header component, none in the layout. The per-show page passes its already-computed Published/Archived pill + crew-link chip into `rightSlot`; the dashboard/settings pages pass title/sub only. This guarantees the header and body render from the **same** show/token snapshot (no second query, no drift) and keeps the per-show pill/chip exactly where their data lives.
- The layout does **not** read the route to synthesize per-show headers; static routes (dashboard, settings) simply render `<AdminPageHeader>` with route-appropriate title/sub from the page.

Per-route header content (each rendered by its page via `<AdminPageHeader>`):

В§2.3.) **Tests:** a post-layout `requireAdmin*` `AdminInfraError` thrown from `/admin` and `/admin/settings` renders via `ADMIN_ROUTE_LOAD_FAILED`, and from `/admin/settings/admins` renders via its existing `ADMIN_EMAIL_LIST_FAILED` вЂ” each with **non-null Doug-facing copy** (not blank, not Next's generic page), exercised with a REAL server-thrown `AdminInfraError` (not a mocked client error object).

### 3.1 Drive connection (read-only health readout)
Panel with: folder icon + **folder name** + a status line + **Healthy/Warn pill** (right) + divider + helper text "Need to switch folders? Re-run setup. Your current shows keep syncing the whole time." + **[Open folder]** + **[Re-run setup]** buttons (`settings.jsx:31-52`).

**Status-line copy MUST derive from `health/reason`, NOT a static "Connected" prefix** (a static "Connected" would contradict the pill when the watch is missing/inactive/expired or a read faulted вЂ” the prototype's `Connected В· 5 shows syncing В· last read 8 min ago` at `settings.jsx:39` is the *positive*-state copy only). Mapping: **positive** в†’ "Connected В· {N} shows syncing В· last read {relative}"; **`not_configured`** в†’ "Connection not set up"; **`watch_inactive`/`watch_expired`** в†’ "Connection needs attention В· last read {relative}"; **`sync_*`/`stale_*`/`sync_unknown`** в†’ "Syncing, but {attentionCount} show(s) need attention В· last read {relative}". **`infra_error`** в†’ "Couldn't read sync status" (via the **new `ADMIN_DRIVE_HEALTH_UNAVAILABLE`** code, В§8.1 вЂ” cataloged, not a literal). The "{attentionCount}" is the count of shows matching the **winning reason's** predicate (the `attentionCount` field above) вЂ” NOT `syncingCount` (the total active count): one stale show among 501 active shows must read "1 show needs attention," never "501." The prefix never says "Connected" unless `health === 'positive'`. **Tests:** (a) the visible status-line copy per health-reason group (no "Connected" prefix on any Warn/infra state); (b) one stale/failing show with `activeCount` в‰« 1 в†’ "1 show needs attention" (not the total).

**Data sources & computation (all reads вЂ” B1 adds NO writes).** All panel reads are wrapped in a single typed-result helper **`fetchDriveConnectionHealth()`** (new, `lib/admin/driveConnectionHealth.ts`) returning a discriminated result that **carries the Warn reason + catalog code** (so the pill picks the right cataloged copy without re-querying or duplicating logic):
```
| { health: 'positive'; folderName: string|null; folderId: string|null; syncingCount: number; lastReadAt: string|null }
| { health: 'warn';
    reason: 'not_configured'|'watch_inactive'|'watch_expired'|'sync_drive_error'|'sync_sheet_unavailable'|'sync_parse_error'|'sync_unknown'|'stale_severe'|'stale_moderate';
    code: MessageCode; folderName: string|null; folderId: string|null; syncingCount: number; attentionCount: number; lastReadAt: string|null }
| { kind: 'infra_error' }
```
Any returned-error or thrown builder/client fault в†’ `infra_error` в†’ Warn pill + cataloged message via the **new `ADMIN_DRIVE_HEALTH_UNAVAILABLE`** code (never a crash, never a false "Healthy", never a literal). The `reason`в†’`code` mapping preserves the **status-specific** catalog codes (NOT a collapsed `DRIVE_FETCH_FAILED` вЂ” a moved sheet vs a parse failure vs a connectivity failure need different recovery guidance, matching `StaleFooter.tsx:57-59`): `not_configured`/`watch_inactive`/`watch_expired` в†’ `WATCH_CHANNEL_ORPHANED`; `sync_drive_error` в†’ `DRIVE_FETCH_FAILED`; `sync_sheet_unavailable` в†’ `SHEET_UNAVAILABLE`; `sync_parse_error` в†’ `PARSE_ERROR_LAST_GOOD`; `sync_unknown` в†’ **`SYNC_STATUS_UNKNOWN`** (new code, В§8.1 three-lockstep вЂ” the enum-drift branch needs cataloged fail-visible copy; settled, not "reuse-or-add-later"); `stale_severe` в†’ `SYNC_DELAYED_SEVERE`; `stale_moderate` в†’ `SYNC_DELAYED_MODERATE`. Every `reason` has a settled `code` (the union and this mapping are exhaustive вЂ” no branch can typecheck-fail or silently collapse). ## 4. Revoke-hang robustness fix

**Bug** (carry-in, RECONCILIATION В§B + ARCHITECT-UPDATE В§6): `RevokeRowButton.tsx` drives a `confirm в†’ resolving в†’ (reset)` UI state machine via `useActionState`. The "RevokingвЂ¦" в†’ reset only fires when the action **returns** a non-ok result (`:62-69` вЂ” the `ui === "resolving"` snap reads the returned result). If `revokeAdminAction` **throws** or the request **hangs**, no result returns, the snap never fires, and the button is stranded on "RevokingвЂ¦" indefinitely. (Env was confirmed NOT the cause вЂ” ARCHITECT-UPDATE В§6.)

**Fix вЂ” the recovery contract belongs at the SERVER-ACTION boundary, not a client-side try/catch** (a React server action submitted via `useActionState` + `<form action={...}>` routes a THROWN error to the nearest error boundary, NOT back to local component state вЂ” a client-side catch around the dispatch cannot intercept it). So:
1. **Typed-result, don't throw вЂ” but NARROWLY scoped (don't swallow auth/control-flow):** the catch wraps ONLY the `revokeAdminEmail(...)` data-layer call (which runs AFTER `requireAdminIdentity()` has already gated, `actions.ts`), catches ONLY `AdminEmailsInfraError`, and **returns a typed `{ kind: 'infra_error' }` outcome** (extend `AdminEmailActionResult`, `actions.ts:47-52`) instead of throwing. **MUST rethrow** (never convert to inline retry): `AdminInfraError` from the `requireAdminIdentity()` gate (a permission/session failure must propagate to the route error boundary, not look like a retryable revoke), Next navigation/control-flow throws (`redirect`/`notFound`/`forbidden` digests), and any unknown error. `useActionState` receives the typed `infra_error` and the button transitions `resolving в†’ error` with a plain-language retry message вЂ” no error-boundary teardown of the settings section. **Tests:** (a) a gate auth/session `AdminInfraError` still propagates to the boundary (NOT inline retry); (b) a `revokeAdminEmail` `AdminEmailsInfraError` в†’ `{ kind: 'infra_error' }` inline; (c) a Next control-flow throw propagates.
2. **Watchdog for no-response hangs в†’ a CONSERVATIVE "outcome unknown" state, NOT idle/retry:** a React server action cannot be cancelled by resetting local UI вЂ” a merely-slow revoke RPC may still commit AFTER a naive timeout fires. So if no result arrives within a bounded window, the button transitions to an explicit **"Couldn't confirm вЂ” refresh to check"** state that (a) does NOT return to a normal idle/retry state, (b) **suppresses duplicate submits** of the same revoke, and (c) prompts a refresh вЂ” on which the revalidated list (the В§3.2 `revalidatePath` contract) reflects the TRUE post-mutation state. The UI stays conservative until the real outcome is known; it never implies the revoke failed (it may have succeeded late).
**Regression tests exercise the REAL exported action path (not just a mocked client callback):** (a) the action hitting an injected infra fault returns `{ kind: 'infra_error' }` and the button shows the inline error (stash the typed-result change в†’ confirm the throw escapes to the boundary / strands the button); (b) a no-response hang в†’ the conservative "couldn't confirm" state, duplicate submit suppressed; (c) **delayed success after the watchdog fires** вЂ” the action commits late в†’ the row is revalidated/removed (or the UI stays conservative until refresh), proving no stale-active row and no duplicate-revoke path.
