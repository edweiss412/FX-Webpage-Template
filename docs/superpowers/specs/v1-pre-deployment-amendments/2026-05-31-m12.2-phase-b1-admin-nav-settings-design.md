### 2.3 UserMenu (`app.jsx:25-65`)
Avatar button (initials) → popover with: identity head (full name + email) + divider + **Sign out** item. 
- **Identity source + fault-boundary contract (load-bearing):** the UserMenu's identity MUST be obtained **inside the layout's existing `AdminInfraError` try/catch** (`app/admin/layout.tsx:47-71`), NOT via a second independent read during UserMenu render — otherwise a Supabase `getUser()`/RPC fault while rendering the menu falls into Next's generic error path instead of the cataloged admin-session-failure surface. **Mechanism:** the layout replaces its `requireAdmin()` call with a single **`requireAdminIdentity()`** (`lib/auth/requireAdmin.ts:89-150`; returns `{ email }`, email from the Supabase JWT canonicalized at `:134`; it gates identically AND yields the email) inside the same try/catch, and passes the resulting `email` down into the nav shell / UserMenu as a **prop**. **Scope of "one read":** this is the LAYOUT's identity read, for the nav-chrome/UserMenu only. In Next App Router a server layout CANNOT pass server-only props to its child page, so a page that needs identity for its OWN logic (notably the embedded Administrators self-revoke policy, §3.2) does its OWN `requireAdminIdentity()` inside that page — covered by that segment's error boundary (§2.7: `app/admin/settings/error.tsx`). That is NOT a forbidden "second fetch of the UserMenu's identity"; it is the page owning its own data (consistent with the §2.6 page-owned-header contract). The rule being enforced is narrow: the UserMenu must not trigger an independent identity read OUTSIDE the layout catch. The prototype's "Doug Pemberton / Production Manager" is mock; **real data = the canonical admin email only** — render the email as the primary identity line; omit name/role (or derive a display name from the email local-part) rather than invent data. **Guard + correct-audience copy (load-bearing):** an `AdminInfraError` from `requireAdminIdentity()` renders the layout's cataloged error surface (`:51-67`). **But the current layout catch renders `messageFor(err.code)` and falls back to `entry.crewFacing` when `dougFacing` is null** — and the identity-fault code (`ADMIN_SESSION_LOOKUP_FAILED`) has `dougFacing: null` + a crew/show-misconfiguration `crewFacing`, so as-is it would show Doug *crew* copy on the admin shell. B1 changes the layout catch to render the **new `ADMIN_ROUTE_LOAD_FAILED`** (Doug-facing, §8.1) for the identity-read fault — NOT `messageFor(err.code)`'s wrong-audience fallback. (This supersedes the R19 "layout can use err.code" note: server-side serialization is fine, but the *content* of that code is wrong-audience — so the layout uses the fixed Doug-facing code too, consistent with the client boundaries.) **Test:** a `getUser()`/identity infra fault renders the cataloged surface with the **actual `ADMIN_ROUTE_LOAD_FAILED` Doug-facing copy** (assert the copy text, not just the `admin-layout-infra-error` test id), never crew/show-misconfiguration copy, never Next's generic page.
- **Sign out** posts to `app/auth/sign-out/route.ts` (POST, `:89-127` — same-origin gated, calls `supabase.auth.signOut()`, clears cookies, 303 → `/auth/sign-in`). The menu item submits a same-origin POST form (not a link — GET returns 405 at the route).

### 2.4 NotifBell (`app.jsx:16-23`)
Bell icon with a count badge. 
- **Count source:** number of unresolved, non-info `admin_alerts` — the SAME filter the AlertBanner uses (`.is("resolved_at", null)` + the `INFO_SEVERITY_CODES` exclusion derived from `MESSAGE_CATALOG`, `components/admin/AlertBanner.tsx:57-59,97-99`). The AlertBanner **already runs this exact `head:true` count inline** for its "+N more" chip (`AlertBanner.tsx:142-147`); B1 **extracts that count query into the shared helper** (§8.1 row 1) that feeds BOTH the bell badge and the banner chip, so they cannot drift (not net-new query logic — a refactor-to-shared).
- **Click behavior:** navigates to `/admin#alerts` (the existing scroll anchor wrapping the AlertBanner, `layout.tsx:92`). On the dashboard this scrolls to the banner; from any other route it routes to `/admin` then to the anchor. No dropdown panel in B1 (YAGNI; the banner is the alert surface).
- **Guards (infra error MUST be visually distinct from zero — no silent failure):** the helper returns a typed result so the bell can distinguish three states: (a) **count = 0** → bell renders with **no badge** (clean state; affordance always present); (b) **count > 0** → numeric badge; (c) **infra_error** → a **distinct degraded bell** (a small warn-tone indicator / "!" affordance, NOT a clean no-badge bell) that links to `/admin#alerts` — so a broken `admin_alerts`/RLS/schema path is *visible* exactly when operators need it, never disguised as "no unresolved alerts." Its accessible label + any visible copy resolve through a **NEW catalog code `ADMIN_ALERT_COUNT_FAILED`** — no existing code fits (`ADMIN_ALERT_NOT_FOUND` / `ALERT_REQUIRES_SHOW_SCOPED_RESOLVE` are unrelated), so B1 adds it via the §12.4 three-lockstep (see §8.1); copy via `lib/messages/lookup.ts`, no raw codes (invariant 5).
- **Banner-detail read must ALSO be fail-visible (the bell creates a NEW cross-read contradiction B1 must close):** the bell count is a `head:true` count, but the AlertBanner's topmost-alert DETAIL SELECT joins `shows(slug)` and currently returns **null on error** (`AlertBanner.tsx:61-78,125-128`). Different failure domains: a join/RLS/schema fault could make the count succeed (positive bell → routes to `/admin#alerts`) while the banner shows **nothing** — hiding alert context exactly when degraded. So **B1 changes the AlertBanner**: when the detail SELECT errors, render a **cataloged degraded banner** (via `ADMIN_ALERT_COUNT_FAILED`, the same shared alert-summary surface) instead of `null`. A positive bell count must never route to an empty alert surface. (This reverses the earlier "AlertBanner unchanged" stance — the new bell makes the banner's silent-null a real contradiction; the no-silent-failure principle now applies to BOTH the bell count and the banner detail read.) **Test:** count succeeds but the banner detail SELECT throws/returns error → degraded banner renders (not empty), surfaced via the cataloged code.
- **Cap:** badge displays the exact count up to 9; **10+ renders "9+"**.

### 2.6 Page header / breadcrumbs (`app.jsx:155-204`)

**Data-ownership contract (load-bearing — prevents re-fetch / stale-header bugs).** The prototype renders one global `PageHead` from a `HEADERS` map + `window.DATA` (`app.jsx:155-204`), but that only works because the prototype has global mock data. In the real app, per-show header data (the show row, `published`/`archived`, the share token, crew-link eligibility/URL) is **page-owned** — fetched inside `app/admin/show/[slug]/page.tsx`. Therefore:
- **Layout-owned** (`app/admin/layout.tsx`): the persistent nav chrome only (top bar / mobile bar / bottom tabs / UserMenu / NotifBell / dark toggle). Its only data needs are the admin identity (§2.3) and the alert count (§2.4) — neither requires per-show data.
- **Page-owned**: each page renders its own header via a shared presentational `<AdminPageHeader title sub crumb? backHref? rightSlot?>` component (new, `components/admin/nav/AdminPageHeader.tsx`) populated **from props the page already has** — no data fetching inside the header component, none in the layout. The per-show page passes its already-computed Published/Archived pill + crew-link chip into `rightSlot`; the dashboard/settings pages pass title/sub only. This guarantees the header and body render from the **same** show/token snapshot (no second query, no drift) and keeps the per-show pill/chip exactly where their data lives.
- The layout does **not** read the route to synthesize per-show headers; static routes (dashboard, settings) simply render `<AdminPageHeader>` with route-appropriate title/sub from the page.

Per-route header content (each rendered by its page via `<AdminPageHeader>`):
| Route | Header |
|---|---|
| `/admin` (dashboard) | Title "Dashboard" + sub "Your live shows and anything that needs review." (matches Phase A `Dashboard.tsx`). |
| `/admin/settings` | Title "Settings" + sub "Manage your Drive connection, who can administer, and how the app behaves." |
| `/admin/show/[slug]` | Breadcrumb "Admin › Active shows" + **Back to dashboard** ghost button + Published/Archived pill + crew-link chip. **Phase A already ships the in-body `← Admin home` back-link (`app/admin/show/[slug]/page.tsx:263`), the title (`:267`), and the Published/Archived `StatusIndicator` (`:243,:274`).** B1 consolidates the breadcrumb + back-link into the page's `<AdminPageHeader>` (via `crumb`/`backHref` props) and **removes the now-duplicated standalone in-body `← Admin home`** (`:263`); the Published/Archived pill + crew-link chip pass through `rightSlot` (single source — rendered once, in the header, from the page's own data). Mobile reorders: title → sub → crew-link chip → pill (`app.jsx:174-180`); desktop: title + pill inline, then sub (`:182-190`). |
| `/admin/show/staged/[stagedId]`, onboarding/staged | No breadcrumb-with-crumb; plain title (these are slug-less candidate surfaces, master spec §9.1 `:2613`). |

**Guard:** unknown route → render a plain title from a sensible default (no crash); the header block must tolerate a route with no `HEADERS` entry.

### 2.7 Admin route-gate topology (load-bearing — the layout catch is NOT the only admin read)
Folding the identity read into the layout catch (§2.3) covers the layout's own read, but **every `/admin/*` page still performs its own admin gate AFTER the layout renders** (defense-in-depth, per the layout docstring). A post-layout `AdminInfraError` from any of these would fall into Next's generic error path unless that route segment has a cataloged error boundary. B1 keeps the page-level gates (defense-in-depth is intentional) AND guarantees a cataloged surface for every admin route:

| Route segment | Existing page-level gate | Error boundary covering a post-layout `AdminInfraError` |
|---|---|---|
| layout (all `/admin/*`) | `requireAdminIdentity()` inside the `:47-71` catch (§2.3) | the layout's own cataloged `admin-layout-infra-error` (`:51-67`) |
| `/admin` (dashboard) | `requireAdmin()` `app/admin/page.tsx:82` | **none today → B1 adds `app/admin/error.tsx`** (catch-all for the segment) |
| `/admin/settings` | `requireAdmin()` `app/admin/settings/page.tsx:37` | **B1 adds `app/admin/settings/error.tsx`** (§3.2) |
| `/admin/settings/admins` | `requireAdminIdentity()` `app/admin/settings/admins/page.tsx:38` | existing `app/admin/settings/admins/error.tsx` |
| `/admin/dev` | `requireAdmin()` `app/admin/dev/page.tsx:60` | inherits `app/admin/error.tsx` (build-gated route; §3.3) |
| `/admin/show/[slug]` | (per-show gate) | inherits `app/admin/error.tsx` |
| `/admin/show/staged/[stagedId]` | (admin gate) | inherits `app/admin/error.tsx` |
| `/admin/onboarding/staged/[wizardSessionId]/[driveFileId]` | (admin gate) | inherits `app/admin/error.tsx` |
| `/admin/show/[slug]/preview/[crewId]` | (admin gate + Phase-A archived rejection) | inherits `app/admin/error.tsx` |

**This enumerates EVERY current `app/admin/**/page.tsx` segment** — the catch-all `app/admin/error.tsx` structurally covers every one that lacks a closer `error.tsx` (settings/admins is the only closer boundary). Any page that ALSO renders its own inline infra fallback keeps it; the catch-all is the floor, not a replacement. (If a future admin route is added, it inherits the catch-all automatically — but the plan's route inventory check must confirm coverage.)

**Strategy = keep gates + add the catch-all `app/admin/error.tsx`** (Next.js error boundaries nest: `app/admin/error.tsx` catches any `/admin/*` segment without a closer `error.tsx`). **Client error boundaries MUST use a FIXED catalog code, NOT `err.code`:** `error.tsx` files are client components and Next serializes the error crossing to them as `Error & { digest? }` — custom fields like a thrown `AdminInfraError.code` do NOT reliably survive (production strips/digests). The existing `app/admin/settings/admins/error.tsx` already does this correctly — it renders `getDougFacing("ADMIN_EMAIL_LIST_FAILED")` (a fixed code, `:49`), NOT `err.code`. Each boundary renders ONE fixed, SETTLED code (per-segment topology — one code per boundary, no ambiguity):
- **`app/admin/error.tsx`** (new, catch-all) → **`ADMIN_ROUTE_LOAD_FAILED`** (new code).
- **`app/admin/settings/error.tsx`** (new) → **`ADMIN_ROUTE_LOAD_FAILED`** (same generic — segment evident from URL).
- **`app/admin/settings/admins/error.tsx`** (existing) → **KEEPS `ADMIN_EMAIL_LIST_FAILED`** unchanged (its more-specific administrator-list copy; the generic code does NOT override it). Note: B1 only EDITS this code's `helpfulContext` for surface-neutrality (§8.1), it does not repoint the boundary.

**Do NOT reuse `ADMIN_SESSION_LOOKUP_FAILED`** for the new boundaries — it has `dougFacing: null` + crew-facing show-misconfiguration copy, so it would render a blank/wrong admin surface. `ADMIN_ROUTE_LOAD_FAILED` Doug-facing copy ≈ "This admin page couldn't load. Refresh in a moment; if it keeps failing, contact the developer." The **layout** catch (`:47-71`) also renders the fixed **`ADMIN_ROUTE_LOAD_FAILED`** copy for the identity-read fault — NOT `messageFor(err.code)`. (Server-side it *could* read `err.code` without a serialization problem, but the identity-fault code's catalog content is wrong-audience — `ADMIN_SESSION_LOOKUP_FAILED` is `dougFacing: null` + crew-facing — so the layout uses the fixed Doug-facing code, consistent with the client boundaries. §2.3.) **Tests:** a post-layout `requireAdmin*` `AdminInfraError` thrown from `/admin` and `/admin/settings` renders via `ADMIN_ROUTE_LOAD_FAILED`, and from `/admin/settings/admins` renders via its existing `ADMIN_EMAIL_LIST_FAILED` — each with **non-null Doug-facing copy** (not blank, not Next's generic page), exercised with a REAL server-thrown `AdminInfraError` (not a mocked client error object).

## 3. Settings shell (`/admin/settings`)

Source: `admin/settings.jsx`. Rebuilds `app/admin/settings/page.tsx` (currently re-run-setup + Administrators-link, `:60-113`). **Max content width 740px** (`settings.jsx:28`). Three sections; the prototype's "Preferences" toggles are deferred (§1.2), so B1's Preferences-area content is the **Developer-tools row only** (no toggle controls) — see §3.3.

### 3.1 Drive connection (read-only health readout)
Panel with: folder icon + **folder name** + a status line + **Healthy/Warn pill** (right) + divider + helper text "Need to switch folders? Re-run setup. Your current shows keep syncing the whole time." + **[Open folder]** + **[Re-run setup]** buttons (`settings.jsx:31-52`).

**Status-line copy MUST derive from `health/reason`, NOT a static "Connected" prefix** (a static "Connected" would contradict the pill when the watch is missing/inactive/expired or a read faulted — the prototype's `Connected · 5 shows syncing · last read 8 min ago` at `settings.jsx:39` is the *positive*-state copy only). Mapping: **positive** → "Connected · {N} shows syncing · last read {relative}"; **`not_configured`** → "Connection not set up"; **`watch_inactive`/`watch_expired`** → "Connection needs attention · last read {relative}"; **`sync_*`/`stale_*`/`sync_unknown`** → "Syncing, but {attentionCount} show(s) need attention · last read {relative}". **`infra_error`** → "Couldn't read sync status". The "{attentionCount}" is the count of shows matching the **winning reason's** predicate (the `attentionCount` field above) — NOT `syncingCount` (the total active count): one stale show among 501 active shows must read "1 show needs attention," never "501." The prefix never says "Connected" unless `health === 'positive'`. **Tests:** (a) the visible status-line copy per health-reason group (no "Connected" prefix on any Warn/infra state); (b) one stale/failing show with `activeCount` ≫ 1 → "1 show needs attention" (not the total).

**Data sources & computation (all reads — B1 adds NO writes).** All panel reads are wrapped in a single typed-result helper **`fetchDriveConnectionHealth()`** (new, `lib/admin/driveConnectionHealth.ts`) returning a discriminated result that **carries the Warn reason + catalog code** (so the pill picks the right cataloged copy without re-querying or duplicating logic):
```
| { health: 'positive'; folderName: string|null; folderId: string|null; syncingCount: number; lastReadAt: string|null }
| { health: 'warn';
    reason: 'not_configured'|'watch_inactive'|'watch_expired'|'sync_drive_error'|'sync_sheet_unavailable'|'sync_parse_error'|'sync_unknown'|'stale_severe'|'stale_moderate';
    code: MessageCode; folderName: string|null; folderId: string|null; syncingCount: number; attentionCount: number; lastReadAt: string|null }
| { kind: 'infra_error' }
```
Any returned-error or thrown builder/client fault → `infra_error` → Warn pill + plain message (never a crash, never a false "Healthy"). The `reason`→`code` mapping preserves the **status-specific** catalog codes (NOT a collapsed `DRIVE_FETCH_FAILED` — a moved sheet vs a parse failure vs a connectivity failure need different recovery guidance, matching `StaleFooter.tsx:57-59`): `not_configured`/`watch_inactive`/`watch_expired` → `WATCH_CHANNEL_ORPHANED`; `sync_drive_error` → `DRIVE_FETCH_FAILED`; `sync_sheet_unavailable` → `SHEET_UNAVAILABLE`; `sync_parse_error` → `PARSE_ERROR_LAST_GOOD`; `sync_unknown` → **`SYNC_STATUS_UNKNOWN`** (new code, §8.1 three-lockstep — the enum-drift branch needs cataloged fail-visible copy; settled, not "reuse-or-add-later"); `stale_severe` → `SYNC_DELAYED_SEVERE`; `stale_moderate` → `SYNC_DELAYED_MODERATE`. Every `reason` has a settled `code` (the union and this mapping are exhaustive — no branch can typecheck-fail or silently collapse). This helper is a Supabase call boundary registered in §8.1 (invariant 9). Field derivation:
- **Folder name:** `app_settings.watched_folder_name` (`supabase/migrations/20260501001000_internal_and_admin.sql:235`; the table also has `watched_folder_id` `:234` and `watched_folder_set_at` `:237`). The existing typed reader is `getActiveWatchedFolderId()` returning `ActiveWatchedFolderResult` (already an `infra_error`-typed result, `lib/appSettings/getWatchedFolderId.ts:9-38`) — B1 extends it / adds a sibling to also surface the stored `watched_folder_name`. The name IS stored at onboarding, so display it directly. **Guard — name null** (unlikely post-onboarding): neutral fallback label ("Your show-sheets folder") + the Open-folder link. (Fetching the *live* Drive name is out of scope — no Drive-name read path exists; do not add one. The stored name may be stale if renamed in Drive; acceptable for B1.)
- **"N shows syncing" (display):** count of active shows (`shows WHERE archived = false`), reusing the bounded-count pattern from Phase A `fetchDashboardData`. **Guard — 0:** "No shows syncing yet."
- **"last read {relative}" (display only):** **max** `shows.last_synced_at` across active shows (column `…initial_public_schema.sql:22`), rendered relative ("8 minutes ago"). This is a DISPLAY value only — it MUST NOT drive the health decision (see below). **Guard — never synced (all null):** "Not synced yet."

## 4. Revoke-hang robustness fix

**Bug** (carry-in, RECONCILIATION §B + ARCHITECT-UPDATE §6): `RevokeRowButton.tsx` drives a `confirm → resolving → (reset)` UI state machine via `useActionState`. The "Revoking…" → reset only fires when the action **returns** a non-ok result (`:62-69` — the `ui === "resolving"` snap reads the returned result). If `revokeAdminAction` **throws** or the request **hangs**, no result returns, the snap never fires, and the button is stranded on "Revoking…" indefinitely. (Env was confirmed NOT the cause — ARCHITECT-UPDATE §6.)

**Fix — the recovery contract belongs at the SERVER-ACTION boundary, not a client-side try/catch** (a React server action submitted via `useActionState` + `<form action={...}>` routes a THROWN error to the nearest error boundary, NOT back to local component state — a client-side catch around the dispatch cannot intercept it). So:
1. **Typed-result, don't throw — but NARROWLY scoped (don't swallow auth/control-flow):** the catch wraps ONLY the `revokeAdminEmail(...)` data-layer call (which runs AFTER `requireAdminIdentity()` has already gated, `actions.ts`), catches ONLY `AdminEmailsInfraError`, and **returns a typed `{ kind: 'infra_error' }` outcome** (extend `AdminEmailActionResult`, `actions.ts:47-52`) instead of throwing. **MUST rethrow** (never convert to inline retry): `AdminInfraError` from the `requireAdminIdentity()` gate (a permission/session failure must propagate to the route error boundary, not look like a retryable revoke), Next navigation/control-flow throws (`redirect`/`notFound`/`forbidden` digests), and any unknown error. `useActionState` receives the typed `infra_error` and the button transitions `resolving → error` with a plain-language retry message — no error-boundary teardown of the settings section. **Tests:** (a) a gate auth/session `AdminInfraError` still propagates to the boundary (NOT inline retry); (b) a `revokeAdminEmail` `AdminEmailsInfraError` → `{ kind: 'infra_error' }` inline; (c) a Next control-flow throw propagates.
2. **Watchdog for no-response hangs → a CONSERVATIVE "outcome unknown" state, NOT idle/retry:** a React server action cannot be cancelled by resetting local UI — a merely-slow revoke RPC may still commit AFTER a naive timeout fires. So if no result arrives within a bounded window, the button transitions to an explicit **"Couldn't confirm — refresh to check"** state that (a) does NOT return to a normal idle/retry state, (b) **suppresses duplicate submits** of the same revoke, and (c) prompts a refresh — on which the revalidated list (the §3.2 `revalidatePath` contract) reflects the TRUE post-mutation state. The UI stays conservative until the real outcome is known; it never implies the revoke failed (it may have succeeded late).
**Regression tests exercise the REAL exported action path (not just a mocked client callback):** (a) the action hitting an injected infra fault returns `{ kind: 'infra_error' }` and the button shows the inline error (stash the typed-result change → confirm the throw escapes to the boundary / strands the button); (b) a no-response hang → the conservative "couldn't confirm" state, duplicate submit suppressed; (c) **delayed success after the watchdog fires** — the action commits late → the row is revalidated/removed (or the UI stays conservative until refresh), proving no stale-active row and no duplicate-revoke path.
