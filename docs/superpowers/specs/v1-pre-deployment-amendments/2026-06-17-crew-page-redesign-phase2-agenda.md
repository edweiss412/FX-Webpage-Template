# Crew Page Redesign — Phase 2: AGENDA run-of-show enrichment

**Status:** draft → adversarial review (cross-model, Codex). User reviews waived per owner mandate.
**Depends on:** Phase 1 spec (`2026-06-15-crew-page-redesign-phase1-design.md`, adversarial-APPROVED). Phase 2 ships **only after** Phase 1's Schedule section + `resolveKeyTimes` anchor strip exist.
**Scope:** one optional, fail-soft parser block + one JSONB column + one projection field + one **enrichment** of the Phase-1 Schedule section. No new route, no new auth surface, no UI-shell change.

---

## 1. Summary

The sheet's AGENDA run-of-show — a per-day grid of `START/FINISH/TRT/TITLE/ROOM/AV` blocks — is **filled with real session content once a show's agenda is locked** (verified **live via the gsheets MCP**, 2026-06-17: filled in 2 of the 7 real shows — East Coast Family Office and RIA Investment Forum; empty auto-time-skeleton in the other 5 + the standardized-template copies, which are mostly future-dated / not yet locked). _(The original audit's "AGENDA empty in all 7 sheets" was **wrong** — the run-of-show is a real, production-used source. An empty AGENDA is a valid result, not a defect: it reflects the project's phased "anchor-times-now / titles-later" design.)_ Phase 2 parses that grid into a structured per-day timeline and **enriches** the Phase-1 crew Schedule section: a day with parsed agenda entries renders a richer run-of-show list; a day without (empty grid, or a show that never filled it) falls back to the Phase-1 `resolveKeyTimes` anchor strip (Load-in / Program / Strike + per-room set/show/strike). The parser is **fail-soft** — a missing/empty/malformed AGENDA grid yields **no** `run_of_show` and the anchor strip is unaffected. Nothing about Phase 1 regresses; Phase 2 is purely additive.

## 2. Resolved decisions

- **D-1. New parser block `parseAgenda`** reads the AGENDA tab grid (when present in the source markdown) and emits `ShowRow.run_of_show: Record<ISODate, AgendaEntry[]> | null`. Added to `parseSheet`'s sequential block list (`lib/parser/index.ts:311`), alongside the existing `parseAgendaLinks` (which is unchanged — AGENDA **LINK** rows and the AGENDA **tab grid** are distinct surfaces).
- **D-2. Fail-soft, never a hard error.** A missing AGENDA tab, an all-empty grid, `#REF!` day banners, ragged per-block row counts, trailing-space headers, or any parse fault → `run_of_show = null` (or the offending day/block omitted), recorded as a **warning** (not a `hardError`). The show still ingests; the crew page still renders the anchor strip. The AGENDA tab is the **most fragile** sheet surface (formula-driven), so robustness is the primary contract.
- **D-3. Storage = new `shows_internal.run_of_show jsonb` column** (nullable, default `null`), keyed by ISO date → `AgendaEntry[]`. **NOT on `public.shows`** (R1-HIGH): `shows` is crew-readable via the `crew_read` RLS policy (`can_read_show(id)` = crew-membership-by-email, **no** per-day `DateRestriction` check — `supabase/migrations/20260501002000_rls_policies.sql:230`), so a crew member with a Supabase session could `select run_of_show` directly via PostgREST and bypass the projection's per-day gate. `shows_internal` is **admin-only** (`admin_only` RLS `using (is_admin())` — same file `:62-65`) and physically separated, exactly the `financials` defense-in-depth model (`getShowForViewer.ts:28-38`). A direct non-admin/non-service PostgREST read returns **zero rows**; the only read path is the service-role projection (below), which applies the per-day gate. (Crew-page viewers in fact hold only a signed picker cookie — no Supabase session at all — but storing in `shows_internal` closes the bypass for the signed-in-by-email case too, and is the correct home for per-day-gated content.)
- **D-4. Projection field `ShowForViewer.runOfShow`** — `getShowForViewer` reads `shows_internal.run_of_show` via the **service-role client** (which bypasses RLS, `:200`/`:235`) **for every viewer** (unlike `financials`, which is read only `if(isLead)` — `run_of_show` is date-gated, not lead-gated), then emits `ShowForViewer.runOfShow` **gated by the same `DateRestriction`** as the Schedule section (Phase 1 §4.2 / wp-16): the projection **drops every ISO-date key the viewer is not entitled to see** — `unknown_asterisk` → no keys (cannot infer show days); `explicit` → only assigned days; `none` → all. Same intersection logic the Schedule section already applies to dates; the projection (not RLS) is the per-day enforcement, and `shows_internal`'s RLS is the direct-access backstop.
- **D-5. UI = enrichment of the Phase-1 Schedule section, not a new component tree.** For each rendered Schedule day, **if** `runOfShow[isoDate]` is non-empty → render the run-of-show list; **else** render the Phase-1 anchor strip for that day. The two are mutually exclusive **per day** (a show can have agenda for some days, anchors for others). No change to the section nav, the shell, or any other section.
- **D-6. Caps (display cap, not parser truncation — R1-MEDIUM).** The **parser stores all** emitted entries per day (bounded only by a 200/day safety cap, §4.1 step 6); the **UI** renders at most **20** entries per day and computes the overflow disclosure (`+N more`, `N = stored.length − 20`) **from the stored array** — so the count is always correct (truncating at the parser would make `+N more` un-computable). Entry `title` truncates at **80** chars in the display with a `<details>` expander (mirrors Notes truncation); the full title stays stored.
- **D-7. Admin-only observability reuse.** A `run_of_show` **parse** failure (sheet-side) surfaces as a parser **warning** in the ingestion path (existing `pending_ingestions` / sync surface), **not** a new admin-alert code. A **fetch** failure of the `shows_internal.run_of_show` column at read time is covered by the existing Phase-1 `getShowForViewer` `tileErrors` + `CrewShell` projection alert (the new column is read in the same query). **No new `admin_alerts` code, no §12.4 catalog change.**
- **D-8. No Right-Now hero change.** `RightNowHero` / `buildRightNowContext` / `selectRightNowState` are **untouched** — the hero consumes `dates` + anchor times only (Phase 1 §4.3). The run-of-show enriches the **Schedule** section, not the hero.

## 4. Design

### §4.1 — The AGENDA grid contract (real structure — verified LIVE via the gsheets MCP, 2026-06-17)

Every AGENDA tab across the test corpus (10 sheets: real shows + standardized-template copies) is the **same deterministic template** — **verified live via the gsheets MCP (2026-06-17)**, not inferred from the markdown. The markdown "raggedness" I first saw was a sheet→markdown **trailing-empty-trim artifact**, not real structural variance.

- **Rows 1–5 are fixed:** row 1 = blank or a `BACK TO INFO` nav link; **rows 2–4 = three stacked BANNER rows** (row 2 = **day-TYPE** `TRAVEL DAY`/`SET DAY`/`DAY 1`/`DAY 2`/`DAY 3`; row 3 = **DATE** e.g. `5/15/24`; row 4 = **day-NAME** e.g. `Wednesday`), **each value anchored ONLY to the starting column of its day block** (trailing columns within the block's span are blank); **row 5 = HEADER**; **data rows start row 6**.
- **The HEADER is NOT a usable primary key** — its labels are **duplicated** (`TITLE`×3, `ROOM`×3, `START`/`FINISH`/`TRT`/`AV`×2, `TIME`×2) and `START` carries a **trailing space** (`"START "`) in every sheet. Block boundaries come from the **banner**, not the header.
- **Per-day blocks are VARIABLE-WIDTH but DETERMINISTIC by day-type:** `TRAVEL DAY` = 3 cols (`NAME|ARRIVAL|FLIGHT#`, idx 0–2); `SET DAY` = 3 cols (`TIME|TITLE|ROOM`, idx 3–5); each **show day `DAY N` = 6 cols** (`START|FINISH|TRT|TITLE|ROOM|AV`). The **first show-day block invariably starts at col index 6** (col G); show-day `TITLE` columns at idx **9 / 15 / 21** for DAY 1/2/3. The run-of-show lives **only** in the `DAY N` show blocks (`parseAgenda` skips `TRAVEL`/`SET` blocks — they carry crew arrival / set times, not sessions).
- **Day → ISO date** comes from the **DATE banner (row 3)** at the block-start column (M/D/YY → ISO, e.g. `5/15/24` → `2024-05-15`), with the **day-NAME banner (row 4)** as cross-check, reconciled against `parseDates`'s `dates.showDays[]` (`lib/parser/blocks/dates.ts:48-72`). The AGENDA banner carries its dates **inline** — the DATES/INFO tab is only a fallback for a broken banner.
- **`#REF!`** appears in the **TRAVEL-DAY banner** date/day-name cells of the standardized-template copies (broken travel-day formula); never in show-day data. Tolerate it: fall back to day-NAME→`dates`, else skip that block (warning).
- **Fill state (verified):** the run-of-show is **filled in production once a show's agenda is locked** (East Coast 5/13–15/24 — "Opening Keynote", "Managing Risk and Liquidity", Breakouts A–H; RIA Central 6/25/25 — "Attendee Registration and Breakfast", …) and **empty** (auto-time skeleton only; `TITLE`/`ROOM`/`AV` blank) for shows not yet locked (8/10 of the test corpus, mostly future-dated/template copies). **An empty AGENDA is a valid result → `run_of_show = null`** (the project's phased "anchor-times-now / titles-later" design), not a parse failure.
- **No normalized continuation table exists** in any live sheet (`A30:Y200` empty everywhere). (A stale historical markdown fixture showed an 8-col `EVENT/DAY` table; that is **not** the current template — `parseAgenda` does not depend on it.)

**Parse algorithm (`parseAgenda`), banner-anchored, fail-soft throughout:**
1. Locate the AGENDA grid in the markdown (the wide table preceded by the 3 banner rows + the `NAME | ARRIVAL | FLIGHT# …` header). **Absent → `null`** (warning-free).
2. **Walk the day-TYPE banner (row 2) left→right.** Each non-empty cell **opens a day block**; the block's column span = `[this column, next non-empty banner column)`. `TRAVEL`/`SET` → skip. `DAY N` → a 6-col show block `[START, FINISH, TRT, TITLE, ROOM, AV]` (TITLE = block-start + 3). **Dual-form fallback:** the live sheets always carry separate banner rows, but the sheet→markdown conversion can collapse the banner into the header as `<DayName>/<token>` cell prefixes (observed in one fixture, `2025-10-consultants-roundtable.md`: `Wednesday/START`, `#REF!/NAME`). If no day-TYPE banner row is present, derive the block boundaries + day-name from the **header-cell prefixes** instead (split each header cell on `/`, trim a leading `#REF!/`; a new day-name prefix opens a block). Either way the run-of-show columns are the same `START/FINISH/TRT/TITLE/ROOM/AV`.
3. **Resolve each show-day's ISO date** from the DATE banner (row 3) at the block-start column (M/D/YY → ISO), day-NAME banner (row 4) as cross-check; reconcile to `dates`. A `#REF!`/unresolvable banner date → day-NAME→`dates` fallback, else **skip the block** (warning); never invent a date.
4. **Walk data rows from row 6 down, per block to that block's OWN last non-empty row** (per-day row counts DIFFER — DAY 2 often runs longer). Read the block's `[START, FINISH, TRT, TITLE, ROOM, AV]` by **absolute column index**; **treat a short/trimmed row as right-padded blanks** (the markdown trims trailing empties — a short row is NOT malformed). Iff **TITLE** is non-empty, emit `AgendaEntry { start, finish?, trt?, title, room?, av? }` (display strings, **never** re-parsed to `Date`; embedded `\n`/`\r\n` inside TITLE/AV preserved).
5. Group entries by resolved ISO date → `Record<ISODate, AgendaEntry[]>` (sheet row order). A day with zero filled TITLE rows contributes **no key** (absent, not `[]`). **No** block yields any entry → the whole result is **`null`** (the column stores `null`, not `{}`). **Never collect the SET-DAY `TITLE` (idx 4)** — it is a 3-col set-day block, not a show-day session.
6. **No parser-level truncation** (cap is at the UI, §4.3) — the parser stores all emitted entries per day, bounded only by a high safety cap (**200/day**) to prevent unbounded storage abuse; exceeding it records a warning. (The UI's 20-row display cap + `+N more` count is computed from the stored array, §4.3 / D-6.)

**`AgendaEntry` type (`lib/parser/types.ts`):** `{ start: string; finish?: string; trt?: string; title: string; room?: string; av?: string }`. `title` is the only required field (the "filled" signal). All fields are sheet-display strings; none are re-typed.

**Do-not-double-parse:** the `TRAVEL`/`SET` banner blocks (crew arrival `NAME/ARRIVAL/FLIGHT#`, set `TIME/TITLE/ROOM`) are the existing crew/event parsers' surface; `parseAgenda` reads **only** the `DAY N` show blocks and never the travel/set blocks (no overlap with `crew_members.flight_info`).

### §4.2 — Storage + projection

- **Migration** adds `shows_internal.run_of_show jsonb` (nullable, default `null`) — on the **admin-only** internal table (R1-HIGH / D-3), not `public.shows`. Idempotent (`add column if not exists`). Applied locally **and surgically to the validation project** + `pnpm gen:schema-manifest` regenerated and committed, per the migration-reaches-validation discipline (the `validation-schema-parity` gate then asserts the column live). `shows_internal`'s existing `admin_only` RLS is unchanged (a new column inherits it); the table is already in the PostgREST-lockdown posture (admin-only SELECT). Tier×domain + CHECK/enum matrices **N/A** (one nullable JSONB column, no CHECK/enum/FK).
- **Sync write path** persists `parsed.run_of_show` to `shows_internal.run_of_show` on ingest. `shows_internal` is already upserted on ingest (it holds `parse_warnings`/`raw_unrecognized`/`financials`), so `run_of_show` joins that existing upsert — no new write surface, same per-show advisory lock. A `null` overwrites a prior value (a show whose AGENDA was emptied loses its run-of-show — correct).
- **Projection** `getShowForViewer` reads `shows_internal.run_of_show` via the **service-role client** (bypasses RLS, `:200`/`:235`) **unconditionally for every viewer** (NOT `if(isLead)` like `financials` — `run_of_show` is date-gated, not lead-gated), then emits `ShowForViewer.runOfShow: Record<ISODate, AgendaEntry[]> | null` **after applying the viewer `DateRestriction` intersection** (D-4): the projection drops any ISO-date key the viewer is not entitled to see (same gate the Schedule day set uses). A malformed stored value (not an object/array shape) → `runOfShow = null` + a `tileErrors["run_of_show"]` entry (so the Phase-1 `CrewShell` projection alert observes it; §4.4).

### §4.3 — Schedule-section enrichment (UI)

The Phase-1 **Schedule** `*Section` (Server Component, `components/crew/sections/`) gains a per-day branch:

- For each rendered Schedule day (already gated + capped by Phase 1), **if `runOfShow[isoDate]` is present and non-empty** → render a **run-of-show list** (`data-testid="run-of-show-<isoDate>"`): each entry is a row `START–FINISH · TITLE` with `ROOM` and an `AV` badge when present, in sheet order. **Else** → render the Phase-1 `resolveKeyTimes` anchor strip for that day (unchanged).
- **Mutually exclusive per day:** a day renders **either** the run-of-show list **or** the anchor strip, **never both** (D-5). The decision is `runOfShow[isoDate]?.length > 0`.
- **Sentinel hiding:** each optional field (`room`, `av`, `finish`, `trt`) routes through `shouldHideGenericOptional` (Phase 1's sentinel contract) — a `""`/`TBD`/`N/A`/`TBA` value hides that field, not the whole entry (the entry shows iff `title` is real, which the parser already guarantees). The Schedule section is already in the `_metaSentinelHidingContract` walk (Phase 1 extended it to `components/crew/`), so the new field reads are covered.
- **Caps (D-6):** at most 20 entries; a 21st+ collapses into a `+N more` overflow stub with the correct `data-testid`/count (`= length − 20`). `title` > 80 chars → `data-testid="agenda-title-truncated"` + `<details>` body. Migrate/extend the Phase-1 `CardinalityCapBoundary` matrix to the run-of-show list.
- **No links rendered raw.** If a `title`/`room`/`av` string contains a URL, it renders **text-only** (no `https://`/`drive.google.com`/`docs.google.com` substring in the crew DOM), reusing the Phase-1 opening-reel URL-strip guard (§4.15) — agenda cells are free text and could paste a link.

### §4.4 — Error handling + observability

| Fault | Crew sees | Admin signal |
|---|---|---|
| AGENDA tab absent (most shows) | anchor strip (Phase 1) | none (not an error) |
| AGENDA grid malformed at **parse** (sheet) | anchor strip | parser **warning** in the ingestion surface (`pending_ingestions`); `run_of_show` stays `null` |
| `shows_internal.run_of_show` **fetch** fails at read | anchor strip (runOfShow falls to `null`) | Phase-1 `tileErrors["run_of_show"]` → `CrewShell` `TILE_PROJECTION_FETCH_FAILED` (union-merged; the new key joins the existing set — **no new code**) |
| stored value malformed (shape) | anchor strip | `tileErrors["run_of_show"]` (same as fetch fail) — fail-soft, never a render throw |

**Invariant:** a run-of-show fault **never** removes or corrupts the anchor strip — the anchor strip is the Phase-1 floor and Phase 2 only ever **adds** on top of it. (Tested: §9.)

### §4.5 — Guard conditions (every input)

- `runOfShow = null` → every day renders the anchor strip (Phase-1 behavior, no Phase-2 element in the DOM).
- `runOfShow = {}` (empty object) → treated as `null` (no day has entries) → all anchor strips. (The parser stores `null` not `{}`, but the projection guards both.)
- `runOfShow[date] = []` → that day renders the anchor strip (empty array = no entries). (The parser omits empty-day keys, but the UI guards `?.length > 0`.)
- An entry with `title` present but every other field sentinel/blank → renders a title-only row (no time/room/av), never an empty row.
- `DateRestriction = unknown_asterisk` → projection drops all `runOfShow` keys → all visible days (none) render nothing extra; the Schedule unconfirmed-placeholder (Phase 1) is unchanged.

### §4.6 — Mode boundaries / dimensional invariants / transitions

- **Mode boundary:** the run-of-show list and the anchor strip are the two mutually-exclusive **per-day** modes of the Schedule day. No element is shared between them; the day container renders exactly one.
- **Dimensional invariants:** the run-of-show list is a flow list inside the existing Schedule day container (no fixed-height parent with flex/grid children → no Tailwind-v4 `items-stretch` concern); each entry row is `min-h-tap-min` (44px) if it carries a tap target, else natural height. (No `getBoundingClientRect` parent==child assertion is required because there is no fixed-dimension parent; the Phase-1 Schedule layout test already covers the day container.)
- **Transitions:** the per-day mode (run-of-show vs anchor strip) is **fixed at render** for a given day+viewer — it does not toggle client-side, so **no animation** (instant; nothing to inventory). Section-level crossfade is owned by Phase 1's `CrewSectionTransition`, unchanged.

## 7. Watchpoints / do-not-relitigate (review focus)

1. **Fail-soft is the contract (do-not-relitigate toward a hard error).** A malformed/`#REF!`/ragged AGENDA tab is the **norm** for this fragile surface — it MUST yield a warning + `null`/partial, never a `hardError` that blocks ingest. The anchor strip is the Phase-1 floor and is never removed by a Phase-2 fault (§4.4 invariant).
2. **No new admin-alert code / no §12.4 change.** Parse faults are parser **warnings** (existing ingestion surface); read faults reuse the Phase-1 `CrewShell` `TILE_PROJECTION_FETCH_FAILED` via the union-merged `tileErrors["run_of_show"]` key. Phase 2 adds **no** `admin_alerts` code, **no** catalog lockstep. (A reviewer asking for a new code is out of contract — the new column rides the existing projection read.)
3. **`run_of_show` is `DateRestriction`-gated exactly like the Schedule day set** — the projection drops keys the viewer can't see; `unknown_asterisk` sees none. Same trust boundary as Phase 1's Schedule (no new gate to design).
4. **Per-day mutual exclusivity** (run-of-show XOR anchor strip) — one day never renders both.
5. **Hero untouched** — `RightNowHero`/`buildRightNowContext`/`selectRightNowState` consume `dates` + anchors only; the run-of-show enriches Schedule, not the hero (D-8).
6. **Phase boundary** — Phase 2 ships only after Phase 1's Schedule section + `resolveKeyTimes` exist; the enrichment branch and its fallback both depend on the Phase-1 anchor strip being present.
7. **Grid structure is LIVE-verified, banner-anchored (do-not-relitigate the "too inconsistent" framing).** The §4.1 structure was confirmed against all 7 live `fxav-test-shows` INFO/AGENDA tabs via the gsheets MCP (2026-06-18), not inferred from the markdown. It is **one deterministic banner-anchored template**: the "ragged tables / truncated columns / continuation table" that the markdown fixtures suggest are **sheet→markdown trailing-empty-trim artifacts** — a short row is right-padded, not malformed; **no normalized `EVENT/DAY` continuation table exists in any live sheet** (do not anchor on it). The parser anchors on the **banner rows** (day-type/date/day-name), with the header-prefix form as a dual-form fallback. A reviewer arguing the grid is unparseable, or to parse a continuation table, is out of contract.
8. **`run_of_show` lives in admin-only `shows_internal`, not `public.shows` (do-not-relitigate toward `shows`).** `public.shows` is crew-readable by `can_read_show` (email membership, no per-day gate), so a `shows.run_of_show` column would be directly PostgREST-readable by any crew member, bypassing the projection's `DateRestriction` gate. `shows_internal` (admin-only RLS) makes the only read path the service-role projection — the same physical-separation model as `financials`.
