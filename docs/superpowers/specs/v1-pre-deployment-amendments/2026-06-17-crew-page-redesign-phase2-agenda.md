# Crew Page Redesign — Phase 2: AGENDA run-of-show enrichment

**Status:** draft → adversarial review (cross-model, Codex). User reviews waived per owner mandate.
**Depends on:** Phase 1 spec (`2026-06-15-crew-page-redesign-phase1-design.md`, adversarial-APPROVED). Phase 2 ships **only after** Phase 1's Schedule section + `resolveKeyTimes` anchor strip exist.
**Scope:** one optional, fail-soft parser block + one JSONB column + one projection field + one **enrichment** of the Phase-1 Schedule section. No new route, no new auth surface, no UI-shell change.

---

## 1. Summary

The sheet's AGENDA run-of-show — a per-day grid of `START/FINISH/TRT/TITLE/ROOM/AV` blocks — is **filled with real session content once a show's agenda is locked** (verified **live via the gsheets MCP**, 2026-06-17: filled in 2 of the 7 real shows — East Coast Family Office and RIA Investment Forum; empty auto-time-skeleton in the other 5 + the standardized-template copies, which are mostly future-dated / not yet locked). _(The original audit's "AGENDA empty in all 7 sheets" was **wrong** — the run-of-show is a real, production-used source. An empty AGENDA is a valid result, not a defect: it reflects the project's phased "anchor-times-now / titles-later" design.)_ Phase 2 parses that grid into a structured per-day timeline and **enriches** the Phase-1 crew Schedule section: a day with parsed agenda entries renders a richer run-of-show list; a day without (empty grid, or a show that never filled it) falls back to the Phase-1 `resolveKeyTimes` anchor strip (Load-in / Program / Strike + per-room set/show/strike). The parser is **fail-soft and CONFIRMED-ONLY** (D-2): the crew run-of-show is a purely additive overlay that shows a day **iff the latest sync positively parsed entries for it**; **every** non-confirmed outcome — never-filled, read-empty (incl. an intentionally-removed agenda), an unresolved block, or an unlocatable grid — renders the always-correct Phase-1 anchor strip. So crew **never see stale/preserved session times** (the structural close of the R17/R21/R22 stale-data class), the admin gets a parser warning (`AGENDA_DAY_EMPTIED`/`AGENDA_BLOCK_UNRESOLVED`/`AGENDA_GRID_MALFORMED`), and a transient fault self-heals on the next sync (the sheet is the source of truth). Nothing about Phase 1 regresses; Phase 2 is purely additive.

## 2. Resolved decisions

- **D-1. New parser block `parseAgenda`** reads the AGENDA tab grid (when present in the source markdown) and **emits `Record<ISODate, AgendaEntry[]> | undefined`** — per-day-resolved (a resolved day is a key with entries or `[]`; an unresolved block is **absent**; `undefined` only when the grid is unlocatable — D-2). The **stored column** `shows_internal.run_of_show` is `Record<ISODate, AgendaEntry[]> | null` (the sync's CONFIRMED-ONLY full-replace produces it — D-2). Added to `parseSheet`'s sequential block list (`lib/parser/index.ts:311`), alongside the existing `parseAgendaLinks` (unchanged — AGENDA **LINK** rows and the AGENDA **tab grid** are distinct surfaces).
- **D-2. CONFIRMED-ONLY storage — the crew sees a day's run-of-show iff the LATEST sync positively parsed it (R4/R5/R17/R21/R22-HIGH — the canonical, counterexample-free data-retention rule; do-not-relitigate, see §4.4 retention matrix + watchpoint 12).** `parseAgenda` returns `Record<ISODate, AgendaEntry[]> | undefined`. A day is **CONFIRMED** iff the parse produced **non-empty entries** for it; everything else (read-empty `[]`, an unresolved/ambiguous block, or an unlocatable whole grid) is **NOT confirmed**. The crew run-of-show is a **purely additive overlay on the always-present anchor floor: it shows ONLY confirmed days; any non-confirmed day renders the Phase-1 anchor strip** (always current — sourced from `rooms`). Because nothing non-confirmed is ever shown, **no stale/preserved agenda can reach crew** — closing the R17/R21/R22 stale-data class structurally (there is no preserve-and-show path left to leak).
  - **The sync write = the latest parse's confirmed days (FULL REPLACE), under the per-show advisory lock** (invariant 2): if `parsed.runOfShow === undefined` (grid unlocatable) → write **`null`** (no day confirmed) + `AGENDA_GRID_MALFORMED`. Otherwise write **exactly the confirmed (non-empty) days** of `parsed.runOfShow` (`merged = { d: entries | for each d where parsed.runOfShow[d].length > 0 }`, or `null` if none) — a read-empty `[]` day and an unresolved/absent block are simply **not written** (so they render anchors). Emit `AGENDA_DAY_EMPTIED` when a **previously-stored** day is now read-empty, `AGENDA_BLOCK_UNRESOLVED`/`AGENDA_DAY_AMBIGUOUS` for an unresolved block (admin freshness signals — the data itself is just the confirmed set). There is **no per-day preserve/merge** of prior entries: the stored value always equals what the latest successful parse confirmed.
  - **Why CONFIRMED-ONLY (R22 — the structural resolution after R17→R21→R22 on this one vector).** The run-of-show is **operational** (crew act on session times/rooms), so showing it requires **current confirmation it's accurate**. The parser cannot infer operator intent, and an intentional removal can take **any** form — blank TITLE cells (read-empty), a deleted/renamed tab or broken header (unlocatable grid), or a changed template — so **any "preserve last-known-good and keep showing it" rule leaks stale operational data on some removal shape** (R21 closed the read-empty shape; R22 showed the unlocatable-grid shape still leaked). The only rule with **no** stale counterexample is "show only what the latest sync confirmed." **No permanent data loss:** the **sheet is the source of truth**, so a *transient* fault (a momentary unlocatable grid / title-drop on a still-filled sheet) self-heals — the next sync re-reads and re-confirms. The cost is a **brief, safe coarsening to the correct anchor strip** during a transient fault window; this is acceptable because the anchors carry the key times (Load-in/Program/Strike) and are never wrong. **No admin "clear" action is needed** (any removal auto-coarsens to anchors on the next sync). **Converter-regression blast radius is bounded + loud:** a converter change that breaks grid location is caught by the §8 pinned-fixture tests pre-merge; even if it reached prod, every show coarsens to **correct anchors** + emits a loud `AGENDA_GRID_MALFORMED` admin alert per show (a self-announcing, recoverable, non-dangerous degradation — strictly safer than silently showing stale-rich).
  - **NO write-time date-domain prune (R12):** the AGENDA parse confirms a day independent of the DATES parse; a removed/non-current date is dropped at **PROJECTION read** (D-4), not at write. (Orthogonal to confirmed-vs-not — it's the *date-domain*, not the agenda *content*. A confirmed day stays stored even if its date transiently drops from `dates.showDays`; it's just hidden at read until the date returns.)
  - **Net:** confirmed (non-empty parse) → **stored + shown**; read-empty / unresolved / unlocatable → **not stored → anchors** (+ the matching admin warning). Crew **only ever see latest-sync-confirmed run-of-show** — never preserved, never stale. All paths are fail-soft (no `hardError`).
  - **D-2 governs storage only — the DISPLAY rule (D-5 / §4.4) is mechanically simple:** a Schedule day renders its run-of-show **iff `runOfShow[isoDate]?.length > 0`**, else the Phase-1 anchor strip. Because the sync stores **only latest-confirmed days** (above), `length > 0` holds **only** for confirmed days — there is **no "preserved last-known-good" display case**; every non-confirmed day has no stored entries → anchors.
- **D-3. Storage = new `shows_internal.run_of_show jsonb` column** (nullable, default `null`), keyed by ISO date → `AgendaEntry[]`. **NOT on `public.shows`** (R1-HIGH): `shows` is crew-readable via the `crew_read` RLS policy (`can_read_show(id)` = crew-membership-by-email, **no** per-day `DateRestriction` check — `supabase/migrations/20260501002000_rls_policies.sql:230`), so a crew member with a Supabase session could `select run_of_show` directly via PostgREST and bypass the projection's per-day gate. `shows_internal` is **admin-only** (`admin_only` RLS `using (is_admin())` — same file `:62-65`) and physically separated, exactly the `financials` defense-in-depth model (`getShowForViewer.ts:28-38`). A direct non-admin/non-service PostgREST read returns **zero rows**; the only read path is the service-role projection (below), which applies the per-day gate. (Crew-page viewers in fact hold only a signed picker cookie — no Supabase session at all — but storing in `shows_internal` closes the bypass for the signed-in-by-email case too, and is the correct home for per-day-gated content.)
- **D-4. Projection field `ShowForViewer.runOfShow`** — `getShowForViewer` reads `shows_internal.run_of_show` via the **service-role client** (which bypasses RLS, `:200`/`:235`) **for every viewer** (unlike `financials`, which is read only `if(isLead)` — `run_of_show` is date-gated, not lead-gated), then emits `ShowForViewer.runOfShow` = the stored keys **intersected with (a) the current `dates.showDays`** (drop any `run_of_show` key for a date the show **no longer has** — a removed/changed date is hidden at read, **R10/R12**, without ever erasing storage) **AND (b) the viewer's `DateRestriction`** (Phase 1 §4.2 / wp-16): `unknown_asterisk` → no keys (cannot infer show days); `explicit` → only assigned days; `none` → all **current** show days. Same intersection logic the Schedule section already applies to dates; the projection (not RLS, not a destructive sync prune) is the per-day + current-date enforcement, and `shows_internal`'s RLS is the direct-access backstop. (Reading the stored `shows.dates.showDays`, so a transient/partial DATES parse only *hides* a real day for that render — storage is intact, and the next clean sync restores both `dates` and the day's visibility.)
- **D-5. UI = enrichment of the Phase-1 Schedule section, not a new component tree.** For each rendered Schedule day, **if** `runOfShow[isoDate]` is non-empty → render the run-of-show list; **else** render the Phase-1 anchor strip for that day. The two are mutually exclusive **per day** (a show can have agenda for some days, anchors for others). No change to the section nav, the shell, or any other section.
- **D-6. Caps — two layers (R1/R13-MEDIUM).** **(1) Storage caps** (parser, §4.1 step 6): 200 entries/day, per-field length caps (`title` ≤ 300, `room`/`av` ≤ 120, time fields ≤ 40), and a ≤ 32 KB/day serialized-byte ceiling — bounding the JSONB by size (a pathological cell can't fail the sync or bloat the projection). **(2) Display cap** (UI, §4.3): the UI renders at most **20** entries per day and computes the overflow disclosure (`+N more`, `N = stored.length − 20`) **from the stored array** (the count is always correct because the parser does **not** count-truncate below the 200 safety cap); entry `title` is **display-**truncated at **80** chars with a `<details>` expander (mirrors Notes truncation) over the already-storage-capped value.
- **D-7. Observability via the existing `ParseWarning` channel — concrete codes named (R4-MEDIUM).** A `run_of_show` **parse** fault emits a `ParseWarning` (`lib/parser/types.ts:1` — `{ severity, code, message, blockRef?, rawSnippet? }`), aggregated into `ParsedSheet.warnings` → persisted to **`shows_internal.parse_warnings`** (the existing admin-internal warnings surface, alongside `UNKNOWN_ROLE_TOKEN` etc.). `ParseWarning.code` is a **free-string parser-internal code** (NOT the §12.4 `admin_alerts` catalog, NOT a crew-facing code — parser warnings never render in crew UI, so no invariant-5 / `no-raw-codes` concern). New codes, all `severity:"warn"`, with `blockRef:{kind:"agenda", index}`: **`AGENDA_GRID_MALFORMED`** (table/token-header not locatable → `undefined`), **`AGENDA_BLOCK_UNRESOLVED`** (a block's date/day-name can't resolve — `#REF!`, no `dates` match, multi-table mis-alignment → block skipped), **`AGENDA_DAY_AMBIGUOUS`** (day-name matches ≥2 same-weekday dates → block skipped, R2), **`AGENDA_DAY_TRUNCATED`** (a day hit a storage cap — the 200/day count cap, a per-field length cap, or the 32 KB/day serialized-byte ceiling, §4.1 step 6), **`AGENDA_DAY_EMPTIED`** (a **previously-stored** day is no longer confirmed because it parsed **read-empty** → it is **not stored** → crew see anchors, never stale (R22); this warning is the admin's freshness signal that the source went blank — possibly an intentional removal, possibly a transient title-drop that the next sync re-confirms). A **fetch** failure of `shows_internal.run_of_show` at read time is covered by the existing Phase-1 `getShowForViewer` `tileErrors` + `CrewShell` projection alert (the new column is read in the same query). **No new `admin_alerts` code, no §12.4 catalog change** (the §12.4 lockstep is for `admin_alerts` codes only — these are not admin-alert codes). **BUT they are NOT registry-less (R10):** new `code:` literals in `lib/parser` ARE extracted by `scripts/extract-internal-code-enums.ts` into `lib/messages/__generated__/internal-code-enums.ts` and asserted by `tests/cross-cutting/no-raw-codes.test.ts` (the `x2-no-raw-codes` gate). So **run `pnpm gen:internal-code-enums` and commit the regenerated manifest in the SAME commit** that introduces the `AGENDA_*` codes (else x2 fails). They remain admin-internal (→ `shows_internal.parse_warnings`), never crew-facing.
- **D-8. No Right-Now hero change.** `RightNowHero` / `buildRightNowContext` / `selectRightNowState` are **untouched** — the hero consumes `dates` + anchor times only (Phase 1 §4.3). The run-of-show enriches the **Schedule** section, not the hero.

## 4. Design

### §4.1 — The AGENDA grid contract (real structure — verified LIVE via the gsheets MCP, 2026-06-17)

Every AGENDA tab across the test corpus (10 sheets: real shows + standardized-template copies) is the **same deterministic template** — **verified live via the gsheets MCP (2026-06-17)**, not inferred from the markdown. The markdown "raggedness" I first saw was a sheet→markdown **trailing-empty-trim artifact**, not real structural variance.

- **Rows 1–5 are fixed:** row 1 = blank or a `BACK TO INFO` nav link; **rows 2–4 = three stacked BANNER rows** (row 2 = **day-TYPE** `TRAVEL DAY`/`SET DAY`/`DAY 1`/`DAY 2`/`DAY 3`; row 3 = **DATE** e.g. `5/15/24`; row 4 = **day-NAME** e.g. `Wednesday`), **each value anchored ONLY to the starting column of its day block** (trailing columns within the block's span are blank); **row 5 = HEADER**; **data rows start row 6**.
- **The token-header alone is NOT a usable boundary key** — its labels are **duplicated** (`TITLE`×3, `ROOM`×3, `START`/`FINISH`/`TRT`/`AV`×2, `TIME`×2) and `START` carries a **trailing space** (`"START "`). Block **boundaries** come from the **DATE row** (one block per dated column); the token-header **classifies** each block (show-day `START…` vs travel/set). (In the SHEET a day-TYPE banner row also exists, but the markdown the parser consumes usually drops it — see "Markdown representation" below.)
- **Per-day blocks are VARIABLE-WIDTH but DETERMINISTIC by day-type:** `TRAVEL DAY` = 3 cols (`NAME|ARRIVAL|FLIGHT#`, idx 0–2); `SET DAY` = 3 cols (`TIME|TITLE|ROOM`, idx 3–5); each **show day `DAY N` = 6 cols** (`START|FINISH|TRT|TITLE|ROOM|AV`). The **first show-day block invariably starts at col index 6** (col G); show-day `TITLE` columns at idx **9 / 15 / 21** for DAY 1/2/3. The run-of-show lives **only** in the `DAY N` show blocks (`parseAgenda` skips `TRAVEL`/`SET` blocks — they carry crew arrival / set times, not sessions).
- **Day → ISO date** comes from the **DATE banner (row 3)** at the block-start column (M/D/YY → ISO, e.g. `5/15/24` → `2024-05-15`), with the **day-NAME banner (row 4)** as cross-check, reconciled against `parseDates`'s `dates.showDays[]` (`lib/parser/blocks/dates.ts:48-72`). The AGENDA banner carries its dates **inline** — the DATES/INFO tab is only a fallback for a broken banner.
- **`#REF!`** appears in the **TRAVEL-DAY banner** date/day-name cells of the standardized-template copies (broken travel-day formula); never in show-day data. Tolerate it: fall back to day-NAME→`dates`, else skip that block (warning).
- **Fill state (verified):** the run-of-show is **filled in production once a show's agenda is locked** (East Coast 5/13–15/24 — "Opening Keynote", "Managing Risk and Liquidity", Breakouts A–H; RIA Central 6/25/25 — "Attendee Registration and Breakfast", …) and **empty** (auto-time skeleton only; `TITLE`/`ROOM`/`AV` blank) for shows not yet locked (8/10 of the test corpus, mostly future-dated/template copies). **An empty AGENDA is a valid result → `run_of_show = null`** (the project's phased "anchor-times-now / titles-later" design), not a parse failure.
- **No normalized continuation table exists** in any live sheet (`A30:Y200` empty everywhere). (A stale historical markdown fixture showed an 8-col `EVENT/DAY` table; that is **not** the current template — `parseAgenda` does not depend on it.)

**Markdown representation (what `parseAgenda` ACTUALLY consumes).** The parser runs on the **sheet→markdown conversion**, not the sheet, and the converter does **not** preserve the grid 1:1 — confirmed against the committed fixtures: (a) **WHICH banner row the converter promotes to the markdown-table HEADER row VARIES** — the **filled** East Coast markdown promotes the **DATE row** (day-name + token-header as body rows, `2024-05-east-coast-family-office.md:84-87`), whereas the **empty** fixtures promote the **day-TYPE row** (`TRAVEL DAY`/`DAY 1`), with the DATE + day-name + token-header all as **body** rows. So the parser must **NOT** assume which banner row is the md-table header. (b) the day-TYPE row is sometimes the md-header (empty fixtures) and sometimes dropped (East Coast) — the parser never relies on it. (c) trailing-empty cells are trimmed per row (ragged). (d) some **stale** fixtures split the grid across multiple mis-aligned markdown tables and carry a normalized `START|END|TRT|TITLE|ROOM|AV|EVENT|DAY` table (`2025-03-dci-rpas-central.md:335-377`) — a shape the **current** clean live template does **not** emit. **So the parser anchors on the `NAME|ARRIVAL|…|START…` token-header (reliably present in EVERY shape — as a body OR header row) and finds the DATE row + day-name row BY CONTENT among that table's rows (cells matching `M/D/YY` = the DATE row; weekday names = the day-name row), independent of which banner the converter promoted to the table header, and NOT a fixed column stride.** **MANDATORY: the implementation regenerates the filled fixtures (East Coast, RIA) from the CURRENT converter before building** and pins those shapes; the committed fixtures are stale/inconsistent and are kept only as **fail-soft robustness inputs** (a malformed/multi-table conversion → skip the unparseable tables / preserve, never a crash).

**Parse algorithm (`parseAgenda`), token-header anchored (shape-independent), fail-soft throughout:**
1. Locate the AGENDA grid: the markdown table containing a **token-header row** matching `NAME | ARRIVAL | FLIGHT# | TIME | TITLE | ROOM | START | FINISH | TRT | …` — **matched after normalizing each cell: strip an optional leading `<prefix>/` segment (including `#REF!/`) AND trailing whitespace** (R9). This locates **both** the plain token-header **and the header-prefix form** (`#REF!/NAME`, `Tuesday/TIME`, `Wednesday/START` — `2025-10-consultants-roundtable.md:210`), which has **no separate DATE row** (the date/day-name live in the cell prefixes; step 2's dual-form resolves them). The token-header is the distinctive, reliably-present anchor (it may be the md-table header OR a body row). Within that table, identify the **DATE row** (the row whose cells are `M/D/YY` dates) and the **day-NAME row** (weekday names) **BY CONTENT** — scanning all rows, **NOT** by which row the converter promoted to the markdown-table header (R8: filled fixtures promote the DATE row, empty fixtures promote the day-TYPE row); a **prefix-form** table has neither separate row (use the prefixes). **No token-header table found (even after prefix-stripping) → `undefined` + an `AGENDA_GRID_MALFORMED` warning** — **no day is confirmed**, so the sync stores `null` and crew see anchors (D-2 CONFIRMED-ONLY; the next sync re-confirms if it was a transient conversion fault — the sheet is the source of truth). **A located grid (either form) with all-blank show-day TITLEs → a `Record` of all-`[]` keys** (step 5 — the parser never returns `null`): those days are **not confirmed**, so they are **not stored** (→ anchors), with `AGENDA_DAY_EMPTIED` for any day that was previously stored. **In all non-confirmed cases the crew see the always-current anchor strip — never a stale/preserved agenda (R22).**
2. **Block boundaries from the DATE row** (found by content in step 1; **never** the day-TYPE row; NOT a fixed stride): each non-empty `M/D/YY` cell in the DATE row **opens a block**, spanning `[this column, next dated column)`. Classify each block by its **token-header** columns: `START|FINISH|TRT|TITLE|ROOM|AV` (or a right-trimmed tail) = a **show day** → parse; `NAME|ARRIVAL|FLIGHT#` or `TIME|TITLE|ROOM` = travel/set → **skip**. **Dual-form fallback:** if a conversion instead merges day-names into header-cell prefixes (`Wednesday/START`, observed in `2025-10-consultants-roundtable.md`; `#REF!/NAME` tolerated), derive boundaries + day-name from the prefixes (split on `/`, trim `#REF!/`). The run-of-show columns are `START/FINISH/TRT/TITLE/ROOM/AV` either way.
3. **Resolve each show-day's ISO date** from the DATE banner (row 3) at the block-start column (M/D/YY → ISO), day-NAME banner (row 4) as cross-check; reconcile to `dates`. A `#REF!`/unresolvable banner date → day-NAME→`dates` fallback **only when the day-name matches EXACTLY ONE date in `dates.showDays[]`** (R7: **show-day blocks resolve against show days ONLY** — a `parseAgenda` show block keys a session, so a fallback match in `set`/`travelIn`/`travelOut` is **invalid** and the block is left UNRESOLVED + warning, never keyed to a non-show date — else a lost-date show block could be mis-keyed onto a travel/set date and mis-apply the `DateRestriction` gate); **zero OR multiple same-weekday `showDays` matches → leave the block UNRESOLVED (omit it from the Record — absent → NOT confirmed → the sync does not store that day, so crew see anchors, D-2 — + an `AGENDA_DAY_AMBIGUOUS`/`AGENDA_BLOCK_UNRESOLVED` warning)** — **never guess among ambiguous dates** (R2: a mis-key would mis-apply the ISO-keyed `DateRestriction` gate — hide from the assigned crew member or expose to a wrong-day one). Any other unresolvable block → likewise omitted (absent → not confirmed → anchors) + warning; never invent a date.
4. **Walk data rows from row 6 down, per block to that block's OWN last non-empty row** (per-day row counts DIFFER — DAY 2 often runs longer). Read the block's `[START, FINISH, TRT, TITLE, ROOM, AV]` by **absolute column index**; **treat a short/trimmed row as right-padded blanks** (the markdown trims trailing empties — a short row is NOT malformed). Iff **TITLE is REAL** — non-empty **AND not a generic sentinel** (run it through `shouldHideGenericOptional`: trimmed `""`/`TBD`/`N/A`/`TBA` → **NOT** a real title, R13) — emit `AgendaEntry { start, finish?, trt?, title, room?, av? }` (display strings, **never** re-parsed to `Date`; embedded `\n`/`\r\n` inside TITLE/AV preserved). A row whose TITLE is blank-or-sentinel yields **no** entry — so a **not-yet-finalized** day of `TBD`/`N/A` placeholder titles does **not** count as filled (the day stays `[]` → the Phase-1 anchor strip shows, **not** a run-of-show of placeholders).
5. Group entries by resolved ISO date → `Record<ISODate, AgendaEntry[]>` (sheet row order). **Per-day encoding (D-2 CONFIRMED-ONLY):** a **resolved** show-day with entries → a key with its entries (**confirmed → stored + shown**), **or `[]` if every TITLE was blank-or-sentinel** (no REAL title — **not confirmed** → the sync does **not** store that day, so a `TBD`-only or removed-agenda day renders anchors; `AGENDA_DAY_EMPTIED` if it was previously stored); an **unresolved** block (step 3) → **absent** from the Record (**not confirmed** → not stored → anchors + `AGENDA_BLOCK_UNRESOLVED`); if the **grid is unlocatable** (no token-header) → return **`undefined`** (**nothing confirmed** → stored `null` → anchors + `AGENDA_GRID_MALFORMED`). The crew never see a non-confirmed day's prior entries (R22). **Never collect the SET-DAY `TITLE` (idx 4)** — it is a 3-col set-day block, not a show-day session.
6. **Storage caps bound the JSONB by size, not just count (R13-MEDIUM)** — the parser stores all real entries per day up to **(a) 200 entries/day**, **(b) per-field length caps** (`title` ≤ 300, `room`/`av` ≤ 120, `start`/`finish`/`trt` ≤ 40 chars — truncate at store), and **(c) a serialized-byte ceiling ≤ 32 KB/day** (drop overflow entries). Any of these → an `AGENDA_DAY_TRUNCATED` warning. This bounds `shows_internal.run_of_show` by content length so a pathological huge cell cannot fail the sync write or bloat the crew projection. (The UI's 20-row display cap + `+N more` count, and the 80-char display truncation, are **separate/presentational** — §4.3 / D-6 — computed from the already-storage-capped value.)

**`AgendaEntry` type (`lib/parser/types.ts`):** `{ start: string; finish?: string; trt?: string; title: string; room?: string; av?: string }`. `title` is the only required field and is the **filled signal — it must be REAL** (the parser's step-4 emit gate runs it through `shouldHideGenericOptional`, so a `""`/`TBD`/`N/A`/`TBA` TITLE never produces an entry). All fields are sheet-display strings; none are re-typed. **Storage caps (R13-MEDIUM, applied at the parser before storing):** each string field is truncated to a stored max (`title` ≤ **300** chars, `room`/`av` ≤ **120**, `start`/`finish`/`trt` ≤ **40**) and each day is bounded to a **serialized-byte ceiling (≤ 32 KB/day)** — entries beyond the byte ceiling (or the 200/day count cap, §4.1 step 6) are dropped with an `AGENDA_DAY_TRUNCATED` warning. This **bounds the `shows_internal.run_of_show` JSONB by content length, not just row count**, so a pathological huge cell can't fail the sync write or bloat the crew projection. (The UI's 80-char *display* truncation, D-6, is separate and presentational — it operates on the already-storage-capped value.)

**Do-not-double-parse:** the `TRAVEL`/`SET` banner blocks (crew arrival `NAME/ARRIVAL/FLIGHT#`, set `TIME/TITLE/ROOM`) are the existing crew/event parsers' surface; `parseAgenda` reads **only** the `DAY N` show blocks and never the travel/set blocks (no overlap with `crew_members.flight_info`).

Because the sync's `run_of_show` write is a **read-modify-merge** (D-2), a concurrent unlocked PostgREST write can **race and silently lose/corrupt** the merge (advisory locks only serialize *other lock-holders*). R6 argued this was a pre-existing, tolerable posture; **R16 correctly rejects that** — the project's structural-defense-calibration rule says ship the lockdown rather than re-argue the threat model. **Phase 2 therefore adds a REVOKE migration:** `revoke insert, update, delete on public.shows_internal from anon, authenticated;` (idempotent; **SELECT grant + `admin_only` RLS left intact**; `service_role` keeps `all privileges` — `:60`). This makes the **service-role sync, under the per-show advisory lock, the single serialized writer** of `run_of_show` (and the rest of `shows_internal`). **Verified safe (live-code, do-not-relitigate the blast radius):** the ONLY writer of `shows_internal` is the service-role sync (`lib/sync/runScheduledCronSync.ts:1278` — the `financials/parse_warnings/raw_unrecognized` insert); `lib/data/getShowForViewer.ts:482` only **reads** (service-role). **No** authenticated/anon app code mutates the table, so the REVOKE has **zero functional impact** — it only removes the racy manual path. **Register `shows_internal` in `RPC_GATED_TABLES`** (`tests/db/postgrest-dml-lockdown.test.ts:124` — `{ table, closed_at: "<this migration>:<line>", selectAnon: true, selectAuthenticated: true, postBody: { show_id: <uuid>, run_of_show: {…} }, rowFilter }`) **in the same migration commit**: the bidirectional meta-test (`:650` every REVOKE has a registry row; `:674` every registry row has a live REVOKE) fails otherwise. This closes the `shows_internal` portion of `BACKLOG.md` `BL-ADMIN-POSTGREST-DML-LOCKDOWN`.
- Tier×domain + CHECK/enum matrices **N/A** (one nullable JSONB column, no CHECK/enum/FK).
- **Sync write path — CONFIRMED-ONLY full replace (D-2), under the per-show advisory lock** (invariant 2 serializes show mutations): the stored `run_of_show` becomes **exactly the latest parse's confirmed (non-empty) days** — there is no per-day preserve/merge of prior entries. If `parsed.runOfShow === undefined` (grid unlocatable; note the **parser field is camelCase `runOfShow`** on `ParsedSheet`/`ParseResult`; `run_of_show` is **only** the snake_case `shows_internal` DB column, R18/R19) → write **`null`** + `AGENDA_GRID_MALFORMED`. Otherwise write `merged = { d: parsed.runOfShow[d] | for each d where parsed.runOfShow[d].length > 0 }` (or `null` if none). A **read-empty `[]`** day and an **unresolved/absent** block are simply **not written** (→ they render anchors); emit `AGENDA_DAY_EMPTIED` when a **previously-stored** day is now read-empty, `AGENDA_BLOCK_UNRESOLVED`/`AGENDA_DAY_AMBIGUOUS` for an unresolved block. Net: crew only ever see latest-confirmed days; everything else → anchors (R22 — never stale/preserved). A removed/changed show *date* is orthogonal — hidden at PROJECTION read (D-4), **no write-time date prune** (R12). The read uses the prior stored value **only** to decide which days to emit `AGENDA_DAY_EMPTIED` for (observability), **not** to preserve content. `shows_internal` is already upserted on ingest (it holds `parse_warnings`/`raw_unrecognized`/`financials`), so this joins that existing upsert — no new write surface. (Implementation: runs inside the same locked transaction that reads the existing `shows_internal` row.)
- **Projection** `getShowForViewer` reads `shows_internal.run_of_show` via the **service-role client** (bypasses RLS, `:200`/`:235`) **unconditionally for every viewer** (NOT `if(isLead)` like `financials` — `run_of_show` is date-gated, not lead-gated), then runs it through a **strict decoder** (below) and emits `ShowForViewer.runOfShow: Record<ISODate, AgendaEntry[]> | null` **after applying the viewer `DateRestriction` intersection** (D-4): the projection drops any ISO-date key the viewer is not entitled to see (same gate the Schedule day set uses).
- **`decodeRunOfShow` — deep validation, NOT a shallow `typeof === 'object'` guard (R14-MEDIUM).** The column is **schemaless JSONB** written by the sync, so the projection must not trust its shape — a buggy sync, a manual admin PostgREST edit (the §4.2 posture note allows admin DML), or future migration drift could store `{ "2026-01-01": [null] }`, a non-array day value, a non-ISO key, or an entry with non-string fields, and the §4.3 UI branch keys only off `runOfShow[isoDate]?.length > 0` — so an under-validated value turns corrupted storage into a **Schedule render crash** instead of the intended anchor-strip fallback. The decoder is **total** (never throws) and validates **structurally, per layer**:
  - **`null` → `runOfShow = null`, NO `tileErrors`** (the legitimate empty/no-agenda state — the common case).
  - **Top-level not a plain object** (array, string, number, boolean) → `runOfShow = null` **+ `tileErrors["run_of_show"]`** (corruption — the admin alert fires).
  - **Per key:** a key that is not a `YYYY-MM-DD` ISO date → that key **dropped** + corruption-flag set.
  - **Per day value:** not an array → that day **dropped** + corruption-flag set.
  - **Per entry:** must be a plain object whose **`title` is a REAL string** (non-empty AND not a generic sentinel — run through `shouldHideGenericOptional`, mirroring the parser's step-4 emit gate, so a corrupt write that injected `""`/`TBD` can't render) **and** whose every **present** optional field (`start`/`finish`/`trt`/`room`/`av`) is a string — any entry failing this is **dropped**; a day left with **zero** valid entries after filtering is **omitted** from the projected Record (→ that day falls back to the anchor strip).
  - **If ANY corruption was encountered** during decode (bad top-level handled above, or any dropped key/day/entry) → set **`tileErrors["run_of_show"]`** so the Phase-1 `CrewShell` projection alert observes it (§4.4) **while still projecting the valid days** (defense-in-depth: the operator is alerted to the corrupt write, the crew still sees the well-formed data; valid days are not blanked just because a sibling day was malformed). The fields the decoder admits are also **storage-capped at display** by §4.3/D-6 (the decoder validates type, the UI caps length) — the decoder does not need to re-truncate.

### §4.3 — Schedule-section enrichment (UI)

The Phase-1 **Schedule** `*Section` (Server Component, `components/crew/sections/`) gains a per-day branch:

- For each rendered Schedule day (already gated + capped by Phase 1), **if `runOfShow[isoDate]` is present and non-empty** → render a **run-of-show list** (`data-testid="run-of-show-<isoDate>"`): each entry is a row `START–FINISH · TITLE` with `ROOM` and an `AV` badge when present, in sheet order. **Else** → render the Phase-1 `resolveKeyTimes` anchor strip for that day (unchanged).
- **Mutually exclusive per day:** a day renders **either** the run-of-show list **or** the anchor strip, **never both** (D-5). The decision is `runOfShow[isoDate]?.length > 0`.
- **Sentinel hiding:** each optional field (`room`, `av`, `finish`, `trt`) routes through `shouldHideGenericOptional` (Phase 1's sentinel contract) — a `""`/`TBD`/`N/A`/`TBA` value hides that field, not the whole entry (the entry shows iff `title` is real, which the parser already guarantees). The Schedule section is already in the `_metaSentinelHidingContract` walk (Phase 1 extended it to `components/crew/`), so the new field reads are covered.
- **Caps (D-6):** at most 20 entries; a 21st+ collapses into a `+N more` overflow stub with the correct `data-testid`/count (`= length − 20`). `title` > 80 chars → `data-testid="agenda-title-truncated"` + `<details>` body. Migrate/extend the Phase-1 `CardinalityCapBoundary` matrix to the run-of-show list.
- **No links rendered raw — dedicated agenda sanitizer (R17/R18-MEDIUM).** Agenda `title`/`room`/`av` are free text that could paste a link. The DOM invariant is **precisely: no `https://`, `http://`, `drive.google.com`, or `docs.google.com` substring in the crew DOM** (the same contract `stripOpeningReelText` upholds for the opening reel). It is broader than that helper's `DRIVE_URL_RE` (Drive/Docs-only, `lib/visibility/openingReelText.ts:54`), so Phase 2 adds a **dedicated `stripAgendaUrls(value)`** helper (new, `lib/visibility/`) that strips (a) every **schemed** URL `https?:\/\/\S+` — covers Zoom/Teams/signed-CDN/Drive/Docs *with* a scheme, the overwhelmingly common pasted form — **and** (b) **scheme-less** `(drive|docs)\.google\.com\/\S+` (Doug's cells sometimes omit the scheme on Google links), then reuses the orphan-connector + whitespace cleanup. Each agenda string renders through it. **Documented limitation (R18 — do-not-relitigate toward a general bare-domain stripper):** a **scheme-less non-Google** bare-domain typed without `https://` (e.g. `zoom.us/j/123`, `teams.microsoft.com/l/…`) is **NOT** stripped — deliberately, because (i) pasted links carry a scheme in practice and (ii) a general `\w+\.\w+\/\S+` stripper would over-strip legitimate agenda text (`A/V`, a room labeled `5/6`, `Q&A w/ X`). The invariant is exactly "no schemed URL + no Google bare-domain substring", which `stripAgendaUrls` guarantees; widening it to arbitrary bare domains is out of contract (filed nowhere — the schemed-link coverage is the product-correct line).

### §4.4 — Error handling + observability

| Fault | Parser result | Stored `run_of_show` | Crew sees | Admin signal |
|---|---|---|---|---|
| All show-day grids empty, **never filled** (**most shows** — not yet locked) | every day `[]` | stays **`null`** (no-op — nothing stored to preserve) | anchor strip | none (the designed empty state) |
| A day **was filled**, now **READ-EMPTY** (grid located, titles blank — intentional removal **OR** a transient title-cell drop) | that day `[]` | that day **CLEARED** (others kept) | **anchor strip** for that day — never stale-rich (R21); if it was a transient drop, the next sync re-reads the still-filled sheet and re-populates | `ParseWarning` `AGENDA_DAY_EMPTIED` (the freshness signal) |
| Grid **not locatable** (no token-header — conversion fault OR a destructively-removed/renamed tab; indistinguishable) | `undefined` (nothing confirmed) | stored **`null`** (full replace — no day confirmed) | **anchor strip** for all days — never stale, regardless of removal shape (R22); a *transient* conversion fault re-confirms next sync | `ParseWarning` `AGENDA_GRID_MALFORMED` → `shows_internal.parse_warnings` (admin freshness signal; mass `AGENDA_GRID_MALFORMED` = a converter regression to roll back) |
| A single block unresolvable (`#REF!`+ambiguous/no day-name match, mis-aligned) | that day **absent** from the Record (not confirmed) | that day **not stored**; other confirmed days written | **anchor strip** for that day (never stale; R22) | `ParseWarning` `AGENDA_BLOCK_UNRESOLVED`/`AGENDA_DAY_AMBIGUOUS` |
| `shows_internal.run_of_show` **fetch** fails at read | — | (unchanged) | anchor strip (runOfShow → `null`) | Phase-1 `tileErrors["run_of_show"]` → `CrewShell` `TILE_PROJECTION_FETCH_FAILED` (union-merged; **no new code**) |
| stored value malformed at read (`decodeRunOfShow`, R14) | — | (unchanged — read-only) | **non-object top level →** anchor strip (all days); **a single corrupt key/day/entry →** anchor strip **for that day only**, valid sibling days still render their run-of-show | `tileErrors["run_of_show"]` whenever the decoder drops anything (same channel as fetch fail) — fail-soft, never a render throw |

**`run_of_show` is a FIRST-CLASS `failedKeys` domain, not a string piggyback (R20).** The Phase-1 `TILE_PROJECTION_FETCH_FAILED` alert's `failedKeys` = the keys of the render's `tileErrors` with **no alert-level allowlist** (Phase-1 spec §4.13 / wp-44/45), and `tileErrors` is typed `Record<string, string>` (`getShowForViewer.ts:168`, open-ended), so `tileErrors["run_of_show"]` flows into `failedKeys` automatically. Phase 2 makes that membership **explicit**: `run_of_show` is the **6th** projection-fetch domain (after `hotel`/`rooms`/`contacts`/`transportation`/`financials`) and is **viewer-independent** — the `shows_internal.run_of_show` read is unconditional (service-role, every viewer — D-4), like `transportation`/`hotel`/`rooms`/`contacts` and **unlike** the lead-gated `financials` — so a `run_of_show` fetch/decode failure enters `failedKeys` from **any** render (no lead-gated delay). **This needs NO new admin-alert code, NO §12.4 catalog-copy change** (the `TILE_PROJECTION_FETCH_FAILED` `helpfulContext` is domain-agnostic — the failed domains live in dynamic `context.failedKeys`, not the static copy — Phase-1 spec line 317), **NO `_metaAdminAlertCatalog` producer change** (it reuses the existing code/producer), and **NO `upsert_admin_alert` change** (the union-merge already handles arbitrary domains). It **DOES** require the **Phase-1 alert test that enumerates the observable `failedKeys` domains to be extended to include `run_of_show`** as a 6th viewer-independent member (so the contract isn't split-brain — a produced-but-unasserted domain). _Catches: `run_of_show` fetch/decode failures losing admin observability because the alert contract/tests treated the failedKeys domain set as a closed 5-element set._

**Invariants:** (1) a run-of-show fault **never** removes or corrupts the anchor strip — the Phase-1 floor; a day **without** a (confirmed) run-of-show always renders the anchor strip. (2) **CONFIRMED-ONLY — crew see a day's run-of-show iff the LATEST sync confirmed it (R17/R21/R22 — the canonical rule).** Confirmed = the latest parse produced non-empty entries for that day. **Every** non-confirmed outcome — read-empty `[]` (incl. an intentionally-removed agenda), an unresolved/ambiguous block, OR an unlocatable grid — is **not stored** and renders the **always-current anchor strip** (+ the matching admin warning: `AGENDA_DAY_EMPTIED` / `AGENDA_BLOCK_UNRESOLVED` / `AGENDA_GRID_MALFORMED`). Because nothing non-confirmed is shown, **no stale/preserved agenda can ever reach crew, regardless of how an agenda was removed** (blank titles, deleted tab, broken header, changed template — all coarsen to anchors). (3) **No preserve-and-show; no permanent loss either.** There is **no** "show last-known-good through a fault" path (that path leaked stale operational data — R21 closed the read-empty shape, R22 the unlocatable-grid shape; CONFIRMED-ONLY closes the whole class). The **sheet is the source of truth**, so a *transient* fault self-heals — the next sync re-reads and re-confirms; the only cost is a brief, safe coarsening to the correct anchors during the fault window. **v1 ships no crew-facing "stale" badge** (a non-confirmed day shows anchors, which are never wrong). **Do-not-relitigate:** asking for ANY preserve-and-keep-showing behavior on a non-confirmed day re-opens the R17/R21/R22 stale-operational-data class; asking for an admin "clear" gate is unnecessary (non-confirmed auto-coarsens to anchors next sync). (Tested: §9 — every non-confirmed shape — read-empty, unresolved, unlocatable — renders anchors + emits its warning; a confirmed re-sync restores the run-of-show.)

**Per-day data-retention matrix (the complete state space — the structural contract; review against THIS, don't hunt cells one at a time).** For a day `d`, given the parser's per-day outcome and whether `d` is in the current `dates.showDays`:

| Parser outcome for `d` | Storage (write/merge) | Crew (projection+UI), `d` ∈ current `dates.showDays` | Crew, `d` ∉ current `dates.showDays` |
|---|---|---|---|
| entries `E` (filled) | `d = E` (replace) | run-of-show list `E` | hidden (anchor strip) — storage keeps `E` |
| `[]` read-empty / blank-titles, `d` **never stored** | not confirmed → not stored | anchor strip | anchor strip |
| `[]` read-empty / blank-titles, `d` **previously stored** (intentional blanking OR transient title-drop) | not confirmed → **not stored** (full replace drops it) + `AGENDA_DAY_EMPTIED` | anchor strip (never stale; re-confirms next sync if transient) | anchor strip |
| **absent** — unresolved block | not confirmed → **not stored** + `AGENDA_BLOCK_UNRESOLVED` | anchor strip | anchor strip |
| whole grid `undefined` — unlocatable (conversion fault OR destructive tab removal) | nothing confirmed → stored **`null`** + `AGENDA_GRID_MALFORMED` | anchor strip (all days) | anchor strip |

One rule, both axes: **storage** (write) = **only the latest sync's confirmed (non-empty) days** — every non-confirmed outcome (read-empty, unresolved, unlocatable) is **not stored**, so the crew **only ever see confirmed run-of-show; never preserved/stale (R22)**. A filled day replaces. **No permanent loss** — the sheet is the source of truth, so a transient fault re-confirms on the next sync. **Visibility** (read) further intersects stored `∩ current dates.showDays ∩ DateRestriction` (removed/transient/forbidden *dates* hidden non-destructively — R10/R12/D-4; orthogonal to the confirmed/not-confirmed *content* axis). A confirmed day whose date transiently drops from `dates.showDays` stays stored (re-shown when the date returns); it's the only place storage outlives a single parse.

### §4.5 — Guard conditions (every input)

- `runOfShow = null` → every day renders the anchor strip (Phase-1 behavior, no Phase-2 element in the DOM).
- `runOfShow = {}` (empty object) → treated as `null` (no day has entries) → all anchor strips. (The parser stores `null` not `{}`, but the projection guards both.)
- `runOfShow[date] = []` → that day renders the anchor strip (empty array = no entries). (The parser omits empty-day keys, but the UI guards `?.length > 0`.)
- An entry with `title` present but every other field sentinel/blank → renders a title-only row (no time/room/av), never an empty row.
- `DateRestriction = unknown_asterisk` → projection drops all `runOfShow` keys → all visible days (none) render nothing extra; the Schedule unconfirmed-placeholder (Phase 1) is unchanged.

### §4.6 — Mode boundaries / dimensional invariants / transitions

- **Mode boundary:** the run-of-show list and the anchor strip are the two mutually-exclusive **per-day** modes of the Schedule day. No element is shared between them; the day container renders exactly one.
- **Dimensional invariants:** the run-of-show list is a flow list inside the existing Schedule day container (no fixed-height parent with flex/grid children → no Tailwind-v4 `items-stretch` concern); each entry row is `min-h-tap-min` (44px) if it carries a tap target, else natural height. (No `getBoundingClientRect` parent==child assertion is required because there is no fixed-dimension parent; the Phase-1 Schedule layout test already covers the day container.)
- **Transitions:** the per-day mode (run-of-show vs anchor strip) is **fixed at render** for a given day+viewer — it does not toggle client-side, so **no animation** (instant; nothing to inventory). Section-level crossfade is owned by Phase 1's `CrewSectionTransition`, unchanged.
