## 1. Problem

B2 shipped the complete undo backend: auto-publish (cron `lib/sync/runScheduledCronSync.ts` and staged-Apply `lib/sync/applyStaged.ts:1369-1430`) mints `shows.unpublish_token` + `unpublish_token_expires_at` (24h); `lib/sync/unpublishShow.ts` consumes the token inside `lockedShowTx` (sets `archived=true, published=false`, clears the token, raises `SHOW_UNPUBLISHED`); `POST /api/show/[slug]/unpublish?token=…` (`app/api/show/[slug]/unpublish/route.ts:8-27`) exposes it. **Nothing delivers the token to Doug.** The shipped UI copy promises the undo ("You can still undo within 24 hours" — `components/admin/settings/AutoPublishToggle.tsx:74`, `components/admin/StagedReviewCard.tsx:136`), the forbidden-prose registry bans help/catalog from repeating the promise precisely because no surface ships (`tests/help/forbidden-prose-registry.test.ts:96-99` `24-hour-undo-email`; `:108-111` `email-delivery-of-action-link`), and `SHOW_FIRST_PUBLISHED` is a severity=info alert the banner filters out — the event is near-invisible.

M12.2 B3 then shipped a full email subsystem (`lib/notify/`: Resend behind `send.ts`, cron entry `app/api/cron/notify/route.ts`, candidates → deliver with `email_deliveries` dedup, recipients = all non-revoked `admin_emails`, per-feature `app_settings` toggles). The original "no email infra" rationale is stale. Owner decision (2026-06-12): ship the full B2 vision — **email delivery via B3 plus an in-app undo surface** — making the existing copy true rather than rewording it.

## 2. Resolved decisions (owner, 2026-06-12)

1. **Scope: email + in-app** (over copy-only reword and over in-app-only).
2. **New dedicated toggle** `app_settings.alert_on_auto_publish` (default ON) gates the email — not the sync-problems toggle (mislabeled), not always-send.
3. **In-app surfaces: BOTH** the per-show footer actions row AND an inline action on the `SHOW_FIRST_PUBLISHED` alert row (over footer-only and over dashboard-level).
4. **Email link mechanics: token confirm page** (`GET` renders, `POST` mutates) — direct-GET mutation disqualified (mail-client prefetchers would consume the token); redirect-into-admin defeats the tokened-link purpose.
5. `AutoPublishToggle.tsx:74` / `StagedReviewCard.tsx:136` copy **stays untouched** — it becomes true as written (the cleanest DEF-1 resolution).

## 3. Architecture overview

Three entry points, ONE mutation path:

```
email link ──> GET /show/[slug]/unpublish?token=…  (confirm page, renders only)
                      └─ confirm button ─┐
per-show footer "Undo auto-publish" ─────┼──> unpublishShow({ slug, token })   ← lib/sync/unpublishShow.ts:251,
SHOW_FIRST_PUBLISHED alert row action ───┘        (inside lockedShowTx — existing single holder)
```

The in-app server actions read the STORED token service-role-side and call `unpublishShow({slug, token})` — never a tokenless bypass — so consumption semantics (single-use, expiry, `UNPUBLISH_TOKEN_CONSUMED`/`EXPIRED` outcomes) are identical on every path. The existing `POST /api/show/[slug]/unpublish` route stays as-is (B2-shipped surface; the confirm page uses a server action rather than fetching it, but both call the same function).

## 4. Email leg (B3 extension)

### 4.1 New candidate kind

`lib/notify/detect/` gains an `auto_publish_undo` candidate (4th member of `RealtimeCandidate` — current union at `lib/notify/detect/candidates.ts:41-44`: show/global/ingestion). Detection SQL (same direct-postgres pattern as the existing detectors, `candidates.ts:76-82` `databaseUrl()`):

```sql
select s.id as show_id, s.slug, s.title,
       s.unpublish_token as token,
       s.unpublish_token_expires_at as expires_at
  from public.shows s
 where s.unpublish_token is not null
   and s.unpublish_token_expires_at > now()
```

- **dedupKey = `${show_id}:${expires_at ISO}`** — distinct per mint (a later unarchive→auto-publish cycle gets a fresh email), stable across cron re-ticks within one window, and contains NO token material (the bearer token never enters `email_deliveries`; it appears only in the rendered email body/link).
- A candidate detected but already delivered is skipped by the existing `(kind, dedup_key, recipient)` unique-index upsert path (`lib/notify/deliver.ts:171-203`, index at `supabase/migrations/20260602000004_b3_email_deliveries.sql:19`).
- Detection reads ONLY (no locked-table writes; invariant 2 untouched).

### 4.2 Per-kind toggle gating in `runRealtimeNotify`

Today `runRealtimeNotify` short-circuits the ENTIRE realtime pass on the single `alert_on_sync_problems` toggle (`lib/notify/runNotify.ts:176-183`). Change: fetch BOTH toggles, then **filter candidates by kind→gate** — sync-problem/global/ingestion candidates require `alert_on_sync_problems`; `auto_publish_undo` candidates require `alert_on_auto_publish`. When both toggles are off → skipped with a combined reason; when one is off → that kind's candidates are dropped, the rest deliver. The existing skip reason `alert_on_sync_problems_off` keeps firing ONLY when sync-problem candidates were the ones dropped and nothing remained (exact result-shape compatibility pinned by the existing runNotify tests, extended).

### 4.3 Template + delivery

`lib/notify/templates/autoPublishUndo.ts` (mirrors `renderRealtimeProblem`'s `RenderedEmail` contract — `templates/realtimeProblem.ts:18,38`): subject "FXAV: <title> published itself"; body: which show, when the window closes (absolute time + "about N hours"), one primary link `${origin}/show/${slug}/unpublish?token=${token}`, one sentence on what undo does (takes the show offline; crew links switch off until republished), one sentence that ignoring it keeps the show live. `origin` comes from the existing `config.origin` (`runNotify.ts:198` deliver call). HTML + text parts, `escapeHtml` for interpolations.

`deliver.ts` gains the kind branch writing `email_deliveries` rows with `kind='auto_publish_undo'`, `show_id`, `triggered_codes=['SHOW_FIRST_PUBLISHED']`, context carrying slug/title/expiry (NOT the token). Idempotency keys via the existing `baseKey/reissueKey` (`lib/notify/idempotencyKey.ts:5-11`) — no new key scheme.

### 4.4 `email_deliveries.kind` CHECK migration (CHECK-matrix discipline)

Current inline CHECK: `kind in ('realtime_problem','digest')` (`20260602000004_b3_email_deliveries.sql:3`). New migration `2026….sql`:

```sql
alter table public.email_deliveries drop constraint if exists email_deliveries_kind_check;
alter table public.email_deliveries add constraint email_deliveries_kind_check
  check (kind in ('realtime_problem','digest','auto_publish_undo'));
```

Matrix: (a) kind values × CHECK — all three accepted, anything else rejected; (b) NULL kind — column is `not null`, unchanged; (c) transitional window — there is NO `supabase/tables/` mirror for this table (migrations are the only DDL), so no dual-accept window; (d) apply-twice idempotent via `drop … if exists` + `add`; (e) no one-shot migrations reference the old constraint. **Post-migration checklist applies** (AGENTS.md validation-schema-parity): local apply + `pnpm gen:schema-manifest` (commit the manifest) + surgical apply to the validation project (`supabase db query --linked` or psql with the pooler URL) + `notify pgrst, 'reload schema'`.

## 5. Confirm page (new public surface)

**Route:** `app/show/[slug]/unpublish/page.tsx`. Sits beside the crew route `app/show/[slug]/[shareToken]/page.tsx`; Next.js resolves static `unpublish` over dynamic `[shareToken]` deterministically, and a structural test pins that a share token can never collide (token format is generated, never the literal `unpublish`). The page carries NO `data-testid^="help-affordance--"` (the §5.6 crew negative row's sweep is route-scoped to the crew page, but keep the page clean regardless).

**GET (server component):** parse `token` from searchParams; look up the show by slug service-role-side and compare tokens WITHOUT consuming. Render states:
- **Token matches + unexpired:** show title + "Take this show offline?" + plain-language consequence line (crew links switch off until republished from the admin) + ONE confirm button (form bound to a server action) + a "Keep it live" secondary line stating that doing nothing leaves the show live. No admin chrome; minimal standalone layout (the page is reached from email on any device, unauthenticated).
- **Token absent/mismatched/show missing:** a neutral not-found state (no oracle about whether the slug exists — mirror the POST route's 404 posture, `route.ts:11-13,18-20`).
- **Token expired:** `UNPUBLISH_TOKEN_EXPIRED` catalog copy via `lib/messages/lookup.ts` (invariant 5 — never the raw code).
- **Token already consumed** (token columns null but show archived recently): the GET cannot distinguish "consumed" from "never existed" without the token — render the same neutral not-found state; the CONSUMED copy surfaces only on the POST path's result (where `unpublishShow` returns it).

**POST (server action on the page):** calls `unpublishShow({ slug, token })` (`lib/sync/unpublishShow.ts:251`) and renders the outcome in place: success → "**<title>** is now offline. Crew links are switched off; you can publish it again any time from the admin." (link text to the admin per-show page — plain href, fine for the signed-in device case, harmless otherwise); `UNPUBLISH_TOKEN_CONSUMED` / `UNPUBLISH_TOKEN_EXPIRED` → catalog copy; `not_found` → neutral state. **GET performs no mutation — pinned by test** (the concrete failure mode is a refactor to mutate on GET, which mail prefetchers would detonate).

**Guard conditions:** missing `token` param → neutral not-found; empty string → same; malformed slug → same; double-submit (action re-fires after success) → second call returns CONSUMED, rendered as its catalog copy — acceptable and tested.

## 6. In-app leg

### 6.1 Data

The per-show page's show select (`app/admin/show/[slug]/page.tsx:145-148`) adds `unpublish_token_expires_at` (NOT the token itself — the page never needs the secret client-side; the server action re-reads it). `undoWindowOpen = expires_at != null && expires_at > now`.

### 6.2 Footer action (per-show page)

In the quiet sync footer's right-hand action group (beside Archive/Re-sync, `page.tsx:616+`): an "Undo auto-publish" button rendered iff `undoWindowOpen && published && !archived`. Its server action (admin-gated like the sibling actions): service-role read of `unpublish_token` by show id → `unpublishShow({slug, token})` → revalidate; if the token vanished between render and click, the action surfaces the catalog outcome (no crash, no raw code). Plain secondary-button styling matching `ReSyncButton`'s register; instant appear/disappear (transition inventory: no animation — matches the footer's existing pattern; compound case archived-while-window-open is impossible post-undo since undo archives).

### 6.3 Alert-row action (`SHOW_FIRST_PUBLISHED`)

`components/admin/PerShowAlertSection.tsx`: rows whose `code === "SHOW_FIRST_PUBLISHED"` AND `undoWindowOpen` (the section receives the flag via props from the page) render the same "Undo auto-publish" action button after the message + existing affordances — the first action button in that section (new but minimal pattern: reuse the footer's server action; one shared component `components/admin/UndoAutoPublishButton.tsx` consumed by both surfaces so copy/behavior cannot drift). When the window closes or the show is manually published (no token minted — B2: deliberate manual publish mints no token, spec `…b2-show-lifecycle-design.md:158`), no action renders; the alert remains as the historical notice.

### 6.4 Mode boundaries

| Surface | renders iff |
| --- | --- |
| Footer undo button | `undoWindowOpen && published && !archived` |
| Alert-row undo action | same flag, and only on `SHOW_FIRST_PUBLISHED` rows |
| Confirm page confirm button | token param matches stored token and unexpired |
| Email | candidate detected, toggle ON, Resend configured |

Post-undo state: show is archived+unpublished → footer shows the existing archived affordances (Unarchive; "Re-sync is paused…"), both undo affordances gone, alert remains.
