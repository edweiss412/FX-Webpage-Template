## 3. Architecture overview

Three entry points, ONE mutation path:

```
email link ──> GET /show/[slug]/unpublish?token=…&r=…  (confirm page, renders only; r = recipient binding)
                      └─ confirm button ─┐
per-show footer "Undo auto-publish" ─────┼──> unpublishShow({ slug, token })   ← lib/sync/unpublishShow.ts:251,
SHOW_FIRST_PUBLISHED alert row action ───┘        (inside lockedShowTx — existing single holder)
```

The in-app server actions read the STORED token service-role-side and call `unpublishShow({slug, token})` — never a tokenless bypass — so consumption semantics (single-use, expiry, `UNPUBLISH_TOKEN_CONSUMED`/`EXPIRED` outcomes) are identical on every path.

**Atomic recipient re-validation on the PUBLIC legs (R12 — revocation and consumption must serialize):** the click-time binding checks in §5 are pre-checks only; a revocation committing between them and consumption would still lose. The public legs therefore call a new locked wrapper (e.g. `unpublishShowViaEmailedLink({slug, token, r})`) that, INSIDE `unpublishShow`'s existing `lockedShowTx` transaction, re-validates the binding **immediately after the show row is read and BEFORE every token-state branch** — before the null-token→CONSUMED return, the token compare, and the expiry branch (R18: `unpublishShow_unlocked`'s expired branch CLEARS the stored token and its consumed/expired returns leak state — an invalid/revoked `r` must reach NONE of them; binding failure exits with the neutral outcome, token state untouched and unlearned) . **Consumed-token contract on the public legs (R19):** when the stored token columns are already NULL, the current mint does not exist, so `r` is UNDERIVABLE — the wrapper exits with the NEUTRAL outcome before the live code's consumed branch (`unpublishShow.ts:192-198`), never CONSUMED (returning CONSUMED would require validating a binding that cannot be derived, or leaking state to unvalidated callers). `UNPUBLISH_TOKEN_CONSUMED` therefore renders ONLY on the session-authed in-app legs (which call plain `unpublishShow` and need no binding); the public double-submitter — who already saw success render in place — sees neutral on a re-submit. The binding check itself reads the unrevoked `admin_emails` rows **`FOR SHARE`** (a concurrent revocation UPDATE then blocks against this transaction or vice versa — true serialization, not a re-read), HMACing app-side against the tuple; no match → abort WITHOUT consuming, surfaced as the neutral state. The in-app legs keep calling plain `unpublishShow` (session-authed; the acting admin's own mid-flight revocation is governed by the existing `requireAdmin` session semantics — out of scope). Single-holder rule preserved: the wrapper adds statements inside the EXISTING lock holder, no new lock layer. Regression: start a revocation concurrently with the public POST → whichever serializes first wins, and a revoked-before-consume recipient NEVER consumes (token intact). The existing `POST /api/show/[slug]/unpublish` route REMAINS as a surface but its CONTRACT CHANGES (R8 — it must not survive as a binding bypass): it now requires `token` AND `r`, may pre-check `r` for cheap-fail UX, and consumes EXCLUSIVELY via `unpublishShowViaEmailedLink({ slug, token, r })` — the locked wrapper with in-transaction FOR-SHARE re-validation (R15: a pre-check + plain `unpublishShow` would leave the check-then-consume race on this leg) — rejecting bare slug+token POSTs with the neutral 404 WITHOUT consuming the token. Route-level regressions: bare-token rejection; concurrent revocation-vs-POST proving revoked-before-consume leaves the token intact; and invalid-`r` × {live, expired, already-consumed} token states ALL returning neutral with ZERO state change (R18 — in particular the expired-token-clearing side effect must not fire for an invalid `r`). Safe to change: the route shipped with B2 but has had no consumer; this milestone's confirm page is its first.

## 4. Email leg (B3 extension)

### 4.1 New candidate kind

`lib/notify/detect/` gains an `auto_publish_undo` candidate (4th member of `RealtimeCandidate` — current union at `lib/notify/detect/candidates.ts:41-44`: show/global/ingestion). Detection SQL (same direct-postgres pattern as the existing detectors, `candidates.ts:76-82` `databaseUrl()`):

```sql
select s.id as show_id, s.slug, s.title,
       s.unpublish_token as token,
       s.unpublish_token_expires_at as expires_at
  from public.shows s
 where s.unpublish_token is not null
   and s.unpublish_token_expires_at > now()
```

- **dedupKey = `${show_id}:${mintId}` where `mintId = sha256(unpublish_token) hex, first 16 chars`** — exact per mint (R6: `expires_at` has only JS millisecond precision at the mint sites, so a same-ms re-mint would collide; the token hash cannot), colon-free (R4: `deliver.ts:51-52` derives occurrence fields via `split(":").at(-1)`), and one-way (NO recoverable token material enters `email_deliveries` — the hash is non-reversible; the bearer token itself appears only in the rendered email link). The delivery row's `context` carries explicit `expires_at` AND `mintId` fields; reconciliation reads those fields, never parses the key. Regression tests: two mints sharing an `expires_at` millisecond produce distinct dedup keys; the context fields round-trip.
- A candidate detected but already delivered is skipped by the existing `(kind, dedup_key, recipient)` unique-index upsert path (`lib/notify/deliver.ts:171-203`, index at `supabase/migrations/20260602000004_b3_email_deliveries.sql:19`).
- Detection reads ONLY (no locked-table writes; invariant 2 untouched).

### 4.2 Per-kind toggle gating in `runRealtimeNotify`

Today `runRealtimeNotify` short-circuits the ENTIRE realtime pass on the single `alert_on_sync_problems` toggle (`lib/notify/runNotify.ts:176-183`). Change: fetch BOTH toggles, then **filter candidates by kind→gate** — sync-problem/global/ingestion candidates require `alert_on_sync_problems`; `auto_publish_undo` candidates require `alert_on_auto_publish`. When both toggles are off → skipped with a combined reason; when one is off → that kind's candidates are dropped, the rest deliver. The existing skip reason `alert_on_sync_problems_off` keeps firing ONLY when sync-problem candidates were the ones dropped and nothing remained (exact result-shape compatibility pinned by the existing runNotify tests, extended).

**Toggle infra-fault contract (R1 fix — bearer-token emails fail CLOSED):** each toggle getter's infra fault is handled independently, never by failing the whole pass open or closed for the other kind:

| `alert_on_sync_problems` read | `alert_on_auto_publish` read | behavior |
| --- | --- | --- |
| ok | ok | per-kind filtering as above |
| infra_error | ok | sync-problem/global/ingestion candidates DROPPED with a typed `infra_error` source (`getAlertOnSyncProblems`) recorded in the result; `auto_publish_undo` candidates still gate+deliver normally |
| ok | infra_error | `auto_publish_undo` candidates DROPPED (fail closed — never send a bearer-token email when the governing preference is unreadable) with typed source `getAlertOnAutoPublish`; sync-problem delivery proceeds normally |
| infra_error | infra_error | both kinds dropped; result carries both sources |

The result shape extends (typed, discriminable) rather than reusing the current single-source `infra_error` short-circuit (`runNotify.ts:177-179`); per-getter unit tests cover each row of this table, including the row where only one kind's candidates were present. **Cron-status visibility (R27):** the notify cron route returns 500 only on whole-pass `infra_error` today (`app/api/cron/notify/route.ts:16-19`) — a partial per-kind toggle fault recorded inside an otherwise-ok delivery would surface HTTP 200 while bearer emails were silently dropped. The extended `DeliverySummary` shape and the route's `statusFor` change together: ANY recorded toggle infra source (incl. partial per-kind faults) yields 5xx so the scheduler sees the degradation; deliberate toggle-OFF skips remain 200. Notify-route tests cover both. The SAME visibility rule applies to the MAINTENANCE side (R28): a toggle fault recorded during reconciliation (the §4.3b tri-state `unknown`) surfaces in the maintenance step's result such that `statusFor` yields 5xx — either by keeping the maintenance step's `kind: "infra_error"` while carrying the partial reconciliation detail, or by extending `statusFor` to inspect recorded toggle sources in both delivery AND maintenance; the plan picks one, tests pin it.

### 4.3 Template + delivery

`lib/notify/templates/autoPublishUndo.ts` (mirrors `renderRealtimeProblem`'s `RenderedEmail` contract — `templates/realtimeProblem.ts:18,38`): subject "FXAV: <title> published itself"; body: which show, when the window closes (absolute time + "about N hours"), one primary link `${origin}/show/${slug}/unpublish?token=${token}&r=${recipientId}` — **recipient-bound (R6, hardened R7, capability-scoped R10)**: `r = HMAC-SHA256(server pepper, canonical recipient email || show_id || mintId) hex, first 16 chars` — `admin_emails` has no surrogate id (`email` is the PK — `20260514000000_admin_emails_runtime_mutable.sql:16-17`); a raw email must not enter URLs (PII); a plain unkeyed hash would be guessable by anyone knowing another admin's address (R7); and an email-only HMAC would be a long-lived reusable credential valid across every show and future mint (R10) — scoping the HMAC input to the specific show + mint makes `r` a single-capability credential that dies with the token (post-undo/expiry/re-mint, every old `r` is dead by construction). Pepper = `HASH_FOR_LOG_PEPPER`, EXACTLY (R23 — pinned, no sibling env): concretely (R24): `lib/email/hashForLog.ts` exports a server-only `hmacWithHashForLogPepper(input: string): string` (HMAC-SHA256 over its already-validated module-private pepper — the module-load gate throws on missing/<32-char values, `hashForLog.ts:3-10`, so an unset/short pepper is fail-closed at BOOT), `hashForLog` itself refactors onto the same validated source, and the binding helper calls the exported HMAC — never reading `process.env` directly (structural test pins both: no direct env read in the binding path, and the HMAC-SHA256 algorithm); underivable without the server secret; non-reversible — one sentence on what undo does (takes the show offline; crew links switch off until republished), one sentence that ignoring it keeps the show live. `origin` comes from the existing `config.origin` (`runNotify.ts:198` deliver call). HTML + text parts, `escapeHtml` for interpolations. **Per-recipient rendering seam (R17):** the live loop renders ONCE per candidate before iterating recipients (`rendered(candidate, origin)` at `lib/notify/deliver.ts:338-350`; the render signature has no recipient — `:221-239`), which cannot produce a recipient-bound `r`. For `auto_publish_undo`, rendering moves INSIDE the per-recipient path (within `deliverOneRecipient`, after recipient canonicalization and active-recipient validation), so each recipient's email carries their own `r`. Regression: two active recipients → two DISTINCT URLs (per-recipient `r`), and the per-recipient property that matters (R25): revoking recipient B kills B's `r` while A's keeps working — there is deliberately NO cross-recipient-swap failure requirement, because any ACTIVE admin's `r` is a valid capability by design (the validation rule accepts any unrevoked row matching the tuple; using another active admin's `r` IS using their live capability, which the trust model permits). The failing cases remain: revoked recipient's `r`, other-show `r`, prior-mint `r`.

`deliver.ts` gains the kind branch writing `email_deliveries` rows with `kind='auto_publish_undo'`, `show_id`, `triggered_codes=['SHOW_FIRST_PUBLISHED']`, and **context = `{ slug, title, expires_at, mintId }` exactly** (R14 — `mintId` is the field §4.3b's reconciliation keys on; NO raw token field). Tests assert BOTH directions: no token material anywhere in the row, AND `mintId` present — plus the reconciliation regression that a failed row WITHOUT `context.mintId` is treated as non-current while a matching-`mintId` row stays current. Idempotency keys via the existing `baseKey/reissueKey` (`lib/notify/idempotencyKey.ts:5-11`) — no new key scheme.

**Deliver-time currentness guard (R1 fix — the detection→delivery race):** `deliver.ts` re-validates every realtime candidate immediately before sending (`isCandidateCurrent`), and that function knows only the three existing shapes. It gains an `auto_publish_undo` branch: re-read the show and require same `show_id`, same `unpublish_token`, same `unpublish_token_expires_at`, `expires_at > now()`, `published = true`, `archived = false` — otherwise SKIP the send (the token was consumed via an in-app undo, expired, or re-minted between detection and this recipient's send; a re-mint produces its own candidate next tick). Required test: consume the token (in-app path) between detection and delivery and assert NO email is sent and no `email_deliveries` row is written for the stale candidate.

### 4.3b Delivery-failure reconciliation (R2 fix — the kind-aware maintenance layer)

`deliverOneRecipient` opens `EMAIL_DELIVERY_FAILED` on failed sends (`lib/notify/deliver.ts:298-306`), and the maintenance pass reconciles those alerts kind-awarely — today ONLY for the two existing kinds: `EmailDeliveryStateInput` carries just `alertOnSyncProblems`/`dailyReviewDigest` (`lib/notify/detect/emailDeliveryFailed.ts:20-23`), `hasCurrentFailed` treats only `realtime_problem`/`digest` rows as current (`:101-167`), and `EMAIL_NOT_CONFIGURED` opens only when one of those two toggles is enabled (`:209-210`). Without extension, a failed undo email's alert is auto-resolved as non-current on the next tick, and a misconfigured-Resend install with only auto-publish emails enabled never learns it. Contract:

- `EmailDeliveryStateInput` gains `alertOnAutoPublish` — and (R21) ALL THREE channel fields change shape from boolean to a tri-state `{ kind: "enabled" | "disabled" | "unknown" }` (optionally carrying the fault source), because the R11/R16 unknown semantics cannot survive a boolean input: collapsing a faulted read to `false` incorrectly resolves shared alerts / suppresses `EMAIL_NOT_CONFIGURED`, collapsing to `true` pins stale alerts open. `runMaintenance` passes toggle faults through as `unknown` (no coercion); every rule below reads the tri-state. Tests: unknown channels prevent shared-alert resolution AND keep `EMAIL_NOT_CONFIGURED` open under invalid config; the two pre-existing channels' callers migrate shape in the same commit (their existing boolean tests update mechanically).
- `hasCurrentFailed` treats a failed `auto_publish_undo` row as CURRENT while: the toggle is ON, **the failed row's OWN recipient is still an active admin** (`admin_emails.email = e.recipient and revoked_at is null` — the existing per-row strictness at `emailDeliveryFailed.ts:95-100`, R4: "any active recipient exists" would keep a revoked recipient's failure open forever with no possible later success), AND the underlying window is still undoable — predicate: `sha256(show.unpublish_token)` prefix EQUALS the delivery row's `context.mintId` (exact mint identity; the reconciler reads the show row service-role-side and hashes in memory — the secret still never persists), AND `context.expires_at > now()`, with `published=true`, `archived=false`. Regression test: failed recipient revoked while another admin stays active → the alert resolves. **Mint identity is `context.mintId` ALONE (R3 as amended by R6/R13):** `email_deliveries` never stores raw token material; the reconciler establishes exact mint identity by hashing the LIVE `shows.unpublish_token` in memory and comparing to `context.mintId` — `context.expires_at` is ONLY the window timestamp for the `> now()` check, never an identity key (it cannot be: same-ms re-mints share it — regression: two mints sharing an `expires_at` millisecond, only the matching `mintId` keeps the failure current). This is the post-hoc counterpart of §4.3's DELIVER-time guard, where the in-memory candidate still carries the token and full token equality IS checked. It stops being current — and the alert resolves on the normal maintenance path — when the window expires, the token is consumed (in-app undo or a later successful send+click is irrelevant; consumption is what clears it), the toggle turns off, or a subsequent send for the same `(dedup_key, recipient)` succeeds.
- `EMAIL_NOT_CONFIGURED` gating becomes "config invalid AND any of the THREE toggles enabled" — **with the same tri-state unknown semantics (R16):** a faulted toggle read contributes "unknown" for its channel; the global config alert (single alert opened/resolved from the toggle booleans — `emailDeliveryFailed.ts:209-213`) OPENS when config is invalid and any channel is known-enabled OR unknown, and may RESOLVE only when config is valid or every channel is KNOWN disabled. Regression: config invalid + `alert_on_auto_publish` read faults + both other toggles known-off → the alert opens/stays open (the broken bearer-email channel must not be hidden by its own toggle's fault).
- **Maintenance toggle-fault independence (R5):** live `runMaintenance` treats either existing toggle fault as a whole-`emailDelivery` infra_error before reconciliation (`lib/notify/runNotify.ts:152-164`). The extended contract mirrors §4.2's per-kind independence: a fault reading ONE toggle suppresses reconciliation/gating only for the channels that toggle governs (typed source recorded), while channels whose toggles read cleanly reconcile normally — an unreadable `alert_on_sync_problems` must not hide a broken undo-email channel whose own toggle is readable and enabled, and vice versa. **Shared-alert resolution rule (R11):** `EMAIL_DELIVERY_FAILED` is ONE per-scope alert shared by all channels (`emailDeliveryFailed.ts:60-76`, `:191-205`), so per-channel independence must not let a clean channel resolve a scope whose OTHER channel's state is unknown: a channel whose toggle read faulted contributes "unknown" — the scope's alert may RESOLVE only when EVERY channel contributing to that scope is KNOWN non-current; any unknown channel leaves the alert untouched (open stays open). Regressions: open realtime failure + sync-problems toggle FAULTS + undo channel clean → alert stays open; the symmetric undo-channel case; all-known-non-current → resolves. Per-getter fault tests cover each of the three toggles independently faulting.
- Tests: failed undo send keeps the alert open while the window is live; resolves on expiry; resolves on toggle-off; resolves on later success; config-invalid + only `alert_on_auto_publish` enabled OPENS `EMAIL_NOT_CONFIGURED` (and the inverse: all three off → does not open).

### 4.4 `email_deliveries.kind` CHECK migration (CHECK-matrix discipline)

alter table public.email_deliveries drop constraint if exists email_deliveries_kind_check;

## 5. Confirm page (new public surface)

**Recipient binding on the PUBLIC consumption legs (R6 — revocation must retract emailed capability):** an admin revoked after receiving the email must NOT retain the undo capability for the rest of the window — every other admin surface dies at revocation instantly; this one must too. Contract: the public legs (this confirm page's GET render-confirm state, its POST action, AND the pre-existing `POST /api/show/[slug]/unpublish` route) require `token` AND `r`, and consume only when some `admin_emails` row with `revoked_at is null` satisfies `HMAC(pepper, canonical email || show_id || mintId-of-the-CURRENT-stored-token)` prefix-matching `r` (checked at CLICK time, both on GET-render and again inside the POST action; the table is operator-scale, so HMACing each unrevoked row per request is trivial — and because the tuple includes the CURRENT mint, the check inherently fails after consumption/expiry/re-mint). A missing/unknown/revoked/stale `r` renders the neutral not-found state (no revocation oracle). Regressions: (R7) a revoked recipient holding the token AND knowing every other admin's email cannot construct a working `r` (pepper is server-side only); (R10) an `r` minted for ANOTHER show, or for a PRIOR mint of the same show, fails on every public leg without consuming the token.

**Token-at-rest leak closed (R7 side-discovery):** the B2-era `SHOW_FIRST_PUBLISHED` alert producer persists the RAW `unpublish_token` (+ expiry) into `admin_alerts.context` (`lib/sync/runScheduledCronSync.ts:1674-1680`, mirrored in `applyStaged`'s reuse of the same producer args) — a bearer secret at rest in a table every admin session reads, and exactly how a soon-revoked admin learns the token. This milestone STOPS writing the token into alert context (expiry may stay; the in-app alert-row action never needed the context token — its server action re-reads the show row) and ships a one-shot data migration nulling `context->>'unpublish_token'` on existing `SHOW_FIRST_PUBLISHED` alert rows. Test: the producer's context carries no token field; the migration is apply-twice idempotent. **Token-context lockstep sweep (R22):** the raw-token-in-context contract is pinned in canonical B2 prose AND live tests, all of which move in the SAME commit as the producer change: the B2 spec's token-context statements (`2026-06-01-m12.2-phase-b2-show-lifecycle-design.md:237` and the `:487` test requirement) gain amendment notes; the three test files asserting the raw token in alert context (`tests/sync/runScheduledCronSync.adapter.test.ts:171-178`, `tests/sync/applyStaged.test.ts:434-437`, `tests/sync/runManualStageForFirstSeen.test.ts:343-365`) flip to asserting context contains ONLY non-secret fields (expiry retained); and a structural grep guard (in the producer's test file or a meta-test) rejects `unpublish_token` reappearing in any `SHOW_FIRST_PUBLISHED` producer context expectation — so the leak cannot be resurrected by following stale B2 docs. The in-app legs are session-authed and carry no `r`. The raw POST route's contract change (it previously accepted bare slug+token) is safe: it shipped with B2 but has had NO consumer until this milestone — note it in the route's comment and §12.4 is unaffected (the route's outcomes are unchanged codes). Regression test: revoke-after-send → both public legs deny (neutral state), in-app legs still work for a different signed-in admin; un-revoked recipient → consumes normally.

**Route:** `app/show/[slug]/unpublish/page.tsx`. Sits beside the crew route `app/show/[slug]/[shareToken]/page.tsx`; Next.js resolves static `unpublish` over dynamic `[shareToken]` deterministically, and a structural test pins that a share token can never collide (token format is generated, never the literal `unpublish`). The page carries NO `data-testid^="help-affordance--"` (the §5.6 crew negative row's sweep is route-scoped to the crew page, but keep the page clean regardless).

**GET (server component):** parse `token` AND `r` from searchParams. **Exact non-consuming evaluation order (R11 — the binding's HMAC inputs come from the show row, so the row fetch necessarily precedes binding validation; the TOKEN comparison stays last so binding failures never leak token validity):** (1) absent `token` or `r` param → neutral not-found, nothing fetched; (2) fetch the show by slug service-role-side SOLELY to obtain `show_id`, the current stored token (for `mintId` derivation in memory) and expiry — returned/thrown fault → the retry state; show missing or no live token → neutral not-found; (3) validate `r` against unrevoked `admin_emails` rows via the §4.3 tuple HMAC — fault → retry state; no match → neutral not-found, with the SUBMITTED token never compared and nothing logged that distinguishes binding-fail from not-found; (4) only after the binding validates, compare the submitted token to the stored token (constant-time) → match+unexpired renders confirm; mismatch → neutral; expired → the EXPIRED catalog state. Render states (all below require steps 1-3 passed except the infra state, which may arise from any lookup):
- **Binding valid + token matches + unexpired:** show title + "Take this show offline?" + plain-language consequence line (crew links switch off until republished from the admin) + ONE confirm button (form bound to a server action) + a "Keep it live" secondary line stating that doing nothing leaves the show live. No admin chrome; minimal standalone layout (the page is reached from email on any device, unauthenticated).
- **Token absent/mismatched/show missing:** a neutral not-found state (no oracle about whether the slug exists — mirror the POST route's 404 posture, `route.ts:11-13,18-20`).
- **Token expired:** `UNPUBLISH_TOKEN_EXPIRED` catalog copy via `lib/messages/lookup.ts` (invariant 5 — never the raw code).
- **Token already consumed** (token columns null): neutral not-found on GET AND POST — with the mint gone, `r` is underivable, so the public legs can never reach a CONSUMED render (§3's R19 contract; the CONSUMED catalog copy is in-app-only).

**POST (server action on the page):** RE-validates the recipient binding (`r`, carried through the form payload) against unrevoked `admin_emails` BEFORE any token use (R9 — the GET-time check does not protect the POST; revocation between render and click must deny) — binding failure → neutral state, token NOT consumed — then calls the locked wrapper `unpublishShowViaEmailedLink({ slug, token, r })` (§3 — which re-validates the binding atomically inside the transaction; the pre-check exists only for cheap-fail UX) and renders the outcome in place — success, EXPIRED (catalog copy; the expired branch still holds a stored token so the binding IS derivable), neutral (includes consumed/cleared per §3's R19 contract — CONSUMED never renders publicly), or the retry state: success → "**<title>** is now offline. Crew links are switched off; you can publish it again any time from the admin." (link text to the admin per-show page — plain href, fine for the signed-in device case, harmless otherwise); `UNPUBLISH_TOKEN_EXPIRED` → catalog copy; consumed/cleared AND `not_found` → the neutral state (R20: `UNPUBLISH_TOKEN_CONSUMED` never renders or returns on ANY public leg — the API route's current behavior of returning non-success codes, `route.ts:23-25`, and its test pin of CONSUMED, `tests/api/show-unpublish-route.test.ts:37-52`, both change under the §7 route-contract lockstep; new API regression: token+old-r after consumption → neutral 404, no code in the body). **GET performs no mutation — pinned by test** (the concrete failure mode is a refactor to mutate on GET, which mail prefetchers would detonate).

**Infra-fault state (R5 — invariant 9):** the GET's service-role show/token read distinguishes returned-error and thrown-error from benign outcomes. A DB/config fault renders a DISTINCT retry state — plain-language copy ("We couldn't check this link just now. Nothing has changed — try again in a minute."), no raw codes, no oracle — and NEVER the neutral not-found / expired / consumed states (a Doug mid-recovery-window must not be told his link is dead because of a transient fault). Both in-app server actions (footer + alert row) carry the same contract for their service-role token read AND for a non-outcome failure of `unpublishShow` itself: a typed infra result rendered as retryable copy, never a benign state, never a silent no-op. Tests pin BOTH fault shapes (returned `{ error }` and thrown) per surface → retry state, with negative assertions that not_found/consumed/expired do NOT render.

**Guard conditions:** missing `token` OR missing `r` → neutral not-found (R9 regression set: bare slug+token never renders the confirm button, never calls `unpublishShow`, never consumes the token — on the page AND the API route); empty strings → same; malformed slug → same; double-submit (action re-fires after success) → the second call finds the token columns NULL, the binding underivable, and renders NEUTRAL (R19 — the submitter already saw success render in place); tested.
