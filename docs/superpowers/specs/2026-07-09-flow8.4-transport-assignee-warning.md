## 1. Problem

A crew member assigned as a show's driver should see their own ground-transport itinerary (pickup times, vehicle, parking) on their crew page. `transportTileVisible` (`lib/visibility/scopeTiles.ts:177-203`) decides "is this viewer the driver / an assignee?" by fuzzy name match — `namesRefer(transportation.driver_name, viewerName)` and `schedule[*].assigned_names` (`scopeTiles.ts:190-202`). `namesRefer` (`lib/data/nameMatch.ts:63-65`) tolerates first-name / nickname / legal-name / case / trim variance, so the common cases already resolve.

It does **not** tolerate a **hard mis-parse** — a merged-cell overflow that fuses an adjacent spreadsheet cell onto the name, e.g. a driver exported as `"Doug Larson Loadout"` against roster `"Doug Larson"`. `refersSingle` takes the both-multi-token branch (`nameMatch.ts:53`) and compares only the last tokens — `tokCompat("loadout", "larson")` (`nameMatch.ts:37-39`) is neither equal nor a prefix, so `namesRefer` returns `false`. Result: the driver-crew-member silently does not see their own transport block, **and no operator is warned anything is wrong.** The failure is silent — that is the harm this spec removes.

### What this spec is NOT

The audit/backlog headline ("resolve `driver_name`/`assigned_names` to `crew_member` ids at enrich, persist the id set, match visibility by id") rests on two premises the live code invalidates:

1. **No crew_member id exists at enrich time.** `ParseResult.crewMembers: CrewMemberRow[]` (`lib/parser/types.ts:410`) carries `{ name, email, phone, role, role_flags, date_restriction, stage_restriction, flight_info }` (`types.ts:84-93`) — **no `id`**. The stable uuid is DB-assigned at APPLY (`crew_members.id uuid ... default gen_random_uuid()`, `supabase/migrations/20260501000000_initial_public_schema.sql:32`; written by `upsertCrewMembers`, `lib/sync/applyParseResult.ts:129`). There is nothing to persist at enrich.
2. **Match-by-id at render is behavior-neutral.** Resolving `driver_name` → roster id at render would use the *same* `namesRefer` binding that already fails on `"Doug Larson Loadout"`, so it closes nothing.

Persisting ids would require APPLY-time resolution + a DB migration, and even then the mis-parsed name resolves to no id at apply too — the driver stays invisible until an operator fixes the source data. **The operator warning is the only mechanism that closes the silent gap in every design.** This spec ships exactly that mechanism and nothing else.

**Explicitly out of scope** (remain in `BL-TRANSPORT-ID-RESOLUTION`, targeted for the later 8.3 enrich spec): DB migration, persisted resolved-id columns, apply-time id resolution, any change to `transportTileVisible` or the crew-page render, any new `admin_alert` code, any new push/email/telemetry surface. Flow 8.3 (venue timezone) is a separate later spec.

## 2. Design

### 2.1 New enrich step — `lib/sync/enrichTransportAssignees.ts`

A pure, synchronous, best-effort enrichment step. It performs **no I/O** (no Drive, no Supabase — therefore not subject to the `_metaInfraContract` Supabase call-boundary meta-test), reads `result.crewMembers` + `result.transportation`, and mutates `result.warnings` only. It never throws.

```ts
export function enrichTransportAssignees(result: ParseResult): void
```

Called from `enrichWithDrivePins` (`lib/sync/enrichWithDrivePins.ts`) immediately after `await enrichVenueGeocode(result)` (`enrichWithDrivePins.ts:423`), before `return result` (`:425`). No `await` (the function is synchronous). Placing it in `enrichWithDrivePins` means all sync paths (onboarding scan + cron) inherit it, exactly like `enrichAgenda` / `enrichVenueGeocode` (`enrichWithDrivePins.ts:413-423`).

### 2.2 The predicate — near-miss AND not-cleanly-resolved

For each **candidate assignee name** — `transportation.driver_name` (when non-null) and every string in every `schedule[*].assigned_names` (`types.ts:187-192`) — classify it. Two primitives:

- **Significant tokens** of a name = normalized (lowercase, diacritic-folded, suffix-stripped like `nameMatch.ts:17` `SUFFIX_RE`) tokens of length ≥ 2. Computed by a helper **private to the enrich module** (§2.3) — the length ≥ 2 filter never touches `namesRefer`, which keeps one-letter tokens for initial matching.
- **`namesRefer(roster, candidate)`** — the unchanged fuzzy matcher (`nameMatch.ts:63`). Its multi-token rule is **surname-only** (`nameMatch.ts:53`), which is the lenient behavior that both (a) resolves nicknames/initials and (b) causes the overflow-mismatch this predicate must catch.

Classification of a non-sentinel candidate `C`:

1. **Skip (sentinel)** if `shouldHideGenericOptional(C)` is true (empty / `TBD` / `N/A` / `TBA` / `-` / `—` — `lib/visibility/emptyState.ts:79`, backed by `GENERIC_OPTIONAL_HIDE` at `:56`, the same helper Flow 8.1 used). Not a real assignment.
2. **Not a near-miss → skip** if `C` shares **no** significant token with **any** roster member. This is an unrelated external / charter driver (`"ABC Charters"`) — warning on it would be chronic noise, so it is silently ignored.
3. **Cleanly resolved → no warn** if there exists a roster member `M` with **both** `namesRefer(M.name, C)` **and** `significantTokens(C).length <= significantTokens(M.name).length`. The token-count guard is load-bearing: it admits abbreviations / nicknames / variants (which have **no more** significant tokens than the member they match) while **rejecting overflow fusions** (which have strictly more).
4. **Otherwise → warn.** `C` shares a token with the roster (rules out external) but no roster member resolves it cleanly — it is either unresolved-but-near (`"Doug Larson Loadout"`) or an overflow that only surname-matches a wrong/partial member (`"Doug Larson John Smith"`).

**Why the token-count guard (R2 finding).** The old "resolved to anyone → safe" gate was wrong: a merged-cell fusion `"Doug Larson John Smith"` (roster has both `"Doug Larson"` and `"John Smith"`) surname-matches `"John Smith"` — verified `namesRefer("Doug Larson John Smith","John Smith") === true`, `namesRefer(...,"Doug Larson") === false`. So it "resolved," suppressing the warning, while Doug still could not see his transport and John saw the wrong itinerary. The overflow signature is **more significant tokens than the matched member**: `sig("Doug Larson John Smith") = 4 > sig("John Smith") = 2` → not clean → warn. A nickname variant `"Bill Werner"` vs `"William Werner"` has **equal** counts (`2 == 2`) → clean → no warn.

Worked examples (all `namesRefer`/count values verified against the live matcher):

| Candidate | Roster has | resolves via `namesRefer` to | sig-token counts | Outcome |
|---|---|---|---|---|
| `Doug Larson Loadout` | `Doug Larson` | (none — surname `loadout`≠`larson`) | 3 vs — | **warn** (near, unresolved) |
| `Doug Larson John Smith` | `Doug Larson`, `John Smith` | `John Smith` only | 4 > 2 | **warn** (overflow, wrong recipient) |
| `Doug` | `Doug Larson` | `Doug Larson` | 1 ≤ 2 | clean → no warn |
| `Douglas Larson` | `Doug Larson` | `Doug Larson` | 2 ≤ 2 | clean → no warn |
| `Bill Werner` | `William Werner` | `William Werner` | 2 ≤ 2 | clean → no warn |
| `J` | `John Larson` | `John Larson` (initial) | 0 ≤ 2 | clean → no warn |
| `  doug larson ` | `Doug Larson` | `Doug Larson` | 2 ≤ 2 | clean → no warn |
| `ABC Charters` | `Doug Larson` | — | shares 0 tokens | skip (external) |
| `` / `TBD` / `—` | any | — | — | skip (sentinel) |

### 2.3 Token primitive — private to the enrich module, `nameMatch.ts` untouched

**`lib/data/nameMatch.ts` is NOT modified.** `namesRefer` / `toks` / `refersSingle` keep their exact current behavior — critically, `toks` (`nameMatch.ts:22-33`) retains one-letter tokens so `namesRefer` still matches initials by prefix (an initial-only assignee resolves). Refactoring the shared matcher to drop one-letter tokens would silently change hotel AND transport visibility globally; this spec forbids that.

The near-miss check uses a **separate, private** helper inside `lib/sync/enrichTransportAssignees.ts` (not exported, not shared with `namesRefer`):

```ts
// private to enrichTransportAssignees.ts
function significantTokens(s: string): string[] // lowercase, strip SUFFIX_RE-style suffixes, keep tokens length >= 2
```

`enrichTransportAssignees` calls the unchanged `namesRefer` for the **resolved?** check and its own `significantTokens` for the **near-miss?** check. The two concerns are deliberately decoupled: `namesRefer` decides visibility (initials included); `significantTokens` decides whether an unresolved name looks like a garbled crew name (initials excluded, since a lone initial is not a meaningful near-miss signal). An initial-only-matching regression test (§7 test 9) pins that `namesRefer` behavior is unchanged.

### 2.4 The warning — one aggregate per show

The step emits **at most one** warning per show (keeps the data-gap `total` / badge count to +1, one staged-review row):

```ts
result.warnings.push({
  severity: "warn",
  code: "TRAVEL_TRANSPORT_NAME_UNMATCHED",
  message: `Transport names people not on the crew list: ${quotedList} — check the spelling or add them to the crew.`,
});
```

- `quotedList` = the distinct **warned** names (the §2.2 rule-4 set — near-miss AND not-cleanly-resolved), de-duplicated by normalized form, in first-seen order, each wrapped in straight double-quotes, joined by `, `. **Capped at 5 names**; a 6th-and-beyond count is appended as `, and N more` (N = distinct-count − 5).
- The message is **plain language, never the raw code** (invariant 5). Warnings live in `shows_internal.parse_warnings` and are **admin-only** (spec `docs/superpowers/specs/2026-04-30-fxav-crew-pages-v1.md:645-649`); the crew page renders zero warnings. Naming the assignee is safe — the operator already sees these names in their own sheet. No crew-facing copy (`crewFacing: null`).
- Emitted only when ≥ 1 warned name exists; zero warned names → no warning pushed (idempotent, silent-clean).

### 2.5 Guard conditions

| Condition | Behavior |
|---|---|
| `result.transportation === null` | return, no warning (nothing seeded) |
| `result.crewMembers.length === 0` | return, no warning (cannot resolve against an empty roster; every name would spuriously "not match") |
| `driver_name === null` | driver contributes no candidate |
| `assigned_names === []` (a leg) | that leg contributes no candidates |
| candidate is a sentinel (`shouldHideGenericOptional`) | skipped |
| candidate shares no significant token with any roster member | skipped (external driver) |
| candidate cleanly resolved (`namesRefer` to a member M **and** `sig(C) ≤ sig(M)`) | not warned |
| candidate near-miss but not cleanly resolved (unresolved, OR overflow `sig(C) > sig(M)`) | **warned** |
| all candidates clean / external / sentinel | no warning pushed |
| any internal throw | swallowed (`try/catch`), no warning — best-effort, never breaks the sync |
| > 5 distinct warned names | message shows first 5 + `, and N more` |

### 2.6 Flag lifecycle — `gateExempt`

`TRAVEL_TRANSPORT_NAME_UNMATCHED` is registered in `GAP_CLASSES` (`lib/parser/dataGaps.ts:30-65`) with **`gateExempt: true`**.

| Column | Value |
|---|---|
| **storage** | `GAP_CLASSES` entry `{ code, label, gateExempt: true }` (`dataGaps.ts`) |
| **write path** | `result.warnings.push(...)` in `enrichTransportAssignees.ts` → persisted to `shows_internal.parse_warnings` on apply |
| **read paths** | `isDataQualityWarning` / `DATA_GAP_CODES` gate (`dataGaps.ts:88-90`) → staged-review card (`app/admin/show/staged/[stagedId]/page.tsx:164-176`, renders `.message`), wizard Step 3, changes feed, per-show panel, `summarizeDataGaps` count |
| **effect of `gateExempt`** | counts toward the data-gap `total` (badge/chip/digest-visible) but **never trips the push-alert regression gate** (`isQualityRegression` / `buildRegressionPayload`) — honored via the `gateExempt` flag, same as `VENUE_GEOCODE_UNRESOLVED` (`dataGaps.ts:56-60`) |

**Why `gateExempt: true`:** the user-selected shape is "staged-review page flags it" — a passive, operator-checked data-quality signal, not a push. Staged/badge visibility comes from `GAP_CLASSES` membership and is independent of `gateExempt`; `gateExempt` governs only push-regression participation. `true` keeps this spec's promise of **no new push/telemetry surface** (email delivery is config-blocked anyway — `lib/notify/config.ts:7`). See §6 Watchpoints for the reviewer preempt on this choice (it is a deliberate scope decision, not an oversight that this class is not transient).

## 6. Watchpoints (disagreement-loop preempts for the reviewer)

- **`gateExempt: true` is deliberate, not an oversight.** The reviewer may argue a non-transient data issue "should" trip the push-regression gate (unlike the transient geocode case). It is intentionally exempt because (a) the user-selected deliverable is staged-review visibility, not a push; (b) this spec's scope explicitly excludes any new push/telemetry surface; (c) staged/badge/digest visibility is preserved via `GAP_CLASSES` membership regardless. The `gateExempt` flag governs only push-regression participation (`dataGaps.ts:56-60`).
- **Match-by-id is intentionally NOT implemented.** §1 shows it is behavior-neutral against the fuzzy match already shipped, and id-persistence is architecturally infeasible at enrich (no id exists). Deferred to 8.3 in `BL-TRANSPORT-ID-RESOLUTION`. Do not relitigate as a gap.
- **The near-miss gate is required, not scope creep.** A plain "not-cleanly-resolved → warn" would fire on every legitimate external/charter driver (§2.2). The gate is what makes the alert trustworthy.
- **Aggregate (one warning/show), not per-name.** Bounds the data-gap count to +1 and keeps a single staged row; the message still names up to 5 specific strings so each is fixable.
- **No `transportTileVisible` change.** The regression pins characterize the *shared* `namesRefer` tolerance via the new enrich classification (red-first, new code) — they do not add green-only tests to unchanged render code (invariant 1).
- **Code named `TRAVEL_TRANSPORT_NAME_UNMATCHED` deliberately** so its `TRAVEL` prefix maps to the existing `crew-schedule` help family with **no `app/help/errors/_families.ts` edit** — keeping the milestone entirely out of `app/` and therefore out of the invariant-8 UI dual-gate. Mirrors the sibling `TRAVEL_FLIGHT_NAME_UNMATCHED`. Do not relitigate as a "should be TRANSPORT_*" naming nit — the prefix choice is load-bearing.
- **`lib/data/nameMatch.ts` is untouched.** The near-miss check uses a private `significantTokens` helper in the enrich module; `namesRefer`/`toks`/`refersSingle` keep one-letter tokens for initial matching. An initial-only regression test (§7 test 9) pins no behavior change. Do not relitigate as a shared-matcher refactor.

## 7. Testing / acceptance

TDD per task. Concrete failure mode each test catches (anti-tautology):

1. **Known-gap fixture (headline):** transportation `driver_name: "Doug Larson Loadout"`, roster `["Doug Larson"]` → exactly one `TRAVEL_TRANSPORT_NAME_UNMATCHED` warning whose `.message` contains `"Doug Larson Loadout"`. Catches: the silent-driver regression re-appearing.
2. **Resolved tolerance (no false warning):** driver `"Doug"` / `"Douglas Larson"` / `"  doug larson "`, and assigned `["Bill Werner"]` vs roster `["William Werner"]` → **no** warning. Catches: over-warning on names `transportTileVisible` already shows (the BL tolerance matrix, red-first via the enrich classifier). Nickname `"Bill Werner"` (2 sig-tokens) vs `"William Werner"` (2) specifically pins the token-count guard's clean side (`2 <= 2`).
3. **External driver (near-miss gate):** driver `"ABC Charters"`, roster `["Doug Larson"]` → **no** warning. Catches: chronic false-positive noise on legit external drivers.
3b. **Overflow / wrong-recipient (R2 finding):** driver `"Doug Larson John Smith"`, roster `["Doug Larson", "John Smith"]` → **exactly one** warning whose `.message` contains `"Doug Larson John Smith"`. Catches: the fusion that surname-`namesRefer`-resolves to `John Smith` (verified `true`) yet hides Doug's transport — the token-count guard (`sig 4 > 2`) must warn despite "resolving." This is the specific hole a "resolved-to-anyone → safe" gate would miss.
4. **Sentinel skip:** driver `""` / `"TBD"` / `"—"` → no warning. Catches: warning on non-assignments.
5. **Empty-roster / null-transport guards:** roster `[]` or `transportation: null` → no warning. Catches: spurious warnings when there is nothing to resolve against.
6. **Aggregate + cap:** 6 distinct warned names → one warning, message names 5 + `, and 1 more`. Catches: count inflation and unbounded message growth.
7. **assigned_names path:** a warned name only in `schedule[*].assigned_names` (driver clean) → warned. Catches: missing the leg-assignee code path.
8. **Lockstep gates:** x1, x2, `errors-grouping` (family pin), `dataGapsClassCompleteness` (partition), `catalogDocsValidator` all green with the new code.
9. **Initial-only matching unchanged (nameMatch untouched, non-tautological):** assert `namesRefer("J", "John Larson") === true` — a case that depends **solely** on the one-letter token `J` matching `John` by prefix (verified live; single-token branch `nameMatch.ts:47`), so it fails if a refactor drops one-letter tokens (unlike `"J Larson"`, which would still pass via the shared `Larson` surname path). Pair with an enrich test: assigned name `"J"`, roster `["John Larson"]` → **no** warning (`J` resolves; `significantTokens("J")` is empty → shares no significant token anyway → skip). Catches: a regression to `namesRefer`'s one-letter behavior AND proves `significantTokens` is genuinely private (dropping `J`) while `nameMatch.ts` is untouched.

Test files: new `tests/sync/enrichTransportAssignees.test.ts` (unit, the classification matrix + guards + cap); extend `tests/help/errors-grouping.test.tsx` (family pin); extend `tests/parser/dataGapsClassCompleteness.test.ts` (partition); `tests/cross-cutting/codes.test.ts` picks up the catalog row automatically.
