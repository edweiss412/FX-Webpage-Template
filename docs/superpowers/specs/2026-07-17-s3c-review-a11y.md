## 2. S3C-1 fix — dual-channel status dots

Add a **shape/fill channel** and a **text channel** so the flagged state is conveyed three ways (hue, shape, text), never hue alone.

### Visual (layout-safe, both states occupy an identical 8px box)

- **Flagged (review):** filled disc — `size-2 shrink-0 rounded-pill bg-status-review` (amber, solid; the higher-salience form draws the eye to the section that needs a look).
- **Clean (positive):** hollow **ring** — `size-2 shrink-0 rounded-pill border-[1.5px] border-status-positive bg-transparent` (no fill; the empty center recedes and is the shape signal).

Filled-disc-vs-ring is a fill/shape distinction, not a hue distinction; both render in an identical `8×8` bounding box (border-box default, `border-[1.5px]` sits inside the 8px) → **zero layout shift** between states (no `rotate-45` diagonal-bounding-box growth). Rationale for the direction: the attention-needing state is the visually heavier one (solid amber pops; the "no issues" ring recedes) — salience follows importance. This also keeps the flagged dot's existing `bg-status-review` class, so only the clean-dot styling changes.

Introduce a single helper in `ShowReviewSurface.tsx` that returns the FULL class string per state (replacing the `rounded-pill ${dotToneClass(id)}` interpolation at both dot sites):

```ts
/** §S3C-1: dot appearance carries BOTH hue and fill/shape so "needs review"
 *  is never signalled by color alone (WCAG 1.4.1). Flagged = filled amber disc
 *  (higher salience); clean = hollow teal ring (recedes). Both occupy an
 *  identical 8px box (no layout shift). */
function dotClass(id: SectionId): string {
  const review = id === "warnings" ? hasWarnRow : flagged.has(id);
  return review
    ? "size-2 shrink-0 rounded-pill bg-status-review"
    : "size-2 shrink-0 rounded-pill border-[1.5px] border-status-positive bg-transparent";
}
```

`dotToneClass` is retired (its only two callers are the two dot sites). Both dot `<span>`s become `className={dotClass(s.id)}` and gain a stable `data-testid` for the layout assertion:

- side-rail: `data-testid={`wizard-step3-card-${dfid}-review-rail-dot-${s.id}`}`
- chip-rail: `data-testid={`wizard-step3-card-${dfid}-review-chip-dot-${s.id}`}`

### Text channel (AT)

The dots stay `aria-hidden` (decorative once text exists). Append an `sr-only` status suffix to each nav button's accessible name, present only when the section is dotted (`!s.hideDot`):

```tsx
{!s.hideDot ? (
  <span className="sr-only">
    {(s.id === "warnings" ? hasWarnRow : flagged.has(s.id)) ? " — needs review" : " — no issues"}
  </span>
) : null}
```

Placed inside both the side-rail `<button>` (opens at `:650`, body `:650-689`) and the chip-rail `<button>` (opens at `:731`, body `:731-752`), after the visible label span, before/after the dot (order within the button only affects concatenated accessible-name reading order; place AFTER the label so it reads "Rooms — needs review"). `sr-only` is the Tailwind v4 built-in visually-hidden utility (used across the repo).

**Guard conditions (S3C-1):**

- Section with `hideDot === true` (registry opt-out): renders neither dot nor sr-only suffix (unchanged). No text, no shape — correct (those sections carry no status).
- Empty `flagged` set (all clean): every dotted section renders the filled positive dot + " — no issues". No ring anywhere.
- `warnings` section: status keyed off `hasWarnRow` (not `flagged.has("warnings")`), preserving the existing `:229` special case exactly.
- The change is class-string + one sr-only span per rail; no data flow, count, or registry change.

### DESIGN.md update (invariant-8 UI surface)

Document the dual-channel rule in the status-dot section (§1.3 / the status-token table area): "Section-nav status dots carry redundant non-color channels — clean = filled dot, needs-review = hollow ring — plus an `sr-only` text status on the nav control, so review state is never signalled by hue alone (WCAG 1.4.1)." One or two sentences; no token value change (reuses `--color-status-positive` / `--color-status-review`).

## 3. S3C-2 fix — portal + background inert

### Portal the modal to `document.body`

**Mount model:** the modal has **no `open` prop** — the parent conditionally MOUNTS it (`components/admin/wizard/Step3SheetCard.tsx:589`: `{detailsOpen ? <Step3ReviewModal … /> : null}`). So "mounted" == "open"; closing = unmount. No `open`-gated branches exist inside the component.

Gate on mount to keep SSR/hydration safe, then portal the entire modal tree (the `fixed inset-0 z-50` root at `:563` and everything under it). `useHasMounted` already exists in-repo — `lib/a11y/useHasMounted.ts:21` (`export function useHasMounted(): boolean`); do NOT add a new hook. It must be called unconditionally at the top of the component (Rules of Hooks), before the final `return`:

```tsx
const mounted = useHasMounted();
// ... all existing hooks/derivations unchanged ...
const tree = (<div data-testid={`wizard-step3-card-${dfid}-review-modal`} role="dialog" …>…</div>);
return mounted ? createPortal(tree, document.body) : tree;
```

First server/client paint renders the tree in place (SSR parity — identical markup, no hydration mismatch); the post-mount client render portals it to `body`. (Returning `null` pre-mount is also acceptable since the modal is client-interaction-only, but rendering the same tree in place first is the stricter hydration-safe choice.)

The modal is fully self-contained (fixed positioning is viewport-relative once un-confined; scroll-spy root is the content pane INSIDE the modal; focus trap + drag handlers use refs and its own elements; entrance animations key off `[data-step3-review-scrim]` / `[data-step3-review-panel]` on its own nodes). Portaling to `body` therefore preserves all behavior and additionally removes the PageTransition-transform confinement.

### Background `inert`

Add a stable, non-visual hook attribute to the admin shell wrapper so the modal can inert it without coupling to a test id:

- `app/admin/layout.tsx` — the two admin-shell root `<div>`s (onboarding branch opens ~`:155`, normal branch ~`:171-172`; each carries `data-testid="admin-layout"` at `:156`/`:173`): add `data-inert-root=""` alongside the existing `data-testid="admin-layout"`. Non-visual, non-behavioral; only one branch renders at a time.

In `Step3ReviewModal`, a **mount/unmount** effect (empty deps — the component is only mounted while the dialog is open) sets `inert` + `aria-hidden="true"` on every `[data-inert-root]` element and restores prior values on unmount:

```tsx
useEffect(() => {
  const roots = Array.from(document.querySelectorAll<HTMLElement>("[data-inert-root]"));
  const prev = roots.map((el) => ({ el, inert: el.inert, ariaHidden: el.getAttribute("aria-hidden") }));
  for (const el of roots) { el.inert = true; el.setAttribute("aria-hidden", "true"); }
  return () => {
    for (const { el, inert, ariaHidden } of prev) {
      el.inert = inert;
      if (ariaHidden === null) el.removeAttribute("aria-hidden");
      else el.setAttribute("aria-hidden", ariaHidden);
    }
  };
}, []);
```

This effect runs client-side only (effects never fire during SSR), so `document` access is safe. It sits alongside the existing body-scroll-lock effect (`:286-292`), which is already a mount/unmount effect — same lifecycle. The portal ensures the modal itself is a `body` child (sibling of `[data-inert-root]`), so inerting the shell never inerts the dialog.

**Guard conditions (S3C-2):**

- SSR / pre-mount: effects do not run on the server; the first render emits the in-place tree (hydration-safe), and both the portal move and the inert effect fire only after client mount.
- Closed: the parent renders `null` (unmounts) — the whole component (portal + inert effect) is gone; the unmount cleanup restores the shell's prior `inert`/`aria-hidden`.
- Zero `[data-inert-root]` matches (defensive — e.g. modal used outside admin someday): effect maps over an empty array; focus trap + `aria-modal` still contain keyboard/most AT. No throw.
- Nested/second modal open: each instance captures/restores its own `prev` snapshot; `el.inert` is idempotent (already-true stays true, restored to the captured prior). The repo opens at most one review modal at a time.
- `prefers-reduced-motion`: unchanged — entrance animation collapse already handled in globals.css; portal/inert add no animation.
