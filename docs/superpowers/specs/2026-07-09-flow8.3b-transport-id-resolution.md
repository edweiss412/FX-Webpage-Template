# Flow 8.3b — id-based transport visibility (garble-proof "is this my ride?")

**Status:** DRAFT · **Date:** 2026-07-09 · **Class:** CREW VISIBILITY
**Backlog:** closes the still-deferred half of `BL-TRANSPORT-ID-RESOLUTION` (`BACKLOG.md:333`) — the id-based visibility matching + the defensive regression pins moved out of `flow8-self-serve-trio` at plan R11. The enrich-time admin warning half already shipped (Flow 8.4, PR #374).
**Audit item:** `docs/audits/e2e-real-world-variation-preparedness-2026-07-07.md` §Flow 8 (a transport name mis-parse must not hide a driver's own itinerary).

---

## 1. Problem

`transportTileVisible` (`lib/visibility/scopeTiles.ts:177-211`) decides whether a crew viewer sees the transportation tile by **fuzzy name match** (`namesReferAny` → `namesRefer`, `lib/data/nameMatch.ts:63-80`) of the viewer's name aliases against `transportation.driver_name` and each `schedule[*].assigned_names[]`.

`namesRefer` closes the common variance (nickname / legal-name / case / trim / prefix). It does NOT close a **hard mis-parse** — a merged-cell overflow that fuses an adjacent token onto the name. The canonical live case: a driver stored as `"Doug Larson Loadout"` (the load-out column bled into the driver cell) vs roster `"Doug Larson"`. `namesRefer` returns **false** here because the multi-token rule compares last tokens `"loadout" !== "larson"` (`nameMatch.ts:50-53`, verified). The driver then **silently does not see their own ground-transport block**.

Flow 8.4 (PR #374) made this **non-silent for the admin**: `enrichTransportAssignees` emits an aggregate `TRAVEL_TRANSPORT_NAME_UNMATCHED` data-gap warning (gateExempt) naming the un-matchable assignees. But the driver stays blind until the operator fixes the sheet. This spec closes that gap: **the driver sees their own ride regardless of the garble**, while the 8.4 admin warning is left exactly as-is.

---

## 2. Ratified design decisions (from brainstorming, user-approved 2026-07-09)

1. **Read-time, in-memory resolution (NOT apply-time persistence).** `getShowForViewer` already fetches both the roster (`crew_members`) and `transportation`; it resolves names → crew ids in-memory on each read. **No migration, no new column, no `schema-manifest` regen, no validation-parity dance.** The resolved id set is recomputed per read (the roster is tiny). Persisting the ids at apply-time was the backlog's original framing; it is functionally equivalent for this bug (the garbled string lives in `transportation.driver_name` either way) and was rejected as unnecessary DB surface. **This is a deliberate departure from the `BL-TRANSPORT-ID-RESOLUTION` "persist at apply" wording** and is the ratified approach for this spec (Watchpoint W1).
2. **Resolution reuses the Flow 8.4 `covers()` matcher; the 8.4 warning is untouched.** The strict whole-name-subset predicate `covers` (today private in `enrichTransportAssignees.ts:26-28`) is what actually resolves `"Doug Larson Loadout"` → Doug. `namesRefer` alone cannot (Decision-2 rejected in brainstorming). **No new `admin_alert` code, no new message-catalog code, no telemetry surface.**
3. **`covers` and `namesRefer` catch DIFFERENT classes — the predicate UNIONs both.** Neither subsumes the other (§5). Visibility = `isAdmin OR id-match(covers-resolved owners) OR name-fuzzy(namesRefer)`. Over-match is benign (UX-not-security, owner determination master-spec amendment 2026-05-23, cited at `scopeTiles.ts:168-170`); under-match (hiding a viewer's own ride) is the only harm — so a union that can only ADD visibility is strictly safe.

## 3. Architecture & data flow

```
crew_members (roster, with ids)  ┐
                                 ├─► resolveTransportOwners(transportation, crewMembers): string[]
transportation (driver_name,     ┘        (pure, read-time, in getShowForViewer)
  schedule[*].assigned_names)                     │
                                                  ▼
                              ShowForViewer.transportationOwnerIds: string[]
                              ShowForViewer.viewerId: string | null
                                                  │
                4 crew section components pass data.{viewerId, transportationOwnerIds}
                                                  ▼
   transportTileVisible({ transportation, viewerId, transportationOwnerIds,
                          viewerName, viewerNameAliases, isAdmin }): boolean
       = isAdmin
         OR (viewerId ∈ transportationOwnerIds)          ← NEW: garble-proof id path
         OR namesReferAny(driver_name, aliases)           ← existing name fallback
         OR schedule.some(s => s.assigned_names.some(n => namesReferAny(n, aliases)))
```

No DB read is added. `resolveTransportOwners` is pure over data already in memory.

**New:**

```ts
import type { TransportationRow } from "@/lib/parser/types";
import { namesRefer } from "@/lib/data/nameMatch";
import { shouldHideGenericOptional } from "@/lib/visibility/emptyState"; // canonicalize-exempt: assignee sentinel, not an email

/** Minimal structural roster shape — reads id + name + sheet_name ONLY. Fed a
 *  SERVER-ONLY roster built inside getShowForViewer (§4.3, `ownerResolveRoster`) — NOT
 *  `ShowForViewer["crewMembers"]`: `sheet_name` is deliberately KEPT OFF the returned/
 *  client-visible projection (data minimization — Codex plan-review R5; exposing every
 *  member's pre-override name to the crew-page payload is a privacy regression). The ids
 *  are DB-assigned at apply, which is why this resolver is read-time-only and never runs
 *  in the enrich/parse pass (NOT ParseResult["crewMembers"] — CrewMemberRow, types.ts:84,
 *  has no id).
 *
 *  `sheet_name` (§3.5, `crew_members.sheet_name`, `admin_field_overrides.sql:60`) is
 *  the PRE-override parsed name, present only while a name override is active — the
 *  name transport rows still key on. Resolving against BOTH `name` and `sheet_name`
 *  is REQUIRED: a surname-changing override PLUS a garbled cell (e.g. sheet
 *  "Old Surname Loadout" while current name is "New Surname") defeats BOTH the id
 *  path (covers/namesRefer vs current name → fail) AND the render-time alias fallback
 *  (namesRefer chokes on the garble) — the exact silent-hide class this spec closes.
 *  Mirrors the render-path alias set `[live name, sheet_name?]` (getShowForViewer.ts:345). */
type ResolvableCrew = ReadonlyArray<{ id: string; name: string; sheet_name: string | null }>;

/**
 * Read-time resolution of free-text transport assignee names → the set of crew
 * member ids that should see the transport tile. Pure; reads names + ids only;
 * never throws. Empty when nothing resolves.
 *
 * A candidate name resolves to a crew id when `covers` (whole-name subset —
 * catches the "Doug Larson Loadout" ⊇ "Doug Larson" garble) OR `namesRefer`
 * (nickname / prefix — catches "Bill Werner" ↔ "William Werner" that `covers`
 * misses), matched against EITHER of that member's aliases (current name +
 * pre-override sheet_name). Union, because neither matcher subsumes the other
 * (§5). A candidate that covers ≥2 roster members (a merged multi-person cell)
 * resolves to ALL of them — benign over-match (UX-not-security).
 */
export function resolveTransportOwners(
  transportation: TransportationRow | null,
  crewMembers: ResolvableCrew,
): string[] {
  if (!transportation) return [];
  if (crewMembers.length === 0) return [];
  const roster = crewMembers.map((c) => {
    // [current name, pre-override sheet_name?] — the alias set transport rows may key on.
    const aliases = c.sheet_name && c.sheet_name.trim().length > 0 ? [c.name, c.sheet_name] : [c.name];
    return { id: c.id, aliases: aliases.map((a) => ({ name: a, sig: significantTokens(a) })) };
  });
  // Runtime-defensive: driver_name/schedule/assigned_names are STRING-typed but arrive
  // from decoded JSONB and can be corrupt (e.g. `assigned_names: [null]`). This resolver
  // runs inside the page-critical projection, so it MUST be total — a non-string element
  // must not throw and crash the whole crew page. Filter to strings + guard the arrays.
  const candidates: string[] = [];
  if (typeof transportation.driver_name === "string") candidates.push(transportation.driver_name);
  if (Array.isArray(transportation.schedule)) {
    for (const leg of transportation.schedule) {
      if (leg && Array.isArray(leg.assigned_names)) {
        for (const n of leg.assigned_names) if (typeof n === "string") candidates.push(n);
      }
    }
  }
  const owners = new Set<string>();
  for (const candidate of candidates) {
    if (shouldHideGenericOptional(candidate)) continue; // rule 1 — whole-value sentinel ("N/A", "TBD", "TBA", "-", "—", "")
    for (const sub of candidate.split("/")) {
      const name = sub.trim();
      if (name.length === 0) continue;
      if (shouldHideGenericOptional(name)) continue; // defensive — a sentinel part in a slash-joined value
      const cSig = significantTokens(name);
      if (cSig.length === 0) continue; // rule 2 — no significant token (a lone initial "A"/"N" would else match
      //                                  namesRefer's initial support and resolve a sentinel to a real crew id)
      for (const m of roster) {
        if (m.aliases.some((a) => covers(cSig, a.sig) || namesRefer(a.name, name))) owners.add(m.id);
      }
    }
  }
  return [...owners];
}
```

The two sentinel/initial guards mirror `enrichTransportAssignees.isWarned` rules 1-2 (`enrichTransportAssignees.ts:32,34`) exactly, so resolution and the 8.4 warning treat sentinels identically. **Critical:** without the `cSig.length === 0` guard, `"N/A"` → split → `"A"` → `namesRefer("Alice Adams", "A")` is TRUE (initials are intentionally supported in `nameMatch.ts:37-38`), leaking a transport tile to unassigned crew (Codex spec-review R2).

`crew_members.{id,name,sheet_name}`: `initial_public_schema.sql:32,34` + `admin_field_overrides.sql:60`. `TransportationRow.schedule[*].assigned_names` shape: `lib/parser/types.ts:191-196`.

### 4.5 `transportTileVisible` (`scopeTiles.ts:177-211`)

Add two params; insert the id path before the existing name branches:

```ts
export function transportTileVisible(opts: {
  transportation: TransportationRow | null;
  viewerId: string | null;                 // NEW
  transportationOwnerIds: string[];        // NEW
  viewerName: string | null;
  viewerNameAliases: string[];
  isAdmin: boolean;
}): boolean {
  const { transportation, viewerId, transportationOwnerIds, viewerName, viewerNameAliases, isAdmin } = opts;
  if (!transportation) return false;
  if (isAdmin) return true;
  // NEW — garble-proof id path (BL-TRANSPORT-ID-RESOLUTION). A viewer resolved as
  // a transport owner sees the tile even when driver_name is mis-parsed such that
  // namesRefer fails (e.g. "Doug Larson Loadout"). Inert when viewerId is null or
  // the owner set is empty. typeof/Array.isArray guards tolerate an OLD cached
  // ShowForViewer shape (pre-deploy entry lacking these fields → undefined) without
  // throwing — defense-in-depth alongside the §4.3 cache-key bump (Codex plan-review R6).
  if (typeof viewerId === "string" && Array.isArray(transportationOwnerIds) && transportationOwnerIds.includes(viewerId)) return true;
  if (!viewerName) return false;
  // Name fallback. typeof/Array.isArray guards mirror resolveTransportOwners: the name
  // branch reads the SAME decoded JSONB and MUST NOT crash the crew page on a corrupt
  // row (`driver_name` non-string, `assigned_names: [null]`) — namesRefer does `.split`.
  if (typeof transportation.driver_name === "string" && namesReferAny(transportation.driver_name, viewerNameAliases)) return true;
  return (
    Array.isArray(transportation.schedule) &&
    transportation.schedule.some(
      (s) =>
        s != null &&
        Array.isArray(s.assigned_names) &&
        s.assigned_names.some((n) => typeof n === "string" && namesReferAny(n, viewerNameAliases)),
    )
  );
}
```

## 5. Matcher semantics — why UNION (worked examples)

`covers` (whole-name subset, candidate ⊇ member) and `namesRefer` (surname-anchored, tolerant of nickname/prefix) catch **disjoint** failure classes:

| Candidate (sheet) | Roster member | `covers`? | `namesRefer`? | Owner via | Correct? |
|---|---|---|---|---|---|
| `"Doug Larson Loadout"` | `"Doug Larson"` | ✅ (`[doug,larson]` ⊆ `[doug,larson,loadout]`) | ❌ (`loadout≠larson`) | **covers** | ✅ visible |
| `"Bill Werner"` | `"William Werner"` | ❌ (`william ∉ [bill,werner]`) | ✅ (`werner===werner`) | **namesRefer** | ✅ visible |
| `"Doug"` (first-name only) | `"Doug Larson"` | ❌ (`larson ∉ [doug]`) | ✅ (single-token→first token) | **namesRefer** | ✅ visible |
| `"Doug Larson Bill Werner"` (merged) | `"Doug Larson"` AND `"Bill Werner"` | ✅ both | mixed | **covers** (both) | ✅ both visible (benign) |
| `"Doug Larson Loadout"` | member `name:"Doug Newname", sheet_name:"Doug Larson"` (renamed) | ✅ vs `sheet_name` alias | ❌ | **covers (sheet_name alias)** | ✅ visible (name-override + garble; §4.1) |
| `"ABC Charters"` (external) | — | ❌ (0) | ❌ (0) | none | ✅ not visible (external driver) |
| `"N/A"` / `"TBD"` / `"TBA"` / `"-"` / `""` (sentinel) | any (e.g. `"Alice Adams"`) | — | skipped | **none — `shouldHideGenericOptional` + zero-significant-token guard** | ✅ not visible (would leak via `namesRefer("Alice Adams","A")` without the guard) |

The id path can only ADD an owner id; it never removes a name-fuzzy match. So the union is monotone-safe: it strictly reduces under-match (the harm) and any added over-match is benign per the UX-not-security posture.

## 6. Guard conditions (every new input)

`resolveTransportOwners(transportation, crewMembers)`:
- `transportation === null` → `[]`.
- `crewMembers.length === 0` → `[]` (cannot resolve against an empty roster).
- `driver_name === null` → driver contributes no candidate.
- `schedule === []` or a leg's `assigned_names === []` → contributes nothing.
- **malformed decoded JSONB** — `driver_name` not a string, `schedule` not an array, or an `assigned_names` element not a string (e.g. `[null]`) → skipped by `typeof`/`Array.isArray` guards; the resolver is TOTAL (never throws) so `getShowForViewer`'s page-critical projection can't crash on a corrupt transportation row (Codex plan-review R1). Contributes no id; owner set is simply whatever the valid candidates resolve to.
- candidate is `""` / whitespace / a `/`-split sub that trims empty → skipped.
- `covers` + `namesRefer` both false for a candidate → contributes no id (name fallback in the predicate may still catch it live, but that path uses the viewer's aliases, not the roster canonical name — both are exercised).
- duplicate ids (a viewer who is both driver and leg-tagged) → deduped via `Set`.

`transportTileVisible` new inputs:
- `viewerId === null` (admin, or a viewer with no crew row) → id path is inert; `isAdmin`/name paths unchanged.
- `transportationOwnerIds === []` → id path inert.
- `transportation === null` → `false` (unchanged early return; owner set is `[]` in that case regardless).
- **malformed decoded JSONB in the NAME branch** — `driver_name` non-string, `schedule` non-array, `assigned_names` element non-string (`[null]`) → the predicate's typeof/`Array.isArray` guards skip it (never throws). BOTH transport-visibility consumers of the corrupt row (the resolver AND this predicate's name fallback) are total, so §6's page-safety promise holds end-to-end, not just inside the resolver (Codex plan-review R4 HIGH — the name branch does `namesRefer` → `.split`, which would else crash the whole crew page).
