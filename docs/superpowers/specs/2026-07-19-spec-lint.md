# spec:lint — mechanize the checkable subset of spec/plan pre-review passes

**Date:** 2026-07-19 · **Backlog:** `BL-SPEC-LINT` (`BACKLOG.md:7-11`) · **Kind:** tooling (no UI, no DB, no runtime surface)

R1–R3 adversarial rounds are dominated by non-compliance with passes `AGENTS.md` already mandates (live-code citation grep, numeric sweep, copy rules, mandatory sections), not by rule gaps. This spec defines `pnpm spec:lint <doc>`: a deterministic linter for spec/plan markdown that hard-fails on the mechanically checkable subset and emits advisory reports (the transcript artifact) for the fuzzy remainder.

## 1. Scope

### 1.1 Explicitly resolved scope decisions (do not relitigate)

Items 1–9 ratified at the brainstorming gate (user-approved, 2026-07-19); items 10–12 ratified during adversarial round 1:

1. **Mixed posture.** Deterministic checks exit non-zero; fuzzy heuristics are advisory-only and never affect the exit code. Rationale: hard-failing heuristics get bypassed; a pure reporter re-permits non-compliance.
2. **No CI gate.** `spec:lint` is a point-in-time pre-dispatch tool. Line-number citations drift as code merges after the spec ships; wiring this into CI would false-fail every historical spec forever. Enforcement is process-level: the review-dispatch discipline in `AGENTS.md:102` is amended to require attached `spec:lint` output.
3. **Symbol-proximity matching is advisory, not hard.** Prose legitimately cites a line for context without naming a symbol on the same spec line.
4. **Copy scan scope = quoted spans + code-block string literals.** Spec prose itself may use em-dashes freely; only quoted user-visible copy and string literals inside fenced code blocks are scanned.
5. **UI-spec detection is automatic** (citation paths under `app/` excluding `app/api/`, or `components/`), waivable via `<!-- spec-lint: not-ui — <reason> -->`.
6. **`--` (double hyphen) and straight-apostrophe findings are ADVISORY, not hard**, even though `DESIGN.md:332` bans `--` in copy. Rationale: quoted CLI flags (`"--json"`, `"--reveal-email"`) are pervasive in this repo's specs and would hard-fail constantly; and the curly-apostrophe convention is not codified in `DESIGN.md` at all — it exists only as test-comment convention (e.g. `tests/app/admin/revokeHang.test.tsx:151` "Curly apostrophe (&rsquo;) per DESIGN.md typography"). Em-dash (U+2014) in scanned spans is the sole hard copy failure.
7. **No new dependencies.** Line/regex-based scanning plus small state machines over house-style markdown; no remark/unified.
8. **Plans skip section-presence checks** (§1.1 and UI sections are spec-mandates, not plan-mandates). Citations, numerics, and copy checks run for both kinds.
9. **Zero-match and ambiguous bare-filename citations are hard failures.** A citation that cannot be resolved to exactly one file cannot be verified, which defeats the check's purpose.
10. **Target corpus = newly authored docs at dispatch time; legacy-corpus compatibility is a NON-GOAL.** The existing ~390-doc corpus contains citation and fence forms this grammar rejects; those docs are never re-linted (item 2). What IS required is fail-closed behavior: a span that looks citation-shaped but does not parse is a hard `CITATION_MALFORMED` finding (waivable, §3), never silent prose. Typos cannot bypass verification by being unparseable.
11. **Citations resolve only within the tracked-file set.** Every resolved citation path must be a member of `git ls-files` output at the repo root. Absolute paths, `..` segments, and backslashes are `CITATION_MALFORMED`; a well-formed path not in the tracked set is `CITATION_FILE_MISSING`. Consequence: the linter never reads a file outside the repository (no traversal, no symlink escape, no untracked-file reads), so advisory printouts cannot leak non-repo content into a review transcript.
12. **Inline-code spans are line-local.** CommonMark technically allows multi-line inline code; house docs do not use it. A backtick span never crosses a line boundary in this model.

### 1.2 Out of scope

- Semantic verification that a cited line's *content* supports the prose claim (stays with the model's citation pass; the advisory printout assists it).
- Linting non-superpowers docs (`README.md`, `AGENTS.md` themselves) and the legacy corpus (§1.1 item 10).
- Auto-fixing. Report-only.
- The apostrophe rule's codification into `DESIGN.md` (separate docs decision; the linter treats it as advisory convention).

## 2. CLI contract

```
pnpm spec:lint <path-to-doc.md> [--kind spec|plan] [--json]
```

**Arguments and flags (all violations exit 2):**

- Exactly one positional argument (zero or more than one = usage error). It must be a readable file, not a directory.
- `--kind` takes exactly `spec` or `plan`. When present it WINS over path inference (no conflict error — explicit beats inferred, stated here so a doc under `specs/` can be linted as a plan deliberately). When absent: path containing `/specs/` → `spec`; `/plans/` → `plan`; neither → usage error naming the missing flag.
- `--json` toggles output shape (§7). Unknown flags and duplicated flags are usage errors.

**Exit codes:** `0` = clean or advisories only; `1` = one or more hard findings; `2` = usage or infrastructure error (unreadable doc, `git ls-files` failure, no repo root). Divergence from the observe CLI stated explicitly: observe collapses all errors to exit 1 (`scripts/observe.ts:45`); `spec:lint` reserves 1 for findings so automation can distinguish "doc has problems" from "tool could not run."

**Streams:** report on stdout; usage/infra errors on stderr — a JSON object (`{"error": <message>}`) when `--json`, plain text otherwise, mirroring the observe convention (`scripts/observe.ts:40-45`).

Adapter pattern mirrors the observe CLI (thin `scripts/observe.ts` adapter over a pure core, `scripts/observe.ts:1-32` imports): `scripts/spec-lint.ts` owns argv, fs, repo-root discovery (`git rev-parse --show-toplevel` from the doc's directory), `git ls-files`, printing, and `process.exit`; all logic lives in `lib/specLint/**` (§7 purity contract).

**Encoding/EOL:** doc and cited files are read as UTF-8 with invalid sequences replaced (U+FFFD); CRLF is normalized to LF before line-splitting; a trailing final newline does not create a phantom last line (split on `\n`, drop one trailing empty element). A file's line count for EOF checks is the length of that array.

## 3. Document model (shared by all checks)

`lib/specLint/parse.ts` produces a line-indexed model:

- **Fenced regions** (CommonMark-subset state machine): a fence opens on a line whose first non-whitespace (≤3 spaces indent) is a run of ≥3 backticks or ≥3 tildes; the info string is the remainder of that line. It closes on a line holding only a run of the SAME character at LEAST the opener's length (≤3 spaces indent). Runs longer than 3 nest correctly: a ````` ```` ````` fence containing ``` ``` ``` lines stays open until a ≥4-backtick closer. Unterminated fence → everything to EOF is fenced.
- **Inline-code spans** (line-local, §1.1 item 12): on non-fenced lines, a span opens with a run of N backticks and closes at the next run of exactly N backticks on the same line (CommonMark delimiter-run rule, so ``` `` … ` … `` ``` double-backtick spans containing single backticks parse correctly). Unclosed runs are literal text.
- **Headings:** ATX headings (`#`–`######`) with depth + text, non-fenced lines only.
- **Waivers:** a non-fenced line whose entire trimmed content is one HTML comment of the exact shapes below. Waiver-shaped text inside fenced regions or inline-code spans is inert. Recognized shapes:
  - `<!-- spec-lint: ignore — <reason> -->` — suppresses **hard** findings whose doc line is the next non-blank, non-waiver line. Advisories are never suppressed (they don't gate).
  - `<!-- spec-lint: not-ui — <reason> -->` — anywhere in the doc; disables UI-section requirements (§6).
  - Empty `<reason>` → hard `WAIVER_MISSING_REASON`, which is **unsuppressible** (no waiver, including itself or a preceding one, can suppress it — otherwise an empty waiver would launder itself).
  - A waiver that suppresses zero findings → advisory `WAIVER_UNUSED` (stale waivers rot; surfaced, not gated).

Guard conditions: empty doc (zero bytes or only blank lines) → hard `EMPTY_DOC` at docLine 1.

## 4. Check 1 — citations

**Candidate detection (fail-closed, §1.1 item 10).** Within inline-code spans only (fenced regions are EXCLUDED — they carry example output and hypothetical code whose paths are illustrative). A span is a **citation candidate** when either:

- **line-bearing shape:** content matches `/^[^\s`]+:\d/` AND the substring before the first `:` contains `.` or `/` AND is not purely digits (excludes times like `12:30`), AND contains no glob/expansion chars (`*`, `{`, `$`); or
- **path-only shape:** content contains no whitespace or glob chars, contains `/`, and its final segment contains a `.` (e.g. `lib/messages/lookup.ts`). Branch names (`feat/spec-lint`) have no dotted final segment → prose. Bare filenames without `/` and without `:line` are NOT candidates (too prose-ambiguous).

Known accepted false-positive class: host:port-shaped spans (`example.com:8080`) become candidates and fail as malformed/missing; the §3 `ignore` waiver covers the rare legitimate case.

**Well-formedness.** A line-bearing candidate must match `/^(?<path>[^\s:`]+):(?<start>[1-9]\d*)(?:-(?<end>[1-9]\d*))?$/`. Anything else — comma line-lists (`x.ts:12,34`), multiple colons, zero/leading-zero lines, trailing text — is hard `CITATION_MALFORMED` (split multi-line citations into separate spans). Paths that are absolute, contain `..` segments, or contain `\` are also `CITATION_MALFORMED` (§1.1 item 11).

**Resolution** (`lib/specLint/citations.ts`, injected resolver — §7):

1. Path contains `/` → must be an exact member of the tracked-file set. Not tracked = `FAIL CITATION_FILE_MISSING` (this is also the containment boundary, §1.1 item 11).
2. Bare filename (no `/`, line-bearing only) → resolve, in order: (a) the most recent earlier citation in the doc whose basename matches (the established shorthand pattern, e.g. `readShowChangeFeed.ts:124-169` at `docs/superpowers/specs/2026-07-19-published-show-alerts.md:98` after the full-path citation `lib/sync/feed/readShowChangeFeed.ts:286-310` at line 33 of the same doc); (b) unique basename match in the tracked-file set. Zero matches = `FAIL CITATION_FILE_MISSING`; >1 match with no earlier anchor = `FAIL CITATION_AMBIGUOUS` (lists candidates).
   <!-- spec-lint: ignore — next line deliberately names a file that no longer exists, as evidence -->
   Live proof of the failure class: the shipped published-show-alerts spec cites `PerShowAlertSection.tsx` (its line 89), whose file no longer exists in the tree — a stale citation this check would surface today; this very line carries the §3 `ignore` waiver so the dogfood run stays clean.
3. Line checks on resolved files: tracked but unreadable = `FAIL CITATION_UNREADABLE`; `start` or `end` beyond EOF (§2 line-count rule) = `FAIL CITATION_LINE_OUT_OF_RANGE` (reports actual file length); `start > end` = `FAIL CITATION_RANGE_INVERTED`. (`start`/`end` ≥ 1 is enforced by the well-formedness grammar.)

**Symbol proximity (advisory).** For each resolved line-bearing citation: collect the other inline-code spans on the same doc line that match an identifier shape (`/^[A-Za-z_$][A-Za-z0-9_$.]*$/`, excluding citation candidates). If at least one candidate exists and none appears as a substring within the cited file's lines `[start-5, end+5]` (clamped to file bounds), emit `ADVISORY CITATION_SYMBOL_UNMATCHED`, printing the first cited line trimmed to 160 chars so the author can eyeball the claim. No identifier spans on the doc line → no advisory (nothing to match against).

## 5. Check 2 — numerics

Both parts advisory (`lib/specLint/numerics.ts`). Scanned over non-fenced lines only; matches inside citation-candidate spans are excluded.

**Numeric lexicon (deterministic):** matches of `/\b\d+(?:\.\d+)?\b/` EXCEPT those inside: ISO dates (`\d{4}-\d{2}-\d{2}`), version strings (`v?\d+\.\d+\.\d+`), clock times (`\d+:\d+`), hex literals (`0x[0-9a-fA-F]+`). No sign, exponent, or thousands-separator handling — a literal like `1,000` scans as `1` and `000`, accepted noise in an advisory-only check.

**Noun-anchored mismatch.** From lexicon matches followed by whitespace + `/[a-z][a-z-]{2,}/`, form (number, noun) pairs; normalize the noun by lowercasing and stripping one trailing `s`. For each normalized noun bound to ≥2 distinct numbers, emit `ADVISORY NUMERIC_NOUN_MISMATCH` listing every occurrence (`doc line N: "45 codes"`). Intentionally over-reports ("3 rounds" vs "22 rounds" may both be true) — a review aid, not a gate (§1.1 item 1).

**Inventory.** Every lexicon match, grouped by value, each occurrence with doc line + a context snippet (surrounding text trimmed to 80 chars). NOT findings: emitted as a separate `inventory` array in `--json` output and an `INVENTORY` section in the text report; never counted in the summary line (§7). This is the mechanized replacement for the hand-built numeric-sweep transcript (`AGENTS.md:102`).

## 6. Check 3 — copy rules · Check 4 — sections

**Copy scan spans** (`lib/specLint/copyRules.ts`):

- **Prose:** text inside double-quote pairs — straight `"…"` paired with `"`, curly `“…”` paired curly-to-curly — on non-fenced lines. An unpaired opener on a line → `ADVISORY COPY_UNPAIRED_QUOTE` and the line's remainder is not scanned (surfaced instead of silently skipped).
- **Fenced code blocks** with a JS-family info string (`ts`, `tsx`, `typescript`, `js`, `jsx`, `javascript`, `mjs`, `cjs`, `json`): string literals delimited by `'`, `"`, or `` ` ``, honoring backslash escapes (`\"`, `\'`, `` \` ``). Template literals track backtick parity ACROSS lines within one fence: lines interior to an open template literal are scanned wholesale until the closing backtick (multiline copy strings do not escape the scan). Interpolation contents are scanned as raw text (not parsed). Other/absent info strings (`sql`, `sh`, bare fences, output transcripts) are not scanned.

**Findings within scanned spans:**

- Em-dash `—` (U+2014): `FAIL COPY_EM_DASH` (`DESIGN.md:332` — "No em dashes."; hard per §1.1 item 6).
- `--` (double hyphen): `ADVISORY COPY_DOUBLE_HYPHEN` (banned by `DESIGN.md:332` but advisory per §1.1 item 6 — CLI-flag false positives).
- Straight apostrophe `'` inside **prose** double-quoted spans only: `ADVISORY COPY_STRAIGHT_APOSTROPHE` (uncodified curly-quote convention, §1.1 item 6). Not applied to code-fence literals (code legitimately uses `'` everywhere).

**Sections** (`lib/specLint/sections.ts`), `kind = spec` only (plans skip, §1.1 item 8); missing-section findings report docLine 1:

- A heading whose text matches `/resolved scope/i` must exist: absent = `FAIL SECTION_MISSING_RESOLVED_SCOPE`. Convention anchor: `docs/superpowers/specs/2026-07-19-published-show-alerts.md:21` ("### 1.1 Explicitly resolved scope decisions (do not relitigate)").
- **UI-spec detection:** any *resolved* citation path (§4) starting with `app/` but not `app/api/`, or starting with `components/` → the doc is a UI spec → headings matching `/dimensional invariants/i` and `/transition inventory/i` are both required; each absence is a separate `FAIL SECTION_MISSING_UI` finding. The `not-ui` waiver (§3) disables this entire sub-check; the waiver plus ≥1 UI citation is fine by design (that is the waiver's purpose — incidental citations).

## 7. Architecture, result contract & purity

```
scripts/spec-lint.ts          # CLI adapter: argv, read doc, real FileResolver, print, exit
lib/specLint/types.ts         # Finding + report types
lib/specLint/parse.ts         # document model (§3)
lib/specLint/citations.ts     # §4
lib/specLint/numerics.ts      # §5
lib/specLint/copyRules.ts     # §6 copy
lib/specLint/sections.ts      # §6 sections
lib/specLint/run.ts           # orchestrator: (docText, kind, resolver) → LintResult
```

**Result contract.** `LintResult = { findings: Finding[], inventory: InventoryGroup[] }`. `Finding = { check: 'citations'|'numerics'|'copy'|'sections'|'document', code: string, severity: 'fail'|'advisory', docLine: number (1-based, always ≥1; whole-doc findings such as `EMPTY_DOC` and missing sections use 1), message: string, detail?: string }`. Ordering is fully deterministic: by check (document, citations, numerics, copy, sections), then docLine, then code, then extraction column. The text report groups by check, tags `FAIL`/`ADVISORY`, prints `INVENTORY` last, and ends with `N hard, M advisory` counting finding OCCURRENCES (inventory excluded). `--json` emits `LintResult` verbatim. The full output is the compliance artifact attached to review dispatches.

**FileResolver injection:** `{ fileExists(path), readFileLines(path), listTrackedFiles() }`, all rooted at the repo root the adapter discovered. The CLI adapter backs it with `node:fs` + `git ls-files`; tests back it with in-memory fixture maps. The core only ever calls `readFileLines` on tracked paths (§1.1 item 11). Nothing under `lib/specLint/**` imports `node:fs`, `node:child_process`, or `node:process` — pinned by a structural meta-test (§8), the same recursive-walk posture as the observe read-core pin (`tests/observe/_metaReadOnlyQueryCore.test.ts:11` walk comment, `:22` suite).

`run.ts` is deterministic: same doc + same resolver state → same `LintResult`.

**Invariant applicability (declared per AGENTS.md):** no advisory locks (no DB), no email boundary, no §12.4 codes (finding codes like `CITATION_FILE_MISSING` are CLI-report identifiers printed by a dev tool, not user-visible app UI routed through `lib/messages/lookup.ts`), no mutation surface (invariant 10 N/A — read-only CLI, no HTTP route or server action), no Supabase calls (invariant 9 N/A), no UI files (invariant 8 / impeccable N/A, Opus-routing N/A).

## 8. Testing

All under `tests/specLint/` — auto-included by `BASE_INCLUDE` (`vitest.projects.ts:20`, `tests/**/*.test.ts`) and therefore run by the sharded unit-suite CI job (`.github/workflows/unit-suite.yml:80` runs `vitest run --shard` over the default include set); no new workflow wiring. TDD per task.

- **Per-check unit tests** with in-memory fixture docs + fixture resolver. Anti-tautology: every fixture plants a specific defect and the assertion names the exact expected finding code + doc line derived from the fixture's construction (never "some finding exists"). Clean-fixture tests assert zero findings for that check. Required branch coverage:
  - *Document model:* empty doc; unterminated fence; 4-backtick fence enclosing 3-backtick lines (stays open); tilde fences; double-backtick inline span containing a single backtick; waiver inside a fence (inert); indented waiver (recognized after trim); waiver with trailing prose on the line (not a waiver).
  - *Citations:* candidate vs prose discrimination (`12:30` prose, `feat/spec-lint` prose, `lib/x.ts:12` candidate, `example.com:8080` candidate→missing); malformed forms (comma list, absolute path, `..`, zero line, trailing text); tracked-set containment (well-formed path outside tracked set fails; resolver never asked to read it — spy assertion); bare-filename anchored resolution, unique-basename resolution, zero-match, ambiguous; EOF boundary (cite last line passes, last+1 fails, with and without trailing final newline in the fixture file); inverted range; unreadable tracked file; symbol-proximity match, unmatch, window clamping at file head/tail, no-identifier-spans (no advisory).
  - *Numerics:* lexicon exclusions (date, version, time, hex, inside citations); noun mismatch across distinct numbers; singular/plural normalization; inventory grouping + 80-char snippet; inventory absent from findings/summary.
  - *Copy:* em-dash in prose quotes (straight + curly pairing); em-dash in each JS-family tag incl. `typescript`; multiline template literal interior line; escaped delimiters; unpaired-quote advisory; `sql`/bare fences unscanned; double-hyphen advisory; straight-apostrophe advisory in prose only.
  - *Sections:* spec missing §1.1-class heading; plan skipping all section checks; UI detection positive (`components/` citation), negative (`app/api/` only), waived (`not-ui` present + UI citation); missing-UI-section findings at docLine 1.
  - *Waivers:* `ignore` suppressing next-line hard finding; NOT suppressing advisories; empty-reason waiver unsuppressible; `WAIVER_UNUSED` advisory.
  - *Contract:* deterministic ordering under shuffled input construction; exit-code mapping (advisories-only → 0) at the run.ts level.
- **CLI adapter test:** spawn `tsx scripts/spec-lint.ts` against small committed fixture docs under `tests/specLint/fixtures/` asserting: failing doc → exit 1; clean doc → exit 0; no-kind-inferable path without `--kind` → exit 2; unknown flag → exit 2; zero/two positionals → exit 2; `--json` shape parses and stderr JSON error shape on usage error. Fixture docs cite files inside `tests/specLint/fixtures/` themselves so citations stay stable forever (never live app code, which drifts).
- **Structural meta-test** `tests/specLint/_metaPureCore.test.ts`: recursive walk of `lib/specLint/` (fails-by-default for new files) asserting no `node:fs` / `node:child_process` / `node:process` imports.
- **Dogfood check (plan-time task):** run `pnpm spec:lint docs/superpowers/specs/2026-07-19-spec-lint.md` — this spec must itself exit 0 with no hard findings.
