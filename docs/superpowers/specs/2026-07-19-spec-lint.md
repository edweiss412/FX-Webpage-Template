# spec:lint — mechanize the checkable subset of spec/plan pre-review passes

**Date:** 2026-07-19 · **Backlog:** `BL-SPEC-LINT` (`BACKLOG.md:7-11`) · **Kind:** tooling (no UI, no DB, no runtime surface)

R1–R3 adversarial rounds are dominated by non-compliance with passes `AGENTS.md` already mandates (live-code citation grep, numeric sweep, copy rules, mandatory sections), not by rule gaps. This spec defines `pnpm spec:lint <doc>`: a deterministic linter for spec/plan markdown that hard-fails on the mechanically checkable subset and emits advisory reports (the transcript artifact) for the fuzzy remainder.

## 1. Scope

### 1.1 Explicitly resolved scope decisions (do not relitigate)

All ratified at the brainstorming gate (user-approved, 2026-07-19):

1. **Mixed posture.** Deterministic checks exit non-zero; fuzzy heuristics are advisory-only and never affect the exit code. Rationale: hard-failing heuristics get bypassed; a pure reporter re-permits non-compliance.
2. **No CI gate.** `spec:lint` is a point-in-time pre-dispatch tool. Line-number citations drift as code merges after the spec ships; wiring this into CI would false-fail every historical spec forever. Enforcement is process-level: the review-dispatch discipline in `AGENTS.md:102` is amended to require attached `spec:lint` output.
3. **Symbol-proximity matching is advisory, not hard.** Prose legitimately cites a line for context without naming a symbol on the same spec line.
4. **Copy scan scope = quoted spans + code-block string literals.** Spec prose itself may use em-dashes freely; only quoted user-visible copy and string literals inside fenced code blocks are scanned.
5. **UI-spec detection is automatic** (citation paths under `app/` excluding `app/api/`, or `components/`), waivable via `<!-- spec-lint: not-ui — <reason> -->`.
6. **`--` (double hyphen) and straight-apostrophe findings are ADVISORY, not hard**, even though `DESIGN.md:332` bans `--` in copy. Rationale: quoted CLI flags (`"--json"`, `"--reveal-email"`) are pervasive in this repo's specs and would hard-fail constantly; and the curly-apostrophe convention is not codified in `DESIGN.md` at all — it exists only as test-comment convention (e.g. `tests/app/admin/revokeHang.test.tsx:151` "Curly apostrophe (&rsquo;) per DESIGN.md typography"). Em-dash (U+2014) in scanned spans is the sole hard copy failure.
7. **No new dependencies.** Line/regex-based scanning over house-style markdown; no remark/unified.
8. **Plans skip section-presence checks** (§1.1 and UI sections are spec-mandates, not plan-mandates). Citations, numerics, and copy checks run for both kinds.
9. **Zero-match and ambiguous bare-filename citations are hard failures.** A citation that cannot be resolved to exactly one file cannot be verified, which defeats the check's purpose.

### 1.2 Out of scope

- Semantic verification that a cited line's *content* supports the prose claim (stays with the model's citation pass; the advisory printout assists it).
- Linting non-superpowers docs (`README.md`, `AGENTS.md` themselves).
- Auto-fixing. Report-only.
- The apostrophe rule's codification into `DESIGN.md` (separate docs decision; the linter treats it as advisory convention).

## 2. CLI contract

```
pnpm spec:lint <path-to-doc.md> [--kind spec|plan] [--json]
```

- **Kind inference:** path contains `/specs/` → `spec`; `/plans/` → `plan`; otherwise `--kind` is required. Undeterminable kind, unreadable doc, or unknown flag = usage error, exit 2 (fail-closed, matching the observe CLI's invalid-flag posture, `AGENTS.md` telemetry section).
- **Exit codes:** `0` = clean or advisories only; `1` = one or more hard findings; `2` = usage error.
- **Output:** human-readable report grouped by check, each finding tagged `FAIL` or `ADVISORY` with `doc line N`. Ends with a summary line (`N hard, M advisory`). `--json` emits the typed findings array. The full output is the compliance artifact attached to review dispatches.

Adapter pattern mirrors the observe CLI: `scripts/spec-lint.ts` is a thin adapter (argv, fs, `git ls-files`, process.exit); all logic lives in `lib/specLint/**` as pure functions (see §7 architecture / purity contract).

## 3. Document model (shared by all checks)

`lib/specLint/parse.ts` produces a line-indexed model:

- **Fenced regions:** lines inside ``` fences, with the fence's info string (language tag).
- **Inline-code spans:** backtick spans per line (outside fenced regions).
- **Headings:** ATX headings (`#`–`######`) with depth + text.
- **Waivers:** HTML comments of the exact shapes
  - `<!-- spec-lint: ignore — <reason> -->` — suppresses **hard** findings whose doc line is this comment's line or the line immediately following it. Advisories are never suppressed (they don't gate).
  - `<!-- spec-lint: not-ui — <reason> -->` — anywhere in the doc; disables UI-section requirements (§6).
  - A waiver comment with an empty reason is itself a hard finding (`WAIVER_MISSING_REASON`) — un-reasoned waivers defeat the audit trail.

Guard conditions: empty doc → hard finding `EMPTY_DOC`. Unterminated fence → everything after the opening fence is fenced (CommonMark behavior); no crash. CRLF input is normalized to LF before line-splitting.

## 4. Check 1 — citations

**Extraction.** Within inline-code spans only (fenced regions are EXCLUDED — they carry example output and hypothetical code whose paths are illustrative):

```
/^(?<path>[A-Za-z0-9_.\/\[\]-]+\.(?:ts|tsx|js|mjs|cjs|sql|md|sh|css|json|ya?ml))(?::(?<start>\d+)(?:-(?<end>\d+))?)?$/
```

applied to each backtick span's full content; spans that are not exactly a `path` or `path:line[-range]` token are not citations (so `` `lookup.ts should be used` `` is prose, not a citation). Spans with a path but no line number are **path-only citations**: existence-checked (hard) but skipped by line/symbol logic.

**Resolution** (`lib/specLint/citations.ts`, using the injected resolver — §7):

1. Path contains `/` → resolve relative to repo root. Missing file = `FAIL CITATION_FILE_MISSING`.
2. Bare filename (no `/`) → resolve, in order: (a) the most recent earlier citation in the doc whose basename matches (the established shorthand pattern, e.g. `readShowChangeFeed.ts:124-169` at `docs/superpowers/specs/2026-07-19-published-show-alerts.md:98` after the full-path citation `lib/sync/feed/readShowChangeFeed.ts:286-310` at line 33 of the same doc); (b) unique basename match in the tracked-file list (`git ls-files`). Zero matches = `FAIL CITATION_FILE_MISSING`; >1 match with no earlier anchor = `FAIL CITATION_AMBIGUOUS` (lists candidates).
   <!-- spec-lint: ignore — next line deliberately names a file that no longer exists, as evidence -->
   Live proof of the failure class: that same shipped spec cites `PerShowAlertSection.tsx` (its line 89), whose file no longer exists in the tree — a stale citation this check would surface today, and this very line carries the §3 `ignore` waiver so the dogfood run stays clean.
3. Line checks on resolved files: `start` or `end` beyond EOF = `FAIL CITATION_LINE_OUT_OF_RANGE` (reports actual file length); `start > end` = `FAIL CITATION_RANGE_INVERTED`; `start = 0` = `FAIL CITATION_LINE_OUT_OF_RANGE` (citations are 1-indexed).

**Symbol proximity (advisory).** For each line-bearing citation: collect the other inline-code spans on the same doc line that match an identifier shape (`/^[A-Za-z_$][A-Za-z0-9_$.]*$/`, excluding other citations). If at least one candidate exists and none appears as a substring within the cited file's lines `[start-5, end+5]` (clamped to file bounds), emit `ADVISORY CITATION_SYMBOL_UNMATCHED`, printing the cited line(s) (first line of the range, trimmed to 160 chars) so the author can eyeball the claim. No candidates on the doc line → no advisory (nothing to match against).

## 5. Check 2 — numerics

Both parts advisory (`lib/specLint/numerics.ts`). Scanned over prose lines only (fenced regions excluded; numbers inside citation tokens excluded).

**Noun-anchored mismatch.** Extract `/(\d+)\s+([a-z][a-z-]{2,})/g` pairs; normalize the noun by lowercasing and stripping one trailing `s`. For each normalized noun bound to ≥2 distinct numbers, emit `ADVISORY NUMERIC_NOUN_MISMATCH` listing every occurrence (`doc line N: "45 codes"`). This intentionally over-reports ("3 rounds" vs "22 rounds" may both be true) — it is a review aid, not a gate, which is exactly why it is advisory (§1.1 item 1).

**Inventory.** Every integer or decimal literal on prose lines (same exclusions), grouped by value, each occurrence with doc line + context snippet (surrounding text trimmed to 80 chars). Emitted under an `INVENTORY` heading in the report. This is the mechanized replacement for the hand-built numeric-sweep transcript (`AGENTS.md:102`).

## 6. Check 3 — copy rules · Check 4 — sections

**Copy scan spans** (`lib/specLint/copyRules.ts`):

- Prose: text inside double-quote pairs — straight `"…"` and curly `“…”` — outside fenced regions. Unpaired quote on a line → that line is not scanned (no span).
- Fenced code blocks with a JS-family info string (`ts`, `tsx`, `js`, `jsx`, `mjs`, `cjs`, `json`): single-, double-, and backtick-quoted string literals (line-local regex extraction; template-literal interpolations are not parsed — the raw span text is scanned as-is). Other/absent info strings (`sql`, `sh`, ```` ``` ```` alone, output transcripts) are not scanned.

**Findings within scanned spans:**

- Em-dash `—` (U+2014): `FAIL COPY_EM_DASH` (`DESIGN.md:332` — "No em dashes."; hard per §1.1 item 6).
- `--` (double hyphen): `ADVISORY COPY_DOUBLE_HYPHEN` (banned by `DESIGN.md:332` but advisory per §1.1 item 6 — CLI-flag false positives).
- Straight apostrophe `'` inside **prose** double-quoted spans only: `ADVISORY COPY_STRAIGHT_APOSTROPHE` (uncodified curly-quote convention, §1.1 item 6). Not applied to code-fence string literals (code legitimately uses `'` everywhere).

**Sections** (`lib/specLint/sections.ts`), `kind = spec` only (plans skip, §1.1 item 8):

- A heading whose text matches `/resolved scope/i` must exist: absent = `FAIL SECTION_MISSING_RESOLVED_SCOPE`. Convention anchor: `docs/superpowers/specs/2026-07-19-published-show-alerts.md:21` ("### 1.1 Explicitly resolved scope decisions (do not relitigate)").
- **UI-spec detection:** any *resolved* citation path (§4) starting with `app/` but not `app/api/`, or starting with `components/` → the doc is a UI spec → headings matching `/dimensional invariants/i` and `/transition inventory/i` are both required; each absence is a separate `FAIL SECTION_MISSING_UI` finding. The `not-ui` waiver (§3) disables this entire sub-check; the waiver plus ≥1 UI citation is fine by design (that is the waiver's purpose — incidental citations).

## 7. Architecture & purity contract

```
scripts/spec-lint.ts          # CLI adapter: argv, read doc, real FileResolver, print, exit
lib/specLint/types.ts         # Finding {check, code, severity: 'fail'|'advisory', docLine, message, detail?}
lib/specLint/parse.ts         # document model (§3)
lib/specLint/citations.ts     # §4
lib/specLint/numerics.ts      # §5
lib/specLint/copyRules.ts     # §6 copy
lib/specLint/sections.ts      # §6 sections
lib/specLint/run.ts           # orchestrator: (docText, kind, resolver) → Finding[]
```

- **FileResolver injection:** `{ fileExists(path), readFileLines(path), listTrackedFiles() }`. The CLI adapter backs it with `node:fs` + `git ls-files`; tests back it with in-memory fixture maps. Nothing under `lib/specLint/**` imports `node:fs`, `node:child_process`, or `node:process` — pinned by a structural meta-test (§8), the same posture as the observe read-core pin (`tests/observe/_metaReadOnlyQueryCore.test.ts`).
- `run.ts` is deterministic: same doc + same resolver state → same findings, ordered by (check, docLine).

**Invariant applicability (declared per AGENTS.md):** no advisory locks (no DB), no email boundary, no §12.4 codes (finding codes like `CITATION_FILE_MISSING` are CLI-report identifiers printed by a dev tool, not user-visible app UI routed through `lib/messages/lookup.ts`), no mutation surface (invariant 10 N/A — pure read-only CLI, no HTTP route or server action), no Supabase calls (invariant 9 N/A), no UI files (invariant 8 / impeccable N/A, Opus-routing N/A).

## 8. Testing

All under `tests/specLint/` — auto-included by `BASE_INCLUDE` (`vitest.projects.ts:20`, `tests/**/*.test.ts`); runs in the standard unit-suite CI job, no new workflow wiring. TDD per task.

- **Per-check unit tests** with in-memory fixture docs + fixture resolver. Anti-tautology: every fixture plants a specific defect and the assertion names the exact expected finding code + doc line derived from the fixture's construction (never "some finding exists"). Clean-fixture tests assert `[]` for that check. Boundary/guard cases per §3–§6: empty doc, unterminated fence, citation at exactly EOF (pass) and EOF+1 (fail), `start=0`, inverted range, ambiguous bare filename, unpaired quote line, waiver with/without reason, waiver on same vs following vs distant line, plan-kind skipping sections, `app/api/` citation NOT triggering UI detection, `not-ui` waiver suppressing UI-section failures.
- **CLI adapter test:** spawn `tsx scripts/spec-lint.ts` against small committed fixture docs under `tests/specLint/fixtures/` (a failing doc, a clean doc, a no-kind-inferable path) asserting exit codes 1/0/2 and report shape. Fixture docs cite files inside `tests/specLint/fixtures/` themselves so citations stay stable forever (never live app code, which drifts).
- **Structural meta-test** `tests/specLint/_metaPureCore.test.ts`: walks every file under `lib/specLint/` (filesystem walk, fails-by-default for new files) asserting no `node:fs` / `node:child_process` / `node:process` imports.
- **Dogfood check (plan-time task):** run `pnpm spec:lint docs/superpowers/specs/2026-07-19-spec-lint.md` — this spec must itself exit 0 with no hard findings.
