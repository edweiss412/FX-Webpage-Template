# spec:lint — mechanize the checkable subset of spec/plan pre-review passes

<!-- spec-lint: ignore — bare BACKLOG.md means the repo-root file; docs/superpowers/plans/BACKLOG.md shadows the basename -->
**Date:** 2026-07-19 · **Backlog:** `BL-SPEC-LINT` (`BACKLOG.md:7-11`) · **Kind:** tooling (no UI, no DB, no runtime surface)

R1–R3 adversarial rounds are dominated by non-compliance with passes `AGENTS.md` already mandates (live-code citation grep, numeric sweep, copy rules, mandatory sections), not by rule gaps. This spec defines `pnpm spec:lint <doc>`: a deterministic linter for spec/plan markdown that hard-fails on the mechanically checkable subset and emits advisory reports (the transcript artifact) for the fuzzy remainder.

## 1. Scope

### 1.1 Explicitly resolved scope decisions (do not relitigate)

Items 1–9 ratified at the brainstorming gate (user-approved, 2026-07-19); items 10–12 ratified during adversarial round 1; items 13–16 during round 2; item 17 during round 4. Rounds 3–4 otherwise refined mechanics inside already-ratified items (backslash/drive-letter candidacy, segment-level `..` rule, resolved-anchor requirement, UTF-16 column semantics, run-input identity, EOF waiver stacks).

1. **Mixed posture.** Deterministic checks exit non-zero; fuzzy heuristics are advisory-only and never affect the exit code. Rationale: hard-failing heuristics get bypassed; a pure reporter re-permits non-compliance.
2. **No CI gate.** `spec:lint` is a point-in-time pre-dispatch tool. Line-number citations drift as code merges after the spec ships; wiring this into CI would false-fail historical docs forever. Enforcement is process-level: the review-dispatch discipline in `AGENTS.md:102` is amended to require attached `spec:lint` output.
3. **Symbol-proximity matching is advisory, not hard.** Prose legitimately cites a line for context without naming a symbol on the same spec line.
4. **Copy scan scope = quoted prose spans + JS-family fence content.** Spec prose itself may use em-dashes freely.
5. **UI-spec detection is automatic** (citation paths under `app/` excluding `app/api/`, or `components/`), waivable via `<!-- spec-lint: not-ui — <reason> -->`.
6. **`--` (double hyphen) and straight-apostrophe findings are ADVISORY, not hard**, even though `DESIGN.md:332` bans `--` in copy. Rationale: quoted CLI flags (`"--json"`) would hard-fail; and the curly-apostrophe convention is not codified in `DESIGN.md` — it exists only as test-comment convention (e.g. `tests/app/admin/revokeHang.test.tsx:151`). The em-dash class (U+2014, raw or encoded — §6) is the sole hard copy failure.
7. **No new dependencies.** Line/regex-based scanning plus small state machines; no remark/unified.
8. **Plans skip section-presence checks.** Citations, numerics, and copy checks run for both kinds.
9. **Unresolvable line-bearing citations are hard failures** (zero-match and ambiguous-without-anchor alike). A citation that cannot be resolved to exactly one file cannot be verified. Path-only bare-filename citations relax the ambiguity half (§4): ≥1 tracked basename match proves existence, and with no line number there is nothing further to check, so only zero matches is a failure there.
10. **Target corpus = newly authored docs at dispatch time; legacy-corpus compatibility is a NON-GOAL.** The existing corpus (637 tracked `docs/superpowers/**/*.md` files — `git ls-files 'docs/superpowers/**/*.md' | wc -l` → `637`, 2026-07-19) contains citation and fence forms this grammar rejects; those docs are never re-linted (item 2). What IS required is fail-closed behavior WITHIN the declared candidate domain (§4): a candidate span that does not parse is a hard `CITATION_MALFORMED` finding, never silent prose.
11. **Citations resolve only within the tracked-file set** (`git ls-files` at the repo root). Absolute paths, `..` segments, and backslashes are `CITATION_MALFORMED`; a well-formed path not in the tracked set is `CITATION_FILE_MISSING`. Tracked **symlinks** are refused at read time (§7 resolver contract) — the repo does track symlinks (e.g. `docs/superpowers/plans/2026-04-30-fxav-crew-pages-v1/handoffs/M11-user-facing-docs.md` is mode 120000 in `git ls-files -s`), so containment cannot rest on tracking alone. Net effect: the linter never reads content outside the repository.
12. **Line-local models.** Inline-code spans are line-local (CommonMark's multi-line inline code is unsupported — a deliberate simplification of this tool's contract, not a corpus claim). Prose quote pairing is line-local (§6). Multi-line prose quotations are outside the copy scan.
13. **The candidate domain is whitespace-free.** A span containing whitespace is never a citation candidate. A "citation" with spaces in its path is not expressible in this grammar; the mandated citation FORM (`file:line` in one unbroken span) is itself part of the discipline being enforced. This is a domain boundary, not a bypass (item 10 defines fail-closed relative to this domain).
14. **JS-family fences are scanned whole for the em-dash class** — no string-literal extraction. Comments, regex literals, template nesting, interpolation, and unterminated strings therefore need no defined semantics; an em-dash anywhere in a JS-family fence is a finding. False positives (an em-dash in an example code comment) are handled by the fence-level waiver (§3). This trades a small false-positive surface for the elimination of an entire undefined-behavior class.
15. **Numeric inventory groups by raw matched text** (`1`, `1.0`, `1.00` are distinct groups). Value-equivalence folding is heuristic territory and the inventory is an advisory transcript, not a gate.
16. **The linted doc itself need not be tracked** — the tool's purpose is pre-dispatch linting of not-yet-committed docs. The doc must live inside the repo working tree (§2); its content is trusted input (the operator chose it).
17. **Repo containment is the only HARD input boundary; `docs/superpowers/` residency is usage guidance, not CLI-enforced** (round 4). Rationale: the committed CLI test fixtures live under `tests/specLint/fixtures/docs/superpowers/` and must be lintable end-to-end through the real adapter; a path-prefix gate would make every exit-0/exit-1 fixture exit 2. Kind inference still keys on `/specs/`-vs-`/plans/` segments wherever the doc lives, and a doc without either segment simply requires `--kind`.

### 1.2 Out of scope

- Semantic verification that a cited line's *content* supports the prose claim (stays with the model's citation pass; the advisory printout assists it).
- Linting the legacy corpus (§1.1 item 10) and non-superpowers docs (guidance-scoped; the CLI enforces only repo containment, §1.1 item 17).
- Auto-fixing. Report-only.
- The apostrophe rule's codification into `DESIGN.md`.

## 2. CLI contract

```
pnpm spec:lint <path-to-doc.md> [--kind spec|plan] [--json]
```

**Arguments and flags (every violation below = usage error, exit 2, nothing linted):**

- Exactly one positional argument. It must end in `.md`, be a regular readable file (not a directory; not itself a symlink — checked via `lstat`), and realpath-resolve inside the repo root (§1.1 item 17 — repo containment is the hard boundary; `docs/superpowers/` residency is guidance). The repo root is discovered once, from the CLI's working directory (`git rev-parse --show-toplevel`) — not from the doc's directory, so a doc inside some other nested repository cannot swap the root.
- `--kind` takes exactly `spec` or `plan`. When present it WINS over path inference (explicit beats inferred — a doc under `specs/` can deliberately be linted as a plan; the report records this, §7). When absent: repo-relative path contains a `/specs/` segment and no `/plans/` segment → `spec`; `/plans/` and no `/specs/` → `plan`; both or neither → usage error naming `--kind`.
- `--json` toggles output shape (§7). Unknown flags, duplicated flags, and a repeated positional are usage errors.

**Exit codes:** `0` = clean or advisories only; `1` = one or more hard findings; `2` = usage or infrastructure error (unreadable doc, root discovery failure, `git ls-files` failure). Divergence from the observe CLI stated explicitly: observe collapses errors to exit 1 (`scripts/observe.ts:45`); `spec:lint` reserves 1 for findings so automation can distinguish "doc has problems" from "tool could not run."

**Streams:** report on stdout; usage/infra errors on stderr — a JSON object (`{"error": <message>}`) when `--json`, plain text otherwise, mirroring the observe convention (`scripts/observe.ts:40-45`).

**Precedent.** The observe CLI is the house pattern for a read-only tool whose logic lives behind an injected-dependency seam: `AGENTS.md:133` names `scripts/observe.ts` as "the CLI adapter" over the `lib/observe/query/**` read-core, and its command runner returns `{ stdout, stderr, exitCode }` (`scripts/observe.ts:111-113`) which the entry point prints and exits with (`scripts/observe.ts:259-263`). `spec:lint` follows the same shape one step further: `lib/specLint/**` is fully I/O-free (§7), and `scripts/spec-lint.ts` owns argv, fs, repo-root discovery, `git ls-files`, printing, and `process.exit`.

**Encoding/EOL:** the doc and cited files are read as UTF-8 with invalid sequences replaced (U+FFFD); CRLF is normalized to LF before line-splitting; a trailing final newline does not create a phantom last line (split on `\n`, drop one trailing empty element). A file's line count for EOF checks is the length of that array.

## 3. Document model (shared by all checks)

`lib/specLint/parse.ts` produces a line-indexed model:

- **Fenced regions** (CommonMark-subset state machine): a fence opens on a line whose first non-whitespace (≤3 spaces indent) is a run of ≥3 backticks or ≥3 tildes; the info string is the remainder of that line, normalized by trimming, lowercasing, and taking the first whitespace-delimited word. It closes on a line holding only a run of the SAME character at LEAST the opener's length (≤3 spaces indent). Runs longer than 3 nest correctly: a ````` ```` ````` fence containing ``` ``` ``` lines stays open until a ≥4-backtick closer. Unterminated fence → everything to EOF is fenced.
- **Inline-code spans** (line-local, §1.1 item 12): on non-fenced lines, a span opens with a run of N backticks and closes at the next run of exactly N backticks on the same line (CommonMark delimiter-run rule, so ``` `` … ` … `` ``` double-backtick spans containing single backticks parse correctly). Unclosed runs are literal text.
- **Headings:** ATX headings (`#`–`######`) with depth + text, non-fenced lines only.
- **Waivers:** a non-fenced line whose entire trimmed content is one HTML comment of the exact shapes below. Waiver-shaped text inside fenced regions or inline-code spans is inert. Recognized shapes:
  - `<!-- spec-lint: ignore — <reason> -->` — suppresses **hard** findings attributed to the waiver's TARGET: the next line that is non-blank and not itself a waiver. If the target line OPENS a fenced region, the waiver covers hard findings on every line of that region (this is how an example fence containing a deliberate em-dash stays clean, §1.1 item 14). Advisories are never suppressed (they don't gate).
  - **Stacking:** a waiver's TARGET skips blank lines and other waiver lines, so several `ignore` waivers can share one target even when separated by blank lines or an interleaved `not-ui` waiver — a stack is the maximal run of waiver/blank lines ending immediately before the target. A stack that reaches EOF has NO target: every `ignore` waiver in it emits `WAIVER_UNUSED` (a `not-ui` waiver there is unaffected — it is doc-global). The target's hard findings are suppressed once; EVERY `ignore` waiver in that stack is marked *used* iff the target produced ≥1 suppressed finding, otherwise EACH emits advisory `WAIVER_UNUSED` (stale waivers rot; surfaced, not gated). Deterministic by construction — no processing-order dependence.
  - `<!-- spec-lint: not-ui — <reason> -->` — disables UI-section requirements (§6). The FIRST such waiver (lowest doc line) is the active one; any later duplicate emits `WAIVER_UNUSED`.
  - Empty `<reason>` → hard `WAIVER_MISSING_REASON`, which is **unsuppressible** (no waiver, including itself or a preceding one, can suppress it — otherwise an empty waiver would launder itself).

Guard conditions: empty doc (zero bytes or only blank lines) → hard `EMPTY_DOC` at docLine 1, column 1.

## 4. Check 1 — citations

**Candidate domain (fail-closed within it, §1.1 items 10/13).** Only inline-code spans on non-fenced lines are considered (fenced regions carry example output and hypothetical code whose paths are illustrative). A span whose content contains whitespace, a backtick, or any glob/expansion char (`*`, `{`, `$`) is never a candidate. **Backslash-bearing spans** get their own rule, ordered before the two forms below and covering every Windows-flavored shape in one place. A span containing a backslash (or opening with a drive-letter shape — an ASCII letter, a colon, then `/` or a backslash) is a candidate, and always hard `CITATION_MALFORMED`, iff BOTH hold: (a) it consists solely of path-safe characters — letters, digits, `_`, `.`, `-`, `,`, `(`, `)`, `[`, `]`, the colon, and the two separators `/` and backslash — so regex-flavored spans containing `+`, `?`, `|`, `^`, or quotes stay prose; and (b) it shows citation intent — it contains a colon ANYWHERE (any suffix: digits, `L12`, empty, non-numeric), OR it has a dotted final segment when split on both separators. This catches "dir\file.ts:12", "dir\file.ts:L12", "dir/sub\file.ts:", "dir\sub/file.ts", "C:dir\file.ts" (colon, path-only drive-relative), and UNC "\\server\share\file.ts:abc", while escape-sequence spans like `\u2014` (no colon, final segment "u2014" undotted) remain prose. Among the remaining backslash-free spans, a span is a **citation candidate** iff:

- **colon form:** it contains at least one colon, and the prefix before the FIRST colon either (a) contains `.` or `/` and is not purely digits (excludes clock times like `12:30` and ratios like `3:1`), or (b) is empty (catches the colon-only shorthand ":22" — always malformed, write the full path), or (c) is a single ASCII letter — the drive-relative shape with no backslash ("C:file.ts", "C:dir/file.ts:12", "C:file.ts:L12") — which is always hard `CITATION_MALFORMED` (a one-letter path resolves nowhere and the double-colon forms fail the grammar; the rare single-letter prose span like "k:2" is a waiver case). EVERYTHING after the first colon participates in well-formedness below — "Foo.ts:L12", "Foo.ts:", "x.ts:12,34", "x.ts:12:14" are all candidates and all malformed, never silent prose; or
- **path-only form (slashed):** no colon, contains `/`, and its final segment contains a `.` (`lib/messages/lookup.ts` yes; branch name `feat/spec-lint` no); or
- **path-only form (bare):** no colon, no `/`, and it matches `basename.ext` where `ext` is in the allowlist `ts tsx js jsx mjs cjs mdx md sql sh css json yml yaml toml html` (`canonicalize.ts` yes; `lookup.ts` yes; `v1.2` no).

Known accepted false-positive class: host:port-shaped spans ("example.com:8080") are candidates and will fail as malformed or missing; the §3 `ignore` waiver covers the rare legitimate case.

**Well-formedness.** A colon-form candidate must match `path COLON start [HYPHEN end]` where `start`/`end` are `[1-9][0-9]*` (single line or single closed range) and `path` is non-empty and passes the path rule; anything else = hard `CITATION_MALFORMED`. **Path rule** (shared by colon-form and both path-only forms): no leading `/`, no backslash anywhere, and no slash-delimited SEGMENT equal to `..` or `.` — checked by splitting on `/`, so a dotted filename like "foo..bar.ts" stays legal while a traversal segment never does.

**Resolution** (`lib/specLint/citations.ts`, injected resolver — §7):

1. Slashed path (colon form or path-only) → must be an exact member of the tracked-file set. Not tracked = `FAIL CITATION_FILE_MISSING` (also the containment boundary, §1.1 item 11).
2. Bare filename, colon form (line-bearing) → resolve, in order: (a) the most recent earlier citation in the doc that itself RESOLVED to exactly one tracked file and whose basename matches — unresolved, missing, or ambiguous earlier citations can never anchor (the established shorthand pattern, e.g. `readShowChangeFeed.ts:124-169` at `docs/superpowers/specs/2026-07-19-published-show-alerts.md:98`, anchored by the resolved full-path citation `lib/sync/feed/readShowChangeFeed.ts:286-310` at line 33 of the same doc); (b) unique basename match in the tracked-file set. Zero matches = `FAIL CITATION_FILE_MISSING`; >1 match with no anchor = `FAIL CITATION_AMBIGUOUS` (detail lists candidates).
3. Bare filename, path-only form → ≥1 tracked basename match proves existence (no uniqueness demand — no line to verify, §1.1 item 9); zero matches = `FAIL CITATION_FILE_MISSING`.
   <!-- spec-lint: ignore — next line deliberately names a file that no longer exists, as evidence -->

This very line carries the §3 `ignore` waiver so the dogfood run stays clean, and the waiver is *used* (the finding on this line is suppressed), exercising §3 end-to-end.
4. Line checks on resolved line-bearing citations: resolver returns no content for a tracked-but-unreadable path or a tracked symlink (§7) = `FAIL CITATION_UNREADABLE` (detail says which); `start` or `end` beyond EOF (§2 line-count rule) = `FAIL CITATION_LINE_OUT_OF_RANGE` (reports actual file length); both may co-occur with `start > end` = `FAIL CITATION_RANGE_INVERTED` (a range can be simultaneously inverted and out of range; both findings emit).

**Symbol proximity (advisory).** Runs ONLY for line-bearing citations with zero hard findings (resolved, readable, in-range, non-inverted — invalid coordinates have no well-defined window or printable first line, so those citations get their hard finding and nothing else). For each such citation: collect the other inline-code spans on the same doc line that match an identifier shape (`/^[A-Za-z_$][A-Za-z0-9_$.]*$/`, excluding citation candidates). If at least one exists and none appears as a substring within the cited file's lines `[start-5, end+5]` (clamped to file bounds), emit `ADVISORY CITATION_SYMBOL_UNMATCHED`, printing the first cited line trimmed to 160 chars. No identifier spans on the doc line → no advisory.

## 5. Check 2 — numerics

Both parts advisory (`lib/specLint/numerics.ts`). Scanned over non-fenced lines only; matches inside citation-candidate spans are excluded.

**Numeric lexicon (deterministic):** matches of `/\b\d+(?:\.\d+)?\b/` EXCEPT those inside: ISO dates (`\d{4}-\d{2}-\d{2}`), version strings (`v?\d+\.\d+\.\d+`), clock times (`\d+:\d+`), hex literals (`0x[0-9a-fA-F]+`). No sign, exponent, or thousands-separator handling — `1,000` scans as `1` and `000`, accepted noise in an advisory-only channel.

**Noun-anchored mismatch.** From lexicon matches followed by whitespace + `/[a-z][a-z-]{2,}/`, form (number, noun) pairs; normalize the noun by lowercasing and stripping one trailing `s`. For each normalized noun bound to ≥2 distinct raw numbers, emit ONE `ADVISORY NUMERIC_NOUN_MISMATCH` anchored at the FIRST occurrence's (docLine, column), with every occurrence listed in `detail` (`doc line N: "45 codes"`). Intentionally over-reports ("3 rounds" vs "22 rounds" may both be true) — a review aid, not a gate (§1.1 item 1).

**Inventory.** Every lexicon match becomes an occurrence `{ docLine, column, snippet }` (snippet per the §7 definition: a raw slice of the line spanning code units [column-41, column+40), clamped to the line, no ellipsis). Occurrences group by RAW matched text (§1.1 item 15); groups order by numeric value (`Number(raw)`) then raw text; occurrences within a group order by (docLine, column). NOT findings: emitted as the `inventory` array in `--json` and an `INVENTORY` section in the text report; never counted in the summary line (§7). This is the mechanized replacement for the hand-built numeric-sweep transcript (`AGENTS.md:102`).

## 6. Check 3 — copy rules · Check 4 — sections

**Copy scan spans** (`lib/specLint/copyRules.ts`):

- **Prose quotes (line-local, §1.1 item 12):** on non-fenced lines, straight `"` characters pair sequentially left-to-right (1st with 2nd, 3rd with 4th, …); curly `“` pairs with the next `”`. Straight and curly never pair with each other. Leftover unpaired delimiters of either style → `ADVISORY COPY_UNPAIRED_QUOTE` per leftover, and text after an unpaired opener is not scanned (surfaced instead of silently skipped). Markdown prose has no quote-escape concept: a backslash before a quote is literal text, no special-casing.
- **JS-family fences, scanned WHOLE (§1.1 item 14):** fenced regions whose normalized info string (§3) is one of `ts`, `tsx`, `typescript`, `js`, `jsx`, `javascript`, `mjs`, `cjs`, `json` are scanned line-by-line in full — no literal extraction, no comment/regex/template semantics. Other/absent info strings (`sql`, `sh`, bare fences, output transcripts) are not scanned.

**Findings within scanned spans:**

- **Em-dash class** (hard, `DESIGN.md:332` "No em dashes."): raw `—` (U+2014) and its encoded spellings — `&mdash;`, the decimal NCR `&#8212;`, the hex NCR `&#x2014;` (case-insensitive), and the JS escapes `\u2014` and `\u{2014}` — each occurrence = `FAIL COPY_EM_DASH`. Encoded forms count in BOTH span types (an encoded em-dash in copy renders as an em-dash).
- `--` (double hyphen): `ADVISORY COPY_DOUBLE_HYPHEN`, **prose quote spans only** (banned by `DESIGN.md:332` but advisory per §1.1 item 6; not scanned in fences, where `--flag` tokens are pervasive).
- Straight apostrophe `'`: `ADVISORY COPY_STRAIGHT_APOSTROPHE`, **prose quote spans only** (uncodified curly-quote convention, §1.1 item 6; code legitimately uses `'`).

**Sections** (`lib/specLint/sections.ts`), `kind = spec` only (plans skip, §1.1 item 8); missing-section findings report docLine 1, column 1:

- A heading whose text matches `/resolved scope/i` must exist: absent = `FAIL SECTION_MISSING_RESOLVED_SCOPE`. Convention anchor: `docs/superpowers/specs/2026-07-19-published-show-alerts.md:21` ("### 1.1 Explicitly resolved scope decisions (do not relitigate)").
- **UI-spec detection:** any *resolved* citation path (§4) starting with `app/` but not `app/api/`, or starting with `components/` → the doc is a UI spec → a heading matching `/dimensional invariants/i` is required (absent = `FAIL SECTION_MISSING_DIMENSIONAL_INVARIANTS`) and a heading matching `/transition inventory/i` is required (absent = `FAIL SECTION_MISSING_TRANSITION_INVENTORY`) — distinct codes, so ordering never ties (§7). The `not-ui` waiver (§3) disables this entire sub-check; the waiver plus ≥1 UI citation is fine by design (that is the waiver's purpose — incidental citations).

## 7. Architecture, result contract & purity

```
scripts/spec-lint.ts          # CLI adapter: argv, read doc, real FileResolver, print, exit
lib/specLint/types.ts         # Finding, InventoryGroup, LintResult
lib/specLint/parse.ts         # document model (§3)
lib/specLint/citations.ts     # §4
lib/specLint/numerics.ts      # §5
lib/specLint/copyRules.ts     # §6 copy
lib/specLint/sections.ts      # §6 sections
lib/specLint/run.ts           # orchestrator: (doc, resolver) → LintResult; exitCodeForResult()
```

**Result contract.**

```ts
type Severity = "fail" | "advisory";
type Check = "document" | "citations" | "numerics" | "copy" | "sections";
interface Finding {
  check: Check;
  code: string;
  severity: Severity;
  docLine: number;   // 1-based; whole-doc findings (EMPTY_DOC, missing sections) use 1
  column: number;    // 1-based UTF-16 code-unit offset of the trigger (JS string index + 1) - applies to EVERY finding kind; whole-doc findings use 1
  message: string;
  detail?: string;
}
interface InventoryOccurrence { docLine: number; column: number; snippet: string } // column: same UTF-16 code-unit semantics as Finding.column; snippet: raw slice of the line spanning code units [column-41, column+40), clamped to the line, no ellipsis
interface InventoryGroup { raw: string; occurrences: InventoryOccurrence[] }
interface LintResult {
  doc: string;                       // repo-relative path of the linted doc
  kind: "spec" | "plan";
  kindSource: "inferred" | "explicit";
  findings: Finding[];
  inventory: InventoryGroup[];
}
```

Finding ordering is total and deterministic: by check (`document`, `citations`, `numerics`, `copy`, `sections`), then docLine, then column, then code — and no two findings share all four keys (per-token findings differ in column; whole-doc findings differ in code, §6). `exitCodeForResult(result)` is a pure lib function: any `severity: "fail"` → 1, else 0; the adapter alone maps usage/infra conditions to 2 and calls `process.exit`. The text report opens with a header naming `doc`, `kind`, and `kindSource` (the compliance artifact must show WHAT was linted and whether spec checks ran — an explicit `--kind plan` on a specs-path doc is visible, not silent), groups findings by check with `FAIL`/`ADVISORY` tags, prints `INVENTORY` last, and ends with `N hard, M advisory` counting finding occurrences (inventory excluded). `--json` emits `LintResult` verbatim.

**FileResolver contract:** `{ readFileLines(path): string[] | null, listTrackedFiles(): string[] }`, rooted at the adapter-discovered repo root. `readFileLines` returns `null` when the tracked path is unreadable OR is a symlink (adapter checks `lstat` BEFORE opening — tracked symlinks exist, §1.1 item 11, and must not be followed); the core maps `null` to `CITATION_UNREADABLE`. Either method THROWING is an infrastructure fault: the adapter catches, prints to stderr, exits 2 — never a finding. (No `fileExists` member: existence is tracked-set membership, checked by the core against `listTrackedFiles()`.) The core calls `readFileLines` only for paths already validated as tracked (§1.1 item 11); a spy test pins that untracked paths are never read.

Nothing under `lib/specLint/**` imports `node:fs`, `node:child_process`, or `node:process` — pinned by a structural meta-test (§8), the same recursive-walk posture as the observe read-core pin (`tests/observe/_metaReadOnlyQueryCore.test.ts:11` states the recursive-walk rationale; the suite is at `tests/observe/_metaReadOnlyQueryCore.test.ts:22`).

`run.ts` takes `doc: { text: string; repoRelPath: string; kind: "spec" | "plan"; kindSource: "inferred" | "explicit" }` — the adapter owns producing all four (read + decode, path relativization, kind resolution), and the core copies `repoRelPath` (as `doc`), `kind`, and `kindSource` into `LintResult` verbatim; that is how the result carries identity the raw text alone cannot supply. Deterministic: same `doc` value + same resolver state → same `LintResult`.

**Invariant applicability (declared per AGENTS.md):** no advisory locks (no DB), no email boundary, no §12.4 codes (finding codes like `CITATION_FILE_MISSING` are CLI-report identifiers printed by a dev tool, not user-visible app UI routed through `lib/messages/lookup.ts`), no mutation surface (invariant 10 N/A — read-only CLI, no HTTP route or server action), no Supabase calls (invariant 9 N/A), no UI files (invariant 8 / impeccable N/A, Opus-routing N/A).

## 8. Testing

All under `tests/specLint/` — auto-included by `BASE_INCLUDE` (`vitest.projects.ts:20`, `tests/**/*.test.ts`) and therefore run by the sharded unit-suite CI job (`.github/workflows/unit-suite.yml:80` runs `vitest run --shard` over the default include set); no new workflow wiring. TDD per task.

- **Per-check unit tests** with in-memory fixture docs + fixture resolver.
